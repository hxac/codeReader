`link_openfpga_arch` 是 `vpr` 与 `build_fabric` 之间的桥梁：它把 `read_openfpga_arch` 解析出的悬空名字（电路模型名、pb_type 路径等）在 VPR 已建好的 device 里反查、对账，落实成 `t_pb_type*` 指针与 `CircuitModelId`。其主线 `link_arch_template` 是一个 17 步线性编排器，顺序即依赖：先做物理瓦片与引脚查找，再 `annotate_pb_types`、`annotate_pb_graph`，随后绑定 rr_switch/rr_segment/direct 的电路模型，构建 `DeviceRRGSB`（依赖自维护的反向边映射 `RRGraphInEdges`，因 VPR 只存出边）、`MuxLibrary` 与 `TileDirect`，最后做聚类/布局标注及可选的 activity、仿真与比特流设置。最复杂的 `annotate_pb_types` 按 physical mode→interconnect type→physical pb_type→circuit model→mode bits 五段推进，每段走「显式反查 + 隐式推断 + check 闸门」套路；单模式 pb_type 可自动推断 physical mode，多模式必须显式声明。链接结果几乎全部汇入 `VprDeviceAnnotation`——一个不动 VPR 源码、以 VPR 对象指针为键平行挂载的中央账本（存电路模型、physical mode、mode bits、operating↔physical 配对的 index factor/offset 等），体现贯穿 OpenFPGA 的 annotation 模式。命令依赖机制将 link 列为 `build_fabric` 等命令的硬前置，缺它即在 shell 层被拦下。关键选项包括 `--activity_file`（推断时钟周期数或启用 FPGA-SPICE 时必需）与 `--sort_gsb_chan_node_in_edges`（影响 GSB 入边顺序与下游可复现性）。
