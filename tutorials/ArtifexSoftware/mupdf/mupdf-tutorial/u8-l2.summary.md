本讲（u8-l2 压缩编解码器）承接 u8-l1 的流与过滤管线，钻进「洋葱」的每一层，讲解 MuPDF 如何把第三方流式解码库（zlib、libjpeg 等）塞进 `fz_stream` 统一外壳。核心结论是：封装任何第三方解码库都遵循统一的四步模式——状态三件套（下层 `chain` 指针 + 库上下文 + 本地输出缓冲）→ 库要字节时从 chain 拉（`fz_available`/`fz_read_byte`，可零拷贝借用 `chain->rp`）→ 解出字节回填自己的缓冲、令 `stm->rp/wp` 指向它并返回 `*stm->rp++`→ 构造用 `keep_stream`、析构逆序 `drop` 且 `close_*` 不得抛异常。本讲覆盖三个最小 filter 模块：`filter-basic.c`（ASCIIHex/ASCII85/RunLength，纯 C 状态机，无第三方库，rld 内置 RLE 炸弹检测）、`filter-flate.c`（zlib inflate 最小范例，零拷贝喂入、对坏数据宽容）、`filter-dct.c`（libjpeg 最复杂封装，需装 source manager 桥接拉模型、error manager 把 longjmp 转 `fz_throw`，并处理 CMYK 反相与 lazy init）。关键术语与概念包括：filter 即 `fz_stream`、`next` 回调契约、`chain`、源/错误管理器、`fz_new_stream` 构造、按名分发的两条路径（PDF 层 `build_filter` 把 `FlateDecode`/`DCTDecode` 等滤镜名、通用层 `fz_open_image_decomp_stream` 把 `FZ_IMAGE_FLATE/JPEG` 等类型枚举，统一翻成 `fz_open_*` 构造函数，使 PDF/PNG/TIFF 共享同一套编解码实现）。库的内存经钩子统一走 `fz_malloc` 并带 Memento 标签。
