本讲覆盖 MuPDF 的三个交互模块：AcroForm 表单、标注与 PDF JavaScript。核心结论有四：一、AcroForm 字段以字典树挂在 /Root/AcroForm/Fields 下，由可继承的 /FT 与 /Ff 标志位推断出八种控件类型（pdf_widget_type 枚举），/V 存值、/DV 存默认值、/T 拼成点分全名，pdf_set_field_value 按类型分发写入并标脏（文本类走校验，复选类改 /AS）。二、表单控件本质是 /Subtype 为 /Widget 的标注，挂于页面独立的 page->widgets 链表，区别于普通标注的 page->annots。三、延迟外观合成：改属性只设脏标记（needs_new_ap、resynth_required），真正的 /AP 外观流到 pdf_update_annot 或 pdf_update_page 时才由 pdf-appearance.c 合成。四、MuPDF 有两套独立 mujs 引擎：pdf-js.c 是文档内嵌的 Acrobat 风格引擎，由 pdf_enable_js 开启、受 FZ_ENABLE_JS 门控并设 CPU/内存限额；murun.c 则是脚本化整个 C 库的 mutool run 宿主引擎，二者经 PDFDocument.enableJS() 衔接。字段与标注的 /AA/*/JS 事件由 pdf_execute_action_chain 读取，按 event_init→execute→result 三段式与 JS 交互，关闭 JS 时事件被静默跳过但表单仍可用。
