本讲讲解 MuPDF 的对象缓存 `fz_store`：它挂在 context 家族上，跨文档、跨页面、跨线程共享，作用是把已解码的可复用对象（如 pixmap）暂存以避免重复解码，内存吃紧时按 LRU 丢弃。其内部结构是「LRU 双向链表 + 4096 桶哈希表 + 大小计数」，全部由 `FZ_LOCK_ALLOC` 一把锁保护，故天然线程安全。上限由 `fz_new_context` 第三参数决定：`FZ_STORE_DEFAULT`（256 MiB）或 `FZ_STORE_UNLIMITED`（0，永不驱逐）。存取建立在三件套上：`fz_store_type` 虚表描述「如何 keep/drop/比较/哈希键」（让 store 容纳任意类型），值用专属 `drop` 函数指针当「类型身份证」，`fz_find_item` 命中返回的指针已增引用、必须配对 drop；`fz_store_item` 返回非 NULL 表示已被缓存、调用方应改用缓存值（多线程去重）。回收有三条路径：插入时 `ensure_work`（两遍扫描 + 摘链延迟释放），分配失败时 `fz_store_scavenge`（16 阶段逐步加压、挑最大块驱逐、`scavenging` 标志防递归），以及显式 `fz_shrink_store`；驱逐只能针对 `refs == 1` 的项。最后引入 `fz_key_storable`（store_key_refs 计数）与 `reap` 机制：当某对象 `store_key_refs == refs`（只剩 store 内部键引用）时收割孤儿项，可用 `fz_defer_reap_start/end` 批处理。本讲建立的「共享 store + 单锁」模型是 u9-l2 多线程渲染的基础。
