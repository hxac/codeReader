本讲下钻 createManager 如何用 manager.Options 装配 controller-runtime 的 Manager——它聚合 Cache、Client、Leader Election、Metrics/Health Server、Runnables 等所有控制器共享的基础设施。init() 将要 watch 的资源注册进 scheme（未注册者无法缓存），控制器设 NeedLeaderElection=false 让非 leader 副本也跑控制器做热备，并为 Pod status.podIP 建 field index 供后续校验 Agent 连接。核心结论是缓存裁剪：cache transform 在对象进缓存前删字段、对业务透明，分三层——DefaultNamespaces 限定命名空间（并兜底加入控制面自身 namespace）、DefaultTransform 剥所有资源 managedFields、ByObject 对 GatewayClass/Secret/ConfigMap 做字段级裁剪（无目标键命中时 Data 置 nil 但保留对象壳）。其正确性前提是 transform 保留的键集合与业务读取的键常量同源、新增键须两边同步。metrics 默认 BindAddress 为 0 即禁用、按端口 9113 开启，health 默认 8081，均可经反向布尔 flag 关闭；readyz 在首次图构建完成而非进程启动时才就绪，避免配置未下发即接流量。该 Manager 即 u3-l3 控制器注册的挂载容器。
