本讲承接 u7-l1，讲清控制面把 NGINX 配置「文件化」后如何按数据面 Deployment 组织、并把同一份配置一致下发给每个副本。围绕 `internal/controller/nginx/agent/` 下三个最小模块展开：`Deployment`/`DeploymentStore`、`FileService`、`CommandService`，辅以 `DeploymentBroadcaster` 与 `ConnectionsTracker`。

引入的关键术语：`Deployment`（控制面内部按工作负载名索引的「配置 + 广播器 + 每 Pod 错误」聚合体，非 Kubernetes 对象）、`configVersion`（对全量文件元数据算出的内容驱动版本号）、`FileOverview`/`File`、`ConfigApplyRequest`、`unmanaged` 文件标记。

核心结论：① 配置按 Deployment 组织而非按 Pod，多副本共享单一权威文件集，多副本一致性被收敛为单一数据源问题；② 采用「推清单（只含元数据 + configVersion）+ Agent 反向拉内容（`GetFile`/`GetFileStream`）」的混合传输，hash 命中跳过、大文件按 2 MiB 分块；③ 内容不变时 `configVersion` 相同、`rebuildFileOverviews` 返回 `nil`，实现零广播零 reload；④ `CommandService` 只管连接生命周期（`CreateConnection → UpdateDataPlaneStatus → Subscribe`），reload 是 Agent 处理 `ConfigApplyRequest` 的副作用，控制面不下发 reload 命令；⑤ 多 Pod 一致性三件套：单一 Deployment 对象、`Broadcaster.Send` 的「等待全部响应」同步屏障、handler 贯穿整条事务的 `FileLock`，新 Pod 的「订阅 + 初始配置」同锁以防漏更新。每 Pod 错误经聚合后入 statusQueue 并置 `NginxConfigPushed: true`，衔接后续状态回写讲义。
