本讲讲解 NGF 两个一次性辅助子命令 `initialize` 与 `generate-certs`，二者分别跑在数据面 Pod 的 init 容器与 Helm 的 pre-install/pre-upgrade Job 中，共同保证主进程启动时「证书就位、配置就位」。`initialize` 按下标一一对应拷贝 `--source`/`--destination` 初始文件，仅在 NGINX Plus 下额外把来自 downward API 的 `POD_UID`/`CLUSTER_UID` 序列化为 `deployment_ctx.json`（Plus 授权上下文）。`generate-certs` 自签一份 RSA 2048 位、3 年有效期的 CA，再用同一 CA 签发服务端与客户端证书，分别写入 `server-tls` 与 `agent-tls` 两个 Secret，二者共享同一份 `ca.crt` 构成 mTLS 共同信任根。本讲引入的关键术语有：init 容器、mTLS、SAN、deployment context、`RequireAndVerifyClientCert` 强制客户端校验，以及控制面（gRPC 服务端，持 `server-tls`）与数据面 Agent（gRPC 客户端，持 `agent-tls`）的角色划分。核心结论：`generate-certs` 生产信任材料，`initialize` 摆好静态文件，落盘统一走 `OSFileManager` 接口。
