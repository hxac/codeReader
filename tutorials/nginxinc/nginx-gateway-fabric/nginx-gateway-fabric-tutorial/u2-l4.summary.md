本讲聚焦 NGF 控制面在正式工作前的参数校验安全闸，将校验拆为两层：第一层「赋值即校验」，第二层「运行期一致性校验」。涉及三个最小模块。一是校验规则集合，位于 validation.go，是一组无状态纯函数，按命名、网络地址、端口、命令参数分类，强调可单测、可复用（如资源名校验被多个 flag 共用），命名校验复用 K8s 的 DNS-1123 与 Qualified Name 规则，控制器名则焊死在 NGF 固定域名下。二是 pflag Value 自定义类型，位于 validating_types.go，包括 stringValidatingValue、stringSliceValidatingValue、intValidatingValue，通过实现 pflag.Value 接口、在 Set 方法里先校验后赋值，把非法值挡在写入门外，即「准入式校验」；slice 类型用 changed 标志实现「首次覆盖、后续追加」语义。三是跨 flag 一致性校验，用 ensureNoPortCollisions（变参与 map 去重）做端口冲突检测，置于 RunE 最前；条件必填（开启 Plus 须提供 usage-report-secret）与跨命名空间一致性（PLM Secret）分别在 RunE 和 PreRunE 完成。关键结论：默认值不经校验器、合法性由开发者自负；必填检查交给 pflag 的 MarkFlagRequired；两套端口范围区分是否允许特权端口。
