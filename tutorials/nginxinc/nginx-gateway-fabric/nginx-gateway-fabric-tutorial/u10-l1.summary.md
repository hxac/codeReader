本讲是 WAF（F5 NGINX App Protect）集成单元第一讲，聚焦数据模型与 bundle 来源选择，不涉及运行时拉取链路。核心 CRD 是 WAFPolicy，一个 Inherited Attached Policy（继承型附着策略），通过 targetRefs 附着到 Gateway/HTTPRoute/GRPCRoute 并向下继承；它只负责"指路"——用 type 声明来源、用 securityLogs 配日志，规则细节在编译后的 bundle（.tgz，对应 NAP v5）里而不在 CRD 内。

关键术语：spec.type 是四值枚举。四种来源分两类——直接拉 URL（HTTP/NIM/N1C，用 policySource + 同命名空间 Secret 鉴权）与引用 CRD（PLM，用 policyRef.apPolicyRef + 集群级 --plm-storage-* S3 配置）。鉴权头各异：HTTP 用 Basic、NIM 用 Bearer、N1C 用 APIToken、PLM 用 S3 access/secret key。字段间的互斥配对由 CEL 写入 kubebuilder 注解，由 API server 准入层拒绝。

PLM 模式下，APPolicy/APLogConf（appprotect.f5.com 组，由 PLM 组件管理）的 status 经 apis/waf/v1 轻量类型反序列化为 BundleStatus，其 State 为 pending→processing→ready/invalid 状态机，仅 ready 带 S3 location 与 sha256。bundle 完整性靠统一 SHA-256 校验：HTTP 用 expectedChecksum 或 verifyChecksum，NIM/N1C 用平台 API 返回的 hash 自动校验，PLM 用 APPolicy status 的 sha256。失败行为映射到 Conditions：拉取失败→FetchError、校验失败→IntegrityError、未就绪→Pending（fail-closed，withhold 配置下发）、旧 bundle 兜底→StaleBundleWarning、PLM 未配置→Accepted/Invalid。
