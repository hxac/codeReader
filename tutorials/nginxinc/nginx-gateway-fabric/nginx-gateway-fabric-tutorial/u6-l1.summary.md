本讲讲解配置生成器：控制面里把领域无关的渲染模型 `dataplane.Configuration` 翻译成一组 NGINX 文件的唯一组件。核心抽象是 `Generator` 接口（仅用于测试，生产用无状态的 `GeneratorImpl`，持 `plus`/`usageReportConfig`/`logger` 三字段），其主方法 `Generate` 按三段式产出 `[]agent.File`：先写密钥类文件（PEM/bundle/auth，绕过模板链直接 append，权限更严），再走模板执行链，最后写 bundle/cert/auth。

模板执行链是核心机制：12 个 `executeFunc` 各吃整份配置、吐 `[]executeResult{dest, data}`，由 `executeConfigTemplates` 用 `map[string][]byte` **按 dest 分桶合并**——servers/upstreams/maps/split_clients/base_http/telemetry 六个函数都写同一份 `http.conf`，stream 侧同理拼成 `stream.conf`，便于 Agent 做 hash 比对与增量下发。关键术语：`executeResult`、`executeFunc`、分桶合并、复合策略生成器。

目录与 NGINX 上下文严格对应：`main-includes`→main、`events-includes`→events、`conf.d`→http、`stream-conf.d`→stream，被静态 `nginx.conf` 的四条 `include` 通配符拉入对应上下文；而 `secrets`/`includes`/`bundles` 是「数据目录」，不被 include、靠具体指令按绝对路径引用，includes.go 负责按文件名去重。本讲只讲总装与目录组织，server/upstream 块内部指令细节留给 u6-l2、u6-l3、u6-l4，产物下发交给 u7-l1 的 NginxUpdater。
