本讲聚焦 internal/framework/events，讲清事件管线的「中段」：事件的收集、批处理、派发。引入三个抽象：EventBatch（[]any，批处理最小单元）、事件类型（UpsertEvent / DeleteEvent / WAFBundleReconcileEvent）、按批消费的 EventHandler 接口；eventCh 由控制器与 WAF poller 多源汇聚、由 EventLoop 单消费者读取。

核心结论：EventLoop 用双缓冲（currentBatch 与 nextBatch）做批处理——处理一批事件（通常触发一次 NGINX reload，≥200ms 且有副作用）期间，新事件堆进 nextBatch，处理完后翻转合并成一批，把 N 次资源变更压缩成远少于 N 次 reload；handler 按值拿到稳定快照，主循环只写 nextBatch，二者底层数组不重叠，故无需加锁。启动前 FirstEventBatchPreparer 主动 list/get 集群当前状态生成「首批事件」，确保首次生成配置基于完整视图、避免瞬态 404，失败则控制面启动失败。关键约定：EventBatch 允许重复事件，下游 EventHandler 必须幂等，同 Generation 的重复 upsert 不触发重配（呼应 u4-l4）。本讲止于事件被派发，事件如何生成 NGINX 配置留待 u4-l3。
