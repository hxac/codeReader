本讲聚焦 NGF 控制面装配的「注册阶段」——在 Manager 建好后、处理器消费事件之前，决定 watch 哪些资源、启用哪些控制器。核心载体是 `ctlrCfg` 注册表单元：`objectType`（watch 谁）、`options`（predicate/field index/filter/only-metadata）、`requireCRDCheck`+`crdGVK`（是否先探测 CRD）、`name`（控制器命名，多实例同 Kind 须手写）。`registerControllers` 把所有需求收集成切片，经框架层 `controller.Register` 统一注册，后者挂谓词、配 field index、用自研 Reconciler 把资源变更翻译成 `UpsertEvent`/`DeleteEvent`。CRD 存在性发现分两层：底层 `crd.Checker.CheckCRDsExist` 用 API discovery 探测一组 GVK，并按 GroupVersion 分组把 O(n) 次调用压成 O(g) 次，「不存在」视为正常状态；上层 `filterControllersByCRDExistence` 据此过滤注册表，并对必需的 ReferenceGrant 做 v1→v1beta1 回退。特性开关（`Plus`/`ExperimentalFeatures`/`Snippets`/`PLMStorageConfig`/`InferenceExtension`）与 CRD check 构成叠加双闸门，flag 表意图、CRD 表能力，InferencePool 为靠 flag 单独控制的例外。产出的 `discoveredCRDs` 还流向变更处理器与首批事件 list 准备，是连接注册阶段与下游事件/图构建阶段的纽带。新增 watch 类型须同步更新注册表与首批 list 两处。
