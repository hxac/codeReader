本讲介绍 NGF 控制面中与主配置链路平行的 **Provisioner（资源置备器）** 子系统：主循环负责「数据面 NGINX 的配置内容」，Provisioner 负责「数据面工作负载本身的生老病死」，二者仅通过 `Provisioner` 接口的唯一方法 `RegisterGateway` 异步衔接。`NginxProvisioner` 是其实现，所有写集群方法首行均受 `isLeader()` 门控，非 leader 只记账（`resourcesToDeleteOnStartup`），当选后由 `Enable` 补删。Provisioner 自带一条挂在 `LeaderOrNonLeader` 上的独立事件循环，watch Gateway 及其创建的全部数据面资源，以 NGF 标签谓词认领自家资源，首批事件中 `GatewayList` 必须首位以建立归属地图。`store`（`NginxResources` 台账）只存 ObjectMeta，用 `gatewayChanged`（比较 Valid/Source/EffectiveNginxProxy/Listeners 四维、监听器忽略顺序）避免无谓重建，并以 resourceVersion 短路防回流死循环。生命周期由 `buildNginxResourceObjects`（按 secrets→configmaps→SA→Service→workload→HPA→PDB 顺序产出，全以 Gateway 为 owner）驱动 `provisionNginx` 建/更新，agent ConfigMap 变更时打 `restartedAt` 注解滚动重启；无效 Gateway 走 `deprovisionNginxForInvalidGateway`，误删走 `reprovisionNginx`。最简 OSS 场景下一个有效 Gateway 至少创建 6 个对象：agent TLS Secret、2 个 ConfigMap、ServiceAccount、Service、Deployment；回收则叠加 owner-reference GC 与主动 deprovision 双重机制，并联动清空 `DeploymentStore`。
