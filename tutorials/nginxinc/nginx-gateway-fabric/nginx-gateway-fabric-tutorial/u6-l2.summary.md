本讲回答「生成的配置如何被 NGINX 加载」这一衔接问题，核心结论是：NGF 的 `nginx.conf` 是一份随镜像内置、永不改写的**静态骨架**，仅定义 main/events/http/stream 四大上下文，并用四条 `include` 通配（main-includes、events-includes、conf.d、stream-conf.d）把可变内容外包给生成器；`load_module` 必须排在最前，决定了 main 的 include 位置。引入的关键术语包括：NGINX 上下文层级、include 文本展开、按 dest 分桶合并。三套骨架模板各管一个上下文地基——main（按需 `load_module`、`error_log`）、events（仅 `worker_connections`）、base http（http2、resolver、map、健康检查 server、日志、OIDC）。模板用 Go `text/template` 在包加载时 `Parse` 成全局变量，经 `MustExecuteTemplate` 渲染。include 机制将 snippet/policy/authz 转为独立 `.conf` 文件、以 `Name` 为键先到先得去重，骨架只渲染 `include <Name>;` 引用行；`includes/` 是数据目录，靠绝对路径引用、不在通配 include 内。业务 server/upstream 留给 u6-l3，stream 留给 u6-l4。
