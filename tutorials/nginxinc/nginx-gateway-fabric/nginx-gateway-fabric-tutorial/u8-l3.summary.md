本讲承接 u8-l2 的策略附着，回答「已校验附着的 `Policy` 对象如何变成 NGINX 配置文本」。核心是组合模式三件套：`Generator` 接口（五个方法 `GenerateFor{Main|HTTP|Server|Location|InternalLocation}` 对应五个 NGINX 上下文）、`CompositeGenerator`（仅做转发与拼接，自身不生成指令）、`UnimplementedGenerator`（嵌入后让生成器只重写关心的上下文）。产物为 `[]policies.File`，经 `createIncludesFromPolicyGenerateResult` 转成 `Include` 挂到对应上下文。字段→指令有三种范式：直接用 Spec（clientsettings）、经中间结构 `getXxxSettings` 适配（proxysettings 的布尔/拼接、ratelimit 的 zone 命名与默认值）、按 location 类型分发模板（observability）。ratelimit 最复杂：一个策略横跨 http（`limit_req_zone`）与 server/location（`limit_req`）两上下文，靠「影子策略 + 注解」解决 Route 级跨上下文 zone 声明。关键区分：指令型策略走 Generator，属性型策略（UpstreamSettingsPolicy）走 Processor 改 upstream 渲染参数。校验侧对称：`Validator` + `CompositeValidator`（按 GVK 路由）+ `NewManager` 注册，提供 `Validate`/`ValidateGlobalSettings`/`Conflicts`，防注入正则与 genericValidator 是生成前的安全闸门，`Conflicts` 字段级判定服务继承裁决，未注册校验器即 panic（fail-fast）。
