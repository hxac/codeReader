本讲解析 NGF 自定义 CRD 与 Gateway API 策略附着机制。CRD 分两类：**配置类资源**（`NginxProxy`/`NginxGateway`，经 `parametersRef` 引用，无 `targetRef` 与 `PolicyStatus`，配置数据面/控制面本身）与**策略类资源**（ClientSettingsPolicy、SnippetsPolicy、WAFPolicy、ObservabilityPolicy 等，经 `targetRef(s)` 附着到流量对象），二者共享 `policies.Policy` 接口，故图层能用同一套代码处理。策略附着遵循 GEP-713，分 **Direct**（仅作用于直接目标，不继承）与 **Inherited**（挂 Gateway 时向下继承到 Route）两种语义，由 kubebuilder 标签 `policy=direct|inherited` 标注，与 `targetRef`/`targetRefs` 单复数正交。图层中策略处理是 `BuildGraph` 的最后一步：`processPolicies` 解析 → `markConflictedPolicies` 按优先级裁决冲突 → `attachPolicies` 按 Kind 分发到 Gateway/Route/Service（经 `ValidateGlobalSettings` 校验对 NginxProxy 全局配置的依赖）→ 回写 `*PolicyAffected` 条件。NginxProxy 经 `processNginxProxies` 校验后，由 `buildEffectiveNginxProxy` 用 JSON marshal/unmarshal 合并 GatewayClass 级与 Gateway 级两份（Gateway 覆盖前者），再以 `cleanupEffectiveNginxProxy` 修补切片清空与字段互斥，产出每个 Gateway 的 `EffectiveNginxProxy`。全程沿用 Source+Valid+Conditions 节点三元组范式，策略另以 `InvalidForGateways` 表达多 Gateway 下的细粒度有效性。
