本讲从用户视角梳理 NIC 的路由资源模型，核心是 Ingress 与 VirtualServer 的取舍：Ingress 以跨控制器可移植换取窄能力（仅 host/path 前缀路由与 TLS），扩展依赖类型不安全、不可复用的注解；VirtualServer 放弃可移植换取 NGINX 全能力的结构化表达，将 upstream 提升为一等对象，route 按名引用 action（pass/proxy/redirect/return），原生支持流量切分（splits，权重和为 100）、内容路由（matches）、头部改写与多种路径匹配。

VirtualServerRoute 是被 VirtualServer 装配的子路由片段（提供 subroutes），支持按名字（可跨命名空间）与按标签选择器两种引用方式，其 host 必须与宿主一致，用于模块化与跨团队复用。TransportServer 负责四层负载均衡（支持 TCP/UDP/TLS_PASSTHROUGH），路由退化为「listener 收到连接 → action.pass」，因端口是全局稀缺资源，必须引用 GlobalConfiguration 注册的 listener。GlobalConfiguration 解耦「在哪监听」（一张 listener 注册表，含 name+port+protocol），Policy 解耦「施加什么策略」（强类型、可复用的 CRD，仅服务 VS/VSR，互斥表达单一策略类型）。L7 与 L4 的不对称贯穿全程。本讲仅讲资源模型，控制器的 watch 与配置翻译留待后续单元。
