本讲为 NIC 仓库建立「空间地图」。核心结论：顶层目录按「它是什么」分组——代码（cmd/internal/pkg）、生成产物（config）、构建交付（build/charts/deployments）、示例文档（examples/docs）、测试（tests）、辅助（hack/grafana/tools.go/Makefile）。

引入的关键概念：

一、`internal/` 是 Go 的可见性围栏，承载全部私有实现，三大核心子系统构成单向数据流且层间不可越界：`internal/k8s`（控制器：Informer 监听、事件入队、sync 调谐、status 回写）→ `internal/configs`（配置生成：把扩展资源翻译成 NGINX 配置并渲染 .tmpl）→ `internal/nginx`（进程管理：启动/reload/校验/回滚）。控制器把资源组装成 VirtualServerEx 交给配置层。

二、`pkg/` 是对外门面，须区分手写真相源与生成代码：`pkg/apis/.../types.go`（带 kubebuilder 标注，CRD 源头）派生出 `pkg/client/`（typed clientset/informers/listers）与 `config/crd/bases/`（12 个 CRD YAML），均带 DO NOT EDIT，靠 `make update-codegen`、`make update-crds` 再生。

三、`internal/configs/` 下分 `version1/`（Ingress 模板）与 `version2/`（VirtualServer/TransportServer 模板），各含 OSS/Plus 两套 .tmpl。`deployments/`（按部署拓扑分目录的原始清单）、`charts/nginx-ingress`（Helm）、`examples/`（可 apply 的活示例）、`tests/`（pytest 集成测试）共同回答「如何安装与使用」。

四、五层架构（数据模型→校验→控制器→配置生成→进程管理）的 ASCII 图与 Key Source Files 表是给目录归类、对号入座的权威标尺。
