本讲聚焦感知层（Informer）与调度层（taskQueue）之间的「过滤阀」——事件处理器（event handler）。核心结论：每种资源都通过 `createXHandlers` 工厂返回一个 `ResourceEventHandlerFuncs` 三件套（Add/Delete/Update），其唯一职责是把值得处理的对象经由 `lbc.AddSyncQueue` 入队，不做配置生成、不写 status、不 reload。因为 taskQueue 无指数退避、无最大重试，变更检测（change detection）成为 UpdateFunc 的性能关键：NIC 给出四档策略——Service 定制字段比较（`hasServicedChanged`）、Ingress 归一化后比整个对象（`hasChanges` 抹平 Status 与 ResourceVersion，防止 status 回写引发 reload 风暴）、Policy/VS 只比 Spec、TransportServer 直接比整个对象。DeleteFunc 必须统一处理 `cache.DeletedFinalStateUnknown`（tombstone 墓碑）——双重类型断言加静默丢弃，防止 watch 重连补发删除事件时漏处理或 panic。Service 主动打破模板：因 K8s 改 Service port 不更新 EndpointSlice，须自行检查 port/externalName/headless 变化。最终确立「轻 handler + 重 sync」的设计取向，业务复杂度全部推迟到单 worker 串行的 sync 层。
