本讲阐明 NIC 把 CRD 字段校验拆成两层：第一层是声明式 OpenAPI/CEL schema，由 API Server 在准入时强制执行（不合法不入 etcd）；第二层是命令式 Go 校验，由控制器运行期在 `validation` 包内手写，失败则把资源剔出内存模型并将 `.status.state` 置为 `Rejected`。Go 层按「每种资源一个 Validator」组织，Validator 是带运行期开关（isPlus、isCertManagerEnabled 等）的结构体，用 `field.ErrorList` 一次性收集所有错误。schema 的真相源是 `types.go` 上的 kubebuilder 标记——`Enum`→`enum:`、`Pattern`→`pattern:`、`XValidation`→`x-kubernetes-validations:`（CEL 跨字段约束），由 `controller-gen` 生成到 `config/crd/bases/*.yaml`，禁止手改。关键结论：能进 schema 的尽量进 schema（更早失败），但跨资源引用、依赖运行期开关、NGINX 格式解析只能留在 Go 层。改 `types.go` 后必须 `make update-codegen`（补 DeepCopy 与 client）再 `make update-crds`（生成 CRD），顺序不可颠倒，`verify-codegen` 在 CI 防漂移。
