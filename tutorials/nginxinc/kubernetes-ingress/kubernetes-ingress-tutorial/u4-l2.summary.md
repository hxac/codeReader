本讲讲解 NGINX 全局配置的 ConfigMap 解析子系统，核心是回答「worker_processes、日志格式等 NGINX 自身全局参数从哪里来」。引入三套按生命周期区分的配置结构：StaticConfigParams（命令行 flags，启动期固定）、ConfigParams（`-nginx-configmaps` 指向的 ConfigMap，运行期可改触发 reload）、MGMTConfigParams（`-mgmt-configmap`，仅 Plus）。其中 ConfigParams 是「大杂烩」，带 Main 前缀的字段进入 nginx.conf 的 main/http 上下文，无前缀字段作 Ingress/VirtualServer 的资源级默认值。主角函数 ParseConfigMap 采用「默认值兜底 + 逐 key if-exists 覆盖 + best-effort（非法 key 记 Event、置 configOk=false 但不中断）」模式。三源最终经适配函数 GenerateNginxMainConfig 合并、计算派生字段（如 zone-sync 域名）成模板输入 MainConfig。完整链路为五跳：ConfigMap.Data → ConfigParams → Configurator.CfgParams → MainConfig → nginx.tmpl → nginx.conf。类型转换统一封装在 GetMapKeyAsBool/Int 与 ParseSize/Time 中，遵循三返回值约定；zone-sync/OIDC/OTel/MGMT 等有跨字段约束的 key 被抽成独立子块解析函数，parseStringField 以柯里化风格消除 OIDC 样板代码。
