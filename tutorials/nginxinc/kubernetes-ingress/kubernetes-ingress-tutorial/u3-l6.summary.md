本讲剖析控制器的「世界模型」`Configuration`（`configuration.go`）：用两类 map 分层——原始资源仓库（`virtualServers` 等）与裁决索引（`hosts`、`listenerHosts`，记录谁真正拥有 host），外加问题簿 `hostProblems`，全由一把 `sync.RWMutex` 保护。所有 `AddOrUpdate*`/`Delete*` 走统一三段式：改仓库 → `rebuildHosts` 全量重算索引 → 做 diff，产出两类核心产物：`ResourceChange`（驱动 `Configurator` 写 nginx.conf）与 `ConfigurationProblem`（驱动上层回写 `.status.state` 并发 Event）。多资源抢同一 host 时由 `chooseObjectMetaWinner` 裁决，先比 `CreationTimestamp` 再比 `UID`，结果稳定可复现；赢家进 `hosts`，输家挂 warning 并得 Warning 级 problem、永不生成配置，从根本上杜绝坏配置落地。`delete` change 永远排在 `addOrUpdate` 之前，避免 reload 中间态出现重复 `server_name`；启动期用 `startupComplete` 跳过逐次重建，由 `CompleteStartup` 做唯一一次。`reference_checkers.go` 以统一接口 `resourceReferenceChecker`、经 `findResourcesForResourceReference` 驱动，遍历 `hosts` 与 `listenerHosts` 两层索引反查「Service/Secret/Policy 变了影响谁」，并将命中资源重新入队。本讲只裁决「生成谁、拒绝谁」，不真正写 nginx.conf。
