本讲义解析 `LoadBalancerController` 的生命周期，覆盖四个模块：结构体字段分组、构造装配、Run 启动序列、Stop 优雅关闭。

核心结论：该结构体是 reconcile 循环的宿主，字段按「连接与缓存 / 工作入口 / 内存与配置 / 状态与选举 / 运行期标志」五组划分。NewLoadBalancerController 遵循「构造与运行分离」：只建对象、注册 handler、建 informer（未 Start），关键接线是把 `syncQueue` 绑定到 `sync` 处理函数，启动推迟到 `Run()`。`Run()` 的铁律是先 `cache.WaitForCacheSync`、再 `preSyncSecrets`、最后 `syncQueue.Run`，以 `<-ctx.Done()` 阻塞收尾。`isNginxReady` 是启动优化标志：首次排空队列后做一次性 host 重建与单次 reload 即置 true（pod 已能转发流量），被推迟的状态写回收回后台 10 worker 并行刷写，把「就绪」与「状态写回」解耦。`Stop()` 经一次 `cancel()` 由 context 串起全部后台 goroutine，配合 `main.handleTermination` 处理 SIGTERM、设置 `ShuttingDown`。

引入的关键术语（供承接）：LoadBalancerController、taskQueue、namespacedInformer / SharedInformerFactory、cache.WaitForCacheSync、isNginxReady、leaderElector（仅 leader 写 status，follower 被 waitForLeadership 拦截）、ShuttingDown、pendingStatus 切片；informer 内部 list-watch、队列 worker 循环、sync 按 Kind 分发、leader 选举细节均留待后续讲义展开。
