本讲承接 u5-l2 的 Provider 抽象，解决代码奖励阶段「多训练进程直连云沙箱触发账号/IP 级限流」的问题。直连模式下每进程各自维护本地信号量、缺乏全局协调，聚合成不受控的请求风暴。Router 模式用三招化解：**批量**（整批脚本打包成一次 `POST /execute_batch`）、**单一出口**（全部请求只出自 Router 节点一个 IP）、**全局闸门**（用一个 `asyncio.Semaphore(max_num_sandboxes)` 统一限流）。核心模块包括：客户端 `RoutedSandbox`/`RoutedMorphSandbox`（伪装成后端 `Sandbox.run_code` 的薄 HTTP 客户端，E2B 版乐观无超时、返回 `Execution`，Morph 版防御性带超时、返回极简对象）；`E2BProvider`/`MorphProvider` 内部以 `e2b_router_url`/`morph_router_url` 是否非 `None` 在直连与 Router 间切换，该分支忽略 `num_parallel` 且训练节点无需沙箱凭据；以及 `scripts/e2b_router.py`（FastAPI 常驻服务，凭 `asyncio.wait_for` 硬超时、`finally sandbox.kill()` 防泄漏）。关键语义差异：Router 模式脚本解析失败记 `None`（跳过样本），直连模式记 `0.0`（惩罚），混用需注意。该 Router 通过 Slurm 长驻 CPU 节点部署，所有训练任务共享同一 URL。
