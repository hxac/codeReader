本讲是 u3「模型与模板」单元首篇，聚焦模型侧加载机制，核心是讲清「一个模型 id 字符串如何变成 `(model, processor)`」并选定对话模板。ms-swift 把结构相同的一族模型抽象成一个 `model_type`，用全局字典 `MODEL_MAPPING: Dict[str, ModelMeta]` 做映射；该表由各 `models/*.py` 在**导入时**调用 `register_model` 填充（「导入即注册」，配合 `_LazyModule` 懒加载），注册默认禁止重复以做防呆校验。

元信息分两层：`ModelMeta` 是可复用的**静态档案**（loader/template/model_arch/is_multimodal 等），`ModelInfo` 是一次性的**运行时档案**（model_dir/dtype/quant/max_len）。「id → model_type」走三道防线：显式传入 > 本地 `args.json` > `config.json` 的 `architectures` 反查；匹配不到时纯文本模型走 `template='dummy'` 兜底、多模态直接报错。

加载由 `ModelLoader.load()` 编排成线性流水线（config → processor → model → 收尾），子类（如 `QwenLoader`）只覆盖少数方法即可定制；收尾时把 `model_meta`/`model_info` **挂回 model 对象**。顶层入口 `get_model_processor` 串起「匹配 + 加载」，`get_processor` 是其 `load_model=False` 的薄封装。最后 `ModelMeta.template` 经 `model.model_meta` 流转，被模板系统 `get_template_meta` 读取，实现模型加载与对话模板的**延迟耦合**——这是衔接 u3-l3 模板体系的桥梁。
