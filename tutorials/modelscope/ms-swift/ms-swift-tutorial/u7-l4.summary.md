本讲解决 GRPO 在多轮场景（生成→环境反馈→再生成）的落地。核心抽象是夹在训练器与推理引擎间的状态机 RolloutScheduler：其默认 run() 实现单轮，MultiTurnScheduler 提供「生成→钩子→判停→step」的多轮循环，并分全定制（覆写 run）与部分定制（实现 step）两条路径；判停三级优先 on_turn_end['done'] > check_finished > max_turns 兜底。关键术语 response_token_ids 与 response_loss_mask 决定每轮 token 是否参与训练。环境侧引入 Gym 风格的 Env 抽象（reset 返回含 system_message，step 接收 Messages，即「对话即动作」）；GYMScheduler 借通用钩子 on_trajectory_start / on_turn_end 挂入环境，用 uuid 一轨迹一环境，--use_gym_env true 时环境 total_reward 作为一列奖励 gym_reward 直接并入奖励矩阵。colocate 模式由与后端无关的 agent_loop.run_multi_turn 驱动，用临时事件循环跑异步钩子、用 gather_fn 保证多 rank 终止一致；外部 HTTP/WebSocket 服务经 OpenEnvWrapper（同步薄封装，不继承 Env）与 OpenEnvScheduler（parse_action / format_observation 做 LLM 文本与 dict 转换）接入。
