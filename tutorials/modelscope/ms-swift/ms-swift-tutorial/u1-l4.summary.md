本讲讲解 ms-swift 的 CLI 分发链路，仅涉及 `swift/cli/` 一层薄代码。`swift` 命令经 `setup.py` 的 `console_scripts` 注册到 `swift.cli.main:cli_main`，`megatron` 命令同理指向各自入口。`cli_main` 是一个「发射器」：它用路由表 `ROUTE_MAPPING` 把子命令（如 `sft`/`infer`/`export`）映射到脚本文件路径（查表前会把下划线转连字符），再通过 `subprocess.run` 重启该脚本，自身不做真正训练。是否套 `torchrun` 由 `use_torchrun()`（读取环境变量 `NPROC_PER_NODE`/`NNODES`）与子命令类型共同决定，仅 `pt/sft/rlhf/infer` 及 megatron 命令走多进程分支，`export`/`eval` 等即便设了多卡变量也忽略。`parse_yaml_args` 把 YAML/JSON 配置就地展开为 `--key value` 参数（列表直接展开、字典转 JSON 字符串），并通过 `ENV` 块注入环境变量，shell 已存在的变量优先、不覆盖。终端打印的 `run sh:` 即框架最终执行的真实命令，是排查问题的第一线索。结论上确立了「main.py 负责调度、sft.py 等负责干活」的分工，为后续讲义中 `SftArguments` 解析、`sft_main` 训练器流程奠定前置基础。
