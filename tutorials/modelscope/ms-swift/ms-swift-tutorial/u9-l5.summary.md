本讲讲解 ms-swift 中 Ray 的角色定位与两套独立实现。核心结论：Ray 不切模型，而是做「编排」——把多个进程/角色调度到集群 GPU 上并维护通信，与负责并行算法的 DeepSpeed/FSDP/Megatron 正交。

ms-swift 有两套互不调用的 Ray 栈。一是 Swift Ray（`swift/ray_utils/`），面向 HF Trainer，靠静态工具箱 `RayHelper` 的 `@RayHelper.worker` / `@RayHelper.function` 装饰器实现「双模透明」：运行时据 `ray_inited()`/`is_worker()` 判定，单机时装饰器退化为空操作，driver 上则把类包成 `ray.remote` 并远程派发方法；`dispatch`（all/slice）控分参、`collect` 控收果，slice 天然数据并行。`RayArguments`（`use_ray`/`ray_exp_name`/`device_groups`）混入 `BaseArguments`，由 CLI 早期的 `try_init_ray()` 用最小 argparse 触发 `RayHelper.initialize` 与 `ResourceManager` 建 placement group。

二是 megatron-ray（`swift/ray/megatron/`），面向 Megatron 的 GRPO/GKD，不复用 `RayHelper`，而用 `MegatronRayPipeline` + `ResourcePool`（`STRICT_PACK` 一节点一 PG）+ `WorkerGroup`（`@dispatch_collect`，按 dp_rank 派发回收）显式编排 train/rollout/teacher 三类角色。关键术语：colocate（共享 GPU，需 offload/sleep 释显存，naive 同步）与 separate（独占 GPU，nccl 同步），DP = gpus/(TP×PP×CP)。两套训练功能等价，Ray 价值在自动化多机/多角色编排。
