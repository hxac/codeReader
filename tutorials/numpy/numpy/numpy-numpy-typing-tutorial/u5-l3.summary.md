本讲聚焦 `numpy/_typing/_add_docstring.py`，解决一个纯文档工程问题：PEP 695 `type` 语句定义的类型别名（如 `NDArray`、`ArrayLike`、`DTypeLike`）没有挂文档字符串的语法，其 `__doc__` 为空，sphinx 的 `autofunction`/`autoclass` 都用不上，且 sphinx 不存在 `autotypealias` 指令。

核心方案是「收集 → 转换 → 拼接」三段式流水线：`add_newdoc(name, value, doc)` 把名字、取值、文档打包成三元组塞进模块级列表 `_docstrings_list`，仅作登记处；`_parse_docstrings()` 是机制核心，先用 `textwrap.dedent` 去缩进并整体加 4 空格，再用正则识别 numpy 风格的「标题 + 下划线」段落——`Examples` 段改写成 `.. rubric::`、其它段（如 `See Also`）改写成 `.. admonition::`，最后把每条包装成 sphinx data 域块 `.. data:: 名字` + `:value: 取值`。引入的关键概念包括 sphinx data 域（`py` 域里描述模块级数据的 `.. data::` 指令族）、`:value:` 选项、以及 numpy docstring 风格。取值策略上，`ArrayLike`/`DTypeLike` 手写为 `typing.Union[...]` 以掩盖难看的私有联合表示，而 `NDArray` 因 `repr` 恰好简洁而直接使用。闭环在 `numpy/typing/__init__.py` 末尾完成：`__doc__ += _docstrings` 把 reST 文本拼进模块文档，供 sphinx `automodule:: numpy.typing` 一次性渲染；真正有 `__doc__` 的 `NBitBase` 类则单独用 `.. autoclass::` 自动拉取。新增别名只需追加一句 `add_newdoc` 即可自动生效，整个机制与类型检查无关。
