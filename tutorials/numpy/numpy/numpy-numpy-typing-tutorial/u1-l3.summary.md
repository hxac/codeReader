本讲讲解 PEP 561 如何让类型信息随包分发：它区分两种模式——第三方维护的独立桩包（types-foo）与库自带的 inline 类型包；NumPy 选后者，pip install 即得类型，注解与实现同源同步。核心有三个最小模块：其一，PEP 561 是「交付机制」，与规定注解写法的 PEP 484 互补；其二，py.typed 是一个须放在包根目录的空标记文件，靠「存在」本身宣告本包自带类型，检查器据此采用包内注解并忽略任何第三方桩包，删除它不影响运行时；其三，.pyi 桩文件只含类型信息、无运行逻辑，当同一模块的 .py 与 .pyi 并存时，类型检查器优先读 .pyi 并忽略 .py 注解。numpy/typing 同时拥有 __init__.py（含 __getattr__/__dir__、文档拼接、PytestTester 等运行时设施）与精简的 __init__.pyi（仅 import 与 __all__），是「双轨制」的样本，两份 __all__ 必须一致以保持检查时与运行时公共面认知相同。test_isfile.py 检查已安装包里 py.typed 与各子包 __init__.pyi 是否齐全，是防止桩文件漏装导致类型静默退化的兜底测试。本讲由此闭合「类型从源码走到用户机器、再被检查器发现」的完整链路。
