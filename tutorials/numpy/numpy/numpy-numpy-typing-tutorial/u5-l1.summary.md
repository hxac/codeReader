本讲以 `numpy._typing._ufunc` 与 `numpy.typing` 为最小样本，讲清同一模块并存 `.py` 与 `.pyi` 时类型检查器与运行时的分野。核心结论是双轨制：CPython 执行 `.py`，类型检查器优先读 `.pyi`、忽略 `.py`，两轨「名字对齐、内容分家」——对外暴露的名字必须一致，而实现内容可完全不同。

由此引入三个关键概念。一是**双轨制（dual-track）**：`_ufunc.py` 仅 3 行别名（三个名字都等于 `ufunc`）与 828 行 `_ufunc.pyi` 重载并存，说明运行时根本不需要那些丰富类型类。二是 **`type_check_only` 幻影类**：`typing` 装饰器声明「只在类型检查时存在」的虚构子类（如 `_UFunc_Nin2_Nout1`），配合 `.py` 中的同名占位别名维持导入图合法。三是 **`__init__.pyi` 包级桩**：给检查器一个干净、显式、最小的入口，屏蔽 `__init__.py` 里的 `__getattr__`/`__dir__`、文档拼接、`PytestTester` 等动态逻辑，且两轨的 `__all__` 必须一致。最终 `np.add` 等的精确返回类型完全由桩文件链决定，运行时那条轨仅让导入合法、对类型推断无贡献。
