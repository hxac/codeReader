本讲区分「运行时类型测试」与 u6-l1 的「静态类型测试」：前者验证类型对象在真实 Python 进程中能否被正常使用，而非注解是否合法。核心四点：PEP 695 的 `type` 别名产生 `TypeAliasType`「盒子」，须经 `__value__` 剥壳后才能用 `get_args`/`get_origin`/`get_type_hints` 拆解并完成注解往返；普通类 `NBitBase` 无盒子须手工构造。`TYPES` 字典配 `test_keys` 把测试清单与公共 `__all__` 绑死以防脱节。`@runtime_checkable` 协议（`_SupportsArray` 等）的运行时 `isinstance`/`issubclass` 是浅层的，只看方法名是否存在。`test_isfile.py` 则是打包完整性兜底，确认 `py.typed` 标记与各 `__init__.pyi` 桩文件确实随包安装，把类型「静默退化」变为可见失败。
