本讲聚焦 `numpy/typing/__init__.py` 的模块文档字符串（辅以 `_nbit_base.py`/`.pyi` 与 reveal 夹具），讲清数值「精度关系」的现代类型表达法。核心结论是：自 NumPy 2.3 起 `NBitBase` 被弃用，官方改推两种标准 Python 手法。

老方案 `NBitBase` 把「精度」本身当作不变泛型参数：标量类写成接收精度标签的泛型（如 `np.floating[T]`，`T` 是 `_32Bit` 这类叶子），优点是能对精度做集合运算（`np.floating[T1 | T2]` 表达混合精度取并集），代价是需维护 `_8Bit < … < _128Bit` 精度层次、`__init_subclass__` 名字白名单，以及与运行时不一致的桩文件（`.pyi` 多出 `_256Bit`/`_80Bit`）。

现代方案一「**bound TypeVar**」：把上界从精度标签换成具体标量类，即 `TypeVar("S", bound=np.floating)`，签名 `def f(a: S, b: S) -> S`，适合「同精度进出」。现代方案二「**`@overload`**」：每条存根签名声明一对（输入类型→输出类型），最后一条实现签名取最宽入参与最宽返回兜底，适合「不同精度一一映射」（如复数相位 `complex64→float32`）。边界在于：TypeVar 只能保证同型，表达不了映射或精度提升；精度提升只能用 `@overload` 逐条手写，这正是弃用 `NBitBase` 后失去的精度集合运算能力。引入的承接术语包括：精度叶子、不变泛型参数、精度并集、上界为标量类的 TypeVar、`@overload` 存根签名与兜底实现。
