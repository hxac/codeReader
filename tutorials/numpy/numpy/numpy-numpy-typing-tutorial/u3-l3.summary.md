本讲围绕 `numpy/_typing/_dtype_like.py`，拆解「带 dtype 属性的对象」这一 `DTypeLike` 构件。核心是两个对称的 `@runtime_checkable` 泛型协议：`_HasDType` 刻画带 `dtype` 属性的对象，`_HasNumPyDType` 刻画带 `__numpy_dtype__` 属性的对象。后者是 NumPy 为避免与第三方（如 Pandas）的通用 `.dtype` 冲突而引入的专属协议。两者都用 `@property` 声明属性，表达「无参读取」语义；结构子类型下，具体类可用普通属性、property 或描述符满足，无需继承。运行时 `np.dtype(obj)` 经 C 端 `_try_convert_from_dtype_attr` 先查 `__numpy_dtype__`、缺失才回退 `dtype`。`_SupportsDType[DTypeT]` 是这两个协议的 PEP 695 联合别名，借类型参数 `DTypeT`（约束为 `np.dtype` 子类型）让 dtype 元素类型沿协议流动，使 `np.dtype(obj)` 的返回类型可被精确推断。公共 `DTypeLike` 用 `_SupportsDType[np.dtype]`（宽松、弃精度），私有可参数化的 `_DTypeLike[ScalarT]` 用 `_SupportsDType[np.dtype[ScalarT]]`（带精度、供收窄别名复用），体现「外松内紧」。再导出边界：`numpy._typing.__init__` 仅暴露 `_HasDType` 与 `_SupportsDType`，`_HasNumPyDType` 留作内部积木。
