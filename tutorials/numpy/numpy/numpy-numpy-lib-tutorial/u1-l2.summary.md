本讲承接 u1-l1，解析 `np.pad` 这类公开函数的导入与分发机制，拆解三个最小模块。其一，**两层结构**：真正实现藏在私有 `_xxx_impl.py`（如 `_arraypad_impl`），对外由只有一两行的**薄再导出模块**（`npyio.py`/`format.py`/`stride_tricks.py`，用 `from ._xxx_impl import ...` 搬名字）暴露稳定接口；部分函数（如 `pad`）无薄模块，由顶层 `numpy/__init__.py` 直接 `from .lib._arraypad_impl import pad` 取名，并经 `set(...__all__)` 收集进 `np.` 命名空间。其二，**dispatcher + impl 双函数写法**：公开函数以 `@array_function_dispatch(_xxx_dispatcher)` 装饰，dispatcher 只返回参与运算的数组参数（如 `(array,)`），背后接的是 NEP-18 的 `__array_function__` 协议；装饰器定义于 `numpy/_core/overrides.py`，把函数包成 `_ArrayFunctionDispatcher`（C 实现），并通过 `verify_matching_signatures` 在导入期强制 dispatcher 与实现签名一致，原始实现可经 `_implementation` 取回。其三，**模块级 `__getattr__`（PEP 562）**：拦截 NumPy 2.0 移除/私有化的旧别名（`emath`/`arraypad`/`utils`/`arrayterator` 等），抛带迁移指引的 `AttributeError`（`name=None` 抑制异常链），将模糊失败转为有向导的失败。核心结论：`_impl` 出实现 → 薄模块/顶层再导出给名字 → `array_function_dispatch` 给派发能力 → `__getattr__` 给迁移指引，构成 numpy.lib 的导入分发骨架。
