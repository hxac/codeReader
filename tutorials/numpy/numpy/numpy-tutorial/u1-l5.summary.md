本讲精读三个最小模块：`numpy/_pytesttester.py`（包装 pytest 的 `PytestTester` 类）、`numpy/testing/__init__.py`（断言工具汇聚入口）、`numpy/__init__.py`（顶层装配与导入末尾自检），辅以 `conftest.py` 与 `pytest.ini`。核心结论：`numpy.test()` 来自很薄的 `PytestTester`，各子包用 `test = PytestTester(__name__)` 三行样板即可拥有自己的 `test()`，内部调 `pytest.main()` 并返回布尔值；其 `label` 仅文档了 `'fast'`（等价 `-m "not slow"`）与 `'full'`，其他字符串会被当成 pytest 标记名，在 `--strict-markers` 下报错，故 `numpy.test('quick')` 不可用。`numpy.testing` 是为数值数组量身定做的断言集（`assert_allclose`、`assert_array_equal`、`assert_raises`、`assert_warns` 等），实现藏在 `_private/utils.py` 经 `from ._private.utils import *` 再导出，解决了普通 `assert` 遇 `ndarray` 返回布尔数组而 `truth value ambiguous` 的问题。导入末尾的 `_sanity_check` 用一个结果应为 `2.0` 的 `float32` 点积探测错误 BLAS 或混用包管理器导致的数值错误，并转成 `RuntimeError`；macOS 上 `_mac_os_check` 用 `polyfit` 触发 LAPACK 探测有 bug 的 Accelerate。二者受 `__NUMPY_SETUP__` 守卫，仅非构建自身时执行，执行后立刻 `del`。引入了 `PytestTester`、`label`/`marker`/`--strict-markers`、导入时副作用与 sanity check、开发模式 vs 发布模式（有无 `pytest.ini`）等术语。
