本讲义解析 NumPy ufunc 的「快」从何而来，核心结论是性能来自**三层优化**：用通用内联 `npyv_*` 写一份 SIMD 循环（写作层），构建期按 `--cpu-baseline` / `--cpu-dispatch` 编译出 baseline 加多个 dispatch 版本（编译层），导入时探测 CPU 并挑选最合适那份（运行时分发层）。

关键术语：**SIMD**（单指令多数据，宽寄存器一次处理多元素）；**baseline（基线）**——无守卫、永远开启的最低指令集；**dispatch（可分发）**——被 `NPY__CPU_TARGET_` 守卫、仅在 `*.dispatch.c` 可分发源里激活、运行时按需挑的额外指令集。前者导入时校验本机支持否则报错，后者按兴趣度短路选择最高目标。

讲解了一组分发宏：`CURFX` 给符号加目标后缀避免链接冲突、`DECLARE` 前向声明、`CALL` 用 `NPY_CPU_HAVE` 做三元选择链、`INFO` 报告当前/可用目标；以及 Meson 的 `mod_features.multi_targets()` 如何把一份源编译成多份目标码。运行时观察入口是 `np.lib.introspect.opt_func_info()`（读取始终开启的 `__cpu_targets_info__`），改变分发的唯一环境变量是 `NPY_ENABLE/DISABLE_CPU_FEATURES`（须导入前设置），当前 HEAD 并无 `NPY_CPU_DISPATCH_TRACE` 环境变量。量化用 asv（`bench_ufunc.py` 强制覆盖全部 ufunc，`spin bench --compare` 跨提交对比）；性能结论为就地运算快于临时表达式、连续内存走 CONTIG 全速、非连续走 NCONTIG 慢、大数组才体现 SIMD 吞吐而小数组测的是 overhead。
