本讲聚焦 `_multiarray_umath.py` 与 `multiarray.py` 两个兼容垫片，讲 NumPy 1.x/2.x 的 C-ABI 不兼容：用 1.x 头文件编译的扩展，拿到 2.x 运行时的 `_ARRAY_API`/`_UFUNC_API` 函数指针表胶囊（PyCapsule）后，按旧内存布局读取会段错误，无法用异常兜住。故 `_multiarray_umath` 的模块级 `__getattr__` 对这两个名字用 ImportError 硬拦（而非普通废弃属性那种 `DeprecationWarning` 软报警）：在扩展被 import 的瞬间，用 `traceback.format_stack()[:-1]` 收集调用栈、跳过 `frozen importlib` 帧，既写 stderr 又 raise。关键概念是「致命路径硬拦、无害路径放行」：同一个 `_ARRAY_API`，因 pybind11 ≤ 2.11.1 的历史初始化属无害路径，在 `multiarray.py` 顶部被 eager 绑定进 `__dict__` 而静默放行。核心结论：一个名字是静默、报警还是硬拦，由「是否被 eager 绑定进 `__dict__`」与「`__getattr__` 里走哪条分支」共同决定——「绑不绑」即「放不放行」。
