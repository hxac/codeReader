本讲对 `numpy/core` 目录做全局分类，确认该目录共 36 个文件（19 个 `.py` + 17 个 `.pyi`），几乎全部是向后兼容的垫片，按运行时角色归为五类：包入口、工具、纯转发垫片、特殊垫片（eager 绑定）、类型存根。

核心结论：包入口 `__init__.py` 用 `__all__` 只列子模块名、不在顶部 import，强制每次访问都走模块级 `__getattr__` 从而打印 `DeprecationWarning`；工具文件 `_utils.py` 仅提供统一的 `_raise_warning`，是唯一不做转发、也无对应 `.pyi` 的 `.py`。

引入的关键术语：「纯转发垫片」共 14 个，顶层只有 `__getattr__`，其中 13 个用 `None` 判属性缺失，仅 `numeric.py` 用 sentinel；「特殊垫片 / eager 绑定」共 3 个（`_internal.py`、`multiarray.py`、`_multiarray_umath.py`），在顶部额外绑死 pickle 重建函数、ufunc 或 ABI 符号，因这些对象等不起懒加载与警告——其中 `_ARRAY_API` 因属 NumPy 1.x/2.x 不兼容的 C-ABI 符号，需抛 `ImportError` 硬拦。判定纯转发与特殊垫片的可靠依据是用 `ast` 检查顶层是否只有 `__getattr__`。类型存根 `.pyi` 不在运行时执行、不触发警告，分完整再导出（`import *`）、省略并写 NOTE、近空/空三种写法。
