本讲精读 NumPy 字符运算最底层的两个 C++ 头文件：`string_buffer.h` 与 `string_fastsearch.h`。核心是 `ENCODING` 枚举（`ASCII`/`UTF32`/`UTF8`）作为编译期分发开关，让同一套算法按编码各编译一份。关键结论：定长 ufunc（`string_ufuncs.cpp`）只用 `ASCII` 与 `UTF32`，`UTF8` 仅服务于 StringDType（`stringdtype_ufuncs.cpp`）。`getchar<enc>` 三个模板特化回答「一个字符占几字节」：ASCII 恒 1、UTF32 恒 4、UTF8 变长 1–4（首字节查表 `num_bytes_for_utf8_character`）。`codepoint_is*` 按 enc 特化（ASCII 走 `NumPyOS_ascii_*`，Unicode 走 `Py_UNICODE_*`），配合 `IMPLEMENTED_UNARY_FUNCTIONS` 与 `unary_loop` 共用遍历骨架。`Buffer<enc>` 游标类把编码差异藏进运算符重载，使上层算法写成编码无关的字符级逻辑。`string_fastsearch.h` 移植自 CPython，用 Bloom+Horspool（`default_find`）、Two-Way、自适应三档策略支撑 `find`/`index`/`count` 的线性最坏时间子串搜索。本讲原语与 u3-l12 循环注册、u2-l7 Python 查找函数首尾相接，构成完整调用链。
