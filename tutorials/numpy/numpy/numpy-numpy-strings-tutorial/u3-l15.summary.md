本讲义深入剖析 `numpy._core.multiarray` 中 `_vec_string` 这一 C 函数的三段式实现，它是连接 NumPy 字符串数组与 Python 内建字符串方法的「通用桥」，承担 `upper`/`lower`/`mod`/`decode`/`encode`/`translate` 等尚无专用 ufunc 的函数。最小模块有三：一是总入口分发器，按**数组 dtype** 决定取 `bytes` 方法（`bytes_`/`'S'`）还是 `str` 方法（`str_`/`'U'` 与 `StringDType`/`'T'`，因后者标量类型即 Python `str`），再按**有无额外参数**在快慢路径间二选一；二是 `_vec_string_no_args` 快路径，用单迭代器 `PyArray_IterNew`，输出形状等于输入形状，每元素一次 `method(item)` 回调，适用于宽度不变的函数；三是 `_vec_string_with_args` 慢路径，用广播多迭代器 `PyArray_MultiIterFromObjects` 支持参数数组逐位置配对，但每元素需新建 `tuple`，开销更高。核心结论：两条路径都逐元素回到 Python 解释器，性能被回调钉死，远不及原生 ufunc，这正是 `__all__` 注释区分「会变 ufunc」与「大概不会变 ufunc」的依据。引入的关键术语包括：dtype 分发、快慢路径、单迭代器与广播多迭代器、C 路径与 object 中间态还原。
