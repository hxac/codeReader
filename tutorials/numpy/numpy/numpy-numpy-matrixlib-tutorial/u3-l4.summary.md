本讲精读 `np.matrix` 重写的四个形状变换方法 `tolist`/`squeeze`/`flatten`/`ravel`，并引入视图/副本判定工具 `np.may_share_memory`。核心结论：这四个方法的方法体几乎都是一行委托调用，真正维持二维不变量的并非方法逻辑，而是派生时自动触发的 `__array_finalize__`，它用私有通道 `_set_shape((1, N))` 把掉到一维的中间结果补回二维。逐项而言：`tolist` 是例外，先经 `__array__()` 脱壳为普通 ndarray 再转 list，以绕开基类对 `x[0]` 降维的依赖；`squeeze` 把 `(N,1)` 列向量变成 `(1,N)` 行向量，`axis` 仅能触发错误；`flatten` 永远复制、`ravel` 尽量视图，形状同是 `(1,N)` 但内存语义相反。关键分流在于：`m.ravel()` 返回 `(1,N)` matrix，而 `np.ravel(m)` 因源码中对 matrix 的 `isinstance` 特判走 `asarray` 剥子类型，返回 `(N,)` 普通 ndarray。`may_share_memory` 是保守启发式，`True` 仅表示边界相交「可能共享」。
