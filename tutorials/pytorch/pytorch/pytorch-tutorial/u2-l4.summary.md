本讲义讲清了「Python 调用如何走到 C++」这一段路径，覆盖的最小模块包括 `torch/__init__.py`、`torch/_tensor.py`、`torch/_VF.py`、`torch/_C/*.pyi.in` 类型桩模板、构建脚本 `tools/pyi/gen_pyi.py`，以及运行时绑定所在的 `torch/csrc/autograd/python_torch_functions_manual.cpp`。

核心结论：算子有两条等价入口——函数式 `torch.add` 来自 `_C._VariableFunctions`，方法式 `a.add` 来自 `torch._C.TensorBase`。两者在 Python 层是不同对象，却在 C++ 层汇合到同一个 `at::add`。函数式入口并非手写，而是 `__init__.py` 中的「循环 B」逐个把 `_C._VariableFunctions` 成员复制进 `torch` 的 `globals()`，借 `PRIVATE_OPS` 黑名单与 `__module__` 改写控制可见性；方法式入口则经 MRO 继承自 `TensorBase`，少数（如 `__rsub__`）显式转发到 `_VariableFunctions`，另有 `__pos__/__neg__` 等以一行别名绑定。`torch._VF` 是 `_VariableFunctions` 的转发外壳，用于绕过 mypy。

本讲引入的关键术语：类型桩（`.pyi`，仅供静态检查器）、`.pyi.in` 模板与 `${function_hints}` 等占位符、运行时绑定（C++ 方法表 `THPVariableFunctions`）、`_VariableFunctions`、`TensorBase`、`PRIVATE_OPS`。两条轨道都源自 `native_functions.yaml`，由生成器保证一致。本讲止步于 `at::add`，Dispatcher 分发留给 Unit 3。
