本讲剖析 lmdeploy 的路由器 `archs.py`。它读取 HuggingFace `config.json`，依次回答三个问题：用哪个后端、用哪种引擎配置对象、用哪种 Pipeline 类。一切判定的起点是 **arch 名字**（即 `architectures[0]`，模型类名），`get_model_arch` 用「architectures → auto_map → language_config.auto_map」三级兜底提取它。`autoget_backend` 优先 TurboMind、不支持则回退 PyTorch，依据是 TurboMind 是否安装与 `is_supported`（查 `triton_models/` 目录或 arch 命中 `SUPPORTED_ARCHS` 表）。`autoget_backend_config` 把用户显式传 `PytorchEngineConfig` 视为最高优先短路，并对跨后端做字段搬迁（如 `block_size` ↔ `cache_block_seq_len`）。`check_vl_llm` 靠多模态 arch 白名单区分纯文本/多模态，`get_task` 据此在 `AsyncEngine` 与 `VLAsyncEngine` 间选择，`language_model_only` 可强制走纯文本；引擎类采用延迟 import 以防循环导入。核心结论：arch 名字是贯穿全栈的「模型身份证」，后端选择、PyTorch patch 映射、TurboMind 分发都是它的不同查表动作。
