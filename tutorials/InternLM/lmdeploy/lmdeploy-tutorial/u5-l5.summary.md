本讲是 PyTorch 后端算子链路的终点，解读 `kernels/` 的设备子包布局：`cuda` 为 Triton 融合算子主力，`default` 是纯 PyTorch 兜底（仅暴露采样、权重量化等最小算子集），`dlinfer` 覆盖昇腾等非 cuda 设备。核心是 `dispatcher.py` 的 `FunctionDispatcher`：按运行时设备延迟 `importlib` 加载、缓存到 `impl_map`，设备切换时作废、加载失败回退 `default`，并以 `make_caller`+`exec` 生成签名正确的统一入口。重点精读 `cuda/w8a8_triton_kernels.py`：`_linear` 用 `@triton.jit`+`@triton.autotune` 做分块 INT8 GEMM，grid 等于输出分块数，每 program 处理一块并就地乘回逐 token 的 `rms_scale` 与逐通道的 `linear_scale`；`rms_norm_dynamic_quant` 把 RMSNorm 与逐 token 量化融合进单 kernel 以减少显存往返。本讲承接 u5-l4，闭合「backends Impl → kernels」末层。
