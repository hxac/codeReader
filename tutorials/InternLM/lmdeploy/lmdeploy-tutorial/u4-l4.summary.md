本讲解析 PyTorch 后端的 Scheduler，它回答 EngineLoop 每一步「让谁跑、跑 prefill 还是 decode」。核心定位：调度器只做决策、不碰张量，持有 block_manager、block_trie、state_manager 三件 KV 家当与按状态分桶的 seq_manager，产出 SchedulerOutput.running 交由 InputsMaker 装车。

引入的关键概念：持续批处理（continuous batching）把请求加入/退出降到每步 forward，同 batch 可混跑 prefill 与 decode，是高吞吐根源；序列状态机以 WAITING/READY/RUNNING/STOPPED 四态分桶驱动调度。schedule() 为唯一入口，按调用方已决的 is_prefill 分派——_schedule_decoding 为 READY 序列补块，先驱逐 hanging/waiting，不够则抢占 batch 末尾 running（recompute 模式打回 WAITING 重算）；_schedule_prefill 是准入循环，过批容量、token 预算、长 prefill 三道门槛，依固定顺序 match→驱逐→allocate→publish，前缀缓存命中可试探性救活超预算候选并支持回滚。max_prefill_token_num（默认 8192）身兼单批 token 预算上限与长上下文分块单块上限二职，保护 TTFT 与显存。
