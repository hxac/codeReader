本讲剖析 LMDeploy PyTorch 后端的「模型 Patch 重写机制」。核心动机是：原生 HF 模型不支持 Paged Attention、张量并行、量化线性层与 CUDA Graph，因此 LMDeploy 采用「先换类、后灌权重」策略——不加载 HF 原模型再逐层改，而是在实例化时直接换成同名不同实现的优化类。

关键术语包括：**arch 名**（`config.json` 的 `architectures` 字段）、**qualname**（全限定名）、**MODULE_MAP**（arch → 实现类 qualname 字符串的纯数据注册表，用字符串而非类是为延迟导入）。

主链路为 `build_patched_model → build_model_from_hf_config → _get_model_class`，查表优先级是 `auto_map` → `architectures`，每个 arch 先经 `_raise_if_removed_model` 校验。`_get_module_map` 将三表合一：主表 ⊕ 设备专属表 ⊕ 用户自定义表。

重写有两条路径：**整模型级直接替换**（主链路，按 arch 精确匹配）与**子模块级替换**（`get_rewrite_cls` 走 full name → class name → submodname 三级降级 + 正则兜底），两者共用 MODULE_MAP。

三张辅助表分工：`DEVICE_SPECIAL_MODULE_MAP`（非 cuda 设备覆盖）、`CUSTOM_MODULE_MAP`（用户扩展，优先级最高，可不改源码支持新模型）、`REMOVED_MODEL_MAP`（已下线模型的清晰报错）。合并优先级为 `MODULE_MAP < 设备表 < CUSTOM_MODULE_MAP`。
