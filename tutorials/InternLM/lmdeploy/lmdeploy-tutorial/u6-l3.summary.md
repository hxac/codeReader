本讲拆解 TurboMind 后端加载 HuggingFace 模型时「配置解析」与「权重读取」两件事。核心结论是：仓库已无独立的 `lmdeploy convert` 命令，转换在 `pipeline()` 加载时**进程内即时完成**，不再产出独立 TM 权重目录。

关键入口 `converter.get_tm_config` 是「定调函数」：六步把 `dtype / model_format / group_size / session_len` 从半填的 `engine_config` + HF 配置敲定，**就地改写 engine_config**，并造出源模型对象与 `WeightFormatResolver`。需厘清三个术语：`dtype`（计算精度，解析优先级为硬件能力→HF dtype→torch_dtype，不支持 bf16 则降级 fp16）、`model_format`（磁盘权重/量化格式，如 hf/awq/gptq/fp8/mxfp4/compressed-tensors）、`group_size`（量化分组大小，常为 128）。

权重侧采用贯穿全讲的「策略对象 + 解析器」哲学：`weight_format.py` 用六个 `WeightFormat` 子类（各声明 `suffix_map/weight_dtype/has_zero_point` 与 `accepts/normalize/pack`）屏蔽格式差异，量化格式排前、`TrivialFormat` 兜底在后，AWQ/GPTQ/compressed-tensors 会强制 fp16。`WeightFormatResolver.resolve` 按优先级逐个询问，命中即产出统一 TM 布局的 `Linear` 权重束。由此源模型代码（如 `models/llama.py`）对量化与否完全无知。

另需注意：磁盘加载职责已从 `loader.py` 搬到 `checkpoint.py`——`create_checkpoint` 按六级优先级挑 safetensors/pytorch 后端，`Prefix` 做路径算术，`Checkpoint` 提供 `get/has/pop/keys`；遗留的 `loader.py` 仅 `StateDictLoader` 服务于在线 RL 的 `update_params` 队列路径。整条链路为 `_from_hf → get_tm_config + ModelLoader.export → create_checkpoint → model.model(Prefix) → resolver.resolve`。
