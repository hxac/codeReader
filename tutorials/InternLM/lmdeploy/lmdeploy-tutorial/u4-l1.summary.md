本讲剖析 PyTorch 后端核心 `Engine` 类，覆盖四个模块。`Engine` 采用 Actor 模型，构造期装配 executor/scheduler/req_manager 但不启动循环；生命周期三件套分工为 `start()` 幂等拉起 async_loop 主循环、`stop()` 仅取消主循环任务、`close()` 额外清理 cuBLAS 显存并经 `_loop_finally` 收尾。

`RequestManager` 是跨执行流的「信箱」：`RequestSender` 把 Request 入队并等待 Response.event，主循环在 `step()` 中按 `request_priority` 顺序（如 STOP_ENGINE 先于 ADD_MESSAGE）将请求分桶派发到 callbacks 表，保证调度器状态仅在主循环单流修改、天然线程安全。`_bind_request_manager` 以命令模式把 ADD_SESSION/STOP_SESSION/END_SESSION/ADD_MESSAGE 四类请求绑定到 `_on_*` 回调。

`_on_add_message` 是推理真正入口：校验会话→预处理多模态输入→新建或续写 SchedulerSequence→裁剪 max_new_tokens→把 resp 挂到序列上供 forward 回信。关键区分：引擎面 Response 带 asyncio.Event，与用户面同名类不同；`cuda_stream_id` 仅为双后端 API 对齐保留的空形参。派发由 EngineLoop.preprocess_loop 反复调 step() 驱动。
