本讲讲解 `chisel3.experimental.hierarchy` 的 Definition/Instance/Instantiate 三件套，把「定义模块」与「实例化模块」在 API 层显式分离。`Definition(new Mod)` 在隔离的 `DynamicContext` 里把模块体细化一次（持 `Proto` 原型，即蓝图），可被多份共享；`Instance(defn)` 不再细化，仅用伪模块 `ModuleClone` 克隆端口、指向新实例（持 `Clone`）。可包装的类型须 `T <: BaseModule with IsInstantiable`，由 `@instantiable` 宏自动补该父类，并由 `@public` 宏把字段改写成 `_lookup` 扩展方法——故 `i.in` 实为查表调用。`Instantiate(new Mod(args))` 是带缓存的糖：按参数结构去重（靠 `DataBox` 把 `Data` 转结构相等），但不能用命名参数、内部类缓存失效。查表由 `Lookupable` 类型类驱动：`Data`/`Module` 被克隆进调用者上下文（每实例不同，结果缓存于 `Hierarchy.cache`），而 `IsLookupable` 与基础类型原样返回（所有实例相同）；`_lookup` 的 `MacroGenerated` 门禁强制只能经 `@public` 触发。相比传统 `Module(new ...)` 每次重细化、内部不可见，hierarchy 方式定义一次、内部 `@public` 字段可跨层按名访问。
