本讲是单元 6（标准库 chisel3.util）首篇，讲解 chisel3.util 几乎所有生成器（Queue、Arbiter、流水线寄存器等）共同依赖的握手接口——Ready/Valid（解耦）协议。涉及的最小模块与类：抽象基类 ReadyValidIO（位于 ReadyValidIO.scala）、具体子类 DecoupledIO 与工厂 Decoupled（Decoupled.scala）、IrrevocableIO 与工厂 Irrevocable（Irrevocable.scala）、独立接口 Valid 与工厂 Valid（Valid.scala）。

核心结论：ReadyValidIO 把方向写死在字段里——ready 为 Input、valid/bits 为 Output，因此「原样使用即生产者」，消费者需用 Flipped 整体翻转。一笔数据传输当且仅当同一时钟沿 ready 与 valid 同时为高，该条件即 fire（＝ready && valid）；ready 为低即反压，会直接降低带宽。便捷方法 fire/enq/deq/noenq/nodeq 通过伴生对象的隐式类统一注入所有 ReadyValidIO 子类，只是连线语法糖、不引入额外硬件。

三种变体按承诺由弱到强：Valid（无 ready、单向广播，直接继承 Bundle，其 fire 退化为 valid）；DecoupledIO（三信号、对时机无承诺，最宽松、默认之选）；IrrevocableIO（三信号外加「不可撤回」承诺：valid 高时 bits 不变、且不拉低直到 ready 拉高过一次）。前两者结构相同可经工厂互转（升级须在消费侧、降级须在生产侧，各带方向 require 检查），Valid 因缺 ready 与另两者结构不同，不能直接 <> 对接。命名澄清：仓库中不存在 ValidIO 类，规格中该词指 Valid。这些概念是后续 u6-l2 Queue（正确传递反压的 FIFO）等生成器的前置基础。
