本讲解答「NVDLA 固定规格写在哪、如何变成 RTL 可用的宏」。核心是 spec/defs 配置体系，含三个最小模块：用户开关层、翻译校验层、生成层。

`nv_full.spec` 是规格的单一事实来源，用约 32 个 `#define` 用户开关（分特性开关、数值档位、接口参数三类）描述 nvdlav1 固定特性，末尾 `#include "projects.spec"` 委托翻译。`projects.spec` 是翻译+校验表，把用户开关翻成标准 `%define NVDLA_* [值]` 内部宏，并用 `#else #error` 强制每个二选一开关都作出选择，实现构建期早失败（局限：仅校验「至少选一个」，因 CPP `#if/#elif` 短路不查「重复选」）。

`Makefile` 编排两级流水线：CPP 先把 spec 求值成中间 `project.def`（只剩 `%define`），defgen（Perl，核心正则 `^\s*%define\s*(\w+)\s*(\d+)?`）再按 `-b c/v/pl/py` 后端加前缀，改写成 `project.h`/`project.vh`，最后删中间文件。

关键术语：三层宏名（带后缀的用户开关、带 `NVDLA_` 前缀的内部宏、最终头文件宏）、`%define` 中间语言、vcp 配置裁剪。最关键结论是两类宏消费路径的分野：ENABLE 类特性宏经 `vcp -imacros project.h` 真实裁剪 RTL 的 `#ifdef`（配置裁剪发生在 CPP 阶段，故 RTL 用 `#ifdef` 而非 `` `ifdef ``）；数值尺寸宏（如 `NVDLA_MAC_ATOMIC_C_SIZE`）主要喂养 C-model 配置头 `nvdla_config_large.h`，vmod 侧阵列规模已烤成字面量——改 spec 数值并不能动态重配 RTL。
