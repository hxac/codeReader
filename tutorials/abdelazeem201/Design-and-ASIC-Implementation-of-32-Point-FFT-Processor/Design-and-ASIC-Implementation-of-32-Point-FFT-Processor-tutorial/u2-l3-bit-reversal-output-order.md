# 位反转与输出顺序还原

## 1. 本讲目标

学完本讲后，你应当能够：

- 说清楚为什么 radix-2 DIF 流水线吐出的样本是「乱序」的，并且这个乱序有严格规律——**位反转（bit-reversal）**。
- 逐字符读懂 `SIM/FFT.py` 末尾那句 `r = int('{:05b}'.format(i)[::-1], 2)`，并能自己写一个 5 位位反转函数。
- 解释为什么硬件需要一个独立的「排序模块」把输出整理成正常顺序，并在 `RTL/FFT.v` 的大 `case(y_1)` 里找到它与软件位反转的对应关系。

## 2. 前置知识

承接 u2-l1：radix-2 DIF 把 32 点 DFT 分成 5 级蝶形，每级做「加减分支 + 乘旋转因子」。本讲只关心一件事：**5 级算完之后，结果到底按什么顺序排布**。

请先记住两点：

- DIF 的特征是「先加减、后乘旋转因子」。这种结构让最终结果**不是**按频率 \(0,1,2,\dots,31\) 排列，而是按频率编号的**二进制位反转顺序**排列。
- 项目里「对的」输出顺序是自然顺序 \(0,1,2,\dots,31\)（testbench 的 golden 文件就是这个顺序），所以算完之后必须做一次「还原」。

本讲不需要你懂硬件——我们先用 Python 把算法层面的位反转讲透，最后才点一下它在 RTL 里的落点。

## 4. 核心概念与源码讲解

### 4.1 位反转索引的概念

#### 4.1.1 概念说明

把一个整数写成 \(m\) 位二进制，再把这串二进制「左右翻转」读出来，得到的新数就是它的**位反转索引**。

例如在 5 位二进制下：

\[
\text{bitrev}_5(3) = \text{bitrev}_5(00011_2) = 11000_2 = 24
\]

所以 3 和 24 互为位反转。

**为什么 radix-2 DIF 的输出天然呈位反转顺序？** 直觉解释：DIF 每一级都在按「频率索引的某一位」做二分——第一级按最低位 bit0（偶/奇频率）分开，第二级按 bit1 分开……一直到 bit4。**最早被分出去的是最低位，最后才决定最高位。** 于是样本物理产出的先后顺序是「最低位优先」读的，而频率编号是「最高位优先」写的——两者正好是位反转关系。

结论：经过 5 级蝶形后，**第 \(i\) 个吐出的样本，对应的真实频率是 \(\text{bitrev}_5(i)\)，而不是 \(i\)**。

#### 4.1.2 核心流程

32 点（5 位）的完整位反转映射节选如下：

| 自然索引 \(i\) | 二进制 | 反转二进制 | \(\text{bitrev}_5(i)\) |
|---|---|---|---|
| 0 | 00000 | 00000 | 0 |
| 1 | 00001 | 10000 | 16 |
| 2 | 00010 | 01000 | 8 |
| 3 | 00011 | 11000 | 24 |
| 16 | 10000 | 00001 | 1 |
| 31 | 11111 | 11111 | 31 |

位反转有两个实用性质：

1. **自反性**：\(\text{bitrev}(\text{bitrev}(i)) = i\)，即它自己的逆操作。
2. **固定置换**：对 \(N=32\)，\(0\dots31\) 这 32 个数被一一对应重新排列，既不重复也不遗漏。

#### 4.1.3 源码精读

`FFT.py` 在 5 级蝶形算完之后，正是用位反转把结果「摆正」：

[SIM/FFT.py:189-196](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/SIM/FFT.py#L189-L196) —— 位反转还原并打印最终结果（`final_ans_r` / `final_ans_i`）。

核心一行：

[SIM/FFT.py:191](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/SIM/FFT.py#L191) —— 把产出序号 `i` 位反转为自然频率 `r`。

随后 `final_ans_r[r] = stage5_r[i]`：把第 `i` 个产出的样本写进自然位置 `r`。这跟 4.1.1 的结论完全一致——产出位 `i` 对应频率 \(\text{bitrev}_5(i)\)。

#### 4.1.4 代码实践

**目标**：用纸笔验证位反转的自反性。

**步骤**：

1. 取 \(i = 13\)，写成 5 位二进制 `01101`。
2. 翻转得 `10110` = 22，即 \(\text{bitrev}_5(13) = 22\)。
3. 再翻一次：`10110` → `01101` = 13，回到原值。

**预期结果**：连续位反转两次回到原数，验证 \(\text{bitrev}(\text{bitrev}(i)) = i\)。

#### 4.1.5 小练习与答案

**Q1**：5 位位反转下，\(\text{bitrev}_5(10) =\) ？

**答**：10 = `01010`，翻转 = `01010` = 10。这种「镜像对称」的数位反转后不变。

**Q2**：0..31 中，位反转后保持不变的数有几个？

**答**：5 位二进制形如 `abcba`（第 1 位=第 5 位、第 2 位=第 4 位，第 3 位任意）才对称，自由位有 a、b、c 共 3 个，所以有 \(2^3 = 8\) 个。

### 4.2 Python 位反转还原实现

#### 4.2.1 概念说明

`FFT.py` 用了一个非常 Pythonic 的「一行式」做位反转。本节把它拆开，让你看懂每一个字符。它的思路是把「数值 → 定宽二进制字符串 → 反转字符串 → 回到数值」四步串起来。

#### 4.2.2 核心流程

伪代码：

```
输入 i（0..31）
s     = 把 i 格式化为 5 位二进制字符串（高位补 0）
s_rev = 把 s 反转
r     = 把 s_rev 当作二进制解析回整数
```

四步对应到 Python：

- `'{:05b}'.format(i)`：格式说明符 `05b` 表示「二进制（`b`）、最小宽度 5、不足补 0」。例如 `i=3` → `'00011'`。
- `[::-1]`：切片步长 −1，即整串反转。`'00011'` → `'11000'`。
- `int(..., 2)`：按二进制解析为整数。`'11000'` → `24`。

#### 4.2.3 源码精读

[SIM/FFT.py:190-193](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/SIM/FFT.py#L190-L193) —— 循环遍历每一个产出样本 `i`，算出 `r`，把实部、虚部写进自然位置 `r`。

注意：循环变量 `i` 走的是「产出顺序」0..31，而写入下标 `r` 是「自然顺序」。这正是位反转还原的本质——**读位反转顺序、写自然顺序**。

#### 4.2.4 代码实践（本讲必做）

**目标**：写一个独立函数 `bitrev5`，验证它与 `FFT.py` 第 191 行完全等价，并打印若干映射。

**操作步骤**：新建 `bitrev.py`，内容如下（示例代码）：

```python
# 示例代码：5 位位反转函数
def bitrev5(i):
    return int('{:05b}'.format(i)[::-1], 2)

for i in range(32):
    print(i, '->', bitrev5(i))
```

运行 `python3 bitrev.py`。

**需要观察的现象**：终端逐行打印 `0 -> 0`、`1 -> 16`、`2 -> 8`、`3 -> 24` ……

**预期结果**：`0→0`、`1→16`、`3→24` 这几条必须命中（与 4.1.2 的表格一致）。由于 `bitrev5` 的函数体与 `FFT.py` 第 191 行逐字符相同，两者对所有 32 个 `i` 都必然一致。

> 待本地验证：本环境未执行该命令，但函数体与源码第 191 行完全相同，逻辑等价性可静态确认。

#### 4.2.5 小练习与答案

**Q1**：把 `'{:05b}'` 改成 `'{:04b}'` 会怎样？

**答**：只取 4 位、宽度 4，`bitrev` 退化成 4 位（\(N=16\)）版本，对 32 点就是错的。要点：**宽度必须等于 \(\log_2 N = 5\)**。

**Q2**：不用字符串，纯算术怎么实现 `bitrev5`？

**答**：逐位提取再移位拼接：

```python
# 示例代码：纯算术版 5 位位反转
def bitrev5(i):
    r = 0
    for k in range(5):
        r = (r << 1) | ((i >> k) & 1)
    return r
```

### 4.3 正常顺序输出的需求与硬件对应

#### 4.3.1 概念说明

**为什么必须还原成自然顺序？** 因为 testbench（见 u1-l3）拿 golden 文件逐行比对时，第 \(k\) 行就是频率 \(k\) 的期望值。如果硬件直接吐位反转顺序，第 \(k\) 拍给的其实是频率 \(\text{bitrev}(k)\)，跟 golden 对不上，SNR 会直接崩。

软件可以优雅地用一行位反转下标解决；硬件里「动态算位反转 + 写 RAM」需要额外的地址生成器。本项目的设计者选了一个更直接的方案：**把整个 32 项置换「展开」成一张查找表，写进一个大 `case` 语句**。这就是 `FFT.v` 里那段看起来很长的 `case(y_1)`。

#### 4.3.2 核心流程

硬件排序模块的工作流：

1. 第 5 级蝶形按**位反转顺序**逐拍吐出 `out_r` / `out_i`。
2. 用计数器 `y_1`（= `count_y − 1`）标记「这是第几个到达的样本」。
3. `case(y_1)` 查表：第 `y_1` 个样本 → 写进二维缓冲 `result_r[目的索引]`。
4. 32 个样本到齐（`over` 标志拉高）后，按**自然顺序**逐拍读出 `result_r[0..31]` 作为 `dout`。

所以硬件没有「算」位反转，而是「查」一张预先算好的位反转地址表。

#### 4.3.3 源码精读

[RTL/FFT.v:37-40](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/RTL/FFT.v#L37-L40) —— 声明 `result_r` / `result_i` 两个 32 项数组作为排序缓冲（`_ns` 是下一拍的组合逻辑值）。

[RTL/FFT.v:56-60](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/RTL/FFT.v#L56-L60) —— `y_1` 是到达样本序号：`y_1 = count_y − 1`（当 `count_y>0`）。

[RTL/FFT.v:313-458](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/RTL/FFT.v#L313-L458) —— 大 `case(y_1)`：每个分支把当前 `out_r[23:8]` 写进 `result_r_ns[目的索引]`。这就是「硬件版位反转器」。

节选前几条：

```verilog
5'd0 : result_r_ns[31] = out_r[23:8];
5'd1 : result_r_ns[15] = out_r[23:8];
5'd2 : result_r_ns[7 ] = out_r[23:8];
5'd3 : result_r_ns[23] = out_r[23:8];
```

把全部 32 条的目的索引按 `y_1` 顺序列出来是：

```
31, 15, 7, 23, 3, 19, 11, 27, 1, 17, 9, 25, 5, 21, 13, 29,
 0, 16, 8, 24, 4, 20, 12, 28, 2, 18,10, 26, 6, 22, 14, 30
```

**进阶观察**（可选）：把这张表和软件对照，可发现规律——**目的索引 \(D = (\text{bitrev}_5(y_1) - 1) \bmod 32\)**。例如：

- `y_1=16`：\(\text{bitrev}_5(10000)=00001=1\)，\(1-1=0\) → 对应 `5'd16 : result_r_ns[0]`。
- `y_1=1`：\(\text{bitrev}_5(00001)=10000=16\)，\(16-1=15\) → 对应 `5'd1 : result_r_ns[15]`。
- `y_1=0`：\(\text{bitrev}_5(0)=0\)，\(0-1 \equiv 31 \pmod{32}\) → 对应 `5'd0 : result_r_ns[31]`。

这里的「−1」来自 `y_1 = count_y − 1` 的一拍偏移（计数器比数据有效早一拍），完整时序推导放在 u4-l2 / u4-l3。要点是：**硬件 `case` 表与软件位反转是同一个置换**。

[RTL/FFT.v:303-306](https://github.com/abdelazeem201/Design-and-ASIC-Implementation-of-32-Point-FFT-Processor/blob/89ce7665eca8b77a0c48e610dab11928108566b6/RTL/FFT.v#L303-L306) —— 排序完成后，按 `y_1_delay` 自然顺序读 `result_r` 作为 `dout` 输出。

#### 4.3.4 代码实践

**目标**：从 `FFT.v` 提取 `case(y_1)` 的 `y_1 → 目的索引` 映射，用 Python 复现并验证它等价于位反转换置。

**步骤**：

1. 在 `FFT.v` 第 313–458 行，把每个 `5'dK : result_r_ns[D]` 里的 `K` 和 `D` 抄成一张表。
2. 运行下面这段示例代码：

```python
# 示例代码：从 case 表提取硬件置换 hw[K] = D
hw = {0:31,1:15,2:7,3:23,4:3,5:19,6:11,7:27,8:1,9:17,10:9,11:25,
      12:5,13:21,14:13,15:29,16:0,17:16,18:8,19:24,20:4,21:20,
```

**预期结果**：打印 `True`，说明硬件大 `case` 确实是位反转换置（带一拍偏移）。

> 待本地验证。

#### 4.3.5 小练习与答案

**Q1**：为什么硬件用 `case` 查表，而不是像软件那样动态算 `bitrev`？

**答**：硬件算 5 位位反转需要额外的地址生成逻辑，且可能多耗一拍；展开成静态 `case` 表后，综合下来是一张纯组合查找表，时序干净、面积可控，对 32 点这种固定规模非常划算。

**Q2**：`result_r` 为什么要 32 项、且等 32 个样本到齐（`over` 拉高）才开始输出？

**答**：位反转换置是「全到齐才能确定每个自然位置写谁」的全局置换——最晚到的样本（`y_1` 最大那个）可能要写 `result_r[0]`。所以必须缓冲整帧 32 个样本，再按自然顺序读出。

## 5. 综合实践

把位反转「端到端」走一遍，串起本讲三个模块。

**任务**：用 Python 模拟「DIF 产出 → 位反转还原 → 自然顺序输出」全过程（不用真算 FFT，构造一个可识别的伪数据即可）。

```python
# 示例代码：端到端位反转还原演示
N = 32

def bitrev(i):
    return int('{:05b}'.format(i)[::-1], 2)

# 假装 stage5[i] 是 DIF 第 i 个产出样本，值取 i*10 方便追踪
stage5 = [i * 10 for i in range(N)]

# 位反转还原：产出位 i 的样本属于频率 bitrev(i)
final_ans = [0] * N
for i in range(N):
    final_ans[bitrev(i)] = stage5[i]

print("自然顺序输出 final_ans:")
for k in range(N):
    print(f"freq {k:2d}  <- 产出位 {bitrev(k):2d}  值 {final_ans[k]}")
```

**观察与验证**：

- `final_ans[0]` 应等于 `stage5[0] = 0`（\(\text{bitrev}(0)=0\)）。
- `final_ans[16]` 应等于 `stage5[1] = 10`（因为 \(\text{bitrev}(1)=16\)，即产出位 1 的样本属于频率 16）。
- `final_ans` 会**单调递增**吗？不会。因为 `final_ans[k] = stage5[bitrev(k)]`，而 \(\text{bitrev}(k)\) 并不单调，所以输出会「乱跳着填」，但填完后恰好是完整的自然顺序 \(0,10,20,\dots,310\)。请实际跑一遍确认这一点。

## 6. 本讲小结

- radix-2 DIF 每级按频率的某一位二分，导致 5 级算完后样本按**位反转顺序**产出：产出位 \(i\) 对应频率 \(\text{bitrev}_5(i)\)。
- `FFT.py` 用一行 `r = int('{:05b}'.format(i)[::-1], 2)` 做位反转，把产出位 `i` 的样本写进自然位置 `r`。
- 位反转是**固定置换**且**自反**；\(N=32\) 时必须用 5 位宽度。
- 硬件 `FFT.v` 不动态算位反转，而是把整个置换展开成 `case(y_1)` 查找表，先写到 `result_r` 缓冲，32 个样本到齐后顺序读出。
- 硬件 `case` 表的目的索引满足 \(D = (\text{bitrev}_5(y_1) - 1) \bmod 32\)，与软件是同一个置换（「−1」来自 `count_y` 的一拍偏移）。
- golden 文件是自然顺序，所以这一步还原是「算得对」的必要条件。
