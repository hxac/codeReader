## 2. 前置知识

本讲几乎不需要任何前置知识，但下面几个日常概念会让你读得更顺：

- **加密（encryption）**：把一段"看得懂的信息（明文）"用某种方法变成"别人看不懂的形式（密文）"。
- **密钥（key）**：加密时用到的一串秘密数字，就像锁和钥匙——同样的锁，用不同的钥匙锁出来的结果不同。
- **对称加密**：加密和解密用的是 **同一把钥匙**。与之相对的是"非对称加密"（比如公钥/私钥），本仓库属于前者。
- **块（block）**：一段固定长度的数据。AES 每次只处理一个固定大小的块，本讲后面会看到它固定是 128 位（16 字节）。
- **硬件描述语言 Verilog**：一种用来"画"数字电路的文字语言。你写下的每一行 Verilog，最终会被综合（翻译）成芯片里真实的逻辑门。本手册从下一篇开始会大量阅读 Verilog，本篇只做概念铺垫。

如果你对"字节（byte）、位（bit）、十六进制（0x...）"这些词陌生，只要记住：**1 字节 = 8 位**，十六进制 `0x63` 就是一个 8 位字节，这就够读懂本讲了。

### 4.2 AES 算法背景

#### 4.2.1 概念说明

**AES（Advanced Encryption Standard，高级加密标准）** 是一种 **对称分组密码**，由美国 NIST 于 2001 年正式确立（标准号 FIPS-197），如今几乎是全世界最常用的加密算法之一，广泛用于 Wi-Fi（WPA2）、硬盘加密、TLS/SSL 等。

理解 AES，抓住三个关键词：

1. **对称**：加密和解密用同一把密钥。
2. **分组（块）**：AES 不是一位一位地处理数据，而是把数据切成固定大小的 **块**，每块 128 位（16 字节），一次加密一块。
3. **密钥长度决定轮数**：密钥越长，加密轮数越多，越安全也越慢。AES 允许三种密钥长度：

| 密钥长度 | 密钥位数 | 加密轮数 \(N_r\) | 本仓库是否支持 |
| --- | --- | --- | --- |
| AES-128 | 128 位 | 10 | ✅ 支持 |
| AES-192 | 192 位 | 12 | ❌ 不支持 |
| AES-256 | 256 位 | 14 | ✅ 支持 |

> 本仓库 **只支持 AES-128 和 AES-256**，跳过了 AES-192。这从源码的 `keylen` 二选一开关也能印证。

#### 4.2.2 核心流程

一次 AES 加密，可以粗略看成"先和密钥异或一下，然后重复执行若干 **轮** 变换"。一轮内部包含四个基本变换：

- **SubBytes（字节代换）**：用一张 256 项的查找表（S-box）逐字节替换。
- **ShiftRows（行移位）**：把 16 字节按矩阵的行做循环移位，打乱位置。
- **MixColumns（列混淆）**：对每一列做有限域上的矩阵乘法，让字节相互扩散。
- **AddRoundKey（轮密钥加）**：把当前块和这一轮的"轮密钥"做按位异或。

完整的加密流程用伪代码表示：

```
加密 AES_encrypt(明文块 P, 密钥 K):
    1. 由 K 通过"密钥扩展"生成 (Nr+1) 个轮密钥 RK[0..Nr]
    2. state = P
    3. state = AddRoundKey(state, RK[0])      # 初始轮：只异或
    4. for round = 1 to Nr-1:                  # 中间 (Nr-1) 轮：四步全做
           state = SubBytes(state)
           state = ShiftRows(state)
           state = MixColumns(state)
           state = AddRoundKey(state, RK[round])
    5. state = SubBytes(state)                 # 最终轮：不做 MixColumns
       state = ShiftRows(state)
       state = AddRoundKey(state, RK[Nr])
    6. return state                            # 这就是密文块 C
```

解密是加密的"逆过程"：用逆变换（InvSubBytes、InvShiftRows、InvMixColumns）按相反顺序应用。

关于"轮数"的数学关系，可以记一个简洁的式子——密钥长度（以 32 位字为单位记作 \(N_k\)）与轮数 \(N_r\) 的对应：

\[ N_r = N_k + 6 \quad \text{其中 } N_k \in \{4,6,8\} \]

- AES-128：\(N_k=4\)，\(N_r=10\)。
- AES-256：\(N_k=8\)，\(N_r=14\)。

> 数学小贴士：AES 的"乘法"不是普通十进制乘法，而是在一个叫 **GF(\(2^8\))** 的有限域（伽罗瓦域）里做的——本质上是一组带"异或"和"特殊移位"规则的 8 位运算。本讲只需知道"有这么一套字节级运算"即可；具体的 S-box 查表和 MixColumns 矩阵乘法，会在进阶篇 [u2-l2 S-box ROM](u2-l2-sbox-and-inverse-sbox-rom.md) 与 [u2-l4 加密四变换函数](u2-l4-encipher-datapath-functions.md) 里结合源码细讲。

把上面的抽象概念落回到本仓库的真实代码上。前面已经看到密钥存储，这里再看密钥和块的寄存器声明，体会"块固定 128 位、密钥最多 256 位"：

**预期结果**：

- 数据块 = \(4 \times 32 = 128\) 位 → 正好一个 AES 块。
- 密钥 = \(8 \times 32 = 256\) 位 → 容得下 AES-256；选 AES-128 时只用前 4 个字。

#### 4.2.5 小练习与答案

**练习 1**：AES-256 要执行多少轮加密？其中"中间轮"有几轮？

**参考答案**：AES-256 共 \(N_r=14\) 轮。除掉初始轮（1 次 AddRoundKey）和最终轮（1 轮，无 MixColumns）外，中间轮有 \(14-1=13\) 轮——这 13 轮每轮都完整执行 SubBytes / ShiftRows / MixColumns / AddRoundKey 四步。

**练习 2**：如果我换成 AES-128，数据块的大小会变成 128 位吗？

**参考答案**：不会变。AES 的 **数据块固定是 128 位**，与密钥长度无关；改变的只是密钥长度（128 位 vs 256 位）和加密轮数（10 轮 vs 14 轮）。源码里 `block_reg` 永远是 4 个字，而 `key_reg` 的实际使用字数才随 `keylen` 变化。

### 4.3 ECB 工作模式

#### 4.3.1 概念说明

AES 本身只定义了"怎么加密 **一个 128 位的块**"。但现实里我们要加密的数据往往远不止一个块（比如一个文件）。**工作模式（mode of operation）** 就是规定"如何把多个块串起来处理"的方法。最简单的工作模式叫 **ECB（Electronic Codebook，电子密码本）**。

ECB 的规则简单到一句话：**每个 128 位块都用同一把密钥独立加密，块与块之间互不影响**。

本仓库实现的正是 **单块 ECB**：它一次只吃进去一个 128 位明文块，吐出一个 128 位密文块。这正对应顶层模块"写一个 BLOCK、读一个 RESULT"的接口形态。

ECB 的优缺点很鲜明：

| 特性 | 说明 |
| --- | --- |
| ✅ 优点 | 极度简单；各块可独立加密，天然支持并行 |
| ❌ 缺点 | 相同的明文块会被加密成相同的密文块，从而 **泄露数据模式**（经典示例：用 ECB 加密一张位图，轮廓依然清晰可见） |
| 适用场景 | 单块加密、密钥封装，或作为更复杂模式的内部构件 |

> 正因 ECB 有模式泄露问题，工程里处理长数据通常改用 CBC、CTR、GCM 等带"链式反馈"或"计数器"的模式。但本仓库聚焦"把 AES 单块核做对、做省电"，所以选择 ECB 单块是合理的起点。

#### 4.3.2 核心流程

ECB 单块加解密的黑盒可以用下面这张图表示：

```
            ┌───────────────────────────────┐
明文块 P ──▶│                               │──▶ 密文块 C
(128 位)    │       AES 单块核 (ECB)        │    (128 位)
            │   C = AES_Encrypt(P, K)       │
密钥   K ──▶│                               │
(128/256)   └───────────────────────────────┘

            ┌───────────────────────────────┐
密文块 C ──▶│                               │──▶ 明文块 P
(128 位)    │       AES 单块核 (ECB)        │    (128 位)
            │   P = AES_Decrypt(C, K)       │
密钥   K ──▶│                               │
(128/256)   └───────────────────────────────┘
```

注意三点：

1. **加密和解密共用同一把密钥 K**（对称）。
2. **数据块进/出永远是 128 位**，只有密钥 K 有 128/256 两种规格。
3. ECB **没有初始向量（IV）**，也没有块间链接，所以同一组 \((P, K)\) 永远得到同一个 \(C\)。

#### 4.3.3 源码精读

ECB"单块"的接口形态，在顶层模块的地址映射里体现得非常直观——正好 4 个输入字、4 个输出字：

- [rtl/aes.v:46-50](https://github.com/abdelazeem201/ASIC-implementation-of-AES/blob/585f265f91338bea074cebb8c31a3eb602e7cdae/rtl/aes.v#L46-L50)：`ADDR_BLOCK0`(0x20) ~ `ADDR_BLOCK3`(0x23) 是 **明文/密文块的 4 个字**（输入），`ADDR_RESULT0`(0x30) ~ `ADDR_RESULT3`(0x33) 是 **结果的 4 个字**（输出）。一个块进、一个块出，正是 ECB 单块接口。

至于"什么时候算加密、什么时候算解密、什么时候开始、什么时候结束"，则由控制与状态寄存器管理：

- [rtl/aes.v:31-37](https://github.com/abdelazeem201/ASIC-implementation-of-AES/blob/585f265f91338bea074cebb8c31a3eb602e7cdae/rtl/aes.v#L31-L37)：`ADDR_CTRL` 里的 `CTRL_INIT_BIT`（触发密钥扩展初始化）、`CTRL_NEXT_BIT`（触发一次加/解密），以及 `ADDR_STATUS` 里的 `STATUS_READY_BIT`（核是否空闲可用）、`STATUS_VALID_BIT`（结果是否已就绪有效）。

这就是一次 ECB 单块操作的控制语义：写好密钥与配置后，先触发 `init` 做密钥扩展，再写一个块、触发 `next`，等 `valid` 拉高后读 `RESULT`。这套"写寄存器—触发—读结果"的访问序列，会在 [u1-l4 顶层接口与地址映射](u1-l4-top-interface-and-address-map.md) 里完整讲解。

#### 4.3.5 小练习与答案

**练习 1**：为什么说 ECB 不适合直接加密一幅图片这类"多块、有重复内容"的数据？

**参考答案**：因为 ECB 下每个块独立加密、相同明文块得到相同密文块。图片里大片相同/相近的像素会产生大片相同/相近的密文块，导致原图的轮廓和模式依然能被看出来，安全性不足。

**练习 2**：使用本核做一次 ECB 加密，需要用户提供"初始向量 IV"吗？为什么？

**参考答案**：不需要。ECB 模式没有块间链接，也就没有 IV 的概念；只需提供密钥 K 和明文块 P 即可得到密文块 C。（需要 IV 的是 CBC 等模式，本核未实现。）

## 6. 本讲小结

- 本仓库是一个用 **Verilog** 实现的 **AES 加解密硬件核**，目标是低功耗、较高吞吐，面向 FPGA 与 ASIC。
- AES 是 **对称分组密码**：数据块固定 **128 位**；密钥有 128/192/256 三种，本仓库支持 **128 和 256** 两种，对应 10 轮与 14 轮加密。
- 一次 AES 加密 = 初始 AddRoundKey + 若干"四变换轮"（SubBytes / ShiftRows / MixColumns / AddRoundKey）+ 最终轮（无 MixColumns）。
- 本仓库采用 **ECB 单块模式**：一个 128 位块进、一个 128 位块出，无初始向量、块间独立；简单但不宜直接加密有重复模式的数据。
- 顶层接口用 **写 BLOCK / KEY + 触发 init/next + 读 RESULT** 的寄存器访问方式呈现，恰好对应 ECB 单块语义。
- 重要习惯：**文档可能不完整**（README 只写了 256 位密钥），读硬件项目要养成"以源码为准"的意识。
