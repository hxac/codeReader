本讲围绕 MLC LLM 的对话模板协议展开，主角是 `Conversation`（定义于 `conversation_protocol.py`，继承 Pydantic BaseModel）。它同时承担两个职责：一是「模板」——描述模型期望的提示格式契约；二是「历史」——承载当前请求的多轮 `messages`。讲义拆为三个最小模块：字段结构、`as_prompt()` 拼装流程、stop 控制与多模态。核心字段分五组：系统提示（`system_template`/`system_message` 解耦、`system_prefix_token_ids`、`add_role_after_system_message`）、角色与模板（`roles`、`role_templates`，默认即 `MessagePlaceholders` 占位符）、消息与分隔符（`messages` 为 `(role,content)` 列表、`seps` 长度限 1 或 2 且 user 后用 `seps[0]`、assistant 后用 `seps[1]`、`role_content_sep`、`role_empty_sep`）、停止条件（`stop_str`/`stop_token_ids`）、函数调用。关键结论：`as_prompt()` 先渲染系统提示，再逐条拼「角色头+sep+内容+消息分隔符」，用 `separators[role=="assistant"]` 布尔下标选分隔符，`content is None` 时只留角色头给模型接写；它返回的是 `List[Union[str, ImageData]]` 而非字符串，因多模态 `image_url` 会混入 `ImageData`（需传 config）。停止条件不在 `as_prompt` 内消费，而是被引擎读出作「额外停止条件」注入采样配置——体现「模板是数据、消费在引擎」的分工。`strip_reasoning_in_history` 模仿 Qwen3 剥掉历史中最后一条 user 之前的 `<think>` 块。本讲为 u6-l2（注册表）与 u6-l3（OpenAI 兼容协议）的地基。
