本讲义打开 `ThreadedEngine` 的「心跳」`Step()`，讲清 MLC 推理引擎如何把 prefill/decode/verify 等行为拆成可插拔的动作。它分三个最小模块：第一，`EngineActionObj` 接口——每个动作只暴露一个纯虚方法 `Step(EngineState estate) → Array<Request>`，遵循「判定与收集 → 调模型 → 采样 → 写回状态」四段式骨架，无事可做时返回空数组，把「判断+执行」合一以避免竞态。第二，`EngineImpl::Step()` 调度循环是职责链：持有引擎创建时一次性装配的有序动作表 `actions_`，每次从优先级高到低遍历，第一个返回非空的动作获胜并执行后立即 return，即「每步只跑一个动作」；走到底则断言 `running_queue` 必须为空。该循环由 `ThreadedEngine::RunBackgroundLoop` 反复驱动。第三，`action_commons` 提供跨动作公共工具：`CreateEngineActions` 按 `speculative_mode`/`disaggregation`/`spec_draft_length` 装配动作链（列表顺序即优先级，`NewRequestPrefill` 几乎总在最前以压低 TTFT，分离式动作插最前）；`ActionStepPostProcess` 统一做步后处理，把 token 流式回调与完成清理解耦出动作，一次性打包回调减少 FFI 跨界；`PreemptLastRunningRequestStateEntry` 在显存不足时把最低优先级 running 请求「回退到 pending、保留已生成 token、释放 KV、插回 waiting 队首」。关键术语：动作（`EngineActionObj`）、步（`Step()`）、四段式骨架、动作表 `actions_`、职责链调度、抢占、首 token 在 prefill 顺便采样（TTFT）。`AutoSpecDecode` 展示动作可嵌套（动作套动作），全 11 个 Action 按进入/解码/推测/校验/分离五阶段分类。
