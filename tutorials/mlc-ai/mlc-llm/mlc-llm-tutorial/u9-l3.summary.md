本讲打开引擎「请求状态」黑盒。核心是把请求拆成两层：`Request` 是用户提交的**不可变契约**（id / inputs / prompt_tokens / generation_cfg），不可变以便在节点间转发重启（分离式推理）；可变进度全部放进状态层。状态层是四层包含：Request → `RequestState`（一棵树）→ `RequestStateEntry`（树节点＝一条生成分支，内含按模型隔离的 mstates 数组）→ `RequestModelState`（单模型视图）。`n>1`（多分支生成）时建成 `(n+1)` 节点的树（1 根共享 prompt 前缀 + n 子发散），entry 向量保持拓扑序。`RequestModelState` 承载两种 token：`committed_tokens`（已敲定、不可撤销，由 CommitToken 写）与 `draft_output_tokens`（推测解码草稿、待大模型 verify，由 AddDraftToken 写），两者互不干扰。`RequestStateStatus` 三态机 `kPending → kAlive → kFinished` 大致对应 waiting_queue → running_queue → 出队，但条目级 status 与请求级队列归属是两层概念。显存不足时**抢占（preemption）**回退队尾请求为 kPending、清空 draft、释放 KV，却把 `committed_tokens` 合并回 inputs 保留已生成内容，并插回 waiting 队首优先重调度——丢草稿与 KV，不丢用户产出；完成判定自底向上收敛，根 finished 时整请求出队。
