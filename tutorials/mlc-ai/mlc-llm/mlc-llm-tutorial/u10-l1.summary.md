本讲聚焦 `ModelObj` 暴露的分页 KV cache 管理接口，含三模块。一是分页 KV cache：借鉴 OS 虚拟内存，将 KV 切成固定大小 page（默认 `page_size=16`），用页表把逻辑序列映射到物理 page 并配空闲页池按需领还，把内部碎片限制在末页、彻底消除外部碎片；`CreateKVCache` 经 `FunctionTable` 调 model lib 建池，运行期按函数存在性在 `create_flashinfer_paged_kv_cache` 与 `create_tir_paged_kv_cache` 间选路。二是序列五种生命周期操作：`AddNewSequence`、`ForkSequence`（写时复制共享前缀，`fork_pos` 取自 `PrefixCacheMatchedResult.prefilled_offset`）、`RemoveSequence`（引用计数回收）、`PopNFromKVCache`（末尾回退）、`CommitAcceptedTokenTreeNodesToKVCache`（推测解码剪枝），均绑定到 `vm.builtin.kv_state_*` 全局函数。三是配置：`max_total_sequence_length` 决定全局 KV 预算（÷`page_size`=总页数）、`max_single_sequence_length` 定单条上限、`max_num_sequence` 定并发上限，由 `InferForKVCache` 在显存预算内反推。核心结论：`ForkSequence` 不复制数据、不额外占 KV 预算，是前缀缓存与并发生成的物理基础，使逻辑并发可超「物理容量÷平均长度」的朴素估算。
