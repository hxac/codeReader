# 实验分层与编译流水线映射

## 1. 本讲目标

本讲要回答一个看似简单却很关键的问题：**这个文档站为什么要把「写一个编译器」拆成 Lv0–Lv9 这么多阶段？这些阶段和「SysY → Koopa IR → RISC-V」这条流水线是什么关系？**

学完本讲，你应当能够：

- 说清楚文档把整个编译器实践拆成「十个阶段」的设计意图，以及这种增量式拆分为什么必要。

- 用「前端 / 中端 / 后端」的三段式模型描述编译器的结构，理解中间表示（IR）为什么能降低开发成本。
- 知道 Lv9+ 是可选的加分内容，并能列出它包含的几个进阶方向。

> 提示：本讲是「读文档」的讲义，不涉及运行代码。重点是理解**内容组织方式**和**编译器架构**这两条线索是如何在目录结构里交织在一起的。

## 4. 核心概念与源码讲解

### 4.1 十阶段分层设计：为什么把编译器拆成 10 步

#### 4.1.1 概念说明

「把 SysY 编译到 RISC-V」听起来是一句话，但真正动手时会发现这一步跨得太大了：从源代码到汇编，中间要处理词法、语法、语义、中间表示、寄存器、指令选择……一口气全做完，几乎没有人能在第一次就写对。

文档作者采取的策略是**增量构建（incremental）**：先让编译器只能处理最简单的程序（一个 `main` 函数里 `return 0`），再逐阶段给它「喂」更复杂的语言特性，每次只新增一小块能力。这种「由易到难、逐步增量」的拆分，正是 Lv0–Lv9 这十个阶段的来源。

#### 4.1.2 核心流程

整个实践被设计成两层拆分：

1. **先按流水线拆成两大步**：

   \[
   \text{SysY} \;\xrightarrow{\text{第 1 步}}\; \text{Koopa IR} \;\xrightarrow{\text{第 2 步}}\; \text{RISC-V}
   \]

   之所以要先经过 Koopa IR 而不是直接 SysY → RISC-V，是因为「直接翻译」难度太大、难以调试；引入一个中间层后，前后端解耦，可以分别测试（这一点在 4.3 节细讲）。

2. **再把两大步进一步细分成十个阶段（Lv0–Lv9）**，每一阶段只引入一种新的语言特性。各阶段的难度大致递增，但并不是严格的线性依赖——例如 Lv6（`if`）和 Lv7（`while`）都属于控制流，可以相对独立地理解。

#### 4.1.5 小练习与答案

**练习 1**：lab.md 说「Lv0–Lv2」搭建一个「只能处理 `main` 函数和 `return` 的简易编译器」，那么 Lv0、Lv1、Lv2 各自的侧重点是什么？

> **参考答案**：Lv0 侧重**环境配置**（Docker、Koopa IR 简介、RISC-V 简介、选编程语言，见 toc.md Lv0 下属四节）；Lv1 侧重**前端 + 中端**（把 `main`/`return` 解析成 AST 并生成 Koopa IR）；Lv2 侧重**后端**（把 Koopa IR 生成成 RISC-V 汇编）。三者合起来才是一个能端到端跑通的「最小编译器」。

**练习 2**：为什么不直接让编译器一次支持全部 SysY 语法？

> **参考答案**：因为「SysY → RISC-V」跨度太大，一次性实现既难写对、也难调试。增量拆分让每一阶段只新增一小块能力，可以针对每一阶段单独写测试、单独排错，大幅降低心智负担。

### 4.2 Lab ↔ 编译流水线映射：每个 Lv 给编译器加了什么

#### 4.2.1 概念说明

光知道「有十个阶段」还不够，关键要建立「**阶段 ↔ 能力 ↔ 流水线位置**」的三元映射。这一节就是本讲的核心实践内容：把 Lv1–Lv9 每一阶段新增的语言特性，对应到它给编译器增加的能力，并标注它主要落在流水线的哪一段。

判断「落在哪一段」的经验法则：

- 凡是「**解析新的 SysY 语法、生成对应的 Koopa IR**」的工作，主要落在**前端 + 中端**（SysY → Koopa IR 这半边）。
- 凡是「**把 Koopa IR 翻译成 RISC-V**」的工作，落在**后端**（Koopa IR → RISC-V 这半边）。
- 课程的后端一旦在 Lv2 搭好，**后续阶段大多只需扩展前端 + 中端**；只有当出现全新类型的 IR 值（如变量需要栈槽、数组需要地址计算、函数需要调用约定）时，后端才需要配合改动。

#### 4.2.2 核心流程

下面这张表是本节的「答案」，但它不是凭空给的——4.2.4 的实践会要求你自己重建一张类似的表。先看结论：

| 阶段 | 新增能力（来自各 Lv 的 README） | 主要落在 |
|------|--------------------------------|---------|
| Lv1 | `main` 函数 + `return`；首次跑通 SysY → Koopa IR（词法/语法分析 + IR 生成） | 前端 + 中端 |
| Lv2 | 首次跑通 Koopa IR → RISC-V（目标代码生成） | 后端 |
| Lv3 | 一元/算术/比较/逻辑表达式 | 前端 + 中端（后端复用 `add`/`sub` 等指令） |
| Lv4 | 常量/变量定义、赋值语句（引入变量，需要在栈上分配空间） | 前端 + 中端，后端需配合变量寻址 |
| Lv5 | 语句块与嵌套作用域 | 前端 + 中端 |
| Lv6 | `if/else` + 短路求值（分支控制流） | 前端 + 中端，后端：分支跳转 |
| Lv7 | `while` + `break`/`continue`（循环控制流） | 前端 + 中端，后端：跳转 |
| Lv8 | 函数定义/调用、SysY 库函数、全局变量 | 前端 + 中端 + 后端（函数调用约定） |
| Lv9 | 数组（一维/多维、数组参数） | 前端 + 中端 + 后端（地址计算） |

> 注意：上表里「前端 + 中端」总是成对出现，是因为新增一种 SysY 语法，几乎一定要同时改解析器（前端）和 IR 生成器（中端）。后端是否要动，取决于这种新特性是否带来后端没见过的新东西。

各阶段新增能力的权威描述，就在每个 `lv*/README.md` 的开头第一句。例如：

> **参考答案**：Lv1 对应前半边（SysY → Koopa IR），Lv2 对应后半边（Koopa IR → RISC-V）。lv2-code-gen/README.md 里明确写着「实现一个能处理 `main` 函数和 `return` 语句的编译器，同时输出编译后的 RISC-V 汇编」，并直接复用 Lv1 的文法规范。

**练习 2**：Lv9（数组）为什么被认为「前端 + 中端 + 后端」都要改，而不只是前端 + 中端？

### 4.3 前端 / 中端 / 后端：编译器的三段式结构

#### 4.3.1 概念说明

前面反复提到「前端 / 中端 / 后端」，这一节给出它们的准确定义。这些定义来自 `docs/lv1-main/structure.md`（Lv1.1「编译器的结构」），是理解整条流水线的概念骨架。

需要先区分两个层面的「编译」：

- **广义的编译**（如 `gcc hello.c -o hello`）：编译 → 汇编 → 链接，三步。
- **课程里要实现的编译**：只做第一步——把 SysY 源代码编译到 RISC-V 汇编。汇编和链接由现成工具完成。

课程要实现的这个「编译器」，内部又分成三段：

- **前端（frontend）**：词法分析 + 语法分析，把源代码变成 AST（抽象语法树）；再语义分析，检查语义错误。
- **中端（middle-end）**：把 AST 转换成 IR（中间表示），并在 IR 上做一些机器无关优化。
- **后端（backend）**：把 IR 转换成目标平台的汇编，并做一些机器相关优化。

#### 4.3.2 核心流程

把三段串起来，就是完整的编译流水线：

\[
\underbrace{\text{SysY 源码}}_{\text{文本}}
\;\xrightarrow{\text{前端}}\;
\underbrace{\text{AST}}_{\text{树}}
\;\xrightarrow{\text{中端}}\;
\underbrace{\text{Koopa IR}}_{\text{中间表示}}
\;\xrightarrow{\text{后端}}\;
\underbrace{\text{RISC-V 汇编}}_{\text{目标代码}}
\]

为什么要特意插入一个 IR，而不是直接 AST → 汇编？因为 IR 能把「源语言」和「目标机器」解耦。假设要支持 \(M\) 种源语言、\(N\) 种目标指令集：

- 不用 IR：需要 \(M \times N\) 个翻译模块。
- 用统一 IR：只需 \(M + N\) 个模块（\(M\) 个「语言 → IR」+ \(N\) 个「IR → 机器」）。

当 \(M\) 和 \(N\) 都大于 1 时，\(M + N\) 远小于 \(M \times N\)，这就是 LLVM、也是 Koopa IR 采用 IR 中间层的根本动机。

三段式的原始定义就在 structure.md 的开头：

>
> [docs/lv1-main/structure.md:19-23](https://github.com/pku-minic/online-doc/blob/d172f8994369fba0f826ac3f1c2f4d6bf63c9ea5/docs/lv1-main/structure.md#L19-L23)

#### 4.3.5 小练习与答案

**练习 1**：structure.md 说课程编译器「只涉及编译，不涉及汇编和链接」。那么学生写的程序最终怎么变成可执行文件？

> **参考答案**：学生的编译器只负责把 SysY 编译成 RISC-V 汇编（`.s` 文件）；之后的汇编（ assembler 把 `.s` 变成目标文件）和链接（linker 变成可执行文件）由现成工具链（如 RISC-V 的 `as`/`ld`，或在课程提供的 Docker 环境里）完成，不需要学生实现。

**练习 2**：用一句话解释「为什么有了 AST 还要再设计一个 IR」。

> **参考答案**：AST 形式上太接近源语言、包含分支/循环/结构体等高级语义，一步到位变成汇编很复杂；IR 抽象层次更低、更接近机器但又没低到汇编，既方便做优化，又能把「源语言」和「目标机器」解耦（M+N 优于 M×N）。

### 4.4 可选 Lv9+：加分项与进阶方向

#### 4.4.1 概念说明

Lv0–Lv9 是**必做**内容，全部完成后编译器就能处理所有合法的 SysY 程序。在此之上，文档还准备了一章 **Lv9+（lv9p-reincarnation，「新的开始」）**，这是**可选**内容：不做不扣分，做了能加分。

Lv9+ 与前九章有两个显著不同：

1. **不再是手把手指导**：前九章是 step-by-step 教学，Lv9+ 只给简要指导或参考文献，需要学生自行探索。
2. **小节之间无顺序依赖**：Lv9+ 内部各小节可任意顺序阅读，难度大致按出现顺序递增。

#### 4.4.2 核心流程

Lv9+ 提供六个进阶方向，可以分成两类：

- **扩展能力**：Lv9+.1「你的编译器超强的」——尝试编译更复杂的程序，甚至扩展更多语法。
- **提升质量 / 性能**：
  - Lv9+.2 寄存器分配（实现更复杂的分配算法，改善目标代码质量）
  - Lv9+.3 优化（在 Koopa IR 层面做各种优化）
  - Lv9+.4 SSA 形式（把 IR 提升到 SSA，简化分析与优化）
  - Lv9+.5 性能测试（量化寄存器分配和优化带来的效果）
- **更远方向**：Lv9+.6「向着更远处进发」——编译领域更多课题的引路。

注意 Lv9+ 的目录名是 `lv9p-reincarnation`，其中 `p` = plus，对应「Lv9+」这个名字（详见 u1-l3 对目录命名规律的说明）。

#### 4.4.5 小练习与答案

**练习 1**：Lv9+ 的目录名为什么是 `lv9p-reincarnation` 而不是 `lv10-xxx`？

> **参考答案**：因为它属于「Lv9+」（plus），不是第 10 个必做阶段。命名上用 `p` 表示 plus，与 `lv9p-reincarnation` 这个目录名一致（见 u1-l3 的目录命名规律）。而 `reincarnation`（重生/新的开始）对应 README 的标题「新的开始」——完成 Lv9 后编译器已成型，Lv9+ 是在此基础上「重新出发」做进阶。

**练习 2**：一个只想拿基础满分的同学，需要做 Lv9+ 吗？

> **参考答案**：不需要。Lv9+ 明确标注「不做不扣分」，只有想加分、或对编译器进阶（优化、寄存器分配等）感兴趣的同学才需要做。但 README 末尾建议即便不选做，也推荐阅读，尤其是 Lv9+.1。

## 6. 本讲小结

- 文档把「写一个编译器」先按流水线拆成两大步（SysY → Koopa IR → RISC-V），再细分成 **Lv0–Lv9 十个增量阶段**，每一阶段只新增一小块语言能力，降低实现与调试难度。
- 每个阶段都对应一个 `lv*-` 内容目录（见 toc.md），目录下属的小节就是该阶段要学的具体步骤；`lv9p-` 表示可选的 Lv9+。
- 编译器内部是**前端 / 中端 / 后端**三段式：前端做词法/语法分析得到 AST，中端把 AST 转成 Koopa IR，后端把 IR 生成 RISC-V 汇编。
- **Lv1 首次建立前端 + 中端，Lv2 首次建立后端**；后续 Lv3–Lv9 大多主要扩展前端 + 中端，只有引入新存储/寻址/调用机制的阶段（Lv4 变量、Lv8 函数、Lv9 数组）才明显需要后端配合。
- 引入 Koopa IR 这一层是为了把「源语言」和「目标机器」解耦，把 \(M \times N\) 个翻译模块降到 \(M + N\) 个。
- **Lv9+ 是可选加分内容**，方向集中在优化、SSA、寄存器分配、性能测试等中后端进阶主题，不再是手把手指导。
