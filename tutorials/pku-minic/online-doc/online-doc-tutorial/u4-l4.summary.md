本讲是链接检查器四连讲第三篇，承接链接提取，专攻远程链接（http/https/`//` 开头）的校验；代码集中于 `check_links.py` 的远程校验函数组，参数取自 `check-links.toml` 的 `[http]` 段，网络能力由 `httpx` 提供。

它含五个最小模块：异步并发与限流、HEAD/GET 回退、HTML 判定、远程锚点验证、状态码接受规则。`check_remote_links` 用 `asyncio.gather` 并发派出所有去重 URL，以 `asyncio.Semaphore` 与 `httpx.Limits` 双重限流，`workers` 控并发度（本仓库 16）。单条链接走「先 HEAD 后 GET」回退：HEAD 不读正文最省；需验锚点或 HEAD 状态码不被接受时升级 GET，HEAD 异常则整体退回 GET，不读正文时 GET 带 `Range: bytes=0-0` 只取 1 字节。

引入的关键术语：Semaphore、httpx.Limits、workers、urldefrag、fragment（`#` 后缀，区别于本地侧 `?id=`）、`is_html_content`、`parse_remote_anchors`（复用本地 `parse_html_anchors` 与 `expand_anchor_forms` 做多形式集合相交匹配）、`evaluate_remote_response`、`RemoteResult`、`status_is_accepted`、`accepted_statuses`（401/403/429）、`accepted_status_ranges`（[200,399]）。

核心反直觉结论：返回 403 的链接被判通过——状态码被 `accepted_statuses` 放行，且即便带 fragment 也走 `evaluate_remote_response` 第 3 分支标「fragment not verified」不再比对锚点，因拒绝页正文多为错误页；仅 HTML 响应才验锚点，非 HTML 与 `#:~:text=` 文本片段直接跳过，锚点完全对不上才报 `fragment not found`。
