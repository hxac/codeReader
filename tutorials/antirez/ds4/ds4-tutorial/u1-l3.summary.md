本讲为约 11 万行的 ds4（DwarfStar）C 代码库绘制一张「源码地图」，只读文件头注释、目录列表与构建脚本，不展开算法。核心结论是项目的三层结构：根目录平铺的 `.c`/`.m`/`.cu`/`.h` 分为三类——**引擎核心**（`ds4.c` 等约 2.7 万行）、**GPU 后端**（`ds4_metal.m`/`ds4_cuda.cu`/`ds4_rocm.cu`，共用 `ds4_gpu.h` 接口、采用张量常驻设备的命令模型）、**前端二进制**（`ds4`、`ds4-server`、`ds4-bench`、`ds4-eval`、`ds4-agent` 五个）。关键拼装关系：每个二进制 = 「自己的前端 `.o` + 公共辅助 `.o` + 共享的 `CORE_OBJS`」，而 `CORE_OBJS` = `ds4.o` + `ds4_distributed.o` + `ds4_ssd.o` + 一个可替换的后端 `.o`；CPU 构建特殊，用 `-DDS4_NO_GPU` 把 `ds4.c` 编成 `ds4_cpu.o` 且不含后端对象。引入的关键术语：`CORE_OBJS`（引擎核心对象集）、前端二进制、GPU 后端对象、`ds4_engine`（已加载模型）/`ds4_session`（可变推理时间线）这对公共窄边界、rax（借自 Redis 的基数树）、mmap 权重加载。子目录按后缀归拢：`metal/`（运行时编译的 GPU 内核）与 `rocm/`（编译期 include 的头）参与运行期，`gguf-tools/` 是离线模型构建工具。辅助层中 `README.md`/`AGENT.md` 为向导，`Makefile` 描述构建（Linux 上裸 `make` 只打印帮助），`AGENT.md` 的 `Layout` 节是目录速览入口但声明不完整。
