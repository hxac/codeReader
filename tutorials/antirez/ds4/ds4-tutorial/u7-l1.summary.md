本讲讲清 `ds4-server` 的架构骨架与线程模型。核心结论是：服务器是「假并发」——**请求解析并发、推理串行**。进程内有三种线程：主线程负责初始化、监听与派生连接；每条 HTTP 连接一条客户端线程（`client_main`）负责读 HTTP、解析 JSON 并把作业塞入共享 FIFO 队列；全局唯一 graph worker（`worker_main`）串行取出作业、跑 `generate_job`（prefill+decode+写回）、置 `done` 唤醒对应 client。两者用互斥锁+条件变量衔接，且 `session` 无锁——因为它只在 worker 线程被读写，「单 worker」与「单 session」互为因果。第二条核心约束是**单活 KV session**：进程内只持有一个 engine 和一个可变 `ds4_session`，靠 `ds4_session_common_prefix` 与多级回退链（协议活状态→活 token 前缀→磁盘快照→冷 prefill）实现前缀复用，让无状态客户端重发整段对话时只增量 prefill 后缀。串行推理带来**队头阻塞**，长 prompt 会拖慢后续请求首 token 延迟。最后 `--chdir` 在 `ds4_engine_open` 前执行进程级 chdir，使 `metal/*.metal`、默认模型与 `--kv-disk-dir` 等相对路径从项目树解析。引入的关键术语：client 线程、graph worker、FIFO 作业队列、栈上 job、单活 KV checkpoint、多级回退链、前缀复用、队头阻塞。
