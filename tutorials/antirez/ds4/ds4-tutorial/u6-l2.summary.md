本讲讲解 ds4 中的「多 token 预测（MTP）投机解码」路径，仅在贪婪（argmax）解码、单机 GPU、非分布式、非 SSD 流式条件下生效。它把 DeepSeek V4 训练时内置的 `mtp.0` 单层 transformer 块加载为独立的轻量 GGUF，充当**草案模型（drafter/draft）**，把 43 层主模型当作**验证模型（verifier/target）**。核心立论是「MTP 是草案器而非采样器，主模型永远是输出流的唯一权威」。讲义覆盖三个最小模块：(1) 草案来源——主模型每次 decode 真实 token 时搭便车算出首个草案 `mtp_draft_token`，使 `drafts[0]` 近乎免费且可用主模型刚算出的 logit 免费验证；(2) 接受/拒绝逻辑——`ds4_session_eval_speculative_argmax` 四步状态机先提交首 token、免费验证 `drafts[0]`、递归起草后缀、再做批量前向验证，贪婪规则下草案被接受当且仅当主模型该位置 argmax 相等，连续匹配构成接受前缀，**部分接受（partial accept）**时必须用 `spec_frontier_snapshot/restore` 回滚主模型压缩 KV 前沿以防污染；(3) **置信度门控 `--mtp-margin`**——用草案 logit 的 top0−top1 边际预测是否「白跑」，过低则跳过 `drafts[1]` 退回普通 decode，这是速度旋钮而非正确性旋钮，严格模式（`--quality`/`DS4_MTP_STRICT`）下禁用并改走精确验证器以复现官方向量。最终结论：MTP 标注为实验性，提速依赖不可控的接受率、至多轻微提升，且 draft 状态不属权威 KV、不随磁盘快照持久化，加载后由下次 decode 重建。
