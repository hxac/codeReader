本讲拆解 `ds4-server`/`ds4-agent` 写入磁盘的 **KVC 缓存文件**（`.kv` 文件）的字节布局，只讲格式本身，不展开 payload 内部序列化（u8-l3）和保存淘汰策略（u8-l2）。

一个 `.kv` 文件由五段顺序拼接：**48 字节固定头 + 4 字节 text_bytes(u32 小端) + 渲染文本 + DS4 payload + 可选 KTM trailer**，总大小为 `48+4+text+payload+trailer`，全部小端。

**固定头**（由 `fill_header`/`read_header` 镜像读写）自描述：magic `"KVC"` + version + payload ABI(=2) 三道校验，字段含量化档(quant_bits)、reason 保存原因码、ext_flags 扩展位、model_id、tokens、hits、ctx_size、created_at、last_used、payload_bytes。

**渲染文本**是核心查找身份：它的 SHA1 当文件名（`<40hex>.kv`），本身存入文件。文件名只编码查找键，权威的 token 序列与图状态由 **DS4 payload** 承载。加载时做 sha、前缀、model_id 三重校验，命中后以 payload 为准、仅对新文本后缀重新分词。

**KTM(tool-id map)** 是可选尾部，由 `ext_flags` bit0 标记、经 trailer hooks 由服务器侧写入/读出，kvstore 不解析其内部；先恢复 payload 再读 KTM，丢失它只损失前缀复用、不影响正确性。
