本讲解析 ds4 构建系统如何让同一份源码跑在 Metal/CUDA/ROCm/CPU 四条路径上。核心结论：Makefile 用 `ifeq ($(UNAME_S),Darwin)` 把构建拆成 macOS 与 Linux 两套独立规则，二者共享变量 `CORE_OBJS = ds4.o + ds4_distributed.o + ds4_ssd.o + 一个后端 .o`，其中第四个对象在 `ds4_metal.o`/`ds4_cuda.o`/`ds4_rocm.o` 之间切换；CPU 构建则没有后端对象，且首对象变为 `ds4_cpu.o`。Linux 上裸 `make` 仅打印帮助（因 CUDA 必须显式指定目标架构），真正的后端目标是 `cuda-spark`/`cuda-generic`/`cuda CUDA_ARCH=`/`strix-halo`/`cpu`，其中 `strix-halo` 通过命令行覆盖 `DS4_LINK`/`DS4_LINK_LIBS` 等变量把链接器换成 hipcc。引入的关键术语：后端枚举 `DS4_BACKEND_METAL/CUDA/CPU`（ROCm 复用 CUDA 这一位）、用 nvcc/hipcc 做最终链接的 `DS4_LINK` 链接器抽象、两个编译期开关 `-DDS4_NO_GPU`（令引擎核心自身退化为 CPU 后端，源文件重编成 `*_cpu.o` 并用 `#ifdef` 切除 GPU 代码）与 `-DDS4_ROCM_BUILD`（激活 ROCm 专属优化）。`default_backend()` 与 `parse_backend()` 把编译期宏与运行期默认后端串联起来，集中体现了「后端可替换、引擎核心稳定」的分层架构。
