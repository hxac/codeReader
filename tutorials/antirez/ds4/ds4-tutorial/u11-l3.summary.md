本讲讲清 ds4 如何用测试向量做推理正确性回归，覆盖三个模块：测试向量、C runner（ds4_test）、严格比对。两套互补向量是核心概念：官方向量（official.vec）对齐官方实现，但官方 API 不暴露完整 logits，greedy 下选中 token 的 logprob 被掩码为 0.0、其余 top-20 候选一律标成 -9999.0 哨兵值，fetcher 过滤后每步只剩 1 个 token，故只能验证贪婪 token 序列与一个 -4.0 弱概率下限；本地 golden 向量（local-golden.vec）从一次正常 Metal 运行用只读旁路 API（ds4_session_top_logprobs / argmax / copy_logits）抓取 64 个真实 float logit，用 top1 精确相等加 top-k 重叠数加 max_abs≤8.0 的宽松容差，能发现「贪婪 token 没变但分布已坏」的实质性后端漂移。ds4_test 通过 #include 真实 ds4_server.c 拉进产品代码、懒加载复用 fast/quality 引擎，--server 是唯一不依赖 GPU 的测试组。严格比对的含义是：钉死 DS4_METAL_PREFILL_CHUNK（官方 2048 / 本地 golden 4096）并关掉 metal4 与流式快速路径，因为 chunk 边界同时是 KV checkpoint 推进点与 logit 写出点；cache-pressure 测试则反向打开 SSD 流式加小缓存聚焦复现 issue #384 的确定性错误 logits。
