本讲围绕 nginx 全局上下文容器 `ngx_cycle_t` 及其构造函数 `ngx_init_cycle` 展开。`ngx_cycle_t` 将进程级共享状态（配置数组 `conf_ctx`、模块表 `modules`、监听端口 `listening`、共享内存、连接运行时）收拢进一个挂在专属内存池上的结构体，全局指针 `ngx_cycle` 指向当前 cycle；reload 时构造全新 cycle，成功才切换、失败则丢弃，保证不影响线上。

`ngx_init_cycle` 是一条七阶段线性装配线：建池与容器 → `ngx_cycle_modules` 装载模块 → 核心模块 `create_conf` 建空壳填哨兵 → `ngx_conf_parse` 解析配置 → 核心模块 `init_conf` 用 `ngx_conf_init_value` 把仍是哨兵的字段补默认 → 物化文件与共享内存 → `ngx_open_listening_sockets` 打开端口 → `ngx_init_modules` 调 `init_module` 提交；阶段 0-4 失败直接销毁池，阶段 5-7 失败 `goto failed` 先回滚物化资源。

需区分两套模块回调：`ngx_core_module_t` 的 `create_conf`/`init_conf` 是配置层回调（仅核心模块，在阶段 2/4）；`ngx_module_t` 的 `init_module`/`init_process` 是进程运行时层回调（对所有模块，`init_module` 在阶段 7）。核心结论有三：一是配置解析只负责填登记表，真正物化（bind 端口、分配共享内存）发生在解析之后；二是 `nginx -t` 本质是跑完整条装配线（含真正 bind 与 init_module）后返回，「syntax is ok」仅代表文本通过；三是 reload 不丢连接的根基，在于新旧 cycle 按同地址复用监听 fd（标 `remain=1`），并按「同名同 size 同 tag」复用共享内存。
