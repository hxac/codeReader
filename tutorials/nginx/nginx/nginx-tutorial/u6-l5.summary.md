本讲聚焦 nginx 请求处理 11 个 phase 中的 `FIND_CONFIG` 阶段，回答「请求选哪个 location」与「location 内未写明指令的值从哪来」两个问题。它拆成五个最小模块：① 配置期由 `ngx_http_core_location` 解析 `=`、`^~`、`~`、`~*`、`@` 五种修饰符，分别置 `exact_match`/`noregex`/`regex`/`named` 位标志；② 配置期 `ngx_http_init_locations` 将 location 三分类——静态类建二叉树存入 `static_locations`，正则进 `regex_locations` 数组，named 进 server 级 `named_locations` 数组；③ 运行期 `ngx_http_core_find_location` 按优先级调度，形成「精确 > 最长前缀 > 正则」链，`^~` 经 `noregex` 关闭正则这一关；④ `ngx_http_core_find_static_location` 用 `ngx_filename_cmp` 做 O(log n) 二叉搜索，靠 `node->tree` 子树取最长前缀、靠 `auto_redirect` 触发补斜杠 301；⑤ `merge_loc_conf` 回调沿「http→server→location→嵌套」树自顶向下递归，凭 `NGX_CONF_UNSET` 哨兵让子 location 就近继承父级或硬编码默认值。关键术语：`r->loc_conf`、`clcf`、静态前缀树、named location 内部重定向（重置 phase 游标到 `location_rewrite_index`）。
