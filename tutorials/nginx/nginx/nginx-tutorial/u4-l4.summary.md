本讲义讲解 nginx 的操作系统抽象层 `src/os/unix`，它把 Linux/FreeBSD 等多系统差异收拢，对上暴露统一接口。核心分两条文件分工：`*_config.h`（如 `ngx_linux_config.h`）负责编译期能力探测与系统头文件包含，依赖 `auto/configure` 生成的 `NGX_HAVE_*` 宏做条件编译；`*_init.c`（如 `ngx_linux_init.c`）负责运行时初始化并构造 I/O 接口表。关键概念 `ngx_os_io_t` 是一张函数指针表，含 `recv/recv_chain/send/send_chain` 等字段与 `NGX_IO_SENDFILE` 标志；`ngx_posix_init.c` 给出用 `ngx_writev_chain` 的保守默认表，Linux 表在编译期探测到 sendfile 后升级为 `ngx_linux_sendfile_chain`，运行时由 `ngx_os_specific_init` 切换。文件与 socket 差异多用宏映射（`ngx_open_file`→`open`，零开销），仅 `ngx_tcp_nopush` 等行为差异项写成函数（Linux 用 `TCP_CORK`，FreeBSD 用 `TCP_NOPUSH`）。内存方面区分三种原语：`ngx_alloc`/`ngx_calloc` 裸 malloc 包装、`ngx_memalign` 对齐分配（无能力时静默退化，服务于 `O_DIRECT` 与 false sharing）、`ngx_alloc_buf` 从池分配 buf 结构体（数据缓冲由 `ngx_create_temp_buf` 另行分配）。三条 I/O 快路径共享「合并相邻 buf 成 iovec」思想：`ngx_readv_chain` 用 `readv` scatter 读 socket，`ngx_writev_chain` 用 `writev` gather 写内存 buf（遇 `in_file` 报 ALERT），`ngx_linux_sendfile_chain` 对文件 buf 用 `sendfile` 零拷贝、对内存 header 用 `writev` 并以 TCP_CORK 打包。根本分工结论：读 socket 选 readv（数据须进用户态供 HTTP 解析），发静态文件选 sendfile（文件已在内核页缓存、可零拷贝）。
