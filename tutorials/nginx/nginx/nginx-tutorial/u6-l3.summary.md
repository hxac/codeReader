本讲打开「字节流如何变成结构化请求对象」的黑盒，围绕三个最小模块：请求行解析器 `ngx_http_parse_request_line`、头部解析器 `ngx_http_parse_header_line`、请求体读取框架 `ngx_http_read_client_request_body`。

核心结论：nginx 用逐字符、可重入的状态机解析请求行与头部，状态记在 `r->state`，跨 TCP 分段续接，解析器只挪指针、几乎零拷贝。请求行用 `usual[]` 位图分流 URI 字符、按已读长度定长比对方法名，区分 origin/absolute/authority 三种 URI 形态，版本号合成为 `http_version`（1.1→1001）。头部解析同时做增量哈希（`hash*31+小写字符`）与小写化，配合 `headers_in_hash` 实现「头部名→处理函数」的 O(1) 分发，空行返回 `NGX_HTTP_PARSE_HEADER_DONE`。

关键架构区分：解析器被动看字符；`ngx_http_request.c` 中的驱动者主动读 socket、调解析器、按需扩展 large header buffer，超限返回 414/431。请求体与头部是两套机制——头部同步读全，请求体惰性、按需读取，经 `request_body_filter` 分流到 `length_filter`（Content-Length）或 `chunked_filter`（内部再调 `ngx_http_parse_chunked`），汇入 `save_filter`，遵循「内存缓冲优先、超 `client_body_buffer_size` 溢写临时文件」。读取全程异步：未读完挂超时定时器、返回 `NGX_AGAIN` 并以 `r->main->count++` 守护请求不被释放，读完后用 `post_handler` 回调业务模块；chunked 流收齐后 `content_length_n` 即真实长度，对业务模块等同于带 Content-Length。
