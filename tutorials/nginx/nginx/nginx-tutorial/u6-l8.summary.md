本讲以 `ngx_http_static_module` 为样本，讲清静态文件如何作为 HTTP 请求处理的终点工作。核心模块有三：静态处理、open_file_cache、特殊响应生成。

CONTENT 阶段有「专属处理者」机制：若 location 配了 `clcf->handler`（如 `proxy_pass`），则 `r->content_handler` 非空，CONTENT 阶段只跑它；否则退化为「轮流尝试」static/index 等候选模块。static 通过 `postconfiguration` 把 handler 注册进 CONTENT phase 数组。

`ngx_http_static_handler` 是一条线性同步主链：方法白名单（仅 GET/HEAD/POST）→ `ngx_http_map_uri_to_path` 把 URI 映射为磁盘路径（区分 root/alias）→ `ngx_open_cached_file` 带缓存打开文件，失败按 errno 映射成 404/403/500 → 目录则构造 `Location` 头返回 301 → 非普通文件 404、POST 则 405 → 组装 `in_file=1` 的 buf（只记 fd 和区间，不读入内存）交给 `ngx_http_output_filter`，最终由 `sendfile()` 零拷贝送出。

`open_file_cache` 用「红黑树（按 crc32 查找，碰撞时 strcmp 兜底）+ LRU 队列」缓存 open+stat 结果，连失败结果（404）也缓存，命中即省掉 `open()`。

static handler 只返回错误码，HTML 错误页由 `ngx_http_special_response_handler` 生成：先处理 `error_page` 内部重定向（靠 `r->error_page` 与 `r->uri_changes` 防递归），否则用 `ngx_http_error_pages[]` 内置页拼装，尾部受 `server_tokens` 控制版本号显示。关键术语：content handler、`r->content_handler`、in_file buf、open_file_cache、error_page 内部重定向、特殊响应。
