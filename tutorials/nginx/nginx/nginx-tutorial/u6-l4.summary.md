本讲拆解 nginx 的 phase 机制：把一次 HTTP 请求硬编码成 11 个有序阶段（`ngx_http_phases` 枚举：POST_READ、SERVER_REWRITE、FIND_CONFIG、REWRITE、POST_REWRITE、PREACCESS、ACCESS、POST_ACCESS、PRECONTENT、CONTENT、LOG），阶段数量与顺序写死、不可配置，用户只能在业务阶段挂 handler。配置期两层结构：`cmcf->phases[]` 是「每阶段一个 handler 数组」的二维容器，解析结束后由 `ngx_http_init_phase_handlers` 拍平成一维引擎 `cmcf->phase_engine.handlers`，每项为 `(checker, handler, next)` 三元组。运行期主循环 `ngx_http_core_run_phases` 靠游标 `r->phase_handler` 逐项调 checker，主循环只认 checker 返回值（OK 交出请求、非 OK 继续）；checker 解释 handler 返回值：DECLINED→下一项、OK→跳 next 进下一阶段、AGAIN/DONE→异步挂起、ERROR→finalize。普通请求从 0 起，内部重定向从 `server_rewrite_index` 重跑。FIND_CONFIG/POST_REWRITE/POST_ACCESS 为框架阶段，仅做选 location、防循环、汇总鉴权。CONTENT 阶段特殊：`r->content_handler`（来自 `clcf->handler`，如 proxy_pass）非空时独占响应生成，否则 static/index 等候选轮流尝试。LOG 不走主循环，由 `ngx_http_log_request` 在请求结束时单独触发。
