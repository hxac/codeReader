本讲承接 u3-l1 的指令解析链，回答两个问题：单参数指令的 set 回调如何复用，以及带地址的指令文本如何转为内核可用的地址结构。核心分三块：第一，nginx 提供一组通用 slot 函数（如 `ngx_conf_set_flag_slot`、`set_num_slot`、`set_msec_slot`），借助 `offsetof` 把「字段位置」在编译期固化进指令描述符的 `offset` 字段，运行时以 `p + cmd->offset` 反射式定位，使一套代码服务成百上千条「写一个字段」的指令；分发器 `ngx_conf_handler` 先用 `cmd->conf` 选出本模块某层配置结构体，再由 slot 用 `offset` 选字段，两者同为「基址+编译期偏移」。第二，每种 slot 对应特定 C 类型、解析器与「未设置」哨兵（`NGX_CONF_UNSET*`），`create_*_conf` 须将字段初始化为对应哨兵，才能支撑重复判定与后续 merge 继承；msec/sec 仅差 `ngx_parse_time` 的 `is_sec` 参数，enum/bitmask 则把 `cmd->post` 复用为「名字→值」表。第三，`ngx_inet.c` 的解析链：`ngx_inet_addr` 用状态机把 IPv4 文本转为网络字节序整数；`ngx_ptocidr` 拼掩码并规范化地址（带主机位返回 `NGX_DONE`）；`ngx_cidr_match` 用 `(addr & mask) == net` 判定命中；`ngx_parse_addr` 产出含 `sockaddr` 的 `ngx_addr_t`（不含端口），是 `ngx_parse_url` 的子步骤，带端口与主机名的完整解析留给后续 upstream 讲义。
