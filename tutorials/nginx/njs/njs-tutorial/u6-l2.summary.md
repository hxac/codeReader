本讲阐明 njs 的「双实现」铁律。根因是内置引擎与 QuickJS 的值类型（njs_value_t vs JSValue）和模块体系（外部对象机制 vs 标准 ES Module）根本不兼容，故 external/ 下每个扩展功能（fs/crypto/xml 等）都成对提供 njs_xxx_module.c 与 qjs_xxx_module.c。引入两种注册结构：njs_module_t{name,preinit,init}（init 返回状态码、两阶段被 njs_vm_create 调用）与 qjs_module_t{name,init}（init 返回 JSModuleDef*、单阶段被 qjs_new_context 调用）；声明表则分用 njs_external_t 与 JSCFunctionListEntry，均靠 magic 复用同一 C 函数。构建链路为 auto/modules、auto/qjs_modules 两条独立流水线，经 shell 变量汇总成 njs_modules[]/qjs_modules[] 全局数组，分别编入 libnjs.a/libqjs.a 后共同链进 build/njs。核心结论：修改一处行为须先判明同步集——业务逻辑尽量复用底层 helper（一处改、两引擎受益），否则 njs_/qjs_ 必须双改；仅新增或删除方法时才需同步两张声明表与两份构建清单。运行期 JS 可经 njs.engine 做能力分支。
