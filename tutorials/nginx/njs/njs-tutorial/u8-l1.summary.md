本讲是 NGINX 集成单元首讲，讲清 `nginx/ngx_js.c` 这个**共享绑定层**如何被 HTTP 与 Stream 两个模块复用：靠构建胶水 `nginx/config` 把同一份 `NJS_SRCS` 编进两模块（静态链接去重、动态链接则各编一份，QuickJS 类 id 用静态枚举避符号冲突）。核心引擎抽象 **`ngx_engine_t`** 用 `union` 持底层句柄（`njs_vm_t`/`JSContext`）、用七个函数指针组成手工 vtable，由 `ngx_create_engine(opts)` 工厂按引擎类型分流，上层只经指针调用、对引擎透明。标准时序为「配置期 `ngx_js_init_conf_vm` 建一个模板 `ngx_engine_t` → 请求期 `engine->clone` 出独立副本 → 请求结束 `engine->destroy` 销毁」。两引擎克隆分工不同：`ngx_njs_clone` 一行 `njs_vm_clone` 复用模板；`ngx_qjs_clone` 无 clone，只能新建 runtime/上下文并重放 `precompiled` 字节码，还支持 `reuse_queue` 复用上下文。「绑 `r`/`s`」属模块专属，故真正克隆回调是模块里 4 参数包装 `ngx_engine_njs_clone`/`ngx_engine_qjs_clone`，先调共享层 3 参数克隆、再绑对象。`js_import`/`js_engine`/`js_path` 解析回调定义在共享层、被两模块共用，且**只在解析期收集**名/路径，真正拼引导脚本、编译进模板延迟到合并结束的 `ngx_js_init_conf_vm`。关键术语：共享绑定层、`ngx_engine_t`、vtable、模板 VM、`ngx_engine_opts_t`、`ngx_js_ctx_t`、`NGX_JS_COMMON_LOC_CONF`、precompiled、reuse_queue。
