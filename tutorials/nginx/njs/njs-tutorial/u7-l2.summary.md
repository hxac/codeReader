本讲剖析 njs 的加密能力，揭示两套名字相同但完全不同的加密 API：Node 风格的可导入 `crypto` 模块（`createHash`/`createHmac`）与 Web 风格的全局 `crypto` 对象（`crypto.subtle`，即 WebCrypto）。前者照搬 Node 流式 API，把 OpenSSL 的 `EVP_MD_CTX`/`HMAC_CTX` 包装成带 `update`/`digest`/`copy` 的对象，靠 `magic8`（njs 引擎）或 `magic`（QuickJS）让一个 C 函数同时服务 Hash 与 Hmac；后者遵循 W3C WebCrypto 规范，操作挂在 `SubtleCrypto` 上并统一返回 `Promise`，密钥封装为不可变的 `CryptoKey`（内部 `njs_webcrypto_key_t`，持 `EVP_PKEY*` 或原始字节），靠一张算法注册表 `njs_webcrypto_alg[]` 管理支持的 usage 与格式。其异步语义由 `njs_webcrypto_result` + `njs_promise_trampoline` 经 jobs 队列投递实现。关键结论有二：OpenSSL 是可选依赖，`--no-openssl` 或探测失败时两模块整体不编入，`AES-KW`/`Ed25519` 等还受构建期特性宏守卫，而 `njs_openssl.h` 抹平多版本 API 差异；两套 API 在 njs 与 QuickJS 引擎下各有成对实现，回收机制一用池 cleanup 一用 finalizer。
