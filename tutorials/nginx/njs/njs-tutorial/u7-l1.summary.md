本讲以 `fs`（文件系统）扩展模块为案例，讲解 njs「一个功能、两份实现」的双引擎铁律：内置引擎用 `external/njs_fs_module.c`（外部原型 + `njs_module_t`），QuickJS 用 `external/qjs_fs_module.c`（注册类 + `qjs_module_t`），对外 API 形状一致、内部机制不同，靠两张「声明表」（`njs_ext_fs[]`/`qjs_fs_export[]`）定义。

核心结论有三：一、同一组 `readFile`/`stat`/`mkdir` 等方法各含同步、回调、Promise 三套写法，且常共用同一个 C 函数；二、用 `magic8` 位编码（`magic = (mode<<2)|calltype`）区分调用约定（DIRECT/PROMISE/CALLBACK）与操作变体（STAT/LSTAT/FSTAT、TRUNC/APPEND），再由 `njs_fs_result` 三路分发；三、Promise/回调的「异步」仅是把已算好的结果投递进 `njs_vm_enqueue_job`（承接 u4-l5 的 jobs 队列），真实文件 IO 同步完成。

关键术语：`Stats`/`Dirent`/`FileHandle` 三类宿主对象（内置引擎靠属性处理器与 `magic32` 复用，低 4 位选字段、高 4 位选 number/Date；QuickJS 靠类 id 与 exotic methods）、`proto_id` 外部原型句柄、`fs.promises` 子对象。另指出 sandbox 安全开关在两引擎下实现不同：内置引擎 `njs_fs_init` 检测到 sandbox 即整体不注册 fs，QuickJS 在 CLI 选项期直接拒绝 `-s`，故 sandbox 不可作为跨引擎可移植保证。
