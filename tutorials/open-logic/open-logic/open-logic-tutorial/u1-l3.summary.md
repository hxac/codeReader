本讲围绕 Open Logic 的「获取—编译—集成到厂商工具」工程链路，拆为四个最小模块。其一，**子模块获取**：`fix`（定点）区域依赖第三方库 `en_cl_fix`（MIT），通过 git submodule 引入于 `3rdParty/en_cl_fix/`，故须用 `git clone --recurse-submodules` 或下载 `CompleteSources.zip`，否则该区域缺文件。其二，**单库编译策略**：官方建议把所需各区域连同依赖编译进同一个 VHDL 库，库名约定为 `olo`，依赖顺序由根目录的 `compile_order.txt`（93 个文件）规定——package 先于使用它的 entity，`en_cl_fix` 先于 `olo_fix`。其三，**厂商导入脚本**：每个厂商在 `tools/<厂商>/` 下有脚本（Vivado/Quartus 为 TCL，Efinity/Yosys 为 Python），统一完成「由脚本路径反推仓库根→加入四区域及 en_cl_fix 源码→登记到库 `olo`→设为 VHDL-2008」；其中仅支持 scoped constraints 的 Vivado 会自动导入 `base`/`intf` 的时序约束，Quartus/Gowin/Libero/Efinity 需手动加约束。其四，**跨语言实例化**：Open Logic 虽为 VHDL-2008，但官方支持从 Verilog/SystemVerilog 实例化；要点是不可依赖 VHDL 端口默认值，所有输入端口须在 Verilog 中显式赋值（Vivado 会把未连端口当作 0），且不同工具对 Verilog 场景库名要求不同——Libero 用 `work`、Efinity 用 `default`。关键术语：库名 `olo`、`en_cl_fix` 子模块、`compile_order.txt`、导入（import）、scoped constraints、跨语言实例化。
