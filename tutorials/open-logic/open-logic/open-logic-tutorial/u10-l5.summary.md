本讲收束第 10 单元，把零散的工程化能力串成「写代码 → 进工具 → 过 CI → 发版本」流水线。核心四块：厂商导入脚本（`tools/<厂商>/import_sources.*`）统一完成加源、进库 `olo`、设 VHDL-2008、加约束，其中**仅 AMD Vivado 凭 scoped constraints（`read_xdc -ref` 绑定实体、`-unmanaged` + `PROCESSING_ORDER LATE`）自动套时序约束**，其他厂商需用户手工挂；FuseSoC 用 `.core` 文件（CAPI=2 格式，声明 fileset/依赖/target/provider）描述 IP，`UpdateCoreFiles.py` 借 Jinja2 模板批量生成 dev（指向本地 `vhdl/`、库名 `open-logic-dev`、无 provider）与 stable（指向 `src/<area>/vhdl/`、库名 `open-logic`、带 GitHub provider）两套，区域依赖如 fix→base+en_cl_fix 用 `^` 兼容前缀声明；CI 采**免费 GitHub 轨（HDL-Check 含 GHDL/NVC 仿真、VSG lint、综合 YAML dry-run；Doc-Check）高频跑、付费 AWS 轨（覆盖率 95% 门禁、综合、FuseSoC、参考设计构建）仅 PR 到 main 或定期跑**，fork PR 需人工批准；贡献一律 PR 到 `develop`，合并到 `main` 触发付费门禁，发布时 `release.yml` 自动打包含子模块的 `CompleteSources.zip`。stable core 验证只能定期跑，因其依赖新 release tag——体现 CI 与发布时序耦合。整体哲学：机器可推导的部分（脚本通配、core 生成器）自动化，人只维护单一真相源（实体+测试+文档+综合 YAML）。
