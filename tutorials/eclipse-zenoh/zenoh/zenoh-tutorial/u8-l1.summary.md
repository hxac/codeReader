本讲立起 Zenoh 路由的三大最小模块：Gateway（总装车间）、Face/FaceState（路由单元）、Tables/TablesData（共享状态）。核心结论是三者咬合关系：Gateway 只持一把 `Arc<TablesLock>`，并通过 `new_session`/`new_transport_unicast` 等工厂方法把传输或本地 Session「缝合」成一张 Face；Face 是廉价克隆句柄（持 `Arc<TablesLock>`+`Arc<FaceState>`），`WeakFace` 用于打破循环引用。

关键术语：Gateway 负责按 `region × 角色` 实例化多套 HAT（路由策略）；FaceState 是真实状态，Face 双向使用——实现 `Primitives` 作为入站入口（被 DeMux 调用），又持有 `EPrimitives`（Mux/McastMux）作为出口，这就是「Mux/DeMux 缝在同一张 Face」的落点。Tables 由全局 `TablesData`（含资源树根 `root_res`、`faces` 注册表、`routes_version` 缓存版本号）与按 region 的 `hats` 组成，经 `TablesLock` 三把锁（`tables`/`ctrl_lock`/`queries_lock`）管控并发。

资源匹配核心：`Resource` 树是 key expression 的 trie，其 `face_ctxs: IntHashMap<FaceId, Arc<FaceContext>>` 记录「谁对该 key 感兴趣」，`FaceContext.subs/qabl/token` 是具体形态。一条声明的完整动作是「登记 + 失效 + 传播」：`declare_subscriber` 解析/创建 Resource、写本地 HAT 结构、`disable_data_routes` 令缓存路由失效、`propagate_subscriber` 跨 region 传播。全程遵循「锁内收集声明 → 出锁再发送」纪律；数据投递经 `route_data` 查表，用 `egress_filter` 与 `max(gwys)` 选主做去重。本讲刻意回避各 HAT 具体策略，留待下一讲。
