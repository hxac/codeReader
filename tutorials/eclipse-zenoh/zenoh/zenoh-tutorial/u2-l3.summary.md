本讲拆解 `zenoh::Config` 的内部结构与 WhatAmI 三种节点角色，分三个最小模块。

配置结构上，`zenoh::Config` 是对内部 `zenoh_config::Config` 的私有 newtype 封装，所有字段不公开，只能用 `from_file` / `from_json5` 加载、`insert_json5` / `get_json` / `remove` 以「字符串键 + JSON5 值」读写。配置是一棵以斜杠分层、带校验的弱类型键值树（如 `mode`、`listen/endpoints`、`scouting/multicast/enabled`）。这样设计是因为配置结构本身被声明为 unstable，内部重构不破坏用户代码，配合 `#[serde(deny_unknown_fields)]` 拦截拼错的键。底层读写由 `validated_struct` crate 的 `ValidatedMap` trait 实现；`from_file` 按扩展名分派到 `json5` / `serde_yaml`（`.toml` 仅 unstable 支持）。术语 serde 是其序列化基础。

角色上，`WhatAmI` 是 Router / Peer / Client 三种节点角色，默认 Peer，由顶层 `mode` 决定；本端角色只能通过配置观察，公开 API 仅能读到对端角色。角色用比特位编码，`WhatAmIMatcher` 以位掩码表达角色集合，serde 下分别序列化为字符串与字符串数组。`ModeDependentValue<T>`（`Unique` / `Dependent`）让同一配置项随角色取不同值，是配置文件「单值或 `{router,peer,client}`」写法的底层依据。

默认值有两个来源：`defaults.rs` 是运行时真默认（`mode=Peer`、router 监听 `7447`、peer 监听随机端口、client 不监听），`DEFAULT_CONFIG.json5` 是给人看的带注释参考清单、不宜整份拷贝。进入 Runtime 前的 `expanded()` 会补全缺失的 `id` 与 `mode`，保证运行时一定有值。
