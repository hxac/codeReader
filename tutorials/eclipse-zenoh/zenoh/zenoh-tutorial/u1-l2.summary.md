本讲承接《u1-l1》，动手把 Zenoh 跑起来，覆盖三个最小模块：构建说明、examples crate 的组织方式、z_pub/z_sub 主流程。

构建上，Zenoh 是标准 cargo workspace，最低 Rust 1.75，编译用 `cargo build --release --all-targets`（`--release` 保性能、`--all-targets` 才会编出 example），运行示例用 `cargo run --example <名字>`（需 `--` 传参）或直接跑 `target/release/examples/` 下的二进制。

`examples` 是名为 `zenoh-examples` 的正式 crate（workspace 成员、`publish = false`），兼具教学与测试职能。它对 `zenoh` 依赖用 `default-features = false` 再由自己的 feature 重新启用，以收拢开关；每个示例用 `[[example]]` 声明，靠 `required-features` 门控（如 `z_pub_shm` 需 `shared-memory`+`unstable`）。`examples/src/lib.rs` 定义 `CommonArgs`（含 `-e/--connect`、`-m/--mode`、`-l/--listen`、`--no-multicast-scouting` 等），通过 clap `#[command(flatten)]` 嵌入各示例，并实现 `From<CommonArgs> for Config` 把 CLI 参数翻译成 Zenoh `Config`。

z_pub/z_sub 体现 Zenoh 应用「三步走」：`zenoh::open(config)` 打开 `Session` → `declare_publisher`/`declare_subscriber` 声明实体 → 循环 `publisher.put()` / `subscriber.recv_async()` 收发，数据以 `Sample`（含 key、payload、kind、Encoding）为单位。核心结论：发布端与订阅端不写「找对方」的代码，匹配完全由 key expression 相交关系决定——默认 `z_pub` 发 `demo/example/zenoh-rs-pub`、`z_sub` 订 `demo/example/**` 天然匹配；`*` 匹配单层、`**` 匹配多层，改 key 即改变送达关系。本讲为后续 `Session`/`Key Expression`/`WhatAmI 角色` 等单元铺垫了直觉。
