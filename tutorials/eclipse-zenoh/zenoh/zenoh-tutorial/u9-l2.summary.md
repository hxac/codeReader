本讲进入 Zenoh 协议栈的传输层，核心是统一管理 unicast（点对点）与 multicast（组播）的 `TransportManager`。它由 `Runtime` 在 `zenoh::open` 阶段经 `builder().from_config(&config).build(handler)` 唯一构造，对外用统一入口 `add_listener`，内部靠 `locator_inspector.is_multicast()` 自动分派到两类子系统，把 Link 层的裸字节连接升级为带保活、序列号、分片与优先级分道的逻辑传输。

关键概念：`TransportUnicast` 按**对端 zid** 索引，一条传输对应一个对端；`TransportMulticast` 按 **Locator**（多播组）索引，一条传输对应一组多个 peer，且「监听」即「加入组」。保活核心是公式 `KeepAlive 间隔 = lease / keep_alive`（默认 10s/4=2.5s），TX 主动发心跳、RX 用 `lease` 判死链超时断开。unicast 另用 `max_sessions`、`accept_pending`/`accept_timeout`、`open_timeout` 约束会话数与防 DoS；multicast 改用 `join_interval` 做组播心跳，默认不开 QoS。`schedule` 是上层 `Mux` 把消息送上网的最终调用点。

本次新增：`TransportManagerState` 在 `uring+linux` feature 下持有 `uring: Option<Uring>`，由 `Uring::new(batch_size, link_rx_buffer_size)` 算出单批次（`batch_size+2`）与批次数量初始化；失败时只打 warn 并置 `None`、逐链路回退 tokio RX——io_uring 是加速器而非必需品。建连握手、批处理分片、uring RX 分派细节分别留给 u9-l3、u9-l4、u9-l5。
