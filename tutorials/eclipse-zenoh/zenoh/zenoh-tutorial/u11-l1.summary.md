本讲剖析官方路由器二进制 `zenohd`：它本质是「Zenoh runtime + 插件管理器」的一层薄壳，`main` 流程为 init_logging → clap 的 `Args::parse()` → `config_from_args()` → `zenoh::open(config).wait()` → `std::thread::park()` 常驻后台（故称 daemon）。

关键概念：clap derive「字段即参数」，12 个字段即 12 类 CLI；`config_from_args` 是配置合并器，优先级为「基础配置（空 key 的 `--cfg ':{...}'` > `--config FILE` > 默认）→ 结构化 CLI 覆盖（`--listen`/`--connect` 整体替换而非追加）→ 非空 key 的 `--cfg KEY:VALUE` 最后执行、优先级最高」；mode 未设时强制设为 Router，并强制开启 adminspace 与 plugins_loading；其结构化字段访问靠 `unstable` feature 门控的 `Deref`，普通应用只能用 `insert_json5`。

插件机制：`zenoh::open` 内 `RuntimeBuilder::build` 在 `plugins` feature 下分两阶段——`load_plugins`（runtime 组装中段、读配置加载库）→ `start_plugins`（runtime 就绪后把 `DynamicRuntime` 交给插件启动）。引入术语静态插件（编译期链接）与动态插件（运行时加载动态库，须与 zenohd 同 Rust 版本/依赖/feature 否则 ABI 不兼容而崩溃），`required` 插件失败会 panic 退出。`Plugin` trait 接口细节留待 u11-l2。
