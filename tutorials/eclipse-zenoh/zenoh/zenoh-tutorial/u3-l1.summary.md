本讲承接《u2-l1 Session》《u2-l2 Key Expression》，专讲 Pub/Sub（data in motion）主链路，Query/Reply 留到 u4。核心结论有四：其一，发布侧有两种姿势——长生命周期 `Publisher`（`Session::declare_publisher` 后反复 `put`/`delete`，QoS 只配一次，适合周期发送）与一次性快捷方法 `Session::put`/`Session::delete`（源码上即「临时声明 Publisher 再发一次」的语法糖）。其二，`declare_*` 与 `put`/`delete` 返回的都是 builder，必须 `.await`/`.wait()` 才 resolve 真正执行，否则仅触发 `#[must_use]` 警告；`Publisher`、`Subscriber` 均在 `Drop` 时自动 undeclare。其三，`Subscriber<Handler>` 借 `Deref` 透明暴露默认 handler（`DefaultHandler = FifoChannel`）的 `recv_async()`，惯用写法是 `while let Ok(sample) = sub.recv_async().await`，sender 全部丢弃即返回 `Err` 自然退出。其四，`Sample` 是 Pub/Sub 数据单元，关键字段为 `key_expr`、`payload`（`ZBytes`）、`kind`、`encoding`、`timestamp`、`attachment`；`SampleKind` 只有 `Put`/`Delete` 两值，`Delete` 的 payload 为空，仅表示「该 key 数据已清理」。投递与否完全由两端 key expression 的相交关系决定，而非地址。Handler 细节留待 u3-l2，QoS 留待 u3-l3，ZBytes 留待 u5-l1。
