本讲讲解 Zenoh 的 Matching（对端匹配感知）：让 Publisher/Querier 主动感知此刻网络中是否存在匹配的 Subscriber/Queryable，从而按需发送、省带宽省电。覆盖三个最小模块：MatchingStatus、MatchingListener、节流发布。

核心结论：默认 Publisher 是「盲目发送」的，Matching 提供两种 API。`matching_status()` 是一次性的「拉」查询，返回仅含一个布尔的 `MatchingStatus`（用 `matching()` 取值）；`matching_listener()` 是长期的「推」监听，注册时立即探测当前状态，之后仅在布尔翻转时回调（边沿触发）。匹配判定基于 key expression 的 `intersects` 关系，按内部 `MatchingStatusType`（Subscribers / Queryables）与 `Locality`（SessionLocal / Remote / Any）分派。

引入的关键术语：`MatchingStatus`、`MatchingStatusType`、`MatchingListener`、边沿触发、`.callback()`/`.with()`/`.background()` 三种取数姿势、`MatchingListener` 与 `Subscriber` 同构（Deref 到 handler、Drop 自动反声明）。落地应用为节流发布：用监听器回调维护一个共享原子布尔，发布热循环只读布尔决定是否 `put`，做到「有订阅者才发、无则休眠」。此外，complete Queryable 的匹配用 `includes` 而非 `intersects` 判定。远端匹配机制留待后续内核讲义展开。
