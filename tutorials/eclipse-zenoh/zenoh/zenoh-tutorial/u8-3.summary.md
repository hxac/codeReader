本讲承接上讲的 Gateway/Face/Tables 与 HAT，讲清 dispatcher 子模块如何把一条消息路由到对端 face。核心四块：Resource 是 key expression 前缀树，`get_matches` 做 chunk 级相交匹配、仅返回被声明过（有 context）的节点，其 `FaceContext` 的 `subs/qabl/token` 位是全部路由判断的依据；pubsub 的 `route_data` 把 `Push` 委托给 HAT 的 `compute_data_route`，只向 `subs` 置位且异 region 的 face 转发，路由结果按 `RoutesVersion` 缓存、`declare_*` 时失效重算；queries 的 `route_query` 按 `QueryTarget`（All/AllComplete/BestMatching）选 queryable，用 `pending_queries` 与 `QueryCleanup` 管理待应答与超时，应答时把本地 qid 翻译回发起方 src_qid；interest 是「对声明的兴趣」，Future 兴趣存于 `face_hat.remote_interests`、在 `maybe_propagate_subscriber` 充当门禁，只给匹配兴趣的 face 传播声明、其 `subs` 才被置位，故 interest 不走数据快路径、只在声明时刻塑造订阅表（`remote_key_interests` 另用于短 keyexpr 带宽优化）；token 是 liveliness 在路由层的落点，机制与 subscriber 同构但更轻。贯穿规律：dispatcher 负责「登记+失效+委托+发送」，HAT 负责「算路由」，并发上遵循锁内读路由、出锁再发送。
