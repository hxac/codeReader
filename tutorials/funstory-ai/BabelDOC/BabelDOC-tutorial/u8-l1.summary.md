本讲讲 BabelDOC 翻译 PDF 所需外部资源的「来源、校验、缓存、离线分发」，覆盖四个最小模块：资源清单与校验、上游择优下载、warmup 预热、离线包生成与恢复。核心结论：BabelDOC 不把字体、ONNX 版面模型、CMap、tiktoken 词表这四类上百 MB 的二进制塞进 pip 包，而是随包内置清单（manifest）与 SHA3-256 指纹，运行时按需下载到 `~/.cache/babeldoc/`。校验基石 `verify_file` 流式（1MB/块）算哈希比对，损坏即重下；`download_file` 叠加 tenacity 重试。多上游（GitHub/HuggingFace/ModelScope）用 `asyncio.as_completed` 赛跑择优并缓存胜者，但 GitHub 只存字体不存模型，故模型下载单独 `exclude_upstream=["github"]`。`--warmup` 经 `asyncio.gather` 一次性下全并预热 tiktoken，且在 CLI 校验翻译服务之前短路，无需 API key。离线包以权威清单的 SHA3-256 作为文件名 tag（不可改名），生成端打包前校验、恢复端 tag 校验加解出后校验，三重把关支撑气隙部署。全模块 async 函数经 `run_coro`（新线程加独立事件循环）桥接给同步主链路。资源如何被下游使用不在本讲范围。
