本讲聚焦 BabelDOC 唯一直接对话 LLM 的「翻译器服务」，覆盖四个最小模块：翻译器骨架、翻译缓存、漏桶限流、token 统计。`BaseTranslator` 采用模板方法模式，对外暴露 `translate()` 统一负责「查缓存→限流→翻译→写缓存」，子类（唯一内置的 `OpenAITranslator`）只实现 `do_translate()` 调 OpenAI 兼容 API；核心结论是缓存命中时既不调 LLM 也不耗限流配额。关键术语：`OpenAITranslator` 固定 `temperature=0` 并用 `tenacity` 对 429 指数退避重试；翻译缓存以 peewee + SQLite 落盘到 `~/.cache/babeldoc/cache.v1.db`，唯一键 `(engine, params, text)` 经递归排序稳定化，按 0.1% 概率清理并保留最新 5 万行；漏桶 `RateLimiter` 用 `time.monotonic()` 把并发请求平滑成匀速，全局单例 `_translate_rate_limiter` 经 `set_translate_rate_limiter` 设置，所有翻译器共享同一 QPS 配额；token 统计用 `AtomicInteger` 线程安全累加四类计数，`update_token_count` 整体 try/except 包裹以保证「统计失败不影响翻译」。另需区分本地翻译缓存与上游 prompt 缓存命中。本讲不涉及整篇 PDF 的批量并发编排，留待 u6-l2 的 `ILTranslator`。
