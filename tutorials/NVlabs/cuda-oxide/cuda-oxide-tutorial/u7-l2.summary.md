本讲覆盖 cuda-oxide 的 GPU 调试链与设备端 debug intrinsics。`cargo oxide debug` 是 cuda-gdb 的一条龙入口：叠加 `CARGO_PROFILE_RELEASE_DEBUG=2` 出带 DWARF 的 release 制品，按「PATH → toolkit root → 回退路径」发现并拉起 cuda-gdb；#330 让其解析 Cargo 的 `compiler-artifact` JSON 精确发现可执行文件，补齐 `--bin`/`--features`，歧义时报错。设备端 debug intrinsics（`clock`/`clock64`/`globaltimer`/`trap`/`breakpoint`/`prof_trigger` 与 `gpu_assert!` 宏）在源码里全是 `#[inline(never)]`+`unreachable!()` 桩，由编译器按全限定名识别后降级：计时类走 LLVM intrinsic，`trap`/`breakpoint`/`prof_trigger` 走 convergent 内联 PTX；`prof_trigger::<N>()` 借 const generic 把事件号做成 PTX 立即数，mir-importer 专门拦截抠值。`#[launch_bounds(max,min)]` 经「标记函数注入 → mir-importer 扫描 → 函数属性 `maxntid`/`minctasm` → PTX `.maxntid`/`.minnctapersm`」落地，是性能/占空提示而非安全契约。核心结论：`trap()` 总终止内核，`breakpoint()` 调试器下为暂停点、离开调试器变陷阱，`gpu_assert!` 当前退化为 `trap()`。这些是「设备桩模型」的又一族，与 `#[launch_contract]` 的活设备校验严格区分。
