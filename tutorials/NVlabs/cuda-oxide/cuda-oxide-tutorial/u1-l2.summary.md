本讲只读根 `Cargo.toml` 与 `README.md`，建立 cuda-oxide 的 crate 地图。workspace 共 **19 个成员**，按注释分为 Core(12)、FFI bindings(5)、内部命名契约(`reserved-oxide-symbols`，1)、Fuzzer(`fuzzer`，1) 四组。引入的关键分类视角：**用户面 / 编译器 / 工具三层**，并在用户面内区分**设备端**（`cuda-device`，编译进 PTX、GPU 执行）与**宿主端**（`cuda-core`/`cuda-host`/`cuda-async`/`cuda-bindings`，CPU 执行），`cuda-macros` 属编译期过程宏。编译器 crate 与流水线工位一一对应：`mir-importer`(MIR→dialect-mir) → `mir-lower`(→LLVM dialect) → `llvm-export`(→.ll)，`dialect-mir`/`dialect-nvvm` 负责方言建模。核心结论：最关键的 `rustc-codegen-cuda` **编外于 workspace**，因它是被 rustc `dlopen` 的 **dylib 插件**，靠 `#![feature(rustc_private)]` 访问 rustc 内部 crate，用空 `[workspace]` 主动退出父 workspace，且必须手动同步 pliron 的 git rev。设计哲学是「把 CUDA 在场推迟到运行时」（`dlopen` 或纯逻辑），唯 `cuda-bindings` 因 `bindgen` 读 `cuda.h` 而编译期依赖 Toolkit。本讲交付物为一张四层 crate 分层依赖草图，供 u2/u3/u4 切入深挖。
