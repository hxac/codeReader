本讲是「宿主运行时」单元第三篇，覆盖 `cuda-async` crate 的异步执行模型。核心抽象是 `DeviceOperation` trait——它**惰性、流无关**：构造时只描述「做什么」，不分配 GPU 资源、不携带任何 `CudaStream`，把选流推迟到调度时刻。四条执行路径区分明确：`schedule`（按策略挑流并包成 `DeviceFuture`）、`.await`（线程级默认策略 + future，非阻塞）、`sync`/`sync_on`（`execute` 入队 + `synchronize` 阻塞到流空闲）、`async_on`（当场入队不等）。`vecadd_async`/`load_async` 等用户面 `*_async` 方法由过程宏生成，把一次内核启动包装成 `AsyncKernelLaunch`（实现 `DeviceOperation`），与 u2-l4 同步版的本质区别是**入队被推迟**到 `.sync()`/`.await`。`DeviceFuture` 用三态机 `Idle → Executing → Complete`（加 `Failed` 态）实现 Rust `Future`：GPU 工作**在第一次 poll 的 Idle 分支才真正入队**。完成检测靠 `cuLaunchHostFunc` 主机回调置位原子标志并经 `AtomicWaker` 唤醒执行器，**绝不忙等**；drop 在飞 future 不取消 GPU 工作，而是把结果连同完成事件寄存到 `reclaim` limbo，由后续 `sweep` 非阻塞回收，保证 drop 永不阻塞 GPU 进度。默认调度策略为 `StreamPoolRoundRobin`（默认 4 条流，原子计数器轮询挑流）。
