本讲深潜 PR #314 抽出的 `cuda-oxide-codegen`：cuda-oxide 流水线中唯一不依赖 rustc 的后段，以 dialect-mir IR 为公共边界，接受 Pliron in-memory IR 产出可被 ptxas 接受的 PTX，全程无需 `rustc_private` 或 nightly 工具链。crate 仅暴露两个表面：`experimental`（v1 实验契约——源码级兼容、产物自包含、无链接步骤）与 `#[doc(hidden)] __private`（仅供 mir-importer 复用的内部钩子）。关键 API 包括把 Context 与根 ModuleOp 绑死的 `CodegenModule`、用 `gpu_kernel` 属性标记 PTX 入口的 `mark_kernel_entry`、以及发现并强制配对同主版本 `llc`/`opt` 的 `Toolchain`（下限 LLVM 21）。核心结论是：rustc 前端与 standalone 前端复用同一份后段编排器 `compile_translated_module`，顺序为 verify → mem2reg/unroll → lower → export → llc，两路径的差异（NVVM IR 策略、extern 链接、工具链发现）全吸收在 `ModulePipelineRequest` 字段里；standalone 路径刻板地永不出 NVVM IR、拒绝未解析符号、缺 `opt` 时严格失败。环境变量读取集中于 `BackendOptions::from_env`，仅在 rustc 边界调用一次，后段本身不读 env。
