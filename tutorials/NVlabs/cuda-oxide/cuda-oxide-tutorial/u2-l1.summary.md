本讲拆解 cuda-oxide 的过程宏层，覆盖五个最小模块。`#[kernel]` 是属性宏，把普通函数改名为带保留前缀的 `cuda_oxide_kernel_246e25db_<name>`（加 `#[no_mangle]`），生成见证类型 `__<name>_CudaKernel` 与 `CudaKernel` 实现；但 PTX `.entry` 符号仍是干净的裸函数名。`#[cuda_module]` 扫描 inline 模块（#324 起含嵌套 inline 子模块）中的 kernel，生成 `LoadedModule`、`load*`/`from_module` 加载器与每个内核的启动方法，嵌套命名空间通过 `from_parent` 共享同一已加载模块，且全树内核名必须唯一。`#[launch_contract(...)]`（#318）让作者声明 domain/block/动态共享内存/算力等假设：宏据此为 `DisjointSlice` 加品牌化 sealed bound（`__LaunchContractDisjointSlice`），为见证类型实现 `KernelLaunchContract`（`type Config = LaunchConfig1D/2D/3D` 锁死 rank），生成 `prepare_*` → `PreparedLaunch<__name_CudaKernel>` 受检启动链，活设备校验在 `PreparedLaunch::__prepare` 完成。关键术语包括：见证类型、品牌化 sealed trait、raw `LaunchConfig`、`PreparedLaunch`、`_unchecked` 专家路径、artifact 锚符号。安全边界结论：未签约内核同名方法吃 raw config 故 unsafe；签约模块所有 loader 变 unsafe（一次性证明）；签约内核另有 `_unchecked` 逃生口。`reserved-oxide-symbols` crate 是命名契约唯一真相源，靠 `246e25db` 哈希让前缀不可伪造，绑定宏、codegen 后端、lowering/export、运行时加载器四方。
