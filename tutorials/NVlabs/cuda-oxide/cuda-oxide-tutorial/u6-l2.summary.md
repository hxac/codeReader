本讲深潜 mir-importer 的终止符翻译，聚焦 `TerminatorKind::Call` 如何被识别为 GPU intrinsic、闭包调用或普通函数调用。核心入口是 `translate_call`，它呈漏斗结构：先跳过死分支、处理闭包与函数项，再拦截需从 callee const generic 抠参数的「特殊常量泛型 intrinsic」（prof_trigger、unroll、DynamicSharedArray、core atomic、assert_inhabited），最后交 `try_dispatch_intrinsic` 大 `match`，剩余即普通 `mir.call`。intrinsic 发射逻辑按功能类别拆进 `terminator/intrinsics/` 下 21 个子模块（目录是目标态、迁移渐进），所有 handler 用 `emit_store_result_and_goto` 统一收尾。重点剖析两类：atomic 模块同时服务 `cuda_device::atomic::*`（作用域编进类型名、ordering 运行期参数）与 `core::sync::atomic::*`（ordering 为 const generic、作用域恒 System）两条前端，合流到同一批 `NvvmAtomicRmwOp`，但两者 ordering 判别值表在 Acquire/Release 上互换须分别映射；wmma 模块用「`extract_array_registers` 拆 fragment 数组为标量寄存器 → 建 mma op → `MirConstructArrayOp` 装回数组」三段范式处理各 dtype，本轮新增 f16 `m16n8k16`、tf32 `m16n8k8`、s8 `m16n8k32` 三条。引入的关键术语：terminator 漏斗分派、设备桩（`#[inline(never)]`+`unreachable!()`，靠名字识别调用改写为 op）、fragment 拆装、alloca/load-store 写回模型。新增一条 intrinsic 需在「设备层→dialect op→importer 翻译→importer 分派→lowering」四层落地。
