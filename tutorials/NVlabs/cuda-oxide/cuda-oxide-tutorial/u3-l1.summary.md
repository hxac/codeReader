本讲聚焦 cuda-oxide 宿主运行时的地基 `cuda-core` crate，把以 `CUcontext`/`CUstream`/`CUmodule`/`CUevent` 为中心的 C 接口包成带生命周期的 Rust RAII 类型，共四个最小模块：`CudaContext` 保留设备主上下文并管理线程绑定；`CudaStream` 提供非阻塞流与 fork/join 依赖图；`CudaModule`/`CudaFunction` 加载 PTX 并取内核句柄；`CudaEvent` 做跨流同步与 GPU 端计时。四类共享三条暗线：每个类型持有一份 `Arc<CudaContext>` 实现链式保活，方法调驱动前先 `bind_to_thread()`，错误统一经 `IntoResult` 转成 `DriverError`。关键结论包括：线程绑定是所有驱动调用的隐式前置条件；`join` 本质是 `record_event`+`wait` 的语法糖；`CudaFunction` 持有 `Arc<CudaModule>` 防止模块被提前卸载；粘性错误用原子变量延迟补报；计时必须用 `CU_EVENT_DEFAULT`，默认的 `DISABLE_TIMING` 不能 `elapsed_ms`。本讲（承接 #318）最重要的新概念是「活设备能力查询」：`CudaContext` 与 `CudaFunction` 新增函数属性、设备上限、cooperative/cluster 支持等懒查询，构成类型化启动契约 `prepare_*` 产出 `PreparedLaunch` 见证类型时的宿主侧受检验证数据源，把 kernel 启动从调用方自证升级为运行期可证。
