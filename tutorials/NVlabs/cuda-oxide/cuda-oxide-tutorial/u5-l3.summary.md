本讲承接 u5-l1（warp 原语）与 u2-l3（共享内存与同步），从「线程间通信」升级到「线程与硬件异步单元之间的通信」，核心是 mbarrier 异步屏障与 cp.async 异步拷贝两类原语。

关键结论：`sync_threads()` 是阻塞式、只认线程的屏障，无法跟踪硬件 DMA 事务；而 mbarrier 是共享内存中的 8 字节硬件屏障，完成条件为「期望到达数齐 ∧ 期望事务字节数齐」两者取逻辑与，故能感知异步搬运。其 arrive/wait 协议围绕 token（相位令牌）、phase（相位位，循环复用）、expect_tx（TMA 事务字节计数）展开；无 token 生产者（如 `tcgen05.commit`）改用 parity 奇偶等待；等待原语首选带调度提示的 `try_wait` 而非纯探测 `test_wait`。

cp.async 把搬运的「下单」与「完成」解耦：线程下单即走、硬件后台 DMA，完成靠每线程的 `commit_group`/`wait_group`/`wait_all` 或 mbarrier，等单类内联 PTX 必须带 `clobber("memory")` 防重排；zfill 变体按整块拷、无效部分硬件补零（`src_size ∈ 0..=cp_size`）。两者组合即「搬运-计算重叠」的双缓冲软件流水线（`wait_all` 是等死模式，不产生重叠）。

关键术语：mbarrier、token、phase、expect_tx、parity、cp.async、zfill、commit_group/wait_group、`fence_proxy_async_shared_cta`（generic proxy 与 async proxy 间搭桥，TMA 必备）、typestate 封装 `ManagedBarrier`、`TmaBarrier0/1`。所有设备端 API 仍是「编译器识别桩」（`#[inline(never)]` + `unreachable!()`），由 mir-importer 译成 `dialect-nvvm` 的 `CpAsyncCa*Op`/`Mbarrier*Op`/`Barrier0Op`，再由 mir-lower 降级为 PTX。本讲为 u5-l5 的 TMA（`cp.async.bulk.tensor`）与 u5-l4 的集群屏障铺路。
