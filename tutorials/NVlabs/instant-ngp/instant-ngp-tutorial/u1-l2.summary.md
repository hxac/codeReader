本讲绘制 instant-ngp 仓库的目录地图。仓库根目录职责分明：`src/` 放 CUDA/C++ 实现，`include/neural-graphics-primitives/` 放头文件声明，二者近乎一一对应；`configs/` 按四种基元模式分目录存放网络 JSON 配置；`dependencies/` 装第三方库；`scripts/` 是 Python 自动化；`data/` 是示例数据。理解这些目录协作的钥匙是 `CMakeLists.txt`：它把 `src/*.cu` 编译成静态库 `ngp`，再链接出 `instant-ngp` 可执行文件和 `pyngp` Python 模块，编译时通过 `NGP_SOURCES` 与 `NGP_INCLUDE_DIRECTORIES` 管理源文件与头文件搜索路径。

关键概念：`ETestbedMode` 枚举（`Nerf/Sdf/Image/Volume/None`）定义于 `common.h`，`configs/` 的四个子目录与之严格对应。`dependencies/` 含 21 个库，分两类——10 个 git 子模块（克隆须加 `--recursive`，否则配置阶段致命报错）与 vendored 直接拷入目录；其中 `tiny-cuda-nn` 是承载底层神经网络算法的必选依赖，GPU 架构选择委托给其 `TCNN_CUDA_ARCHITECTURES`；GUI/OptiX/Python/Vulkan/RTC 五项为可选能力，由 `NGP_BUILD_WITH_*` 开关控制。

核心结论：整个项目围绕「上帝对象」`Testbed` 类组织，其声明在 `testbed.h`（1294 行），骨架实现在 `testbed.cu`（5672 行，全仓库最大），承载帧循环、文件分发、网络重建、快照、GUI 等共用逻辑；四个 `testbed_*.cu` 是按模式拆分的成员函数实现。`m_testbed_mode` 是几乎所有分发逻辑的开关。后续讲义将逐层拆解 `Testbed` 的主循环、文件加载（`load_file`）、网络重建（`reset_network`）与各模式专属逻辑。
