本讲讲的是 BUS/RT 的「嵌入模式」：在自己的 Tokio 进程里直接 new 一个 `Broker`，它只是一个普通 Rust 值，内核 `BrokerDb` 装着三张路由表（clients / broadcasts / subscriptions）和收发计数。`Broker::new()` 等价于 `Broker::create(&Options::default())`，唯一区别是 `Options` 可开启 `force_register`（名字冲突时踢掉旧实例）。

通信要先注册客户端：`Broker::register_client(name)` 创建一个 `BusRtClientKind::Internal` 的内部客户端，返回 `broker::Client` 句柄（持有 `Arc<BusRtClient>` 与入站通道接收端 `rx`）。名字含 `%%`（`SECONDARY_SEP`）时取分隔符前为主名，由此区分 primary / secondary 客户端；身份比较基于 `sha256(name)` 摘要。客户端通过 `async_channel::bounded(8192)` 有界通道收消息，`take_event_channel()` 拿走接收端交给消费循环，形成天然背压。

关键结论：`broker::Client` 实现统一 `AsyncClient` trait，但没有连接概念——`is_connected` 恒真、`ping` 空操作、`get_timeout` 为 `None`、确认立刻兑现。满队列时内部客户端**阻塞发送方**，而外部 IPC 客户端被**强制断连**，这是两者最关键的行为差异。官方 `examples/inter_thread.rs` 是最小蓝本，演示点对点 `send` 与 `send_broadcast` 广播掩码（`worker.*`）。三张路由表、secondary 客户端、广播/订阅掩码的细节留给 u3-l2、u3-l3 详讲。
