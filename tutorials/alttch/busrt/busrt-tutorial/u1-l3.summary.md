本讲建立 BUS/RT 的「源码导航地图」，只依赖 `src/lib.rs` 与 `Cargo.toml` 两个索引文件。核心结论有三：其一，`lib.rs` 顶部的协议常量（`OP_*` 操作码、`ERR_*` 错误码、`GREETINGS` 握手魔数、`PROTOCOL_VERSION`、`PING_FRAME`）与核心类型别名 `Frame = Arc<FrameData>`、`EventChannel` **始终编译**，是整个库不依赖任何 feature 的公共契约。其二，`lib.rs` 后半部分用 `#[cfg(feature = "...")]` 守卫声明各模块，形成「feature→文件」映射：`broker.rs`/`ipc.rs`/`cursors.rs` 为单 feature 守卫，而 `rpc`、`sync` 因异步/同步共用而用 `any(rpc, rpc-sync)`、`any(ipc-sync, rpc-sync)`，且模块内部还有二级守卫（`rpc/async_client.rs` 需 `rpc`，`sync/rpc.rs` 需 `rpc-sync`）；始终编译的基础模块为 `borrow`、`common`。其三，`Cargo.toml` 的 `[features]` 通过**传递依赖**驱动编译：`broker-rpc = broker + rpc + rmp-serde`、`cursors = rpc + uuid`、`server → broker-rpc`、`cli → ipc + rpc`。关键陷阱：`full` **不含** `cursors`、`rpc-sync`、`rt`，也**不构建**两个二进制 `busrtd`（feature `server`）与 `busrt`（feature `cli`），后者由 `[[bin]] required-features` 门控；docs.rs 的 feature 集（含 `rpc-sync`、`cursors`）比 `full` 更全。本讲引入的术语 `OP_*`/`ERR_*`/`Frame`/`FrameData`/`AsyncClient` trait/`RpcEvent`/`TopicBroker` 等将由后续单元（核心类型、零拷贝 Cow、线上协议）承接展开。
