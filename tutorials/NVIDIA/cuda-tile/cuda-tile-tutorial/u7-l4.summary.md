本讲聚焦 CUDA Tile 的「字节码版本（bytecode version）」，强调它与项目发布版本、LLVM commit 是三套独立的概念，不能混为一谈。字节码版本用 `BytecodeVersion` 值类型刻画为三段式 `major.minor.tag`，按字典序构成全序；其合法集合与所有版本门代码均由 `.td` 声明经 tblgen 生成，体现「单一数据源」。关键常量有四个：`kCurrentVersion=13.3`（最新可写）、`kCurrentCompatibilityVersion=13.1`（最广兼容目标）、`kMinSupportedVersion=13.1`（读取端最老接受线）、`kUnifiedBitfieldVersion=13.3`（可选参数编码方式分界）。

兼容机制分两道门：粗粒度门由 opcode 永不重新编号的铁律与「版本→最大 opcode」映射支撑，用 `isOpcodeAvailableInVersion` 做 O(1) 比较，拦截整条新操作（新操作总是追加在 opcode 末尾）；细粒度门由操作、属性、操作数、结果、类型各自的 `sinceVersion` 标注驱动，处理「老操作长出新东西」。兼容是双向的——前向使旧工具遇新特性时报错或静默降级（如 13.3 Producer 段被丢弃），后向使新读取器能升级旧形态（如 13.1 的 `print` 读成 13.2 的 `print_tko` 且不破坏 SSA 编号）。读取端报错话术用 `file version is <T>`，写入端用 `targeting <T>`，便于区分「文件造假」与「目标版本写错」。`test/Bytecode/versioning/` 用仅在测试构建启用的 250.0/250.1 演练逐版本演进的全部分支。
