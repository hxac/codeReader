本讲解决 CUDA Tile 如何「绑定」到具体 MLIR/LLVM 版本的问题。核心结论是：CUDA Tile 寄生在 MLIR 上，直接使用其内部 API 与头文件，属于强耦合共生，而非松耦合依赖。因此它必须把上游 LLVM/MLIR **锁定到具体一个 commit**（当前为 `57109befac92...`，定义在 `cmake/IncludeLLVM.cmake` 中）。为追踪上游频繁的 breaking 改动，CUDA Tile 引入**兼容区间**机制：上游每个破坏性 commit 触发一个新 patch 版本，每个 CUDA Tile 版本对应主干上一段左闭右开区间；`get-cuda-tile-version-for-llvm-hash.sh` 可据 LLVM commit 反查应使用的 CUDA Tile 版本。注意 CUDA Tile 版本号、LLVM commit hash、字节码版本三者不同。

获取 LLVM 有三种互斥方式：**自动下载**（默认，最慢）、**本地源码**（`CUDA_TILE_USE_LLVM_SOURCE_DIR`）、**预编译库**（`CUDA_TILE_USE_LLVM_INSTALL_DIR`，最快），分派为「预编译 vs 源码」与源码内「本地 vs 下载」的二级嵌套。拿到 LLVM 后须执行 `find_package(MLIR)`，Python 绑定校验必须排在其后，因为该步会回填 `MLIR_ENABLE_BINDINGS_PYTHON`；从源码构建时此开关自动透传，预编译库则需自行保证。最后，`ccache` 经 `CUDA_TILE_ENABLE_CCACHE` 同时加速 CUDA Tile 与 LLVM，跨编译则禁止从源码构建 LLVM，必须走预编译库与 `CUDA_TILE_USE_NATIVE_LLVM_INSTALL_DIR` 以产出宿主 `cuda-tile-tblgen`。后续讲义（u2-l2 方言定义、u2-l3 TableGen）将承接这些术语。
