本讲回答「为 cuda_tile 方言新增操作/类型/属性时，`.tilebc` 的读写代码从何而来」——全部由构建期工具 cuda-tile-tblgen 自动生成，自身不参与运行时序列化。

核心模块有四组。三份 `.td` 文件（BytecodeOpcodes/TypeOpcodes/AttrOpcodes）是「操作→opcode、类型→TypeTag、属性→AttributeTag」的单一数据源，编码一旦分配即 FROZEN、永不重新编号，新编码只能尾部追加。写入端后端 BytecodeGen 为每个操作生成 `write<OpName>` 与 TypeSwitch 分派、Opcode 枚举及版本→最大 opcode 表，并在构建期强校验「每个操作必须登记 opcode」。读取端 BytecodeReaderGen 是其镜像，生成 `parse<OpName>` 与 `switch(opcode)`，多出「读旧文件时为新字段填默认值」与「保护 SSA 编号」两职责。BytecodeTypeCodeGen/AttrCodeGen 负责类型可选参数位域（13.3 unified bitfield）与逐枚举值版本检查。

关键术语：PublicOpcode/BytecodeTypeTag/BytecodeAttrTag、FROZEN 编码、flags 按版本序分配位、读写对称、`.inc` 的 `#define GEN_xxx`+`#include` 切片技巧、`isOpcodeAvailableInVersion`/`isAttrTagAvailableInVersion` 版本判定。结论：新增操作人工只需「定义语义 + 分配一个冻结 opcode」两步，其余胶水全部自动生成。
