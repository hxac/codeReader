本讲（u1-l2）讲解 CUDA Tile 的仓库目录布局与 CMake 构建骨架，覆盖三个最小模块。其一，顶层 CMakeLists.txt 声明 C++17 标准，用 `set_cuda_tile_build_type()` 将构建类型默认为 Release 并做白名单校验；功能由 `option` 控制，TESTING、BINDINGS_PYTHON 默认 OFF，CAPI、TOOLS 默认 ON，使最小构建尽量轻量。其二，LLVM/MLIR 配置块把上游锁定到具体 commit（这是上一讲「兼容区间」机制在构建层的落地），提供三种获取方式：自动下载并编译（默认、最慢）、本地源码（`CUDA_TILE_USE_LLVM_SOURCE_DIR`）、预编译库（`CUDA_TILE_USE_LLVM_INSTALL_DIR`），后两者互斥；Python 绑定校验须在 `find_package(MLIR)` 之后，因其依赖 MLIR 侧的 `MLIR_ENABLE_BINDINGS_PYTHON`。其三，子目录注册顺序是构建的关键：代码生成工具 `cuda-tile-tblgen` 必须最先构建，它用 TableGen 把 `.td` 翻译成 `.inc` 胶水代码，随后 include/lib 才能 `#include` 这些产物，再按开关注册 test/tools/python，核心依赖链为 tblgen→include/lib→tools/python/test；开启测试时 `TILE_IR_INCLUDE_TESTS` 宏会条件化生成仅测试用的操作。
