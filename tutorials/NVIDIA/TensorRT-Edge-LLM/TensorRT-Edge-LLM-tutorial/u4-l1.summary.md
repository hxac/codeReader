本讲打开 `build` 这一步的黑盒，定位「C++ 引擎构建器」为三段式流水线（检查点→Python 导出 ONNX→**引擎构建 engine**→运行时）的第二阶段，职责是把 ONNX 编成 TensorRT engine，本质是针对具体 GPU 做算子融合与 kernel 选优。

核心是 `LLMBuilder::build()` 的**八阶段线性流程**：插件加载、配置解析、网络创建（strongly-typed）、模型类型检测、ONNX 解析、优化 profile 设置、引擎编译（`buildSerializedNetwork`）、文件管理（拷贝 sidecar）。引入的关键术语：checkpoint、ONNX、engine、optimization profile（min/opt/max）、strongly-typed network、prefill/decode 双 profile、`LLMBuilderConfig`、`edgellm_version`。

核心结论：① ONNX 与 engine 不可跨 TensorRT/Edge-LLM 版本及 GPU 型号移植，由 `version::checkVersion` 比对 `edgellm_version` 做程序级强制——升级须重新 export+rebuild；② 阶段 6 用 `setupLLMOptimizationProfiles` 决策树按模型类型（vanilla/spec/dflash/gemma4-mtp）分发，并建 prefill 与 decode 双 profile；③ 投机解码（EAGLE3/MTP/DFlash）让八阶段跑两遍，分产 `spec_base.engine`/`spec_draft.engine`，仅文件名与 profile 分支不同。`LLMBuilder` 是编排者，底层 TensorRT 调用抽进 `builderUtils` 供多 builder 复用。
