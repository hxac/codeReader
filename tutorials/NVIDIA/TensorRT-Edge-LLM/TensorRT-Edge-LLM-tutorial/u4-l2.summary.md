本讲深入构建器八阶段流程的第 6 阶段「优化 profile 设置」，并连带覆盖第 4 阶段「模型类型检测」。核心模块有三：其一，优化 profile（min/opt/max 三元组，告诉 TensorRT 动态维度的运行时变化范围）；其二，双阶段 profile——LLM builder 一律创建 context（prefill）与 generation（decode）两个 profile，挂载顺序决定索引 0/1，运行时切换以让 prefill 的长序列与 decode 的单 token 各自命中最优 kernel。关键修正：本实现里两阶段共享 maxBatchSize，真正差异在序列长度维（prefill 到 maxInputLen、decode 为 1）。其二，模型类型分发有两层，共用 spec_decode_type/engine_role/maxLoraRank 三判据：profile 分发是一棵带提前返回的决策树（dflash/gemma4_mtp draft 自成一套；其余走通用底盘 common+rope，再 vanilla/spec 二选一，按需叠加 PLE/Deepstack/LoRA）；ONNX 选择为 maxLoraRank>0 时解析 lora_model.onnx，否则 model.onnx，parseConfig 强校验三者一致。其三，VisualBuilder 是独立 builder，只配单 profile（无 prefill/decode 之分），按 model_type 在 5 个 setup*ViTProfile 间 switch 分发，InternVL/Phi4 要求 image tokens 为 256 倍数。重要纠错：代码不存在 setupVLMProfiles，VLM 语言侧复用 setupVanillaProfiles。另外 EAGLE 的 base/draft 必须共用同一 engineDir，靠 spec_base/spec_draft、base_config/draft_config 命名与 draft 外部权重前缀防冲突。
