本讲讲解 EdgeLLM 如何把已建好空壳、灌好权重的 nn.Module 经 dynamo 导出固化成一张下游 C++ 引擎可消费的 ONNX 图，核心是三个文件的分工。`onnx_custom_schemas.py` 为 20 个融合算子（attention/Mamba/MoE/量化 GEMM）登记「身份证」，说明为什么需要 `trt`、`trt_edgellm` 两个自定义算子域，并幂等注册 schema（since_version 统一钉 23）保证 ONNX 校验认识这些节点；FP8 故意走标准 QuantizeLinear/DequantizeLinear 而非自定义 schema。`dynamo_translations.py` 提供 `custom_translation_table`——一张「FX 自定义算子 → onnxscript `@script()` 函数」映射表，翻译有「1 对 1」（如 Int4GroupwiseGemmPlugin）与「1 对多」（如 NVFP4 激活量化展开成量化加两次反量子图）两种形态。`export.py` 用 `torch.onnx.export(dynamo=True, opset_version=24)` 导出同时覆盖 prefill/decode 的单图，KV cache 与 Mamba 状态以图 I/O 暴露；因 dynamo 导出器面向通用 runtime、TRT 解析器更严格，导出后须跑一串兼容性后处理（NVFP4 dtype 由 INT8 改写为 FLOAT4E2M1、attention 可选输入裁剪、共享 DQL scale 去重、initializer dtype 修正）。关键直觉：`_permissive_inline_opset` 取 max 解决 opset 18/21 冲突；追踪前预提取 FP8 KV scale 为常量避免数据依赖。视觉/音频编码器经 `export_encoder.py` 复用同一 dynamo 底座与翻译表，差异仅在模型构建与 I/O spec 来源。本讲产出的合法、TRT 友好的 ONNX 图即交给下游 C++ 构建器解析的契约。
