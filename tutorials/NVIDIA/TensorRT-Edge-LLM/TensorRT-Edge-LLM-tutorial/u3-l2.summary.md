本讲讲解用户面向的量化命令行 `tensorrt-edgellm-quantize` 的三个子命令：`llm`（普通 LLM 与多模态各组件）、`draft`（独立 EAGLE3/DFlash 草稿模型）、`qwen3-omni`（Omni-MoE 联合 NVFP4）。核心设计是按组件拆分量化旋钮：backbone、lm_head、KV cache、视觉/音频/CP 塔各有 `--*_quantization`，使 backbone 与 lm_head 可用不同精度（混合精度），且每部件仅暴露经端到端精度验证的格式。关键结论：CLI 能生产的格式（fp8、int4_awq、nvfp4、mxfp8、int8_sq）只是导出侧 `QUANT_*` 常量全集的子集，多出的 int4_gptq、int4_awq_modelopt、mixed_precision 源自外部预量化检查点，本包只读不写。方法名经 `build_quant_config` 翻译为底层 ModelOpt 配方（由 `_BACKBONE_CFG_MAP`/`_LM_HEAD_CFG_MAP` 决定可组合范围），导出侧再由 `QuantConfig` 与 `module_quant_type` 落到每个 Linear 类。格式权衡的核心维度是位宽、是否量化激活（如 int4_awq 为 W4A16 仅权重、int8_sq 为 W8A8）与分组大小。投机解码 draft 量化分两路：EAGLE3/DFlash 走 `draft` 子命令（按 `dflash_config` 判别，默认 fp8/nvfp4）；Qwen3.5 MTP 因派生自 base，由 `llm` 子命令内部检测 `mtp_num_hidden_layers` 后自动先于 base 量化。所有产物统一为带 `hf_quant_config.json` 的 HF 检查点，不产出 ONNX 或 engine。
