本讲是 Python 导出前端（u2 单元）的第一篇，也是后续所有 Python 篇的公共地基。它讲清楚导出器拿到 HF 检查点目录后，如何解析出一份可消费的 `ModelConfig`，对应两个源码文件、三个最小模块。第一，`checkpoint_utils.load_checkpoint_config_dicts` 把检查点目录读成规整的 `(root_dict, llm_dict)` 两个字典：先用 `AutoConfig` 再回退原始 JSON 以兼容未注册类型，用 `_promote_llm_subconfig` 把多模态里藏于 `text_config`/`thinker_config` 等嵌套子对象的 LLM 配置「提升」上来，并做 RoPE 兼容回填；判定子配置是否为 LLM 的朴素判据是同时具备 `hidden_size` 与 `num_attention_heads`。第二，`config.py` 把 `llm_dict` 解析成强类型、扁平的 `ModelConfig`，覆盖架构尺寸、RoPE（多级兜底）、逐层 `layer_types`、Mamba/GDN 子配置、MoE、投机解码与张量并行等字段；`from_pretrained` 是唯一工厂入口。关键术语：`model_type`（模型族主键，仅原样读出）、`layer_types`（混合模型逐层标注 `attention`/`mamba`/`gdn`/`mlp`/`moe`）、混合模型（hybrid）。两个值得承接的设计细节：Mamba 的 `conv_dim` 在 JSON 缺失时会读 safetensors 的 `conv1d` 权重形状打破循环依赖；`has_qk_norm` 等采用「以权重 key 事实为准」而非按 `model_type` 字符串判断的探测策略，新模型无需改码即可被识别。第三，`QuantConfig` 用九个量化常量（`fp16` 默认、`fp8`、`mxfp8`、`nvfp4`、`int4_awq`、`int4_awq_modelopt`、`int4_gptq`、`int8_sq`、`mixed_precision`）描述格式，`_parse_quant` 按「外部 `hf_quant_config.json` 优先于内嵌 `quantization_config`」判定，混合精度取出现最多的 algo 为主导类型、逐层例外写入 `layer_overrides`，并用真实权重事实修正 `excluded` 列表。`module_quant_type` 是「某个 Linear 最终用什么精度」的唯一真相来源，被 `make_linear` 与 ONNX 导出校验共同消费。核心结论：`ModelConfig` 把 HF 散落各处的字段摊平成统一结构，换取下游只认 `ModelConfig`、不必关心模型族的统一性；解析阶段允许读权重形状元数据但不加载张量数据。
