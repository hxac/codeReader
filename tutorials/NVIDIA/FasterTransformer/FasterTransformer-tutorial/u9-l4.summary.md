本讲承接 u9-l2，讲清 FT 为何在官方 CUTLASS 之上维护 `cutlass_extensions`：补齐 cuBLAS 做不到的混合精度反量化 GEMM（weight-only）、分组持久化 GEMM（MoE）、可定制融合 epilogue（int8 缩放）三类能力。核心模块是 MoE GEMM 与 int8 GEMM。MoE 的难点是每个 expert 分到的 token 行数运行时才确定、且各不相同；解法是 grouped GEMM：用前缀和数组 `total_rows_before_expert` 作「问题描述符」，所有专家共用一份连续输入/输出，由一个持久化 kernel 一次启动，`MoeProblemVisitor` 在设备端（`GroupScheduleMode::kDeviceOnly`）用 warp 内前缀和把 tile 跨专家负载均衡地分给线程块。这把 E 次启动合并为一次并避免小专家导致 SM 空闲，代价是禁用 split-k。fc1 用 `moe_gemm_bias_act`（带偏置+激活），fc2 用 `moe_gemm`（无偏置）。int8 GEMM 是 w8a8 路径（s8×s8→T、int32 累加、`EpilogueVisitorPerRowPerCol` 融合 alpha 缩放，仅 sm80、仅 TN），与 MoE 共用同一份 `get_candidate_configs`/`estimate_best_config_from_occupancies` 启发式，差别仅在 `num_experts`（MoE 为真实数、int8 为 1）与 `split_k_limit`（MoE=1、int8=7）。两路径分别由 `BUILD_CUTLASS_MOE`、`BUILD_CUTLASS_MIXED_GEMM`（默认 ON）守卫，因 CUTLASS 模板实例化极重，用「一个 `.cu` 一个类型特化」控制编译爆炸。关键术语：cutlass_extensions、grouped/persistent GEMM、MoeProblemVisitor、total_rows_before_expert、Mma/Epilogue、w8a8、epilogue visitor、occupancy 启发式。
