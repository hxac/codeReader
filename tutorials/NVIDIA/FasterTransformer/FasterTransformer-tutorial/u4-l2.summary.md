本讲专门回答「`Bert::forward` 到底处理了多少 token」，核心是 Effective FasterTransformer 如何去除 padding 带来的无效计算。批处理时短句被补零到 batch 内最长句长度 S，padding 位置的 GEMM 全是浪费，收益正比于平均长度 s̄ 与最大长度 S 之比；线性层工作量可降到 s̄/S。

核心思想是 **compact/uncompact**：进 BERT 前用 `invokeRemovePadding` 把有效 token 挤成紧凑缓冲 `[M_eff, H]`（M_eff=Σs_i），跑完 N 层再用 `invokeRebuildPadding` 散回 `[B,S,H]`。两者共享一张 `padding_offset` 表，由 `getPaddingOffsetAndCuSeqLensKernel` 用单线程串行跑 `O(B)` 前缀和构造，并通过 pinned 内存 + CPU 自旋把有效 token 数回传 host——这是整个流程唯一的 CPU↔GPU 同步点，因后续 GEMM 的 M 维运行期才确定。

去 padding 后 attention 边界信息丢失，需另传两种表：朴素 attention 用 `invokeBuildEncoderAttentionMask` 生成 `[B,1,S,S]` 的 0/1 mask；TensorRT 融合 MHA 不读 mask，改用 `invokeGetTrtPaddingOffset` 生成的累积长度数组，compact 时为 `[B+1]`、padded 时为 `[2B+1]`。

v5.0 起这些操作全部封装进 `Bert::forward` 的 `switch(attention_type)`，去 padding 在层循环前、恢复在层循环后各做一次，中间各层对 padding 无感知。`AttentionType` 是「FUSED/UNFUSED」与「是否带 PADDED」的正交组合，由 `getAttentionType` 自动推导，两维相互独立，且 fused 不可用时会降级为 unfused 但保留 padding 决策。
