本讲梳理 CUTLASS 仓库的目录地图，结论是「目录名≈命名空间≈职责」。仓库顶层分两块：`include/` 是 header-only 库本体（唯一需加入编译 include path 的目录），其余 `examples/`、`tools/`、`test/`、`python/`、`docs/` 围绕它服务。库本体内部并列两套核心库：`include/cutlass/`（经典库，`cutlass::` 命名空间）按「计算阶段」与「抽象层次」交叉组织——`gemm/conv/epilogue/layout/transform` 对应阶段，`arch/thread/warp/threadblock/device/kernel/collective` 对应层次，是 GEMM 四层分解的文件镜像；`include/cute/`（3.x 引入的 CuTe 布局代数，`cute::` 命名空间）以 `Layout`（形状+步长的纯函数）、`Tensor`（Layout 加数据指针）、`Atom`（硬件指令封装）三大抽象把算法与硬件解耦，子目录分 `algorithm/`（copy/gemm 等通用算法）、`arch/`（裸 PTX）、`atom/`（指令 traits）。关键判断：3.x 起 `cutlass::gemm` 已反向 `#include "cute/layout.hpp"`，即 cutlass 建在 cute 之上，cute 是更底层地基；且 `cute/arch/` 文件名直接编码架构版本（sm70/sm80/sm90/sm100）。辅助目录中，`examples/`（约上百个，手写可读教材）、`tools/library/`（由 `python/cutlass_library` 批量生成的实例库，供 `cutlass_profiler` 调用）、`test/unit/`（Google Test，目标 `test_unit`）构成「学—测—验」三件套。
