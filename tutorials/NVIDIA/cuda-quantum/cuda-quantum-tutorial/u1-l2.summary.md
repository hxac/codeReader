本讲是后续所有讲义的「目录地图」，核心结论是：CUDA-Q 仓库由根 `CMakeLists.txt` 的 `add_subdirectory` 拼装出四大子系统——`cudaq/`（编译器）、`runtime/`（C++ 运行时与 nvqir 模拟器，是公共底盘）、`python/`（Python 前端）、`realtime/`（FPGA/GPU 实时控制），其中前两者受 `CUDAQ_ENABLE_PROJECTS` 控制、按需开关，测试目录则受 `CUDAQ_BUILD_TESTS` 控制。本讲引入的关键概念与导航锚点包括：`cudaq/` 采用标准 MLIR 布局（`include/` 声明、`lib/` 实现、`tools/` 工具、`test/` 回归测试），方言用 TableGen（`.td`）声明，所有变换 Pass 集中在 `lib/Optimizer/Transforms/`（约 68 个），工具链为 `cudaq-quake`→`cudaq-opt`→`cudaq-translate`，由 `nvq++` bash 脚本编排；`runtime/` 分 `common/`（公共类型）、`cudaq/`（qis、算法原语 sample/observe、平台 platform）、`nvqir/`（模拟器后端，后端切换是链接期任务，继承自 `CircuitSimulator`）；Python 前端经 `kernel/ast_bridge.py` 将 Python 函数翻成 Quake，与 C++ 端 `ASTBridge.cpp` 镜像合流；示例在 `docs/sphinx/examples/`，测试分三类：`cudaq/test/`（FileCheck 编译器测试）、`unittests/`（GoogleTest 运行时）、`targettests/`（跨目标端到端，含众多硬件厂商目录）。基本功是先用「四象限」判断文件归属子系统，再判断其处于 include/lib/tools/test 哪一层。
