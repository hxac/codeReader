# MLIR 基础与 CUDA-Q 编译总览

## 1. 本讲目标

本讲是「编译器」单元（单元 4）的入口。前面几讲你已经学会写 `__qpu__` 内核（u1-l4）和 `@cudaq.kernel` 内核（u1-l5），并用 `cudaq::sample` 把它们跑起来。但「C++ 源码是怎么变成可在模拟器或 QPU 上执行的东西的」这一段一直是个黑盒。

学完本讲，你应当能够：

1. 用自己的话解释 MLIR 的**方言（Dialect）、操作（Operation）、Pass、Pipeline** 四个核心概念，并说出它们在 CUDA-Q 里的具体体现。
2. 画出 CUDA-Q 从 C++ 源码到可执行文件的**完整编译流水线**，标注每个阶段的产物（Quake MLIR → 优化后的 MLIR → QIR/LLVM IR → 目标代码）落在哪个工具里。
3. 实际动手用 `cudaq-quake` / `cudaq-opt` / `cudaq-translate` 三个工具**观察和变换中间 IR**，而不再只依赖 `nvq++` 一键编译。

本讲只做「全景总览」与「工具入门」，不深入任何一个 Pass 的实现细节——那是 u4-l2 ~ u4-l7 的任务。

## 2. 前置知识

在进入 MLIR 之前，先用三段话把背景补齐。

**什么是中间表示（IR）。** 编译器通常不是把源码一步翻译成机器码，而是先翻译成某种「中间表示」，再在中间表示上做分析和优化，最后才生成目标代码。LLVM IR 就是最著名的中间表示之一。把源码 → IR → 目标代码这条链路切成多层，可以让每层只关注一类问题，降低编译器复杂度。

**什么是 MLIR。** MLIR（Multi-Level Intermediate Representation）是 LLVM 项目推出的「元 IR 框架」。它的关键设计是：**不规定一种固定的 IR，而是允许你定义自己的「方言」**。不同抽象层级可以用不同方言来表示，再通过「lowering」一层层往下降。例如 CUDA-Q 就在 MLIR 上自定义了量子方言 Quake 和经典计算方言 CC，最终降到 LLVM IR。

**CUDA-Q 的两个前端共享同一条后端链路。** 无论你用 C++（`__qpu__`）还是 Python（`@cudaq.kernel`）写内核，最终都会变成同一种 Quake MLIR，再走同一条优化与 lowering 流水线（Python 前端在 u5 单元细讲）。所以本讲虽然是「编译总览」，对两套前端的读者都适用。

如果你对「内核是什么、`cudaq::sample` 怎么用」已经陌生，建议先回顾 u1-l4 与 u1-l5。

## 4. 核心概念与源码讲解

本讲拆成三个最小模块：**MLIR 基础概念**、**CUDA-Q 编译流水线**、**中间产物观察工具**。

### 4.1 MLIR 基础：方言、操作、Pass、Pipeline

#### 4.1.1 概念说明

MLIR 用四个抽象贯穿整个编译流程：

- **方言（Dialect）**：一组操作、类型、属性的「命名空间」。可以理解为「某一种 IR 的方言变体」。CUDA-Q 自带两个核心方言：建模量子计算的 **Quake**，和建模 C++ 经典计算的 **CC**。此外还会用到 MLIR 自带的 `func`（函数）、`arith`（算术）、`cf`（控制流）、`LLVM`（LLVM IR 对接）等标准方言。每个操作都带方言前缀，例如 `quake.h`、`cc.loop`、`arith.constant`。
- **操作（Operation）**：IR 的最小组成单元，大致相当于「一条指令 + 它的输入输出 + 它的元信息」。一段 Quake 代码 `%q0 = quake.alloca` 里的 `quake.alloca` 就是一个操作。
- **Pass（通道）**：一段「输入 IR → 输出 IR」的变换程序。可以是化简（如 `canonicalize`）、内联（如 `aggressive-inlining`）、循环展开（如 `loop-unroll`）等。Pass 是编译器做实际工作的单位。
- **Pipeline（流水线）**：一组按顺序执行的 Pass。Pipeline 既可以用文本写（如 `--pass-pipeline="builtin.module(canonicalize,cse)"`），也可以在 C++ 里用 `PassManager` 组装。

MLIR 的核心思想是**渐进式 lowering（progressive lowering）**：一份代码从高层抽象（贴近源语言）逐层降到低层抽象（贴近机器），每一层用合适的方言表示。CUDA-Q 的典型降层路径是：

\[ \text{C++ AST} \;\longrightarrow\; \text{Quake + CC MLIR} \;\longrightarrow\; \text{QIR / LLVM IR} \;\longrightarrow\; \text{目标代码} \]

抽象层级逐级降低，每一层只解决自己那一段的问题。

#### 4.1.2 核心流程

MLIR 程序的执行（编译）流程可以概括为：

1. **解析**：把文本或 AST 变成一个 `ModuleOp`（整个编译单元的根操作），里面是一棵操作树。
2. **注册方言**：在 `MLIRContext` 里登记所有用到的方言，否则操作无法被识别。
3. **跑 Pipeline**：`PassManager` 按 Pass 顺序遍历并改写 IR；每个 Pass 看到的都是「合法的 IR」，输出也必须是合法 IR。
4. **降低 / 输出**：最终一层 lowering 把 IR 翻译成 LLVM IR 或其他目标格式。

关键点：**Pass 之间通过 IR 解耦**。一个 Pass 不需要知道前一个 Pass 的内部状态，只看当前的 IR。这就是为什么 CUDA-Q 能把几十个 Pass 自由组合成不同目标的流水线。

#### 4.1.3 源码精读

CUDA-Q 的方言定义在 TableGen（`.td`）文件里，而 `Overview.md` 在最开头就点明了 MLIR 编译器的定位：

> The compiler maps the C++ abstract syntax tree (AST) generated by Clang to internal MLIR dialects for quantum and classical computing. These dialects are further lowered to LLVM adherent to the Quantum Intermediate Representation (QIR) specification.

这段话对应 [Overview.md:L14-L18](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/Overview.md#L14-L18)，中文意思即「编译器把 Clang 生成的 C++ AST 映射到内部 MLIR 方言（Quake/CC），再进一步降低到符合 QIR 规范的 LLVM IR」。

Quake 方言同时支持两种比特建模方式，[Overview.md:L20-L27](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/Overview.md#L20-L27) 说明：内存语义（memory-semantics，量子操作是对比特引用的副作用）与值语义（value-semantics，量子操作消费并产生新的比特值）。两种语义的区别是 Quake 设计的核心，本讲先建立印象，细节留到 u4-l2。

至于「Pass 串成 Pipeline」的真实代码，`Pipelines.cpp` 里大量出现 `pm.addNestedPass<func::FuncOp>(...)` 与 `pm.addPass(...)`，每一行就是把一个 Pass 追加到流水线。例如 AOT（预先编译）入口：

```cpp
void cudaq::opt::addAOTPipelineConvertToQIR(PassManager &pm,
                                            StringRef convertTo) {
  if (convertTo.empty())
    convertTo = "qir";
  ::createTargetCodegenPipeline</*JIT=*/false>(pm, convertTo);
}
```

见 [cudaq/lib/Optimizer/CodeGen/Pipelines.cpp:L137-L142](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/lib/Optimizer/CodeGen/Pipelines.cpp#L137-L142)。这里 `pm` 就是 MLIR 的 `PassManager`，`addAOTPipelineConvertToQIR` 是「把 Quake 降到 QIR」的整条 AOT 流水线入口。

#### 4.1.4 代码实践

**目标**：亲眼看到「一段 C++ 被翻译成带方言前缀的操作」，建立「方言/操作」的直观感受。

**步骤**：

1. 准备一个最小内核（示例代码，可自行命名 `hello.cpp`）：

   ```cpp
   // 示例代码
   #include <cudaq.h>
   struct hello {
     void operator()() __qpu__ {
       cudaq::qubit q;
       h(q);
       mz(q);
     }
   };
   ```

2. 用 `cudaq-quake` 把它翻译成 Quake MLIR（关于该工具的细节见 4.3）：

   ```bash
   cudaq-quake hello.cpp -o hello.qke
   cat hello.qke
   ```

3. 在输出里找出带 `quake.` 前缀的操作（如 `quake.alloca`、`quake.h`、`quake.mz`）和带 `cc.` / `func.` / `arith.` 前缀的操作。

**需要观察的现象**：`hello.qke` 是一段可读的文本 IR，每个操作都以「方言名.操作名」开头；`ModuleOp` 是顶层容器，内核是一个 `func.func`。

**预期结果**：你能指出哪些操作属于 Quake 方言（量子部分）、哪些属于 CC/func/arith 方言（经典部分）。

> 如果环境中没有可运行的 `cudaq-quake`，本步骤可改为「源码阅读型实践」：阅读 [cudaq/tools/cudaq-quake/cudaq-quake.cpp:L504-L520](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-quake/cudaq-quake.cpp#L504-L520)，确认翻译完成后会跑 `VerifierPass`、`Canonicalizer`、`SymbolDCE` 三个 Pass，并标注「待本地验证」。

#### 4.1.5 小练习与答案

**练习 1**：下列四个概念，哪些是「变换 IR 的程序」？哪些是「IR 的命名空间」？
(a) Dialect　(b) Operation　(c) Pass　(d) Pipeline

**答案**：(c) Pass 是「变换 IR 的程序」；(d) Pipeline 是「一串 Pass 的有序组合」，也属于变换范畴；(a) Dialect 是「IR 的命名空间」；(b) Operation 是「IR 的组成单元」，不是命名空间也不是变换程序。

**练习 2**：为什么 CUDA-Q 要自定义 Quake 方言，而不是直接用 LLVM IR 来表达量子计算？

**答案**：LLVM IR 是面向经典 CPU 的低层 IR，没有「量子比特」「量子门」「测量」这种一等概念；如果在 LLVM IR 层做量子优化（比如消除相邻的两个 H 门），会因为信息丢失而极难正确实现（详见 4.2.2 的反例）。Quake 在更高抽象层保留量子语义，便于做专门的量子优化，最后再 lowering 到 LLVM IR。

### 4.2 CUDA-Q 编译流水线：从 C++ AST 到 QIR

#### 4.2.1 概念说明

CUDA-Q 的编译器是一个 **MLIR 编译器**，但它不只有一个工具，而是被故意拆成了几个可独立使用的小工具，由 `nvq++` 脚本编排。`Overview.md` 把这套设计称为「模块化的编译工作流」。

四个核心角色：

- **ASTBridge**：用 Clang 的 `RecursiveASTVisitor` 遍历 C++ AST，把它翻译成 Quake/CC MLIR。它是「C++ → MLIR」的桥。
- **cudaq-quake**：承载 ASTBridge 的工具，输入 `.cpp`，输出 Quake MLIR（默认）或 LLVM IR（`--emit-llvm-file`）。
- **cudaq-opt**：读入 Quake MLIR，跑用户指定的 Pass/Pipeline，输出变换后的 Quake MLIR。
- **cudaq-translate**：把 Quake MLIR 降低（lowering）到目标「传输层格式」，主要是 QIR/LLVM IR，也支持 OpenQASM2 与 IQM JSON。
- **nvq++**：一个 bash 脚本，把上述工具按正确顺序与参数串起来，最终产出可执行文件。

一个关键名词：**QIR（Quantum Intermediate Representation）** 是基于 LLVM IR 的开放量子中间表示规范。CUDA-Q 把 Quake lowering 到符合 QIR 规范的 LLVM IR，这样就能复用 LLVM 后端生成目标代码，也能让模拟器（`libnvqir`）按统一接口执行。

#### 4.2.2 核心流程

`Overview.md` 把 `nvq++` 的工作流明确写成四步：

1. 通过 Clang `ASTConsumer` 把 CUDA-Q 的 C++ 内核映射成 Quake MLIR；
2. 把所有 Quake 内核**注册到运行时**，供量子 IR 内省；
3. **改写**原始 C++ 内核入口函数，让它调用一个内部的运行时内核启动函数，从而命中指定的 `quantum_platform`；
4. lowering 到 QIR 并链接，产出可执行文件或目标代码。

对应 [Overview.md:L152-L159](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/Overview.md#L152-L159)。用一张数据流图表示：

```text
   hello.cpp  ──[1] ASTBridge ──▶  Quake MLIR (.qke)
                                       │
                                       │ [2] 注册内核到运行时
                                       │ [3] 改写入口为 launch 函数
                                       ▼
                                  cudaq-opt (可选 Pass)
                                       │
                                       ▼
   cudaq-translate ─[4] lowering─▶  QIR / LLVM IR (.ll)
                                       │
                                       ▼
                            LLC + 链接 ▶  可执行文件
```

**为什么要这样拆？** 因为 lowering 是有损的：一旦降到 QIR/LLVM IR，量子优化的机会就大幅减少。所以 CUDA-Q 把「能在 Quake 层做的优化」尽量放在前面，把 lowering 放在最后。这也解释了为什么 Quake 要支持「值语义」——为了让优化器能看清数据依赖。

`quake-dialect.md` 用一个反例说明了这一点：假如在内存语义下，同一个比特 `%q0` 上先后施加两个 H 门，中间夹一个对整个 `%veq`（包含 `%q0`）的测量，朴素的「相邻 H∘H 消去」优化会**错误地**把两个 H 删掉，因为它没察觉到中间的测量。对应原文见 [docs/sphinx/specification/quake-dialect.md:L62-L80](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/docs/sphinx/specification/quake-dialect.md#L62-L80)。

用数学式表达这个错误：设两个 H 之间有一段含测量的演化 \(M\)，正确的复合作用是

\[ H \cdot M \cdot H \neq I \]

而朴素优化会把首尾两个 H 当作 \(H\cdot H = I\) 直接消去，丢掉了中间的 \(M\)。值语义通过让每段操作产出**新的、独立命名的值**，把数据依赖显式化，从而避免这类错误。

#### 4.2.3 源码精读

四步工作流在 `nvq++` 脚本里有非常具体的对应。下面三行是每个 `.cpp` 源文件实际经历的三次工具调用：

```bash
# [1] C++ → Quake MLIR (+ 经典 LLVM IR)
run ${TOOLBIN}cudaq-quake ... --emit-llvm-file $i -o ${file}.qke

# 可选：在 Quake 层跑优化 Pass
run ${TOOLBIN}cudaq-opt ... --pass-pipeline="${OPT_PASSES}" ${QUAKE_IN} -o ${DCL_FILE}

# [4] Quake MLIR → QIR/LLVM IR
run ${TOOLBIN}cudaq-translate ... --convert-to=${PLATFORM_TRANSPORT_LAYER} ${QUAKE_IN} -o ${QUAKELL_FILE}
```

分别见 [cudaq/tools/nvqpp/nvq++.in:L916](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/nvqpp/nvq++.in#L916)、[cudaq/tools/nvqpp/nvq++.in:L925](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/nvqpp/nvq++.in#L925)、[cudaq/tools/nvqpp/nvq++.in:L932](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/nvqpp/nvq++.in#L932)。注意 `${file}.qke` 就是 Quake MLIR 文件，`.ll` 是 LLVM IR 文本文件，`--convert-to=${PLATFORM_TRANSPORT_LAYER}` 决定 lowering 到哪种 QIR profile。

lowering 阶段真实的 Pass 顺序写在 `Pipelines.cpp`。QIR 有四种 profile：`qir` / `qir-full`（完整 QIR）、`qir-base`（Base Profile）、`qir-adaptive`（Adaptive Profile）。分流逻辑在 `addQIRConversionPipeline`：

```cpp
if (convertFields.first == "qir" || convertFields.first == "qir-full") {
  cudaq::opt::addConvertToQIRAPIPipeline(pm, "full:" + ...);
} else if (convertFields.first == "qir-base") {
  pm.addNestedPass<func::FuncOp>(cudaq::opt::createDelayMeasurements());
  cudaq::opt::addConvertToQIRAPIPipeline(pm, "base-profile:" + ...);
} else if (convertFields.first == "qir-adaptive") {
  cudaq::opt::addConvertToQIRAPIPipeline(pm, "adaptive-profile:" + ...);
}
```

见 [cudaq/lib/Optimizer/CodeGen/Pipelines.cpp:L31-L47](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/lib/Optimizer/CodeGen/Pipelines.cpp#L31-L47)。可以看到不同 profile 走不同的 Pass 组合，例如 `qir-base` 会先 `DelayMeasurements`（把测量延后到线路末端，Base Profile 的要求）。

在进入 QIR lowering 之前，还有一段公共的「目标代码生成」预处理 `createCommonTargetCodegenPipeline`，依次做：应用控制取反 → 激进内联 → canonicalize → 展开 unwind → 展开测量 → 经典 mem2reg → canonicalize/CSE → 添加 dealloc → 循环归一化与展开 → 再次内联……见 [cudaq/lib/Optimizer/CodeGen/Pipelines.cpp:L49-L95](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/lib/Optimizer/CodeGen/Pipelines.cpp#L49-L95)。本讲不需要理解每个 Pass，只要建立一个印象：**lowering 不是一步完成的，而是几十个 Pass 排成的一条长流水线**。每个 Pass 的细节会分散在 u4-l6（优化 Pass）与 u4-l7（CodeGen）讲。

> 关于运行时与模拟器：lowering 出来的 QIR 最终由 `libnvqir` 执行，它实现 QIR 规范定义的量子指令集（QIS）并委托给可扩展的 `CircuitSimulator` API；切换后端模拟器是链接期任务，由 `nvq++` 隐式处理（见 [Overview.md:L96-L102](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/Overview.md#L96-L102)）。这部分在单元 6 详讲。

#### 4.2.4 代码实践

**目标**：对照四步工作流，跟踪一个真实示例 `static_kernel.cpp` 的编译过程。

**步骤**：

1. 阅读 [docs/sphinx/examples/cpp/basics/static_kernel.cpp:L1-L40](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/docs/sphinx/examples/cpp/basics/static_kernel.cpp#L1-L40)，确认这是一个 GHZ 内核：先对 `q[0]` 加 H，再用循环施加一连串受控非门 `x<cudaq::ctrl>(q[i], q[i+1])`，最后测量。

2. 对照 4.2.2 的数据流图，把这个内核的编译过程「拆」成四步，并在每一步旁标注「发生在哪个工具里 / 产物是什么」。

**需要观察的现象**：你会注意到这个内核里有**经典循环**（`for (int i = ...)`) 和**量子门**两类逻辑，它们在 Quake MLIR 里会分别落到 CC 方言与 Quake 方言。

**预期结果**：能口述「`for` 循环由 CC 方言建模，循环展开后变成一串 `quake.x` 受控门，最后 lowering 到 QIR」。**待本地验证**：具体 IR 形态留到 4.3 实践里亲眼确认。

#### 4.2.5 小练习与答案

**练习 1**：`nvq++` 四步工作流里，哪一步把 C++ 内核入口改写成「调用运行时 launch 函数」？这一步为什么必要？

**答案**：第三步。原始内核函数体是「量子逻辑」，不能直接在经典 CPU 上调用；为了让宿主代码能像调用普通函数一样触发内核、并命中指定的 `quantum_platform`（模拟器或远程 QPU），必须把入口改写成调用运行时的内核启动函数。

**练习 2**：为什么 lowering（到 QIR）要放在流水线最后，而不是最前？

**答案**：lowering 是有损降层，降到 LLVM IR 后量子结构与数据依赖被抹平，量子专属优化（如门消去、受控门分解）就难以再做。把 lowering 放最后，意味着所有能在 Quake 高层做的优化都先做完，最大化保留了优化机会。

**练习 3**：`qir-base` 与 `qir-full` 的 lowering 流水线有一个明显不同的 Pass，是哪一个？

**答案**：`qir-base` 多了 `createDelayMeasurements()`（延迟测量），因为 Base Profile 要求所有测量集中在线路末端；完整 QIR 没有这个限制。见 [cudaq/lib/Optimizer/CodeGen/Pipelines.cpp:L36-L39](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/lib/Optimizer/CodeGen/Pipelines.cpp#L36-L39)。

### 4.3 中间产物观察工具：cudaq-quake / cudaq-opt / cudaq-translate

#### 4.3.1 概念说明

`nvq++` 是「一键编译」，但它把中间产物都藏起来了。当你想调试编译过程、理解某个优化、或排查「为什么我的内核被编译成了这样」时，就需要直接用底层三个工具。三者职责清晰分工：

- `cudaq-quake`：**前端**。输入 C++，输出 Quake MLIR。它内部驱动 Clang，跑 ASTBridge。
- `cudaq-opt`：**中端变换器**。输入 Quake MLIR，输出（变换后的）Quake MLIR。它是「调试单个 Pass」的主要工具。
- `cudaq-translate`：**后端翻译器**。输入 Quake MLIR，输出 QIR / OpenQASM2 / IQM JSON 等目标格式。

这三个工具是「观察中间产物」的最直接窗口，也是后续 u4-l2 ~ u4-l7 反复要用的实验台。

#### 4.3.2 核心流程

三者的协作链路：

```text
  .cpp ──cudaq-quake──▶ .qke (Quake MLIR)
                            │
                            ├──cudaq-opt──▶ .qke' (优化后的 Quake MLIR)
                            │
                            └──cudaq-translate --convert-to=qir ──▶ .ll (LLVM IR)
```

每个工具的常用入口与关键选项（均来自源码，可直接对照）：

- `cudaq-quake` 的选项定义在 [cudaq/tools/cudaq-quake/cudaq-quake.cpp:L60-L123](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-quake/cudaq-quake.cpp#L60-L123)：`-o`（输出文件）、`--filter=<名字>`（只转换指定内核）、`--emit-llvm-file`（同时输出经典部分的 LLVM IR）、`--no-simplify`（关闭桥接后的化简 Pass）、`--ast-dump`（打印 Clang AST）、`-I/-D/-Xcudaq`（透传给 Clang）、`--std`（默认 c++20）。
- `cudaq-opt` 的实现在 [cudaq/tools/cudaq-opt/cudaq-opt.cpp:L55-L86](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-opt/cudaq-opt.cpp#L55-L86)。它复用 MLIR 的 `MlirOptMain`，并调用 `cudaq::registerAllPasses()` 注册所有 CUDA-Q 专属 Pass，因此既支持 `--canonicalize`、`--cse` 这类单 Pass，也支持 `--pass-pipeline="..."`。
- `cudaq-translate` 的 `--convert-to` 选项见 [cudaq/tools/cudaq-translate/cudaq-translate.cpp:L67-L81](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-translate/cudaq-translate.cpp#L67-L81)，取值为 `qir` / `qir-full` / `qir-adaptive` / `qir-base` / `openqasm2` / `iqm`，默认 `qir:0.1`。

`cudaq-quake` 在翻译完成后，默认会跑一段小流水线（验证 + canonicalize + 死符号消除）来清理 ASTBridge 留下的冗余，见 [cudaq/tools/cudaq-quake/cudaq-quake.cpp:L504-L520](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-quake/cudaq-quake.cpp#L504-L520)。这意味着默认输出的 `.qke` 已经是「清理过一遍」的 Quake；想看「最原始」的桥接输出，需要加 `--no-simplify`。这是本讲综合实践的关键。

#### 4.3.3 源码精读

`cudaq-quake` 的核心是把 C++ 代码喂给 Clang 并触发 ASTBridge。它的 `main` 根据选项选择三种「动作」之一：

```cpp
if (auto rc = emitLLVM
                  ? runTool<CudaQAction>(module, mangledKernelNameMap, ...)
                  : (llvmOnly ? runTool<InterceptCudaQAction>(...)
                              : runTool<cudaq::ASTBridgeAction>(...)))
  return rc;
```

见 [cudaq/tools/cudaq-quake/cudaq-quake.cpp:L473-L482](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-quake/cudaq-quake.cpp#L473-L482)。默认（不带 emit 开关）走 `ASTBridgeAction`，只产出 Quake MLIR；`--emit-llvm-file` 走 `CudaQAction`，同时产出经典代码的 LLVM IR（这是 `nvq++` 实际用的模式，对应 nvq++.in:L916 的 `--emit-llvm-file`）。

`cudaq-translate` 则根据 `--convert-to` 的值，用 `StringSwitch` 选择对应的 lowering 路径：

```cpp
llvm::StringSwitch<std::function<void()>>(convertPair.first)
    .Cases({"qir", "qir-full", "qir-adaptive", "qir-base"},
           [&]() {
             cudaq::opt::addAggressiveInlining(pm);
             cudaq::opt::createTargetFinalizePipeline(pm);
             cudaq::opt::addAOTPipelineConvertToQIR(pm, convertValue);
           })
    .Case("openqasm2", [&]() { ... cudaq::opt::addPipelineTranslateToOpenQASM(pm); ... })
    .Case("iqm",       [&]() { ... cudaq::opt::addPipelineTranslateToIQMJson(pm); ... })
    ...
```

见 [cudaq/tools/cudaq-translate/cudaq-translate.cpp:L179-L204](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-translate/cudaq-translate.cpp#L179-L204)。可以看到 lowering 到 QIR 会先做 `AggressiveInlining` 和 `TargetFinalizePipeline`，再进入 `addAOTPipelineConvertToQIR`（即 4.2 里那条长流水线）；最后再 `translateModuleToLLVMIR` 把 LLVM-IR 方言翻译成真正的 LLVM IR，见 [cudaq/tools/cudaq-translate/cudaq-translate.cpp:L228-L231](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-translate/cudaq-translate.cpp#L228-L231)。

#### 4.3.4 代码实践

**目标**：用三个工具手动跑一遍编译链路，并用 `cudaq-opt --canonicalize` 观察 Pass 前后的 IR 变化。

**步骤**：

1. 准备示例 `ghz.cpp`，直接复用仓库示例 [docs/sphinx/examples/cpp/basics/static_kernel.cpp](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/docs/sphinx/examples/cpp/basics/static_kernel.cpp)（GHZ 内核）。

2. **生成「最原始」的 Quake**（关闭桥接后的化简）：

   ```bash
   cudaq-quake --no-simplify ghz.cpp -o ghz.raw.qke
   ```

3. **生成「默认清理过」的 Quake**（不带 `--no-simplify`）：

   ```bash
   cudaq-quake ghz.cpp -o ghz.simple.qke
   ```

4. **对比两者**，观察 `cudaq-quake` 内置的 canonicalize + SymbolDCE 做了什么清理：

   ```bash
   diff ghz.raw.qke ghz.simple.qke
   ```

5. **用 `cudaq-opt` 显式跑 `--canonicalize`**，把原始 IR 喂给 Pass：

   ```bash
   cudaq-opt --canonicalize ghz.raw.qke -o ghz.opt.qke
   diff ghz.simple.qke ghz.opt.qke
   ```

6. （可选）继续 lowering 到 QIR 看看：

   ```bash
   cudaq-translate --convert-to=qir ghz.simple.qke -o ghz.ll
   head -n 40 ghz.ll
   ```

**需要观察的现象**：
- `ghz.raw.qke` 里能找到 `for` 循环对应的 CC 控制结构与逐个 `quake.x` 受控门。
- 步骤 4 的 diff 应当显示清理 Pass 消除了一些临时值或死符号（变化可能很小，取决于内核）。
- 步骤 5 中 `cudaq-opt --canonicalize` 的输出，与 `cudaq-quake` 默认输出（已含一次 canonicalize）应当非常接近——这正好印证 4.3.2 所说「`cudaq-quake` 默认就跑了一次 canonicalize」。
- 步骤 6 的 `ghz.ll` 里会出现 `__quantum__qis__h`、`__quantum__qis__cnot` 之类的 QIR 函数调用。

**预期结果**：你能画出 `ghz.cpp → ghz.raw.qke → ghz.opt.qke → ghz.ll` 这条链路，并指出每一步发生在哪个工具、跑了什么 Pass。

> **重要**：不同内核在 `--canonicalize` 下的可见变化差别很大。如果 diff 几乎为空，说明该内核的可化简冗余很少，这本身就是一个有效结论。若想看到更明显的变化，可在内核里刻意引入冗余（例如连续两次 `h(q)`），再重跑步骤 5 对比。**待本地验证**：本实践需要已构建好的 CUDA-Q 工具链（参考 u1-l3）。

#### 4.3.5 小练习与答案

**练习 1**：`cudaq-quake`、`cudaq-opt`、`cudaq-translate` 三者的输入输出类型分别是什么？

**答案**：`cudaq-quake` 输入 C++（`.cpp`），输出 Quake MLIR（`.qke`，可选附加 LLVM IR）；`cudaq-opt` 输入 Quake MLIR，输出 Quake MLIR；`cudaq-translate` 输入 Quake MLIR，输出 QIR/LLVM IR（`.ll`）或 OpenQASM2 / IQM JSON。

**练习 2**：为什么 `cudaq-quake` 默认就要跑一次 `canonicalize` + `SymbolDCE`？怎样关掉它？

**答案**：ASTBridge 在翻译过程中会产生一些临时中间值和未使用的符号，默认跑这两个 Pass 是为了输出干净的 Quake 给下游。加 `--no-simplify` 可关闭，得到「最原始」的桥接输出，便于调试桥接本身。见 [cudaq/tools/cudaq-quake/cudaq-quake.cpp:L504-L510](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-quake/cudaq-quake.cpp#L504-L510)。

**练习 3**：`cudaq-translate --convert-to=openqasm2` 与 `--convert-to=qir` 走的是同一条 lowering 流水线吗？

**答案**：不是。源码里 `qir`/`qir-full`/`qir-adaptive`/`qir-base` 走 `addAOTPipelineConvertToQIR`，而 `openqasm2` 走 `createTargetFinalizePipeline` + `addPipelineTranslateToOpenQASM`，`iqm` 又是另一条。见 [cudaq/tools/cudaq-translate/cudaq-translate.cpp:L179-L204](https://github.com/NVIDIA/cuda-quantum/blob/61face2b9a41d1ef9b6ea6e7941bc22441c75ab9/cudaq/tools/cudaq-translate/cudaq-translate.cpp#L179-L204)。

## 6. 本讲小结

- MLIR 的四个核心抽象是**方言、操作、Pass、Pipeline**；CUDA-Q 在 MLIR 上自定义了量子方言 **Quake** 和经典方言 **CC**，并复用 `func`/`arith`/`LLVM` 等标准方言。
- CUDA-Q 编译是一个**渐进式 lowering** 流程：C++ AST → Quake/CC MLIR → QIR/LLVM IR → 目标代码；lowering 故意放在最后，以最大化 Quake 层的优化机会。
- `nvq++` 把编译编排成四步：映射到 Quake、注册内核到运行时、改写入口为 launch、lowering 到 QIR 并链接，对应 `Overview.md` 与 `nvq++` 脚本里的三次工具调用。
- 三个底层工具各司其职：`cudaq-quake`（前端，C++→Quake）、`cudaq-opt`（中端变换，可单跑 Pass）、`cudaq-translate`（后端翻译，Quake→QIR/OpenQASM2/IQM）。
- `cudaq-quake` 默认就会跑一次 `canonicalize + SymbolDCE` 清理输出，加 `--no-simplify` 可看到最原始的桥接结果——这是调试编译过程的入口。
- Quake 同时支持**内存语义**与**值语义**两种比特建模；值语义把数据依赖显式化，是后续量子优化的基础（细节留到 u4-l2）。

## 7. 下一步学习建议

本讲建立了编译流水线的「全景图」，但每一个具体环节都还是黑盒。建议按以下顺序深入：

1. **u4-l2 Quake 方言：操作与语义**——精读 `QuakeOps.td`，正式学透内存语义与值语义，理解 `alloca`/门/`apply`/测量等操作的定义。
2. **u4-l3 CC 方言：经典计算抽象**——看清内核里的循环、数组等 C++ 结构在 MLIR 里如何建模。
3. **u4-l4 AST Bridge：从 C++ 到 Quake**——回到 `cudaq-quake` 内部，看 ASTBridge 是怎么逐节点把 C++ 翻译成 Quake/CC 的。
4. **u4-l6 优化 Pass 流水线** 与 **u4-l7 CodeGen：从 Quake 到 QIR/LLVM**——拆开本讲里那条「几十个 Pass 排成长龙」的流水线，逐个理解。
5. 在阅读以上讲义时，**随时用本讲教的三个工具**做实验：每学一个 Pass，就用 `cudaq-opt` 单跑它，对比前后 IR。这是理解编译器最有效的办法。
