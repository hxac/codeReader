## 2. 前置知识

本讲面向零基础读者，但有几个名词先解释清楚，理解起来会更顺：

- **深度学习推理（Inference）**：把训练好的神经网络模型部署到设备上，输入一张图/一段语音，输出分类、检测框等结果。与之相对的是「训练（Training）」。
- **边缘推理（Edge Inference）**：在「靠近数据源」的设备（摄像头、机器人、车载芯片、FPGA 板卡）上做推理，而不是把数据传到云端。边缘推理的核心诉求是**低延迟、低功耗、小体积**。
- **TPU**：Tensor Processing Unit，张量处理器，一种专门为神经网络矩阵运算设计的加速芯片。这里的 **EEP-TPU** 是厂商自研的 TPU 产品线。
- **FPGA**：Field Programmable Gate Array，现场可编程门阵列。一种芯片，其内部逻辑可以在出厂后重新「编程」配置，常用于硬件原型验证和早期产品。
- **ZynqMP**：Xilinx（现 AMD）的 SoC 芯片系列，把 ARM 处理器（PS）和 FPGA 可编程逻辑（PL）集成在同一颗芯片上，本项目的 demo 就是跑在 ZynqMP 上的。
- **精度（FP16 / FP8 / INT8）**：数据用多少位来表示。FP16 = 16 位浮点，INT8 = 8 位整数。位数越低，占用内存越少、计算越快，但精度可能略损。

> 如果你已经熟悉这些词，可以直接跳到第 3 节。

### 4.2 V3+ 核心特性

#### 4.2.1 概念说明

V3+ 的特性被作者归纳成三大类，正好对应边缘推理的三大诉求。理解这三大类的「卖点逻辑」比死记每一条更重要：

| 特性大类 | 解决的核心问题 | 关键词 |
| :--- | :--- | :--- |
| **High Efficiency Computing（高能效）** | 单位功耗算得多 | 可重构架构、Only Take Once、低内存占用 |
| **Low Latency Computing（低延迟）** | 单张图算得快 | Dataflow、混合精度、近存计算、多核多线程 |
| **General Purpose Computing（通用性）** | 能跑的网络多 | CNN/LSTM/RNN、FP16/FP8/INT8、上百种算子、多框架编译器 |

几个对初学者陌生的词先解释：

- **Reconfigurable architecture（可重构架构）**：TPU 内部的计算阵列可以根据网络结构动态配置，从而对不同网络都能高效利用硬件资源。
- **Only Take Once data usage scheme**：字面理解是「数据只取一次」，即尽量让数据在片内只从外部存储搬一次就反复使用，减少昂贵的访存开销。
- **Dataflow architecture（数据流架构）**：让数据像流水线一样在计算单元间流动，前一层的输出直接喂给下一层，减少中间结果的回写。
- **Near memory computing（近存计算）**：把计算单元放在靠近存储的地方，缩短数据搬运距离。
- **Mixing-precision computing（混合精度）**：同一网络里不同层可以用不同精度（如 FP16 与 INT8 混合），在精度和速度之间权衡。

#### 4.2.2 核心流程

可以把三大特性理解为「一个推理请求到达 TPU 后，被这三组技术共同加速」：

```text
输入张量进入 TPU
   │
   ├── 高能效：可重构阵列 + 数据复用 → 用更少功耗完成同样计算
   │
   ├── 低延迟：Dataflow 流水线 + 混合精度 + 多核多线程 → 更快出结果
   │
   └── 通用性：上百种算子 + 多精度 + 多框架编译器 → 能跑各种网络
   │
   ▼
输出结果（分类/检测/分割...）
```

### 4.3 支持的算法与精度

#### 4.3.1 概念说明

这一块回答两个问题：**TPU 能跑哪些网络？用什么精度跑？**

**支持的算法类型**（来自 General Purpose Computing 小节）：

- 网络大类：**CNN、LSTM、RNN**（卷积网络 + 循环网络，覆盖视觉和序列任务）。
- 典型网络：YOLO 全系列（目标检测）、ICNet（语义分割）、MobileNet / Shufflenet（轻量分类）、Resnet（经典分类）、LSTM（序列）。
- 算子：**上百种原生算子**，并支持算子扩展。

**支持的精度**：

- **FP16**（16 位浮点）：精度高，速度中等。
- **FP8**（8 位浮点）：介于两者之间的新精度。
- **INT8**（8 位整数）：速度最快、最省内存，精度略损。

注意一个重要细节：本仓库的**免费 IP 实际只编译支持 FP16 和 INT8**（FP8 是商用 V3+ 架构支持的能力，免费版未必开放）。这一点在 IP 包构成里会写明。

**支持的框架与编译链路**：

- 编译器支持 **Pytorch、Caffe、Darknet、NCNN** 框架。
- 工作模式：用框架训练好的模型 → 经编译器转成 TPU 可执行的 bin → 在板卡上推理。

### 4.4 性能数据解读

#### 4.4.1 概念说明

README 给出了一张性能表，这是**理解 TPU 价值最直观的入口**。先看清表头的前提条件：

> 测试平台是**单核 ASIC 设计**，**1GHz 频率**，算力标称为 **2 TFLOPS（FP16）/ 4 TOPS（INT8）**。

这里有两个数最关键：

- **2 TFLOPS**：每秒 2 万亿次**浮点**运算（FP16 时）。
- **4 TOPS**：每秒 4 万亿次**整数**运算（INT8 时）。

注意 **INT8 的算力（4 TOPS）正好是 FP16（2 TFLOPS）的 2 倍**——这正是 INT8 延迟更低的根本原因之一。

性能表内容如下：

| 算法 | 输入尺寸 | 精度 | 延迟 |
| :---: | :---: | :---: | :---: |
| MobileNetV2-1.0 | 224×224×3 | FP16 | 2.01 ms |
| MobileNetV2-1.0 | 224×224×3 | INT8 | 1.5 ms |
| Resnet50 | 224×224×3 | FP16 | 9.52 ms |
| Resnet50 | 224×224×3 | INT8 | 6.19 ms |
| Yolov5s | 640×640×3 | FP16 | 21.82 ms |
| Yolov5s | 640×640×3 | INT8 | 14.19 ms |

#### 4.4.2 核心流程

**为什么 INT8 比 FP16 快？** 可以从两个角度定量理解：

**角度一：算力翻倍。** 同一颗 1GHz 芯片，INT8 吞吐是 FP16 的 2 倍。粗略地，若一个网络的计算量主要被峰值算力限制，那么：

\[
\text{延迟}_{\text{INT8}} \approx \frac{1}{2} \times \text{延迟}_{\text{FP16}}
\]

以 MobileNetV2 为例：\(2.01 / 2 = 1.005\) ms，实际是 1.5 ms。没到理想的一半，是因为现实中还有**访存、控制、不能完全并行**等开销，但方向是对的——INT8 明显更快。

**角度二：带宽更省。** INT8 每个数只占 1 字节，FP16 占 2 字节。权重和激活值的搬运量减半，对带宽受限的边缘场景尤其友好。

**为什么八核能跑到 0.56ms？** README 在表后补充：

> 「在八核 ASIC 设计里，MobileNetV2-1.0 的 INT8 延迟是 **0.56 ms**。」

单核 INT8 是 1.5 ms，八核降到 0.56 ms，加速比约为：

\[
\text{加速比} = \frac{1.5}{0.56} \approx 2.68\times
\]

为什么 8 个核没有带来 8 倍加速？用 **Amdahl 定律**可以直观解释。若网络中可并行的部分占比为 \(p\)，则 \(n\) 核的理论加速比上限为：

\[
\text{加速比}(n) = \frac{1}{(1-p) + \dfrac{p}{n}}
\]

把 \(n=8\)、加速比 \(=2.68\) 代入反推，可得 \(p \approx 0.72\)。也就是说，网络大约 72% 的部分能被多核并行加速，剩下的串行部分（数据搬运、调度、不可并行的层）拖累了线性扩展。这是非常典型的边缘加速器行为——**多核多线程显著提速，但受限于串行与带宽，无法做到完全线性。**

> ⚠️ 上面的 Amdahl 反推只是一个**教学性的近似计算**，真实芯片还涉及核间通信、负载均衡、DDR 带宽争用等复杂因素，并非精确建模。它用来帮你建立直觉：多核不是「几核就快几倍」。

#### 4.4.3 源码精读

性能表与多核补充说明：

#### 4.4.4 代码实践

> **源码阅读 + 手算型实践：性能三问**

这是本讲的核心实践，对应学习目标里的「定量解释」。

1. **实践目标**：用 README 性能表的数据，亲手算出三个结论。
2. **操作步骤**：
   - **问题 A（INT8 vs FP16）**：对 MobileNetV2、Resnet50、Yolov5s 三个网络，分别计算 INT8/FP16 延迟比，验证「INT8 更快」是否普遍成立。
   - **问题 B（算力翻倍）**：用「2 TFLOPS vs 4 TOPS」解释为什么 INT8 理论上能快一倍，并结合 A 的结果说明现实为何打折扣。
   - **问题 C（多核加速）**：用单核 INT8（1.5ms）与八核（0.56ms）算出加速比，并用 Amdahl 定律反推可并行占比，解释「八核为何不到 8 倍」。
3. **需要观察的现象**：三个网络的 INT8/FP16 比值都落在 0.6–0.66 区间，说明 INT8 普遍更快但都没有到理想的 0.5。
4. **预期结果**：
   - A：MobileNetV2 \(1.5/2.01 \approx 0.746\)；Resnet50 \(6.19/9.52 \approx 0.650\)；Yolov5s \(14.19/21.82 \approx 0.650\)。三个都 < 1，INT8 普遍更快。
   - B：INT8 算力是 FP16 的 2 倍（4 vs 2），理想延迟应为一半；实际比值 0.65–0.75，多出来的部分来自访存与不可并行开销。
   - C：八核加速比 \(1.5/0.56 \approx 2.68\times\)；Amdahl 反推 \(p \approx 0.72\)，说明约 72% 可并行，串行部分限制了线性扩展。

> 说明：以上计算都可在纸或计算器上完成，无需硬件或运行环境。**待本地验证**的部分仅指「在 FPGA 实板上的实际延迟」——它受频率与带宽影响，会与 ASIC 表不同。

#### 4.4.5 小练习与答案

**练习 1**：同样是 Yolov5s，FP16 是 21.82ms，INT8 是 14.19ms。如果把 INT8 延迟的降低完全归因于「算力翻倍」，理论上应该是多少 ms？实际为何偏高？

> **参考答案**：理论上一半，即 \(21.82 / 2 = 10.91\) ms。实际 14.19 ms 偏高，说明延迟并非只由峰值算力决定——数据搬运（带宽）、层间调度、不可并行部分都会贡献额外时间，所以达不到理想的 2 倍提速。

**练习 2**：README 说性能「高度依赖 IP 频率和 DDR 带宽」。请推测：把同一颗 TPU 从 ASIC（1GHz）搬到频率较低的 FPGA 上，性能表里的数字会变大还是变小？为什么？

> **参考答案**：会**变大（延迟变高）**。因为频率降低意味着单位时间内能完成的运算更少；同时 FPGA 平台的 DDR 带宽通常也不如 ASIC，进一步拉高访存受限网络的延迟。所以 FPGA 实测不能直接套用这张 ASIC 表。

## 6. 本讲小结

- **FREE-TPU V3+** 是已量产商用芯片 IP「EEP-TPU V3+」的**免费评估版**，定位是**深度学习边缘推理**，跑在 Xilinx **ZynqMP FPGA** 上。
- 它与商用版的**唯一关键区别**是加了一个**「一小时停机限制」**，用于防止商业滥用；架构能力本身相同。
- V3+ 的特性归为三类：**高能效**（可重构、数据复用）、**低延迟**（dataflow、混合精度、多核多线程）、**通用性**（CNN/LSTM/RNN、多精度、多框架编译器）。
- 免费版加密 IP 实际开放 **FP16 + INT8** 两种精度；编译器支持 **Pytorch/Caffe/Darknet/NCNN**。
- 性能表显示 **INT8 普遍比 FP16 快**（根因是 INT8 算力 4 TOPS 是 FP16 2 TFLOPS 的两倍，且更省带宽）。
- **多核多线程**能把 MobileNetV2 INT8 从单核 1.5ms 降到八核 0.56ms（约 2.68×加速），受 Amdahl 定律约束，无法线性扩展。
- 性能高度依赖**配置、IP 频率、DDR 带宽**——FPGA 实测会与 ASIC 性能表不同。
