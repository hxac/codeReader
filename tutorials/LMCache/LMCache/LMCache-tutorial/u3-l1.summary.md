本讲是 LMCache 多进程（MP）架构总览。核心动机为 no fate-sharing（不与引擎共命运）：把 KV cache 管理从 vLLM 引擎进程拆为独立 daemon，避免引擎崩溃连累 cache、把 CPU 密集簿记（哈希/淘汰/序列化）移出 GPU 关键路径，并支撑多 worker/多实例共享 cache、PD 分离与 MoE 加速。

MP 含三类进程：vLLM engine 进程（Scheduler 查命中 + Worker 搬 KV）、MP cache server daemon、MP coordinator daemon（可选）。分属两条职责线：`v1/multiprocess/` 是单实例 cache 运行时，worker 经 ZMQ（控制消息）+ CUDA IPC/SHM（GPU 张量零拷贝）连它；`v1/mp_coordinator/` 是舰队级协调，server 经 HTTP/REST opt-in 注册。

关键术语：MPCacheServer 是「组合器」，组装可插拔 EngineModule（Lookup/P2P/Management/Transfer/Blend），共享 MPCacheServerContext，线程池分 AFFINITY（GPU 搬运）与 NORMAL（CPU 簿记）两类；coordinator 是 FastAPI/uvicorn 进程，挂 registry/quota_manager/eviction_manager/blend_directory 等协作者及 health、eviction 后台循环，注册表临时、集成 best-effort。两种 daemon 通信形态（ZMQ 二进制协议 vs REST）迥异，细节留待后续讲义展开。
