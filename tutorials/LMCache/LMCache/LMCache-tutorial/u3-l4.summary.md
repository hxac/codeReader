本讲聚焦 LMCache 多进程架构下「进程之间用什么字节/结构对话」，系统拆解**三套并存的通信协议**及其各自边界：①独立 KV server 的裸 TCP 二进制协议（`v1/protocol.py`），采用「定长头部 + 变长数据体」设计，`ClientMetaMessage` 用 `struct` 打成 186 字节（9 个整数 + 150 字节 key），`ServerMetaMessage` 为 36 字节，`ServerReturnCode` 借用 200/400 但并非 HTTP；②MP daemon 的 ZMQ + msgspec RPC 协议，核心是 `RequestType` 枚举、`ProtocolDefinition`（payload/response/handler_type）、`HandlerType`（SYNC/BLOCKING）三件套，`multiprocess/protocol.py` 仅作门面委托给 `protocols/` 子包，启动期由 `initialize_protocols()` 做「枚举 ⇄ 定义」双向一致性校验、不一致即 fail-fast；③面向控制面的 HTTP/REST，靠「文件名 `_api` 结尾 + 模块级 `router`」约定自动发现路由，`schemas.py` 的 Pydantic 模型作为 coordinator 与 mp server 双端共享的线缆契约。核心结论：三套协议分属数据面与控制面、抽象层级不同，一套尺寸不能通吃——其中 MP 协议只传 GPU block id 与 CUDA event IPC handle，KV 张量字节从不过 ZMQ（零拷贝）；`BlendMatchRequest` 等热点字段则在 JSON 之外用 `encode_tokens` 做紧凑二进制编码。本讲为后续 u4 的 `EncodedObjectKey` 等线缆表示打下基础。
