本讲拆解 worker 进程与 cache daemon 之间的跨进程传输管道，覆盖四个最小模块。mq.py 用 ZMQ 的 ROUTER（server）/DEALER（client）搭出 multipart 帧 RPC 通道，每帧为 [identity, request_uid, request_type, payloads]，靠 request_uid 把响应配对回对应的 Future；真正的 ZMQ 收发全在后台 ClientPollingLoop 单线程完成，调用方只拿 Future。handler 分 SYNC（主循环直接跑）与 BLOCKING（丢线程池），BLOCKING 又分普通池（CPU 簿记）与亲和池（AffinityThreadPool，让同一客户端落同一线程，免去共享 GPU 临时缓冲加锁）。futures.py 的 MessagingFuture 是纯 CPU「事件+结果」等待；CUDAMessagingFuture 额外 synchronize 一个跨进程 CUDA Event，把"字节到达"升级为"GPU 搬完"。GPU 零拷贝靠 DeviceIPCWrapper（CudaIPCWrapper）：KV 张量从不过 ZMQ，REGISTER_KV_CACHE 一次性把 IPC handle 发给 server，server 用 to_tensor() 映射同一显存，此后 STORE/RETRIEVE 只发 block id。CPU 零拷贝则靠 posix_shm.py 按同名段 shm_open+mmap 共享同一物理内存。核心结论：控制流（uid/type/key/block id）走消息队列、用 Future 异步回收，KV 张量始终零拷贝。
