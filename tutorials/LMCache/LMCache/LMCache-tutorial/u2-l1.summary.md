本讲承接 u1-l3，讲透 LMCache 如何用一个统一的 `torch_dev` 抽象同时支持 NVIDIA CUDA、Intel XPU、Habana HPU、Moore Threads MUSA 及纯 CPU 主机。核心是「检测一次，全局复用」：进程启动时由 `_detect_device()` 探测当前硬件，产出两个全局变量 `torch_dev`（对应 torch 设备模块）和 `torch_device_type`（字符串类型），业务层统一 `from lmcache import torch_dev` 即可硬件无关地运行。该检测点位于 `lmcache/__init__.py` 导入期，称为 monkey patch 点。

抽象分三层：入口层负责检测、中间层（engine/storage）硬件无关、底层 gpu_connector 每硬件一套。关键机制有：`DeviceSpec` 注册表 `_DEVICE_REGISTRY` 通过 `discover_subclasses` 自动收集硬件子类，「定义即注册」，且 `DeviceSpec` 自身可实例化为「全 False」兜底；`get_backend()` 把 Python fallback 与硬件算子合并成模块并替换 `sys.modules["lmcache.c_ops"]`（先 fallback 后硬件覆盖）；两条降级路径——有 torch 无卡返回 `StubCPUDevice`（no-op，`is_available()` 为 False），无 torch 返回 `(None,"cpu")` 的 CLI-only 模式；底层 `CreateGPUConnector` 按 `torch_device_type` 集中路由连接器并延迟导入按需加载；第二套自动发现是 `platform/_registry.py` 的 KV IPC wrapper 工厂表，懒加载、双重检查锁定、靠 `_is_default_wrapper` 去重，服务多进程跨实例 KV 传递。新增硬件仅需新增 `DeviceSpec` 子类、GPU 连接器及路由分支，中间层无需改动。
