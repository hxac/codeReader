本讲用两个最简 API 让读者首次跑通 FlashInfer kernel。`single_decode_with_kv_cache` 处理「单请求、单个 query 向量」的解码，q 形状为 [num_qo_heads, head_dim]；`single_prefill_with_kv_cache` 处理「单请求、一段 query 序列」的预填充，q 多一个 qo_len 维度，并引入 causal 因果掩码与 backend（fa2/fa3）选择。两者均要求张量在 CUDA 设备、num_qo_heads 为 num_kv_heads 的整数倍（支持 GQA），KV 默认 NHD 布局。核心结论：首次调用触发 JIT 编译（ninja+nvcc），之后靠两级缓存避免重编译——第一级为进程内 @functools.cache（键为编译期参数元组），第二级为磁盘 .so（键为 URI）。URI 只编码 dtype、head_dim、posenc 等编译期参数，不含 kv_len、头数等运行期形状，故仅改 dtype/head_dim 才会重新编译。编译期特化与运行期参数的区分是理解整个项目缓存行为的钥匙。
