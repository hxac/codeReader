本讲梳理 FlashInfer 融合 MoE 的三套并立后端。**CUTLASS**（OSS MoE GEMM，SM89-SM120 广架构兜底）按 `device_arch` 分派到五个 `gen_*_sm{89,90,100,103,120}_module`（同 major 共用一份，如 100≈110、120≈121），差异仅在 nvcc flags 与 `supported_major_versions`；JIT 时由 `generate_gemm_operations` 现场实例化 CUTLASS 模板为 `.generated.cu` 再编译。**trtllm-gen 后端**仅 SM100，只编译薄启动器，运行期加载/下载预编译 cubin（清单见 `flashinferMetaInfo.h`，路径烧进 `-DTLLM_GEN_GEMM_CUBIN_PATH`），且可内置路由（`FromLogits` 模式）。**CuTe-DSL 后端**仅 SM100/103 NVFP4，不经 C++ JIT，kernel 来自 `nvidia-cutlass-dsl` pip 包，走 sort→GEMM1+act→async memset→GEMM2+finalize 四步显式管线。关键结论：CUTLASS 与 CuTe-DSL 只吃预算好的 `topk_ids`/`topk_weights`（计算后端），trtllm 融合度更高；三者共用 `AutoTuner`/`TunableRunner` 但 tactic 来源各异（实例化模板 / `trtllm_get_valid_moe_configs` / Python 元组表）；统一 API 用 `BackendOptions`+各配置类 `supported(arch)` 声明式门控，默认 trtllm 优先、CUTLASS 兜底、CuTe-DSL 仅 SM100/103。
