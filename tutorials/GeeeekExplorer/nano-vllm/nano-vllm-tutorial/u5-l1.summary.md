本讲聚焦 nano-vllm decode 阶段的最后一项加速手段——CUDA Graph，仅覆盖两个最小模块：`ModelRunner.capture_cudagraph`（预录图）与 `ModelRunner.run_model`（分流与回放）。核心问题是 **kernel launch 开销**：decode 每步只处理极少 token，GPU 常在等 CPU 逐个启动 kernel。CUDA Graph 把整串 kernel 录成一张图，`replay()` 时一次提交，几乎消灭 launch 开销。因 decode 形状只随 batch size 变化而 prefill 形状多变，故 prefill 永远走 eager、只有 decode 用图。`capture_cudagraph` 引入的关键设计：分档点 `[1,2,4,8,16,...,max_bs]`（`max_bs=min(max_num_seqs,512)`）、用 `reversed` 从最大档开始由首张图捐出共享的 `graph_pool`、每档捕获前必须 warmup 消化惰性初始化。图绑定静态张量，故这些张量（`input_ids`/`positions`/`slot_mapping`/`context_lens`/`block_tables`/`outputs`）被收进 `graph_vars` 作为图的「插口」。`run_model` 以 `is_prefill or enforce_eager or size(0)>512` 三选一决定是否走图，走图时执行 copy-in → `graph.replay()` → copy-out。实际 batch size 向上取整到分档点，多出的填充行靠 `slot_mapping.fill_(-1)`（哨兵值，Triton 内核跳过）与 `context_lens.zero_()` 保证无害。LM 头（`compute_logits`）因张量并行下含 `dist.gather` 跨 rank 通信而留在图外。
