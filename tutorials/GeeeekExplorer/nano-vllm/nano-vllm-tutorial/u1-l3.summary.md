本讲剖析 nano-vllm 从 `llm.generate()` 到推理主循环的骨架，覆盖三个最小模块：编排层 `generate`、迭代层 `step`、决策层 `Scheduler`。`generate` 不做计算，只负责把所有 prompt 经 `add_request`（`tokenizer.encode` + 封装 `Sequence`）一次性入队，再 `while not is_finished()` 反复调 `step`，最后按单调递增的 `seq_id` 排序、解码返回。`is_finished` 以 `waiting`/`running` 双队列均空为判据。

一次 `step` 是严格的「三段式」：`schedule`（决策跑哪些序列、prefill 还是 decode）→ `run`（`model_runner.call` 前向采样）→ `postprocess`（写回 token、判定结束）。关键设计是 `num_tokens` 用正负号同时编码阶段与产出量：prefill 步取各序列 `num_scheduled_tokens` 之和（正），decode 步取 `-len(seqs)`（负，因 decode 每序列每步仅产 1 个 token），配合 `if num_tokens > 0` 分别统计 prefill/decode 吞吐。

`Scheduler.schedule` 遵循「prefill 优先于 decode」：只要 `waiting` 有可调度序列就做 prefill 并立即返回 `is_prefill=True`，否则才处理 `running` 做 decode。受 `max_num_seqs`、`max_num_batched_tokens` 约束，并允许首条 prompt 分块。`postprocess` 检测 eos 或达 `max_tokens` 将序列置为 `FINISHED`。序列三态 `WAITING`/`RUNNING`/`FINISHED`、前缀缓存、分块 prefill、抢占等细节留给后续讲义。
