本讲聚焦 `attention.py`，讲清 Attention 层如何补齐 paged KV cache 的「写 K/V」与「读 K/V」闭环。Attention 模块不含可训练参数，仅持有本层的 k_cache/v_cache 视图；其 forward 在 prefill 与 decode 两条路径间分流：prefill 走 `flash_attn_varlen_func`（靠 cu_seqlens 做变长打包），decode 走 `flash_attn_with_kvcache`（靠 context_lens 与 block_table，并对 q 做 unsqueeze(1) 补 seqlen_q=1）。当前缀缓存命中或分块 prefill 的后续块时，prefill 会把 k/v 重指向整张 cache，改读「前缀+刚写入后缀」而非直接用新算出的 K/V。新算出的 K/V 经启动器 `store_kvcache` 调用 Triton 内核 `store_kvcache_kernel` 写入物理槽位：前者用四条 stride 断言保证每 token 的 KV 是连续的 D=num_kv_heads×head_dim 个元素、cache 按 D 对齐，以 slot×D 定位；后者每个 program 写一个 token，并跳过 slot=-1 的 CUDA Graph 填充位以防污染 cache。关键术语包括 slot=block_id×block_size+offset、GQA、varlen 打包、slot_mapping，以及 K/V 共用同一 slot 而分挂 kv_cache 的 [0,layer] 与 [1,layer]。
