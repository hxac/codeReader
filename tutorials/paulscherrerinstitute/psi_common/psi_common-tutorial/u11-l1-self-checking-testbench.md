# 编写自校验测试平台（psi_tb 用法）

## 1. 本讲目标

前面十讲的讲义都在教「如何读懂一个组件」。但当你要给 psi_common 贡献一个新组件、或把库里的组件搬进自己的项目时，光有组件是不够的——你还得能**证明它是对的**。PSI 库对这一点有硬性要求：贡献代码必须附带**自校验测试平台**（self-checking testbench），否则不予合并。

本讲以 [`psi_common_pl_stage_tb`](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd)（即 u7-l1 精读过的 `pl_stage` 的测试平台）为范本，拆解 PSI 库测试平台的「通用骨架」。读完本讲你应当能够：

1. 说出 PSI 库自校验测试平台由哪几块拼成、各自职责，并能照着搭出一个新 TB 的空壳。
2. 解释 `TbRunning` / `ProcessDone` / `testcase` / `done` 这组信号是如何在「激励进程」与「检查进程」之间做协调、又如何让仿真在跑完后**自动干净停机**的。
3. 理解「`assert ... report "###ERROR###: ..." severity error`」这条约定，以及回归脚本为何能用一个字符串 `###ERROR###` 判定全库所有 TB 的成败。
4. 会用 `psi_tb` 工具包里的 `StdlvCompareInt`（比较）与 `print`（打印）来写检查代码，而不是手写一堆 `assert` + `to_integer`。

本讲依赖 u1-l3（仿真运行与回归脚本）和 u7-l1（`pl_stage` 组件本身）。本讲**只讲测试平台的写法**，不重复 `pl_stage` 的内部实现。

## 2. 前置知识

先建立三个直觉，再去看源码就不会乱。

**直觉一：「自校验」意味着 TB 自己判断对错，不靠人眼看波形。** 传统调试型 TB 是「给激励、看波形、人肉对比」；自校验 TB 则是「给激励、在 TB 里用代码把实际输出和期望值逐拍比对，错了就报错」。前者离不开人，后者可以塞进回归脚本里成百上千个一起跑、由脚本统一收口成败——这正是 PSI 库回归测试的基础（回顾 u1-l3：`run.tcl` 跑完所有 TB 后用 `run_check_errors "###ERROR###"` 收集结果）。

**直觉二：仿真必须「自己停下来」，否则回归脚本会卡死。** 一个永远不结束的 TB，会让 `run_tb -all` 永远等下去。PSI 文档明确要求：TB 跑完后必须**所有进程停住、时钟停掉**（见 [doc/old/ch1_introduction/ch1_introduction.md:79-91](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/doc/old/ch1_introduction/ch1_introduction.md#L79-L91)）。所以 TB 里有一套专门的「停机协议」——这就是后面要讲的 `TbRunning` 与 `ProcessDone`。

**直觉三：激励和检查最好分成两个进程，再用握手信号对齐。** 如果把「制造输入」和「检查输出」写进同一个进程，遇到带反压、带延迟的数据通路就会非常别扭——你得一边数着自己发了几个、一边猜现在该收第几个。PSI 库的写法是拆成 `p_stim`（只管发）和 `p_check`（只管收并比对）两个进程，用一个 `testcase` 信号广播「现在跑第几个用例」、一个 `done` 信号回传「这个用例检查完了」，让两者按用例粒度对齐。这套手法在库内所有 TB 里高度一致。

## 4. 核心概念与源码讲解

### 4.1 TB 总体结构

#### 4.1.1 概念说明

PSI 库里几乎每一个测试平台长得都一样——它们都是由一个叫 `TbGen.py` 的 Python 脚本**生成骨架**、再由人工**填充用例**得到的。范本 TB 顶部就写明了这一点：

> `-- Testbench generated by TbGen.py`（[L8-L9](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L8-L9)）

这解释了为什么文件里到处是 `!DO NOT EDIT!`（不要编辑）注释：那些是生成器产出的「公共脚手架」（时钟、复位、停机控制、`ProcessDone` 汇报），所有 TB 一模一样；而 `-- Handwritten`（手写）标记的 `testcase`/`done` 信号和 `p_stim`/`p_check` 进程体才是每个组件各不相同的部分。

一个 PSI 风格 TB 的固定八段结构：

1. **库与 use 子句**：引入 `psi_tb` 工具包。
2. **实体声明**：TB 自己也是个 entity，generic 用来在回归脚本里换参数跑多遍。
3. **架构声明区**：被测信号、TB 控制信号、手写同步信号。
4. **DUT 例化**：把被测组件接上信号。
5. **TB 控制进程 `p_tb_control`**：管「什么时候停机」。
6. **时钟进程**：靠 `TbRunning` 驱动。
7. **复位进程**：上电后释放复位。
8. **激励进程 `p_stim` 与检查进程 `p_check`**：真正跑用例的地方。

#### 4.1.2 核心流程

```
实体 (generic => 回归脚本可换参)
  └─ 架构
       ├─ p_clock   : while TbRunning loop 翻转 clk  end loop;   ← 时钟靠 TbRunning 驱动
       ├─ p_rst     : 上电等待 → 释放复位
       ├─ p_tb_ctrl : 等复位释放 → 等 ProcessDone 全 1 → TbRunning<=false（停机）
       ├─ p_stim    : 发激励 → 每个 case 末尾等 done → 全发完置 ProcessDone(0)
       └─ p_check   : 等 testcase 广播 → 比对 → 每个 case 末尾置 done → 全查完置 ProcessDone(1)
```

关键在于：**两个工作进程（stim/check）各跑各的，用 `ProcessDone` 向控制进程汇报「我完成了」，控制进程等两个都完成后关掉时钟，整个仿真自然结束。**

#### 4.1.3 源码精读

**库与 use 子句**——这里把 `psi_tb` 工具包拉进来（`StdlvCompareInt`、`print` 都来自这里，详见 4.4）：

[psi_common_pl_stage_tb.vhd:20-22](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L20-L22) —— `library work; use work.psi_tb_compare_pkg.all; use work.psi_tb_txt_util.all;`，注意它们编译进 `work` 库（与组件同库），这要求回归脚本先把 psi_tb 的源码加进来。

**实体声明**——TB 自己是个带 generic 的 entity，唯一的 generic `handle_rdy_g` 会在回归脚本里被换成 `true`/`false` 各跑一遍（覆盖 `pl_stage` 的两条实现分支）：

#### 4.1.5 小练习与答案

**练习**：为什么 TB 的实体要带 generic `handle_rdy_g`，而它最终又映射到 `pl_stage` 的 `use_rdy_g`？能不能直接在 TB 里把 `use_rdy_g` 写死成 `true`？

**答案**：因为 `pl_stage` 有「带反压」和「朴素寄存器」两套 `if generate` 实现（u7-l1），两套都要测才算覆盖完整。把开关做成 TB 的 generic，回归脚本就能用 [config.tcl:320-322](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/config.tcl#L320-L322) 的 `tb_run_add_arguments` 换成 `true`/`false` 各编译运行一遍，**一份 TB 代码覆盖两条分支**。写死成 `true` 会漏掉 `use_rdy_g=false` 的整条朴素路径。

### 4.2 多进程协调

#### 4.2.1 概念说明

「让仿真自己停下来」听起来简单，做起来要解决两个问题：**(a) 时钟怎么停？(b) 谁来拍板说『都跑完了』？** PSI 库用一组约定好的信号回答这两个问题，这是整个 TB 骨架里最值得学的一块。

核心是四个信号：

| 信号 | 类型 | 作用 |
|:--|:--|:--|
| `TbRunning` | `boolean` | 时钟进程的「电源开关」。为 `true` 时钟才翻转，置 `false` 后时钟停、仿真结束。 |
| `ProcessDone` | `std_logic_vector(0 to 1)` | 每个工作进程一位，跑完把自己的位置 `1`。这里 0 号=`p_stim`，1 号=`p_check`。 |
| `AllProcessesDone_c` | 常量 `(others => '1')` | 「全部完成」的判据，`ProcessDone` 等于它即停机。 |
| `testcase` / `done` | `integer` / `boolean` | `p_stim` 与 `p_check` 之间的「按用例对齐」握手（手写部分）。 |

#### 4.2.2 核心流程

**停机协议**（生成器产出，所有 TB 一致）：

```
p_tb_control:
    等到 rst_i = '0'            # 复位释放后才开始监听
    等到 ProcessDone = AllProcessesDone_c   # 两个工作进程都汇报完成
    TbRunning <= false          # 关时钟 → 仿真自然结束
    wait;                       # 进程挂起

p_clock:
    while TbRunning loop
        等半个周期; clk <= not clk
    end loop
    wait;
```

**用例级握手**（手写部分，`p_stim` 与 `p_check` 之间）：

```
p_stim:                          p_check:
    testcase <= 0                  wait until testcase = 0   # 收到广播
    ... 发激励 ...                  done <= False
    if done /= true then           ... 等输出、比对 ...
        wait until done = true     done <= True              # 回传「查完了」
    end if
```

`testcase` 初值是 `-1`（见 [L62](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L62)），所以第一个 `wait until testcase = 0` 不会提前误触发；每跑完一个用例，stim 用 `if done /= true then wait until done = true` 等检查方放行，再进入下一个用例。

#### 4.2.3 源码精读

**控制信号声明**——注意常量 `TbProcNr_stim_c=0`、`TbProcNr_check_c=1` 给 `ProcessDone` 的两位起了名字，避免到处写魔法数字：

[psi_common_pl_stage_tb.vhd:42-48](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L42-L48) —— `TbRunning`、`ProcessDone`、`AllProcessesDone_c` 与两个进程编号常量。

**停机控制进程**——只有 4 行有效逻辑，却是「仿真自动结束」的全部秘密：

[psi_common_pl_stage_tb.vhd:86-92](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L86-L92) —— `p_tb_control`：等复位释放 → 等 `ProcessDone = AllProcessesDone_c` → `TbRunning <= false`。

**时钟进程**——半周期由频率算出（`0.5 * (1 sec) / Frequency_c`，`Frequency_c = 100e6` 即 100 MHz、半周期 5 ns），循环条件就是 `while TbRunning`：

[psi_common_pl_stage_tb.vhd:97-105](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L97-L105) —— 时钟靠 `TbRunning` 驱动，`TbRunning` 一假，循环退出、时钟停。

**复位进程**——等 1 us 后再等两个上升沿才释放复位，注释明说「确保复位至少覆盖一个时钟沿」：

[psi_common_pl_stage_tb.vhd:110-118](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L110-L118) —— 上电复位时序。

**用例级握手的实际样子**——stim 这边广播 `testcase <= 0`，结尾 `if done /= true then wait until done = true; end if` 等检查方放行：

[psi_common_pl_stage_tb.vhd:130-145](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L130-L145) —— `print(">> Single Samples")` 打印用例名，`testcase <= 0` 广播用例号，发完激励后用 `done` 等待检查完成。

check 这边镜像对应：`wait until testcase = 0` 收广播 → `done <= False` → 比对 → `done <= True`：

[psi_common_pl_stage_tb.vhd:220-229](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L220-L229) —— check 进程对用例 0 的处理：收广播、置 `done`、比对、放行。

**两个工作进程的收尾**——各自跑完所有用例后，把自己的 `ProcessDone` 位置 1，然后 `wait`（挂起，不再耗仿真时间）：

[psi_common_pl_stage_tb.vhd:208-210](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L208-L210) —— `p_stim` 收尾：`ProcessDone(TbProcNr_stim_c) <= '1'; wait;`。

[psi_common_pl_stage_tb.vhd:289-291](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L289-L291) —— `p_check` 收尾：`ProcessDone(TbProcNr_check_c) <= '1'; wait;`。

两位都置 1 后，`p_tb_control` 看到 `ProcessDone = AllProcessesDone_c`，置 `TbRunning <= false`，时钟停、仿真结束。整条链自洽，无需任何人为干预。

#### 4.2.4 代码实践

**实践目标**：跟踪一次「用例 0」的 stim/check 握手，看清两者如何按拍对齐。

**操作步骤**：

**需要观察的现象**：`done` 像一个「通行闸」——check 没置 `True` 之前，stim 卡在 `wait until done = true` 不往下走，保证 stim 不会在 check 还没核完上一个用例时就冲进下一个。

**预期结果**：你会理解 `testcase`/`done` 是一对**「广播 + 应答」**信号，本质上是把两个独立进程按用例边界串行化。本实践为源码阅读型，无需运行。

#### 4.2.5 小练习与答案

**练习 1**：如果把 `p_tb_control` 里的 `wait until rst_i = '0'` 删掉，会发生什么？

**答案**：`ProcessDone` 在复位释放前一直是 `00`（初值），但删掉这句后控制进程会立即去等 `ProcessDone = AllProcessesDone_c`——这本身不会误触发（因为工作进程也会 `wait until rst_i = '0'` 才开始）。真正的问题是复位语义被破坏：控制进程不再以「复位已释放」作为计时起点。保留它是为了明确「停机倒计时从复位释放后开始」。

**练习 2**：`ProcessDone` 为什么是 `std_logic_vector` 而不是一个简单的 `integer` 计数器？

**答案**：因为每个工作进程**只关心、只写自己那一位**，互不干扰，无需任何仲裁或读改写；控制进程只需做一次 `ProcessDone = AllProcessesDone_c` 的相等比较。若用整数计数器，多个进程要 `+1` 就得串行化或引入信号分辨函数，复杂且易错。位向量是「N 个独立完成标志」最直接的建模。

### 4.3 自检与报告约定（`###ERROR###`）

#### 4.3.1 概念说明

自校验 TB 判错的统一手法是 VHDL 的 `assert ... report ... severity` 语句：当 `assert` 后的条件为**假**时，触发 `report` 打印消息、按 `severity`（`note/warning/error/failure`）上报。PSI 库在这一点上加了一条**全库统一的字符串约定**：

> 凡是真正出错时的报告消息，**必须以 `###ERROR###:` 开头**。

这条约定写在贡献规范里（[ch1_introduction.md:84](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/doc/old/ch1_introduction/ch1_introduction.md#L84)）。它的价值在于**让回归脚本能用一次字符串搜索判定全库所有 TB 的成败**：`run.tcl` 跑完所有 TB 后执行 `run_check_errors "###ERROR###"`（[run.tcl:32](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/run.tcl#L32)），只要任何 TB 的任何输出里出现这个前缀，回归即判失败。于是「六十多个 TB、上百次运行」的成败被压缩成一句 grep——这就是统一约定的力量。

文档还要求：**只在真有问题时才用 `error`/`failure`/`fatal` 级别**（[ch1_introduction.md:83](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/doc/old/ch1_introduction/ch1_introduction.md#L83)）。换句话说，进度提示走 `print`（note 级），真正的失败才走带 `###ERROR###` 前缀的 `severity error`。

#### 4.3.2 核心流程

```
TB 里发现实际 ≠ 期望
   └─ assert <条件为真> report "###ERROR###: <说明>" severity error
            │ 条件为假时触发
            ▼
仿真器把消息（含 ###ERROR### 前缀）打印到日志
   └─ run.tcl: run_check_errors "###ERROR###"
            │ 在所有 TB 输出里 grep 这个前缀
            ▼
命中 → 回归失败；未命中 → 回归通过
```

注意范本 TB 里 `assert` 条件常常写成「`A or not handle_rdy_g`」这种形式——意思是「**只有在 `handle_rdy_g` 为真时这个断言才生效**」。这是一种把断言和 generic 分支绑在一起的常用技巧，避免在 `use_rdy_g=false`（根本没有 `rdy_o` 语义）的运行里误报。

#### 4.3.3 源码精读

**`###ERROR###` 断言的典型写法**——检查「`rdy_o` 不该掉」，且带 `or not handle_rdy_g` 守卫，使该断言仅在带反压的分支里生效：

[psi_common_pl_stage_tb.vhd:134](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L134) —— `assert rdy_o = '1' or not handle_rdy_g report "###ERROR###: rdy_o went low unexpectedly" severity error;`。

另一处在检查进程里，确认「valid 一旦拉高、在握手前不应掉」（这是 AXI-S「源端不能撤 VLD」规则的回归检查）：

[psi_common_pl_stage_tb.vhd:276](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L276) —— `assert vld_o = '1' report "###ERROR###: vld_o went low" severity error;`。

**回归脚本如何用这个前缀收口**——编译、跑完所有 TB 后，用 `###ERROR###` 在输出里搜错误：

[run.tcl:14-32](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/run.tcl#L14-L32) —— `init` → `source ./config.tcl` → `compile_files -all -clean` → `run_tb -all` → `run_check_errors "###ERROR###"`。

**贡献规范原文**——把这条约定的「为什么」讲得很清楚：

[ch1_introduction.md:79-84](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/doc/old/ch1_introduction/ch1_introduction.md#L79-L84) —— 自校验 TB 四条要求：必须自检、跑完自动停、只对真问题用高级别、出错消息以 `###ERROR###:` 开头（因为回归脚本要搜这个串）。

#### 4.3.4 代码实践

**实践目标**：亲手触发一次 `###ERROR###`，看清它如何被回归脚本捕获。

**操作步骤**：

1. 复制一份 `psi_common_pl_stage_tb.vhd` 到你自己的实验目录（**不要改原文件**）。
2. 在检查进程里把某条 `StdlvCompareInt(i, dat_o, "Wrong Data")`（如 [L227](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L227)）的期望值故意改错，例如把 `i` 改成 `i+1`。
3. 按你本地的 PsiSim 工作副本结构（u1-l3）跑回归或交互仿真。

**需要观察的现象**：仿真日志里应出现一条以 `###ERROR###:` 开头的报告（`StdlvCompareInt` 比较失败时正是用这个前缀上报，见 4.4），`run_check_errors "###ERROR###"` 会据此把这次回归判为失败。

**预期结果**：你把期望值改回去后，`###ERROR###` 消失、回归恢复通过。这一加一减就证明了「前缀约定 + grep」这套判定机制是闭环的。

> 注意：本实践依赖本地配好的 PsiSim + psi_tb 兄弟仓库（u1-l3）。若本地未配置仿真环境，可改为**源码阅读型实践**：在 [config.tcl](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/config.tcl) 里统计有多少个 TB 用了 `###ERROR###` 约定（提示：它们都来自各自 TB 里的 `assert`/`StdlvCompareInt`），运行结果**待本地验证**。

#### 4.3.5 小练习与答案

**练习**：为什么 PSI 库不直接用 `severity failure`（失败级别，仿真器通常立即停止）来报错，而是统一用 `severity error` + `###ERROR###` 前缀？

**答案**：用 `error` 级别让仿真**继续跑完**当前 TB 的剩余用例，从而一次运行能暴露**尽可能多**的错误（而不是遇到第一个错就停、修完再跑才知道第二个错）。成败的最终判定交给回归脚本对 `###ERROR###` 前缀的搜索——它只看「有没有出过错」，不看「仿真有没有中途停」。这套「软失败 + 事后 grep」是 PSI 库成百上千次回归能高效收敛的关键设计。

### 4.4 psi_tb 工具包

#### 4.4.1 概念说明

`psi_tb` 是一个**独立于 psi_common 的兄弟仓库**（u1-l2、u1-l3 已介绍），专门收集「写测试平台时反复要用、但又不该塞进可综合库」的工具：字符串打印、数值比较、AXI/I2C 的 BFM（总线功能模型）、活动检测等。范本 TB 只用了其中最基础的两个包：

- `psi_tb_txt_util`：提供 `print(...)`，把字符串打印到仿真日志（本质是 `report ... severity note`），用于打用例名、进度。
- `psi_tb_compare_pkg`：提供 `StdlvCompareInt(expected, actual, msg)` 等**比较并自动上报**的函数——期望值与实际 `std_logic_vector` 比对，不等就用 `###ERROR###:` 前缀上报（这正是 4.3 约定的落地实现）。

为什么要单独的工具包？因为「把整数和 slv 比对、不一致就按统一格式报错」这件事在每个 TB 里都要做几十次，手写 `assert to_integer(unsigned(x)) = v else report ...` 既啰嗦又容易写错前缀。`psi_tb` 把它收敛成一个调用，保证全库的比较与上报风格一致。

#### 4.4.2 核心流程

```
# 打印进度（note 级，不影响成败判定）
print(">> Single Samples")

# 比较并自报错（失败时自动 ###ERROR### 前缀 + severity error）
wait until rising_edge(clk_i) and vld_o = '1';   # 等到一个有效输出
StdlvCompareInt(期望整数, dat_o, "Wrong Data");    # 比对 dat_o 是否等于期望
```

`StdlvCompareInt` 内部做的是：把期望整数转成与 `dat_o` 等宽的 slv，逐位比较；不等则 `report "###ERROR###: <msg> ..."` 且 `severity error`。所以**你不必手写 `###ERROR###` 前缀**——比较类检查交给 `psi_tb`，前缀自动带上。

#### 4.4.3 源码精读

**引入工具包**——范本 TB 在 use 子句里拉进两个包：

[psi_common_pl_stage_tb.vhd:21-22](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L21-L22) —— `use work.psi_tb_compare_pkg.all; use work.psi_tb_txt_util.all;`。

> 这两个包的源文件本身不在 psi_common 仓库里，而在兄弟仓库 `psi_tb/hdl/` 下；回归脚本负责把它们编译进 `work` 库，见 [config.tcl:26-27](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/config.tcl#L26-L27)（`psi_tb/hdl/psi_tb_txt_util.vhd`、`psi_tb/hdl/psi_tb_compare_pkg.vhd` 在 `lib` 源组里）。这也是为什么跑仿真必须按 u1-l3 的「工作副本结构」把 psi_tb 摆成兄弟目录。

#### 4.4.4 代码实践

**实践目标**：体会 `StdlvCompareInt` 比手写 `assert` 省了多少事。

**操作步骤**：

1. 看 [L227](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/testbench/psi_common_pl_stage_tb/psi_common_pl_stage_tb.vhd#L227) 的一行 `StdlvCompareInt(i, dat_o, "Wrong Data");`。
2. 假设没有这个函数，等价的「裸 VHDL」应写成（**示例代码**，非项目原有）：
   ```vhdl
   -- 示例代码：手写等价比较
   assert to_integer(unsigned(dat_o)) = i
     report "###ERROR###: Wrong Data (got " & integer'image(to_integer(unsigned(dat_o))) &
            ", expected " & integer'image(i) & ")"
     severity error;
   ```
3. 对比两版的字数与出错概率（手写版要自己拼 `###ERROR###` 前缀、自己转整数、自己拼消息）。

**需要观察的现象**：`psi_tb` 的一个函数调用，等价于 3~4 行啰嗦且易错的手写断言，且前缀由函数统一保证。

**预期结果**：你会理解为什么 PSI 库**几乎所有数值比较都走 `psi_tb`**——它既是省事，更是为了保证「前缀不写错、上报格式统一」。本实践为源码阅读型，无需运行。

#### 4.4.5 小练习与答案

**练习**：范本 TB 里 `StdlvCompareInt` 的第三个参数 `"Wrong Data"` 是干什么用的？如果同一个用例里有多处比较都传 `"Wrong Data"`，出错时你怎么定位是哪一处？

**答案**：第三个参数是**出错消息的标签**，`StdlvCompareInt` 在不等时把它拼进 `###ERROR###: ...` 报告里，方便人快速知道「是哪一类检查挂了」。如果多处都用同一个标签，定位会困难——这是范本 TB 的一个小不足。改进做法是给每处比较传**更具区分度**的标签（例如带循环变量或用例名），这样出错日志能直接指到具体位置；这也是写自检代码时可顺手优化的细节。

## 5. 综合实践

**任务**：仿照 `psi_common_pl_stage_tb`，为 psi_common 里任意一个简单组件（建议挑一个纯组合或单进程的，如 `psi_common_debouncer` 或 `psi_common_ser_par`）搭一个**自校验 TB 的空壳框架**，把本讲四块知识串起来用一遍。

**要求框架必须包含**（先不填具体用例，只搭能编译、能自动停机的骨架）：

1. **库与 use**：引入 `ieee` 三个标准包 + `work.psi_tb_compare_pkg.all` + `work.psi_tb_txt_util.all`。
2. **实体**：带至少一个 generic（例如把被测组件的某个关键 generic 提到 TB 顶层，方便回归换参）。
3. **架构声明区**：
   - 被测信号（`clk_i/rst_i/...`，按你选的组件端口来）；
   - TB 控制信号 `TbRunning`、`ProcessDone`（位数 = 你的工作进程数）、`AllProcessesDone_c`、进程编号常量；
   - 手写同步信号 `testcase : integer := -1`、`done : boolean := False`。
4. **DUT 例化**：例化你选的组件。
5. **三个脚手架进程**：照抄 `p_tb_control`、`p_clock_Clk`（频率自定）、`p_rst_Rst` 的写法。
6. **两个工作进程** `p_stim` / `p_check`：各写一个占位用例——stim 里 `print(">> Case 0"); testcase <= 0; ... if done /= true then wait until done = true; end if;`，结尾 `ProcessDone(stim位) <= '1'; wait;`；check 里 `wait until testcase = 0; done <= False; ...至少一处 StdlvCompareInt(...) 比对...; done <= True;`，结尾 `ProcessDone(check位) <= '1'; wait;`。

**验收标准**（自检清单）：

- 编译能过（假设本地已配 psi_tb）。
- 仿真能**自己停下来**（`TbRunning` 最终被置 `false`，时钟停）。
- 把 check 里那处 `StdlvCompareInt` 的期望值故意改错，日志应出现 `###ERROR###:`；改回应消失。
- 在 [config.tcl](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/config.tcl) 里照 [pl_stage_tb 的注册方式](https://github.com/paulscherrerinstitute/psi_common/blob/98c2fcc75fa11edfe837cf0de885da95d94c871f/sim/config.tcl#L319-L323)给你的 TB 加 `add_sources`（tb 组）和 `create_tb_run`，确认回归能跑到它。

> 提示：最稳妥的起步方式是**复制 `psi_common_pl_stage_tb.vhd`、改名、把 DUT 换成你的组件、把 stim/check 的用例体清空**——这样脚手架部分（生成器产出的那 3 个进程和所有控制信号）天然正确，你只需专注业务。能运行的结果**待本地验证**（依赖本地 PsiSim + psi_tb 工作副本）。

## 6. 本讲小结

- PSI 库的测试平台由 `TbGen.py` 生成**公共脚手架**（时钟、复位、停机控制、`ProcessDone`）、由人工填**业务**（`p_stim`/`p_check` 与 `testcase`/`done`），呈现「八段结构」。
- 仿真自动停机靠 `TbRunning`（驱动时钟）+ `ProcessDone`/`AllProcessesDone_c`（汇报完成）：两个工作进程都置位后，控制进程关掉时钟。
- `p_stim` 与 `p_check` 用 `testcase`（广播用例号）和 `done`（应答「查完了」）做按用例粒度的同步，初值 `-1`/`False` 防误触发。
- 全库统一的**判错约定**是 `report "###ERROR###: ..." severity error`，回归脚本 `run.tcl` 用 `run_check_errors "###ERROR###"` 一次 grep 收口所有 TB 成败；用 `error` 而非 `failure` 是为了一次跑出尽量多的错。
- `psi_tb` 工具包（兄弟仓库）提供 `print`（打印）与 `StdlvCompareInt`（比对并自动带 `###ERROR###` 前缀上报），让检查代码简短且前缀不会写错；新 TB 必须登记进 `sim/config.tcl` 才会被回归跑到。
- `assert` 条件里常带 `or not <generic>` 守卫，让断言只在对应分支生效，是绑定 generic 分支与检查的常用技巧。
