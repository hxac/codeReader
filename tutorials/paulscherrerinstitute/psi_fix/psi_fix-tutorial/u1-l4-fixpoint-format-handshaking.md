# 定点数格式与握手约定

## 1. 本讲目标

学完本讲，你应当能够：

- 用 `[s, i, f]` 三元组描述任意一个 psi_fix 定点数，并计算出它占用的**总位宽**与**可表示数值范围**。

本讲不写 VHDL 代码逻辑，只建立贯穿全库的「共同语言」。后续每一篇讲义都会用到这三个约定，所以本讲是 u2（定点包）、u3（测试方法论）乃至所有组件讲义的前置基础。

## 2. 前置知识

在进入正文前，先用三段话建立直觉。如果你已经看过 u1-l1（项目概览）和 u1-l2（目录结构），可以直接跳到第 3 节。

### 2.1 什么是「定点数」

真实世界的物理量（温度、电压、声音采样）是连续的实数，但 FPGA 里只有 0 和 1。**浮点数（float）**用一个指数和小数部分来逼近很大或很小的实数，范围大但硬件实现昂贵；**定点数（fixed-point）**则把小数点的位置**固定**下来：约定好「这 18 位里，最高位是符号位、中间 0 位是整数位、低 17 位是小数位」，于是同一串比特就有了唯一确定的数值。

定点数牺牲了表示范围，换来的是「加法器、乘法器可以直接用普通整数硬件实现，资源极省」。这正是 DSP 在 FPGA 上的主流做法，也是整个 psi_fix 库存在的理由。

### 2.2 为什么需要「握手」

DSP 组件通常是**流式**的：数据一个样本接一个样本地进来，处理后一个样本接一个样本地出去。但上下游的速度未必一致——上游可能时快时慢，下游也可能临时处理不过来。如果大家各发各的，就会丢数据或读到垃圾。

**握手（handshaking）**就是一套约定：发送方举起「我这次的数据是有效的」旗子（valid），接收方举起「我现在能收」的旗子（ready），两面旗子同时举起的那一个时钟周期，数据才真正完成传递。psi_fix 采用工业界事实标准的 **AXI4-Stream（简称 AXI-S）** 协议。

### 2.3 为什么需要命名约定

一个组件可能有十几个端口。如果名字随心所欲（`Data`、`valid_in`、`readyOut`），阅读者每次都要猜方向。psi_fix 用**后缀**机械地编码方向：`_i` 表示输入、`_o` 表示输出、`_g` 表示编译期参数（generic）。看到 `dat_i` 就立刻知道「这是输入数据」，不需要查文档。这点在 u1-l1 的贡献规范里也是强制要求。

## 4. 核心概念与源码讲解

### 4.1 定点格式 [s, i, f]

#### 4.1.1 概念说明

psi_fix 用一个三元组来完整描述一个定点数格式：

\[
\text{格式} = [s,\ i,\ f]
\]

- \(s\) ：**符号位**个数，只能是 0（无符号）或 1（有符号，二进制补码）。
- \(i\) ：**整数位**个数（小数点左边的位数，可正可负）。
- \(f\) ：**小数位**个数（小数点右边的位数）。

三者直接相加就是**总位宽**：

\[
W = s + i + f
\]

例如格式 `[1, 0, 17]`：有符号、0 个整数位、17 个小数位，总位宽 \(W = 1 + 0 + 17 = 18\) 位。

小数点的「重量」由 \(f\) 决定：最低位（LSB）的权重是 \(2^{-f}\)。所以这种格式的**可表示范围**为：

- 无符号 `[0, i, f]`：\(\big[\,0,\ \ 2^{i} - 2^{-f}\,\big]\)
- 有符号 `[1, i, f]`：\(\big[\,-2^{i},\ \ 2^{i} - 2^{-f}\,\big]\)

对 `[1, 0, 17]` 而言，范围是 \(\big[-1.0,\ \ 1.0 - 2^{-17}\big] \approx [-1.0,\ 0.99999]\)，即「能表示 -1.0，但不能表示 +1.0」。这个小细节后面解释乘法位增长时会用到。

这个三元组在源码里就是一个 record，名字叫 `psi_fix_fmt_t`：

[hdl/psi_fix_pkg.vhd:27-31](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_pkg.vhd#L27-L31) —— 定义了 `psi_fix_fmt_t` 这个 record，三个字段 `s / i / f`，其中 `s` 被约束为 `0 to 1`，`i` 和 `f` 是任意整数。

#### 4.1.2 核心流程

给定一个格式，推导总位宽只需一步加法；推导可表示范围只需代入上面的两个公式。整个库到处用 `psi_fix_size(fmt)` 这个函数来「由格式算位宽」，从而声明信号宽度：

[hdl/psi_fix_pkg.vhd:77-78](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_pkg.vhd#L77-L78) —— `psi_fix_size(fmt)` 返回该格式的总位数，等价于 \(s+i+f\)。

例如在 `psi_fix_mov_avg` 里，输入数据端口的位宽不是写死的常数，而是「根据输入格式动态算出来」的：

```vhdl
dat_i : in  std_logic_vector(psi_fix_size(in_fmt_g) - 1 downto 0);
```

[hdl/psi_fix_mov_avg.vhd:36-39](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_mov_avg.vhd#L36-L39) —— `dat_i` 的宽度由 `psi_fix_size(in_fmt_g)` 决定；`in_fmt_g / out_fmt_g` 本身就是 `psi_fix_fmt_t` 类型的 generic（[第 24-25 行](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_mov_avg.vhd#L24-L25)）。这意味着换一种定点精度，只需改 generic、无需改代码。

#### 4.1.3 源码精读

定点运算函数的签名都遵循同一套模板：传入「操作数 + 操作数格式 + 结果格式 +（可选）舍入/饱和模式」，返回结果比特向量。以加、减、乘三个最常用函数为例：

[hdl/psi_fix_pkg.vhd:99-120](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_pkg.vhd#L99-L120) —— `psi_fix_add` / `psi_fix_sub` / `psi_fix_mult` 三个函数的声明。注意它们的 `rnd`（舍入）和 `sat`（饱和）参数都有**默认值** `psi_fix_trunc` / `psi_fix_wrap`，即「不写就默认截断 + 不饱和」。这是为什么 tips.md 反复强调「只在必要时才做舍入/饱和」——默认行为是最省资源、但最容易溢出的。

#### 4.1.4 代码实践

**实践目标**：亲手把 `[s,i,f]` 翻译成位宽与范围，建立对格式的肌肉记忆。

**操作步骤**：

1. 取格式 `[1, 0, 17]`，按公式写出总位宽 \(W\)。
2. 代入有符号范围公式，写出它的最小值与最大值。
3. 再取一个无符号例子 `[0, 8, 8]`（tips.md 里反复用的例子），写出位宽与范围。

**需要观察的现象**：

- `[1, 0, 17]`：\(W = 18\)，范围 \([-1.0,\ 1.0 - 2^{-17}]\)。
- `[0, 8, 8]`：\(W = 16\)，范围 \([0,\ 2^8 - 2^{-8}] = [0,\ 255.99609]\)。

**预期结果**：你能不查表地由三元组推出版宽与范围。这就是后续读懂任何 psi_fix 组件 generic 的前提。

#### 4.1.5 小练习与答案

**练习 1**：格式 `[1, 3, 12]` 的总位宽和可表示范围是多少？

**答案**：\(W = 1 + 3 + 12 = 16\) 位；有符号范围 \([-2^3,\ 2^3 - 2^{-12}] = [-8,\ 7.99976]\)。

**练习 2**：一个 16 位无符号数，希望表示 `0 ~ 65535` 的纯整数（无小数），格式应写成什么？

**答案**：`[0, 16, 0]`。整数位 = 16，小数位 = 0，无符号位。

### 4.2 位增长规则

#### 4.2.1 概念说明

做定点运算时，结果的「自然格式」往往比输入更宽——这就是**位增长（bit growth）**。如果无视它、直接把结果塞回原格式，就会**溢出**。tips.md 把「在未被饱和保护的地方发生溢出」列为 DSP 设计里最糟糕的事，并点名了 Ariane-5 火箭爆炸这个著名案例作为警示。

关键规则有三条（直接来自 tips.md「Get ranges correct」一节）：

1. **加法 / 减法**：整数位 +1，小数位取两者最大值。
2. **舍入（rounding）**：可能溢出，因此做舍入的那一步要再 +1 整数位。
3. **乘法**：两个有符号数相乘，整数位相加后**还要再 +1**，小数位直接相加。

#### 4.2.2 核心流程

用公式把三条规则写清楚。设两个输入格式分别为 \([s_a, i_a, f_a]\) 与 \([s_b, i_b, f_b]\)。

**加法 / 减法**（结果一定带符号）：

\[
\text{结果格式} = [1,\ \max(i_a, i_b) + 1,\ \max(f_a, f_b)]
\]

直观理解：两个同范围数相加，最坏情况是「最大值 + 最大值」，需要多 1 个整数位承载进位；小数位对齐到更细的那一个。

**乘法**（符号位 = 两符号位之和，但通常约定为有符号 1 位）：

\[
[1, a, b] \times [1, c, d] = [1,\ a + c + 1,\ b + d]
\]

整数位多出来的那一个，是为了容纳 \((-1.0) \times (-1.0) = +1.0\) 这个极端结果——而 +1.0 在 `[1, a, b]` 这种格式里是**表示不出来**的（其最大值是 \(2^a - 2^{-b} < 2^a\)），必须多 1 个整数位才能放下。

**舍入**：舍入要加一个「舍入常数」再做截断，加常数本身可能进位，所以舍入步骤的中间格式要比目标格式多 1 个整数位。

> 数据流：`原始格式 --运算--> 自然增长格式 --(可选)舍入--> 舍入中间格式 --(可选)饱和--> 目标格式`。每一步都可能变宽或变窄，关键是**绝不能在未被饱和保护的地方缩窄**。

#### 4.2.3 源码精读

tips.md 用一段真实代码演示了「手动拆分（Manual Splitting）」，把「加法 → 舍入 → 饱和」拆成三级，每级的中间格式都遵循位增长规则：

[doc/files/tips.md:82-100](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/doc/files/tips.md#L82-L100) —— 关键看这几行常量声明：

- `addFmt_c := (1, 9, 8)` —— 加法结果，整数位 8→9（**+1**）。
- `rndFmt_c := (1, 10, 8)` —— 舍入中间格式，整数位 9→10（再 **+1**，因为加舍入常数可能进位）。
- `rFmt_c := (1, 8, 0)` —— 最终目标格式，靠最后一步饱和压回去。

这正是「加法 +1、舍入再 +1」两条规则的真实落地。规则本身在 tips.md 上一节用文字总结过：

[doc/files/tips.md:162-167](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/doc/files/tips.md#L162-L167) —— 三条警示：①舍入可致溢出；②有符号数取负可致溢出（`[1,0,x]` 含 -1.0 但不含 +1.0）；③两个有符号数相乘多 1 个整数位，即 `[1,a,b]×[1,c,d] = [1,a+b+1,c+d]`。

真实组件 `psi_fix_mov_avg` 内部也严格遵循这套规则。它先做「差分」（当前样本减延迟样本），差分结果的格式就是把输入整数位 +1：

[hdl/psi_fix_mov_avg.vhd:50-51](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_mov_avg.vhd#L50-L51) —— `DiffFmt_c := (1, in_fmt_g.I + 1, in_fmt_g.F)` 正是「加/减法整数位 +1」；随后的 `SumFmt_c` 则按累加抽头数 `AdditionalBits_c = log2(taps)` 继续增长整数位。这就是位增长规则在工程代码里的直接体现。

#### 4.2.4 代码实践（本讲主实践任务）

**实践目标**：用位增长规则手算 `[1, 0, 17] × [1, 0, 17]` 的理论最大格式，并解释乘法为何多出一个整数位。

**操作步骤**：

1. 套用乘法公式 \([1, a, b] \times [1, c, d] = [1, a+c+1, b+d]\)，代入 \(a=c=0,\ b=d=17\)。
2. 写出结果格式与总位宽。
3. 用「极端乘积」论证为什么必须 +1 整数位。

**推导过程**：

\[
[1,0,17] \times [1,0,17] = [1,\ 0+0+1,\ 17+17] = [1,\ 1,\ 34]
\]

总位宽 \(W = 1 + 1 + 34 = 36\) 位。

**为什么多出一个整数位**：输入 `[1,0,17]` 的范围是 \([-1.0,\ 1.0 - 2^{-17}]\)，它**能表示 -1.0，但不能表示 +1.0**。两个极端值相乘时，最坏情况是：

\[
(-1.0) \times (-1.0) = +1.0
\]

而 +1.0 在 `[1, 0, 34]`（整数位仍是 0）里**放不下**——该格式最大只能到 \(1 - 2^{-34} < 1\)。为了让 +1.0 这个合法结果不溢出，必须把整数位从 0 抬到 1，即 `[1, 1, 34]`，其范围 \([-2,\ 2 - 2^{-34}]\) 才能容纳 +1.0。

**预期结果**：结果格式 `[1, 1, 34]`，36 位；额外整数位源于「两个负的最小值相乘会得到一个正的最大值，该最大值超出原格式上界」。

> 待本地验证：u2-l3 会教你在 Python 模型里用 `psi_fix_mult` 实际跑一遍，把这里的理论格式与 `model/psi_fix_pkg.py` 的输出逐位对照。

#### 4.2.5 小练习与答案

**练习 1**：`[1, 8, 8] + [1, 8, 8]` 的自然结果格式是什么？

**答案**：`[1, 9, 8]`。整数位 +1（8→9），小数位不变。

**练习 2**：tips.md 警告「舍入可能溢出」。请用 `[0, 0, 17]` 舍入到 `[0, 0, 16]` 解释这一点。

**答案**：`[0, 0, 17]` 的最大值是 \(1.0 - 2^{-17}\)，舍入（四舍五入）后会变成 \(1.0\)，而 `[0, 0, 16]` 的上界是 \(1.0 - 2^{-16} < 1.0\)，于是发生溢出。所以舍入步骤的中间格式要 +1 整数位。

**练习 3**：为什么 tips.md 强烈建议「基于理论最大值」而非「假设输入永远小于 0.7」来选格式？

**答案**：因为前置组件一旦更换，输入范围假设就会失效，导致未被饱和保护的位置突然溢出。基于输入格式本身能推出的理论最大值才是稳健的（tips.md 标注为 IMPORTANT）。

### 4.3 AXI-S 握手与命名约定

#### 4.3.1 概念说明

psi_fix 全库统一采用 **AXI4-Stream（AXI-S）** 握手协议，由三根信号协同完成一次数据传递：

| AXI-S 标准名 | psi_fix 常见同义名 | 方向（相对当前组件） | 含义 |
| --- | --- | --- | --- |
| TDATA | `dat_i` / `dat_o`（或应用相关名） | 输入 / 输出 | 数据本体 |
| TVALID | `vld_i`（输入）、`vld_o`（输出）、`str_i` | 主→从 | 「本次数据有效」 |
| TREADY | `rdy_i`（从设备给出，作输入）、`rdy_o`（本组件给出，作输出） | 从→主 | 「我现在能接收」 |

关键时序规则（introduction.md 明确列出）：

- 一个时钟周期内 **TVALID 与 TREADY 同时为高**，数据才真正完成传递。
- **主设备不许等 TREADY 才拉 TVALID**——TVALID 必须自主声明。
- TVALID 一旦拉高，在握手完成前**必须保持**。
- **从设备可以等 TVALID 才拉 TREADY**，也可以提前拉高 TREADY 后又撤销。
- 不是所有组件都实现反压（backpressure），有些省略 TREADY；但凡实现了的，都遵循上述规则。

#### 4.3.2 核心流程

用一段伪时序描述一次握手：

```
cycle : vld(Master) | rdy(Slave) | 是否传递
  1   :     0       |     1      |  否（主没数据）
  2   :     1       |     0      |  否（从没准备好，主必须保持 vld=1）
  3   :     1       |     1      |  是 ← 本次传递发生
  4   :     1       |     1      |  是 ← 又传一次
  5   :     0       |     1      |  否（主没数据了）
```

可以看到：谁先拉高不重要，只要某一拍两者都为高，就传递。这与 AXI-S 标准完全一致。

#### 4.3.3 源码精读

introduction.md 的「Handshaking Signals」章节给出了完整定义：

[doc/files/introduction.md:104-118](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/doc/files/introduction.md#L104-L118) —— 明确「PSI 库使用 AXI4-Stream 握手」，并列出四条时序铁律：主不等从才拉 valid；valid 拉高后必须保持到握手；从可等 valid 才拉 ready；从可提前拉 ready 后再撤销。同一拍 TVALID 与 TREADY 都高才传递。

命名上，psi_fix **没有照搬** AXI-S 的 `TDATA/TVALID/TREADY` 大写名，而是用了更贴近硬件工程师习惯的小写同义词：

[doc/files/introduction.md:124-129](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/doc/files/introduction.md#L124-L129) —— 命名对照表：TDATA→`dat_i/dat_o`、TVALID→`vld_i/str_i`、TREADY→`rdy_i/rdy_o`。并特别说明：PSI 倾向用**多个有语义的数据信号**共享同一组握手，而不是 AXI-S 那样把所有数据打包进一个 TDATA 大向量——这提升了可读性。

方向后缀 `_i / _o / _g` 则在贡献规范里被列为强制项：

[doc/files/introduction.md:80-83](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/doc/files/introduction.md#L80-L83) —— 强制要求端口带后缀：输入 `_i`、输出 `_o`、generic `_g`；并要求用 snake_case（不得 camelCase）。

真实组件 `psi_fix_mov_avg` 的端口就是这套约定的范本：

[hdl/psi_fix_mov_avg.vhd:32-40](https://github.com/paulscherrerinstitute/psi_fix/blob/821049e0375cb950a74f0ccec4b4993d5d960899/hdl/psi_fix_mov_avg.vhd#L32-L40) —— 端口区：`clk_i / rst_i`（控制）、`dat_i / vld_i`（输入数据 + 输入有效）、`dat_o / vld_o`（输出数据 + 输出有效）。注意这个组件**没有 `rdy` 信号**——它省略了反压功能，符合 introduction.md「not all entities may implement all optional features」的说法。输入输出都只有 `vld`，表示「数据有效就处理，不等待」。

#### 4.3.4 代码实践

**实践目标**：在真实组件上识别握手信号与方向后缀，判断它是否实现了完整反压。

**操作步骤**：

1. 打开 `hdl/psi_fix_mov_avg.vhd` 的端口区（第 32-40 行）。
2. 逐个端口标注：它是 `_i` 还是 `_o`？它属于 TDATA / TVALID / TREADY 中的哪一类（或控制类）？
3. 检查是否存在 `rdy_i` 或 `rdy_o`。

**需要观察的现象 / 预期结果**：

| 端口 | 后缀（方向） | 语义类别 |
| --- | --- | --- |
| `clk_i` | `_i`（输入） | 控制信号 |
| `rst_i` | `_i`（输入） | 控制信号 |
| `dat_i` | `_i`（输入） | TDATA（数据） |
| `vld_i` | `_i`（输入） | TVALID |
| `dat_o` | `_o`（输出） | TDATA（数据） |
| `vld_o` | `_o`（输出） | TVALID |

结论：`mov_avg` **没有** TREADY 信号，因此它是一个「只做 valid、不做反压」的简化 AXI-S 端口——上游必须保证不会在它处理不过来时持续灌数据。这与 introduction.md 所说「可选特性可不实现」一致。

#### 4.3.5 小练习与答案

**练习 1**：一个组件端口名为 `dat_i / vld_i / rdy_o`。请说明这个组件扮演主还是从，以及握手如何完成。

**答案**：`rdy_o` 表示「本组件作为从设备，向主设备输出 ready」。主设备拉 `vld_i`，本组件准备好时拉 `rdy_o`，两者同拍为高即完成一次接收。本组件是从。

**练习 2**：为什么 AXI-S 规定「主不许等 TREADY 才拉 TVALID」？

**答案**：若双方都等对方先拉高，就会死锁（主等从的 ready，从等主的 valid）。所以规定 valid 必须自主、无条件先行声明，把主动权交给数据真正存在的一方。

**练习 3**：`out_regs_g` 这个名字里，`_g` 代表什么？它和 `out_fmt_g` 在性质上有何共同点？

**答案**：`_g` 代表 generic（编译期参数）。`out_regs_g`（输出寄存器级数）和 `out_fmt_g`（输出定点格式）都是 generic，都在**综合前**就固定下来，运行时不可改——这与运行时可变的端口信号（`dat_i` 等）形成对照。

## 6. 本讲小结

- psi_fix 用三元组 `[s, i, f]` 描述定点格式：\(s\) 符号位、\(i\) 整数位、\(f\) 小数位；总位宽 \(W = s + i + f\)，由 `psi_fix_size()` 在代码里动态算出。
- 有符号 `[1,i,f]` 的范围是 \([-2^i,\ 2^i - 2^{-f}]\)；记住「能表示 -1.0，但不能表示 +1.0」这个不对称性，它是乘法位增长的根因。
- 位增长三规则：加/减法整数位 +1；舍入再 +1；两个有符号数相乘整数位相加后再 +1（\([1,a,b]\times[1,c,d]=[1,a+c+1,b+d]\)）。
- 默认舍入/饱和是 `trunc / wrap`（最省资源、但最易溢出），所以 tips.md 反复强调只在必要处才做量化。
- psi_fix 全库用 AXI4-Stream 握手：`vld`（TVALID）+ `rdy`（TREADY）同拍为高才传递；`dat` 是 TDATA；许多组件省略 `rdy` 做无反压简化。
- 命名约定强制：输入 `_i`、输出 `_o`、generic `_g`，且一律 snake_case——文件名即索引、后缀即方向。
