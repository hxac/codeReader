# 顶层多流 testbench：psi_ms_daq_axi_tb

## 1. 本讲目标

学完本讲后，读者应该能够：

- 说清楚顶层 testbench `psi_ms_daq_axi_tb` 与前面 u5-l2 讲的模块级 TB（input/dma/sm）在**职责上的根本区别**：模块级 TB 只验证单个子模块，而顶层 TB 把整个 `psi_ms_daq_axi` IP 当作 DUT，验证「AXI 寄存器配置 → 多路流数据输入 → DMA 写 DDR → 中断回报 → 数据回读校验」的**端到端**链路。
- 理解顶层 TB 的两大支柱：用 `psi_tb_axi_pkg` 提供的 AXI 过程当「软件 CPU」驱动寄存器；用一块共享的字节数组 `Memory` 当「DDR 内存模型」接收并校验 DMA 写入。
- 掌握 `psi_ms_daq_axi_tb_str0_pkg` ~ `str3_pkg` 四个「流数据包」如何分别描述一路流的数据生成、窗口校验与中断处理，且四路覆盖了环形/线性、覆盖/不覆盖、四种记录模式的不同组合。
- 了解单流变体 `psi_ms_daq_axi_1s_tb` 为何单独存在，以及它与多流 TB 共享了哪些代码、又改了哪些 generic。

## 2. 前置知识

本讲是仿真验证单元（u5）的最后一篇，默认读者已经具备以下认知（若没有，建议先读对应讲义）：

- **u5-l1**：PsiSim 仿真骨架、`config.tcl`/`run.tcl`/`ciFlow.py` 如何组织一次回归、`###ERROR###` 失败约定。本讲的两个 TB（`psi_ms_daq_axi_tb`、`psi_ms_daq_axi_1s_tb`）就是在 `config.tcl` 里用 `create_tb_run` 注册的两次仿真。
- **u5-l2**：PSI 的 TbGen 风格「顶层 TB + tb_pkg + case 文件」三层骨架，以及 `ApplyStrData`/`CheckAcqData` 这类高层过程的思想。顶层多流 TB 同样是 TbGen 生成（见文件头注释 `Testbench generated by TbGen.py`），但**没有**再拆 case 文件——它的「场景」直接写在四个 `str*_pkg` 里。
- **u3-l5**：寄存器映射。本讲的 `axi_tb_pkg` 把 GCFG/IRQVEC/IRQENA/STRENA/MAXLVL/POSTTRIG/MODE/LASTWIN 以及上下文/窗口寄存器的地址重新声明了一遍（作为 TB 自己的「地址字典」），必须对照 u3-l5 才能理解这些地址的硬件含义。
- **u4-l1 / u4-l2**：硬件中断生成（IRQ FIFO、TfDone）与 C 驱动的 `HandleIrq`。顶层 TB 的 `IrqHandler` 过程其实是「用 VHDL 重写一遍最小的 CPU 中断服务例程」，逻辑与 C 驱动一一对应。

还需要几个 VHDL/PSI 库的基础概念：

- **`t_aslv8`**：`psi_common_array_pkg` 定义的「字节数组」类型，即 `array of std_logic_vector(7 downto 0)`。本讲用它实现一片**字节寻址**的伪 DDR。
- **`axi_ms_r` / `axi_sm_r`**：`psi_tb_axi_pkg` 定义的 AXI master/slave 记录类型（含 awaddr/wdata/bresp 等全部字段）。TB 把所有 AXI 信号打包成这两个记录，过程调用时只传「ms（请求）」和「sm（响应）」两个参数，避免几十根信号一一罗列。
- **`axi_single_write` / `axi_single_read` / `axi_single_expect`**：`psi_tb_axi_pkg` 提供的「单拍 AXI 读写」过程，封装了完整的 AW/W/B 或 AR/R 握手。

## 4. 核心概念与源码讲解

### 4.1 公共包：寄存器映射与共享 Memory 数组

#### 4.1.1 概念说明

顶层 TB 要同时扮演两个角色：**一是发起 AXI 访问的 CPU**（写寄存器配置 IP、读寄存器查状态、应答中断），**二是接收 DMA 写入的 DDR**。这两件事都需要一块「全局可见的地方」：

- DDR 侧需要一片所有进程都能读写的内存——这就是 `Memory`。
- CPU 侧需要一份「寄存器地址字典」，把可读的常量地址喂给 `axi_single_write` 等过程——这就是寄存器映射常量。

公共包 `psi_ms_daq_axi_tb_pkg` 把这两样东西集中放在一起，让四个 `str*_pkg` 和两个 TB 主体都能 `use` 它，从而共享同一份真相。

#### 4.1.2 核心流程

公共包的内容分四块，自顶向下：

1. **全局量**：`Memory` 数组、`MaxWindows_c`、打印开关。
2. **寄存器地址映射**：三段式地址（全局寄存器区、流上下文区、窗口上下文区）。
3. **底层 AXI 过程**：`AxiWrite32`/`AxiRead32`/`AxiExpect32`/`AxiWriteAndRead32`，薄封装 `psi_tb_axi_pkg`。
4. **高层 `Hl*` 过程**：把「写一个寄存器」提升为「配置一个流」「查一个窗口的样本数」等语义动作。

#### 4.1.3 源码精读

**共享 Memory 数组**——本讲最关键的对象。它是一片 64 KiB 的字节数组，声明为 `signal`（不是 variable），因此所有进程可见：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd:32-34](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd#L32-L34)

```vhdl
constant MemSize_c    : integer := 16#10000#;
signal   Memory       : t_aslv8(0 to MemSize_c-1);
constant MaxWindows_c : integer := 16;
```

- `MemSize_c = 0x10000`（65536 字节）定义伪 DDR 大小。
- `Memory` 是 `t_aslv8`，即「每个元素一个字节」。DMA 写入的地址直接当数组下标用，因此这是**字节寻址**模型，和真实 DDR 一致。
- `MaxWindows_c = 16` 是 TB 传给 DUT 的 `MaxWindows_g`（见后文 DUT 例化），同时驱动窗口上下文区的步长计算。

**寄存器地址映射**——TB 自己的「地址字典」。它把硬件 `reg_axi` 暴露的寄存器地址重述为常量，分三段：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd:41-78](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd#L41-L78)

关键常量含义（对照 u3-l5 的硬件寄存器）：

| 常量 | 值 | 含义 |
|------|----|------|
| `REG_CONF_GCFG_ADDR` | 0x000 | 全局配置（使能位 + IRQ 使能位，写 0x0101 同时开两者） |
| `REG_CONF_IRQVEC_ADDR` | 0x010 | 中断向量（读查中断源、写 1 清零） |
| `REG_CONF_IRQENA_ADDR` | 0x014 | 中断使能掩码 |
| `REG_CONF_STRENA_ADDR` | 0x020 | 各流录制使能 |
| `REG_CONF_Xn_STEP` | 0x10 | 每流寄存器步长（MAXLVL/POSTTRIG/MODE/LASTWIN 共用） |
| `REG_CONF_MODEn` | 0x200 | 每流 MODE 寄存器基址 |
| `REG_CTX_REGION` | 0x1000 | 流上下文区基址（SCFG/BUFSTART/WINSIZE/PTR） |
| `REG_CTX_Xn_STEP` | 0x20 | 每流上下文步长 |
| `REG_WIN_REGION` | 0x4000 | 窗口上下文区基址（WINCNT/WINLAST/TSLO/TSHI） |
| `REG_WIN_STRn_STEP` | `MaxWindows_c*0x10` | 每流窗口区步长（向上取整到 2 的幂，对照 u3-l4） |

`VAL_MODE_*` 与 `VAL_SCFG_*` 常量则是这些寄存器里各 bit 位的「语义编码」，例如 `VAL_MODE_ARM = 1*2**8`（MODE 寄存器 bit 8 = Arm）、`VAL_MODE_RECORDING = 1*2**16`（bit 16 只读回 IsRecording）。

**底层 AXI 过程**是对 `psi_tb_axi_pkg` 的薄封装，例如 `AxiWrite32` 直接调用 `axi_single_write`：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd:251-258](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd#L251-L258)

`AxiWriteAndRead32` 则「先写、等 400 ns、再 expect 读回」，用来确认寄存器真的写进去了——这是 TB 里最常见的自检手段。

**高层 `Hl*` 过程**把多步寄存器访问组合成语义动作。最典型的是 `HlConfStream`，它一次写完一个流的 BUFSTART、WINSIZE、SCFG 三个上下文寄存器，并把 ringbuf/overwrite/wincnt 三个布尔/整数拼成 SCFG 的位段：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd:322-346](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd#L322-L346)

注意 `v := v + (2**SFT_SCFG_WINCNT)*(wincnt-1)`——这里写的是 **wincnt-1**，与 C 驱动 `Str_Configure` 写 WINCNT 的约定完全一致（u1-l4 已点出）。这种「TB 与驱动用同一套位段语义」是端到端验证可信的基础。

另一个值得注意的过程是 `HlGetWinCnt` / `HlIsTrigWin`，它们读同一个 WINCNT 寄存器却给出两种语义：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd:408-432](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd#L408-L432)

- `HlGetWinCnt` 用 `IntAnd(v, MSK_WIN_WINCNT_CNT)`（掩码 0x7FFFFFFF）取低 31 位，得到窗口已写样本数。
- `HlIsTrigWin` 却判断 `v < 0`——因为 VHDL 里 `integer` 是 32 位有符号，WINCNT 的**最高位（bit 31）正是触发标志**（u3-l3、u3-l4 已说明），置 1 时整个 32 位整数变负。这是 TB 巧妙利用「有符号整数符号位 = 触发位」的惯用法。

#### 4.1.4 代码实践

**实践目标**：确认寄存器地址字典与硬件 `reg_axi` 的地址译码一致。

**操作步骤**：

1. 打开 [tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd:41-78](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_pkg.vhd#L41-L78)，抄下 `REG_CONF_MODEn = 0x208`、`REG_CONF_Xn_STEP = 0x10`。
2. 计算 stream 2 的 MODE 寄存器地址：`0x208 + 0x10*2 = 0x228`。
3. 打开 `hdl/psi_ms_daq_reg_axi.vhd`，找到每流寄存器的地址译码（对照 u3-l5 讲过的三段译码），确认 stream 2 的 MODE 偏移确实是 0x228。

**预期结果**：两边地址对齐。这是「TB 地址字典」与「硬件地址译码」互为镜像的体现；若不一致，TB 写的寄存器会落到错误位置，后续校验必然报 `###ERROR###`。

**需要观察的现象**：`REG_WIN_STRn_STEP = MaxWindows_c*0x10 = 16*16 = 256 = 0x100`。这与 u3-l4 讲的「窗口维向上取整到 2 的幂」一致——因为 `MaxWindows_c=16` 恰好是 2 的幂，所以流间的窗口区步长就是 `窗口数 × 每窗口字节数`。

### 4.2 顶层 TB 的进程骨架：stimulus、control 与内存模型

#### 4.2.1 概念说明

`psi_ms_daq_axi_tb.vhd` 的 `architecture sim` 里挂着一组并发进程，它们各自扮演系统中的一个角色，靠共享信号（`reg_axi_ms`/`reg_axi_sm`、`axi_ms`/`axi_sm`、`Memory`、`Irq`、`Str_*`）协同。理解顶层 TB 的关键是先认清**谁在驱动哪组信号**：

| 进程 | 角色 | 驱动的信号 |
|------|------|-----------|
| `i_dut` | 被测 IP 本身 | （DUT 例化） |
| `p_mem` | DDR 内存模型（AXI Slave 端） | `axi_sm`（M_AXI 的响应侧）、`Memory` |
| `p_clk_*` | 时钟生成 | `M_Axi_Aclk`、`S_Axi_Aclk`、`Str_Clk` |
| `p_regacc` | CPU：配置 + 中断服务 + 主循环 | `reg_axi_ms`（S_AXI 的请求侧） |
| `p_str0..3` | 4 路流数据源 | `Str_Vld`、`Str_Trig`、`Str*_Data` |
| `g_ts` | 时间戳计数器 | `Timestamp` |

这与真实 SoC 系统几乎一一对应：`p_regacc` 是 PS 上的 CPU 固件，`p_mem` 是 DDR，`p_str*` 是 PL 上的数据源，`i_dut` 是 DAQ IP。

#### 4.2.2 核心流程

顶层 TB 的端到端时序可以画成：

```
p_regacc:                      p_str0..3:               p_mem:                i_dut:
 释放复位 ────────────────────► (流源开始产出样本)                              (各模块脱离复位)
 写 IRQENA/STRENA/GCFG ──────►                                              (寄存器配置生效)
 调 Str*Setup 配置 4 路 ─────►                                              (上下文 RAM 被填)
 写 GCFG=0x0101 使能 ────────►                                              (状态机开始调度)
                              ┌─ 产数据 ──► Str_Vld/Trig ──► input 逻辑 ──► DMA ──► M_AXI 写 ──► p_mem 存入 Memory
 <── Irq='1' (有窗口完成) ◄───┤                                                          │
   调 IrqHandler:                                                                       │
     读 IRQVEC → 清 ──►                                                                │
     分发到 Str*Handler:                                                                │
       读 WINCNT/WINLAST/TS ──►                                                         │
       逐字节比对 Memory[...] ◄─────────────────────────────────────────────────────────┘
 循环 150 us 后停，断言 Str*WinCheck 次数
```

#### 4.2.3 源码精读

**DUT 例化**——顶层 TB 把整个 IP 例化为 `i_dut`，generic 决定了这是一颗「4 流、混合位宽、混合优先级」的 IP：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd:158-175](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L158-L175)

关键 generic 取值（对照 u1-l3）：

- `Streams_g => StrCount_c`，而 `StrCount_c = 4`（[L51](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L51)）。
- `StreamWidth_g => (0=>8, 1=>16, 2=>16, 3=>32)`：故意让 4 路位宽不同，覆盖字节拼字（u2-l2 的 `WconvFactor_c`）。
- `StreamPrio_g => (0=>1, 1=>3, 2=>2, 3=>2)`：优先级各不相同，专门验证 u3-l1 的三级仲裁。
- `IntDataWidth_g => 128`、`AxiDataWidth_g => DATA_WIDTH (=128)`：内部数据宽度与 AXI 数据宽度都设 128，验证 u4-l4 的可参数化数据宽度。
- `MaxWindows_g => MaxWindows_c (=16)`、`MaxBurstSize_g => 128`、`MinBurstSize_g => 16`。

**内存模型 `p_mem`**——这是「DDR」本身。它实现了一个最小的 AXI4 写通道 Slave：等 AW 握手 → 拉高 `wready` → 按 `awlen+1` 拍接收数据 → 按 `wstrb` 逐字节写入 `Memory` → 回 B 通道 OKAY 响应：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd:266-297](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L266-L297)

重点看写入那一小段：

```vhdl
Address_v := to_integer(unsigned(axi_ms.awaddr));
Size_v    := to_integer(unsigned(axi_ms.awlen))+1;
for qw in 0 to Size_v-1 loop
    wait until rising_edge(M_Axi_Aclk) and axi_ms.wvalid = '1';
    for byte in 0 to 7 loop
        if axi_ms.wstrb(byte) = '1' then
            Memory(Address_v+qw*8+byte) <= axi_ms.wdata(byte*8+7 downto byte*8);
        end if;
    end loop;
end loop;
```

- `Memory` 的下标是 `Address_v + qw*8 + byte`，即**绝对字节地址**，这正是各 `Str*Handler` 后续逐字节读回校验时所用的同一套地址。
- 末尾 `StdlCompare(1, axi_ms.wlast, ...)` 校验突发确实以 WLast 结尾——这是内存模型自带的一项协议自检，失败会打印 `###ERROR###`。
- 注意此处的 `for byte in 0 to 7` 每拍只搬运低 8 字节（64 位）；本 TB 的 `AxiDataWidth_g=128`，每拍实际携带 16 字节。**这是一个值得在本地波形里确认的简化点**（见 4.2.4），但不妨碍理解「Memory 是字节准确的接收端」这一核心事实。

**CPU 主进程 `p_regacc`**——TB 的总指挥，负责复位、配置、中断服务与收尾断言：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd:337-388](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L337-L388)

它的三段式结构：

1. **初始化与配置**（[L346-362](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L346-L362)）：释放复位 → `AxiExpect32(0x010, 0, ...)` 自检初始 IRQVEC 为 0 → 写 IRQENA=0x000F、STRENA=0x000F（开 4 路中断与录制）→ 依次调 `Str0Setup`..`Str3Setup` 配置每路 → 写 GCFG=0x0101 全局使能。
2. **运行主循环**（[L366-379](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L366-L379)）：循环 150 µs，每个 `S_Axi_Aclk` 上升沿检查 `Irq`，若拉高则调 `IrqHandler`，随后调各路 `Str*Update`（给状态机喂「适时 Arm」之类的边沿动作）。
3. **收尾断言**（[L383-386](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L383-L386)）：检查每路的 `Str*WinCheck`（成功校验过的窗口数）达到预期，否则报 `###ERROR###`。

**`IrqHandler` 过程**——用 VHDL 写的「最小 CPU 中断服务例程」，与 u4-l2 的 C 驱动 `PsiMsDaq_HandleIrq` 逐句对应：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd:124-151](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L124-L151)

它做的事：读 IRQVEC（`AxiRead32(0x010, v, ...)`）→ 把同一个值写回 IRQVEC（W1C 应答，拉低中断线）→ 遍历 4 位中断向量，哪位为 1 就调对应的 `Str0Handler`..`Str3Handler`。注意末尾还多等了 6 拍，确保中断线真的被清掉再返回（注释 `Delay to ensure IRQ is cleared`）。

**时钟生成**——三组时钟模拟真实 SoC 的异步时钟域：`M_Axi_Aclk` 200 MHz、`S_Axi_Aclk` 166 MHz、4 路 `Str_Clk` 分别 250/125/80/200 MHz（[L52](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L52)）。流时钟用 `generate` 每路一个进程（[L323-332](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L323-L332)），刻意不同频是为了压测 u2-l2 讲的跨时钟域 FIFO。

#### 4.2.4 代码实践

**实践目标**：把端到端流程与进程角色对上号，并定位「内存模型每拍搬运字节」这一简化点。

**操作步骤**：

1. 在 [psi_ms_daq_axi_tb.vhd:266-297](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L266-L297) 的 `p_mem` 里，标注出 AW 握手、W 数据接收、B 响应三段，指出哪一行把数据真正落进 `Memory`。
2. 跟着 `p_regacc`（[L351-362](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L351-L362)）列出一次完整的「CPU 配置序列」：写了哪几个地址、各写什么值、为何 GCFG 要最后写。
3. 用仿真器（Modelsim/Questa）跑 `psi_ms_daq_axi_tb`，在 `p_mem` 写 `Memory` 那一行设断点，观察一次 32 位样本（如 stream 0）落入的字节地址，再对照 `Str0Handler` 里读 `Memory(addr)` 的地址，确认两者指向同一字节。

**需要观察的现象**：

- GCFG=0x0101 必须在 `Str*Setup` 之后写——因为配置上下文时流还不应开始录制。
- `p_mem` 的内层循环 `for byte in 0 to 7` 只覆盖每拍的低 8 字节。在 `AxiDataWidth_g=128` 的配置下，每拍 W 数据有 16 字节，**高 8 字节是否进入 `Memory`、以及地址递增 `qw*8` 是否与 16 字节节拍对齐，需在本地波形确认**（待本地验证）。这正是「TB 内存模型是简化模型」的体现：它够用来校验本 TB 选定的数据图样，但并非一个全宽度精确的 DDR 模型。

**预期结果**：能在波形里看到 CPU 写寄存器 → DUT 在 `Str_Clk` 域收样 → `M_Axi_Aclk` 域发写突发 → `p_mem` 把字节写进 `Memory` → 中断拉高 → `IrqHandler` 读回校验的完整闭环。

### 4.3 str0~str3：四路流数据包如何定义「期望」

#### 4.3.1 概念说明

模块级 TB（u5-l2）的激励与校验写在 `tb_pkg` 的公共过程里；顶层 TB 不一样——它把**每一路流**的「数据怎么生成、窗口怎么校验、中断怎么处理」整体封装到一个独立的 `str*_pkg` 里。这样做的理由很直接：4 路流的配置（位宽、模式、缓冲策略）差异极大，混在一起会无法维护。

每个 `str*_pkg` 都提供四个过程，名字遵循同一约定：

| 过程 | 何时被调 | 干什么 |
|------|---------|--------|
| `StrNSetup` | `p_regacc` 初始化阶段 | 写该流的 POSTTRIG/MODE/上下文寄存器 |
| `StrNUpdate` | 主循环每拍 | 喂「适时 Arm」等边沿动作（可为空） |
| `StrNData` | 独立数据进程 | 在 `Str_Clk` 域驱动 `Vld`/`Trig`/`Data` |
| `StrNHandler` | `IrqHandler` 分发时 | 读窗口上下文、逐字节校验 `Memory` |

四个流刻意覆盖了不同的「配置矩阵」：

| 流 | 位宽 | 模式 | ringbuf | overwrite | 校验重点 |
|----|------|------|---------|-----------|---------|
| str0 | 8 | Continuous | 是 | 否 | 环形回绕、IRQ 使能时序、清窗口后继续录 |
| str1 | 16 | Manual | 否 | 否 | 成块数据、Arm 时机、首 100 样本在 Arm 之前 |
| str2 | 16 | SingleShot | 否 | 是 | 触发窗口、后触发样本、frame 号在高字节 |
| str3 | 32 | TriggerMask | 是 | 是 | 前/后触发解包、跨窗口回绕 |

#### 4.3.2 核心流程

以 str0 为例（其余结构相同），一个流数据包的内部状态机是：

```
Str0Setup:  写 MAXLVL 期望=0、POSTTRIG=3、MODE=Continuous、HlConfStream(ringbuf,win=3,size=100)
             ↓
Str0Sample: 每拍 data=(data+1) mod 256；15µs 后在 data=30/60/90 处发 trig 脉冲
             ↓ (硬件把窗口写满或遇触发 → 发 IRQ)
Str0Handler: 读 MAXLVL/CURWIN/LASTWIN → 对每个含触发的窗口：
               读 WINCNT/WINLAST/TSLO/TSHI → 按 winstart..winend 回绕逐字节比 Memory
               → HlClrWinCnt 释放窗口 → Str0NextWin++、Str0WinCheck++
```

注意 `Str0Handler` 末尾会调 `HlClrWinCnt`（写 WINCNT=0），这正是 u4-l2 讲的「MarkAsFree」的 TB 对应物——**窗口必须被软件释放，硬件才会继续往里写**（winOverwrite=false 时的窗口保护，u4-l5）。

校验逻辑 `Str0Handler` 用一个 `while ... loop` 遍历所有「已完成且含触发」的窗口，对每个窗口按 `winstart..winend` **环形回绕**地比对字节：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str0_pkg.vhd:165-223](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str0_pkg.vhd#L165-L223)

注意回绕那两行：

```vhdl
if addr = winend then
    addr := winstart;     -- 环形：到窗口尾绕回头
else
    addr := addr + 1;
end if;
```

这正是 u4-l3「环形缓冲解包」在 TB 侧的镜像：因为 str0 是 ringbuf，写指针会在窗口内回绕，校验时也必须按同一回绕规则读。每个 case（`when 0/1/2/3`）检查不同的现象：窗口写满、连续触发的时间戳差、清空后再次写满并最终禁用该流（`Str0Disabled := true`，并清 IRQENA/STRENA 的相应位）。

**str1——线性 + Manual + 成块数据**。数据以「50 样本突发 + 1 µs 间隔」的方式到达，无触发；校验时确认「前 100 个样本是在 Arm 之前到达、不该被记录」：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str1_pkg.vhd:139-144](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str1_pkg.vhd#L139-L144)

`StdlvCompareInt(spl+100, ...)`——期望值是 `spl+100`，说明第 `spl` 个被记录的样本其实是数据源的第 `spl+100` 个样本（前 100 个因未 Arm 而丢弃）。Arm 时机由 `Str1Update` 在 `Str1DataCnt = 99` 时触发（[L172-175](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str1_pkg.vhd#L172-L175)）。

**str2——线性 + 覆盖 + SingleShot + 触发**。每帧 200 样本、帧首发触发；因为 `overwrite=true`，窗口可被覆盖重写。校验时按后触发样本数倒着读，且每样本占 2 字节（16 位）、高字节存帧号：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str2_pkg.vhd:161-170](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str2_pkg.vhd#L161-L170)

`Memory(spladdr)` 是低字节（CNT）、`Memory(spladdr+1)` 是高字节（FRAME），地址步进 `spladdr-2`。

**str3——环形 + 覆盖 + TriggerMask + 32 位**。最复杂的一路：32 位样本占 4 字节，校验**跨窗口**的前/后触发解包，且窗口内地址回绕：

[tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str3_pkg.vhd:151-183](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str3_pkg.vhd#L151-L183)

回绕规则（注意是按样本步进 4 字节回绕）：

```vhdl
if spladdr = Str3BufStart_c+curwin*Str3WinSize_c then
    spladdr := Str3BufStart_c+(curwin+1)*Str3WinSize_c-4;   -- 绕到窗口尾
else
    spladdr := spladdr - 4;
end if;
```

这段先校验后触发样本（本窗口），再继续往前校验前触发样本（上一窗口），把 u4-l3「两段拼接解包」的环形几何在 TB 侧落到了实处。

**实践目标**：把四路流的「配置矩阵」与各自校验重点对齐。

**操作步骤**：

1. 在四个 `str*_pkg` 里分别找到 `StrNSetup`，抄下每路的 `HlSetMode(...)` 与 `HlConfStream(ringbuf=>..., overwrite=>..., wincnt=>..., winsize=>...)` 取值，填入 4.3.1 的表格。

**预期结果**：能说清「为什么必须有四路覆盖不同配置」——单一路无法覆盖环形回绕、覆盖写、四种记录模式、不同位宽的字节拼字等所有路径。

### 4.4 单流变体 psi_ms_daq_axi_1s_tb

#### 4.4.1 概念说明

`psi_ms_daq_axi_1s_tb` 是顶层 TB 的「精简版」：`Streams_g=1`、只用 str0、AXI 数据宽度 64。它存在的意义是验证**单流、窄总线**这一最常见部署配置下 IP 仍正常工作——多流 TB 的 `Streams_g=4`、`AxiDataWidth_g=128` 是高配，不能代表所有用户场景。

它**复用**多流 TB 的公共包 `psi_ms_daq_axi_tb_pkg`（共享同一份 `Memory`、寄存器地址字典、`Hl*` 过程），但**自带**一个 `psi_ms_daq_axi_1s_tb_str0_pkg`（路径在 `tb/psi_ms_daq_axi_1s/`，名字带 `_1s_`），因为这个 str0 的内存布局/校验细节与多流版可能不同。

#### 4.4.2 核心流程

单流变体的进程骨架与多流版**几乎逐行相同**：同样有 `p_mem`、`p_regacc`、`p_str0`、`g_ts`、时钟进程，同样有 `IrqHandler` 过程。差别只在三处：

1. `StrCount_c = 1`、`ClkFreq_c = (0=>250.0e6)`（只有一路时钟）。
2. `DATA_WIDTH = 64`（AXI 总线 64 位），且 DUT 例化**不传** `IntDataWidth_g` 与 `AxiDataWidth_g`——直接用顶层默认值。
3. `IrqHandler` 的 `case` 只剩 `when 0 => Str0Handler`，`p_regacc` 只调 `Str0Setup`/`Str0Update`，IRQENA/STRENA 只写 `0x0001`。

1. 并排打开 [psi_ms_daq_axi_tb.vhd](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd) 与 [psi_ms_daq_axi_1s_tb.vhd](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi_1s/psi_ms_daq_axi_1s_tb.vhd)。
2. 列出所有差异点：`StrCount_c`、`ClkFreq_c`、`DATA_WIDTH`、DUT generic map（哪些被省略）、`IrqHandler` 的 case 分支数、`p_regacc` 调用的 `Str*Setup`/`Str*Update` 数量、IRQENA/STRENA 写入值。

**需要观察的现象**：单流版 `p_regacc` 里 `AxiWriteAndRead32(0x0014, 0x0001, ...)` 与 `0x0020` 只写最低位（只开 stream 0），而多流版写 `0x000F`（开 4 路）。

**预期结果**：能总结出「单流 TB 是多流 TB 的子集」——除 generic 与位宽外，骨架完全相同。这也是 TbGen 生成器的价值：同一模板产出不同规模的 TB。

#### 4.4.5 小练习与答案

**练习 1**：顶层 TB 为何要把 `Memory` 声明成 `signal` 而不是普通 `variable`？

**参考答案**：`Memory` 要被 `p_mem`（写）和多个 `Str*Handler`（读）这些**不同进程**访问。VHDL 里 `variable` 只在单个进程内可见；只有 `signal` 能跨进程共享（`signal` 在 `architecture` 的声明区里对所有进程可见）。把它放公共包里声明为 `signal`，是为了让 `p_mem` 写入的字节立刻对所有 handler 进程可见，从而实现「内存模型」语义。

**练习 2**：`HlIsTrigWin` 用 `v < 0` 判断一个窗口是否含触发，这依赖什么前提？

**参考答案**：依赖 WINCNT 寄存器的**最高位（bit 31）是触发标志**（u3-l3、u3-l4），且 VHDL 的 `integer` 是 32 位**有符号**类型。`axi_single_read` 把 32 位读回值转成 `integer`，当 bit 31=1 时该整数就是负数，于是 `v < 0` 等价于「触发位置 1」。若把 WINCNT 改成 `unsigned` 或用 `std_logic_vector` 比较，这个惯用法就会失效。

**练习 3**：如果要让多流 TB 增加第 5 路流（stream 4），至少要改哪些文件、哪些位置？

**参考答案**：①新增 `tb/psi_ms_daq_axi/psi_ms_daq_axi_tb_str4_pkg.vhd`（仿 str0~str3，定义 `Str4Setup/Update/Data/Handler` 与 `Memory4` 别名，并分配一段不冲突的 `Str4BufStart_c`）；②在 `psi_ms_daq_axi_tb.vhd` 的 `use` 区加 `use work.psi_ms_daq_axi_tb_str4_pkg.all`（[L29-32](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L29-L32) 后追加）；③把 `StrCount_c` 从 4 改成 5（[L51](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/tb/psi_ms_daq_axi/psi_ms_daq_axi_tb.vhd#L51)），并在 DUT 的 `StreamWidth_g/StreamPrio_g/StreamBuffer_g/StreamTimeout_g/StreamTsFifoDepth_g/StreamUseTs_g` 数组里补索引 4；④新增 `Str4_Data` 信号、`p_str4` 数据进程、port map 里 `Str_Data(4)(...)`；⑤`IrqHandler` 的 `case` 加 `when 4 => Str4Handler`、`p_regacc` 加 `Str4Setup`/`Str4Update`、收尾断言加 `Str4WinCheck` 检查、IRQENA/STRENA 掩码从 `0x000F` 改成 `0x001F`；⑥在 `sim/config.tcl` 的 `tb` 组里 `add_sources` 追加新包路径（[L87-94](https://github.com/paulscherrerinstitute/psi_multi_stream_daq/blob/f4c0185f318d9f7cc9fa8edd109b00cc1764dcf9/sim/config.tcl#L87-L94)）。

## 6. 本讲小结

- 顶层 TB `psi_ms_daq_axi_tb` 以整个 `psi_ms_daq_axi` IP 为 DUT，验证「AXI 配置 → 多路流输入 → DMA 写 DDR → 中断 → 回读校验」的端到端链路，与 u5-l2 的模块级 TB 互补。
- 公共包 `psi_ms_daq_axi_tb_pkg` 提供两大支柱：一块所有进程共享的字节数组 `Memory`（伪 DDR），以及一套与硬件 `reg_axi` 地址译码互为镜像的寄存器地址字典和 `Hl*` 高层过程。
- `p_regacc` 是「CPU」，`p_mem` 是「DDR 内存模型」（字节准确、带 WLast 自检），`p_str0..3` 是 4 路异步流数据源，`IrqHandler` 是用 VHDL 重写的最小中断服务例程，与 C 驱动 `HandleIrq` 逐句对应。
- 四个 `str*_pkg` 各封装一路流的数据生成、Setup、Update 与 Handler，刻意覆盖环形/线性、覆盖/不覆盖、Continuous/Manual/SingleShot/TriggerMask 四种模式与 8/16/32 三种位宽的不同组合。
- `HlIsTrigWin` 用 `v < 0` 判触发位、`HlConfStream` 写 `wincnt-1` 等惯用法，与 C 驱动和硬件寄存器语义完全对齐——这是端到端验证可信的根基。
- 单流变体 `psi_ms_daq_axi_1s_tb` 复用公共包、自带 str0 包，`Streams_g=1`、AXI 宽度 64、DUT 用默认 `IntDataWidth_g`，验证最常见精简配置；其进程骨架与多流版几乎逐行相同，是 TbGen 同模板不同规模的产物。
