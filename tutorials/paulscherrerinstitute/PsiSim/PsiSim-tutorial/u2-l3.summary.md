本讲围绕 `PsiSim.tcl` 讲解测试运行的数据模型 `TbRuns`，它是与 `Sources`（描述「要编译什么」）平行的另一个 list-of-dict 结构，描述「要仿真什么、怎么仿真」。PsiSim 用两个变量管理测试运行：`ThisTbRun` 是「草稿/编辑缓冲区」，`TbRuns` 是「定稿登记表」。`add_tb_run` 依赖 TCL 的按值复制把草稿快照进登记表，故后续 `create_tb_run` 整体重建草稿不会影响已入库的 run。定义一个 run 采用两段式编程模型：`create_tb_run`（用 `dict create` 一次性写满 11 个字段默认值）→ 可选配置命令 → `add_tb_run`（定稿）；配置命令必须夹在中间，这条契约靠数据流隐式维护而非显式校验，漏写 `add_tb_run` 或事后改字段都会静默丢失配置。`ThisTbRun` 的 11 个字段为：`TB_NAME`、`TB_LIB`（默认取 `CurrentLib`）、`TB_ARGS`（默认含一个空串的 list，保证至少跑一次）、`PRESCRIPT_CMD/PATH/ARGS`、`POSTSCRIPT_CMD/PATH/ARGS`、`TIME_LIMIT`（`"None"` 表示跑到底）、`SKIP`。核心结论：单个 run 触发的仿真次数等于 `len(TB_ARGS)`；前后脚本把所有参数组「包」在一起、只各跑一次（而非每组各跑一次）；`tb_run_skip` 用 `lsearch` 匹配当前 `Simulator`，缺省值为 `"all"`（不传参会静默全跳过），且大小写敏感，须用 `"GHDL"`/`"Vivado"`/`"Modelsim"`/`"all"`。这些字段与运行逻辑由下一篇 u2-l6 的 `run_tb` 消费。
