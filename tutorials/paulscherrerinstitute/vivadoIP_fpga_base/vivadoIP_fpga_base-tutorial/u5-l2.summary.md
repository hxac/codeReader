本讲承接 u5-l1，回答两个遗留问题：`base_addr` 从何而来、`fpga_base.c` 如何变为可链接库，并剖析三个均无运行期逻辑的装配文件。`src/Makefile` 是被 Vitis 注入变量的编译模板：`COMPILER`/`ARCHIVER`/`*_FLAGS` 故意留空、由 Vitis 按 CPU（mb-gcc 等）回填，用模式规则把 `.c` 编成 `RELEASEDIR` 下的 `.o`、把 `.h` 拷进 BSP 的 `include/`；新版不再自行用 `$(ARCHIVER) -r` 归档 `libxil.a`，改由顶层 BSP Makefile 统一归档（为兼容新 Vitis）。`data/fpga_base.tcl` 的 `generate` 调用 Xilinx 自带的 `xdefine_include_file`，为每个实例在 `xparameters.h` 写出 `XPAR_FPGA_BASE_*` 四类宏（`NUM_INSTANCES`/`DEVICE_ID`/`C_BASEADDR`/`C_HIGHADDR`），此即 `base_addr` 的真正来源。`data/fpga_base.mdd` 是 PSF 驱动描述文件，其 `supported_peripherals` 是 Vitis“为外设找驱动”的匹配键。三者经 `component.xml` 的 `xilinx_softwaredriver_view_fileset` 打包进 IP，构成“识别→生成地址→编译装配”完整自动化链路。核心约定：`fpga_base` 这个名字须在 `component.xml`、`.mdd`、`.tcl`、entity 四处一致，且驱动库版本 1.0 独立于 IP 版本 1.4。
