本讲是「版本管理与仓库维护」单元第一讲，只读 `Changelog.md` 与 `.gitmodules` 两个文件，聚焦集合仓库 `psi_fpga_all` 如何固定子模块版本。核心结论有三点。其一，`Changelog.md` 按 release（形如 `## 2020.2 (20.10.2020)`）分段，段内按 `TCL`/`VHDL`/`Python`/`VivadoIP` 四大类分组，每行一个子模块名加版本号。release 分两种记录模式：**完整快照**（如 `2020.2`、`2019.4`，列出几乎全部库）与**增量记录**（如 `2021.1`、`2020.1`，只记本次升级的库）；某 release 没列某库不等于不存在，需向下回溯最近一次记录。其二，版本号多为 SemVer（`MAJOR.MINOR.PATCH`），须按点分段转整数比较，字符串比较会出错（如 `2.7.1 < 2.13.0`）；少数老库用旧式 tag，`en_cl_fix` 则是带 Original Location 的 fork。其三，发布策略为「整体快照 + 个别更新」：约每 3 个月（实际 3~10 个月浮动）发布一组协调一致、互相兼容的指针，两次发布间不保证各子模块最新；不保证最新的根本原因是用版本固定换取兼容性，用户可单独更新某子模块并自担风险。最后需分清两套事实来源：`Changelog.md` 是人工维护的人类账本，父仓库 gitlink（mode 160000）才是机器认定的真实指针，单独更新只改 gitlink、不自动改 Changelog，故两者可能不一致。
