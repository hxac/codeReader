本讲解析 cuTile 编译期显式定序访存的 `token_order_pass`。核心动因是：纯计算靠 SSA def-use 边天然定序，而访存操作之间没有这条边，GPU 内存模型下后端可能任意重排导致「写—读—写」语义出错。cuTile 的解法是给每个访存操作配一个输入 token 和输出 token，用 `MakeToken`（根 token）、`JoinTokens`（扇入合并）把访存串成 **memory token chain**；带 token 的 `TileLoad`/`TileStore`/`LoadPointer`/`StorePointer`/`TileAtomic*` 统称 **tko 操作**，其语义是「输入 token 就绪才能执行」。pass 分两阶段：先用 `_get_block_memory_effects` 自底向上汇总每个块的 `MemoryEffects`（按 `alias_set` 归类效应、acquire 位取或），再用 `token_map`（键为 `AliasTokenKey(alias_set, LAST_OP|LAST_STORE)` 与 `ACQUIRE_TOKEN_KEY`）递归穿线。关键定序规则：load 只等上一条 store（连续 load 可并行），store/原子等上一条任意访存（防 RAW/WAR/WAW）；`_collect_join_tokens` 决定何时插入 join。token 像携带值一样流过循环（LOAD 加 1 个、STORE 加 2 个携带 token）、分支（合并两支效应后分配结果 token）与终止符。另有 **循环并行 store 优化**：LayerNorm/RMSNorm 模式（单一 store、索引为归纳变量的单射、不相交、非 ALIAS_UNIVERSE）被特判为只依赖循环前 token，解开串行。调试开关 `CUDA_TILE_TESTING_DISABLE_TOKEN_ORDER` 可跳过本 pass 用于二分定位，但生产环境依赖性访存可能出错。本 pass 紧随 `add_divby_pass`、共享 u6-l2 的 `dataflow_result`，且必须早于 `hoist_loop_invariants`。
