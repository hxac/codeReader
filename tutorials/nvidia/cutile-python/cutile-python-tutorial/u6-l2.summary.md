本讲围绕 cuTile 优化流水线中的「数据分析 + 信息注入」展开，核心是数据流分析与整除性传播。为支持 GPU 访存对齐优化，编译器需在编译期知道每个变量「至少能被谁整除」。

`dataflow_analysis` 为每个 IR 变量维护一个 `DataPredicate`，刻画两个维度：`alias_set`（指针别名位集，供后续 token 排序）与 `div_by`（整除性，供对齐优化）。分析采用「种子 + 传播 + 不动点」三步：从参数约束写入初始谓词，按操作类型套用代数传播规则（整数加/减取 gcd、乘取乘积、abs/neg 保持、整字面量取绝对值；浮点不参与），用 `_Tracker.update` 的单调 `unify`（仅当信息变化才置 `dirty`）反复扫描直到稳定。整除性沿 def-use 链流动，循环携带值需多轮才能收敛。

种子来自 `ArrayConstraint`，一次生成 1+2·ndim 个谓词；静态形状/步长常量被直接折叠为 `div_by`（shape_constant folding），无需特化 IR 即可吃对齐红利。嵌套 `TupleConstraint` 由 `_register_tuple_params` 递归展开，用 offset 游标在扁平 Var 序列上对齐叶子；list 消费 2 个 Var。

`add_divby_pass` 只对访存操作（MakeTensorView/LoadPointer/StorePointer）的输入，用 `divisor & -divisor` 提取最大 2 的幂因子（封顶 1024），物化为 `AssumeDivBy` 操作并重映射操作数。用户也可用 `ct.assume_divisible_by` 手动注入；若被断言的已知整常量不整除，编译期抛 `TileTypeError`。流水线顺序为 dataflow → add_divby → token_order，三者共享同一份 `dataflow_result`。
