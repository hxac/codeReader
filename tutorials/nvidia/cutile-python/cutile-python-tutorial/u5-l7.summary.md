本讲把 cuTile 的用户 API 与底层 Tile IR 缝合起来，核心是「stub 签名 + impl 实现」的注册与分派机制。`@stub` 装饰器只是给函数贴上 `_cutile_python_stub` 标志，提供签名与文档占位，函数体在编译期从不执行；`is_stub` 沿 `__wrapped__` 链判定。真正的 IR 实现由 `@impl(stub)` 挂到 `ImplRegistry` 上，该容器持有 `op_implementations`（唯一实现）、`_overloaded_implementations`（按操作数类型的重载实现）与 `unflatten_aggregate_implementations` 三张表，并通过线程局部的 `get_current()`/`as_current()` 提供当前注册表。`@impl` 支持 `fixed_args`（用 `partial` 预绑参数，一份实现服务多个 stub）、`overload`（声明重载模式）、`min_version`（版本门控），并经 `_verify_params_match` 强制 stub 与 impl 签名对齐、记录错误上下文。

`tile_impl_registry` 在 `ops.py` 浅合并 core / static_eval / arithmetic / control_flow 四个键空间正交的子注册表。多态由 `overload_dispatcher` 与 `WILDCARD` 实现：被装饰的生成器 yield 操作数类型对作为重载键，`_find_overload` 按非 WILDCARD 个数（优先级）选最具体重载——`operator.add` 对 tile 做算术、对 tuple 做拼接即此机制。端到端追踪显示，`a + b`（经 ast2hir 映射为 `operator.add` 后走重载分派）与 `ct.add(a, b)`（直接命中 stub 唯一实现）两条路径在 `binary_arithmetic_tensorlike` 汇合，最终发射 `RawBinaryArithmeticOperation`；常量操作数则在编译期折叠，不下沉到 raw 操作。此外，`build_tuple` 与 `loosely_typed_const` 的 `result_var` 参数可复用入口变量充当结果，省去多余赋值。
