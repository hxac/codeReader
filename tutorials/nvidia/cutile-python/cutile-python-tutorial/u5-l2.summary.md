本讲拆解 cuTile 编译前端总入口 `compile_tile`（位于 `_compile.py`，被 `kernel._compile` 调用，外层包全局 `RLock` 保证并发安全）。其主体是「线性向下 + 三处短路 return」的流水线四段：前端准备（规范化 `AnnotatedFunction`、补 mangled symbol、探测 `sm_arch` / `bytecode_version`）→ AST→HIR（`get_function_hir` 只做一次，与 signature 无关）→ IR 生成 → 后端（IR→字节码→cubin）→ 磁盘缓存。三个开关 `return_cubin` / `return_bytecode` / `return_final_ir` 决定短路位置，使同一函数既服务 JIT、AOT 又服务调试。

关键概念：`_IrKeeper` 按 signature 懒生成并 memo final IR，`keep_all` 控制是否记忆；`_create_kernel_parameters` 本次重写为递归版，以 `ParameterAnnotationNode` 注解树取代旧的扁平 `constant_parameter_mask`，让 constraint×annotation 并行递归——tuple 参数「先拆后装」（`build_tuple`）、Constant 烘焙为 `loosely_typed_const` 并从运行时签名消失、静态形状经 `_resolve_static_shape_axes` 折进 `ArrayTy.shape`；`_transform_ir` 是固定顺序的优化 pass 流水线（核心约束：循环不变量外提必须在 token 排序之后，数据流分析结果被 divby 与 token_order 共享）；`compile_cubin` 调用外部 `tileiras` 子进程生成 cubin，SQLite 缓存（key 含编译器版本/架构/优化等级/字节码/device-debug）命中即跳过编译。最关键的洞见：IR 生成是被后端步骤「按需拉动」的，而非前端主动完成。
