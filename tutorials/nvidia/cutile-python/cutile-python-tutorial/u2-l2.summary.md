本讲承接 u2-l1 执行模型，回答「block 算的是什么数据」——cuTile 的根本数据单元是**全局数组（global array）**，一个 array-based、不暴露指针的模型，带来边界安全、能高效降级到 TMA 等硬件访存、并与 NumPy 习惯一致。

三个核心模块：**（1）全局数组**——放在 GPU 全局显存、由 host 分配的可读写多维数组，核心属性为 `shape`/`strides`/`dtype`。关键所有权规则：**新数组只能由 host 分配并作为参数传入 `ct.launch`；tile 代码内只能对已有数组创建视图（如 `Array.slice`、`tiled_view`），绝不拷贝、绝不新建底层存储**；多个数组参数的显存不得重叠。其中 `shape`/`strides` 是运行时 `int32`（性能考量，单维上限约 21 亿），`dtype`/`ndim` 是编译期常量，这决定了各自能用在哪。**（2）strided 内存布局**——逻辑下标经地址公式 `base_addr + sizeof(dtype)·Σ(stride_k·index_k)` 映射到物理地址；cuTile 的 strides 以**元素**为单位，而外部 CUDA Array Interface 以字节为单位，C++ 桥接层用 `stride/dtype_bytewidth` 换算，缺省时按行优先 `fill_row_major_strides` 补全。**（3）外部张量入口**——cuTile 自身不分配 GPU 张量，宿主框架张量经 **DLPack** 与 **CUDA Array Interface** 两套零拷贝协议进入内核；`classify_arg` 按 torch 快路径（`TorchTensorDlpack`）→ `__dlpack__`（`DlpackArray`）→ `__cuda_array_interface__`（`CudaArray`）的优先级判定，三类最终统一归一为同一种内部 `Array`；DLPack 路径强制校验设备为 CUDA，CPU 数组会被拒绝。这些术语（global array、strides、视图、DLPack/CAI、`ParameterKind::Array`）将在 u2-l3（tile）及后续 load/store 范式中被承接。
