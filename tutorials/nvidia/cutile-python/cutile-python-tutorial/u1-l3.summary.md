本讲俯瞰 cuTile Python 仓库的结构与构建系统。顶层目录分家清晰：`src/` 存纯 Python 编译器源码，`cext/` 存 C++ 运行时桥接源码，`docs/`/`samples/`/`test/`/`experimental/` 分别装文档、示例、测试与独立打包的实验功能。`src/cuda/tile/` 的子包归为「前端 / 优化 / 后端 / 运行时」四类：`_passes` 同时承载前端降级（ast2hir/hir2ir）与优化 pass；`_ir` 是贯穿三类的 Tile IR 骨架；`_bytecode`/`compilation`/`_cache` 属后端；`_execution`/`_context`/`tune`/`jax`/`_cext` 属运行时。关键术语包括 src 布局、命名空间包、可编辑构建。构建链为三层接力：`pyproject.toml` 用显式 `include` 白名单决定打包哪些子包（版本从 `VERSION` 文件单一来源读取）；`setup.py` 的 `BuildExtWithCmake` 拦截 `build_ext` 调用 CMake，顶层 `CMakeLists.txt` 找 Python/CUDA Toolkit/DLPack/XLA，`cext/CMakeLists.txt` 先编 `_cext_static` 静态库再编 `_cext` MODULE。可编辑构建以符号链接把 `.so` 链回源码树，使重编即生效。核心结论：`_cext` 是 Python 运行时与 CUDA Driver API 之间唯一的桥，`launch`/`TileDispatcher`/`TileContext`/`CallingConvention` 均经它暴露。
