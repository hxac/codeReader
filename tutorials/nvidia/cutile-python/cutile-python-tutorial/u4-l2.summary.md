本讲承接 u4-l1（TiledView 与 persistent 遍历），讲解 cuTile 的非常规索引 API，解决 `load`/`store` 只能取「网格对齐矩形块」的局限，覆盖三个最小模块：`gather`、`scatter`、`load_advanced_indexing`（`store_advanced_indexing` 成对出现）。

`ct.gather`/`ct.scatter` 是最通用的逐点（pointwise）元素空间寻址：`indices` 是长度等于数组秩的元组，各整数 tile 分量按 NumPy 规则广播，结果 tile 形状即广播形状；1D 数组可省略元组直接传单 tile。`mask`、`padding_value`、`check_bounds` 三者通过「有效掩码」协同——`gather` 在越界或 `mask=False` 处返回 `padding_value`，`scatter` 则静默不写；`check_bounds=False` 关闭运行时边界掩码（更快但越界是 UB）。关键陷阱：`scatter` 的重复下标属未定义行为（硬件并行写入顺序不定），直方图等多对一累加必须用 `ct.atomic_add`（u4-l3）。

`ct.load_advanced_indexing`/`store_advanced_indexing` 是「一维稀疏 + 多维连续切片」混合索引：元组中恰好一个 1D 整数 tile 为稀疏维，其余稠密维用 `ct.Slice(start, length)`（length 须为 2 的幂编译期常量，start 可为运行时值）描述连续区间，稠密段仍可走 TMA/合并访存，典型用途是 embedding 查表。

核心结论：读操作的重复下标恒良定义（幂等），写操作的重复下标恒 UB；访存规整度 `load` > `load_advanced_indexing` > `gather`，能规整就规整、一维任意才用 advanced indexing、全逐点才用 gather。
