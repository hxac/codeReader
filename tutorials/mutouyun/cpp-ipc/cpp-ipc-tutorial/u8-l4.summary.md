本讲从应用层视角拆解 libipc 三个进阶 demo，提炼出可复用的业务套路。最小模块有四：(1) chat 群聊靠「两条同名 channel、一条 sender、一条 receiver」实现全双工；由于广播会把消息回声给发送者自己的 receiver，必须用消息里的发送者 id 做「自我过滤」。(2) 跨进程唯一 id 只需把一个 `std::atomic<std::size_t>` 放进 `ipc::shm::handle`，用 `fetch_add` 即得单调递增唯一号，`memory_order_relaxed` 足矣。(3) service/client 请求-响应用「两条命名广播通道、角色互换」拼出——服务端「写 r 读 w」、客户端「读 r 写 w」，本质是把两条单向通道对偶。(4) Windows 跨 Session（Session 0 服务 ↔ 用户会话客户端）需用 `ipc::prefix{"Global\\"}`，经 `make_prefix` 拼进底层共享内存对象名、落入全局命名空间，两端 prefix 必须逐字一致。关键术语：自我过滤、`fetch_add` 唯一 id、角色互换、`prefix` 标签、`reconnect` 重试。核心结论：同名即同一通道、构造即连接、析构即销毁，复杂业务皆是这几条原语的组合；Linux 客户端 recv 空即退出偏脆弱，Windows 版用 reconnect 更健壮。
