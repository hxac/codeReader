本讲讲解 `_schedule_next_batch` 的 decode 半边及两个底层管家。最小模块有四：`DecodeManager` 是极简 dataclass，仅靠 `running_reqs: Set[Req]` 维护「正在 decode 的请求」，自身不含 tensor，体现调度逻辑与张量数据分离；它用 `filter_reqs`（`union` 后按 `can_decode` 过滤）一行完成新请求加入与已结束请求剔除，因 `ChunkedReq` 的 `can_decode` 恒为 `False` 而天然被挡在外。其只读属性 `inflight_tokens = Σ remain_len + (page_size-1)×请求数` 为在途 decode 向未来预留 KV 空间，作为 prefill 的 `reserved_size` 初值以防显存吃光。核心命门是 decode 批必须按 `uid` 排序——`set` 迭代序不确定，多 rank 下会导致 batch 顺序分叉、`all_reduce` 维度错位，提交 `#113` 正是修复此问题。`TableManager` 是 `table_idx` 行号发号器，`allocate` 即 `pop`、`free` 即 `append`，LIFO、O(1)。`page_table`（K/V 寻址）与 `token_pool`（token id 草稿）同形 `(max_running_req+1, aligned_max_seq_len)`，行即请求、列即序列位置；`dummy_req` 复用 `token_pool` 故须零初始化。`max_running_req` 同时约束槽位数与真实请求行数且恒等，`+1` 行专供 dummy。
