本讲讲解 CPU 侧的 tokenizer/detokenizer worker 进程，它把用户文字与模型 token 互转，与 GPU 计算解耦。核心设计是 `tokenize_worker`「一个函数两种身份」：不区分自己是哪种角色，只按收到的消息类型分流。四个最小模块：`TokenizeManager` 负责入向，将 `TokenizeMsg`（裸字符串或对话消息列表，后者套 chat template）encode 成一维 int32 张量，产出 `UserMsg` 送 scheduler；`DetokenizeManager` 负责出向，用 `DecodeStatus` 的三偏移量（`read_offset`/`surr_offset`/`sent_offset`）状态机做流式增量 decode，靠 `endswith("�")` 与 `find_printable_text` 解决中文/emoji 的「半个字」问题，产出 `UserReply` 回 api_server，按 `uid` 维护状态、`finished` 时清理；`AbortMsg` 仅转译为 `AbortBackendMsg`，不经任何 manager。主循环以 `local_bs` 做攒批阈值（当前写死为 1，实际逐条处理，批量 tokenize 为 TODO）。`num_tokenizer` 决定 ZMQ 拓扑：默认 0 为共享模式（单 worker 自 bind 合并地址，同 socket 收两种消息）；大于 0 为独立模式（1 个 detokenizer + N 个 tokenizer，地址分别由 scheduler、api_server bind），detokenizer 因带流式状态而不可水平扩展。
