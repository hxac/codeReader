本讲打开 u4-l1 主循环里 `receive_msg`/`send_result` 两个黑盒，讲解 `SchedulerIOMixin` 的 I/O 装配与多 rank 广播。结论：Mixin 在构造期仅凭 `offline_mode` 与 `tp_info.size` 把这两个方法动态绑定到不同实现（离线短路、单 rank、多 rank），主循环代码对所有拓扑保持不变。分工为「rank 0 对外收发、其余 rank 静默」——只有 rank 0（`is_primary()`）连 tokenizer/detokenizer，结果回送也只有 rank 0 执行，rank 1+ 的 `_reply_tokenizer_rank1` 是空操作。

多 rank 同步走两段式广播：rank 0 用 ZMQ PUB/SUB 把每条原始 msgpack 字节（`get_raw`/`put_raw`）原样转发（省去重复编解码），再用 gloo `broadcast` 同步「消息条数」。核心难点是**为何广播条数而非转发 tensor**：PUB/SUB 异步投递，rank 1 靠 SUB `empty()` 判长度会被时延欺骗而漏收，致 batch 分叉、前向 `all_reduce` 维度对不上；broadcast 一个整数 N 让各 rank 用 `range(N)` 精确收齐等数量、等内容的批次。变长 bytes 宜走 ZMQ，broadcast 宜搬同构大张量，二者职责解耦。

`sync_all_ranks` 是一行 gloo `barrier`（复用 `tp_cpu_group`），用于启动后与关闭前的生命周期级同步，避免 slow-joiner 丢消息。

关键术语：`SchedulerIOMixin`、`tp_cpu_group`、gloo CPU 进程组、PUB/SUB 字节级转发、broadcast 条数同步、`sync_all_ranks`、slow-joiner。
