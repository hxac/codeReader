本讲是 Mini-SGLang 的骨架篇，讲清进程架构与请求生命周期。系统为多进程架构，由四类进程组成：API Server（FastAPI 前端，跑在主进程）、1 个 Detokenizer、num_tokenizers 个 Tokenizer、world_size（= --tp-size）个 Scheduler；默认配置下 Tokenizer 并入 Detokenizer，后台进程总数等于 world_size + num_tokenizers + 1。每张 GPU 对应一个 Scheduler，身份由 DistributedInfo(rank, size) 标识，rank 0 作「队长」独揽与 tokenizer/detokenizer 的交互并向其他 rank 广播，保证各卡处理同一批次，这是张量并行正确性的前提。通信分两条干道：ZMQ 走轻量控制消息（token、元数据，共 5 条 ipc 通道），NCCL/PyNCCL 走重型张量（前向时的 all_reduce/all_gather）；多 rank 广播靠 ZMQ pub/sub 搬消息、CPU 进程组 broadcast 同步消息条数。一个请求经历 8 步生命周期，ZMQ 通道绕成一环，NCCL 只藏在第 5 步 Engine 前向内部。全程不变量是 uid，串起 TokenizeMsg→UserMsg→DetokenizeMsg→UserReply 的请求身份。
