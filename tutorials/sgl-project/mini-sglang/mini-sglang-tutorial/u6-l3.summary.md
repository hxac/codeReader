本讲讲解 `CacheManager`——夹在池存储（`MHAKVCache`）与基数树索引层（`RadixPrefixCache`）之间的纯 CPU 页簿记对象。它持有 `free_slots`（页对齐的 token 下标数组，存 token 下标而非页号以直接写入 `page_table`）与 `prefix_cache`，把请求逻辑位置翻译成池子物理下标。`available_size = evictable_size + |free_slots|×page_size`，只计可淘汰那桶，是 prefill 准入判断的命脉。四个最小模块：`allocate_paged` 用 `div_ceil` 将 `cached_len→device_len` 换算成需新分配页区间，正确跳过跨边界已拥有页，`_allocate` 在空闲不足时按 token 数向基数树 `evict`、返回值再 `[::page_size]` 取页起始并回 `free_slots`；`cache_req` 把算完的 KV 按 `old_handle.cached_len / cached_len / new_handle.cached_len / req.cached_len` 四边界处置——重复段释放防泄漏、新段入树、尾部按 `finished` 决定保留（`lock` 进 decode）或释放；`lazy_free_region` 以实例属性遮蔽类方法 + `del` 回落，把一批 `_free` 延迟到批末一次 `torch.cat` 合并。全程不变量：`free_slots` 永远页对齐、`free_pages + cache_pages == num_pages`，由 `check_integrity` 校验。
