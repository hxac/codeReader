本讲聚焦每张 GPU 上的执行核心 `Engine`，剖析 `Engine.__init__` 这条「有序的显存装配流水线」。最小模块有四：Engine 初始化编排、meta 建图、`_determine_num_pages` 显存估算、`_sync_get_memory` 跨 rank 显存对齐。

核心结论：`__init__` 严格按「量 baseline 显存 → 装模型 → 用差值算 KV cache 页数 → 捕获 CUDA Graph」排序，两次 `_sync_get_memory` 夹着模型加载构成估算支点。采用 meta device 建图（`torch.device("meta")` 零显存搭骨架）配合 `BaseOP.load_state_dict` 以 `setattr` 覆盖真实权重，将加载峰值从约 2× 参数量降到约 1×。

关键术语与概念（供后续承接）：`Engine`（GPU 侧执行器）、page/page_size（KV cache 分配单元）、`memory_ratio`（默认 0.9，为运行时预留余量）、`num_pages` 与 `cache_per_page` 公式（2×head_dim×local_kv_heads×page_size×itemsize×num_layers）、`div_even`（处理 KV head 的 TP 切分与 GQA 复制，保证估算与实际分配一致）、`page_table`（存原始位置下标而非页号，对齐 128 字节）、dummy page/`dummy_req`（供 CUDA Graph 捕获用）。`_sync_get_memory` 用一次 `all_reduce(MIN)` 同时拿 min/max（打包 `[free, −free]` 技巧），差超 2 GiB 即判失衡早 fail；所有 rank 取同一全局聚合值算页数，是 TP 下 KV cache 形状对齐的根本保证。本讲为 u5-l2（forward 与采样）、u5-l3（CUDA Graph）及 u6-l1（KV Cache 池）的前置。
