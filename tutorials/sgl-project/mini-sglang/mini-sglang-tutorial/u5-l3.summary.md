本讲讲解 `engine/graph.py` 中 `GraphRunner` 如何用 CUDA Graph 把 decode 每轮前向录制后反复回放，将 CPU 逐个 launch kernel 的开销压成一次 `replay`。核心结论：图只对 decode 且 `size≤max_graph_bs` 生效——decode 每条请求固定贡献 1 个 query、形状固定可录图，prefill 形状每批皆变无法录制。

关键术语与机制：`GraphCaptureBuffer` 体现「地址固定、内容可变」，捕获时 `set_batch` 让 batch 输入别名到 buffer 切片录下地址，回放时 `copy_from` 改写内容；`_capture_graphs` 对尺寸集 `[1,2,4]+range(8,max_bs+1,8)` 从大到小逐个录制，复用首个最大图的 graph pool 节省显存，录制前先 eager warmup；`pad_batch` 用 `dummy_req` 把真实 batch 补齐到不小于它的最小捕获尺寸，`dummy_req.table_idx` 指向 `page_table` 末行、整行指向专用 dummy page，不污染真实 KV，其 logits 被 `[:size]` 丢弃；注意力后端用 `init_capture_graph`/`prepare_for_capture`/`prepare_for_replay` 三段协议应对每批变化的 page 映射，`HybridBackend` 转发给 decode 后端。
