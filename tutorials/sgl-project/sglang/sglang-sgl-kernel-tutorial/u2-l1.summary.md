本讲追溯 `import sgl_kernel` 在 Python 端发生的事，涉及 `__init__.py`、`load_utils.py`、`debug_utils.py` 三文件。核心结论：包入口最顶层用平台判定把 macOS（仅发 Metal 扩展，连 `import torch` 都不执行）与其余后端分流；CUDA/ROCm/MUSA 通用分支按「加载算子库 → 预载 CUDA 运行时 → re-export 算子」顺序初始化，并以 `torch.version.hip`/`musa` 增补后端专属算子。

关键概念：compute capability 按 `cc = major×10 + minor` 折算，据此在 `sm90`（仅 H100/cc==90，fast math）与 `sm100`（其余所有 NVIDIA 卡及无 GPU，precise math，实为默认通用版）两份产物目录间二选一，对上 CMake 的两个安装目录；`_load_architecture_specific_ops` 用三级回退（架构专属目录 → 扁平回退 → 标准 import），配 `_filter_compiled_extensions`（`.so` 优先于 `.py`）保障健壮加载；`_preload_cuda_library` 用 `ctypes.CDLL(..., RTLD_GLOBAL)` 提前载入 `libcudart.so.X`，让后续 dlopen 的库能解析运行时符号，消除「libcudart.so.12 not found」类报错。另需区分：`SGLANG_KERNEL_API_LOGLEVEL`（依赖 sglang 主包的 `kernel_api_logging`）控制算子调用追踪，与 `load_utils.py` 靠 Python logging 打开的加载日志是两套独立机制。
