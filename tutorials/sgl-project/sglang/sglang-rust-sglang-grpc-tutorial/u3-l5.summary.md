本讲义聚焦 sglang-grpc 中 Tokio 运行时与 CPython GIL 在同进程内的并发协作模型。start_server 把多线程运行时建在独立的 sglang-grpc OS 线程上，由该线程 block_on 驱动 tonic accept 主循环，而具体 RPC async fn 实际跑在 sglang-grpc-tokio worker 池上，阻塞活儿走独立阻塞池。关键术语：Runtime（拥有型运行时）、Handle（引用计数轻量句柄，clone 廉价）。tokio_handle 经 rt.handle().clone() 取出、存入 PyBridge 并随每个 chunk 回调再 clone，根本原因是回调跑在持有 GIL 的 Python 线程上、不能 await，必须借 Handle 把慢通道发送甩回运行时。直接在 async fn 内 Python::with_gil 会占用 worker、削弱并发（W_eff=W-b）且有死锁风险，故 Python 做真活儿的调用用 spawn_blocking 隔离（server.rs 共 6 处），其 .await?? 双问号分别处理 JoinError 与 PyErr。核心结论：with_gil 有四种时机——A 启动期一次性、B 内联投递即返回（如 submit_request）、C spawn_blocking 隔离、D spawn 出的任务内先 await 后拿 GIL；判据是 Python 侧是否立即返回与当前线程是否有运行时上下文，GIL 始终遵循快进快出。
