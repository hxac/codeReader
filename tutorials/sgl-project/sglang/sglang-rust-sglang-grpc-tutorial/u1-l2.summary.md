本讲只讲 sglang-grpc 这个 Rust crate 的「构建链」：它如何从 `.proto` 被编译成 `sglang.srt.grpc._core`（`_core.so`），不涉及任何 gRPC 业务逻辑。

核心四块：`Cargo.toml` 用 `[lib] name="_core" crate-type=["cdylib"]` 决定产物形态，依赖按「Python 互操作（pyo3）/ 异步 gRPC 运行时（tokio、tonic、prost、tokio-stream、async-stream）/ 业务工具（uuid、serde_json、tokenizers、tracing）」三类各司其职，`tonic-build` 因只在编译期生成代码而放在 `[build-dependencies]`；`[profile.release]` 取 `opt-level=2 + thin LTO + strip` 兼顾性能与体积。`build.rs` 唯一职责是用 `tonic_build::configure()`（只生成 server 端、传 `--experimental_allow_proto3_optional`、保存 FileDescriptorSet）把 `.proto` 编成 Rust 代码写入 cargo 注入的 `OUT_DIR`，`src/lib.rs` 再用 `tonic::include_proto!`（包名须与 proto 的 `package` 严格一致）引入。`cargo:rerun-if-changed` 只声明 proto 路径，因为它是 build.rs 唯一会改变输出的外部输入，可避免改 `.rs` 时无谓重跑 protoc。`rust-toolchain.toml` 把工具链钉到 `1.90`（`profile=minimal`）以匹配 `edition=2024`。最后 `pyproject.toml` 声明 grpc 与 multimodal 两个扩展，`setup.py` 的 `BuildRust` 在 `super().run()` 前用 `_selected_rust_extensions` 按 `SGLANG_BUILD_RUST_EXTS` 做大小写不敏感子串匹配过滤（这是构建期变量，直接读 `os.environ`），`=grpc` 只保留 `sglang.srt.grpc._core`、跳过 sglang-mm。
