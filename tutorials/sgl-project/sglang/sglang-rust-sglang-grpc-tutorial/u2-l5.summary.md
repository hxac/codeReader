本讲围绕 bridge.rs 讲清 Rust 异步世界与 Python GIL 世界的解耦骨架。PyBridge 是全局唯一的桥接外壳（非 pyclass），持有 Python 端的 runtime_handle、Tokio 的 tokio_handle、共享账本 state 及配置；真正存放 per-request 动态状态的是 BridgeState——以 rid 为主键的 5 张表（channels/pending_sends/ready_callbacks/ready_signals/terminal_errors）由同一把 std::sync::Mutex 护住，以保证多表读写原子、临界区短且不跨 await。lock_or_recover 把锁中毒转成「记警告 + 取回数据」的有损恢复，避免单次 panic 雪崩拖垮整个服务。create_channel 建容量为 response_channel_capacity 的有界 mpsc 通道，遇重复在途 rid 返回 Duplicate 错误且不动现存通道，注册新通道时则顺手清旧账。submit_request 是数据型 RPC 公共入口，三段式：建通道→跨 GIL 调 runtime_handle.submit_request→失败 remove_channel 回滚，回滚防止僵尸通道、Receiver 永等不到终端 chunk、rid 占用挡重发。关键术语：rid、ResponseChunk（Data/Finished/Error）、TerminalError（ChannelFull/ClientDisconnected/Aborted）、PyO3 回调、Sender/Receiver、GIL。本讲只覆盖骨架；chunk 消费、回调内部细节、背压停泊与中止传播留待后续讲义。
