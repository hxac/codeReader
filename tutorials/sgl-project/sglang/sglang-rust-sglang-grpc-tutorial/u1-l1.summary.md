本讲义定位 sglang-grpc：一个用 Rust + PyO3 编写、被编译成 Python 扩展模块 `sglang.srt.grpc._core` 的进程内原生 gRPC 服务。核心覆盖四个最小模块。其一，crate 清单：`Cargo.toml` 的 `[lib]` 段（`name="_core"`、`crate-type=["cdylib"]`）与 pyo3 的 `extension-module` feature 决定产物为可被 Python C FFI 加载的动态库。其二，PyO3 入口：`src/lib.rs` 的 `#[pymodule] _core` 导出 `start_server` 函数（带 `worker_threads` 等默认参数）、`GrpcServerHandle` 句柄类与 `ChunkSendStatus`，对 Python 隐藏 Tokio/tonic 细节。其三，构建接入：`pyproject.toml` 的 `[[tool.setuptools-rust.ext-modules]]` 通过 `target`/`path`/`binding` 三字段把 crate 编进 sglang wheel，落到 `sglang.srt.grpc._core` 路径。其四，职责边界：SGLang 存在两条 gRPC 路径——本 crate 经 `http_server.py` 调 `start_server` 启动进程内 tonic server 并复用进程内调度器，而 `grpc_server.py` 的 `serve_grpc` 另委托外部 PyPI 包 `smg-grpc-servicer`，两者无关。关键术语 `RuntimeHandle`（Python 瘦句柄）与 `chunk_callback`（Rust 侧 PyO3 对象）构成 Rust→Python→Rust 的双向闭环，本讲义后续只研究前者路径。
