本讲聚焦 `src/app_context.rs`，讲解网关运行时的共享底座 `AppContext`。它把 22 个需全程序共享的组件（worker_registry、policy_registry、三个 storage、job_queue、inflight_tracker 等）打包进 `Arc`；因关键字段皆为 `Arc<...>`，`Clone` 只增计数，到处克隆几乎免费。字段分三类：启动即就绪的 `Arc` 字段、按配置可选的 `Option<Arc>` 字段、靠 `OnceLock` 延迟初始化的字段。其中 `worker_job_queue` 与 `workflow_engines` 必须延迟，以破解 `JobQueue` 与 `AppContext` 的循环依赖——JobQueue 持 `Weak<AppContext>`，AppContext 先建好再 downgrade 并 set；`mcp_manager` 无循环依赖但装配期已填，保留 OnceLock 仅为访问一致。`AppContextBuilder` 用链式 setter 与 `build()` 集中校验必填字段，就地构造 worker_service 与 inflight_tracker；`from_config` 以 14 步 `with_*` 装配链将 `RouterConfig` 实例化为可共享的活状态，步骤间依赖靠 `.expect()` 运行时强制顺序。核心结论：AppContext 是 RouterConfig 的运行期具象化。
