本讲是控制面「Worker 生命周期」单元的地基，回答代码里用什么表示一个后端推理进程。核心是 `Worker` trait：用 trait 对象 `Arc<dyn Worker>` 统一 `BasicWorker` 与 `DPAwareWorker` 等不同形态，约 30 个方法分身份、健康、负载、熔断、模型能力、DP、gRPC 七组，且大量带默认实现。关键结论：worker 是否可用由 `is_available()` 决定，即 `is_healthy()` 与熔断器放行的合取——熔断器被嵌入每个 worker、并以 worker URL 作指标标签。`WorkerType` 取 `Regular`、`Prefill{bootstrap_port}`、`Decode` 三值，仅 `Prefill` 携带 bootstrap 端口，是 PD 分离部署的唯一痕迹。`WorkerMetadata` 打包全部静态属性（含 `HealthConfig`、`ConnectionMode`），默认 models 为空即接受任意模型。`BasicWorkerBuilder` 以链式 API 加 `build()` 收口，造出原子计数器与带 URL 标签的熔断器；`DPAwareWorkerBuilder` 复用该逻辑，仅额外做 URL 改写与 DP 包装。`ModelCard` 把模型字段聚合成有类型卡片，配套 `ModelType` 的 bitflags 使端点能力判断退化为位运算。本讲为后续 `WorkerRegistry`、`WorkerManager` 及注册工作流的地基。
