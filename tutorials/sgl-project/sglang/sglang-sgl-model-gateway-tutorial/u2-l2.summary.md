本讲补齐 u2-l1 留白，聚焦 `src/config/` 三个文件：`types.rs` 管「有什么」（字段）、`validation.rs` 管「对不对」（规则）、`mod.rs` 管「错成什么样」（错误类型）。

`RouterConfig` 是网关的「参数总表」，四十余字段分为路由形态、服务能力、连接池、限流、可靠性、存储/安全六组；未显式赋值的字段一律取自 `impl Default`（默认端口 3001、超时 1800s、`max_concurrent_requests = -1` 即不限、`policy = Random`、`connection_mode = Http`），即「开箱即用」的真实行为。

两个正交维度：`RoutingMode`（Regular / PrefillDecode / OpenAI，描述部署形态）与 `ConnectionMode`（Http / Grpc，描述与 worker 通信协议）。PD 模式的有效策略经 `get_prefill_policy` / `get_decode_policy` 回退到主策略，`worker_count()` 按模式分别统计。

可靠性三件套 `RetryConfig`、`CircuitBreakerConfig`、`HealthCheckConfig` 风格一致（字段+Default）；`disable_retries` / `disable_circuit_breaker` 为软开关，经 `effective_*_config()` 把「关闭」翻译成阈值无效化，校验针对有效值。`HistoryBackend`（Memory/None/Oracle/Postgres/Redis）把外部后端设计成「选了就必须附带凭据」的可选字段。

核心结论：项目以「强类型枚举 + 集中式 `validate()`」对抗配置散乱——非法字符串在解析期被拒，跨字段校验集中在 `ConfigValidator::validate`，在 `builder.build()` 封口执行，校验不过程序不起。`ConfigError` 四变体（`ValidationFailed`、`InvalidValue`、`IncompatibleConfig`、`MissingRequired`）对应四类错法，校验链顺序短路，决定多处问题时先看到哪个错。本讲只讲配置的类型与校验，如何被 `server::startup` 消费留待 u2-l3。
