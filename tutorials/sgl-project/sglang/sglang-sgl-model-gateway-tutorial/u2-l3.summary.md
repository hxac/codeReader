本讲讲解网关从 `main()` 到 `server::startup()` 的启动编排。`main()` 极薄，靠所有权移交把责任交给异步 `startup`，`validate()` 在进 `startup` 前已完成。`startup` 是顺序敏感的装配流水线：日志 → 指标 → Mesh → AppContext → JobQueue/WorkflowEngines 填入 OnceLock → 提交 tokenizer/worker/MCP 三类启动作业 → RouterManager → 健康/限流 → 服务发现 → bind+serve。引入的关键术语：AppContext 的三个 OnceLock 字段（worker_job_queue、workflow_engines、mcp_manager）以「先占位、后填值」惰性初始化，解开「JobQueue 需 `Weak<AppContext>` 而 AppContext 须先存在」的循环依赖；JobQueue 用 mpsc+信号量（默认容量 1000/并发 200），`submit` 仅 `send` 即返回，`InitializeWorkersFromConfig` 扇出为多个 `AddWorker`。故网关能在 worker 未全部注册时即对外 bind，真正就绪判定交给 `/readiness`；`RouterManager::from_config` 按 `enable_igw` 走多路由器容错或单路由器必成两种装配，且不依赖 worker 已注册。
