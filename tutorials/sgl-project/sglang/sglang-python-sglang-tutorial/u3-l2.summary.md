SGLang 默认运行时是多进程架构，以绕开 GIL、隔离 CPU（asyncio）与 GPU（CUDA stream）工作：由主进程的 `TokenizerManager` 加两个子进程 `Scheduler`、`DetokenizerManager`（进程名 `sglang::scheduler`/`sglang::detokenizer`）组成，仅 Scheduler 操作 GPU。`Engine._launch_subprocesses` 是总装配线：先 fork Scheduler 让其后台加载模型，再 fork Detokenizer，再于主进程内构造 TokenizerManager，最后 `wait_for_ready` 阻塞等模型就绪并装上 `SubprocessWatchdog` 守护子进程。

三方经三条 ZMQ PUSH/PULL 通道形成单向环：`scheduler_input_ipc_name`（Tokenizer→Scheduler）、`detokenizer_ipc_name`（Scheduler→Detokenizer）、`tokenizer_ipc_name`（Detokenizer→Tokenizer），通道名集中于 `PortArgs`；约定主进程与 Detokenizer 各 bind 自己的接收 socket、Scheduler 统一 connect。消息用 `msgspec`/msgpack 经 `sock_send/sock_recv` 序列化，非 msgspec 字段（如多模态）经 `PickleWrapper` 包装；载荷基类为带 `rid` 的 `BaseReq` 与带 `rids` 的 `BaseBatchReq`。

一条请求的 token 流为：Tokenizer 分配 `rid`、token 化后 PUSH → Scheduler 组 batch、GPU 前向产出 token id → Detokenizer 增量解码（skip 指针）回传文本 → Tokenizer 的 `handle_loop` 按 `rid` 查 `rid_to_state` 中的 `ReqState` 唤醒异步调用者。Scheduler 的事件循环由 `dispatch_event_loop` 按配置分发 normal/overlap/pp/disagg 等变体，本讲仅展开 `event_loop_normal`；仅 rank 0 Scheduler 参与 ZMQ IPC，其余 worker 只做本 rank 计算并经 NCCL 协作。
