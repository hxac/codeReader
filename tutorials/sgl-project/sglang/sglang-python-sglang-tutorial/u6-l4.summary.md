本讲承接「KV 池长什么样」（u6-l1），回答两个上层问题：池要做多大、运行时每个 token 的 slot 怎么分发与回收。核心模块有四：一是池容量推算，由 `KVCacheConfigurator.configure` 主导，走「预算→容量」反推——`_profile_available_bytes` 量出可用字节（扣模型权重与运行时 slack），再经 `MemoryPoolConfigurator` 的 `coeff + bias` 线性模型换算成 `max_total_num_tokens`（默认 MHA 下 `coeff = cell_size`、`bias = 0`），最后叠加用户 `--max-total-tokens` 上限与流水线并行的 `all_reduce(MIN)` 对齐。二是混合 Mamba 模型的状态池与 KV 池共享预算，需联立方程求解 `max_mamba_cache_size`（记作 K）；本讲重点解释 HEAD 59ef3b1 修正的 +1 padding slot off-by-one——因 `ReqToTokenPool` 与 Mamba 状态张量物理上多分 1 个 slot 给 CUDA Graph dummy 读写兜底（slot 0 永不分配），故求解方程、投机中间状态预留、最终扣减三处都必须用 K+1，否则在显存吃满时超卖。三是运行时 slot 分发集中在 `allocation.py`，extend 与 decode 各走一条路径，均先淘汰再分配、失败 fail-loud，`page_size>1` 走带 Triton kernel 的分页路径。四是三类分配器记账方式不同：非分页（slot free list）、分页（页 free list）、统一内存池 `MultiEndedAllocator`（virtual→physical 映射 + watermark + compaction），均遵守 `BaseTokenToKVPoolAllocator` 的 `alloc/free/backup_state` 协议；而 `registry.py` 注册的是 RadixCache 实现工厂（`--radix-cache-backend`），与注意力后端注册表正交。关键承接术语：`cell_size`、`max_total_num_tokens`、`max_mamba_cache_size (K)`、`+1 padding slot`、`req_to_token` / `out_cache_loc`、`page_size`、三类分配器、`compaction`。
