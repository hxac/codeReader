本讲讲解 SGLang 的两类请求入口：HTTP + OpenAI 兼容 API（`/v1/chat/completions`、`/v1/completions`、`/v1/embeddings`）与同进程嵌入式 API（`sglang.Engine`）。核心结论是：两条路径的多样性在 `GenerateReqInput` 这一统一请求对象上被抹平，最终都汇聚到同一个方法 `tokenizer_manager.generate_request(GenerateReqInput, raw_request)`，由此进入运行时调度。`Engine` 类在主进程拉起 Scheduler/Detokenizer 子进程，`generate/encode` 直接构造 `GenerateReqInput`，零网络开销；HTTP 侧由 `protocol.py` 的 Pydantic 模型定义协议、`serving_chat.py`/`serving_base.py` 的 `handle_request`（校验→转换→分流）将 OpenAI 请求渲染为内部请求。`EngineBase` 抽象统一两类引擎接口形状。关键术语：`GenerateReqInput`（贯穿运行时的通用语言）、汇聚点、前端工厂 `Runtime`/`RuntimeEndpoint`，以及 `sglang.Engine` 经 `__init__.py` 命名空间解析默认指向运行时引擎而非前端 DSL 同名壳。
