本讲讲清 SGLang 两条请求入口：OpenAI 兼容 HTTP API（`/v1/chat/completions`、`/v1/completions`、`/v1/embeddings`）面向跨机器多语言客户端；同进程嵌入式 `sglang.Engine` 零网络开销，适合 RL rollout 与单机场景。核心结论是：两条路径在源码层都汇聚到同一个方法 `tokenizer_manager.generate_request`，入口的多样性在统一的请求对象 `GenerateReqInput` 上被抹平。`Engine` 类在主进程拉起 Scheduler/Detokenizer 子进程，`generate/encode` 把参数打包成 `GenerateReqInput` 后交出；`EngineBase` 抽象使 HTTP 引擎与嵌入式引擎接口形状一致、可互换。本讲引入的关键特性 `return_token_ids`：开启后响应的 choice 会多出 `token_ids`（输出 token 序列）与 `prompt_token_ids`（输入 token 序列）两个整数列表，二者默认为 None 并在序列化时被 pop，以保持与官方 OpenAI 协议字节级兼容；两端流式行为不对称——completions 流式逐块回传增量 token id，而 chat 流式直接 raise ValueError 拒绝。此外澄清命名空间解析：`__init__.py` 用 LazyImport 使 `sglang.Engine` 默认指向运行时引擎 `sglang.srt.entrypoints.engine.Engine`，而非前端 `lang/api.py` 的同名工厂壳，前端另有 `Runtime`/`RuntimeEndpoint` 用于连接远程 HTTP 服务。本讲止步于「入口汇聚点」，运行时内部（TokenizerManager→Scheduler→Detokenizer）机制留待后续单元细讲。
