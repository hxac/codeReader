本讲回答前端解释器持有的 `self.backend` 是什么对象。核心是依赖倒置：`BaseBackend` 抽象基类为所有后端定义统一契约，`generate` / `generate_stream` / `select` 是必须实现方法（基类抛 `NotImplementedError`），其余为可选钩子（默认空实现，让弱后端优雅退化），前端只认接口、不关心底层是 HTTP 服务还是 SDK。

`RuntimeEndpoint` 是 `BaseBackend` 子类，即对运行中 sglang HTTP 服务的 REST 客户端，构造即探活（`/get_model_info`），用 `to_srt_kwargs` 翻译采样参数并组装 `/generate` 请求体，能拿真实 token logprob 且 `support_concate_and_append=True`，能力最完整。`Runtime` 是服务启动器（spawn 子进程跑 `launch_server`、轮询 `/health_generate`，并内部包一个 `RuntimeEndpoint`）而非后端，其 `generate` 签名与后端契约不同。`OpenAI` / `Anthropic` / `LiteLLM` 是第三方后端，走各家 SDK、用各自 `to_*_kwargs` 翻译参数，拿不到 token 级 logprob、`select` 只能退化（chat 模型直接不支持）、并需限流重试。

`set_default_backend` 把后端实例写入 `global_config.default_backend`，`run` / `run_batch` / `trace` 在未传 `backend=` 时回退到它；`Runtime` / `Engine` 在 `api.py` 用懒导入，避免纯前端用户被迫装入运行时依赖。结论：同一份 `@function` 可在自研与第三方后端间无缝切换，自研后端能力完整、第三方为退化版。
