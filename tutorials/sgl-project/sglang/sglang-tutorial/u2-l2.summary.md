本讲讲清 SGLang 配置如何进入运行时及运行期为何只读。ServerArgs 为配置源头,字段以 `A[T, Arg(...), NS(...)]` 标注,NS 标记逻辑命名空间、与字段在文件里的物理位置无关(如 `mem_fraction_static` 属 `schedule`)。`__post_init__` 以有序 `_handle_*` 序列解析:模型代码只声明改动、返回 dict 而不写字段,声明累积进 `_resolved_overrides` 后由 `materialize_declarations` 按后写者胜一次性应用并冻结配置(置 `_declarations_materialized`)。此后只读,裸赋值被重写的 `__setattr__` 拦截;运行期唯一改写入口 `override(source, **fields)` 凭 `_in_override` 令牌放行,并记入可重放 stash 与 mutations 日志,命名空间读者则需走 `get_context().override()`。启动链路为 `sglang serve` → `prepare_server_args`(触发解析与只读化)→ `run_server` 模式分发 → `http_server.launch_server` 拉起 TokenizerManager/Scheduler/Detokenizer 多进程,所有进程各持一份只读快照。本讲为下讲 RuntimeContext 构建配置袋奠基。
