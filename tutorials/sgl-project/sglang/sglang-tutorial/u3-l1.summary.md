本讲打开 Scheduler 黑盒。Scheduler 是每张 GPU 一个的进程级单例，构造完成后立即进入 run_event_loop 阻塞循环，靠事件循环（每圈一轮迭代）把请求编排成批交给 GPU 前向，直到收到 ShutdownReq 才优雅退出。它维护三大运行时状态：waiting_queue（待 prefill）、running_batch（正在 decode）、last_batch（上一轮的批）。run_event_loop 仅负责创建 schedule_stream 并调 dispatch_event_loop，按 enable_overlap 等参数分派到 event_loop_normal（串行）或默认的 event_loop_overlap：后者用 schedule_stream/forward_stream 双 CUDA 流加 result_queue 把结果处理延后一轮，配 WAR 屏障使 CPU 调度与 GPU 前向重叠，即「零开销调度器」，连续 prefill 等场景由 is_disable_overlap_for_batch 自动退化为同步。调度大脑是 get_next_batch_to_run：先 merge_batch 衔接 prefill→decode，再分流到 get_new_batch_prefill（经 PrefillAdder 评估预算与前缀缓存匹配、从等待队列组 prefill 批）和 update_running_batch（维护 decode 批，KV 不足时 retract_decode 回缩请求），用 NextBatchPlan 打包返回 batch_to_run 与更新后的 running_batch。协作组件 request_receiver 以 ZMQ 非阻塞拉取请求、batch_result_processor 消化前向结果并流式发回 Detokenizer，均以「持有并调用」的组合方式接入 Scheduler。一条 Req 的生命周期随之在 waiting_queue↔running_batch 间流转。
