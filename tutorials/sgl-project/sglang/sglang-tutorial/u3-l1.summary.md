本讲打开 u2 系列中被当作黑盒的 Scheduler，刻画其进程定位、事件循环骨架与 prefill/decode 分流。Scheduler 是每张 GPU 一个的进程级单例，持有 GPU worker、KV 缓存与调度策略；进程入口 `run_scheduler_process` 在构造 Scheduler 前**先 `publish(server_args, role="scheduler")`** 把只读配置快照成命名空间袋，再构造、再进入 `run_event_loop` 阻塞循环。它维护三大运行时状态：`waiting_queue`（待 prefill）、`running_batch`（正在 decode）、`last_batch`（上一轮的批）。`run_event_loop` 只建 `schedule_stream` 并调 `dispatch_event_loop`，按 `disaggregation_mode × enable_overlap × pp_size` 分派循环实现；普通服务默认走 `event_loop_overlap`。`event_loop_normal` 是串行循环（收请求→决策→`run_batch`→`process_batch_result`→刷 `last_batch`），CPU/GPU 交替空闲。`event_loop_overlap`（零开销调度器）用 `result_queue` 把结果处理**延后一轮**，让 CPU 调度与 GPU 前向经 `forward_stream`/`schedule_stream` 重叠，并用 `_apply_war_barrier` 保证写后读内存序；连续 prefill 等场景经 `is_disable_overlap_for_batch` 退化为同步。`get_next_batch_to_run` 是调度大脑，先 `merge_batch` 衔接 prefill→decode，再用 `get_new_batch_prefill`（经 `PrefillAdder` 评估预算+前缀匹配）与 `update_running_batch`（KV 不够时 `retract_decode` 回缩）分流，返回 `NextBatchPlan`。配置读取已迁移到命名空间袋：`chunked_prefill_size`/`prefill_max_requests` 走 `get_schedule()`、`disaggregation_mode` 走 `get_disagg()`、`pp_max_micro_batch_size` 走 `get_parallel()`、`enable_metrics` 走 `get_observability()`，热改唯一入口是 `get_context().override()`。关键术语：event loop、iteration、continuous batching、overlap、schedule_stream/forward_stream、retract，为 u3-l2（批数据结构）、u3-l3（调度策略）、u3-l4（overlap 工程细节）承接铺垫。
