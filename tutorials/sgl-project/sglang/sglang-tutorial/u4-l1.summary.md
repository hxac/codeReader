本讲围绕 SGLang 的前缀复用机制，覆盖三个最小模块：`RadixCache` 类、`TreeNode`/`RadixKey` 数据结构、`RadixAttention.forward`。核心结论是 RadixAttention = 基数树前缀缓存 + 统一注意力壳。基数树是一棵按 token 序列前缀压缩的多叉树，公共前缀在树中只存一份；其演化遵循「沿公共前缀下沉、遇分歧则分裂（`_split_node`）、剩余挂为新叶」三条规则。`RadixKey` 封装前缀匹配（指数搜索+二分降为 O(log n)）与子节点查找；`TreeNode` 持有 key、value（KV 池逻辑索引）、`lock_ref`、时间戳与 `priority`。`RadixCache` 暴露 `match_prefix`（找最长命中前缀返回 `device_indices`）、`insert`、`evict`（对 `evictable_leaves` 按 LRU/优先级淘汰）三大 API，及面向 `Req` 的 `cache_finished_req`/`cache_unfinished_req` 封装。`inc_lock_ref`/`dec_lock_ref` 的引用计数保护正在使用的前缀路径不被淘汰，`evictable_size_`/`protected_size_` 是调度器预算评估的账本。缓存层与计算层通过 `ForwardBatch.out_cache_loc` 这一 int64 索引张量解耦：RadixCache 产出命中索引，调度器拼成 `out_cache_loc`，注意力后端据此读写物理 KV 池。`BasePrefixCache` 定义统一契约（`MatchPrefixParams`/`InsertParams`/`MatchResult`），为后续 HiCache 等实现留出替换空间。
