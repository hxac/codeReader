本讲阐明 SGLang 在 RL 后端中的角色：非训练框架，而是负责高效 rollout（用当前策略权重采样响应）与权重热更新（不重启进程地把新权重灌入运行中的引擎）的推理骨干。核心有三：其一，Engine 提供同步 generate（一把批量、阻塞返回，适合离线评测）与异步 async_generate（逐条并发到达调度器、贴近在线流量）两套离线入口，底层共用 tokenizer_manager.generate_request，差别仅在驱动方式与请求到达模式；离线脚本须置于 `if __name__=="__main__"` 守卫下，否则 spawn 多进程会无限递归。其二，权重热更新按权重来源分四类：update_weights_from_tensor（张量直传，veRL/OpenRLHF 常用路径）、from_distributed（经 NCCL 进程组 broadcast，需先 init_weights_update_group 建组、用完 destroy）、from_disk（检查点重载）、from_ipc（ZMQ/checkpoint-engine）；因权重一变旧 KV 缓存失效，默认 flush_cache=True，连续多次更新时中间步骤设 False，配套 get_weights_by_name 做更新后校验、release/resume_memory_occupation 在非 rollout 期让出显存。其三，配置访问已迁移到 runtime_context：dp_size 走 get_parallel()、MultiTokenizerRouter 分支须主动 publish(role="tokenizer")、get_server_info 用 resolved_server_args_dict 叠加 override 后的运行期改动，server_args 退化为只读留档（迁移渐进，仍有少量遗留读取点如 tp_size）。完整闭环为 rollout → 训练 → update_weights → 再 rollout。
