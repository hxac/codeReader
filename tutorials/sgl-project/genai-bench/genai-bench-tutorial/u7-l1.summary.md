本讲聚焦 `DistributedRunner`（`distributed/runner.py`），拆解 genai-bench 的多进程压测架构。核心是「同一份代码、按 `environment.runner` 类型分流」的三态模型：**master**（调度与记账、跑主实验循环，唯一持有 `AggregatedMetricsCollector`）、**worker**（只算单条指标并发送、`greenlet.join()` 永久阻塞、不持有 collector）、**local**（单进程兼任）。`DistributedConfig` 据必填的 `num_workers` 分流：>0 走分布式（关 `TOKENIZERS_PARALLELISM`、注册 `atexit` 清理、fork worker、再建 master 并 `sleep` 等连接），否则走 local。四类 Locust 消息方向分明：master 发 `update_scenario` / `update_batch_size`，worker 发 `request_metrics`（源自 `BaseUser.collect_metrics`），刻意按角色只注册所需 handler。指标闭环为 worker 发 `request_metrics` → master 的 `_create_metrics_handler` 反序列化（Pydantic 校验，坏数据丢弃不崩溃）→ `add_single_request_metrics` 入聚合 → 非阻塞刷仪表盘。worker 日志经 `WorkerRichHandler` → `worker_log_queue` → master 的 `_consume_worker_logs` 统一回传，避免多进程直接打印破坏 `rich.Live` 仪表盘（`worker_log` 消息已注册却无发送方）。另有实验性 CPU 亲和（仅 Linux）与「先礼后兵」的 `cleanup`（terminate→join(10)→kill）。
