本讲解析 `RequestMetricsCollector.calculate_metrics` 如何把一条 `UserResponse` 换算成 `RequestLevelMetrics`。全部延迟指标以三个时间戳为骨架：TTFT（`time_at_first_token − start_time`）反映 prefill 速度、`e2e_latency` 为端到端时长、`output_latency` 为纯生成耗时。公共指标（含 `input_throughput`，分母取 TTFT 并做除零保护）对所有任务都算；仅 chat 任务额外算输出指标，TPOT 分母用 `num_output_tokens − 1`（N 个输出 token 只有 N−1 个相邻间隔落在 output_latency 内），故单请求层 `output_throughput` 与 `output_inference_speed` 恒等于 1/tpot。非 chat 任务（embeddings/rerank/image/tts）走重置分支，把输出字段置 0 而非 None：既规避 `model_validator` 跨进程重建报错，又让 0 作为「不适用」哨兵，供聚合层 `filter_metrics` 以 `tpot != 0` 跳过。`RequestLevelMetrics` 既是计算容器也是跨进程契约，其产物将交由 `AggregatedMetricsCollector` 聚合。
