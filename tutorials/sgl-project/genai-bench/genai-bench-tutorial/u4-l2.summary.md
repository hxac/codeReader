本讲聚焦 `AggregatedMetricsCollector`，承接 u4-l1 的单请求指标 `RequestLevelMetrics`，讲清如何把成百上千条逐请求指标压成一份运行级摘要 `AggregatedMetrics`。一次 **run** 由 `[scenario, concurrency]` 唯一界定；聚合由 `RequestLevelMetrics.model_fields` 驱动，遍历每个数值字段（排除 `error_code`/`error_message`），用 numpy 算出 `min/max/mean/stddev/sum` 与 p25~p99 六个百分位，写入 `StatField`。关键结论有三：其一，整轮吞吐 `mean_output_throughput = Σ输出token / run_duration`，是「总 token ÷ 时长」而非单请求吞吐的均值；错误率、RPS 等元信息依赖统计量、赋值有先后顺序。其二，warmup/cooldown **按请求数（非时间）**剔除头尾，保留区间为半开 `[warmup_number, n-cooldown_number)`，CLI 校验两者之和须 `< 1.0`。其三，`filter_metrics` 把 `output_latency<0.001` 且 `tpot!=0` 的请求的 TPOT/推理速度置 `None`，剔掉「假高 TPOT」，同时不误伤非流式任务 `tpot=0` 的哨兵。最后 `save` 写出三段式 JSON（聚合结果 / 逐请求明细 / `_time_unit`），并用 `TimeUnitConverter` 仅换算延迟字段；`AggregatedMetrics` 的 `model_dump`/`model_validate` 通过 `MetricStats.to_dict`/`from_dict` 与纯 dict 闭环转换。该 JSON 是后续 u4-l3 单位转换与 U6 实验报告的输入。
