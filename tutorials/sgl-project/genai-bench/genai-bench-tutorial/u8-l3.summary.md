本讲从「二次开发」视角串联 genai-bench 的三个扩展点：场景（Scenario，描述单次请求的输入/输出规模）、采样器（Sampler，把场景与数据集揉成 UserRequest，按输入模态分发）与后端用户（User，封装厂商的发送与响应解析）。核心结论是统一的「注册表 + `__init_subclass__`」心智模型，但三者扩展成本递增：场景与采样器靠父类 `__init_subclass__` 在子类定义时自动写入注册表（`Scenario._registry`、`Sampler.modality_registry`），实现「定义即生效」；后端 User 没有自动注册机制，必须手动登记。具体地，新增场景只需设 `scenario_type`（必要时先加枚举成员）、`validation_pattern` 并实现 `sample`/`to_string`/`parse`，且模块须被 import；「枚举有成员但无子类」（如 VIDEO 的 `V`）不会进注册表。新增任务分低摩擦（复用输入模态，仅改 `supported_tasks` 与 `sample()` 分发分支）与高摩擦（新建 Sampler 子类）。新增后端最重，需设 `BACKEND_NAME`、`supported_tasks`、`@task` 方法（末尾必须调 `collect_metrics`），再手动登记到 `API_BACKEND_USER_MAP` 与认证工厂（或 `auth_backend_map`），主流程再把 `auth_provider`/`host`/`api_backend` 注入类属性；可走「复用继承 `OpenAIUser`」或「从零继承 `BaseUser`」两种策略。
