本讲承接 u5-l1 的认证骨架，深入各 ModelAuthProvider 的方法体，覆盖三块：api_key 类认证、云厂商认证（AWS Bedrock / Azure OpenAI / GCP Vertex）与适配器封装。核心结论：api_key 类用老类加 adapter 补出 Authorization: Bearer 头；AWS Bedrock 因 SigV4 由 SDK 签名使 get_headers 恒空，靠 get_credentials 产出 boto3.Session 风格字典喂 SDK；Azure / GCP 都支持静态密钥头与 SDK 接管认证的双模式，且 get_auth_type 为运行时动态值。所有 provider 的 get_config 必须可序列化、脱敏（AWS 据此排除 secret_access_key），因其产物会写入 ExperimentMetadata.auth_config。接口契约由 ABC 强制，三个抽象方法缺一即 TypeError。adapter 遵循「组合优于继承」，持有老对象引用，转发 get_config / get_credentials、补造 get_headers / get_auth_type；OCIModelAuthAdapter 更以类名嗅探推断 OCI 认证子类型，体现「一份凭据两种视图」。下一讲 u5-l3 将讲对称的存储认证 StorageAuthProvider。
