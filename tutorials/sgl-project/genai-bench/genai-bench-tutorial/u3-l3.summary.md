本讲聚焦 genai-bench 的多后端 User 体系。项目用「每个厂商一个 User 子类」应对多厂商差异，子类只需声明 BACKEND_NAME（后端身份证，即 --api-backend 合法取值）与 supported_tasks（任务字符串→方法名路由表，兼声明支持哪些任务）。核心机制是注册表 API_BACKEND_USER_MAP：validate_api_backend 查表把选中类存入 ctx.obj["user_class"]，validate_task 再校验任务兼容性并将其翻译成方法对象；benchmark 函数体则把 auth_provider、host、api_backend 作为类属性注入，供各虚拟用户在 on_start 构建后端客户端。结论：存在继承复用（OCIOpenAIUser 继承 OpenAIUser，仅覆写差异）与从零实现（AWSBedrockUser 继承 BaseUser，自写发送解析、boto3 惰性导入、多模型族适配）两种扩展策略，所有后端的复用下限是 BaseUser 的 sample() 与 collect_metrics()。另指出 vllm/sglang 非新类，而是注册表里指向 OpenAIUser 的别名，后者靠注入的 api_backend 字段在三后端间做行为切换。
