本讲把 `README.md` 当作一张「导航地图」来读，是 u1 系列的收尾基本功（不含数学算法）。核心结论有四点。第一，README 由六大顶层**区段**（`##` 标题）构成：Announcement（里程碑公告，按月份倒序）、Slides（幻灯片，本地资产最集中）、Blog（多为外链）、Videos（YouTube）、Paper（arXiv）、Documentaion（文档站入口，原文有拼写错误）。总体规律是越靠上越时间敏感、越靠下越稳定长青。第二，`## Slides` 用 `###` **子区段**做**事件归类**，事件分 meetup（社区聚会）、学术会议/Dev Day、biweekly developer sync（双周例会）三类，外加 `### Other` 兜底；每条记录统一格式为 `[YYYY-MM-DD] [标题](链接)`。第三，**内链 vs 外链**的判断在每个区段都适用：以 `slides/`、`blogs/`、`./` 相对路径开头的是仓库内资产，以 `https://` 开头的是外部链接；Blog/Videos/Paper 几乎全是外链，唯一的本地长文是 `blogs/Efficient LLM Deployment and Serving.md`。第四，**日期前缀**既用于倒序排列也用于同名消歧，且 Biweekly 下有 `[2025-1-25]` 等非零填充写法，按关键词检索更稳妥。本讲强调「按主题反查」技能：同一主题（如 MLA、调度、受限解码、大规模部署）常横跨多区段、多次活动，应把同主题多条记录当成「**资料簇**」整体阅读，并交付一张「主题→README 位置」速查表，作为后续 u2《按主题绘制资料地图》的直接输入。贯穿后续讲义的检索命令是 `grep -ni '关键词' README.md` 与 `git ls-files slides/`。
