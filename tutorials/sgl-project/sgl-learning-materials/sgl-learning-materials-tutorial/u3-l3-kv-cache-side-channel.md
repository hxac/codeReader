## 2. 前置知识

本讲默认你已经读过前置讲义 `u2-l6`，并了解以下概念。为照顾从零开始的读者，这里用通俗语言再点一遍：

- **KV Cache（键值缓存）**：Transformer 在生成每一个 token 时，需要用注意力机制「回看」之前所有 token。为了避免每生成一个字就把前面全部重算一遍，引擎会把每一层算出的 Key/Value 中间状态缓存下来，这就是 KV Cache。它是推理服务里最值钱、最占显存的资产之一。
- **RadixAttention（基数树注意力）**：SGLang 的招牌机制，把 KV Cache 按 **token 前缀** 组织成一棵基数树（radix tree）。当新请求的前缀命中树里已有的节点，就 **复用** 这段 KV，跳过重新计算。详见 `u3-l1`、`u2-l3`。
- **Cache-Aware Load Balancer / SGLang Router**：挡在多个推理副本前面的负载均衡器，它依据 RadixAttention 的前缀复用情况，把请求优先发给「最可能已经缓存了该前缀」的副本。详见 `u2-l6`。
- **多租户（multi-tenant）**：一台服务器（或一个集群）同时为多个互不信任的用户/客户提供服务。共享基础设施能摊薄成本，但也意味着一个用户的行为可能影响、甚至泄露给另一个用户。
- **时序侧信道（timing side channel）**：一类经典安全攻击。系统对「不同输入」的处理时间如果有可测量的差异，攻击者就能通过 **测量响应时间** 反推出系统内部的某些秘密状态。本讲要讲的，就是把这条通用思路套在「KV Cache 命中 vs 未命中」的时间差上。

> 关键直觉：凡是「命中/未命中」会带来 **可观测时延差** 的优化，都可能是一条侧信道。SGLang 的前缀复用恰好满足这个条件。

### 4.1 KV Cache 共享机制回顾

#### 4.1.1 概念说明

要理解侧信道，先要理解它寄生在哪条「性能优化」上。SGLang 的吞吐优势很大程度来自两件让 KV Cache 「被反复复用」的事：

- **前缀复用（prefix reuse）**：多个请求如果开头相同（例如都用同一段系统提示词 system prompt），它们的前缀 KV 只需计算一次，之后所有请求直接共享。
- **跨请求 / 跨副本共享**：在 RadixAttention 树里，不同用户的请求只要前缀相同就会命中同一个缓存节点；Cache-Aware Router 还会主动把请求调度到「已缓存该前缀」的副本上，进一步放大复用率。

这当然是好事——复用越多，重新计算的 prefill 越少，吞吐越高、延迟越低。但从安全角度看，**「命中」和「未命中」在时间上是不对称的**：命中会显著更快。而「谁的前缀被缓存了」这件事，可能携带了某个用户问过什么的信息。这就是侧信道的种子。

#### 4.1.2 核心流程

把一次请求的延迟拆开看（伪代码，仅示意）：

```
# 示例伪代码：前缀复用下的延迟构成
def serve(request):
    prefix, rest = split(request.prompt)
    if prefix in radix_tree:            # 命中：前缀 KV 已存在
        kv = radix_tree[prefix]          # 直接复用，无需重新计算
        t_prefix = ~0                    # 命中：前缀计算成本近似为 0
    else:                               # 未命中：必须重新 prefill 整段前缀
        kv = prefill(prefix)             # 代价随前缀长度线性增长
        radix_tree.insert(prefix, kv)
        t_prefix = c * len(prefix)
    t_rest = prefill(rest) + decode(rest)
    return t_prefix + t_rest
```

要点：

- 命中时 `t_prefix ≈ 0`；未命中时 `t_prefix = c × 前缀长度`，其中 \(c\) 是每个 token 的 prefill 计算成本。
- 前缀越长，命中与未命中之间的 **时延差** 越大，越容易被外部测量区分。
- 这棵 RadixAttention 树在多租户服务里是 **跨用户共享** 的——这正是下一节攻击成立的前提。

#### 4.1.5 小练习与答案

**练习 1**：为什么说「前缀越长，侧信道越危险」？
**参考答案**：因为前缀越长，命中（复用、几乎不耗时）与未命中（完整重算）之间的时延差越大，时延差越大就越容易被外部测量区分，信噪比越高，攻击越可靠。

**练习 2**：Cache-Aware Router 让「命中」更容易发生，这对安全是好事还是坏事？
**参考答案**：对吞吐是好事，对安全是坏事。Router 主动把请求调度到已缓存该前缀的副本，等于放大了「命中 vs 未命中」在端到端时延上的可观测差异，从而使时序侧信道更敏锐。

### 4.2 时序侧信道攻击原理

#### 4.2.1 概念说明

把上一节的「时延差」交给一个恶意用户，就得到了时序侧信道攻击。核心只有一句话：**「我猜的前缀如果被别人问过、已被缓存，我的请求就会变快；如果没被缓存，就会变慢。于是通过测时间，我能反推出别人问过什么。」**

这种攻击在本仓库的 README 里被直接称作 **Side-Channel KV Cache Attack**（见 4.2.3 的第 100 行）。它的危害指向一类常被称作 **prompt stealing / prompt extraction（提示词窃取）** 的后果：攻击者可能逐步还原出其他租户（或系统提示词）使用过的一段文本。

> 注意标题里的 **Possible**（「Possible Timing Side Channel Of KV Cache」）。这个词提示：这份资料更像是对 **潜在** 漏洞的探讨与可行性分析，而非一份已经放出的完整利用工具。阅读时应把它当作「威胁建模 + 风险论证」来读，具体结论以幻灯片原文为准（待本地验证）。

#### 4.2.2 核心流程

先给一个最小的时延模型。设某段前缀长度为 \(n\) 个 token，每个 token 的 prefill 计算成本为 \(c\)，与该前缀无关的后续处理时间为 \(t_{\text{rest}}\)：

- 未命中（miss）：需要完整重算前缀

\[
t_{\text{miss}} \approx c \cdot n + t_{\text{rest}}
\]

- 命中（hit）：前缀 KV 被复用，省掉前缀计算

\[
t_{\text{hit}} \approx t_{\text{rest}}
\]

于是可观测的时延差为

\[
\Delta t = t_{\text{miss}} - t_{\text{hit}} \approx c \cdot n
\]

攻击者用一个阈值 \(\tau\) 做判决：

\[
\text{判定命中} \iff t < \tau
\]

判决的可靠性取决于信噪比 \(\Delta t / \sigma\)，其中 \(\sigma\) 是网络与系统带来的时延噪声标准差。前缀越长、噪声越小，攻击越可靠；这是为什么长系统提示词、长敏感前缀风险更高。

把单次「测试某个前缀是否被缓存」的能力放大，就得到 **逐 token 还原** 的攻击流程（伪代码，仅示意，**请勿用于未授权系统**）：

```
# 示例伪代码：概念性还原流程，仅用于理解威胁，不可用于真实攻击
secret = ""
while not done:
    for candidate in next_tokens(secret):
        probe = known_prefix + secret + candidate
        t = measure_latency(send(probe))     # 多次采样取中位数降噪
        if t < tau:                           # 命中：说明该前缀确实被缓存过
            secret += candidate               # 把猜中的 token 接上
            break
    # 每轮把搜索空间缩小，逐步逼近目标前缀
```

要点：

- 攻击者并不需要直接读取 KV Cache，只需 **能向服务发请求并计时**。
- 每一次「命中」相当于一个「oracle」回答「这段前缀是否被别人用过」，于是可以把还原一段未知前缀的过程变成 **逐 token 的搜索**（类似经典二分 / 字符猜解）。
- 多次采样取中位数是为了压低网络噪声 \(\sigma\)，提高判决可靠性。

#### 4.2.5 小练习与答案

**练习 1**：为什么攻击者要「多次采样取中位数」而不是用单次测量？
**参考答案**：单次测量受网络抖动、调度噪声影响很大，\(\sigma\) 高；多次采样取中位数可以压低噪声、抬高 \(\Delta t / \sigma\)，让「命中/未命中」的判决更可靠。

**练习 2**：如果服务把「未命中」的请求也人为延迟到和「命中」一样快（或反过来把命中拖慢到和未命中一样慢），攻击会怎样？
**参考答案**：这会抹平 \(\Delta t\)，使时延不再携带「是否命中」的信息，从而破坏侧信道。这正是「时延规整 / 填充」类缓解思路的原理（见 4.3）。

**练习 3**：标题里的「Possible」想提醒读者什么？
**参考答案**：它强调这是对 **潜在** 侧信道的探讨与可行性论证，而非已落地的完整攻击工具；阅读时应以威胁建模的视角看待，具体结论需回到幻灯片原文核实。

### 4.3 缓解措施与运营注意点

#### 4.3.1 概念说明

从上一节的时延模型能直接读出缓解思路的总纲：**让「命中/未命中」不再（或难以）被外部时间测量区分**，或者 **让攻击者拿不到可利用的 oracle**。常见的应对方向有几类（本仓库不含可运行代码，下列为通用工程方向，具体到 SGLang 的可配置项需查阅主仓库 `sgl-project/sglang` 与官方文档站，**待确认**）：

- **隔离 / 分区**：按租户、按用户、按敏感级别拆分 KV Cache 与调度域，避免不受信任的租户之间共享前缀缓存。
- **关闭或限制前缀复用**：在对提示词保密性要求极高的场景，关闭跨请求前缀复用（代价是吞吐下降，这是典型的安全—性能权衡）。
- **时延规整（timing padding）**：在命中路径上补足计算或延迟，使命中与未命中的端到端时延趋于一致，抹平 \(\Delta t\)。
- **访问控制与配额**：对可疑的逐 token 探针行为做速率限制与异常检测（短时间内大量「只差一两个 token」的相似请求是典型特征）。
- **避免把秘密放进可共享前缀**：在架构层不让高价值秘密（如系统提示词）落在会被多租户共享的前缀位置。

> 运营要点：这些方向之间常常冲突（隔离与关闭复用会直接削掉 SGLang 的吞吐卖点）。运维需要根据 **租户是否互信** 来决定开关——例如企业内部同一团队共享一个实例时，前缀复用的收益远大于风险；而面向公网、租户互不信任的 SaaS 场景，则必须考虑隔离或时延规整。**具体幻灯片给出的缓解建议以原文为准（待本地验证）。**

#### 4.3.2 核心流程

把缓解措施按「作用点」映射回 4.2 的攻击链：

| 攻击环节 | 可选缓解 | 代价 |
| --- | --- | --- |
| 前缀被多租户共享 | 按租户/用户隔离 KV Cache 与调度域 | 缓存命中率下降，吞吐降低 |
| 命中带来时延差 \(\Delta t\) | 时延规整 / 填充，抹平 \(\Delta t\) | 平均延迟上升 |
| 攻击者可随意发探针 | 速率限制 + 异常探测（逐 token 相似请求） | 误伤合法高频用户，需调参 |
| 秘密位于可共享前缀 | 架构上避免把秘密放进共享前缀位置 | 需改造应用层 prompt 设计 |

决策伪代码（示例）：

```
# 示例伪代码：运维侧的缓存共享安全决策
if tenants_mutually_trusted:            # 同一团队 / 内网
    keep prefix_reuse = True            # 保留 SGLang 的吞吐红利
    monitor_for_abuse()                 # 仅做被动监控
else:                                   # 公网 / 租户互不信任
    if confidentiality_is_critical:
        prefix_reuse = per_user_only    # 隔离，关闭跨租户共享
        consider timing_padding         # 抹平时延差
    enable rate_limit_on_probe_patterns # 限制逐 token 探针
```

#### 4.3.5 小练习与答案

**练习 1**：如果你的 SGLang 实例只服务一个互信团队，是否还需要担心本讲描述的侧信道？

## 6. 本讲小结

- SGLang 的吞吐优势很大程度来自 **KV Cache 共享**：RadixAttention 的前缀复用 + Cache-Aware Router 的按命中调度（资料入口见 README 第 62/100/118/126 行）。
- 这条优化带来一个 **可观测的时延差**：命中 \(\approx t_{\text{rest}}\)，未命中 \(\approx c\cdot n + t_{\text{rest}}\)，差值 \(\Delta t \approx c\cdot n\)。
- 攻击者可把这个时延差当作 oracle，通过 **测时间 + 逐 token 搜索** 还原他人前缀，构成 **时序侧信道 / prompt 窃取** 风险（本仓库 README 第 100 行明确标注 *Side-Channel KV Cache Attack*，资料由 PR #9 / 提交 `24684ac` 引入）。
- 标题中的 **Possible** 提示这是对潜在漏洞的可行性探讨，阅读应以威胁建模视角对待，具体结论以幻灯片原文为准（待本地验证）。
- 缓解总纲是 **抹平时延差** 或 **切断 oracle**：隔离/分区、关闭跨租户复用、时延规整、速率限制、架构上避免秘密入共享前缀；这些与吞吐直接对冲，需按「租户是否互信」权衡。
- 本仓库是资料库、无运行时代码；真正的可配置参数在主仓库 `sgl-project/sglang` 与文档站（衔接见 `u4-l3`）。
