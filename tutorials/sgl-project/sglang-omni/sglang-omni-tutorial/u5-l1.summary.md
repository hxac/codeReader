本讲回答「框架如何识别磁盘上的模型该用哪份 PipelineConfig」——靠全局注册表 PIPELINE_CONFIG_REGISTRY。它是「架构名 → PipelineConfig 子类」的进程级字典，由 import_pipeline_configs 在模块导入时用 pkgutil 自动扫描 models/ 子包建立（@lru_cache 仅执行一次），实现「新建目录即被发现」的零配置接入。模型包须满足两条硬约定：config.py 必须声明 EntryClass（默认入口配置类，缺失则 AssertionError），且该类的 architecture ClassVar 必须与 HF config.json 的 architectures 字段逐字符相等；architecture_aliases 兼容同一架构的历史多命名，注册时与主名一起展开成多个键。匹配链路 resolve_config_cls_for_model_path 为：权重路径经 AutoConfig / 原始 json / Mistral params 三条回退取得架构名，再调 get_config 查表。本讲还引入 ModelCapabilities——冻结、全布尔、无默认值的静态架构能力声明（五标志），以 CAPABILITIES 符号挂在模型包 __init__.py，查询借助配置类 __module__ 反向定位到包。关键边界：能力标志是「架构级上限」，checkpoint 能否启用由 PipelineConfig 方法（如 supports_uploaded_voice_references）决定，二者可不一致。
