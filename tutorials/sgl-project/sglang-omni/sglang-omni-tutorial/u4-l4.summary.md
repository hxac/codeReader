本讲打开流式语音合成的调度机制，覆盖三个模块：流式调度器骨架、流式 vocoder 模板与具体实现、chunk 累积与 flush 时机。

`StreamingSimpleScheduler` 是带请求级生命周期的调度器骨架，承接 u4-l1 的 SimpleScheduler：保留 `inbox/outbox/start/stop/abort` 五件套契约，额外按 `request_id` 追踪 `new_request → stream_chunk×N → stream_done` 的跨消息状态机；非流式请求仍走 `compute_fn`，且会被刻意挡在流式路径之外。三类进入消息由 `_handle_message` 分流，子类只覆盖五个 hook 表达语义。

核心结论有三点。其一，「先于 payload 到达的 stream_done」由 Stage 的 `can_accept_stream_before_payload` 开关配合调度器的 `_pending_done` 兜底，先挂起、payload 到达再补 flush。其二，`StreamingVocoderBase` 是模板方法基类，把状态注册表、ingest→decode→emit→收尾、abort 释放、迟到 chunk 丢弃固化进基类；`Code2WavScheduler` 则直接继承骨架、自管缓冲与游标，带左上下文窗口解码消除拼接伪影。其三，解码时机为「累积到 `stream_chunk_size` 才解」，判据 `ready = len(code_chunks) - emitted`；首次产出非空音频触发 `code2wav_first_audio`，作为首音频时延锚点。`stream_done` 不可省：负责 flush 尾巴并发送 terminal `result`（流式客户端拿 metadata-only，非流式拿整段音频）。关键术语：`StreamItem`、streaming edge、`should_decode`/`decode_delta` 解耦。
