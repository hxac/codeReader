本讲聚焦纯 TTS 模型（单向文本→语音）如何接入 SGLang-Omni。核心结论：最小管线恒为三阶段——preprocessing（SimpleScheduler，CPU 轻量、每请求一次）→ tts_engine（OmniScheduler，自回归生成语音 codec 码本）→ vocoder（SimpleScheduler + batch_compute_fn，码→波形）；接入按官方 8 步清单落地为 config/stages/request_builders/payload_types 四件套。关键术语：AR 引擎产出 codec 码本而非文本，vocoder 为无状态可批处理解码器。另有几条硬纪律：HTTP 端点采样默认值会静默覆盖模型默认值，须用 explicit_generation_params（Pydantic model_fields_set）配 _IMPLICIT_SAMPLING_DEFAULTS 反向过滤，区分「用户显式传」与「端点填的」；payload_types 用 wire(...) 声明 emit/codec 序列化、DeclarativeStateBase 自动生成 to_dict；radix cache key 必须由 embedding 内容派生（build_embedding_cache_key_ids 用 blake2b 压成 id），否则不同参考音频命中同段 KV 致语音串台；张量勿在 request_builder 随手 .cpu()，只在归属边界搬运；abort 清理须挂在触碰共享状态的多个 scheduler（preprocessing 与 tts_engine 各挂同一幂等回调），因请求可能停在任意阶段。
