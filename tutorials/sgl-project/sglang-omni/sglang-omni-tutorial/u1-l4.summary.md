本讲串联「HTTP API 层」与「内部 Client」，完成从启动服务到拿到第一条回复的闭环。核心结论：`sgl-omni serve` 由 Typer 实现，职责仅是配置翻译——将 flag 经 `ConfigManager` 合并为一份 `PipelineConfig`，再交 `launch_server` 拉起管线、`Client`、FastAPI 与 Uvicorn；其中未知 `--key=value` 也会被当成配置覆盖（`--model-path/--config/--text-only` 三条来源互斥）。`/health` 据运行时 `running` 标志返回 200/503，`/v1/models` 只返回启动时定死的单模型名、不查运行时。请求主链路为 HTTP → FastAPI → Client → Coordinator → Stage，含两次类型转换（`ChatCompletionRequest → GenerateRequest → OmniRequest`）与一次提交（`coordinator.submit`/`stream`）。关键区分：`create_app` 只建 FastAPI 路由，`launch_server` 才管「拉起管线 + 跑 Uvicorn + 关停」的完整生命周期，启动超时由 `SGLANG_OMNI_STARTUP_TIMEOUT` 控制。
