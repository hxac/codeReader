本讲承接 u3-l3，解决「给定一条 stage 边该用哪种后端」的路由问题。核心主角是 `CommRouter`，它把跨 stage 搬运抽象为四种 `TransportKind`：同进程直传对象引用的 `LOCAL_OBJECT`、同节点 GPU↔GPU 走 relay 池的 `CUDA_IPC`、CPU 共享内存 `SHM`、跨节点网络的 `MOONCAKE`。路由决策只凭**局部性**：同进程优先 LOCAL_OBJECT，跨节点选 MOONCAKE，同节点且双方均 GPU 选 CUDA_IPC，其余兜底 SHM。除按「边」决策的 `outbound` 外，还有按实际载荷决策的 `outbound_payload`/`outbound_stream`：全 CPU 张量会务实降级到 SHM，CUDA 张量发给非 GPU 目标则直接报错。`LOCAL_OBJECT` 遵循「引用传递 + 只读」契约，零拷贝、无 relay、无 ACK，仅「同进程且双方非 TP」的边才够资格。`CommEngine` 把路由结果落成字节搬运，按目标分队列串行发送，执行「写 relay→发控制消息→等 ACK」三段式，并守住 **control-before-wait** 不变量：控制消息必须先于 `wait_for_completion` 发出，否则基于**信用（credit）**的后端（NIXL/Mooncake）会因接收方无法获知取货单而**死锁**。`StageIO` 负责 stage 对象与 `DataRef` 间的打包解包（含 dtype 对齐填充），并提供「直接 CUDA IPC」旁路：同 GPU 时把 IPC handle 内嵌进控制消息绕过 relay，可被 `disable_direct_cuda_ipc_payload` 关闭。重要陷阱：跨节点靠 `remote_stage_names` 白名单显式声明，遗漏会被**静默降级**为 cuda_ipc/shm 而不报错。
