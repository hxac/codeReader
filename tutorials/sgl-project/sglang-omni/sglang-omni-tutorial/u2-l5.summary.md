本讲拆解配置结构本身——即「模型定义（提供拓扑）」与「模型无关运行时」之间的契约，核心是三个 schema。StageConfig 是配置层最小积木，字段分五组：身份构造、路由（next/terminal/route_fn）、GPU/并行/进程、fan-in（wait_for/wait_for_fn/merge_fn）、streaming（stream_to/stream_done_to_fn）。贯穿全篇的模式是「静态全集 + 请求感知子集」：静态字段列出所有可能，对应的 `_fn` 在运行时按请求裁剪，返回 None 则暂挂等待。PipelineConfig 组装阶段并提供管线级控制（entry_stage、fused_stages、terminal_stages_fn、runtime_overrides），模型作者经子类化与 ClassVar（如 architecture）接入。CommConfig 只调通信缓冲参数、不选传输后端，传输由 CommRouter 按阶段局部性派生。resolved_entry_stage/terminal_stages/gpu_placement 三个派生属性从 stages 自动算出以保证单一真相；静态校验（_validate_general/_validate_fusion）在构造时即拦下非法拓扑，包括 next/terminal 互斥、fan-in 必带 merge_fn、非 TP 阶段必声明 process、tp_size 与 gpu 列表等长。
