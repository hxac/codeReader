本讲下钻到 OmniScheduler 选好 batch 之后真正「算」的那一层——`ModelRunner`。它是所有 AR（自回归）阶段的共享执行骨架，把「拿到 batch → 跑前向 → 采样 → 抽输出」这条不变主干固化进基类 `execute()`：`build ForwardBatch → before_* 钩子 → custom/标准 forward → post_* 钩子 → finalize`，模型差异收口到命名一致的钩子（`before_*` 原地改 batch、`custom_*_forward` 返回前向结果或 None 走标准路径）。`ForwardBatch` 复用上游 SGLang 已算好的前向输入包，不重造。采样管线在基类统一，顺序固定：重复惩罚 → codec 屏蔽 → 装种子（无种子行用 request_id 派生兜底种子以保 TP 一致）→ 调上游 SGLang `sample` → 可选记 logprob。`ThinkerModelRunner` 在 prefill 的 `custom_prefill_forward`、标准前向之前，把 image/video/audio 占位 token 处的文本 embedding 替换为真正多模态向量并带上 deepstack 视觉向量。「反馈式 AR」概念（设计文档称 `FeedbackARModelRunner`/`FeedbackStrategy`，当前仓库内联而非独立类）由 `QwenTalkerModelRunner` 落地：`before_decode` 写 `_feedback_buffer` → forward 产出码本与反馈 → `post_decode` 抽 `_output_codes`/`_output_embeds`、推流 code2wav 并回填 `pending_feedback_queue`，形成闭环；状态载体为 `SGLangARRequestData` 的两个队列。
