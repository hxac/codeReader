本讲拆解 launch_plan.py：`build_launch_plan` 把已校验 Config 无副作用地折叠成不可变的 `LaunchPlan` 值对象，`plan.kind` 四值决定进程拓扑——worker（进程内直跑）、command（套 torchrun）、supervisor（起 producer+consumer）、managed_supervisor（额外起本地 Mooncake/capture 服务）；cli.py 据此分流，worker 走 `_train`，其余走 `run_commands`。关键概念：`_resolve_role` 派生角色，colocated→all、disaggregated→both、续训→consumer，其中 both 为 launch 专属不入 YAML；`world_size=nnodes×nproc_per_node` 是 worker 与 command 的分水岭（等于 1 则降级为 worker）；`run_commands` 遵循「首个失败即优雅收割」语义，靠 shutdown_grace_s 宽限与信号转发防止子进程孤儿化；`--plan` 可在不占 GPU 的前提下零开销预览整张计划。
