本讲是 SpecForge DataFlow 运行时（specforge/runtime/）的鸟瞰总纲。运行时对外只暴露四个拓扑构建器——build_offline_runtime（colocated 离线）、build_disagg_offline_runtime（disaggregated 离线）、build_disagg_online_producer、build_disagg_online_consumer——由两道分发器 build_training_run → build_disaggregated_run 按「数据模式 × 部署模式」自动路由；真实拓扑只有三种，因在线特征捕获必须由外部 SGLang 完成，online 恒为 disaggregated。

核心结论是「四路径坍缩成一条主链路」Trainer → FeatureDataLoader → TrainerController → TrainerCore：三个承载训练器的构建器共用装配器 _assemble_trainer，拓扑差异几乎全部收敛为两个变量——参考源 ref_source（离线为可重迭代的 refs 列表，在线为消费一次的 queue 流）与特征存储 store（LocalFeatureStore / 共享目录 / Mooncake）。唯一例外是在线 producer，只返回 (workers, drive_producer)，自身不训练。

系统按职责切成四个平面：控制面（调度/记账，只传元数据）、数据面（搬运张量）、推理面（捕获特征、只提交元数据）、训练面（算梯度、对部署形态无感知）。铁律「张量不得进入控制面」由 assert_no_tensors 递归守卫焊成硬约束，FeatureDataLoader 是 refs+store 到 TrainBatch 的唯一桥梁。

引入的关键术语：平面（plane）、参考源（ref source）、quantum（= dp_size × batch_size × accumulation_steps，producer/consumer 窗口握手单位）、SampleRef（纯元数据指针）、TrainBatch（唯一带张量的契约）。在线 disaggregated 标准通路分 producer 池（SGLang 捕获 → 张量直写 Mooncake → 发布 SampleRef）与 consumer 池（rank0 的 RefDistributor 去重并写 SQLite 账本 → 每 rank 一个 inbox → FeatureDataLoader → DPAckController）；RefDistributor 只派发完整 quantum 窗口、绝不派发残缺 optimizer 步，每边界 lockstep 一笔 durable 事务后才删 feature。在线仅有 consumer-only 恢复（复用 retained 账本与精确匹配 checkpoint，不重启 producer），num_epochs 在线时只控制 producer 的 prompt 轮数，consumer 始终 consume-once。
