本讲放大数据面，回答「特征张量存在哪、怎么跨进程搬、怎么变成 TrainBatch」，含四个模块：FeatureStore 契约与 LocalFeatureStore、跨进程后端 SharedDir/Mooncake、离线 ref 来源、收口桥梁 FeatureDataLoader。

FeatureStore 是数据面唯一张量拥有者，只暴露 put/get/release/abort/gc 五个生命周期方法，靠「租约（FeatureHandle）+ 代数（generation）」机制保证：代数进 URI 拒绝旧 ref 别名新数据，原子租约登记防 use-after-free，consume-once 释放配合 max_resident_bytes 背压与 gc 限内存。三种后端（进程内、共享目录、RDMA）实现同一契约，使 trainer/loader 对传输后端完全无感，后端选择只发生在装配层（DISAGG_BACKEND）；retain_on_release 区分在线 consume-once 释放与离线保留多 epoch，Mooncake 额外做硬钉、_freed 立即逻辑释放与 drain_pending_removals 回收。

离线 ref 由 file://（只引用不复制）或 disagg_ingest 写 manifest（写盘前 assert_no_tensors 强制张量不进控制面）产生。FeatureDataLoader 是 SampleRef+Store→TrainBatch 的唯一桥梁，默认 clone_on_fetch 克隆并立刻归还租约；refs 模式输入固定列表、可多 epoch 重迭代且 seek 续训，queue 模式是流、一次性消费且 seek 报错——离线是数据集、在线是流，在线借 defer_ack_until_durable 把 durable ack 让给 DPAckController。
