本讲讲解 SpecForge 运行时「控制面」（调度、去重、租约、记账）与「数据面」（搬运张量）之间的对话语言——一批纯标准库的跨平面契约（cross-plane contracts），全部定义在 `contracts.py` 中，刻意不导入 torch。三大最小模块：(1) 控制面元数据记录 `PromptTask`（待捕获 prompt 工作单元）、`FeatureSpec`（命名张量的形状/类型描述）、`SampleRef`（指向一个样本全部特征的指针，仅用 `feature_store_uri` + `feature_keys` 定位而自身不持任何张量，恒等于单个样本）、`FeatureHandle`（带 `generation` 自增的归还租约令牌），均以 `frozen=True` 声明为不可变值对象；(2) 数据面唯一允许携带张量的契约 `TrainBatch`，故意不冻结，仅存活于 trainer 一侧，是 `FeatureDataLoader` 把若干 ref 取回张量并 collate 后的产物；(3) 运行时守卫 `assert_no_tensors`，递归扫描对象，靠鸭子类型（模块根属 torch/numpy 或同时具备 `dtype`+`shape`+`device`）识别张量并抛 `TypeError`，对纯标量长序列跳过递归以省开销，报错带路径面包屑精确定位。核心结论：控制面只传元数据、张量只在数据面/`TrainBatch`，且张量全程待在 `FeatureStore` 不跨进程边界，使控制面消息可序列化为 JSON 跨节点轻量传递，是 SpecForge 在线分离式训练协同的根基。该守卫被 controller 各入口及 manifest 写盘前调用，把「张量不得进入控制面」从约定焊成硬约束。
