本讲讲清 `AlgorithmRegistration` 的「可执行半边」——provider 端口体系。核心结论是：`AlgorithmProviders` 容器按 `step / model / offline / server_streaming / vocab_mapping_modes` 五组端口组织算法「会动的代码」，且这些端口只描述算法行为、不描述部署拓扑。`make_registration` 在注册期对契约与 provider 做 parity 校验：特征 `(mode, modality)` 键集合必须完全相等，capture layout 物化名必须正好等于 `storage.required_tensors`，任何漂移即 fail-fast；端口「有」与「无」本身也是被强校验的属性（如 P-EAGLE 只在线不离线）。捕获层解析分两条路线：EAGLE3 走「运行覆盖→草稿配置→目标结构兜底公式 `[1, L//2-1, L-4]`」的三级回退，强制 3 层；DFlash 家族读 `dflash_config.target_layer_ids`，层数可配、默认 1。`OfflineCaptureLayout`/`ServerCaptureLayout` 把通用四类来源（`input_ids`/`loss_mask`/`aux_hidden_states`/`last_hidden_states`）经 `materialize()` 翻译成各算法自己的离线 schema 名。最后由 reader/normalizer/collator 与 `defaults.py` 共享钩子把记录对齐成训练 batch。重依赖一律懒导入，使注册表解析保持无副作用。
