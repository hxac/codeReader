本讲是 SpecForge 源码的「索引地图」，目的是让人在 30 秒内定位任意功能的目录。核心结论：全项目只有一个类型化命令入口 `specforge`（含 `train`/`export`/`benchmark` 三子命令），`cli.py` 仅做命令解析与进程生命周期，所有 worker 路径最终汇入组合根 `build_application_run(resolved).run()`。

引入并供后续讲义承接的关键术语有：**类型化配置**（用 Pydantic 描述 YAML，未知字段直接报错）、**组合根**（`application/composition.py`，唯一把配置、算法注册、训练器拼装到一起之处）、**算法注册表**（用 `training.strategy` 查得不可变的 `AlgorithmRegistration`，装配层只消费不查名）、运行时铁律「**控制面只传元数据、数据面才传张量**」、以及**七段配置**（model/data/training/tracking/profiling/runtime/deployment）。一次 `train` 的主轴为 `cli.py → config/schema.py → composition.py → launch.py → training/assembly.py → trainer.py`，旁挂 `runtime/`（DataFlow，含在线引用集中式 dispatcher `ref_distributor.py`）与 `inference/`（采集面）。仓库根 `examples/configs/` 是黄金样例 YAML 来源，`scripts/` 放数据与离线特征准备脚本，旧 `scripts/train_*.py` 已删除而非弃用，无兼容路径。
