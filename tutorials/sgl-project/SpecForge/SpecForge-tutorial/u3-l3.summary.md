本讲剖析 `launch.py` 的四个拓扑构建器。SpecForge 运行形态由两条正交轴线决定：数据模式（online/offline，看 `data.hidden_states_path` 是否填）与部署模式（colocated/disaggregated，看 `deployment.mode`）。因在线特征捕获须交外部 SGLang，online 恒为 disaggregated，真实拓扑仅 colocated offline、disaggregated offline、online 三种。四个构建器由组合根经两道分发器（`assembly.build_training_run` 按 `deployment.mode`、`disaggregated.build_disaggregated_run` 再按 `mode` 与 `role`）路由到位。

核心结论：三个承载训练器的构建器（offline、disagg offline、online consumer）汇聚到同一统一训练路径 `Trainer → FeatureDataLoader → TrainerController → TrainerCore`（由共享 `_assemble_trainer` 实现），拓扑差异仅翻译为 `ref_source`（离线为可重迭代 refs 字典、在线为消费一次 queue 字典）与 `store`（LocalFeatureStore/外部存储/Mooncake）两变量。两特例需承接：离线 disagg producer 不调任何 `build_*`，直接 `ingest_offline_features` 灌特征并写 manifest；在线 producer 唯一不汇聚 Trainer，返回 `(workers, drive_producer)`，靠 quantum（dp×batch×accum）握手与水位校验协同 consumer。关键术语：拓扑构建器、组合根、参考源（ref source）、特征存储、元数据账本、承载训练器的构建器。离线两路无账本、可多 epoch 与断点续训；在线 consumer 用 SQLite/DPAckController 账本、`retain_on_release=True`、`num_epochs=1`，仅 consumer 可恢复。
