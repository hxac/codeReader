本讲围绕 `contracts.py` 讲清算法的「纯数据契约」。`AlgorithmSpec` 是只含静态约束、不含工厂/模型类/运行时对象的登记表，它是 `AlgorithmRegistration` 的「纯契约」半边，另半边是可执行 `providers`（留待 u4-l3）。四件套为 `name`、`draft`（`DraftRequirement`）、`feature_contracts`（多个 `FeatureContract`）、`capabilities`（`AlgorithmCapabilities`）。`FeatureContract` 是心脏，以 `(mode, modality)` 二元组为身份键，`mode` 取 `FeatureMode` 枚举的 `OFFLINE`/`STREAMING`，声明训练所需的 `required_tensors`/`optional_tensors` 与离线 `OfflineStorageContract`；校验规定 OFFLINE 必须带 storage、STREAMING 不得带，required 与 optional 不相交，同一 `(mode, modality)` 键不可重复。`DraftRequirement` 只约束可兼容的草稿架构名（只放名字、不含模型类），`AlgorithmCapabilities` 约束注意力后端、vocab mapping、compact teacher 等算法自身能力。关键不变量：`_assert_pure_value` 递归守卫在 `AlgorithmSpec` 构造末尾拒绝一切 callable/type，保证解析期轻量、可哈希、声明与实现解耦。
