# 投机解码原理

## 1. 本讲目标

本讲是 SpecForge 全部训练逻辑的「物理直觉」基础。SpecForge 的存在意义就是训练一个能让投机解码跑得更快的草稿模型——如果不理解投机解码本身在做什么、为什么能加速、为什么还不改变输出，后面所有关于「特征捕获」「接受率」「草稿树」的内容都会变成无源之水。

学完本讲，你应该能够：

- 说清楚大模型逐 token 生成时的**延迟瓶颈**是什么，以及为什么这是投机解码能成立的物理前提。
- 复述投机解码的 **prefill / drafting / verification 三阶段**，并指出每个阶段由目标模型还是草稿模型负责。
- 用**拒绝采样（rejection sampling）**解释为什么投机解码能保证输出分布与单独使用目标模型完全一致。
- 看懂 SpecForge 官方概念文档 `docs/concepts/speculative_decoding.md` 中的每一段，并能用永久链接定位。

本讲不涉及任何 Python 代码或命令，是一节纯概念课。它直接承接 [u1-l1 项目定位与生态](./u1-l1-project-overview.md) 中引入的「目标模型 / 草稿模型 / 接受率」三个术语。

## 2. 前置知识

在进入源码之前，先用最通俗的方式补两个基础概念。

**（1）自回归生成（autoregressive generation）。**
大语言模型生成文本时，是「一次一个 token」往后写的：每生成一个 token，都要把它接回序列末尾，再预测下一个。生成 100 个 token 就要串行地跑 100 次前向计算。这就是「逐 token」的含义。

**（2）显存带宽瓶颈（memory-bound）。**
一次前向计算的时间，主要花在「把整个模型权重从显存搬到计算单元」上，而不是花在「算乘加」上。换句话说，计算单元经常在**等权重到位**。一个直观推论是：在这一次前向里，你是只算 1 个 token 的最后一步，还是顺带把 5 个 token 的中间结果都算出来，时间差别并不大——因为「搬权重」这件事只发生一次。这一点是本讲后半段「验证多个 token 几乎不增加延迟」的全部根据，务必先记住。

承接 [u1-l1](./u1-l1-project-overview.md) 已建立的术语：

| 术语 | 含义 |
|---|---|
| 目标模型（Target Model） | 真正要服务用户的大模型，输出质量的天花板 |
| 草稿模型（Draft Model） | 小模型，负责「提前猜」接下来几个 token |
| 接受率（acceptance rate） | 草稿猜的 token 中被目标模型采纳的比例 |

> 提示：u1-l1 里还提到 SpecForge 训练产物最终交给 SGLang 去做推理加速。本讲解释的就是「SGLang 端那套加速机制」的原理——SpecForge 训练草稿模型，正是为了让这套机制里的「猜得更准」。

## 4. 核心概念与源码讲解

本讲拆成三个最小模块，正好对应官方文档的三段叙事：

1. **延迟瓶颈分析**——为什么会有投机解码这个想法。
2. **草稿 / 目标协作**——两个模型如何分工，三阶段如何串起来。
3. **验证与拒绝采样**——为什么加速还不改变输出分布。

### 4.1 延迟瓶颈分析

#### 4.1.1 概念说明

大模型推理的痛点不是「算得慢」，而是「搬得慢」。文档把这一点讲得很直白：逐 token 的解码过程**主要被显存带宽卡住**——每生成一个 token，推理引擎都得把整套模型权重重新搬进显存一次。既然瓶颈是「搬权重」而不是「做计算」，那么就有一个可被利用的空隙：在「搬一次权重」的窗口里，其实可以顺带多算几个 token。

投机解码的灵感正源于此：**让一个小模型提前预测接下来的几个 token，再让大模型一次性把这些 token 都验证掉。** 因为验证阶段也是 memory-bound 的，验证多个 token 的耗时和生成单个 token 差不多。

#### 4.1.2 核心流程

用伪流程描述「为什么加速」的因果链：

```text
逐 token 生成 → 每生成 1 个 token 都要搬一次完整权重 → 延迟被显存带宽主导
        │
        ▼
若在「搬一次权重」的窗口里顺带处理 K 个候选 token → 单次前向的产物从 1 个变成最多 K 个
        │
        ▼
平均每个 token 分摊到的「搬权重」时间下降 → 吞吐/延迟显著改善
```

关键定量直觉（不严谨，仅帮助理解）：

- 假设单次前向「搬权重」的固定开销为 \(T_{\text{mem}}\)，每个 token 的额外计算开销为 \(T_{\text{calc}}\)，且 \(T_{\text{mem}} \gg T_{\text{calc}}\)。
- 单独生成 1 个 token 的耗时约为 \(T_{\text{mem}} + T_{\text{calc}}\)。
- 在同一次前向里顺带处理 \(K\) 个 token 的耗时约为 \(T_{\text{mem}} + K \cdot T_{\text{calc}}\)。
- 由于 \(T_{\text{mem}}\) 远大于 \(T_{\text{calc}}\)，后者只比前者多一点点，却可能换来多个被接受的 token。这正是「验证多个 token ≈ 生成一个 token」的数学根源。

#### 4.1.5 小练习与答案

**练习 1**：如果某天出现了「计算瓶颈（compute-bound）」而非「显存带宽瓶颈」的推理场景，投机解码还有意义吗？

**参考答案**：意义会大幅下降。投机解码加速的前提是「搬权重」的开销远大于「多算几个 token」的开销；一旦变成 compute-bound，\(T_{\text{calc}}\) 不再可忽略，验证 \(K\) 个 token 就会线性增加耗时，加速比随之坍缩。这也是为什么投机解码在「大模型 + 小批量」的在线服务场景里收益最高。

**练习 2**：文档说「每生成一个 token 都要把整套权重搬进显存」。请用一句话解释为什么 KV cache 不能消除这个开销。

**参考答案**：KV cache 缓存的是「历史 token 的注意力中间状态」，避免重复计算 attention 的 key/value；但**模型权重本身**（每层的投影矩阵等）每次前向仍需从显存读到计算单元，这部分才是 memory-bound 的开销主体，KV cache 并不替代它。

### 4.2 草稿模型与目标模型的协作

#### 4.2.1 概念说明

有了「为什么能加速」的物理前提，下一步是「谁来猜、谁来验」。投机解码里始终有两个模型：

- **目标模型（Target Model）**：要部署给用户的大模型，质量天花板，负责 prefill 与最终验证。
- **草稿模型（Draft Model）**：小模型，专门用来「提前猜」接下来几个 token。

文档列举了草稿模型的几种常见形态：n-gram 模型、同家族的小型预训练模型（如用 8B 当 70B 的草稿）、或者单独训练的小模型（EAGLE 就属此类，是 SpecForge 的主力，详见下一讲 [u1-l4](./u1-l4-eagle3-concepts.md)）。注意：**草稿模型只是「提议者」，最终拍板的是目标模型**——这一点决定了下一节的「不改变输出」性质。

#### 4.2.2 核心流程

文档把一次投机解码的完整工作流拆成三阶段。用流程式文字描述如下：

```text
[1] prefill（预填充）
      执行者：目标模型
      动作：把整段 prompt 一次性吃进去，算出初始的隐藏状态 / KV cache
      说明：这一步和普通推理的 prefill 完全一样

[2] drafting（草拟）
      执行者：草稿模型
      动作：基于当前状态，自回归地连续预测接下来的 N 个候选 token
      说明：草稿模型很小，drafting 耗时可忽略

[3] verification（验证）
      执行者：目标模型
      动作：把这 N 个候选 token 一次性喂给目标模型做并行验证
            被接受的 token 接到输出序列末尾
            遇到第一个被拒绝的 token 就停下，并用目标模型的分布补一个 token
            然后从「最后一个被接受的 token」出发，回到 [2] 继续循环
      说明：因为 memory-bound，并行验证 N 个 token 的耗时 ≈ 生成 1 个 token
```

一个重要细节：验证阶段**只要碰到第一个被拒绝的候选 token 就终止本轮**，其后草稿模型猜的 token 全部丢弃；下一轮草稿从「最后一个被接受的位置」重新开始猜。这也是为什么「接受率」直接决定加速效果——接受得越多，一轮里被采纳的 token 越多，加速比越高。

#### 4.2.5 小练习与答案

**练习 1**：为什么 drafting 阶段「串行猜 N 个 token」不会成为新瓶颈？

**参考答案**：因为草稿模型远小于目标模型，其单步前向的「搬权重」开销也小得多；即便串行做 N 次，总耗时仍远小于目标模型做一次验证。文档也正是用「the drafting time is insignificant」来描述它。

**练习 2**：verification 阶段遇到第一个被拒绝的 token 后，为什么要把它之后的所有候选 token 全部丢弃，而不能「跳过这个继续用后面的」？

**参考答案**：因为草稿模型是自回归的——后面的候选 token 是基于前面被拒 token 的假设续写出来的，前提一旦不成立，后续预测就失去了依据。所以必须从「最后一个被接受的位置」重新开始新一轮 drafting。

### 4.3 验证阶段与拒绝采样

#### 4.3.1 概念说明

到目前为止，读者最自然的疑问是：「让一个小模型参与生成，输出质量会不会变差、甚至跑偏？」投机解码最优雅的地方就在于：**它保证最终输出分布与单独使用目标模型完全一致**。也就是说，从概率分布的角度看，加不加草稿模型，结果在统计上是无差别的（只是更快）。

实现这一保证的机制叫**拒绝采样（rejection sampling）**：目标模型不是「无脑接受」草稿提议，而是按一个特定的概率规则决定接受/拒绝，必要时再补采一个 token。只要这个规则设计正确，最终每一步采到 token \(x\) 的概率，就恰好等于目标模型自己给出的概率 \(p(x)\)。

#### 4.3.2 核心流程

记目标模型给某 token \(x\) 的概率为 \(p(x)\)，草稿模型给同一 token 的概率为 \(q(x)\)。文档给出的接受规则（用本讲要求的公式格式重写）如下：

- 当 \(q(x) \le p(x)\) 时：该 token **直接接受**。
- 当 \(q(x) > p(x)\) 时：以概率

\[
\frac{p(x)}{q(x)}
\]

接受；若被拒绝（概率为 \(1 - \frac{p(x)}{q(x)}\)），则从「残差分布」中重新采样一个 token 作为本步的最终输出，残差分布为

\[
p'(x) = \frac{\max\bigl(0,\; p(x) - q(x)\bigr)}{\displaystyle\sum_{x'} \max\bigl(0,\; p(x') - q(x')\bigr)}.
\]

等价的、更紧凑的写法是「接受概率」：

\[
r(x) = \min\!\left(1,\; \frac{p(x)}{q(x)}\right).
\]

直观理解这两条规则：

- 若草稿「低估」了这个 token（\(q < p\)），说明目标比草稿更喜欢它，那就**照单全收**。
- 若草稿「高估」了这个 token（\(q > p\)），说明草稿热情过头，目标只按 \(p/q\) 的比例接受；被拒的部分，用「目标比草稿多出来的那截概率」\(p(x)-q(x)\) 重新分配。

把「被接受」和「被拒后补采」两种情况的概率加起来，每一步最终落到 token \(x\) 的总概率恰好等于 \(p(x)\)——这就是输出分布不变的数学内核。文档同时指出，完整的数学证明在原论文的附录里。

文档把这一保证和规则写在 Overview 的收尾处：

> [docs/concepts/speculative_decoding.md:24-25](https://github.com/sgl-project/SpecForge/blob/a4fca140bc5fd12d6db40bc694c1a7dd790da57d/docs/concepts/speculative_decoding.md#L24-L25) —— 这一段说明：投机解码保证输出分布与单独使用目标模型相同，原因是目标模型用拒绝采样决定候选 token 的接受；并给出了规则——记目标概率 \(p(x)\)、草稿概率 \(q(x)\)，若 \(q(x)<p(x)\) 则接受，若 \(q(x)>p(x)\) 则以概率 \(1-p(x)/q(x)\) 拒绝并从 \(p'(x)=\text{norm}(\max(0,p(x)-q(x)))\) 重采样，正确性的数学证明见原论文附录。

#### 4.3.4 代码实践

这是一个**手算型实践**，用来确认你理解了接受规则。

1. **实践目标**：用具体数字走一遍拒绝采样，体会「分布不变」。
2. **操作步骤**：假设词表里只有 3 个候选 token \(\{A, B, C\}\)。在某一步：
   - 目标模型分布 \(p = (0.5,\; 0.3,\; 0.2)\)。
   - 草稿模型提议了 token \(A\)，其分布里 \(q(A) = 0.6\)（即草稿把超过一半的概率压在了 A 上）。
   - 用 4.3.2 的公式计算：token \(A\) 的接受概率 \(r(A)\)；若被拒绝，从残差分布 \(p'\) 重采样时，\(A/B/C\) 各自的概率。
3. **需要观察的现象**：拒绝后重采样时，A 在残差分布里的概率被「削」掉了。
4. **预期结果**：
   - 接受概率：\(r(A) = \min(1,\, p(A)/q(A)) = \min(1,\, 0.5/0.6) = \min(1,\, 0.833) \approx 0.833\)。
   - 残差：\(\max(0, p-q)\) 对 A/B/C 分别为 \(\max(0, 0.5-0.6)=0\)、\(\max(0, 0.3-0)=0.3\)、\(\max(0, 0.2-0)=0.2\)，和为 \(0.5\)。
   - 归一化后的残差分布：\(p'(A)=0/0.5=0\)、\(p'(B)=0.3/0.5=0.6\)、\(p'(C)=0.2/0.5=0.4\)。
   - 综合看 A 落地概率：被接受 \(0.833 \times 1\) 的贡献……本步演示到此即可；关键是看到「A 被高估时，残差里 A 的概率被削为 0」，这正是纠偏机制。精确的「落地概率之和等于 \(p(A)\)」可自行加总验证。
5. 上述数字为手工推演，不依赖任何运行环境；若要观察真实采样轨迹，需要接入一个支持投机解码的推理后端（如 SGLang），标记为「待本地验证」。

#### 4.3.5 小练习与答案

**练习 1**：若草稿模型与目标模型的分布完全相同，即 \(q(x) = p(x)\)，接受概率是多少？这意味着什么？

**参考答案**：\(r(x) = \min(1, p/q) = \min(1, 1) = 1\)，所有候选 token 全部被接受。这意味着草稿与目标「想法一致」时零浪费，加速比最大化——这也是为什么训练一个「尽量贴近目标分布」的草稿模型（SpecForge 的核心目标）能直接提升接受率。

**练习 2**：有人说「既然目标模型最终拍板，那草稿模型随便选一个就行」。这个说法对吗？

**参考答案**：不完全对。输出分布不变确实由拒绝采样保证，与草稿质量无关；但**加速效果**强依赖于草稿质量。草稿越准，接受率越高，每轮被采纳的 token 越多，加速比越大；草稿太差则大量候选被拒，每轮只产出 1 个 token，几乎退化成普通解码，还白白付出 drafting 开销。所以 SpecForge 才要专门训练草稿模型。

## 6. 本讲小结

- 大模型逐 token 解码的延迟主要被**显存带宽**卡住，每生成一个 token 都要把整套权重搬进显存一次，这是投机解码能成立的物理前提。
- 投机解码由**目标模型**和**草稿模型**协作：目标模型负责 prefill 与验证，草稿模型负责快速「提前猜」接下来的几个 token。
- 工作流分 **prefill → drafting → verification** 三阶段；验证遇到第一个被拒 token 即终止本轮，从最后一个被接受处重新开始 drafting。
- 由于验证是 memory-bound，**并行验证 N 个 token 的耗时 ≈ 生成单个 token**，这是加速的来源。
- 目标模型用**拒绝采样**决定接受/拒绝，并在被拒时从残差分布补采，从而**保证最终输出分布与单独使用目标模型完全一致**——加速但不改变结果。
- 草稿模型的质量直接决定**接受率**与加速比，这正是 SpecForge 要专门训练草稿模型的根本原因。

## 7. 下一步学习建议

本讲建立的是「通用的投机解码」直觉，草稿模型还停留在「同家族小模型 / n-gram」这种朴素形态。下一讲 [u1-l4 EAGLE3 特征式草稿原理](./u1-l4-eagle3-concepts.md) 将进入 SpecForge 的**主力方法**：

- 草稿模型不再直接吃 token，而是吃**目标模型的隐藏状态（特征式草拟）**。
- 训练时引入**训练时测试（Training-time Test）**以提升接受率。
- 使用**动态草稿树**只保留最可能被接受的候选。

建议在进入 u1-l4 前，先确保自己能默写出本讲的三阶段流程与拒绝采样的接受概率公式 \(r(x)=\min(1, p(x)/q(x))\)——它们会在 EAGLE3、训练策略（u6-l2）、推理捕获（u7-l5）中反复出现。随后即可进入 [u2 第一次运行训练](./u2-l1-first-run.md)，亲手把一个草稿模型训练起来。
