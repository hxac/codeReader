本讲解析 LanceDB 的 Python 绑定如何把 Rust 核心类型变成可 `import` 的 Python API，分两层：Rust 绑定层 `_lancedb`（PyO3 用 `#[pyclass]`/`#[pymethods]`/`#[pyfunction]` 暴露 `Connection`/`Table`/`Query`，用 `future_into_py` 把 Rust `async` 包成 Python 协程）与纯 Python 封装层 `lancedb`（补 docstring、类型提示、pandas/pyarrow 转换）。异步执行依赖 fork-safe 的 tokio 运行时（`LanceRuntime` + `AtomicPtr` + `pthread_atfork`），同文件的 `block_on` 让同步函数也能驱动 async。Python 侧提供两套等价 API：`AsyncTable`/`AsyncConnection` 与同步的 `LanceTable`/`LanceDBConnection`，后者靠常驻守护线程的后台事件循环 `LOOP`（`asyncio.run_coroutine_threadsafe`）把协程「堵着跑完」，两套共享同一核心。核心结论：同步是异步的阻塞包装，行为完全一致。本次更新重点为命名空间连接的 read-freshness 修复：`rest` 实现改走 Rust 原生构建（`LanceNamespaceDatabase::connect` 装 `ReadFreshnessContextProvider` 注入 `x-lancedb-min-timestamp` 头），并用镜像判定开关 `_route_pushdown_to_rust` 让 `QueryTable` 下推回到自带新鲜度头的 Rust 表，避免纯 Python 下推静默读到陈旧数据。
