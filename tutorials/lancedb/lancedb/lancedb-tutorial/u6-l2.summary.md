本讲聚焦"表集合如何被组织管理"，引入 `Database` trait 作为统一契约：`Connection` 仅是一层外壳，几乎所有方法都一行转发给内部 `Arc<dyn Database>`。连接时按 URI 前缀与 `manifest_enabled` 开关分流到三套实现——默认 `ListingDatabase`（表即文件夹、靠目录列举发现表、无元数据服务）、基于 lance-namespace 的 `LanceNamespaceDatabase`（支持多级命名空间）、以及留待 u6-l3 的 `RemoteDatabase`。核心结论：`ListingDatabase` 内嵌一个指向同根的 `LanceNamespaceDatabase`，故命名空间是其更深的底座而非对立面。两种模式的根本差异在于"表位置由谁决定":listing 由客户端拼 `uri/name.lance`，namespace 由服务端 `declare_table` 分配 location，因而支持多级命名空间、rename_table 与托管提交。引入的关键术语包括：表全局 id 形如 `ns1$t_a`（默认 `$` 分隔，亦为读新鲜度基线 key，承接 u5-l3）、`manifest_enabled` 是 listing 到 namespace 的迁移桥、`managed_versioning` 让表 manifest 提交由命名空间服务端托管。
