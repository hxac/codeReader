本讲把 LanceDB 的「版本」从一句承诺拆成可操作能力，覆盖两个最小模块。其一是 `table` 的版本机制：Lance 采用「不可变文件 + 单调递增版本」模型，任何写操作（add/update/delete/建索引）都会提交一个新版本、版本号 +1，纯读不变；版本是只读快照的编号，由底层 Lance 定义。关键认知是「当前指向哪个版本」是 `Table` 句柄内部的可变状态，而非表属性——时间旅行就是用 `checkout(version)` 或 `checkout_tag(tag)` 把句柄临时钉到旧快照（只读、只影响本句柄、并旁路读一致性），`checkout_latest()` 解除钉定，`restore()` 用旧版本覆盖「最新」（它是写、追加新版本、不删历史，且必须先 checkout）；`tags` 给版本起人类可读别名。其二是读一致性：本地后端由 `read_consistency_interval` 决定三档模式——`None`(Manual)、`Some(0)`(Strong)、`Some(d)`(Eventual)，表永远内部一致，档位只影响跨句柄可见性。其三是远程命名空间后端独有的读新鲜度机制：写时 bump 每表基线、读时注入 `x-lancedb-min-timestamp = max(baseline, now - interval)` 头，绕过服务端缓存确保「读得到自己的写」，本地表该机制为 no-op。核心 API 在本地与远程两套后端一致，真正实现在底层 Lance。
