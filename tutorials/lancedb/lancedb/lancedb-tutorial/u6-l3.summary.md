本讲是「存储后端与架构」单元收尾，专讲远程后端（`remote` 模块）。核心结论是远程后端把每次高层 API 调用翻译成一次 HTTP 往返，发给 LanceDB Cloud/Enterprise，其分层调用链为：`Connection/Table` 委托给 `RemoteDatabase`/`RemoteTable`（翻译成 HTTP 请求），再交给 `RestfulLanceDbClient`（构造请求、注入认证头、重试、发送），最终落到 `reqwest::Client` 完成网络 IO。客户端构造由 `ClientConfig` 汇总超时、mTLS、host 推导与 `HeaderProvider` 动态头部扩展点；`default_headers` 注入 `x-api-key`、`x-lancedb-database`、`x-lancedb-user-id` 等鉴权与路由头。重试由三件套驱动：`RetryConfig`（用户可选）经 `ResolvedRetryConfig` 填默认值，再由 `RetryCounter` 分连接/读取/请求三类独立计数、共享放弃判定，退避公式为 `backoff_factor × 2^request_failures + jitter`。关键安全取舍：写操作不在 5xx 上重试以防重复写入，读操作允许。`ServerVersion` 通过 `phalanx-version` 响应头做能力探测，用 `support_multivector/support_structural_fts/support_multipart_write` 适配多版本服务端。`RemoteTable` 在无状态 HTTP 之上模拟有状态句柄，用 `FreshnessState` 注入 `x-lancedb-min-version`/`x-lancedb-min-timestamp` 实现「读自己的写」，并靠缓冲整批数据让流式写入也能重试。传输用 Arrow IPC Stream（`application/vnd.apache.arrow.stream`）。本讲仅覆盖 `remote` 一个模块。
