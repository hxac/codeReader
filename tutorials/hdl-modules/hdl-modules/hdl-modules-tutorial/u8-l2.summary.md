本讲承接 u1-l4，讲透 `module_*.py` 的仿真钩子 `setup_vunit`：它在 Python 侧只负责「登记测试配置」，不写测试逻辑、不亲自跑仿真。核心模式有四：其一，用嵌套 `for` 循环或 `itertools.product` 把各 generic 取值展开为笛卡尔积（**generic 矩阵**），每组取值经 `self.add_vunit_config(test, generics=...)` 登记为一次仿真配置，非法/无意义组合用 `continue` 提前剔除。其二，引入「按测试名定制 generic」分发模式——以 `if "关键词" in test_name` 给不同用例塞不同 generic，避免笛卡尔积爆炸，fifo 将其抽成生成器 `generate_common_fifo_test_generics` 复用。其三，VHDL 侧测试台分 `run()` elsif 多用例分支与并发进程自检两种风格，共享 `runner_cfg` 魔法 generic 与 `test_runner_setup`/`cleanup`、`test_runner_watchdog` 三件套骨架；**断言式自检**靠 `check_*`/`assert` 量化正确性，看门狗防止死锁卡死 CI。其四，Python 侧靠 `test/conftest.py` 与 `tools/simulate.py` 在最前修 `PYTHONPATH`，`setup_vunit` 在 `add_modules` 时被回调登记，真正仿真在 `vunit_proj.main()` 启动。下一讲 u8-l3 讲同一文件的另一个钩子 `get_build_projects`，做综合侧资源回归。
