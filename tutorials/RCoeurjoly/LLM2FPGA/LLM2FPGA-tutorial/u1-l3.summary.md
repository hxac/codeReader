本讲（u1-l3）解释 LLM2FPGA 为何用 Nix flake 而非 pip/conda 管理工具链：因流水线上的 C++ 重型编译器（torch-MLIR、CIRCT、Yosys+yosys-slang、openXC7、Verilator）版本高度敏感，必须从精确 commit 重建才能保证可复现。核心拆为三模块：一是 flake 的 inputs 与版本 pin——每个工具被钉死到具体 commit/分支，引入 `flake = false`（只拉源码自编译）、`follows`（强制上游用我指定的依赖）、`circt-src` 指向作者 task3 fork 等关键概念；二是 devShell，用 `nix develop` 进入一个带全套工具的交互 shell，经 shellHook 配好 NEXTPNR_XILINX_DIR、PRJXRAY_DB_DIR 等环境变量，且工具版本与 `nix build` 产物完全一致；三是最微妙的「三套独立 LLVM/MLIR」策略：通用稳定 LLVM 21、CIRCT 用的 LLVM、torch-MLIR 单独 pin 的 LLVM（pin 到其 submodule 自带版本），目的是改 torch-mlir 源码时不触发 LLVM 重编。flake 的 outputs 另有 packages（产物）、checks（静态检查）、apps（如 docs-md）三类产物。
