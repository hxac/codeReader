本讲回答「四语言交织仓库如何保证每次提交都不写坏」这一工程问题，答案分两层：一是 CI 里的 `nix flake check`，把四类静态检查汇成一条命令；二是 `docs-md` 应用，把 Org 权威源转成 Markdown 派生物。两者都挂在 `flake.nix` 的 `checks`/`apps` 输出上，由 GitHub Actions 在每次 push/PR 时触发。

核心结论是「薄 CI、厚 flake」：`.github/workflows/ci.yml` 只有 checkout→装 Nix→`LC_ALL=C nix flake check` 三步，不写任何具体 lint 命令。`nix flake check` 本质是枚举并构建 `outputs.checks` 的每个派生；`runCommand` 把任意 lint 命令包成派生，命令失败即派生失败即 CI 红，末尾 `mkdir -p "$out"` 是「检查通过」的物理凭证。

`checks` 里有四个因语言制宜的检查：`nix`（deadnix+statix+nixfmt）、`python`（仅 `compile()` byte-compile，最轻）、`systemverilog`（verilator lint，只针对 matmul testbench，复用降级链产物 `tbDataSv` 与 `matmulSv`）、`shell`（shellcheck，唯一全仓扫描）。关键术语包括 `checks` 输出、`apps` 输出、`runCommand`、`writeShellApplication`、权威源与派生物约定。`docs-md` 是 `apps` 而非 `checks`，用 pandoc 只转换 `README.org`+`deliverables/*.org`+`docs/*.org` 三类面向读者的文档，CI 不强制 `.md` 与 `.org` 同步。
