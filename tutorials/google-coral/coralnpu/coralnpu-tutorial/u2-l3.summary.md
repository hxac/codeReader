本讲解决「没有真实芯片如何运行并观测 `.elf`」的问题。CoralNPU 的 Verilator 仿真有两条入口：命令行工具 `core_mini_axi_sim`（SystemC 体系，README Quick Start 路径，回归测试主力）和 C++ 库 `CoralNPUSimulator`（`hw_sim/` 轻量封装，不依赖 SystemC，供 npusim/Python 调用），二者最终驱动同一个 Verilator 模型 `VCoreMiniAxi`（由 RTL 顶层 `CoreMiniAxi` 生成）。8 个仿真器变体由 `BUILD` 的 `template_rule` 从同一份 `core_mini_axi_sim.cc` 批量生成，差别仅是链接的 testbench 库与 `VERILATOR_MODEL` 宏。核心启动链固定为：加载 ELF（经 AXI slave「写→读→校验」灌入 ITCM/DTCM，并把入口地址写入 CSR `PC_START`，即 `0x30004`）→ 解除时钟门控 → 释放复位（写 `RESET_CONTROL` `0x30000`）→ 等待 `io_halted` → 读 `STATUS`（`0x30008`）确认。`CoralNPUSimulator` 采用三层封装：抽象接口 → `CoreMiniAxiSimulator` → `CoreMiniAxiWrapper` → `hw_primitives`（`Clock::Step` 按上升/下降沿搬运 AXI）。观测内核状态有四种途径：halt/fault/wfi 信号、CSR STATUS、`--instr_trace` 指令轨迹，以及 mailbox 信箱（通过 `WriteCallback`/`ReadCallback` 捕获内核经 master 端口写出的 16 字节数据）。核心结论：`main` 成功返回后 CRT 执行 `mpause` 触发 `io_halted`，是仿真器判断「程序正常跑完」的信号；命令行路径靠 tohost/半主机与指令轨迹看输出，库式路径则靠 mailbox。
