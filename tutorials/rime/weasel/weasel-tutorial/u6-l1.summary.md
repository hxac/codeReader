本讲剖析 Weasel 独立的「配置+部署」进程 WeaselDeployer.exe。它与常驻 WeaselServer 分离，是因为改配置需重新部署、部署期间引擎不可用，须由独立进程经命名管道 IPC 协调 Server 进/出维护模式（StartMaintenance/EndMaintenance）；「申请维护→操作→恢复」这条骨架贯穿全讲。Configurator 是其大脑，用 deployer_initialize 做轻量初始化（而非完整 initialize），同时调用 librime 两套 API：核心 RimeApi（setup/deploy/sync_user_data）与高层 levers 模块的 RimeLeversApi。levers 的核心概念是「覆盖层」：用户个性化以 `__patch` 指令写入 `.custom.yaml`，叠加在发行版默认 `.yaml` 之上；RimeCustomSettings 是其抽象，由 customize_string 写补丁、load/save_settings 读写、is_first_run 控制安装向导。三类设置对话框分写三份配置：方案切换→default.custom.yaml、UI 配色→weasel.custom.yaml、用户词典导出/导入/备份/还原。/deploy、/sync、/dict 三种命令行模式共享同一套维护骨架，差异仅在中间那段操作。Deployer 与 Server 的跨进程握手靠两个命名互斥体：进程级单实例 WeaselDeployerExclusiveMutex 与操作级 WeaselDeployerMutex（被 Server 端 _IsDeployerRunning 探测）。
