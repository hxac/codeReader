本讲覆盖两个测试工程及其构建方式。`test/TestResponseParser` 是响应解析的纯单元测试，需澄清的关键事实是：它用的并非大纲所称的 Google Test，而是 Boost.LightweightTest（断言宏 `BOOST_TEST`/`BOOST_TEST_EQ`，末尾 `boost::report_errors()` 汇总退出码）。每个用例遵循「拼一段宽字符响应文本 → 构造 `ResponseParser(&commit, &ctx, &status)` → 调 `operator()` 逐行解析 → 断言字段」的五步模板，`test_4` 几乎是一份响应行协议活文档。`test/TestWeaselIPC` 则是命名管道端到端联调程序，无测试框架，靠命令行参数（无参 / `/start` / `/stop` / `/console`）让同一 `.exe` 扮演 Client/Server，用自带的 `TestRequestHandler` 替换 `RimeWithWeaselHandler` 以脱离 librime，验证 `eat` 闭包回写、`GetResponseData` 回调读取的管道往返。两工程双轨构建（`weasel.sln`/MSBuild 与 `xmake.lua`），且仅 Debug 配置参与（xmake 用 `is_mode("debug")` 守卫，MSBuild 靠 `ProjectReference`，`TestResponseParser` 仅引 `WeaselIPC`、`TestWeaselIPC` 引四工程）。核心实践结论：写新用例须区分两种正文——`commit/ctx/config` 可手写，而 `style=`/`ctx.cand=` 是 boost 归档，必须用「构造 UIStyle → text_woarchive 序列化 → Styler 的 text_wiarchive 反序列化」的往返方式测试，不可手写输入文本。
