本讲讲「引擎处理结果如何取回前端」。`rimeUpdate()` 是前端从引擎取结果的唯一出口，采用 commit→status→context 三段式消费，每段遵循「`rimeStructInit`→`get_*`→处理→`free_*`」同构模板。第一段 `rimeConsumeCommittedText` 用 `get_commit` 取定稿文字并 `insertText` 上屏，无定稿则整段跳过；第二段 `get_status` 检测 `schema_id` 切换（用 `schemaId` 缓存去重），触发 `loadSettings(for:)` 重载样式，联合主题配置（`panel.inlinePreedit/inlineCandidate`）与 librime 运行时选项（`no_inline`/`inline`）计算 `inlinePreedit`/`inlineCandidate`，再反向同步 `set_option("soft_cursor",...)`；第三段 `get_context` 信息量最大，将 UTF-8 字节偏移（`sel_start`/`cursor_pos`）转成 `String.Index`，按 inline 策略三路输出 marked text（`show`），并读候选/注释/标签（`select_keys` 优先于 `select_labels`）/页码驱动 `showPanel`。核心结论：librime 结构必须用 `rimeStructInit()` 清零并填 `data_size` 做 ABI 校验，每个成功的 `get_*` 必须配对 `free_*`——结构内部字符串/数组由引擎在 C 堆分配，不释放即泄漏。`rimeUpdate` 是按键、选词、翻页、光标移动、ASCII 切换、和弦超时等场景的统一收口，且为纯消费函数（「Update」指更新前端 UI 而非引擎）。
