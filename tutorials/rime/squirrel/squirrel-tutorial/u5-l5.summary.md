本讲讲的是 Squirrel 编译产出 `Squirrel.app` 后，如何变成普通用户双击即可安装的 `.pkg`、安装时如何自动注册输入源，以及发布后如何通过 Sparkle 自动更新，覆盖打包、签名公证、安装脚本、自动更新四个模块。打包上，`pkgbuild` 把编译产物造成组件包（含组件属性表与 `postinstall` 脚本，安装位置为 `/Library/Input Methods`），`productbuild` 再用分发脚本包装成带向导 UI 的最终 `Squirrel.pkg`，版本号统一取自 `CURRENT_PROJECT_VERSION`。为通过 Gatekeeper，需先用 Developer ID Application 证书 `codesign`（含 `--options runtime` 硬化运行时、`--deep`、`--timestamp`、entitlements），再用 Developer ID Installer 证书 `productsign`，随后 `notarytool submit --wait` 公证、`stapler staple` 装订票据，仅在设 `DEV_ID` 时触发。`postinstall` 以 root 身份在铺完文件后依次执行五步：`killall`（以登录用户身份）→ `--register-input-source`（root）→ `--build` 预编译方案（可被 `RIME_NO_PREBUILD` 跳过）→ `--enable-input-source` 与 `--select-input-source`（以登录用户身份），复用「单二进制双身份」设计，体现「系统级注册表 vs 用户级偏好」两层模型。Sparkle 更新依赖 `Info.plist` 的 `SUFeedURL`（appcast 地址）与 `SUPublicEDKey`（EdDSA 公钥）；发布侧用 `sign_update` 对包算签名生成 appcast 的 `edSignature` 与 `length`，客户端周期拉 appcast、下载后用内置公钥本地校验，把信任锚定在公钥而非传输层。本地自测与正式发布是两条有意分流的路径。
