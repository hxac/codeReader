本讲讲解 librime 的组件注册基础设施，支撑「数据驱动装配」：方案配置里写组件名字，引擎运行时按名造对象，源码不必为每种输入法硬编码。核心抽象是统一根基类 `ComponentBase`（仅供多态存放）与 `Class<T, Arg>` 模板——后者把工厂接口以内嵌 `Component` 子类加纯虚 `Create(Initializer)` 绑进目标类，其中 T 为产品类型、Arg 为构造参数类型。`Component<T>` 是「默认工厂」，继承 `T::Component` 并把 Create 实现为 `new T(arg)`，使多数组件一行 `new Component<MyClass>` 即可注册；需要复杂创建逻辑时则手写 `Component` 子类。`Registry` 是进程级单例，一张「名字→工厂」的 `map<string, ComponentBase*>`，语义为 Register 同名覆盖即 delete 旧值（工厂所有权归它）、Find 未命中返回 NULL。`Class<T, Arg>::Require(name)` 是「按名取货」入口：Find 后经 dynamic_cast 还原类型、返回带类型的工厂（不 Create），调用方再 `Create(arg)` 造出产品，类型安全且可反复实例化。引擎装配流水线用此机制把 YAML 处方串变成对象，缺件仅记 ERROR 跳过不崩溃。本讲只讲注册骨架本身；具体组件基类（Processor/Segmentor/Translator/Filter）留 u5-l2，`RIME_REGISTER_MODULE` 与 ModuleManager 加载机制留 u5-l3，Ticket 拆解与 boost::dll 外部插件加载留 u5-l4。
