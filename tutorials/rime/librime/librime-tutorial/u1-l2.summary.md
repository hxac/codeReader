本讲解决「librime 的 C++ 源码如何被编译成可被前端调用的库」，围绕构建系统展开五个最小模块：Makefile 快捷外壳、顶层 CMakeLists 的配置阶段与选项、第三方依赖查找与链接、库产物形态、deps.mk 自建静态依赖。核心结论：librime 以 CMake 为构建系统，`Makefile` 是一层薄壳，`make` 等价于「cmake 配置 + cmake --build」两步，并按 `uname` 自动推算安装前缀和并行核数。关键构建选项有 `BUILD_SHARED_LIBS`（动/静态）、`BUILD_TEST`（默认 ON，依赖 GTest）、`ENABLE_LOGGING`（依赖 glog，可关）、`BUILD_STATIC`（依赖是否走静态）、`BUILD_MERGED_PLUGINS` 与 `ENABLE_EXTERNAL_PLUGINS`（插件合并进主库或独立编译）。引入的关键术语：配置阶段与构建阶段、静态库/动态库、`find_package` 与 `Xxx_STATIC` 驱动的静态/动态选择、librime 自带的 `cmake/Find*.cmake` 查找模块（LevelDb/Marisa/Opencc/Glog/YamlCpp/Gflags）、`configure_file` 将选项渲染为 `build_config.h` 供源码用 `#ifdef` 消费。库产物分三种形态：默认单动态库 `rime`（带 VERSION/SOVERSION 落于 build/lib）、静态库 `rime-static`、拆分库 `rime-dict`/`rime-gears`/`rime-levers`；`tools`/`test`/`sample` 子目录仅在动态库构建时参与。七个第三方依赖（Boost/Glog/YamlCpp/LevelDb/Marisa/OpenCC/GTest）经分组拼装成最终链接列表。可选 `make deps` 可从源码编译六个第三方库为静态库，用于完全自包含构建。本讲产出的 `librime.so` 与命令行工具是后续讲义（u1-l3 分层架构、u1-l5 体验输入流程）的前置。
