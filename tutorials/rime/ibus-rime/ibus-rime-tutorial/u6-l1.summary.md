本讲是「工程化与架构」单元首篇，解决 ibus-rime「能编译」之上的「如何连同全部依赖打包成可分发、可移植的二进制包」问题，覆盖四块。其一，git 子模块：`librime`（输入法引擎）与 `plum`（旧名 `brise`，产出方案与词库，即运行时 `rime-data`）是独立发版、被多前端共用的项目；子模块只存 commit 指针，是「可选的、版本锁定的源码供应」——普通构建用系统包即可不拉，完整打包时才从源码构建。其二，`cmake/FindRimeData.cmake`：编译期顺序敏感地探测候选目录（自定义 prefix 优先于 `/usr/share`），用 `-DRIME_DATA_DIR` 可整体跳过覆盖，路径烘焙进宏 `IBUS_RIME_SHARED_DATA_DIR` 并决定 `ibus_rime.yaml` 安装目标。其三，`BUILD_STATIC` 与 `RIME_DEPS`：静态链接把 librime 的传递依赖闭包（boost、leveldb、opencc、marisa、yaml-cpp、glog、ICU 等）显式列出，因为没有加载器代为传递符号。其四，`package/` 三脚本：`make-package` 出源码 tarball，`make-binpkg-static` 以 `DESTDIR` 暂存按 `librime → brise → ibus-rime` 顺序编排全静态构建到 `pkg/`，`binpkg-install` 落地系统。核心结论：打包脚本与 `CMakeLists.txt` 的 `install()` 规则是同一套路径的声明式与命令式两种表达，四条 ibus-rime 安装命令逐条对应。
