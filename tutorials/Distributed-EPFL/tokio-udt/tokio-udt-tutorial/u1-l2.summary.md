本讲把 tokio-udt 真正跑起来，覆盖三个最小模块：`bin/udt_sender`、`bin/udt_receiver`、`RateControl` 只读指标。核心结论：`src/bin/` 下的 `.rs` 文件由 Cargo 自动发现为 binary（文件名即程序名，无需 `[[bin]]`），用 `cargo run --bin <名字>` 运行。`udt_receiver` 作服务端，`bind` 到 `0.0.0.0:9000` 后循环 `accept`，为每个连接 `spawn` 独立任务累加字节并每秒打印吞吐；`udt_sender` 作客户端，`connect` 到 `127.0.0.1:9000`，死循环发送 1.2 MB 数据块，每秒打印发送条数与两项速率指标。关键术语：binary 自动发现、bind/accept（服务端被动）与 connect（客户端主动）、吞吐（MB/s）、发送周期（Period，相邻包间隔）、拥塞窗口（Window，未确认时在途包数）。两个指标经 `UdtConnection::rate_control()` 返回的写锁 guard 读取，对应 `RateControl` 的 `pkt_send_period`/`congestion_window_size`，初始值约 1µs 与 16。两者随网络反馈动态调节，体现 UDT 区别于定速发送的特性，其调整机制留待 u7 拥塞控制单元；回环环境下拥塞动作可能不明显。
