本讲是 tokio-udt 系列的收尾技术篇，聚焦高速协议收发包「最后一公里」的 Linux 平台快路径优化，核心是三个「同一签名、两套实现」（`cfg(target_os="linux")` 二选一）的函数：`send_m_msg_to`、`receive_packets`、`sleep_until`。其一是批量系统调用 `sendmmsg`/`recvmmsg`，把逐包系统调用压成一次收发多包，省下大量用户态↔内核态切换；接收缓冲 `mss*100` 决定 `recvmmsg` 单次最多收 100 个包。其二是 `nix` 操作 raw fd：因 tokio 的 `UdpSocket` 不暴露批量调用，故用 `as_raw_fd()` 取底层 fd 直连内核，经 `try_io(Interest)` 与 epoll 协作，把 `EWOULDBLOCK` 转成 `WouldBlock` 实现非阻塞。其三是 timerfd 高精度调度：UDT 的 `pkt_send_period` 为微秒级，`tokio::time::sleep` 的时间轮粒度太粗，故 Linux 改用基于内核 hrtimer 的 `tokio_timerfd`，非 Linux 回退到 `sleep_until`。关键结论：这套快路径是纯工程优化，不改变 UDT 协议语义，只让同样协议在 Linux 上跑得更快；`nix`/`socket2` 为全平台依赖，`tokio-timerfd` 仅 Linux 引入。引入的可承接术语：sendmmsg/recvmmsg、raw fd、syscall、timerfd/hrtimer、双实现桥接模式、EWOULDBLOCK→WouldBlock。
