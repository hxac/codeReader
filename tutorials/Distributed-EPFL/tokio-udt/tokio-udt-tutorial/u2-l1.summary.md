本讲精读 `src/connection.rs`，聚焦客户端主路径，拆为三个最小模块。核心结论：`UdtConnection` 是一个**薄壳**，内部仅持有一个 `SocketRef`（即 `Arc<UdtSocket>`），所有逻辑委托给底层 socket，目的是提供干净的用户 API 并实现 `AsyncRead`/`AsyncWrite`，从而像用 `TcpStream` 一样收发 UDT 数据。模块一 `connect`（及 `bind_and_connect`）的核心是 `_bind_and_connect`：经全局 `Udt` 引擎 `new_socket` 建流式 socket → `lookup_host` 解析多地址 → 逐个尝试 `UdtSocket::connect`（第一个成功即止）→ 循环 `wait_for_connection` 等握手。关键术语：`UdtSocket::connect` 只发出首个握手包（`connection_type = 1`）并把状态置为 `Connecting`，握手往返由后台 worker 异步完成。模块二 `wait_for_connection` 采用「持锁判状态 + `connect_notify.notified().await`」避免忙等，外层 `loop` 保证返回时状态确实离开 `Connecting`；状态机经历 `Init → Opened → Connecting → Connected`（或 `Broken`），`is_alive()` 区分存活与已死。模块三的四个 `poll_xxx`（`poll_write` 委托 `send`、`poll_read` 委托 `poll_recv`、`poll_flush` 查 `snd_buffer_is_empty`、`poll_shutdown` 调 `close`）在遇到 `Pending` 时统一采用「`tokio::spawn` 一个任务等对应 `Notify`，条件满足后 `waker.wake()`」的模式。注意 `flush` 仅保证发送缓冲清空，不等价于对端已确认；握手协议细节留待 u8-l1，`spawn+wake` 取舍留待 u8-l3，服务端 `UdtListener` 在下一讲 u2-l2。
