本讲是第 4 单元首篇，打开「UDT 包」黑盒的入口层。核心是 `src/packet.rs` 中 `pub(crate)` 的 `UdtPacket` 枚举：它只有 `Control(UdtControlPacket)` 与 `Data(UdtDataPacket)` 两个变体，把线上两大类包——承载用户数据的「数据包」与承载握手/ACK/NAK/Shutdown 等信令的「控制包」——捏合为单一类型，使收发侧和测试只需面对一个类型，并把所有按类型分发的逻辑集中到一处；它是 crate 内部实现细节，不暴露于公共 API。

最关键的判别手段是「首比特 dispatch」：仅凭首字节最高位（`raw[0] >> 7`）区分两类包——为 0 走数据包解析、为 1 走控制包解析，这是整个协议解析的总开关。`UdtPacket` 的 `serialize`/`deserialize`/`get_dest_socket_id`/`handshake` 四个方法都是 `match self` 的薄封装，自身不含算法，只做分发与委托。接收 worker 链路依次为：`deserialize`（格式分发）→ `get_dest_socket_id`（取出解复用键，抹平两类包字段位置差异）→ `handshake`（识别握手包）→ `process_packet`（业务分发到 `process_ctrl`/`process_data`）。其中 `dest_socket_id == 0` 是发往 listener 的握手广播地址，需由 `handshake()` 识别并路由给 listener；解析失败的包被 `.ok()?` 静默丢弃以应对不可靠 UDP。包内部字段布局留给 u4-l2、u4-l3、u4-l4。
