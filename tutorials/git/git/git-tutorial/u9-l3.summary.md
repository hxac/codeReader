`git status` 的本质是**两次 diff**：`HEAD ↔ 索引`产出「已暂存」一栏（等价 `git diff --cached`），`索引 ↔ 工作树`产出「未暂存」一栏（等价 `git diff`），未跟踪文件则不属于任何 diff，靠目录遍历单独收集。两条 diff 流都以**路径为键**插入同一个 `string_list s->change`，因此同一文件可同时出现在两栏——其根源是 `struct wt_status_change_data` 同时持有 `index_status` 与 `worktree_status` 两个状态位；合并冲突文件的多个 stage 经 `unmerged_mask()` 折叠成 `stagemask`，打印时映射为 `DD/AA/UU` 等双字母码。

收集骨架 `wt_status_collect()` 是 worktree→index（或 initial 走空树全标 ADDED）→untracked 三段式，每段都用 `DIFF_FORMAT_CALLBACK` 把自己挂成 diff 管线的回调消费者（分别调 `run_diff_files` 与 `run_diff_index(..., DIFF_INDEX_CACHED)`），复用整套 diffcore 配对/重命名能力。打印端 `wt_longstatus_print()` 遍历同一张 `s->change` 表四次按字段过滤；短格式则把两字段压成 `XY` 两列。状态码经 `wt_status_diff_status_string()` 翻译成 `modified:`/`new file:` 等标签。

性能瓶颈在于「索引 vs 工作树」diff 要对每条 `cache_entry` 做 `lstat`；`preload-index.c` 用多线程并行 lstat 预先标记 `ce_uptodate`（经 `ie_match_stat()` 判等），让真正 diff 跳过未变文件，受 `core.preloadindex`、`MAX_PARALLEL=20`、`THREAD_COST=500` 三者节制。入口 `cmd_status` 与 `git commit` 的 `run_status` 都走 `collect→print→free` 三步。
