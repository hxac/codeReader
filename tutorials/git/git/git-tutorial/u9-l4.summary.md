本讲把「索引」与「遍历引擎」拼成两件事：commit 的创建与历史的展示。写入侧，`git commit` 由 `cmd_commit` 编排，确定父列表（初始／amend／merge／普通四种情况），其中 tree 哈希直接取自索引的 **cache-tree**（`index->cache_tree->oid`），无需重新构造目录；再由 `commit_tree_extended` → `write_commit_tree` 逐行拼出 `tree`／`parent`／`author`／`committer`／空行／正文，哈希落盘。核心结论是 **commit 对象只是文本指针**，不存改动，且父顺序也参与哈希。展示侧，`git log` 复用 u7-l1 的 `get_revision()`，每个 commit 交给 `log_tree_commit` → `show_log`。pretty 是两套机制：**结构化格式**（`pp_header`/`pp_remainder`，如 medium/full）与**占位符格式**（`USERFORMAT`，`repo_format_commit_message` 的 `%` 循环经 `format_commit_item`→`format_commit_one` 替换），由 `get_commit_format` 按 `--pretty` 是否含 `%` 分类；`oneline` 走结构化而非占位符。`log-tree.c` 是粘合层，把 graph、header、pretty 正文、diff 串成输出；`--graph` 是 `graph.c` 中独立的状态机，每次 `graph_next_line()` 画一行 ASCII，正文列宽由 `graph_show_commit_msg` 前缀对齐。
