本讲讲清 git 如何把外部仓库嵌入主仓库（superproject/超项目）。核心结论：主仓库不持有子模块的历史对象，靠「两条腿」绑定——tree 里一条 gitlink 条目（模式 S_IFGITLINK=0160000，OID 指向子模块某 commit，不是 blob/tree）与一个版本控制的 .gitmodules 文件（记录 name/path/url）。磁盘上子模块 Git 目录被「吸收」进 .git/modules/<name>，工作区 <path>/.git 是 gitfile，由 connect_work_tree_and_git_dir 双向绑定（并回填 core.worktree）。配置层 submodule-config.c 以 (gitmodules_oid, name/path) 二元组为 key 懒加载缓存，因此支持按任意历史版本查询（submodule_from_path→config_from「查缓存→未命中读 blob 解析→再查」三段式），复用通用配置解析器，并对 name/URL 做路径穿越与注入防御。操作层 builtin/submodule--helper.c 由 cmd_submodule__helper 分发：add 克隆并把 gitlink 与 .gitmodules 写进索引，init 把 .gitmodules 的 url 抄到本地 .git/config（幂等、尊重覆盖），update 克隆并检出到 gitlink 指向的 commit，sync 把新 URL 同步到子模块自己的 remote，absorbgitdirs 把就地 .git 搬进 modules/。所有子进程经 prepare_submodule_repo_env 清掉主仓库环境。fetch/push 递归协商与多工作树 worktree 留待后续讲义。
