本讲厘清合并的三层职责。`unpack-trees` 是与策略无关的「多路树 + 索引同步遍历」引擎，靠换 `merge_fn_t` 回调切换语义：`threeway_merge` 只做文件级（entry 级）裁决，用 `same()` 比较 (mode, oid)，命中平凡情形就走 `merged_entry`/`deleted_entry`，仅当双方都改且改得不同时才用 `keep_entry` 把 base/ours/theirs 以 stage 1/2/3 原样留进索引——这就是冲突的记录方式，它**不做内容合并**。stage 编号由 `unpack_single_entry` 按 `head_idx` 分配（1=base、2=ours、3=theirs）。内容级合并是另一层：`merge-blobs` 把三个 blob 读成 `mmfile_t` 后调 `ll_merge → xdiff` 产出带冲突标记的合并结果，服务于 plumbing `git merge-tree`。`resolve-undo` 则在冲突**被解决**（stage 条目移除）时由 `record_resolve_undo` 记下原 stage 的 mode/oid、序列化为索引 `REUC` 扩展，`unmerge_index` 据此重建 stage 1/2/3，支撑 `checkout --merge`、`update-index --unresolve` 与 `rerere`。
