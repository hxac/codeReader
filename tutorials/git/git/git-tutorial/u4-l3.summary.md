本讲承接 u4-l1（index_state／cache_entry）与 u4-l2（cache-tree、split-index），讲解 sparse-index 如何解决巨型单仓库下索引 O(N) 条目的读写瓶颈。核心思路：把 cache-tree 算好的目录折叠从提交时的临时行为升级为索引磁盘格式的常态，将 cone 外连续属于同一目录的 SKIP_WORKTREE 文件条目用一条记录代表。关键术语「稀疏目录条目」本质是普通 cache_entry 的伪装：mode 为 S_IFDIR（判定宏 S_ISSPARSEDIR）、oid 指向该目录的 tree 对象、路径名带末尾斜杠，并挂 CE_SKIP_WORKTREE。索引稀疏程度由枚举 sparse_index_mode 刻画：INDEX_EXPANDED=0（完整，零初始化兼容历史）、INDEX_COLLAPSED（折叠）、INDEX_PARTIALLY_SPARSE（半展开）。核心路径有二：折叠 convert_to_sparse 借 cache-tree 递归，前提是目录不在 cone 内、无冲突条目、无 gitlink、全部挂 SKIP_WORKTREE，任何条件不满足都静默放弃；展开 expand_index 重建索引并按 tree 拆回文件，传 NULL（ensure_full_index）彻底展开，传 cone pattern list 则只展开进入新 cone 的目录。调度挂在读索引（ensure_correct_sparsity，受 command_requires_full_index 控制）与写索引（convert_to_sparse）两个钩子，默认强制完整索引、仅显式适配命令享受加速。sparse-index 硬性依赖 cone 模式 sparse-checkout 与 index.sparse 配置，经 git sparse-checkout --sparse-index 启停，且与 split-index 互斥；clear_skip_worktree_from_present_files 与 expand_to_path 分别负责与工作树自洽修正和单点查询的懒展开。
