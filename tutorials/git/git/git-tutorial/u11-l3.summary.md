本讲是传输协议单元的收尾，聚焦客户端与服务端如何协商出共同提交集合并据此传输最小 pack。核心结论：传输的对象集等于「want 可达」减去「common 可达」，因此协商越准 pack 越小，但每轮协商都要付出网络往返代价，权衡在于用尽量少的轮次把共同集合探到尽量大。`fetch-pack.c`（fetch 客户端）按协议版本分两路：v0/v1 的 `find_common` 用「批量 have → 读 ACK」循环，v2 的 `do_fetch_pack_v2` 是 CHECK_LOCAL/SEND_REQUEST/PROCESS_ACKS/GET_PACK 四状态机。关键报文 want/have/ACK/NAK/done/ready 在 fetch 与 push 中角色相反。决定「下一批发哪个 have」的策略被抽成虚表 `fetch_negotiator`，默认算法用优先队列按提交时间从新到旧通告，靠 COMMON 标志位与 `non_common_revs` 收敛，批量自 INITIAL_FLUSH=16 指数翻倍、配合 in_vain/MAX_IN_VAIN=256 早停。`send-pack.c`（push 客户端）则是镜像的一侧：发完 ref 更新命令后 spawn `pack-objects --revs --stdout`，用 `^old new` 修订参数编码集合差；可选的 `push.negotiate` 先用 `fetch --negotiate-only` 探出 commons 进一步缩小 pack。
