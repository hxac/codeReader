本讲讲解 git 合并（merge）的源码实现，覆盖三个最小模块：ort 合并策略核心、merge-ll 单文件三方合并驱动、builtin/merge 的策略调度。核心结论是：ort 是当前 git 的默认且唯一核心合并策略，旧的 merge-recursive.c 已被移除，`try_merge_strategy` 把 recursive、subtree、ort 三者全部路由到同一份 `merge_ort_recursive`，因此 `git merge -s recursive` 今天跑的就是 ort。

ort 的设计要点是「递归 + in-core」：全程在内存中完成，期间不碰工作树与索引。`merge_ort_internal` 递归地把多个合并基两两合并成不落盘的虚拟提交（virtual commit，无祖先时用空树兜底），塌缩成单一基后交给 `merge_ort_nonrecursive_internal`，后者是「收集三方树信息 → 检测重命名（含 redo 回环）→ 逐路径决策」三段流水线，最后由 `merge_switch_to_result` 落盘并写 AUTO_MERGE 引用。

关键概念包括：base/ours/theirs 三方合并、3 位掩码 filemask/dirmask/match_mask（两方相同取第三方、三方全同则短路、两两皆不同才进内容合并）、冲突写回索引的 stage 1/2/3 机制（record_conflicted_index_entries）。C 风格继承（conflict_info 首成员为 merged_info，以 clean 字段为安全闸门）是其数据表示基础。

merge-ll.c 中 ll_merge 按 .gitattributes 的 merge 属性经 find_ll_merge_driver 选驱动（内置 text/binary/union，或用户自定义），text 驱动调 xdiff 的 xdl_merge 产出 `<<<<<<<` 冲突标记并返回 LL_MERGE_CONFLICT；virtual_ancestor 标志对应 ort 合并合并基时的 call_depth>0。builtin/merge.c 用 all_strategy[] 策略表与标志位（DEFAULT_TWOHEAD、NO_FAST_FORWARD 等）调度，ort 进程内执行而 octopus 等走子进程，多策略时用 evaluate_result 量化择优，快进走 checkout_fast_forward。
