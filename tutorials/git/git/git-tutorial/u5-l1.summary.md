本讲是「引用 refs」单元首讲，只讲后端无关的公共 API 与抽象，具体落地留待 u5-l2/u5-l3。核心模块有三个：一是 `struct ref_store` 抽象接口，二是引用解析 `refs_resolve_ref_unsafe`，三是引用遍历 `refs_for_each_ref`。

引用（ref）是给对象哈希（oid）起的命名指针，分直接引用（存 oid）与符号引用（symref，存 `ref: <另一引用>`）两种；另有伪引用如 `FETCH_HEAD`/`MERGE_HEAD` 走文件系统特例。`struct ref_store` 是「一个仓库的引用存储」的运行时抽象，基类仅含 `be`、`repo`、`gitdir` 三字段；`be` 指向虚表 `struct ref_storage_be`，公共 API 几乎只做 `return refs->be->操作(...)` 的分派，从而把上层与 files/reftable 后端解耦。后端按 `enum ref_storage_format` 注册在 `refs_backends[]`（仅 files 与 reftable 两个；packed 只是 files 内部辅助，不独立可选），由 `get_main_ref_store` 懒加载进 `repository.refs_private`，承接 u2-l2。

`refs_resolve_ref_unsafe` 是最核心的读函数：最多 `SYMREF_MAXDEPTH=5` 轮递归跟随 symref 链直至拿到 oid，由 `READING`/`NO_RECURSE`/`ALLOW_BAD_NAME` 三 flag 控制；`_unsafe` 指返回串可能指向静态缓冲，需 `refs_resolve_refdup` 复制。遍历有迭代器与回调两套风格，由 `do_for_each_ref_iterator` 桥接；结果按 refname 排序，默认开启 `ref_paranoia`——含坏引用而省略悬空 symref。三者共用一张虚表与一套 flag，是 `ref_store` 抽象的统一性所在。本仓库已取消无前缀全局包装，调用方显式传 `ref_store`/`repository`。
