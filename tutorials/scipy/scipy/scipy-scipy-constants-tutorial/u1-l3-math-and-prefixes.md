- 掌握二进制前缀与 SI 前缀的差别（为什么用 `2**10` 而不是 `1e3`），并能用它们做字节、存储容量换算。

## 2. 前置知识

在进入源码前，先建立三个直觉。

**直觉一：常数 vs 因子。** 在 `scipy.constants` 里，所有名字其实都是「一个浮点数」。它们的差别在于**语义**：

- `pi`、`golden` 是「数学常数」，本身就是一个数值。
- `kilo`、`milli`、`kibi` 是「换算因子」，表示「1 个某前缀单位 = 多少个基本单位」。例如 `kilo = 1e3` 表示 1 千克 = 1000 克，所以 `5 * kilo` 就是 5000 个基本单位。

因为是因子，所以使用方式永远是「数值 × 因子」，结果落在 SI 基本单位上（千克、米、秒……）。

**直觉二：字面量 vs CODATA。** 本讲的全部内容（数学常数 + 两类前缀）都是**Python 源码里直接写死的字面量**，例如 `kilo = 1e3`。它们**不来自** CODATA 物理常数数据库。CODATA 提供的是 `c`（光速）、`h`（普朗克常数）这类需要实验测量的物理常数，那是 u2 进阶篇的内容。本讲里唯一的「外部依赖」是标准库的 `math`，用来取 `pi` 和 `sqrt(5)`。

**直觉三：十进制前缀 vs 二进制前缀。**

- SI 前缀是国际单位制（Système International）规定的十进制前缀，相邻「主」前缀差 1000 倍（`1e3`），用于科学和工程。
- 二进制前缀是 IEC 60027-2 标准规定的，相邻差 1024 倍（`2**10`），用于计算机存储和内存（因为计算机按二进制编址）。所以 1 MiB = 1024 KiB，而不是 1000 KB。

理解这三个直觉后，源码就非常容易读懂了。

## 4. 核心概念与源码讲解

### 4.1 数学常数：pi 与 golden_ratio

#### 4.1.1 概念说明

`scipy.constants` 提供了两个数学常数：

- `pi`：圆周率 \( \pi \)，约 3.141592653589793。
- `golden` / `golden_ratio`：黄金比例 \( \varphi \)，约 1.618033988749895。这两个名字是同一个值，只是别名。

黄金比例的数学定义是：

\[ \varphi = \frac{1 + \sqrt{5}}{2} \approx 1.618033988749895 \]

它满足一个优美的性质 \( \varphi^2 = \varphi + 1 \)。为什么 `scipy.constants` 要提供它？因为它在自然界（向日葵花盘螺旋、鹦鹉螺壳）和工程（黄金分割构图、斐波那契搜索）中频繁出现，放一个现成常量省得每次手算。

这两个常数之所以放在 `_constants.py` 最开头，是因为它们和后面的几何/角度换算有关——例如 `degree = pi / 180`（第 147 行）就用到了 `pi`。

#### 4.1.2 核心流程

这两行的「执行流程」其实非常简单，重点在于**值的来源**：

1. 第 10 行 `import math as _math`：把 Python 标准库 `math` 引入为私有别名 `_math`。
2. 第 69 行：`pi = _math.pi`，直接复用 CPython 标准库里精度最高的 `pi`（双精度浮点，约 15–17 位有效数字）。
3. 第 70 行：`golden = golden_ratio = (1 + _math.sqrt(5)) / 2`，用 `_math.sqrt` 计算 \( \sqrt{5} \)，再代入公式。注意 Python 的连续赋值语法 `a = b = expr` 让 `golden` 和 `golden_ratio` 指向同一个 float 对象。

关键点：这些值是**模块加载时计算一次**的字面量表达式，不涉及任何运行时查找、不调用 CODATA。导入 `scipy.constants` 的瞬间它们就已经确定。

#### 4.1.4 代码实践

**实践目标**：验证 `pi` 与 `golden_ratio` 的值，并确认 `golden` 和 `golden_ratio` 是同一个对象。

**操作步骤**：

```python
from scipy.constants import pi, golden, golden_ratio
import math

# 1. 与标准库 math 对照
print("pi      =", pi)
print("math.pi =", math.pi)
print("pi is math.pi:", pi is math.pi)   # 预期 True

# 2. 黄金比例的优美性质：phi**2 == phi + 1
print("golden            =", golden)
print("golden**2         =", golden**2)
print("golden + 1        =", golden + 1)
print("golden**2 - (golden + 1) =", golden**2 - (golden + 1))   # 预期 0.0

# 3. 别名是同一个对象
print("golden is golden_ratio:", golden is golden_ratio)        # 预期 True
```

**需要观察的现象**：

- `pi is math.pi` 应为 `True`，因为 `pi = _math.pi` 是直接赋值同一个 float 对象。
- `golden**2 - (golden + 1)` 应为 `0.0`（在浮点误差范围内），验证 \( \varphi^2 = \varphi + 1 \)。
- `golden is golden_ratio` 应为 `True`，因为它们是连续赋值的同一对象。

**预期结果**：

```
pi      = 3.141592653589793
math.pi = 3.141592653589793
pi is math.pi: True
golden            = 1.618033988749895
golden**2         = 2.618033988749895
golden + 1        = 2.618033988749895
golden**2 - (golden + 1) = 0.0
golden is golden_ratio: True
```

> 说明：以上为依据源码逻辑推断的预期输出。请在本地实际运行确认（浮点运算在个别平台上末位可能略有差异，但 `golden**2 - (golden+1)` 应严格为 `0.0`，因为这两次运算的舍入路径恰好一致）。

#### 4.1.5 小练习与答案

**练习 1**：为什么 `pi` 用 `_math.pi` 而不是手写 `3.141592653589793`？

**参考答案**：因为 `_math.pi` 是 CPython 标准库提供的高精度常量，避免了手写时抄错或截断的风险；同时 `pi = _math.pi` 直接绑定同一个对象，不产生新的 float 计算开销。

**练习 2**：写出用斐波那契数列逼近 \( \varphi \) 的关系，并解释 `golden` 为什么精确而斐波那契比值只是近似。

**参考答案**：\( F_{n+1}/F_n \to \varphi \)（例如 8/5=1.6，13/8=1.625，…）。`golden` 由闭式公式 \( (1+\sqrt{5})/2 \) 一次算出，精度只受双精度浮点限制；斐波那契比值收敛但永远不等于 \( \varphi \)，只是越来越接近。

### 4.2 SI 前缀：quetta 到 quecto

#### 4.2.1 概念说明

国际单位制（SI）为方便表达很大或很小的量，定义了一套**十进制前缀**。例如：

- `kilo = 1e3`：1 千克 = 1000 克。
- `milli = 1e-3`：1 毫米 = 0.001 米。
- `nano = 1e-9`：1 纳米 = 10⁻⁹ 米（芯片工艺常用）。
- `giga = 1e9`：1 吉赫兹 = 10⁹ 赫兹。

`scipy.constants` 把全部 24 个 SI 前缀都定义为换算因子，从最大的 `quetta = 1e30`（2022 年新增）到最小的 `quecto = 1e-30`。使用方式永远是「数值 × 前缀因子」，结果换算到 SI 基本单位。

#### 4.2.2 核心流程

理解 SI 前缀的关键是把握**两组规律**：命名规律与量级规律。

**规律一：大前缀（≥1）与小前缀（<1）的镜像对称。**

把 24 个前缀按指数排开，正负两侧一一对应：

| 指数 | 「大」前缀（乘） | 指数 | 「小」前缀（除） |
| --- | --- | --- | --- |
| \(10^{1}\) | deka | \(10^{-1}\) | deci |
| \(10^{2}\) | hecto | \(10^{-2}\) | centi |
| \(10^{3}\) | kilo | \(10^{-3}\) | milli |
| \(10^{6}\) | mega | \(10^{-6}\) | micro |
| \(10^{9}\) | giga | \(10^{-9}\) | nano |
| \(10^{12}\) | tera | \(10^{-12}\) | pico |
| \(10^{15}\) | peta | \(10^{-15}\) | femto |
| \(10^{18}\) | exa | \(10^{-18}\) | atto |
| \(10^{21}\) | zetta | \(10^{-21}\) | zepto |
| \(10^{24}\) | yotta | \(10^{-24}\) | yocto |
| \(10^{27}\) | ronna | \(10^{-27}\) | ronto |
| \(10^{30}\) | quetta | \(10^{-30}\) | quecto |

可以看到 \(10^{n}\) 和 \(10^{-n}\) 总是成对出现，例如 `quetta`(1e30) ↔ `quecto`(1e-30)、`ronna`(1e27) ↔ `ronto`(1e-27)。

**规律二：相邻前缀的倍数关系并不均匀。**

- 在 \(10^{0}\) 到 \(10^{3}\) 之间，前缀是**每 10 倍**一档：deka(\(10^{1}\))、hecto(\(10^{2}\))、kilo(\(10^{3}\))，用来精细描述 1–1000 这个「日常范围」。
- 从 kilo(\(10^{3}\)) 往外，前缀是**每 1000 倍**一档：kilo、mega、giga、tera… 这些是工程上最常用的「主」前缀，指数都是 3 的倍数。

换句话说，源码里相邻两行的比值要么是 10（在 deka/hecto/kilo 段和 deci/centi/milli 段），要么是 1000（其余段）。

**规律三：2022 年新增的两个最大前缀。**

`ronna`(\(10^{27}\), 符号 R) 和 `quetta`(\(10^{30}\), 符号 Q)，以及对应的最小前缀 `ronto`(\(10^{-27}\), r) 和 `quecto`(\(10^{-30}\), q)，是第 27 届国际计量大会（2022）新加入的，用于描述大数据存储（PB 级以上）和极小尺度（如宇宙学、粒子物理）。源码里它们排在最前和最后，体现了「从大到小」的排列顺序。

#### 4.2.3 源码精读

**第 72–96 行：SI 前缀全表**

[_constants.py:72-96](https://github.com/scipy/scipy/blob/5f09bd719ca35a5c4de9644a097d379e5b3b4165/scipy/constants/_constants.py#L72-L96) — 从 `quetta = 1e30` 到 `quecto = 1e-30`，按「从大到小」顺序定义 24 个 SI 前缀，每个都是纯十进制字面量。注意第 95–96 行的 `ronto = 1e-27` 与 `quecto = 1e-30`，以及第 73–74 行的 `quetta = 1e30` 与 `ronna = 1e27`，正是 2022 年新增的四个前缀。

这 25 行（含注释）就是全部 SI 前缀的实现——没有循环、没有字典、没有查表，就是一串赋值语句。这种「平铺直叙」的写法是有意为之：常量定义追求**可读、可 diff、零运行时开销**，不需要任何抽象。

#### 4.2.5 小练习与答案

**练习 1**：把「7.2 GHz」换算成 Hz，用 `scipy.constants` 的前缀因子写出表达式。

**参考答案**：`7.2 * giga`，结果是 `7.2e9`（7200000000.0）Hz。因为 `giga = 1e9`。

**练习 2**：`hecto` 和 `deka` 为什么单独存在，而不是只保留 kilo/mega 这种 1000 倍的前缀？

**参考答案**：因为 1–1000 这个「小范围」需要更细的刻度（10 倍一档），例如百米跑（hecto）、十进制工程图纸（deka）。如果只有 kilo，那么 100 就只能写成 `0.1 * kilo`，不直观。所以 SI 保留了 deka/hecto/deci/centi 这四个「过渡」前缀。

**练习 3**：`scipy.constants` 里为什么没有 `deca`（deca 是 deka 的另一种拼写）？

**参考答案**：源码只定义了 `deka = 1e1`（第 84 行），没有提供 `deca` 别名。这是 SciPy 的命名约定，与国际单位制的两种官方拼写（美式 deka / 国际 deca）无关，使用时只能用 `deka`。

### 4.3 二进制前缀：kibi 到 yobi

#### 4.3.1 概念说明

计算机里经常出现「1 KB 到底是 1000 字节还是 1024 字节」的混淆。为了消除歧义，IEC 60027-2 标准定义了**二进制前缀**：

- `kibi = 2**10 = 1024`（Ki，注意大写 K）
- `mebi = 2**20 = 1048576`（Mi）
- `gibi = 2**30`（Gi）
- … 一直到 `yobi = 2**80`（Yi）

命名规律很清晰：把对应的 SI 前缀末尾加上 **"bi"**（binary 的缩写）：

| SI 前缀 | 二进制前缀 | 值 |
| --- | --- | --- |
| kilo | kibi | \(2^{10}\) |
| mega | mebi | \(2^{20}\) |
| giga | gibi | \(2^{30}\) |
| tera | tebi | \(2^{40}\) |
| peta | pebi | \(2^{50}\) |
| exa | exbi | \(2^{60}\) |
| zetta | zebi | \(2^{70}\) |
| yotta | yobi | \(2^{80}\) |

所以 1 MiB（mebibyte）= \(2^{20}\) 字节 = 1048576 字节，而 1 MB（megabyte）按 SI 是 \(10^{6}\) = 1000000 字节——两者差约 4.86%。

注意二进制前缀的两个特点：

1. **只定义「乘」的方向**（\(2^{10}\) 到 \(2^{80}\)），没有对应的「除」方向（没有 millibi 之类），因为计算机存储总是以字节为最小单位往大聚合。
2. **目前只到 yobi（\(2^{80}\)）**，没有跟进 SI 在 2022 年新增的 ronna/quetta 对应的 robi/quebi——因为现实中还没有那么大的存储需求。

#### 4.3.2 核心流程

二进制前缀的「流程」就是一条数学公式，每个前缀的值是 2 的幂：

\[ 1\,\text{kibi} = 2^{10} = 1024,\quad 1\,\text{mebi} = 2^{20} = 1048576,\quad \ldots,\quad 1\,\text{yobi} = 2^{80} \]

相邻二进制前缀的比值恒为：

\[ \frac{2^{10(n+1)}}{2^{10n}} = 2^{10} = 1024 \]

也就是说**每跨一档就是 1024 倍**，这是二进制前缀与 SI 前缀最大的区别——SI 主前缀是 1000 倍，二进制前缀是 1024 倍。

由于源码用 `2**10`、`2**20` 这样的整数幂表达式，结果是精确的整数（Python 的 `**` 对整数运算返回精确整数），不存在浮点误差。但要注意：在 `_constants.py` 里这些值会和 SI 前缀一起参与后续的浮点运算（比如 `Btu_th` 的计算），那时会被自动转成 float。

#### 4.3.3 源码精读

**第 98–106 行：二进制前缀全表**

[_constants.py:98-106](https://github.com/scipy/scipy/blob/5f09bd719ca35a5c4de9644a097d379e5b3b4165/scipy/constants/_constants.py#L98-L106) — 从 `kibi = 2**10` 到 `yobi = 2**80`，定义 8 个二进制前缀。注意这里用的是 `2**10`（整数幂），而不是 `1024` 或 `1.024e3`——既保证了精确整数，又让「这是 2 的幂」这层语义一目了然，可读性远胜魔法数字 `1024`。

#### 4.3.4 代码实践

**实践目标**：用二进制前缀换算 1 MiB 等于多少字节，并与 `2**20` 对照；再体会 SI 与二进制前缀的差距。

**操作步骤**：

```python

# 1. 1 mebi 换算成「基本单位」（字节）
print("1*mebi        =", 1 * mebi, "字节")     # 预期 1048576
print("2**20         =", 2**20)                # 预期 1048576，应与上一行相等

# 2. 验证相邻二进制前缀恒为 1024 倍

print("gibi / mebi   =", gibi / mebi)          # 预期 1024.0

# 3. SI 与二进制前缀的差距（著名的 1 MB vs 1 MiB 问题）
print("mebi - mega   =", mebi - mega, "字节")  # 预期 48576.0
print("mebi / mega   =", mebi / mega)          # 预期 1.048576
```

**需要观察的现象**：

- `1*mebi` 应严格等于 `2**20` = `1048576`，验证源码字面量。
- `mebi / kibi` 和 `gibi / mebi` 都是 `1024.0`，验证 1024 倍规律。
- `mebi - mega` = `48576.0`，即 1 MiB 比 1 MB 多出 48576 字节（约 4.86%），这就是为什么硬盘厂商按 SI（1000）标容量、操作系统按二进制（1024）显示，导致「500 GB 硬盘在系统里只有约 465 GB」的现象。

**预期结果**：

```
1*mebi        = 1048576 字节
2**20         = 1048576
mebi / kibi   = 1024.0
gibi / mebi   = 1024.0
mebi - mega   = 48576.0 字节
mebi / mega   = 1.048576
```

> 说明：以上为依据源码字面量推断的输出。请在本地运行确认。

#### 4.3.5 小练习与答案

**练习 1**：把「16 GiB」换算成字节，用二进制前缀因子写出表达式，并给出数值。

**参考答案**：`16 * gibi`。因为 `gibi = 2**30 = 1073741824`，所以结果是 `17179869184` 字节。

**练习 2**：为什么源码用 `2**10` 而不是写 `1024`？

**参考答案**：两个理由——一是精确性，`2**10` 在 Python 中是精确整数运算，不会引入浮点误差；二是可读性，`2**10` 直接表达了「这是 2 的 10 次方」的二进制本质，而 `1024` 是一个需要读者回忆含义的魔法数字。

**练习 3**：二进制前缀为什么没有「小」方向（类似 milli 的除法前缀）？

**参考答案**：因为二进制前缀用于描述存储/内存容量，最小单位是字节，永远只会往大聚合（KiB→MiB→GiB），不存在「1/1024 字节」这种需求，所以 IEC 标准和 `_constants.py` 都只定义乘方向的 8 个前缀。

## 5. 综合实践

把本讲的三块内容（数学常数、SI 前缀、二进制前缀）串起来，完成一个小任务：**写一个「单位助手」脚本，用前缀因子做一组换算，并验证它们与字面量计算一致。**

任务要求：

1. 用 SI 前缀表达「人体神经元直径约 20 微米」换算成米。
2. 用 SI 前缀表达「SSD 容量标称 2 TB（SI）」换算成字节，再用二进制前缀表达「实际格式化后约 1863 GiB」换算成字节，比较两者差距。
3. 用 `pi` 计算一个直径为 `2 * micro` 米的圆的面积，结果用平方纳米（`nano**2`）表示。

参考实现：

```python
from scipy.constants import micro, tera, gibi, nano, pi

# 任务 1：20 微米 -> 米
neuron_m = 20 * micro
print("神经元直径:", neuron_m, "m =", neuron_m * 1e9, "nm")

# 任务 2：2 TB(SI) vs 1863 GiB(二进制)
ssd_si = 2 * tera            # 厂商标称，按 1000 进制
ssd_real = 1863 * gibi       # 系统显示，按 1024 进制
print("2 TB(SI)   =", ssd_si, "字节")
print("1863 GiB   =", ssd_real, "字节")
print("差距       =", ssd_si - ssd_real, "字节")

# 任务 3：直径 2 微米的圆面积，用平方纳米表示
r = (2 * micro) / 2                 # 半径，单位米
area_m2 = pi * r**2                 # 面积，平方米
area_nm2 = area_m2 / (nano**2)      # 换算到平方纳米
print("圆面积:", area_m2, "m^2 =", area_nm2, "nm^2")
```

**验证要点**：

- 任务 1：`20 * micro` 应为 `2e-05` 米。
- 任务 2：`2 * tera` = `2e12`，`1863 * gibi` ≈ `2.0e12`，两者接近但不等，差距反映了 SI 与二进制前缀的系统性差异。
- 任务 3：半径 1 微米的圆，面积 \( \pi r^{2} = \pi \times 10^{-12}\,\text{m}^2 \)，换算到平方纳米（\(1\,\text{nm}^2 = 10^{-18}\,\text{m}^2\)）应为 \( \pi \times 10^{6} \approx 3141592.65 \) nm²。

> 说明：以上数值为依据源码字面量与公式推断，请在本地运行确认。

## 6. 本讲小结

- 本讲全部内容都是**纯 Python 字面量计算**，唯一外部依赖是标准库 `math`（第 10 行 `import math as _math`），不涉及 CODATA 数据库。
- 数学常数只有两个：`pi = _math.pi` 和 `golden = golden_ratio = (1 + _math.sqrt(5)) / 2`（第 69–70 行）。
- SI 前缀共 24 个，从 `quetta = 1e30` 到 `quecto = 1e-30`（第 72–96 行），呈「大/小」镜像对称；主前缀（kilo、mega…）每档 1000 倍，过渡前缀（deka、hecto、deci、centi）每档 10 倍。
- 二进制前缀共 8 个，从 `kibi = 2**10` 到 `yobi = 2**80`（第 98–106 行），每档 1024 倍，只定义乘方向，用于计算机存储。
- 使用方式统一为「数值 × 因子」，因子把任意前缀单位换算到 SI 基本单位。
- 源码写法的细节有信息量：SI 用 `1e3`（浮点字面量），二进制用 `2**10`（整数幂），既保证精度又传达语义。
