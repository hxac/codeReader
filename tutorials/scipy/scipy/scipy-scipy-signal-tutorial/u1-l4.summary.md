本讲义解析 `scipy.signal` 公共命名空间的「编织」过程，核心结论是：扁平 API `scipy.signal.*` 由一条**四层接力流水线**生成，并附带一条向后兼容的弃用侧链。

**主链四层**：第 0 层为 `_filter_design`、`_signaltools` 等私有实现模块（真正算法，如 `butter` 定义于 `_filter_design`）；第 1 层 `_signal_api` 是「**裸 API 聚合层**」，用 `from ._xxx import *` 收集各模块公共函数，并以 `__all__ = [s for s in dir() if not s.startswith('_')]` **自动生成** API 列表，新增函数无需手动注册；第 2 层 `_support_alternative_backends` 对每个函数做两道处理——仅在 `SCIPY_ARRAY_API` 开启且有「**签名委托器**」（`_delegators` 中按 `<名>_signature` 命名）时才用 `delegate_xp` 包一层 CuPy/JAX 转发 wrapper，恒定生效的 `xp_capabilities` 只贴后端能力元数据、改写文档、返回原对象不改行为；第 3 层 `__init__` 用 `import *` 暴露并 `del` 掉脚手架模块。

**侧链**：`filter_design` 等 10 个「**弃用 stub（壳模块）**」无实现，靠模块级 `__getattr__` 调 `_sub_module_deprecation` 发 `DeprecationWarning` 并从私有模块取回函数，计划 v2.0.0 移除。

关键术语：四层接力、裸 API、`dir()` 自动 `__all__`、签名委托器、`xp_capabilities` 能力标注、`CUPY_BLACKLIST`/`capabilities_overrides`、子模块导入副作用（解释 `del` 生效原因）。默认下 `scipy.signal.butter is _filter_design.butter` 为真，但官方推荐扁平路径。
