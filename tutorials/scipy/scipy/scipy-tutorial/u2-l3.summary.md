本讲聚焦 SciPy 对 Python Array API 标准的支持机制，核心原则是「数组类型进、数组类型出」——传 PyTorch 张量进去就用 PyTorch 算、返回的仍是 PyTorch 张量，避免偷偷回退 NumPy 造成 GPU→CPU 的性能陷阱。整套机制由环境变量 `SCIPY_ARRAY_API=1`（必须在导入 SciPy 前设置）开启；不开时 `array_namespace` 直接返回兼容版 NumPy 命名空间 `np_compat` 并跳过全部校验，保证零行为变化、零性能损耗；开启时附带更严格的输入校验（拒绝 `np.matrix`、`MaskedArray`、`object` dtype）。`array_namespace` 是分发心脏，它用带 `lru_cache` 的 `_validate_array_cls` 把每个输入归类为 `_ArrayClsInfo` 枚举标签（skip/numpy/array_like/unknown），分桶后推断公共命名空间 `xp`，跨后端统一推断则委托上游 `array_api_compat.array_namespace`。`_asarray` 是 SciPy 版 `np.asarray`，补回标准缺失的 `order`/`check_finite`/`subok`（对非 NumPy 后端静默忽略 order）；整个 `xp_*` 辅助家族（如 `xp_vector_norm`、`xp_promote`）共同起点都是 `xp = array_namespace(x)`。关键术语还包括：标准命名空间 `xp`、`device=` 跨设备参数、`@xp_capabilities` 能力表装饰器、`scipy_namespace_for` 跨库委托（CuPy→cupyx.scipy）、conftest 的 `xp` fixture 参数化测试。本讲的 Array API 分发（代码看数组类型）与上一讲 uarray 分发（用户显式 set_backend）互补，分界线在于：走 `array_namespace` 的普通函数（如 `fftfreq`、`trim_mean`）无法被 uarray 拦截。
