本讲围绕 SciPy `_lib` 模块下的 `LowLevelCallable` 与 C 回调机制展开，核心是解决 C/Fortran 底层例程（如 `quad`、`generic_filter`）在求值循环里反复调用用户函数时的 **Python 回调开销**：普通 Python 回调每次需拿回 GIL、创建 `PyObject`、打包拆包，开销在微秒级；若把一个编译好的 C 函数指针直接交给底层，则可绕开 Python 与 GIL，仅作一次纯 C 跳转。

关键术语与结论：`LowLevelCallable` 是继承自 `tuple` 的不可变类，把 ctypes / cffi / PyCapsule / Cython 四种异构来源归一为一个内部 PyCapsule（装指针、name 装签名、context 装用户数据），原始对象被存进 tuple 以**保活**底层指针。**签名**是类 C 原型字符串（如 `"double (double, void *)"`），既是协议契约也是路由 key，C 侧用逐字符 `strcmp` 与例程的签名表匹配，不匹配则抛带期望清单的 `ValueError`。C 侧运行时由 `ccallback.h` 的 **prepare / release / obtain 三件套 + `ccallback_t` 结构 + 线程局部存储（TLS）** 构成，thunk 按 `c_function` 是否为空决定「直接 C 调用」还是「拿 GIL 调 Python」。`quad`（`__quadpack.h`）是其最真实消费者，用 `CCALLBACK_OBTAIN` 配合 `ccallback_obtain` 在 Fortran 循环里从 TLS 取回回调。该类被提升到顶层 `scipy` 命名空间，其导入还兼任编译扩展模块的金丝雀检查。
