本讲精读 `scipy/special/__init__.py`，揭示 `scipy.special` 这个统一命名空间是如何由十来个分散子模块拼装而成的。该文件具有「双重身份」：前 783 行是一整段文档字符串，既是按类别（Airy、Bessel、椭圆、Gamma 等）排列的函数目录，也是 Sphinx 渲染官方文档的源材料，其开篇契约明确「几乎所有函数都是 ufunc、遵循广播，例外按小节单独警告」；后半部分则是一串导入语句，负责命名空间总装。关键区分是 `from . import X` 仅绑定子模块对象，而 `from .X import *` 把函数提到顶层货架。`_ufuncs`、`_basic`、`_orthogonal`、`_multiufuncs` 等成对导入；`_support_alternative_backends` 排在主体之后，用以覆盖同名函数、注入 Array API 多后端支持。公开 API 清单 `__all__` 采用「四路聚合 + 14 项手动补丁」策略：四个大模块的 `__all__` 相加，再手动追加来自小专项模块（`_logsumexp`、`_ellip_harm`、`_lambertw`、`_spherical_bessel`）及错误类的名字，其中 `multigammaln` 是唯一必须手动补的特例。文件末尾的旧命名空间（`basic`、`orthogonal`、`specfun`、`sf_error` 等）是历史遗留别名，`dir()` 可见但不在 `__all__`，将在 v2.0.0 移除。本讲确立了一条贯穿后续的核心认识：「能在命名空间调用」与「属于官方公开 API」是两条独立链路，新增函数时二者都要兼顾。
