本讲聚焦 `scipy.fft` 四层架构中的「uarray 分派层」，回答为何 `scipy.fft.fft` 的函数体仅一行 `return (Dispatchable(x, np.ndarray),)` 却能算出真正的 FFT。核心结论是「函数即协议」：函数体并非计算代码，而是「参数提取器」，声明哪些参数可被后端替换。

引入的关键术语：uarray 三要素——domain（路由键，`scipy.fft` 共用 `"numpy.scipy.fft"`）、multimethod（由 C++ 类型 `_Function` 实现，`repr` 为 `<uarray multimethod 'fft'>`）、Dispatchable（给参数贴「可分派 + 目标类型」标签）。`generate_multimethod` 接收提取器、回填器、domain，构造多方法并用 `update_wrapper` 嫁接元信息。

四个最小模块：`_dispatch` 是 `_basic.py` 的一行装饰器，固定 domain 并共享回填器，被 `_realtransforms.py`/`_fftlog.py` 复用，28 个变换共用同一套分派逻辑；`_x_replacer` 是参数回填器，约定「可分派数组恒为首位置参数或关键字 `x`」，在需转换后端时把 `__ua_convert__` 结果塞回 `(args, kwargs)`。关键细节：默认 `_ScipyBackend` 无 `__ua_convert__`，普通 numpy 调用根本不执行提取器；`_backend.py` import 时 `set_global_backend('scipy', try_last=True)` 注册默认后端，并靠 `method.__name__` 查找实现。后端管理的优先级语义留待 u4-l2。
