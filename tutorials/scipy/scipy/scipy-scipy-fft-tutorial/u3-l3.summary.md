本讲剖析实变换后端 `_realtransforms_backend.py` 的核心 `_execute`：一个 8 行的高阶函数适配器，把「用哪个 ducc 内核」当参数传入，统一完成「记录数组出身命名空间 → 转 numpy → 调内核 → 转回原命名空间」四步，被 `dct/idct/dst/idst` 及其 n 变体共 8 个变换复用。关键术语包括**数组命名空间（array_namespace）**——识别输入数组的「出身」桥接器，以及**默认模式 vs `SCIPY_ARRAY_API` 模式**：默认模式下 `array_namespace` 恒返回 numpy，桥接「隐形」，仅转 numpy + 调 ducc 干活；开启后才真正识别异构库。核心结论是与 `_basic_backend._execute_1D` 的对比：两者同处四层架构的后端层、最终都落 ducc，但 FFT 族有可移植的 `xp.fft` 原生实现（可上 GPU，路由三分支繁复），实变换族**数组标准中没有 `xp.dct/dst`**，只能全程经 CPU numpy，故后端无 `is_numpy`/`xp.fft` 分支、最简结构，并被标注 `cpu_only=True`。1-D 变换通过把 `(n, axis)` 对齐到 `(s, axes)` 槽位，使单一 `_execute` 兼顾 1-D 与 N-D。
