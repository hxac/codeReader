本讲聚焦 LightLLM 多进程架构中的 HttpServer 进程，覆盖 HTTP 路由、请求分发、结果回流三个最小模块。核心结论：HTTP 入口基于 FastAPI，端点函数很「薄」，仅做 PD 模式拦截、异常翻译与委托，真正逻辑在 `lightllm_generate`/`chat_completions_impl` 等实现函数中；`api_http.py`（路由层，持有全局 `g_objs`）与 `httpserver/manager.py`（干活层，持有全部 zmq 套接字）分工明确。请求分发由 `HttpServerManager.generate` 完成，依次经历 encode（分词）→ 长度校验（超限时下调 `max_new_tokens` 而非报错）→ 共享内存分配 Req → `transfer_to_next_module` 选路。选路按优先级 visual → audio → multi_level_kv_cache → router，纯文本模型因 `disable_vision=True` 直达 router。关键术语：`g_objs`、`SamplingParams`、`to_group_req_index()`、`index_in_shm_mem`（「对象放共享内存、线上只传索引」）、`GroupReqObjs`、`FinishStatus`、`ReqStatus`、`out_tokens_queue`。结果回流采用「事件+暂存区」中转模式：常驻协程 `handle_loop` 作生产者写 `out_token_info_list` 并 `event.set()`，`_wait_to_token_package` 作消费者被唤醒后逐 token `yield`，最终由流式（SSE）/非流式函数组装成 HTTP 响应，实现闭环。
