本讲讲解 LightLLM 的三级 KV Cache 机制：GPU（L1）→ CPU 内存（L2）→ 磁盘（L3），三级各自 LRU、按最长公共前缀串联查询，用更便宜的存储换取更大缓存容量。一个独立进程 multi_level_kv_cache 插在 HttpServer 与 Router 之间专职做 CPU/磁盘前缀匹配，以「接收线程 + 队列 + 大线程池（磁盘模式 500 workers）+ 0.5s 超时兜底」把慢磁盘 IO 隔离在调度循环外。CPU 缓存按「页」（每页 cpu_cache_token_page_size 个 token）组织，用滚动 xxh3_128 哈希做身份证，靠 _CpuPageStatus 三态机（EMPTY/LOADING/READY）+ 引用计数 + LRU 管理页槽、仅返回页索引而非数据；磁盘层由 DiskCacheWorker 借助 LightMem 落盘/回填。两条对称数据流：offload（GPU→CPU→磁盘）于请求结束时经独立 cuda stream 触发，load 在 prefill 前把命中页拷回 GPU，设有 need_token_num≥128 的拷贝/重算收益阈值；命中长度按 GPU/CPU/磁盘三段分别记账，最终在 HttpServer 汇总。
