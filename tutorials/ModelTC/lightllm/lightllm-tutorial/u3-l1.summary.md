本讲打开 `ModeBackend.init_model` 调用 `get_model(...)` 后的「黑盒」，仅覆盖三个最小模块：**模型基类**、**组件初始化**、**推理状态**。`TpPartBaseModel` 是所有模型族的基类，采用「插槽+组装」的依赖注入设计：基类声明六个默认为 None 的类属性插槽（权重类×2、推理类×3、推理状态类×1），子类只需填入零件类、几乎不写 `__init__`，名字中的 `TpPart` 表示每张 GPU 只持有「张量并行的一片」模型。`__init__` 是一条写死的、有严格顺序依赖的近 20 步初始化流水线：先读 config.json（`repair_config` 统一字段名）→ 建权重空壳 → 建请求/内存管理器 → 用 `mem_manager.size` 回填真实 `max_total_token_num` → 建推理层 → 子类钩子 `_init_custom`（如 llama 初始化 RoPE）→ 加载真实权重 → 建注意力后端 → CUDA Graph 与预热 → 试跑测 OOM → 标记就绪。每次前向则新建一次性的 `InferStateInfo` 上下文：它搬运外部 `ModelInput` 字段、挂接模型级常驻的 `mem_manager`/`req_manager`，再由 `init_some_extra_state`/`init_att_state` 现算派生张量与注意力状态。核心结论是：一个模型 = 子类填好的六个插槽 + 基类一条写死流水线；一次推理 = 一个新建 `InferStateInfo` 穿过推理层。关键术语：`TpPartBaseModel`、插槽、`InferStateInfo`、`ModelInput`、`_init_custom`、`max_total_token_num` 回填，为后续 prefill/decode 主流程讲义奠基。
