本讲打开 Router 主循环里 `generate_new_batch` 的黑盒。LightLLM 把「挑请求成批」抽象成 `req_queue`，主循环只调 `generate_new_batch`（挑批）与 `update_token_load`（发布负载），使调度策略可替换、调用方稳定。`build_req_queue` 按启动参数经优先级链选队列类，绝大多数模式落到 `ChunkedPrefillQueue`——`--disable_chunked_prefill` 也复用它，靠把 `chunked_prefill_size` 设为整条 prompt 长度退化成一次性 prefill；多 DP 组再套一层 `DpQueue`。

chunked prefill 让长 prompt 分块预填充，与 decode 共享每拍 `batch_max_tokens` 预算，避免长 prompt 阻塞 decode；预算初值在 batch 为空时为 0、非空时为老请求的 decode 需求，正是「prefill 与 decode 共享预算」的体现。单请求能否加入由三道闸门把关：KV 峰值 `ok_token_num`、请求数 `ok_req_num`、单拍 prefill 预算 `ok_prefill`，任一不过即停止。同一套峰值公式 peak(i)=b[i]·i+Σa[j] 在准入（含等待区新请求）与对外发布（仅现有 batch）两处复用。`is_busy`（由 `router_token_ratio` 触发的激进/保守状态）渗透进输出长度估计；负载经 `update_token_load` 写入共享内存 `shared_token_load`（含 `current_load`、`estimated_peak_token_count`、`dynamic_max_load`），供 HttpServer 准入读取，形成跨进程背压。
