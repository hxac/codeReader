本讲讲解 LightLLM 三套彼此正交的「结构化输出」机制。**约束解码**（`output_constraint_mode`，取值 `outlines/xgrammar/none`，以 `api_cli.py` 为准）住在 ModelBackend，在 GPU 采样**之前**用 FSM 或文法匹配器把不合法 token 的 logit 压成 `-1000000.0`，从而强制输出满足正则 / JSON Schema / 文法；由 `XgrammarBackend`/`OutlinesConstraintBackend` 替换 prefill/decode/post 三个钩子实现，禁用 overlap。**推理解析**（`reasoning_parser`）与**函数调用解析**（`tool_call_parser`）都住在 HttpServer/OpenAI 适配层（CPU，反 token 化之后），不改采样、纯后处理。二者均采用「外观 + 探测器」结构：前者按思考标记（如 `<think>...</think>`）把文本切成 reasoning/normal 两路，靠 `force_reasoning` 与「hold 住标记前缀」应对无开标记与流式分片；后者把 9 种模型族格式（Qwen/Llama3/Mistral/DeepSeek/GLM 等）归一为统一的 `ToolCallItem`，流式遵循「先发名字、再流参数」，借助 `partial_json_parser` 解析半截 JSON。核心直觉：约束解码是「让模型说对话」，parser 是「把说的话翻译成结构」；组装响应时必须**先 reasoning、后 function_call**，`TOOLS_TAG_LIST` 提供廉价预筛。关键术语：logits 掩码、FSM/文法匹配器、思考标记、工具调用标记、`ToolCallItem`。
