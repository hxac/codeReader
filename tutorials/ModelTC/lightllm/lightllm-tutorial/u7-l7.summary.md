本讲是「分布式部署与扩展特性」单元收尾篇，主题是服务可观测性，拆为三个最小模块：指标采集、指标查询、健康检查。

指标采集采用基于 rpyc 的「中央账本」模型：独立 metric 进程持有唯一的 `CollectorRegistry`（由 `Monitor` 用 `prometheus_client` 登记 Counter/Gauge/Histogram），其余进程（Router、HttpServer）各持一个 `MetricClient` 只写不改结构。`MetricClient` 是后台守护线程加 `queue.Queue(4096)`，写方法只把闭包塞队列立即返回（fire-and-forget），后台线程串行执行 rpyc 调用，队列满即丢弃并告警，绝不阻塞推理调度循环。关键 gauge 由 Router 每拍上报，如 `lightllm_batch_current_size`、`lightllm_queue_size`、`lightllm_num_running_reqs`、`lightllm_batch_pause_size`、`lightllm_batch_current_max_tokens`；请求级 counter/histogram 由 HttpServer 在请求完成时上报；所有指标带固定 label `model_name`。

指标查询通过唯一的读路径 `/metrics` 端点，用 `rpyc.async_` 桥入 async 端点输出标准 Prometheus 文本；配 `--metric_gateway` 时另起线程周期 push 到 pushgateway。

健康检查分两层：被动 `/health` 端点读两个 `SharedInt` 标记（最近成功推理时间、在跑请求数），按「宽限期内出过 token」或「空闲」三分支判定，秒级被动响应；主动 `health_monitor` 进程（需 `--health_monitor` 开启，默认 88s 探一次）周期 GET `/health` 并检查子进程存活，连续 3 次失败即 `SIGKILL` 整棵进程树自愈。健康标记由 HttpServer 维护，构造期暖启动为当前时间，每出一个 token 即刷新，避免冷启动被误判。
