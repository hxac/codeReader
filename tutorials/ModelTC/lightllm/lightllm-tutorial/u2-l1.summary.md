本讲建立贯穿第二单元的「请求流转地图」。LightLLM 以多进程协作为核心，把推理拆成职责严格隔离的若干进程：HttpServer 收请求并分词、Router 负责调度（决定 prefill/decode）、ModelBackend（每 GPU 一个）做 GPU 推理、Detokenization 把 token 解码成文本、Metric 汇聚指标。三类 IPC 各司其职：zmq（PUSH/PULL 投递、PUB/SUB 广播）传轻量通知，rpyc 传需应答的远程调用，共享内存承载 Req 等大块状态、做到零拷贝。一次纯文本请求的数据流为闭环：HttpServer 以 zmq PUSH 发请求索引给 Router，Router 经 rpyc 调 ModelBackend 算 token、再 PUSH 索引给 Detokenization，后者解码后 PUB 唤醒 HttpServer，HttpServer 从共享内存取文本流式返回。核心设计是「对象放共享内存、线上只传索引」，这也是强依赖 --shm-size 的根因。
