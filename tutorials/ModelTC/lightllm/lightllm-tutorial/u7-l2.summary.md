本讲讲清 LightLLM 多模态请求的流转路径。多模态服务在「HttpServer→Router→LLM」主链路前插入一条独立进程链：HttpServer 用 `transfer_to_next_module` 按优先级挑第一棒（visual→audio→multi_level_kv_cache→router），每棒处理完自行 zmq PUSH 转交下一棒。`visualserver`/`audioserver` 是结构对称的独立进程，各持 ViT/音频编码器权重，配独立的 `visual_dp`/`visual_tp`，只对 `embed_ready=False` 的图/音频跑编码器，算完用 Triton kernel 把嵌入 DMA 写入 CPU pinned 共享内存。`embed_cache` 是地基性的独立 rpyc 进程，以 `MD5(字节)+hash(extra_params)` 为键去重，用引用计数+LRU 管理生命周期；每个 `Record` 有 `data`（字节就绪，HttpServer 写）与 `embed`（嵌入就绪，visual/audio 写）两个独立就绪位，分别决定是否跳过写字节与跑 ViT。线上 zmq 只传 `GroupReqIndexes`（索引+`MultimodalParams`），图片字节与嵌入全走共享内存。核心结论：多模态服务 = 主链路前的 visual/audio 进程链 + MD5 去重的中央缓存，对象放共享内存、线上只传索引。
