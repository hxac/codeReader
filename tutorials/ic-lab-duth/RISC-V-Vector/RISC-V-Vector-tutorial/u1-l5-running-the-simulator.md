# u1-l5 端到端跑通仿真

## 1. 本讲目标

学完本讲后，你应该能够：

- 说出 RISC-V² 向量核仿真的**端到端数据流**：从一份伪汇编 CSV，到解码产物，再到驱动器喂入、DUT 运行、波形与性能日志产出。
- 用 `sim_generator.py` 的**三个命令行参数**（csv / AVL / VECTOR_LANES）生成 `decoder_results/` 下的全部产物，并能解释每个产物文件的作用。
- 看懂 `compile_vector_simulator.do` 这份 QuestaSim 脚本里 `vlib / vlog / vsim / log / run -all` 每一步在做什么。
- 知道仿真**何时自动结束**（正常完成 vs. 死锁强制结束），以及如何从 `perf_results/results.log` 与波形里读出总周期数与各类 stall 统计。

本讲是整个入门单元的“收口”：把 u1-l1 到 u1-l4 建立的架构与接口认知，落到“一条命令把仿真跑起来、并看懂结果”这件事上。

## 2. 前置知识

本讲假设你已经学过 **u1-l1（整体架构）** 和 **u1-l2（目录与构建）**。在此基础上，先用通俗语言补几个本讲会用到的硬件仿真术语：

- **DUT（Design Under Test，被测设计）**：这里就是 `vector_top`——我们要验证的向量数据通路。
- **Testbench / TB（测试台）**：包裹 DUT 的“假环境”，负责产生时钟、复位、喂入激励（指令）、检查结果。本工程的 TB 顶层是 `vector_sim_top.sv`。
- **激励（stimulus）**：喂给 DUT 的输入。这里不是真实的标量核译码器，而是一份预先编好的“已译码向量指令”流，由驱动器 `vector_driver.sv` 按节拍推给 DUT。
- **ready/valid 握手**：一种流水线节拍协议。`valid_o=1` 表示发送方有数据，`pop_i=1`（相当于接收方 `ready`）表示接收方这一拍收下了。本讲只要知道“驱动器靠它决定何时推下一条指令”即可，细节留到进阶单元。
- **`$readmemb`**：SystemVerilog 的系统任务，在仿真启动时（`initial`）把一个**纯文本二进制文件**按行读进一个数组/存储器。这是驱动器加载指令的机制。
- **波形（waveform）**：仿真过程中每个信号随时间变化的记录，用 QuestaSim 的波形窗口查看，是定位问题的第一手段。
- **QuestaSim 与 `.do` 脚本**：QuestaSim（也叫 ModelSim 的高级版）是 Mentor 的逻辑仿真器；`.do` 文件是它的 Tcl 命令脚本，把一串 `vlog`/`vsim`/`run` 命令串起来一键执行。

> 一句话定位：本工程**没有公开标量核 RTL**，所以仿真时由 Python 脚本 + 驱动器**冒充**标量核的译码器，把“伪汇编”翻译成 DUT 能吃的已译码指令流。理解了这一点，本讲的全部流程就顺理成章了。

## 4. 核心概念与源码讲解

### 4.1 仿真整体流程：从伪汇编到性能日志

#### 4.1.1 概念说明

整个仿真分成**两个阶段、四类角色**，务必先在脑海里建立这张全局图：

1. **生成阶段（软件，仿真前）**：`sim_generator.py` 把人写的伪汇编 CSV，翻译成 DUT 能直接吃的“已译码信息”，落盘成一堆二进制文本文件。
2. **运行阶段（硬件仿真器，仿真中）**：QuestaSim 编译 RTL + TB；`vector_driver` 在 `initial` 里用 `$readmemb` 把上一阶段的二进制文件读进内部数组，再一拍一拍推给 `vector_top`；`vector_sim_top` 监视结束条件，结束后把性能统计写进 `results.log`。

四类角色：**生成器（Python）→ 产物文件（磁盘）→ 驱动器（读文件/推指令）→ DUT（执行）**。其中“产物文件”是软件与硬件之间的**唯一接口**——这是理解整套框架的关键。

> 为什么这么设计？因为真实系统里，向量指令是由**标量核的译码器**翻译的。本仓库不公开标量核，于是用 Python 脚本**离线**完成这件翻译，把结果存成文件，让 TB 在仿真时回放。这样一来，DUT 本身不需要改一行代码，就能像在真实系统里那样接收已译码指令。

#### 4.1.2 核心流程

端到端数据流（伪代码描述）：

```
[examples/vvadd/instrs.csv]            人写的伪汇编：vld / vadd ...
        │  python sim_generator.py instrs.csv 5000 8
        ▼
[sim_generator.py]
   1. 解析 CSV，去掉注释/空行
   2. 对每条指令查表得到 microop、fu、操作数
   3. 由“目的寄存器个数”算出 maxvl、vl、循环次数 REPEATS
   4. 按 REPEATS 展开，每轮末尾插入 TOTAL_BUBBLE_CYCLES 个气泡
        │  写盘
        ▼
[decoder_results/instrs/*.txt]         13 个二进制产物文件
[decoder_results/autogenerated_params.sv]  SIM_VECTOR_INSTRS = N
        │  QuestaSim: do compile_vector_simulator.do
        ▼
[vector_driver.sv]   $readmemb 把 12 个文件读进数组 → 按 ready/valid 推给 DUT
        │  valid_o / instr_o / pop_i
        ▼
[vector_top (DUT)]   vRRM → vIS → vEX，旁路 vMU
        │
        ▼
[vector_sim_top.sv]  检测 sim_finished / 死锁 → print_perf_stats()
        │
        ▼
[perf_results/results.log]   总周期数、发射数、各类 idle/stall
[wave window]                clk、各级 idle、性能计数器等
```

#### 4.1.3 源码精读

先看 README 对这条流程的一句话总述——它点明了“脚本生成、驱动器回放”的分工：

> A generation script parses the vector instructions and generates the decoded information that a scalar core would produce. Then, the vector driver compiles this information and drives it to the vector datapath.
> —— [vector_simulator/README.md:1-2](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/README.md#L1-L2)

再看驱动器是怎么把产物文件“装回”成指令的——12 个 `$readmemb`，每个对应 `to_vector` 结构体的一个字段（参见 u1-l4）：

[vector_simulator/vector_driver.sv:33-46](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_driver.sv#L33-L46) —— 在 `initial` 中一次性把 12 个二进制文件读进 12 个数组（`data1_list`、`microop_list`、`fu_list`、`vl_list`……），数组下标就是“第几条指令”。

随后驱动器用 `head` 指针逐条取出，拼成 `instr_o`：

[vector_simulator/vector_driver.sv:63-77](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_driver.sv#L63-L77) —— `valid_o = (head < DEPTH) & rst_n & ~instr_is_dummy`，把当前 `head` 处的各字段赋给 `instr_o`。注意 `DEPTH` 参数就是由生成器写出的 `SIM_VECTOR_INSTRS` 传入的（见 4.2）。

这条链路把“软件产物 → 硬件激励”接了起来：**生成器决定喂什么，驱动器决定怎么喂，DUT 负责执行**。

#### 4.1.4 代码实践（源码阅读型）

**目标**：确认产物文件、驱动器、DUT 三者的字段一一对应，建立“接口契约”的直觉。

**步骤**：

1. 打开 [vector_simulator/vector_driver.sv:33-46](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_driver.sv#L33-L46)，列出 12 个 `$readmemb` 各自读的文件名与目标数组。
2. 打开仓库里**已经生成好的** [vector_simulator/decoder_results/instrs/](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/decoder_results/instrs/microop_output.txt) 目录，确认这 12 个文件确实存在（外加一个给人看的 `decoded_output.txt`）。
3. 对照 u1-l4 讲过的 `to_vector` 结构体字段（`dst/src1/src2/data1/data2/immediate/fu/microop/maxvl/vl/reconfigure/valid`），把驱动器第 65–77 行的赋值逐个对上号。

**需要观察的现象**：每个产物文件**第 N 行**对应**第 N 条指令**的同一个字段；文件之间是“按行对齐”的关系。

**预期结果**：例如 `microop_output.txt` 第 4 行是 `0000001`（vadd 的操作码，见 u1-l4），而 `fu_output.txt` 第 4 行是 `10`（INT 功能单元），二者共同描述了“第 4 条指令是 vadd、走整数通路”。（仓库现存产物正是 vvadd / AVL=5000 / LANES=8 的结果，可与下文 4.2.4 互相印证。）

#### 4.1.5 小练习与答案

- **练习 1**：为什么不直接在 TB 里写死几条指令，而要搞一套 Python 生成器 + 文件回放？
  - **答案**：真实系统里向量指令由标量核译码器动态产生；用脚本离线生成“已译码信息”再回放，可以**不改 DUT**就模拟真实喂入方式，还能方便地改变 AVL、lane 数、循环次数等参数做参数化实验。
- **练习 2**：驱动器读 12 个文件，但 `to_vector` 结构体字段不止 12 个（还有 `valid`、`use_mask` 等），这些“多出来”的字段从哪来？
  - **答案**：看 [vector_driver.sv:63-77](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_driver.sv#L63-L77)：`valid` 由 `valid_o` 组合逻辑给出，`use_mask` 直接赋常量 `'0`，它们不需要从文件加载。

### 4.2 sim_generator.py：脚本参数与解码产物

#### 4.2.1 概念说明

`sim_generator.py` 是生成阶段的核心。它做三件事：

1. **翻译**：把每条助记符（`vadd`、`vld`…）查表成 DUT 需要的 `microop`（7 位操作码）和 `fu`（2 位功能单元编码，见 u1-l4 的 `vmacros.sv`）。
2. **展开**：根据应用向量长度 AVL 和 lane 数，算出**单次最多能处理多少元素（maxvl）**，进而算出**需要循环多少次（REPEATS）**——这模拟了标量核围绕向量循环的控制开销。
3. **注气泡**：每个循环末尾插入若干“气泡周期”，模拟标量核在循环边界上执行标量控制指令（判断是否结束、更新指针、分支）时**不发射向量指令**的那些周期。

它还要顺手生成一个 RTL 参数文件，告诉 TB“这次一共要喂多少条指令”。

#### 4.2.2 核心流程

**命令行参数**（顺序固定）：

```
python sim_generator.py <csv 文件> <AVL> <VECTOR_LANES>
```

- 参数 1：输入 CSV 路径（也是产物子目录名的来源）。
- 参数 2：AVL（Application Vector Length，应用向量长度 = 一共要处理多少个元素）。
- 参数 3：VECTOR_LANES（RTL 的向量通道数，必须与 `params.sv` 里的 `VECTOR_LANES` 一致）。

**关键推导**（这是理解产物的核心）。设程序里一共用到 \(N_{\text{vregs}}\) 个不同的**目的**向量寄存器：

\[
\text{max\_vregs} = \left\lfloor \frac{32}{N_{\text{vregs}}} \right\rfloor,\qquad
\text{maxvl} = \text{max\_vregs} \times \text{VECTOR\_LANES}
\]

物理上只有 32 个向量寄存器；硬件循环展开会把它们**轮转复用**（见 u2-l3 的 vRRM）。一个程序用掉的寄存器越多，每个寄存器能被展开的份数（`max_vregs`）就越少，于是单轮 maxvl 就越小。

\[
\text{vl} = \min(\text{maxvl},\ \text{AVL}),\qquad
\text{REPEATS} = \left\lceil \frac{\text{AVL}}{\text{maxvl}} \right\rceil
\]

最终产物里指令总条数：

\[
N_{\text{total}} = 1\ (\text{首条 reconfigure})\ +\ \text{REPEATS} \times \bigl(N_{\text{instr}} + \text{TOTAL\_BUBBLE\_CYCLES}\bigr)
\]

**产出物**：13 个二进制文本文件（每个文件一行一条指令，按字段拆开），加 1 个 `autogenerated_params.sv`（写出 `SIM_VECTOR_INSTRS = N_total`）。

#### 4.2.3 源码精读

**默认参数与气泡周期**。`TOTAL_BUBBLE_CYCLES = 4` 是每个循环末尾注入的气泡数；默认值 AVL=16、LANES=8、文件名 `instrs.csv`：

[sim_generator.py:6-11](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L6-L11) —— 可调的气泡周期与默认参数。README 特别强调：气泡周期数**必须与真实标量循环里的控制指令数匹配**，否则性能指标会失真（[README.md:19](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/README.md#L19)）。

**命令行参数解析**（顺序就是 csv → AVL → VECTOR_LANES）：

[sim_generator.py:288-299](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L288-L299) —— `sys.argv[1/2/3]` 分别覆盖 `FILE_NAME / AVL / VECTOR_LANES`，并用彩色打印回显当前取值。

**maxvl / vl / REPEATS 推导**（对应上面两个公式）：

[sim_generator.py:306-312](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L306-L312) —— `total_vregs` 是不同目的寄存器的个数；`max_vregs = int(32 / total_vregs)`；`maxvl = max_vregs * VECTOR_LANES`；`REPEATS = math.ceil(AVL / maxvl)`。

**助记符 → microop 的查表**（节选 vadd 与 vld）：

[sim_generator.py:71](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L71) 与 [sim_generator.py:106](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L106) —— `vld` 映射到 `1000000`，`vadd` 映射到 `0000001`。这与 u1-l4 讲的 `v_int_op_t` 枚举一致；遇到不认识的助记符会在 [sim_generator.py:150-153](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L150-L153) 报错退出。

[sim_generator.py:462-474](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L462-L474) —— 一次打开 `decoded_output / reconfigure / microop / fu / destination / operand_a/b/c / data1/data2 / immediate / maxvl / vl` 共 13 个文件，产物子目录名由 CSV 文件名（去扩展名）决定：`decoder_results/<subdir>/`。默认 CSV 是 `instrs.csv`，所以子目录就是 `decoder_results/instrs/`——这正是驱动器 `$readmemb` 读的路径。

#### 4.2.4 代码实践（源码阅读型，附真实数值）

**目标**：用 vvadd（AVL=5000，VECTOR_LANES=8）手工算一遍产物结构，再与仓库里**已存在的产物**核对。

**输入程序**（3 条指令，2 个 load 目的 + 1 个 add 目的）：

[vvadd/instrs.csv:7-12](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/examples/vvadd/instrs.csv#L7-L12) —— `vld v0,#0` / `vld v1,#2048` / `vadd v2,v0,v1`。

**手工计算**：

1. 不同的目的寄存器：v0、v1、v2 → \(N_{\text{vregs}} = 3\)。
2. \(\text{max\_vregs} = \lfloor 32/3 \rfloor = 10\)，\(\text{maxvl} = 10 \times 8 = 80\)。
3. \(\text{vl} = \min(80, 5000) = 80\)，\(\text{REPEATS} = \lceil 5000/80 \rceil = 63\)。
4. \(N_{\text{total}} = 1 + 63 \times (3 + 4) = 1 + 441 = 442\)。

**核对仓库现存产物**（这些文件就是用本组参数生成的）：

- [autogenerated_params.sv](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/decoder_results/autogenerated_params.sv)：`SIM_VECTOR_INSTRS = 442` ✅
- [decoded_output.txt:1](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/decoder_results/instrs/decoded_output.txt#L1-L4)：第 1 行 `reconfigure: maxvl == 80` ✅；随后是 vld/vld/vadd，再接 4 行 `BUBBLE CYCLE` ✅。
- [microop_output.txt:1-5](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/decoder_results/instrs/microop_output.txt#L1-L5)：`0000000`(reconfigure) / `1000000`(vld) / `1000000`(vld) / `0000001`(vadd) / `1111111`(气泡) ✅。

**需要观察的现象**：产物文件里“reconfigure → 3 条真指令 → 4 条气泡”的 7 行一组结构，重复 63 次，最前面额外 1 行 reconfigure。

**预期结果**：`decoded_output.txt` 一共 442 行；`microop_output.txt` 同样 442 行，与 `fu_output.txt` 等按行对齐。（本步为纯文件阅读，无需仿真器，结果可立即验证。）

> **注意（待确认）**：`init_main_memory.txt`（vvadd 两个输入数组的初值）**不是** `sim_generator.py` 生成的，而是随仓库预置的。也就是说生成器只负责“指令译码”，真正喂给 load 的数据需要另行准备——换你自己的程序时，这一步要自己造数据。

#### 4.2.5 小练习与答案

- **练习 1**：如果把 vvadd 的 AVL 从 5000 改成 10000（lane 仍为 8），`SIM_VECTOR_INSTRS` 会变成多少？
  - **答案**：maxvl 仍为 80，\(\text{REPEATS}=\lceil 10000/80\rceil=125\)，\(N_{\text{total}}=1+125\times7=876\)。
- **练习 2**：为什么 `maxvl` 依赖“目的寄存器个数”而不是固定值？
  - **答案**：物理向量寄存器只有 32 个，硬件循环展开要轮转复用它们（u2-l3）。程序占用的目的寄存器越多，每个寄存器能展开的份数就越少，单轮能处理的元素数 maxvl 就越小。

### 4.3 编译运行脚本与波形查看

#### 4.3.1 概念说明

产物就绪后，剩下交给 QuestaSim。`compile_vector_simulator.do` 是一键脚本，把“清理 → 建库 → 编译 → 加载 → 记波形 → 运行”串成一条命令。波形由 `wave_simulator.do` 预置，它提前把一批关键信号加进波形窗口，省去你手动 `add wave` 的麻烦。

#### 4.3.2 核心流程

`compile_vector_simulator.do` 的执行顺序：

```
quit -sim                       # 退出当前仿真（若有）
file delete -force work         # 删掉旧的编译库 work/
vlib work                       # 新建工作库 work/
vlog -f files_rtl.f -f files_sim.f +incdir+...   # 编译 RTL + TB 两个文件清单
vsim -novopt work.vector_sim_top -onfinish stop  # 加载 TB 顶层，结束时不立刻退出
log -r /*                       # 记录所有信号到波形
do wave_simulator.do            # 加载预置波形配置
onbreak {wave zoom full}        # 遇到断点时自动缩放波形
run -all                        # 一直运行到 $finish
wave zoom full                  # 全景缩放
```

> 关于 `+incdir`：它告诉编译器去这些目录找 `` `include `` 的文件。三个目录各司其职——`../rtl/shared/`（`params.sv`/`structs.sv`）、`../rtl/vector/`（`vstructs.sv`/`vmacros.sv`）、`../sva/`（断言）。这一点 u1-l2 已详细讲过。

- **练习 1**：`vsim` 加的是 `vector_sim_top` 而不是 `vector_top`，为什么？

- **练习 2**：`log -r /*` 中的 `-r` 是什么意思？为什么仿真规模大时这会很慢？

### 4.4 性能日志与仿真自动结束

#### 4.4.1 概念说明

仿真不会无限跑下去。`vector_sim_top.sv` 实现了**三种结束条件**：

1. **正常完成** `sim_finished`：DUT 全流水线空闲，且驱动器已把全部指令喂完。
2. **空闲超时** `force_end_sim_empty`：指令喂完、但流水线迟迟不空，连续 100 拍判为异常，强制结束。
3. **死锁超时** `force_end_sim_deadlock`：驱动器还在喂有效指令，但发射计数器连续 300 拍没增加（典型的死锁/活锁），强制结束并打印 `Deadlock detected`。

无论哪种结束，都会调用 `print_perf_stats()`，把 7 项性能指标同时打印到控制台和 `perf_results/results.log`。

#### 4.4.2 核心流程

```
每个 posedge clk:
  previous_total_issues <= vis.total_issues        # 记上一拍发射数
  memory_empty   = ~(head < SIM_VECTOR_INSTRS)     # 驱动器是否喂完
  pipeline_empty = memory_empty & ~vis.valid_in     # 喂完且当前无有效指令
  if (pipeline_empty) idle_counter++   else idle_counter=0
  deadlock_en    = valid_instr & (total_issues === previous_total_issues)
  if (deadlock_en)  deadlock_counter++ else deadlock_counter=0

sim_finished           = vector_idle & memory_empty & rst_n
force_end_sim_empty    = (idle_counter == 100)
force_end_sim_deadlock = (deadlock_counter == 300)
sim_stop = 三者之一

wait(sim_stop) -> 若是超时，打印 "Deadlock detected" -> print_perf_stats() -> $finish
```

**性能指标（7 项）**：总周期 `total_cycles`、总发射 `total_issues`、总空闲 `total_idle`、因无有效指令而空闲 `idle_no_valid`、因 pending 而停顿 `stall_pending`、因 locked 而停顿 `stall_locked`、存储侧因 IS 而停顿 `total_ld_stalled_due_is`。前 6 项来自 `vis`，最后一项来自 `vmu_ld_eng`——这些会在进阶单元逐个展开，本讲只要会读数即可。

#### 4.4.3 源码精读

**时钟与复位**（10ns 周期 = 100MHz；复位在头几个上升沿后撤销）：

[vector_sim_top.sv:40-56](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_sim_top.sv#L40-L56) —— `clk` 每 5ns 翻转一次；`rst_n` 先保持 0，若干拍后置 1。

**三种结束条件**：

[vector_sim_top.sv:77-102](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_sim_top.sv#L77-L102) —— 第 77–78 行定义 `memory_empty/pipeline_empty`；第 79–96 行是两个计数器（`idle_counter`、`deadlock_counter`，靠 `deadlock_en` 触发）；第 98–102 行给出 `sim_finished`、两个 `force_end`（阈值分别是 100 和 300）以及它们的“或” `sim_stop`。

#### 4.4.5 小练习与答案

- **练习 1**：死锁检测为什么用“连续 300 拍 `total_issues` 不变”作为判据，而不是“流水线空闲”？
  - **答案**：死锁的典型表现是“有指令在喂、但谁也推进不了”——此时流水线并不空闲（`valid_instr=1`），只是发射数卡住。所以用 `valid_instr & (total_issues === previous_total_issues)` 连续 300 拍来捕捉；而“流水线空闲”由另一个 `idle_counter`（100 拍）单独处理。
- **练习 2**：把死锁阈值 300 调小到 30，会有什么风险？
  - **答案**：正常的长时间停顿（比如一次 cache miss 等待几十拍）可能被误判为死锁，导致仿真提前强制结束、性能统计偏小。阈值要在“能抓真死锁”与“不误伤正常长停顿”之间取平衡。

## 6. 本讲小结

- 仿真分**生成**与**运行**两阶段：`sim_generator.py` 离线把伪汇编译成二进制产物，`vector_driver` 用 `$readmemb` 回放给 DUT，二者以 `decoder_results/` 文件为唯一接口。
- `sim_generator.py` 三个参数依次是 **csv / AVL / VECTOR_LANES**；核心推导是 `maxvl = ⌊32/N_vregs⌋ × VECTOR_LANES`，`REPEATS = ⌈AVL/maxvl⌉`；每轮末尾注入 `TOTAL_BUBBLE_CYCLES=4` 个气泡来模拟标量核的控制开销。
- 产物共 **13 个二进制文件**（按字段拆开、按行对齐）+ 1 个 `autogenerated_params.sv`（写出 `SIM_VECTOR_INSTRS`，决定驱动器数组深度）。
- 一键脚本是 `compile_vector_simulator.do`：`vlib → vlog -f 两份清单 → vsim vector_sim_top → log → do wave_simulator.do → run -all`；波形预置了时钟、各级 idle 与 7 个性能计数器。
- 仿真有**三种结束条件**：正常完成、空闲超时（100 拍）、死锁超时（300 拍，发射数不变）；结束后 `print_perf_stats()` 把 7 项指标写进 `perf_results/results.log`。
- 换自己的程序时，记得**指令译码用生成器、输入数据要自备**（`init_main_memory.txt` 不由生成器产出）。
