本讲解读向量存储单元 VMU 的顶层分派与仲裁逻辑。VMU 是从 vRRM 分叉的访存旁路：它接收 vRRM 改写好的 `memory_remapped_v_instr`，用四路纯组合译码（`is_load`/`is_store`/`is_toepl`/`is_reconf`）将其分派到 load、store、toeplitz 三个并行子引擎；核心依据是 `microop[6]`（`LD_BIT`）切分 load/store，两条 toeplitz 自定义编码优先识别。三个引擎共享唯一的缓存请求端口，VMU 用一个深度为 3 的 `fifo_duth` 作「在途顺序表」，`push_data` 以 `01/10/11` 编码引擎身份，队头 `pop_data` 恒为最老指令，据此按程序顺序仲裁；toeplitz 预取另有「偷空闲周期」特权——当 load/store 本拍未获缓存授权即可越过队头发请求。写回端口在 load 与 toeplitz 间仲裁，由 2 位寄存器 `is_older` 标记更老者优先写回（store 不写回 VRF）；唯一的 unlock 端口被三引擎时分复用，mux 优先级为 toeplitz > load > store，作为 acquire-release 的 release 出口。`reconfigure` 指令特殊：要求三引擎全部排空、三路 `push_*` 同拍同时下发且不入 FIFO。三个子引擎的内部实现留给后续讲义展开。
