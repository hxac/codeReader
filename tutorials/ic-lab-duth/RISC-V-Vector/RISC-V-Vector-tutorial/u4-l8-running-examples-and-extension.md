# u4-l8 示例实战与项目扩展

## 1. 本讲目标

本讲是专家单元的收口，也是整本手册的「毕业设计」。前面二十几讲我们分别拆解了向量数据通路的每一级（vRRM / vIS / vEX / vMU）、每一条机制（重映射、计分板、解耦执行、归约树）和每一层验证（TB、驱动器、SVA）。本讲要把它们全部串起来，完成三件事：

1. **读懂四个真实示例内核**（vvadd / saxpy / dot_product / fir）的伪汇编写法，理解它们各自考察了向量核的哪种能力。
2. **跑通从 CSV 到 `results.log` 的完整端到端流程**，并用一条具体指令（saxpy）在生成器里产生的 870 条记录作为活体证据，验证我们对「VL 展开 + 气泡注入」机制的理解。
3. **识别项目的扩展点与硬限制**——以一条已经全链路实现的整数指令 `vsub` 为参照模板，写出新增一条整数指令的完整改动方案；并结合 README 的 Future Work 讨论浮点 lane、增大 lane、加背压等扩展方向。

学完后，你应当能够独立给这个向量核添加一条新指令，并知道每一步会牵动哪些文件。

## 2. 前置知识

本讲默认你已经学完 u4-l5（`sim_generator.py` 解码原理）与 u2-l8（整数 ALU 与归约树）。这里只做最简回顾：

- **伪汇编到硬件的翻译链**：示例程序用一种「伪汇编」写在 `instrs.csv` 里。离线脚本 `sim_generator.py` 冒充标量核的译码器，把每条助记符查表得到 `{fu, microop}`，再按 AVL 展开成若干轮，每轮末尾插 4 条「气泡」哨兵指令模拟标量核的控制开销。产物是一堆按行对齐的二进制文件，被驱动器回放给 DUT。真操作码其实是 `{fu, microop}` 拼起来的组合，而不是单独的 microop。
- **三层「可用」概念**：一条指令要真正能在硬件上跑通，必须同时满足三层——译码层（生成器认识它）、结构层（`vstructs.sv` 的操作码枚举里有它）、执行层（`v_int_alu.sv` 的 `case` 里有它的实现）、外加验证层（SVA 白名单允许它）。这是 u4-l5 的核心结论「译码层可用 ≠ 执行层可用」。
- **变延迟执行**：简单 ALU 指令 EX1 一拍出结果，VMUL 三拍，VDIV 四拍，归约指令走 EX1–EX4 的归约树。所有指令物理上都走完流水，差异只在最早可转发的那一拍。
- **最多 16 lane**：归约树级数 \(= \lceil \log_2(\text{LANES}) \rceil\)，当前静态生成四级（步长 1/2/4/8），所以 lane 数硬上限是 16。

如果以上任何一点你觉得陌生，建议先回看对应讲义再继续。

## 4. 核心概念与源码讲解

### 4.1 四个示例程序的写法与结构

#### 4.1.1 概念说明

向量核存在的意义是「一条指令处理一批数据」。但写示例程序时，我们并不直接写硬件二进制，而是写一种**伪汇编**：每行一条助记符，操作数用 `vN` 表示向量寄存器、`#K` 表示立即数（通常是字节地址）。这种写法贴近 RISC-V 向量汇编，但又足够简单，能让生成器离线翻译。

四个示例被有意设计成「由易到难、各考一项能力」：

- **vvadd**：纯计算（两条 load + 一条 add），考察基本数据通路。
- **saxpy**：load + 计算 + store 全流程，考察访存岔路 vMU 与解耦执行。
- **dot_product**：含归约指令 `vradd`，考察跨 lane 归约树。
- **fir**：多抽头乘累加、指令间大量独立，考察计分板的乱序发射能力与指令级并行（ILP）。

理解它们的结构，就理解了向量核的应用谱。

#### 4.1.2 核心流程

伪汇编程序的通用结构是：

1. **取数据**：用 `vld vd, #addr` 把一段连续内存装入向量寄存器，`#addr` 是字节地址。
2. **算数据**：用算术/逻辑指令（`vadd`/`vmul`/`vandi`/...）在寄存器间做逐元素运算。
3. **（可选）归约**：用 `vradd`/`vrand` 等把一个寄存器里的多个元素合并成一个。
4. **（可选）存回**：用 `vsw vd, #addr` 把结果写回内存。

需要注意：**示例里没有循环**。循环是由生成器按 AVL 自动「展开」出来的——你在 CSV 里写的是「循环体一次」，生成器会复制它 `REPEATS` 份。这点极易混淆，下一节会详讲。

另一个关键点是**寄存器分组（register grouping）**。每条指令的 `maxvl`（单条指令一次能处理的最大元素数）不是写死的，而是由「这个循环体一共用了几个目的向量寄存器」反推出来的：用到的目的寄存器越多，每个寄存器分到的元素越少。这正是 u2-l3/u2-l6 讲的「动态寄存器分配」在程序层面的体现。

#### 4.1.3 源码精读

**vvadd —— 最朴素的向量加法**（\(z[i]=x[i]+y[i]\)），全程序只有三条指令：

[vvadd/instrs.csv:L8-L12](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/examples/vvadd/instrs.csv#L8-L12) 加载两个数组到 v0、v1，再用 `vadd v2,v0,v1` 相加。注意第二条 `vld v1,#2048`：`#2048=0x800` 是第二个数组的字节起始地址，说明数组 x 和 y 在主存里间隔 2048 字节（512 个 32 位字）。它没有显式 `vsw` 存回——结果留在 v2，仿真上看重的是周期数与功能正确性。

**saxpy —— 全流程示范**（\(y[i]=a\cdot x[i]+y[i]\)，这里用 `vmul v4,v3,v3` 近似）：

[saxpy/instrs.csv:L8-L21](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/examples/saxpy/instrs.csv#L8-L21) 七条指令：三条 `vld`（条件、x、y 分别在字节地址 `#0`、`#640=0x280`、`#1280=0x500`）、一条 `vseq`（生成全 1 掩码）、`vmul` 做乘法、`vadd` 做累加、最后 `vsw v4,#2560=0xA00` 存回。它是四个示例里唯一同时覆盖 load/compute/store 的，所以本讲后面拿它当端到端演示样本。

**dot_product —— 归约指令登场**（\(\sum x[i]\cdot y[i]\)）：

[dot_product/instrs.csv:L8-L18](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/examples/dot_product/instrs.csv#L8-L18) 开头 `vandi v0,v0,#0` 把累加器 v0 清零（「与立即数 0」）；两条 `vld` 取数组；`vmul v3,v1,v2` 逐元素乘；**`vradd v4,v3` 是关键**——它把 v3 里所有元素用归约树加成一个标量放进 v4；最后 `vadd v4,v0` 把这一段的归约结果累加到总和中。这条 `vradd` 走的就是 u2-l8 讲的对数深度归约树（EX1–EX4，步长 1/2/4/8）。

**fir —— 指令级并行密集型**（多抽头 FIR 滤波）：

[fir/instrs.csv:L8-L32](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/examples/fir/instrs.csv#L8-L32) 一连串 `vld`/`vslli`/`vmul`/`vadd`，把多个抽头（tap）的采样相乘再累加。注意它的写法：先把两路抽头相乘、相加（`vadd v5,v3,v2`），同时再去取另外两路（`vld v4`、`vmul v4`、`vld v0`），最后汇合（`vadd v6,...`、`vadd v5,v5,v6`）。这种「交错排布」让相邻指令之间有大量**无数据依赖**的对，正好考验 vIS 计分板能否把它们尽量同周期发射、填满 lane。它是四个示例里 ILP 最丰富的。

#### 4.1.4 代码实践（分析型）

**实践目标**：用生成器的公式，**预测**四个示例在 `AVL=5000, VECTOR_LANES=8` 下各自的 `maxvl` 与 `REPEATS`，不必运行。

**操作步骤**：

1. 对每个示例，数出**不同的目的向量寄存器个数** \(N_{\text{vregs}}\)（看每条指令的第一个操作数，去重）。注意 `vsw` 的第一个操作数（被存的源）也会被生成器计入目的列表——这是 u4-l5 提醒过的隐藏陷阱。
2. 套公式（下一节 4.2 会从源码证实）：
   - \(\text{maxvl} = \lfloor 32 / N_{\text{vregs}} \rfloor \times \text{LANES}\)
   - \(\text{REPEATS} = \lceil \text{AVL} / \text{maxvl} \rceil\)

**预期结果**（按生成器公式推算，待本地验证）：

| 示例 | \(N_{\text{vregs}}\) | maxvl (8 lane) | REPEATS (AVL=5000) |
| --- | --- | --- | --- |
| vvadd | 3（v0,v1,v2） | \(\lfloor 32/3\rfloor\times 8 = 80\) | \(\lceil 5000/80\rceil = 63\) |
| saxpy | 4（v1,v2,v3,v4） | \(8\times 8 = 64\) | \(\lceil 5000/64\rceil = 79\) |
| dot_product | 5（v0,v1,v2,v3,v4） | \(\lfloor 32/5\rfloor\times 8 = 48\) | \(\lceil 5000/48\rceil = 105\) |
| fir | 7（v0..v6） | \(\lfloor 32/7\rfloor\times 8 = 32\) | \(\lceil 5000/32\rceil = 157\) |

**需要观察的现象**：目的寄存器用得越多，`maxvl` 越小，`REPEATS` 越多——也就是说「循环体越复杂，单条指令处理的元素越少，总轮次越多」。这正是寄存器分组的代价。

#### 4.1.5 小练习与答案

**练习 1**：为什么 fir 的 `maxvl` 只有 32，而 vvadd 有 80？

> **答**：fir 的循环体用了 7 个不同目的寄存器，32 个物理向量寄存器被均分成 7 组，每组只能放 \(\lfloor 32/7\rfloor=4\) 个「lane 块」，故 `maxvl=4×8=32`；vvadd 只用 3 个，每组 \(\lfloor 32/3\rfloor=10\) 块，`maxvl=10×8=80`。用到的寄存器越多，单寄存器容量越小。

**练习 2**：如果把 fir 的某些中间寄存器复用（减少 \(N_{\text{vregs}}\)），对性能是绝对好事吗？

> **答**：不一定。减少 \(N_{\text{vregs}}\) 会增大 `maxvl`、减少 `REPEATS`（少付每轮 4 拍气泡开销），但也会引入更多真数据依赖（RAW），可能增加 `stall_pending`。这是一个「气泡开销 vs 冒险停顿」的权衡，需要跑仿真看 `results.log` 才能定夺（见 u4-l7）。

### 4.2 从 CSV 到性能结果的端到端流程

#### 4.2.1 概念说明

「端到端」指从一份人类可读的 `instrs.csv`，直到产出一份含 `total_cycles` 等指标的 `results.log`。这条链路分四段：

1. **生成（Python）**：`sim_generator.py` 读 CSV，算 `maxvl`/`REPEATS`，展开 + 注气泡，写出一批二进制文件和一个 `autogenerated_params.sv`。
2. **回放（驱动器）**：`vector_driver.sv` 用 `$readmemb` 把这些二进制文件灌进 12 个数组，按 ready/valid 握手一条条喂给 `vector_top`。
3. **执行（DUT）**：向量数据通路真正跑这些指令，访存请求落到 `data_cache`/`main_memory`。
4. **度量（TB 顶层）**：`vector_sim_top.sv` 在仿真结束时把内部计数器打印成 `results.log`。

其中第 1 段是软件、第 2–4 段是硬件仿真。理解这条链的最大好处是：你能用第 1 段的公式**预测**第 4 段的数字，再用实际仿真**验证**预测。本节就用 saxpy 的 870 条记录做这次验证。

#### 4.2.2 核心流程

`sim_generator.py` 的主流程（精简伪代码）：

```
读 CSV → instruction_list
N_vregs = 不同目的寄存器数
maxvl   = (32 / N_vregs) * VECTOR_LANES
REPEATS = ceil(AVL / maxvl)

写一条 reconfigure 指令（第 1 行）           # 设置 maxvl
for j in 0..REPEATS-1:
    current_vl = (最后一轮 ? AVL-(REPEATS-1)*maxvl : maxvl)
    for 每条指令:
        写 {microop, fu, dst, src1, src2, imm, vl=current_vl, ...}
    for k in 0..3:                            # TOTAL_BUBBLE_CYCLES=4
        写一条气泡哨兵 {reconfigure=1, fu=11, microop=1111111, vl=0}
写 SIM_VECTOR_INSTRS = 总行数 → autogenerated_params.sv
```

几个要点：

- **第一行恒为 reconfigure**：它把这一段程序的 `maxvl` 写进硬件配置寄存器，并对齐所有字段文件。所以你看到任何 `microop_output.txt` 的第 1 行总是 `0000000`、`fu_output.txt` 第 1 行总是 `00`——那不是 `vsw`，而是 reconfigure。
- **VL 展开是「复制循环体」**：CSV 里写一次的循环体被原样复制 `REPEATS` 份，每份的 `vl` 字段在最后一轮用尾数（`AVL-(REPEATS-1)*maxvl`），其余轮用 `maxvl`。硬件里对应 u2-l6 的 `current_exp_loop` 与 `vl_therm` 尾部掩码。
- **气泡哨兵的精确编码**：五字段合取 `reconfigure=1 & fu=11 & microop=1111111 & vl=0 & maxvl=0`，驱动器识别后「弹出但不发射」（见 u4-l4）。
- **总指令数**：\(\text{SIM\_VECTOR\_INSTRS} = 1 + \text{REPEATS}\times(N_{\text{instr}} + 4)\)。

**输入数据从哪来**——这是新手最容易卡住的地方。生成器**只产出指令流，不产出输入数据**。真正的数组初值由主存模型加载：

[vector_sim_top.sv:L230](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/vector_sim_top.sv#L230) 把 `main_memory` 的 `FILE_NAME` 指到 `../vector_simulator/decoder_results/init_main_memory.txt`；

[main_memory.sv:L71](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/rtl/shared/main_memory.sv#L71) 用 `$readmemh(FILE_NAME, ram)` 把它读进 RAM。所以**你必须自己准备一份 `init_main_memory.txt`**（十六进制、每行一个字），里面填上 saxpy 的 x、y 数组，地址要和 CSV 里的 `#640`、`#1280` 对齐。仓库示例目录里的 `test_mem_*.txt` 只是参考样本，并不自动被加载。

#### 4.2.4 代码实践（运行型）

**实践目标**：按官方步骤把 vvadd 跑起来，看到 `results.log`。

**操作步骤**（来自 [vector_simulator/README.md:L21-L25](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/README.md#L21-L25)）：

1. 把 `vector_simulator/examples/vvadd/instrs.csv` 复制到 `vector_simulator/` 目录，重命名为 `instrs.csv`。
2. 在 `vector_simulator/` 下运行：`python ./sim_generator.py ./instrs.csv 5000 8`（Windows 下把 `./` 换成 `.\`）。
3. 打开 QuestaSim，把工作目录切到 `vector_simulator/`。
4. 执行 `do compile_vector_simulator.do`。它会自动 `vlib`/`vlog`/`vsim`/`log`/`run -all`，结束后加载预置波形。
5. 仿真结束后查看 `vector_simulator/perf_results/results.log`。

**需要观察的现象**：`results.log` 里应有 `total_cycles`、`total_issues`、`total_idle`、`stall_pending`、`stall_locked`、`total_ld_stalled_due_is` 等七项（含义见 u4-l7）。

**预期结果**：对 vvadd（AVL=5000, 8 lane），按公式应有 `SIM_VECTOR_INSTRS = 1 + 63×(3+4) = 442` 条指令记录；`total_issues` 应接近「真指令条数」（即 \(63\times 3=189\)，因为 4×63=252 条气泡不计入发射）。**实际 cycle 数待本地验证**——它会因 load miss、冒险停顿而高于 `total_issues`。

> 若你没有 QuestaSim 许可证，可改做「分析型实践」：只跑第 2 步生成器（纯 Python），用 `decoded_output.txt` 与 `autogenerated_params.sv` 核对 `SIM_VECTOR_INSTRS`，同样能验证本节机制。

#### 4.2.5 小练习与答案

**练习 1**：saxpy 产物 `microop_output.txt` 第 1 行是 `0000000`，有人据此说「saxpy 第一条指令是 `vsw`」，对吗？

> **答**：不对。那是生成器在所有真指令之前写入的 **reconfigure 指令**（`microop=0000000, fu=00, reconfigure=1`），用于设置 `maxvl=64`。`vsw` 的 microop 虽也是 `0000000`，但要靠 `fu=00` 且 `reconfigure=0`、且在循环体中段出现来区分。区分依据是真操作码 `{fu, microop, reconfigure}` 的组合，不是 microop 单字段。

**练习 2**：把 saxpy 的 AVL 从 5000 改成 64（恰好等于 maxvl），`SIM_VECTOR_INSTRS` 会变成多少？

> **答**：`REPEATS = ceil(64/64) = 1`，故 `SIM_VECTOR_INSTRS = 1 + 1×(7+4) = 12`。AVL 再从 64 增到 65，`REPEATS` 跳到 2，总数跳到 `1+2×11=23`——这就是下节综合实践要观察的「阶梯效应」。

### 4.3 扩展方向与硬限制

#### 4.3.1 概念说明

学完前面所有讲义，你大概会问：「我能给这个核加自己的指令吗？能加浮点吗？能加到 32 lane 吗？」本节正面回答这些问题，并把「新增一条整数指令」做成一份可照抄的改动清单。

关键认知是「一条指令要跑通，必须四层都改」：

1. **译码层**（`sim_generator.py` 的 `get_microop`/`get_fu` 两个字典）
2. **结构层**（`vstructs.sv` 的 `v_int_op_t` 枚举）
3. **执行层**（`v_int_alu.sv` 的 `case`）
4. **验证层**（`sva/vector_top_sva.sv` 的 INT_FU 白名单）

少了任何一层，要么生成器不认识、要么硬件算不出、要么断言会误报。好消息是：项目里**已经有一条全链路实现的整数指令 `vsub`**，它就是最好的参照模板。本节先用 `vsub` 把四层走一遍，再让你照着设计一条真正全新的指令。

至于浮点、大 lane、背压，这些都写在 [README.md](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/README.md) 的 Future Work 里，属于「已知想做但还没做」的方向，本节会逐一说明它们的硬限制在哪里。

#### 4.3.2 核心流程

**新增整数指令的四层改动（以 `vsub` 为参照模板）**：

| 层 | 文件 | 要做什么 | vsub 的现成实现 |
| --- | --- | --- | --- |
| 译码 | `sim_generator.py` | 在 `get_microop` 加 `"vsub":"0000101"`，在 `get_fu` 加 `"vsub":"10"` | 已在 L110、L208 |
| 结构 | `rtl/vector/vstructs.sv` | 在 `v_int_op_t` 枚举加 `VSUB = 7'b0000101` | 已在 L129 |
| 执行 | `rtl/vector/v_int_alu.sv` | 在简单 ALU 的 `case` 加 `7'b0000101: result_int = a - b; valid_int_ex1 = valid_i;` | 已在 L137–L141 |
| 验证 | `sva/vector_top_sva.sv` | 在 INT_FU 白名单 `inside { }` 加 `7'b0000101` | 已在 L79 |

注意第三层：`vsub` 是「简单 ALU」，所以放进 EX1 一拍就绪的 `case`（驱动 `valid_int_ex1`），由 [v_int_alu.sv:L847](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/rtl/vector/v_int_alu.sv#L847) 的 `ready_res_ex1_o = valid_int_ex1 | valid_rdc_ex1` 自动声明「EX1 就绪」。如果你加的是乘法类指令，则要改 MUL `case`（EX3 就绪）；除法类改 DIV `case`（EX4 就绪）——这就是「变延迟」在扩展时的体现。

**三个扩展方向与各自的硬限制**：

- **加浮点 lane**：Future Work 明确列出。骨架已留：`vex_pipe` 用 2 位 `fu` 路由，`v_fp_alu` 已是占位模块（见 u2-l7）。限制是要补一套浮点 ALU、扩展操作码枚举与 SVA 白名单。
- **增大 lane（>16）**：受归约树级数限制。当前归约树静态生成四级（步长 1/2/4/8），最多连 16 lane。要超过必须**加背压**（见下条）并新增归约级——这是结构性改动，不是改个参数。
- **加背压 + 解耦 vis/vmu**：Future Work 说当前 vis 与 vmu 之间「时序很紧」（timing pressed），曾尝试解耦但因硬件解压导致面积变小但复杂。加背压是支持 >16 lane 的前提。

#### 4.3.3 源码精读

**vsub 的四层证据链**——

译码层：[sim_generator.py:L110](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L110) `"vsub":"0000101"` 与 [sim_generator.py:L208](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/vector_simulator/sim_generator.py#L208) `"vsub":"10"`。

结构层：[vstructs.sv:L125-L155](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/rtl/vector/vstructs.sv#L125-L155) 的 `v_int_op_t` 枚举里，[vstructs.sv:L129](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/rtl/vector/vstructs.sv#L129) `VSUB = 7'b0000101`。

执行层：[v_int_alu.sv:L137-L141](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/rtl/vector/v_int_alu.sv#L137-L141) 在简单 ALU 的 `case (microop_i)` 里，`7'b0000101` 分支做 `result_int = data_a_u_ex1 - data_b_u_ex1` 并置 `valid_int_ex1 = valid_i`。

验证层：[vector_top_sva.sv:L73-L79](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/sva/vector_top_sva.sv#L73-L79) 的 INT_FU 白名单 `inside { ... }` 列出了所有合法整数 microop，其中第 79 行就是 `7'b0000101`（vsub）。任何不在表里的 microop 流入 INT_FU 都会触发 `$error`。

这条链完整闭合——这就是「一条指令真正可用」的样子。

**Future Work 与硬限制**——

[README.md:L34](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/README.md#L34) 写明 `Current maximum vector lanes supported is 16.`（当前最大 16 lane）。

[README.md:L37-L42](https://github.com/ic-lab-duth/RISC-V-Vector/blob/8ded0f4036bcb34868c2d15475883d03bff37328/README.md#L37-L42) 列出四项 Future Work：对齐新版 RISC-V ISA、用优化硬件替换 MUL/DIV 以降延迟、解耦 vis/vmu 改善时序、**加背压以支持 >16 lane**、**加浮点执行 lane**。后两项就是本节讨论的扩展方向。

#### 4.3.4 代码实践（源码阅读型）

**实践目标**：在源码里把 `vsub` 的四层链亲手核对一遍，确认「译码层可用 = 执行层可用」对 vsub 成立。

**操作步骤**：

1. 打开 `sim_generator.py`，分别跳到 L110 与 L208，确认 `vsub` 的 `microop=0000101`、`fu=10`。
2. 打开 `vstructs.sv` L129，确认枚举里有 `VSUB = 7'b0000101`。
3. 打开 `v_int_alu.sv` L137–141，确认简单 ALU `case` 有 `7'b0000101` 分支且做了减法。
4. 打开 `vector_top_sva.sv` L73–79，确认 INT_FU 白名单包含 `7'b0000101`。

**需要观察的现象**：四处的 7 位编码 `0000101` 完全一致；执行层用的是「无符号减」`data_a_u_ex1 - data_b_u_ex1`（与 `vadd` 的无符号加法对称）。

**预期结果**：四层闭合，`vsub` 可端到端跑通。这也反证了：如果哪一层缺了，指令就会在那一层「断掉」——译码层缺则生成器 `exit(-1)`；结构层缺则枚举无此名（仅影响可读性，硬件按数值匹配）；执行层缺则落入 `default: result_int = 'x`（结果为不定）；验证层缺则 SVA 报错。

#### 4.3.5 小练习与答案

**练习 1**：假设你只改了 `sim_generator.py` 让它认识 `vnot`，却没改 `v_int_alu.sv`，仿真会怎样？

> **答**：生成器能产出 `vnot` 的指令记录，驱动器也能把它喂进 DUT。但 `v_int_alu` 的 `case` 没有对应分支，落入 `default`，`result_int = 'x`、`valid_int_ex1 = 0`，于是结果是不定值且 EX1 不声明就绪——指令「卡」在执行级或写出 X。同时若 SVA 白名单也没加，还会触发 `invalid microop for INT_FU` 报错。这正是「译码层可用 ≠ 执行层可用」。

**练习 2**：为什么把 lane 从 16 加到 32 不能只改 `VECTOR_LANES` 参数？

> **答**：两个原因。其一，归约树由 `vex.sv` 用 `generate` 按 lane 编号静态连出四级（步长 1/2/4/8），32 lane 需要第五级（步长 16），不改归约树则 `vradd` 结果错误。其二，lane 数变化还要同步改 `vstructs.sv` 的 `DUMMY_VECTOR_LANES`（u1-l3 提醒的隐藏耦合），否则结构体字段宽度与实际不一致。要稳妥支持 >16 lane，Future Work 指出的「加背压」是前提。

## 5. 综合实践

这是一个贯穿本讲全部内容的毕业任务，分两部分。

### 第一部分：saxpy 端到端跑通 + AVL 扫描

**任务**：用 saxpy 示例完成一次端到端仿真，再扫描 AVL 观察周期变化趋势。

**步骤**：

1. 复制 `examples/saxpy/instrs.csv` 到 `vector_simulator/instrs.csv`。
2. 准备一份 `decoder_results/init_main_memory.txt`，在字节地址 `0x280`(640) 起填 x 数组、`0x500`(1280) 起填 y 数组（各 5000 个 32 位字，十六进制每行一个）。
3. 对 AVL ∈ {64, 128, 1000, 5000} 分别运行 `python ./sim_generator.py ./instrs.csv <AVL> 8`，记录每次的 `SIM_VECTOR_INSTRS`。
4. 对每个 AVL 跑 `do compile_vector_simulator.do`，从 `results.log` 记录 `total_cycles`。
5. 画一张 `AVL → total_cycles` 图。

**预期趋势（按公式推算，实际待本地验证）**：

- \(\text{REPEATS}=\lceil \text{AVL}/64\rceil\)，所以 AVL 在 \((0,64]\) 内 `REPEATS=1`（cycle 数基本持平，只有一段启动填充）；AVL=65 时 `REPEATS` 跳到 2，cycle 数阶跃上升；之后大致**线性增长**，斜率约等于「每轮 7 条真指令 + 4 拍气泡 + 访存/冒险停顿」。
- 你会观察到一个**阶梯 + 线性**的混合趋势：阶梯发生在 AVL 跨过 `maxvl` 整数倍处，线性段在每个阶梯内部。这正是「VL 展开」在宏观性能上的签名。
- 若你有许可证，再换 `VECTOR_LANES=4` 与 `16` 各跑一次 AVL=5000：lane 翻倍使 `maxvl` 翻倍、`REPEATS` 减半，cycle 数应明显下降——但注意 16 已是上限，不能再大。

> 无许可证时可只做第 3 步（纯 Python），用 `SIM_VECTOR_INSTRS` 的变化代替 `total_cycles` 验证趋势。

### 第二部分：设计一条全新整数指令 `vnot` 的扩展方案

**任务**：参照 `vsub` 的四层模板，为一条**当前不存在**的指令写完整改动方案。我们定义 `vnot vd, vs`（按位取反，\(vd = \sim vs\)），它是单周期简单 ALU 指令（和 `vand` 同类）。

**交付物：一份改动清单**

1. **选一个未被占用的 7 位操作码**。查 `v_int_alu.sv` 的三个 `case`（int/mul/div）与归约、以及 `vector_top_sva.sv` 白名单：简单整数指令组用到 `0000001`–`0000110`、`0010000`–`0011110`、`0100000`–`0100011`，乘除占用 `0000111`–`0001111`、`0100100`。紧邻比较指令组之后的 `7'b0011111` 目前在各层均未出现——选它（**落地前请在四个文件里再 grep 一遍 `0011111` 确认**）。

2. **译码层** `sim_generator.py`：
   - 在 `get_microop`（约 L106–L148）加 `"vnot":"0011111"`；
   - 在 `get_fu`（约 L204–L233）加 `"vnot":"10"`。

3. **结构层** `rtl/vector/vstructs.sv`：在 `v_int_op_t` 枚举（L125–L155）加 `VNOT = 7'b0011111`。

4. **执行层** `rtl/vector/v_int_alu.sv`：在简单 ALU 的 `case (microop_i)`（L116–L250）加一个分支（示例代码，仿照 VAND）：
   ```systemverilog
   7'b0011111 : begin
       // VNOT
       result_int    = ~data_a_u_ex1;
       valid_int_ex1 = valid_i;
   end
   ```
   因为是 EX1 一拍就绪，无需动 `ready_res_ex1_o`（它已含 `valid_int_ex1`）。

5. **验证层** `sva/vector_top_sva.sv`：在 INT_FU 白名单（L74 起）加 `7'b0011111`。

6. **联调**：写一个最小 CSV（如 `vld v1,#0` / `vnot v2,v1` / `vsw v2,#N`），用生成器跑通，确认 `microop_output.txt` 出现 `0011111`，再在 QuestaSim 里验证 v2 = ~v1。

**验收标准**：四层编码一致、SVA 不报错、波形里 v2 逐元素等于 v1 按位取反。至此你已独立完成一次向量核指令扩展。

## 6. 本讲小结

- 四个示例从易到难各有侧重：vvadd 考基本数据通路、saxpy 考 load/compute/store 全流程与解耦执行、dot_product 考跨 lane 归约树、fir 考指令级并行与计分板乱序发射。
- 端到端流程是「生成器（Python）→ 驱动器回放 → DUT 执行 → TB 打分」；生成器先写一条 reconfigure，再把循环体复制 `REPEATS` 份、每轮注 4 条气泡，总行数 `SIM_VECTOR_INSTRS = 1 + REPEATS×(N_instr+4)`。saxpy 的 870 条记录逐项验证了这套公式。
- 输入数据不在生成器产出之列，需自备 `decoder_results/init_main_memory.txt`（由 `main_memory.sv` 的 `$readmemh` 加载），地址要与 CSV 里的 `#addr` 对齐。
- 一条指令要真正可用，**四层缺一不可**：译码层（`sim_generator.py` 两字典）、结构层（`vstructs.sv` 枚举）、执行层（`v_int_alu.sv` 的 `case`）、验证层（SVA 白名单）。`vsub` 是已闭合的参照模板，`vnot` 是可照抄的练习。
- 扩展硬限制来自归约树：lane 数上限 16（级数 \( \lceil\log_2 L\rceil \)），突破需加背压并新增归约级；浮点 lane 骨架已留（`v_fp_alu` 占位）但需补完整实现；vis/vmu 之间时序紧张是已知的优化项。

## 7. 下一步学习建议

本讲已是手册最后一篇。如果你一路做完，建议按以下方向继续深耕：

1. **动手扩展**：落地综合实践里的 `vnot`，再挑战一条乘法类指令（如 `vmulw`，注意它要改 MUL `case`、EX3 就绪），体会「变延迟」对扩展的影响。
2. **性能调优实战**：用 u4-l7 的方法论，对 fir 这种 ILP 密集内核扫描转发点（`VECTOR_FWD_POINT_A/B`）与气泡周期，寻找最优配置，并把结论与波形证据整理成报告。
3. **读论文**：README 引用了 ISCAS 2020 的原文（RISC-V²: A Scalable RISC-V Vector Processor）。结合你对源码的理解去读，会发现「寄存器重映射 + 解耦执行 + 归约树」三大创新在论文里的定量收益，与你在 `results.log` 里看到的 stall 计数遥相呼应。
4. **跟进上游**：关注 Future Work 的几项（对齐新版 RISC-V 向量 ISA、优化 MUL/DIV、解耦 vis/vmu、浮点 lane、>16 lane 背压）——它们既是这个开源项目的路线图，也是你可以贡献的方向。
