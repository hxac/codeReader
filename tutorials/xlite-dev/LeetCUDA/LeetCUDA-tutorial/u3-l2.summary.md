本讲拆解 LeetCUDA 的 `.py` 包装脚本两大标准模式：**正确性验证**与**基准计时**，是 kernel 写完后「算得对不对、有多快」两道关。

验证范式是：构造输入张量，跑自己 kernel，与 **PyTorch 参考实现**（`torch.relu`/`torch.matmul`/`F.leaky_relu`）比对逐元素**最大绝对误差 Max Err**，小于**算子相关阈值**即 PASS，不要求逐比特相同（阈值随精度放宽，转置约 1e-6、fp16 GEMM 放宽到 1.0）。`relu.py` 实际用的是更轻量的「打印前几个值肉眼比对」**冒烟测试**，有盲区，严谨场景应升级为显式 `(out-ref).abs().max()`。注意 kernel 异步启动，读 `out` 前须 `torch.cuda.synchronize()`。

计时脚手架全仓库统一：`out.fill_(0) → warmup 预热 → sync → 计时 → iters 多轮 → sync → 取平均`。两次 `torch.cuda.synchronize()` 是命门——否则只量到「提交命令」时间。轮数按算子单次耗时权衡（relu 1000 轮、GEMM 10 轮）。

`hgemm/bench/prof.py` 把脚手架工业化：**argparse** 参数、**TFLOPS = 2MNK×1e-12 / t** 指标、**预分配最大 buffer + 切片**的 fast profiling、`gc.collect + sleep` 防 OOM 与热降频、与 **cuBLAS** 对比绘图。库加载分两范式：`relu.py` 直接 `load()` JIT；`prof.py` 经 `try_load_hgemm_library` 先试 `toy_hgemm` 预编译包再 JIT。
