本讲覆盖 `_persistent_autotune.py` 的运行时配置查找模块。`autotune=False` 时，前向/反向 launcher 构造 `PersistentConfigRequest` 调用 `lookup_persistent_config`：命中用设备 JSON 的 config，返回 None 则回退写死的默认 config，永不抛异常。查找分三步：① 加载——设备名文件（`{sanitized_name}.json`）优先，缺失时按 `compute_capability` 同架构回退，二者皆空则空列表；② 过滤——direction/kernel/causal/has_attn_bias/has_dropout/use_gemv 等硬过滤，dtype 精确优先且 fp16 可回退 bf16（反向不可），TMA/WS 对老 JSON 用 kernel 名加 `warp_specialize` 推断，头布局不参与过滤、仅作平局偏好；③ 就近匹配——head_dim 用 `nearest_value`（最近、并列取大），seqlen 用 `upper_or_max_value`（最小上界、否则取最大）。全程 `@lru_cache`，cache key 含 `FFPA_TUNED_CONFIG_DIR`、device_index 与 request。关键环境变量 `FFPA_TUNED_CONFIG_DIR`、`FFPA_SKIP_PERSISIT_TUNED_CONFIG`（注意此拼写）、`FFPA_LOGGER_LEVEL`，配合 `debug_once` 观察 selected/cache hit/miss 三态。本讲承接 u8-l2 生成端，由 u8-l4 多卡并行生成闭环。
