本讲围绕两个测试文件，讲解 FFPA「自动调优与持久化配置」链路的回归测试，全部为 CPU 安全的纯 Python 逻辑测试，不跑真实 kernel。

模块一（`test_persistent_autotune_config.py`）：用「monkeypatch 伪造设备名 + `tmp_path` 重定向 config 目录 + `clear_config_cache` 隔离 + `write_config_file` 构造 entry」四件套，把运行时 `lookup_persistent_config` 的命中、缓存、回退、就近匹配变成可执行断言。覆盖三类退化路径：schema 守门（`schema_version` 不符返回空）、dtype 单向回退（fp16→bf16、精确优先、反向不回退）、arch 回退（设备专属优先、按 `compute_capability` 匹配）。就近匹配有严格数学：`nearest_value` 以 `(距离, −v)` 实现并列取大，`upper_or_max_value` 取最小上界否则取最大。

模块二（`test_triton_autotune_mode.py`）：前半锁定 `TritonBackend` 默认值与硬约束——`autotune_mode∈{fast,max}`、`persist_dkdv` 依赖 `enable_tma`、TMA/WS 前后向方向解耦；后半锁定 fast/max 候选数量与剪枝、seqlen 分桶（fast 桶 8192 封顶）与 `exact_autotune_seqlen_keys` 精确 key 切换，以及离线生成器的 payload、方向化 CLI 开关、任务网格、SM90 TMA entry 记录。核心结论：生成端写入的过滤字段必须与运行时查找字段端到端对齐，否则调优结果运行时查不到而回退默认值。
