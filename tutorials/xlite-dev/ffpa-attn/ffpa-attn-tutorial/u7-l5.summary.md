本讲是「手写 CUDA 后端与构建系统」单元收尾篇，核心目标是厘清 env.py 中 `ENABLE_FFPA_*` 开关的「构建期 vs 运行期」边界。最关键结论是：docs/env.md 标为「runtime kernel-selection」的开关里，真正逐调用可变、零重编的只有 `stages` 与 `acc` 两个 int 参数（经 `CUDABackend` 携带到前向调用点）；其余全部在编译 `_C` 时由 `env_cuda_cflags()` 翻译成 nvcc `-D` 宏，再经 `launch_templates.cuh` 的 `#ifdef` 与 `getConfigXXX()` constexpr 函数固化成模板参数，改了必须重编。涉及最小模块：env.py 的 `ENV` 类、launch_templates.cuh、ffpa_attn_api.cc 的运行期分发。引入的关键术语：构建期/运行期两条消费路径、MMA 累加器精度三层语义（运行期 `acc` 0=f16/1=f32 做大选，构建期 `FORCE_QK_F16`/`FORCE_PV_F16` 对 fp16f32 入口做混合精度细化，bf16 三层一致强制 f32 acc，两 FORCE 宏互斥为推荐而非 assert）、五类开关分类（MMA 精度/预取·共享/swizzle/persist/流水线·网格，本质都是 SRAM/寄存器/IO 三方权衡）、`env_cuda_cflags()` 末尾的构建期 assert 拦截非法 persist/share 组合。本讲还指出一个真实缺陷：`LAUNCH_GRID_DNHB` 因 `ENBALE` 拼写错误当前失效。需强调这些开关仅手写 CUDA 后端专用，Triton-only 安装无影响。
