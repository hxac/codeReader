本讲解决 OpenOFDM 的 `dot11` 解码器如何真正接入 Ettus USRP N210（Spartan 3A-DSP）FPGA。核心结论有四：其一，N2x0 接收链为 `rx_frontend → ddc_chain → dsp_rx_glue → vita_rx_chain`，`dot11` 应插在 **DDC 之后、VITA RX 之前**，其 `sample_in/sample_in_strobe` 直接对接 DDC 输出的 20 MSPS 基带 I/Q（每 5 拍一对），与既有 32 位 I/Q 时序约定吻合。其二，利用 USRP 官方预留的 **`dsp_rx_glue` 占位模块**（默认直通），通过自定义 Makefile 定义 `RX_DSP0_MODULE=custom_dsp_rx` 等宏驱动其例化，集成仅触及一份 `Makefile.N210R4.custom` 与一个 `custom_dsp_rx.v` 包装，不改动接收链核心 RTL，可选注释掉第二条 RX 链省资源。其三，`verilog/usrp2/setting_reg.v`（配置总线地址认领终端，复位优先、按 strobe 与 my_addr 匹配锁存、只写不读）与 `ram_2port.v`（真双口 RAM 模板，读带一级寄存，需 delayT 错相位对齐）是从 Ettus vendored 的 GPL **平台胶水模块**，与 Apache 主体分属不同目录与许可证。其四，host 经 UHD 的 `set_user_reg` 配置，地址契约 `common_params.v` 的 `SR_*`（3–6），UHD 与 ZPU 固件均无需改动；但因 FPGA 改吐解码字节、不再回传 RF 采样，`rx_samples_to_file` 失效，`byte_out` 如何送回 host 是遗留的输出路径开放问题。关键术语：DDC/DUC、VITA RX、ZPU、UHD、`dsp_rx_glue` 占位、`set_user_reg`、平台胶水模块。
