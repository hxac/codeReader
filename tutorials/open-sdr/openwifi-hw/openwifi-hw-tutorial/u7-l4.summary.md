本讲覆盖 openwifi 六个自研 IP（xpu/tx_intf/rx_intf/openofdm_tx/openofdm_rx/side_ch）的「改源码→重新打包→集成回顶层」流水线。核心结论：六 IP 是带 `component.xml` 的 Vivado IP 核（VLNV 统一 `user.org:user` 前缀），顶层工程引用 `boards/$BOARD_NAME/ip_repo/` 里已打包的核而非散装 `.v`，故改源码后必须重新打包才进 bitstream。编辑单 IP 入口 `create_vivado_proj.sh` 只是「source 环境+调 vivado」的通用壳（最多透传 7 参），真正建工程、生成 `clock_speed.v`、以 append 模式追加 `*_pre_def.v` 的是各 IP 自己的 `<ip>.tcl`（append 有累积陷阱，迭代前需删旧文件）。打包有两套逐行相同的脚本，唯一差异：`package_ip.tcl`（简单，3 参 `part_string/src_dir/ip_dir`、`glob` 全文件、需调用方预置板级文件）vs `package_ip_complex.tcl`（4 参，多 `ip_tcl_filename/BOARD_NAME`，先 `source` IP 的 `.tcl` 当场生成板级文件），`ip_repo_gen.tcl` 主循环统一用 complex 版。集成回顶层是三段接力：`create_ip_repo.sh`（命令行宏写入 `ip_config/*_pre_def.v`、`BOARD_NAME` 取自目录名）→ `ip_repo_gen.tcl`（生成六份共享文件并循环打包；openofdm_rx 作为外部子模块用 `cat >>` 追加 `pre_def.v` 以保留自带定义，对应提交 b6a3231）→ `openwifi.tcl` 建顶层工程。新增源文件时 complex 路径须手改 `.tcl` 写死的 `add_files` 清单。
