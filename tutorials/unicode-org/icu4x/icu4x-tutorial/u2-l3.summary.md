本讲围绕 Locale 的「解析—规范化—补全」三步，覆盖 `icu_locale_core` 的解析器与 `icu_locale` 的规范化器、扩展器。核心结论有四：其一，Unicode 定义三层递进合规级别 **well-formed / valid / canonical**，而解析器只保证 well-formed 加大小写规范化，valid 与 canonical 因依赖 CLDR 数据留到带数据的 crate——这体现「语法层在 core、语义层在数据 crate」的分层。其二，解析靠 `SubtagIterator` 切字节与 `ParserPosition` 位置状态机，大小写规范化实现在各子标签构造函数（如 `to_ascii_lowercase`），而非 canonicalizer。其三，`LocaleCanonicalizer` 实现 UTS #35 Annex C，以**不动点循环**按优先级反复应用 `Aliases` 别名规则直至无修改，复杂地区借 `maximize` 选最优候选。其四，`LocaleExpander` 实现 likely subtags：maximize 按查表优先级补缺、只补不覆盖；minimize 用「删后再最大化验证」删冗余，分偏地区与偏脚本两种倾向。所有操作统一用 `TransformResult`（Modified/Unmodified）报告是否改动；`new_common` 与 `new_extended` 的差别仅在 `likely_subtags_ext` 是否加载。标准流水线为「解析→规范化→补全」。
