本讲讲解 `icu_locale_core` 的两个核心类型。`LanguageIdentifier`（语言标识符）由 `language`/`script`/`region`/`variants` 四个子标签组成，对应 BCP-47 不带扩展的部分；`Locale` 在其上内嵌一个 `id: LanguageIdentifier` 并额外携带 Unicode 扩展（`-u-...`）。二者是严格子集关系，口诀是「拿不准就用 `Locale`」。构造分两条路：`langid!`/`locale!` 宏在编译期完成（受 `const` 求值限制，至多单变体、单 Unicode 关键字），`.parse()`/`try_from_str` 在运行期完成、能力完整、返回 `Result`；解析只保证 well-formed 与大小写规范化，废弃标签替换等 canonicalize 留给 `LocaleCanonicalizer`。两个类型都故意不实现 `Ord`，排序需显式用 `strict_cmp`（字符串序）或 `total_cmp`（结构序）。特殊常量 `und`（未确定）是 `Locale::UNKNOWN`、回退链的最终落脚点。在格式化 API 中，`Locale` 经 `From<Locale>` 转成组件用 `define_preferences!` 宏生成的偏好类型（只挑本组件关心的扩展关键字），最终再压缩成 `Copy` 的 `DataLocale` 供数据管道查找——这是体积/性能取舍的核心。本讲为 u2 入口，承接 u2-l2（子标签类型）、u2-l3（规范化）、u2-l4（回退链）。
