本讲解决 DeepGEMM 的环境与安装问题，承接 u1-l1 的 JIT 哲学：**安装阶段只编译一个薄宿主模块 `_C`，所有重型 CUDA kernel 推迟到运行时 JIT 编译**。依赖分编译期（C++20 编译器、CUDA Toolkit、CUTLASS 与 fmt 两个 git 子模块）与运行时（SM90/SM100 GPU、PyTorch≥2.1、`nvcc`/`nvrtc` 及随包分发的设备头文件）两层；因 JIT 之故，CUDA Toolkit 同时是构建与运行依赖，克隆须加 `--recursive`。`setup.py` 用 PyTorch 的 `CUDAExtension` 把单文件 `csrc/python_api.cpp` 编译成 `deep_gemm._C`，并通过 `CustomBuildPy` 完成三件准备：复制 CUTLASS/CuTe 头文件进包、生成默认 `envs.py`、生成 `_C.pyi`。两个构建入口 `develop.sh`（原地 `setup.py build` + 软链 `.so`，面向开发者）与 `install.sh`（`bdist_wheel` + `pip install`，面向使用者），其 `CachedWheelsCommand` 仅在 `DG_USE_LOCAL_VERSION=0` 时下载预编译 wheel，默认走本地源码构建。仓库根目录 `CMakeLists.txt` 不参与真实构建，仅供 CMake 类 IDE 索引设备代码。关键术语：`_C` 扩展、CUDAExtension、JIT、子模块、C++ ABI 对齐、`envs.py`、版本号（`__version__`+git 短 SHA，当前 2.6.1）。
