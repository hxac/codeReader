本讲拆解 DeepGEMM 的 Python↔C++ 边界与按架构派发这一段宿主侧胶水代码，覆盖三个最小模块。一是 `_C` 扩展模块的 pybind11 注册：`csrc/python_api.cpp` 作为唯一编译入口，用 `PYBIND11_MODULE` 聚合 gemm、attention、mega 等 7 个功能域各自的 `register_apis(m)`，域内以 `m.def` 绑定 C++ 函数、以 `m.attr` 建立别名（如 `fp8_gemm_nt` 即 `fp8_fp4_gemm_nt`）。二是 `apis/gemm.hpp` 等 API 层统一的「布局校验→类型形状校验→early_return→变换 SF→派发」四步范式：其中 early_return 处理 M/N=0、K=0 以及带累加 C≠D 时的预拷贝，让设备 kernel 始终假设 C/D 同址；断言经 `DG_HOST_ASSERT` 抛 `DGException`，由 pybind11 翻译为 Python 异常而非 abort。三是核心派发开关 `device_runtime->get_arch_major()`：返回缓存的 `cudaDeviceProp.major`（9=Hopper/SM90，10=Blackwell/SM100），`fp8_fp4_gemm_nt` 用「架构 + 变换后 SF dtype（SM90 为 float32，SM100 为打包 UE8M0 的 int32）」双条件派发到 `sm90_fp8_gemm_1d1d/1d2d` 或 `sm100_fp8_fp4_gemm_1d1d`，错配即 `DG_HOST_UNREACHABLE`。关键术语：arch_major、early_return、SF（缩放因子）、宿主 Runtime 类与设备 kernel 模板之分、nt 为唯一原生 kernel 而 nn/tn/tt 仅转置转发。派发终点落在宿主 Runtime 函数（如 `sm90_fp8_gemm_1d1d`），构造 GemmDesc、调 `get_best_config` 后由 `LaunchRuntime::launch` 触发 JIT 编译与启动。
