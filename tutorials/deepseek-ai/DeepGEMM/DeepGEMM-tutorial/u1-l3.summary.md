本讲为后续源码精读提供「地图」，核心是建立 DeepGEMM 的目录划分与三层分层架构。仓库顶层五类目录各司其职：`deep_gemm/`（Python 包，含随包分发的设备头文件 `include/`）、`csrc/`（编译进 `_C` 扩展的宿主 C++）、`tests/`、`scripts/`、`third-party/`。代码分为三层：Python 层（用户面对）、宿主 C++ 层（派发与 JIT 基础设施）、设备 CUDA 层（kernel 模板）。引入的关键术语包括「宿主（host，CPU 侧调度）」「设备（device，GPU 侧计算）」「JIT 桥（运行时代码生成把运行时形状固化为编译期常量）」。关键约定：`csrc/jit_kernels/impls/*.hpp`（宿主 Runtime 类，继承 `LaunchRuntime`，提供 `generate_impl`/`launch_impl` 钩子）与 `deep_gemm/include/deep_gemm/impls/*.cuh`（设备 kernel 模板）命名一一对应，是跨层跳转的依据。端到端调用链为：`__init__.py` → `python_api.cpp`（pybind11 注册）→ `apis/gemm.hpp`（校验、SF 变换、按 `device_runtime->get_arch_major()` 派发，`9` 走 SM90、`10` 走 SM100）→ `jit_kernels/impls/*.hpp`（生成 `.cu` 源码 `#include` 设备 `.cuh` 后编译启动）→ 设备 `.cuh`（tensor core 执行）。连接三层的两座桥是 pybind11 与 JIT 运行时编译。
