本讲是 JIT 编译系统单元首篇，回答宿主拿到 GemmConfig 后、到 GPU tensor core 真正运行之间这段运行时编译发生了什么，并立下三对基础概念：宿主（CPU 侧调度代码）与设备（GPU 侧 .cuh 模板）、编译期与运行时、cubin 与 kernel 符号。核心结论有三：其一，设备 kernel 是高度参数化模板，潜在形状组合呈天文数字，故 DeepGEMM 只编译被请求的形状并缓存，安装阶段零重型 CUDA 编译。其二，Compiler::build 是 JIT 大脑，用 (name, signature, flags, code) 经 FNV-1a 算摘要作缓存目录名，依次查内存缓存、编译进临时目录、fsync、整目录原子 rename、写入运行时缓存；多 rank 并发靠「原子整目录 rename 加 fsync」实现无锁安全。其三，KernelRuntime 把 kernel.cubin 加载成可启动 KernelHandle，守「一个 cubin 恰好一个 kernel」契约，默认走 Driver API 库枚举定位唯一符号。基类 LaunchRuntime 用 generate、launch 两钩子统一各 Runtime 类：generate 调子类 generate_impl 生成源码并注入 include 哈希（使头文件改动可靠触发重编译），launch 组装 LaunchConfig 后调子类 launch_impl。供后续承接的关键术语：GemmConfig、signature、FNV-1a digest、include hash、原子整目录 rename、KernelRuntime、KernelHandle、generate_impl/launch_impl、LaunchConfig。
