本讲梳理 DeepEP 遗留 V1 `Buffer` 的设计与初始化，作为理解 V2 `ElasticBuffer` 演进的基准线。V1 用 `num_nvl_bytes` 与 `num_rdma_bytes` 双缓冲分别承载节点内 NVLink 与节点间 RDMA，外加 `low_latency_mode` 切到纯 RDMA 的第三种布局；workspace 与 MoE 计数器（初值 `-1` 作哨兵）为本地资源，不参与对称共享。对称显存分两条路径建立：RDMA buffer 走 **NVSHMEM**（Python 先编排 `NVSHMEM_*` 环境变量并广播 root unique id，C++ `sync` 里 init+alloc+barrier，并借 IBGDA 通道支撑低延迟）；节点内 NVLink buffer 走更轻量的 **CUDA IPC handle**（本 rank `cudaMalloc`+导出、peer `open_mem_handle` 导入、指针表拷入 GPU，`use_fabric`/`allow_mnnvl` 切换 fabric handle 以适配 MNNVL）。三类内核按链路分工：`intranode`（纯 NVLink）、`internode`（RDMA+NVLink 两级）、`low_latency`（纯 RDMA IBGDA）；前两者共享 `num_channels=num_sms/2` 的「两 block 一 channel」模型，后者用固定双缓冲+send/recv phase+hook 实现零占 SM 重叠。V1 内核安装期编译、性能靠人工 `config_map: num_ranks → Config` 查表，用户须两步式调用 `get_dispatch_layout`+`dispatch(config=...)`——这些正是 V2 用 NCCL Gin、解析式 SM/QP、JIT、`EPHandle` 重构掉的痛点。
