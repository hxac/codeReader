本讲深入 DeepEP V2 的通信底座——**NCCL Gin 后端**与**对称内存上下文（`NCCLSymmetricMemoryContext`）**，这是支撑 dispatch/combine/barrier/engram/AGRS 等所有内核的物理基础。核心覆盖四个模块：其一，NCCL Gin 后端以 header-only、复用已有 `ncclComm_t`（不新建通信世界）的轻量设计替代了 V1 的 NVSHMEM，通过 `ncclDevCommCreate` 把 RDMA 队列（QP）直接交给自定义内核，`EP_DISABLE_GIN` 为其排障回退开关。其二，对称内存由 `symmetric.hpp` 用 CUDA VMM（`cuMemAddressReserve`/`cuMemCreate`/`cuMemMap`/`cuMemSetAccess`）分配可 GPUDirect RDMA 的显存，GPU 段设 `gpuDirectRDMACapable`，并能把 GPU 段与 CPU 段（乃至多 rank CPU 段，靠 `pidfd_getfd` 借 FD 跨进程拼接）拼成连续虚拟地址，分纯 GPU / Elastic / HybridElastic 三类。其三，经 `ncclCommWindowRegister`（集合操作、含 bootstrap barrier、`STRICT_ORDERING`）注册窗口后，用 `ncclGetLsaDevicePointer` 取本 rank 基址 `mapped_window_ptr` 与 NVLink peer 基址表 `nvl_window_ptrs`，支撑 `get_sym_ptr` 以「同一偏移量」做跨 rank 寻址直写对端显存——但仅对 NVLink（LSA）域有效，跨节点 scaleout 走 Gin request。其四，`ElasticBuffer` 构造函数将 `[[[ Workspace ] GPU buffer ] CPU buffer]` 装入窗口，关键关系为 `num_sym_bytes = workspace + buffer`、`num_gpu_bytes = workspace + gpu_buffer`、`num_cpu_bytes = cpu_buffer`；workspace 置于窗口最前（= 基址，便于对称寻址）、对齐 2MB（= VMM 粒度，利于 TMA/RDMA）、构造时必须清零（计数器/信号量初值为 0）。
