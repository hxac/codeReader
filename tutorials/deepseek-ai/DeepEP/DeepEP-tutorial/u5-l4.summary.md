本讲承接 u5-l1 直接模式 Dispatch，聚焦 dispatch 主内核跑完后、`recv_x` 交回用户前，host 侧如何确定接收规模。核心矛盾在于：`recv_x` 的行数 `num_recv_tokens` 由所有 rank 的 `topk_idx` 共同决定，host 在内核执行前无法得知，由此派生三种 host 处理模式。

模式一 `do_cpu_sync=True`：内核（`kDoCPUSync=true`）将计数写入映射到 host 的 workspace，host 用 `while` 循环轮询，靠对合函数 `encode_decode_positive`（f(x)=-x-1）让 0 兼任初值与未就绪哨兵、负号位充当就绪标志，超时（默认 300s）抛异常，分配精确大小的 `recv_x`，是训练前向标准路径。模式三无 CPU sync：按 `num_max_tokens_per_rank * num_ranks` 的最坏上界分配，host 永不阻塞、张量形状固定以兼容 CUDA graph，代价是尾部有无效行（需用 GPU 上的 `psum_num_recv_tokens_per_scaleup_rank[-1]` 切片）。模式二 cached handle：路由不变时把首次产出的 `EPHandle` 整体回传，C++ `cached_mode` 分支将 `num_notify_warps` 置 0、`reuse_slot_indices=true`、跳过轮询与布局张量重分配，`num_recv_tokens` 直接取缓存值，是推理解码快路径。cached 与 CPU sync 互斥。引入的关键术语：`EPHandle`、host workspace、`encode_decode_positive`、`do_cpu_sync`、`cached_mode`、`reuse_slot_indices`、`dst_buffer_slot_idx`、`recv_src_metadata`。
