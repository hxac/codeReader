本讲梳理 DeepEP 仓库的目录分层与跨层调用结构。顶层目录按职责划分为 `deep_ep/`（Python 包，内含随包发布的 C++/CUDA 头文件 `include/`）、`csrc/`（安装时编译的 C++/CUDA 源）、`tests/`、`docs/`、`third-party/`，构建主入口是 `setup.py` 而非 `CMakeLists.txt`。核心结论是区分两条编译路径：`csrc/*` 在 `pip install` 时编进 `_C.so`；而 `deep_ep/include/impls/*.cuh`（如 `dispatch.cuh`）作为 header-only 模板，运行时才被 JIT 实例化。同名 `deep_ep` 既指 Python 包又指 C++ include 前缀，最易混淆。

关键术语包括：pybind11 模块 `_C`（由 `csrc/python_api.cpp` 定义，分派给 jit/legacy/elastic 三组 `register_apis`）、JIT 编译、CRTP 基类 `LaunchRuntime<Derived>`、`ElasticBuffer`。`import deep_ep` 时自动执行 `check_nccl_so()` 校验 NCCL 一致性、`init_jit()` 注入库根/CUDA/NCCL 路径。一次 `buffer.dispatch()` 穿过五层：Python 用户 API → `self.runtime.dispatch`（pybind11）→ `ElasticBuffer::dispatch`（host）→ `launch_dispatch` 启动器（generate+build+launch）→ JIT 编译的 `dispatch_impl<...>` GPU kernel。V2 把 SM 数、rank 数、hidden 等烘焙成模板常量以换取极致优化，这是「启动器在 `csrc/kernels/`、真 kernel 在 `deep_ep/include/impls/`」分工的根本原因。
