本讲讲解 DeepEP 中 `ElasticBuffer.dispatch`/`combine` 的 Python 工作流与路由句柄 `EPHandle`，覆盖到 Python 接口层与 C++ host 层边界。dispatch 按 `topk_idx` 把 token 发往目标专家所在 rank，输入 `x` 支持 BF16 单张量与 FP8 `(data, scale)` 元组，返回 `(recv_x, recv_topk_idx, recv_topk_weights, handle, event)`；combine 为其逆过程，把专家输出按 `topk_weights` 加权归约回原 rank，必须依赖 `handle` 提供的 `recv_src_metadata`、`psum_num_recv_tokens_per_scaleup_rank`、`topk_idx`、`do_expand` 等十余个字段驱动反向路由。`EPHandle` 仅持路由元数据（发货单），关键字段还包括 `psum_num_recv_tokens_per_expert`（expand 与非 expand 语义不同）、`dst_buffer_slot_idx`（cached 复用核心）、`num_recv_tokens_per_expert_list`（CPU 侧列表，仅 CPU sync 或 cached 时有值）。由于 `recv_x` 行数运行时才确定，本讲引入三种 dispatch 模式：CPU 同步（精确，训练前向）、cached handle（复用旧布局，推理解码）、最坏情况上界分配（固定形状，兼容 CUDA graph）。combine 输入强制 BF16 且 16 字节对齐，`topk_weights` 形状随 `do_expand` 在 2D/1D 间切换。真正的 GPU kernel 细节留待 U5/U6。
