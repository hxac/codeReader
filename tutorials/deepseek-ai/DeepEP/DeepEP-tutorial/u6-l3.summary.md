本讲围绕 `EPHandle.deterministic_sort`，解决 DeepEP dispatch 输出的非确定性问题。非确定性源自接收端多 warp/多 channel/多 rank 通过 `atomicAdd` 抢占 buffer 槽位的竞态，导致 `recv_x` 行序随运行抖动，破坏逐位复现与 CUDA graph 复用。核心策略是：在 copy epilogue 写完 `recv_x` 后，按**源 token 全局索引**（`recv_src_metadata[:, 0]`）对其重排，得到与到达顺序无关的规范位序。引入的关键机制与术语：`EventOverlap.register_hook_after_wait`——dispatch 用 `functools.partial` 把排序柯里化为无参回调挂在该 hook 上，用户 `current_stream_wait()` 一返回即在计算流上一次性触发排序。两种模式有别：**非 expand** 用单键全排序，对所有依赖顺序的张量（含 `recv_src_metadata` 行、`channel_linked_list` 指针）套用同一排列；**expand** 用「专家优先、组内按源索引」双键（`expert_idx·B + (-B/2 + src_global_idx)`），只重排 expand 数组并改写 `recv_src_metadata[:, 2:]` 的 slot 指针、不动其行序，以保住 combine 的 compact-token 路由表。**cached dispatch** 下跳过排序。无 CPU sync 时将越界行键置为 `int_max` 使其沉末。
