本讲聚焦 `import deep_ep` 触发的同步初始化链：依次为加载持久化环境变量、`check_nccl_so()` NCCL 一致性校验、`init_jit()` 注入工具链路径、暴露 `Buffer`/`ElasticBuffer` 等 API，顺序固定，NCCL 校验失败即 fail fast。核心机制有三：其一，持久化变量（`EP_JIT_CACHE_DIR`、`EP_NCCL_ROOT_DIR` 等）由 `setup.py` 构建期烘焙进动态生成的 `deep_ep/envs.py`，import 时仅作兜底默认，用户运行时 `export` 永远优先。其二，`check_nccl_so` 通过 `/proc/self/maps` 取"运行时加载的 NCCL"，用 `find_nccl_root()/lib/libnccl.so*` 取"DeepEP 链接的 NCCL"，再 `filecmp.cmp(shallow=False)` 逐字节比对，防止同进程双 NCCL 符号；`EP_SUPPRESS_NCCL_CHECK=1` 可跳过。其三，`init_jit` 把库根/CUDA home/NCCL root 三条 Python 探测到的路径经 `_C.init_jit` 注入 C++ 侧 `Compiler`/`KernelRuntime`/`IncludeParser` 的静态变量，为运行时 JIT 编译 nvcc 命令行、`.cubin` 加载、头文件哈希缓存签名做准备。两条探测函数遵循"Python 探测、C++ 使用"的分工：`find_cuda_home` 走环境变量→`which nvcc`→兜底（刻意不复用 PyTorch 实现以避免触发与 fork 冲突的 CUDA 初始化），`find_nccl_root` 走环境变量→扫描 pip 包元数据，适配 NCCL-as-pip-package。本讲只到路径"钉入"，不涉及真正的内核编译（留待 u4）。
