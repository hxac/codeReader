本讲专讲 decode 的调度元数据 kernel——一个单 warp（1 block × 32 线程）kernel，串行扫一遍 batch，一次产出两张表：供主 kernel 的 `DecodingSchedMeta[num_sm_parts]` 与供 combine 的 `num_splits[batch_size+1]` 前缀和。核心目标是负载均衡：以 `payload = ceil_div(total, num_sm_parts) + fixed_overhead` 为配额，把含每条请求固定开销 `fixed_overhead_num_blocks` 的总工作量平均摊到 `num_sm_parts` 个 SM partition，配额不足以整条吞下请求时就在 block 边界切一刀。`DecodingSchedMeta` 含七个字段——`begin/end_req_idx`（请求范围，闭）、左闭右开的 `begin/end_block_idx`、`begin_split_idx`（accumulate 槽位起点）、`is_first/is_last_req_splitted`——完整描述一张 partition 的工单；单请求 partition 会把首尾标志合并为「或」。`num_splits` 是前缀和，在请求被完整吞下时累加 `now_n_split_idx+1`，使主 kernel 的写槽位与 combine 的读区间经同一张表对齐。空序列先按 1 个 block 计入配额、再于 `end_block_idx` 处用 `seqlens_k==0` 纠正为 0，sparse 空 topk 被强改为 1，末尾 `FLASH_DEVICE_ASSERT` 兜底保证分完且无半截请求。dense 与 sparse 共用此 kernel，差异仅在 `cur_s_k` 来源（真实 `seqlens_k` vs `topk`/`extra_topk`）与 `fixed_overhead`、`num_sm_parts` 取值。
