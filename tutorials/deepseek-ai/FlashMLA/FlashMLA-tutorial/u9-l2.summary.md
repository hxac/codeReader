本讲从「扩展视角」梳理给 FlashMLA 新增一个能力（如 head_dim 或 feature）需改的接线层，不重讲 kernel 算法。核心结论是「三道关卡」：每个能力取值须同时通过运行时校验（TORCH_CHECK）、编译期化（DISPATCH 宏）、构建注册（instantiations/*.cu + setup.py），漏任一道即编译失败、运行崩溃或能力静默失效。引入的关键术语包括：四个 DISPATCH 宏（DISPATCH_HEAD_DIM / MODEL_TYPE / NUM_HEADS / BOOLEAN_FLAG）、ModelType 枚举（V32 / MODEL1）、DecodeFeatures / FwdFeatures 能力位、由 DECLARE_SUPPORTED_FEATURES 声明的支持集合，以及「宏生成实例化点 → instantiation 兑现 → setup.py 编译」的模板注册链。核心不对称在于：decode 路径把 head_dim 折进 ModelType（经 DISPATCH_MODEL_TYPE 编译期化，并绑定 FP8 字节布局），prefill 路径则直接用 DISPATCH_HEAD_DIM 把 d_qk 编译期化、不涉及 ModelType，因此同一条「加 head_dim」诉求在两路径上改动面完全不同。最后给出新增 head_dim 的七步端到端清单：枚举 → 宏 → 接口层四处 → Impl features → 派发表 → kernel+instantiation → setup.py，强调大部分工作是接线而非写算法。
