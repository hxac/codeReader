本讲打开 FlashMLA 从 Python 到 CUDA kernel 的完整调用链，是后续 kernel 深入讲义的全局地图。整条链分四层：Python 包装函数（薄壳，只做轻量校验与 sparse/dense 二分派发，真正重活下放 C++）→ pybind 绑定（`api.cpp` 用 `PYBIND11_MODULE` 注册 5 个绑定）→ C++ 接口函数（`csrc/api/*.h`）→ kernel 命名空间（`sm*::run_*`）。`flash_mla.cuda` 扩展对外暴露 5 个绑定：`dense_decode_fwd`、`sparse_decode_fwd`、`sparse_prefill_fwd`、`dense_prefill_fwd`、`dense_prefill_bwd`，每个绑定一一对应一个接口函数。接口函数统一遵循「架构检查→张量校验→参数装配→派发启动」套路，但派发有两种风格：dense 路径（仅 SM90 decode / SM100 prefill）用直接调用，sparse 路径用基于 feature 集合的 ImplBase 派发器（实现＝支持的 feature 子集）。命名空间编码归属：`sm90::`/`sm100::` 架构专用，`smxx::decode::` 为两架构共用的解码辅助 kernel。关键结论：所有 decode 路径都是「sched_meta → 主 kernel → combine」三段式，prefill 路径无 combine，dense prefill 直接走 CUTLASS 的 `FMHACutlassSM100*Run` 入口。引入的核心术语：pybind11、CUDAExtension、接口函数、kernel 命名空间、派发、Arch、ImplBase、DISPATCH_* 宏、sched_meta 复用模式，供后续讲义承接。
