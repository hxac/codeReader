本讲拆解 3FS 客户端进程内部的「一捆子客户端」：`MgmtdClientForClient`（控制面，自动刷新 `RoutingInfo`、续客户端会话、收配置热更新）、`MetaClient`（元数据 RPC，且内部以 `shared_ptr` 同时持有 mgmtd 与 storage）、`StorageClient`（数据 RPC）、`CoreClient`（底层 CoreService），组装依赖链为 mgmtd → storage → meta。路由信息以「监听者 + `atomic_shared_ptr`」无锁热更新，`StorageClient` 取快照使用。数据热路径上 meta 不参与：`batchRead` 走「校验范围 → `splitReadIOs` 拆分巨型 IO → `selectRoutingTargetForOps` 选 target → `groupOpsByNodeId` 按节点合并成受 `max_batch_size/bytes` 约束的批次 → 并发发送」四段流水线，配「全局 + 单服务器」两级信号量限流，发包前再做路由版本校验。关键术语：`RoutingInfo`、`SlimChainInfo`/`servingTargets`（前缀连续的 SERVING 副本）、`TargetSelectionMode`（默认 `LoadBalance` 按累计 IO 数把读流量在链内副本间均衡，并支持失败感知 failover）、`RoutingTarget`、`IOBuffer`（`registerIOBuffer` 做 pin + `ibv_reg_mr`，`ReadIO.data` 须落在其范围内，storage 用 RDMA Write 单边回填、回包仅含校验和）。偏移换算链为 `getChunkId`（纯算术）→ `getChainId`（查本地路由解出 `ChainRef`/`ChainId`）→ 选 target → RDMA 回填，全程零拷贝。
