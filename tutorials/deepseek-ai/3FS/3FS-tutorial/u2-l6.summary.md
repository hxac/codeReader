本讲讲 3FS 与 FoundationDB（FDB）打交道的胶水层，是 u4（meta）、u3（mgmtd）的前置；FDB 是元数据与路由的唯一事实来源，使 meta/mgmtd 服务无状态。

三个最小模块：① **FDB 封装**：`fdb::DB`/`fdb::Transaction` 把 FDB 的 C API 包成 RAII 对象，核心是 `Result::toTask` 把 `FDBFuture*` 经回调+`Baton` 变成可 `co_await` 的协程；`FDBContext` 在进程级 `selectAPIVersion`→`setupNetwork`→起线程跑 `runNetwork`，是事务能跑通的前提，`readonly` 在 `commit` 入口拦截防误写。② **事务抽象**：`IKVEngine`→`IReadOnlyTransaction`/`IReadWriteTransaction` 解耦上层（测试可换 `MemKV`）；核心语义为快照读 `snapshotGet` 不进冲突范围、冲突读 `get`/`addReadConflict` 进读冲突范围、`set`/`clear` 进写冲突范围，单个 key 经 `TransactionHelper::keyAfter` 扩成左闭右开 range。③ **重试策略**：`FDBTransaction::convertError` 把 `fdb_error_t` 映射为 `TransactionCode`（`1xxx` 段，`kConflict=1001`、`kMaybeCommitted=1006`、`kTooOld=1003`），`TransactionHelper::isRetryable` 判重试性；`FDBRetryStrategy::onError` 优先把退避交给 `fdb_transaction_on_error`，否则用带抖动的指数退避兜底；`OperationDriver::run` 是 meta 侧重试主循环，叠加 `maxRetryCount` 与 `deadline_` 双保险，`maybe_committed` 受 `retryMaybeCommitted` 控制，仅幂等操作（`Idempotent`）允许重试。
