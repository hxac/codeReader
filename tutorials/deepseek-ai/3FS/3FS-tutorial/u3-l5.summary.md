本讲讲解 mgmtd 侧的 Target 状态机与故障检测，核心结论是：mgmtd 不主动探测 SSD 健康，只做两件事——被动收 storage 心跳获取 target 的 local 状态，再周期性地依据最新 local 状态对每条链重算 public 状态并下发。Target 持双状态：对外下发的 public 状态（SERVING/LASTSRV/SYNCING/WAITING/OFFLINE 五种，读写语义各异）和仅存 mgmtd 内存的 local 状态（UPTODATE/ONLINE/OFFLINE 三种）。local 状态是状态机的触发事件（输入），public 状态是输出结论；故障检测由 MgmtdHeartbeatChecker 在心跳超时（heartbeat_fail_interval，默认 60s）时把 local 状态翻成 OFFLINE 实现。重算由 generateNewChain 完成：按旧 public 状态分桶、顺序处理，用「桶里已放过什么」隐式表达前驱状态，并以「有 SERVING 则清空 LASTSRV」收尾，保证链内 public 状态呈固定有序布局，该布局由测试 validatePs 的多条不变量钉死。关键术语包括：LASTSRV（链上所有 serving 副本全死时的最后服务者，链必须等其恢复，rotateLastSrv 是以丢数据为代价的应急手段）、sync-done、full-chunk-replace（被当作会改变 local 状态的外部事件，其数据恢复机制见 u5-l5）。每次重算若改变 public 状态，chainVersion 单调 +1，经 FDB 持久化、写回内存、推进 routingInfoVersion 后下发至 client/storage/meta。
