本讲是 storage 服务单元的开篇，建立服务的总览与启动地图。storage 是 3FS 最重的进程，既扛数据面 RPC 又直接驱动 SSD，被组织成「四层洋葱」：进程入口 main 经两阶段启动骨架拉起 StorageServer（net::Server 子类），其下是 Components——所有运行时对象（StorageOperator、StorageTargets、TargetMap、BufferPool、转发器与后台 Worker）的聚合容器，再用一个引用拿全部依赖以避免网状耦合；最内层是 StorageOperator 与各 Worker。启动严格发生在 beforeStart 钩子：注册 StorageService（RDMA，8000）与 CoreService（TCP，9000）两个服务并挂协程池分流器，随后按 Components::start 固定顺序拉起对象，其中 waitRoutingInfo 必须排在数据恢复与对外服务之前。StorageOperator 是所有数据面 RPC 的中央协调者：本身不直接碰磁盘，而是协调 TargetMap（路由与版本校验）、ReliableUpdate/ReliableForwarding（CRAQ 链式写）、BufferPool、StorageTarget，并按 IB 设备用信号量限流；读写流量经 readPool/updatePool/syncPool/defaultPool 四个隔离协程池分流。后台 Worker 分空间回收、诊断、数据恢复三类，与请求路径通过共享的 StorageTarget 间接协作。StorageTarget 是「target」在单机的化身，落盘执行在 Rust chunk_engine（FFI）与旧 C++ ChunkStore 间二选一。本讲引入的关键术语：target、chunk、CRAQ、Components、StorageOperator、TargetMap、StorageTarget、各 Worker、协程池分流，均由后续讲义（读路径、写路径与 CRAQ、链路由与版本校验、数据恢复、chunk engine）承接。
