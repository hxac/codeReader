本讲讲解 Bedrock 用 TCL 驱动 Xilinx Quad MGT（多千兆收发器，本工程为 Kintex-7 上的 GTX）的配置流程，核心是「文本化取代图形向导」，使 MGT 配置可 diff、可纳入 CI。它呈现三段式结构：入口脚本 `gtx_comms_top.tcl` 声明「Quad 0 各通道跑什么协议」（GT0=以太网、GT1=ChitChat）；生成引擎 `mgt_gen.tcl` 是纯过程库，其核心过程 `add_gt_protocol`（真实签名为 7 参：gt_type、config_file、quad、gt、en8b10b、endrp、pll_type）调 `gen_ip` 经 Vivado API 批量生成 GT IP 实例，并通过 `verilog_define`（等价 `-D` 宏）开一组 `Q{quad}_GT{gt}_*` 开关，把配置翻译成编译期宏；宏驱动 Verilog 包装器 `qgt_wrap.v` 借 `Q_REDEFINE` 与 `GTi_PORTS` 宏让同一份 stub 被 include 4 次、按 Quad 展开成不同端口表与实例。关键术语：Quad（四通道组，共享 REFCLK 与公共 PLL）、CPLL（通道 PLL，低速链路用）与 QPLL（公共 PLL）、`en8b10b` 开关（0=外部 8b/10b 编码，端口为 20 位原始比特；1=内部编码，端口为 16 位数据加 charisk）。顶层 `comms_top.v` 将速率（1.25/2.5 GBd）与编码都不同的两条协议装进同一 Quad，靠 CPLL 输出分频 4→2 实现速率翻倍，并经 `eth_gtx_bridge` 暴露 Local Bus。控制平面由手写（非 newad.py 生成）的 `comms_top_regbank.v` 担任，采用 1 拍寄存读的固定延迟读，让 Host 经以太网 UDP→localbus 控制/观测 ChitChat，未命中地址返回 `0xdeadf00d`。该设计是后续工程集成实战（u7-l4）的前奏。
