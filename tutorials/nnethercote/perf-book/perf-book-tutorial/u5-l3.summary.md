本讲围绕 perf-book 的「Wrapper Types」与「Logging and Debugging」两章展开。核心结论有四条：其一，包装类型（`RefCell`/`Mutex` 等）把无法在编译期证明的不变量推迟到运行期维护，因此每次访问都要付一笔固定开销 c——`Mutex` 涉及原子 CAS 与守卫析构，`RefCell` 较轻但仍非零；若记一次直接访问为单位开销，对 N 个独立包装值各访问一次的开销近似 N·c。其二，当多个包装值总是成组访问时，可把它们合并进同一个包装类型（如把两个 `Arc<Mutex<u32>>` 合为一个 `Arc<Mutex<(u32,u32)>>`），将开销从约 N·c 降到约 c，并附带原子快照与缓存局部性收益；但若各值常被独立并发访问，合并会加大锁粒度反而更慢，故是否合取决于访问模式。其三，减小单次 c 的另一条路是换 `parking_lot` 等替代实现，但须先测量。其四，日志/调试代码的隐藏开销更常来自「喂给它的数据收集代码」而非打印本身，日志未启用时不应为它做多余工作；`assert!` 始终运行、`debug_assert!` 仅在 dev 运行，非安全所必需的热点断言可降级为 `debug_assert!` 以在 release 抹掉检查，而挡住 UB 的安全必需断言绝不可降级。关键术语：包装类型、守卫、固定开销 c、合并包装值、数据收集开销、`debug_assertions` profile 开关。全程贯穿「先测量再优化」纪律。
