# Parallelism——并行与 SIMD

## 1. 本讲目标

前面六讲（u2–u5 及 u6-l1、u6-l2）讨论的优化，本质都是**让一个执行流（一个线程、一个核心）把活干得更快**：消除一次分配、消掉一次边界检查、让编译器自动向量化一段循环。本讲换一个维度——**让多个执行流同时干活**，也就是**并行（parallelism）**。

perf-book 的「Parallelism」章开头用三句话点明了它的全部主张：

> Rust provides excellent support for safe parallel programming, which can lead to large performance improvements. There are a variety of ways to introduce parallelism into a program and the best way for any program will depend greatly on its design. […] an in-depth treatment of parallelism is beyond the scope of this book.

也就是说，perf-book 对并行**只给路线、不给细节**——它明确把深入讲解排除在书外，转而指向三个外部资源（`rayon`、`crossbeam`、*Rust Atomics and Locks*）和一篇 SIMD 综述博客。这是本书「重广度轻深度（breadth over depth）」风格（见 u1-l1）在并行主题上最典型的体现：并行这个主题太依赖具体程序的设计，写「通用最佳实践」几乎没有意义，所以作者选择只树一块路标。

学完后你应当能够：

- 说清**并发（concurrency）与并行（parallelism）的区别**，以及 Rust「安全并行」承诺对性能意味着什么。
- 知道进入线程并行的两个主要入口：**`rayon`（数据并行、工作窃取线程池）**与 **`crossbeam`（底层并发构件）**，以及 *Rust Atomics and Locks* 这本书的定位。
- 理解**并行的加速比上限由程序设计决定**——能用 Amdahl 定律把这条直觉讲成一条公式，并据此判断一段串行代码改并行后「最多能快多少」。
- 理解**并行必然引入同步，而同步原语（`Mutex`/`Arc` 等）的每次访问都要付一笔固定开销**，这会把 u5-l3 讲过的「包装类型开销」问题在多线程下放大成**锁竞争（contention）**。
- 了解 **SIMD（细粒度数据并行）** 作为另一种并行形态的现状，并理解它与 u5-l4 讲过的「看机器码 / 自动向量化」之间的承接关系。

一句话定位：本讲不教你「写出最快的并行程序」，而是教你**读懂 perf-book 这块路标**——它指向哪里、为什么这样选、以及在你自己的程序里如何用前几讲建立的工具（剖析、基准测试、看机器码）去决定「要不要并行、怎么并行」。

## 2. 前置知识

本讲默认你已读过：

- **u5-l3 Wrapper Types 与日志/调试的性能代价**：这是本讲的直接前置（依赖关系 `depends_on: u5-l3`）。本讲的「原语与锁资源」模块完全建立在其结论之上——`Mutex`/`RefCell` 等包装类型每次访问都付一笔固定开销 \(c\)，多个包装值成组访问时可合并。本讲会把它推进到**多线程下的锁竞争**这一更严重的场景。
- **u2-l1 Benchmarking**：本讲的综合实践要求你**用基准测量并行的加速比**，需要会建可重复的基线、理解 wall-time 的方差。
- **u2-l2 Profiling**：判断「该不该并行」需要先剖析热点，确认热点确实是可以并行的工作（如一个可拆分的循环），而非被串行资源（如一次只能一个人写的文件、一把全局锁）卡住的代码。
- **u5-l4 Machine Code**：本讲「SIMD」模块直接承接 u5-l4 讲过的 `core::arch` intrinsics、`-C target-cpu=native` 自动向量化、以及「手写 SIMD 前先看机器码」的纪律。

补充两个本讲要用、但前面没正式给过的概念：

- **并发（concurrency）vs 并行（parallelism）**：并发是**程序的结构**——一个程序被组织成可以独立推进的多个任务（哪怕在单核上靠时间片轮流跑）；并行是**执行的方式**——这些任务真正在多个核心上同一时刻同时跑。Rust 的所有权与 `Send`/`Sync` 机制让你能写出**数据竞争不可能发生**的并发程序，而把它跑在多核上就自动得到安全的并行。本讲关心的是「并行带来的性能提升」，因此聚焦并行。
- **`Send` 与 `Sync`**：这是 Rust 保证线程安全的两个核心标记 trait。`T: Send` 表示 `T` 的所有权可以安全地**转移到**另一个线程；`T: Sync` 表示 `&T` 可以安全地**被多个线程共享**。`Mutex<T>` 把 `T` 变成 `Sync`（于是可跨线程共享可变状态），`Arc<T>` 提供线程安全的引用计数（于是可被多线程共同拥有）。理解它们就理解了为什么并行代码里到处是 `Arc<Mutex<...>>`。

> 术语对照：本讲的「锁竞争（contention）」指多个线程同时争抢同一把锁、被迫排队串行执行的现象；「数据并行（data parallelism）」指对一批**互相独立**的数据做**同一种**运算（如「对数组每个元素求平方和」），这是最容易并行、也最常被 `rayon` 接管的形态。

本讲按 perf-book 这块路标的指向，拆成四个最小模块：

### 4.1 并行的价值与「最佳方式取决于程序设计」

#### 4.1.1 概念说明

perf-book 这一章的开场，是全书对某项技术给出的**最高单句评价之一**——Rust 对**安全并行编程**的支持是「excellent（卓越）」的，它能带来「large performance improvements（巨大的性能提升）」。

这里的「安全」二字是关键。在 C/C++ 里，多线程共享可变数据是**未定义行为（UB）的高发区**——数据竞争（data race）不会编译报错，只会在运行时给出难以复现的错误结果。Rust 用所有权系统加上 `Send`/`Sync` 标记 trait，把**数据竞争在编译期就变成编译错误**。这意味着：在 Rust 里，一段能编译通过的并行代码，其正确性比 C/C++ 的同等代码有强得多的保障，你可以放心地把串行代码改写成并行，而不必整日提心吊胆地与微妙的并发 bug 博斗。**这种「安全」本身就是性能优势**——它降低了并行的工程成本，使「为热点引入并行」变得可行。

但 perf-book 紧接着泼了一盆冷水：

> the best way for any program will depend greatly on its design

这句话是本讲的纲领。它说的不是「并行很难」，而是**「不存在通用的『最佳并行方式』」**。同样一个程序，把它的一个循环并行化、把它的整个流水线并行化、用线程池、用 actor 模型、用 SIMD——这些方案哪个最优，取决于程序**天然的结构**（哪些工作互相独立、数据依赖是什么形状、瓶颈是 CPU 还是内存带宽还是 I/O）。所以 perf-book 选择不写细节：在不知道你程序长什么样的前提下，任何「具体最佳实践」都是空头支票。

> 待确认：如果你的程序瓶颈是磁盘 I/O 或网络，给它套多线程往往**不会**带来「large performance improvements」，因为瓶颈不在 CPU。是否值得并行，必须先用剖析（u2-l2）确认热点是 CPU 计算型的。

#### 4.1.2 核心流程

「最佳方式取决于程序设计」这条直觉，有一条硬邦邦的数学依据——**Amdahl 定律**（u6-l1 已为「加速」引入过它的通用形式 \(S=1/((1-p)+p/s)\)）。把加速来源具体化为「N 个并行核心」，就得到并行加速比上限：

\[
S(N) = \frac{1}{(1-p) + \dfrac{p}{N}}
\]

其中：

- \(p\) 是程序中**可被完美并行化**的工作占比（\(0 \le p \le 1\)）。
- \(1-p\) 是**必须串行**的工作占比。
- \(N\) 是可用的核心数。
- \(S(N)\) 是相对单核串行的最大加速比。

这条公式直接量化了「取决于程序设计」：

1. **\(p\) 是设计出来的**。一个循环里每一步都依赖上一步的累加结果，\(p\) 接近 0；一批互相独立的元素各算各的，\(p\) 接近 1。你能不能把 \(p\) 做大，是算法与数据结构（见 u6-l1）层面的问题，不是「调一个并行参数」能解决的。
2. **即使有无限多核心**（\(N \to \infty\)），加速比也封顶于 \(1/(1-p)\)。若程序有 10% 的串行部分（\(p=0.9\)），则无论你有多少核心，最多只能快 10 倍。这就是为什么「消灭串行段」比「堆核心」重要得多。
3. **真实加速通常低于 \(S(N)\)**。Amdahl 只算「纯并行收益」，没算并行引入的**开销**——线程创建/调度、工作窃取、以及最致命的**同步（锁）开销**（4.3 节）。所以 \(S(N)\) 是天花板，实测往往更低。

把决策过程写成伪代码：

```
1. 剖析程序，定位热点（u2-l2）
2. 判断热点工作是否「互相独立」：
   - 是 → p 大，并行潜力高，进入第 3 步
   - 否 → 先重构算法/数据结构把依赖解开（u6-l1），p 提不上去就别并行
3. 选并行形态：
   - 一批数据做同一种独立运算 → 数据并行（rayon par_iter / SIMD）
   - 多个不同任务可同时推进 → 任务并行（线程池 / crossbeam）
4. 改写后用基准测试（u2-l1）测真实加速比，与 Amdahl 天花板对照
5. 若实测远低于天花板 → 怀疑同步开销/锁竞争（4.3）/ 内存带宽墙
```

#### 4.1.4 代码实践

这是一个**源码阅读 + 推导型实践**，目标是让你亲手算一遍 Amdahl 天花板，建立「能不能并行、最多快多少」的直觉。

1. **实践目标**：用 Amdahl 定律判断一段代码的并行上限，避免盲目并行。
2. **操作步骤**：
   - 选一段你熟悉的、含循环的串行热点（比如一个对 1000 万个元素求平方和的循环）。
   - 估计它的 \(p\)：这个循环里有多少工作是「互相独立、可拆分」的（多数逐元素运算 \(p \approx 1\)），有多少是必须串行的（比如最终把所有结果累加成一个值——累加本身串行，但占比极小）。
   - 取你的机器核心数 \(N\)（用 `nproc` 命令查），代入公式算 \(S(N)\) 和 \(N \to \infty\) 的上限 \(1/(1-p)\)。
3. **需要观察的现象**：你会发现，即便 \(p=0.99\)、\(N=8\)，加速比也只有约 7.5 倍而非 8 倍；而 \(1/(1-p)=100\) 这个「无限核心」上限，说明串行那 1% 是真正的瓶颈。
4. **预期结果**：你会得到一个**「这段代码最多能快几倍」的数字**，作为后面综合实践里实测加速比的对照标尺。若实测明显低于这个数，说明并行开销（同步、调度）吃掉了收益。
5. 本步骤为纯推导，无需运行命令。

#### 4.1.5 小练习与答案

**练习 1**：一个程序有 5% 的工作必须串行（\(p=0.95\)），在 16 核机器上，理论最大加速比是多少？加到 1024 核呢？

**答案**：
\(S(16) = 1/(0.05 + 0.95/16) = 1/(0.05 + 0.059) \approx 9.17\) 倍。
\(S(1024) \approx 1/(0.05 + 0.00093) \approx 19.6\) 倍。
核心从 16 加到 1024（64 倍），加速比只从 9.2 涨到 19.6（约 2.1 倍）——这就是串行段 5% 的统治力。

**练习 2**：为什么 perf-book 说「最佳并行方式取决于程序设计」，而不是直接推荐「用 `rayon` 就对了」？

**答案**：因为并行收益的上限 \(S(N)\) 由可并行占比 \(p\) 决定，而 \(p\) 是程序**结构**的属性，不是某个库能改变的。`rayon` 只能把已经是「互相独立」的工作分摊到多核；若程序的工作天然串行依赖（\(p\) 小），任何并行库都救不了，反而引入的同步开销会让它更慢。所以选并行方案前，必须先理解自己程序的数据依赖形状——这正是「取决于设计」的含义。

### 4.2 线程并行：rayon 与 crossbeam

#### 4.2.1 概念说明

perf-book 把**线程并行（thread-based parallelism）**的入口明确指向两个 crate：

> the documentation for the [`rayon`] and [`crossbeam`] crates is a good place to start.

理解这两个 crate 的分工，是本模块的全部内容。

**`rayon`** 是 Rust 生态里**数据并行**的事实标准。它的核心是一个**工作窃取（work-stealing）线程池**：启动时创建一组（默认等于核心数的）常驻工作线程，当你调用 `par_iter()` 时，它把迭代工作切成许多小块，分发给这些线程；空闲线程会从忙碌线程的队列尾部「窃取」工作，从而自动实现负载均衡。对用户而言，把串行的 `iter()` 换成 `par_iter()`（前提是闭包满足 `Send + Sync` 的要求），就完成了最常见的「数据并行」改造。它的设计哲学是 **fork-join + 数据不共享**：每个线程处理数据的不同部分，最后用归约（`sum`/`reduce`）合并结果，**几乎不需要锁**——这恰好避开了 4.3 节将要讲的锁竞争。

**`crossbeam`** 提供的是更**底层的并发构件（building blocks）**：比标准库更强/更安全的**作用域线程（scoped threads）**、多种**通道（channels）**、无锁队列、基于 epoch 的内存回收等。如果说 `rayon` 是「拿来就能并行一个循环」的高层糖，`crossbeam` 就是「自己设计并发结构」时要用的砖块。perf-book 把两者并列推荐，是因为它们覆盖了线程并行的两个抽象层次：要么用 `rayon` 的现成模式，要么用 `crossbeam` 自己搭。

perf-book 还点了一本书——[*Rust Atomics and Locks*](https://marabos.nl/atomics/)（Mara Bos 著），作为「想深入理解原子操作与锁」的权威读物。它讲的是 `std::sync::atomic`、`Mutex`、`RwLock` 这些**原语（primitives）**的底层原理与正确用法——这正是 4.3 节要讨论的「同步代价」的理论根基。

#### 4.2.2 核心流程

`rayon` 数据并行的典型工作流（伪代码）：

```
串行:
    let total: i64 = data.iter().map(|x| x * x).sum();

并行:
    use rayon::prelude::*;
    let total: i64 = data.par_iter().map(|x| x * x).sum();
```

执行过程：

1. `par_iter()` 把 `data` 切成若干块（按线程数）。
2. 工作窃取线程池把块分发给各核心的工作线程。
3. 每个线程对自己的块独立执行 `map`（计算平方）——**互不干扰、无锁**。
4. `.sum()` 触发**归约（reduction）**：各线程先算出局部和，再合并成全局和。合并这一步是唯一需要协调的地方，`rayon` 内部用树形归约把 \(O(N)\) 的串行合并降到 \(O(\log N)\)。

关键点：步骤 3 占了几乎全部计算量（\(p \approx 1\)），且完全并行；步骤 4 的合并是串行段（\(1-p\) 很小）。这正符合 4.1 节 Amdahl 公式里「\(p\) 大、\(1-p\) 小」的理想形态，所以这类数据并行往往能拿到接近线性的加速比。

**什么时候 `rayon` 帮不上忙**（即 4.1 说的「取决于设计」）：

- 循环里每一步都**依赖上一步**（如递推 `fib`、有跨迭代的累加依赖）——工作不独立，`par_iter` 没法拆。
- 各线程需要**频繁读写同一份共享可变状态**（如往一个共享 `HashMap` 插入）——必须加锁，锁竞争吃掉并行收益（见 4.3）。
- 瓶颈是 **I/O 或内存带宽**而非 CPU——多线程不会让磁盘转得更快，反而增加争用。

#### 4.2.4 代码实践

这是本讲的**主实践**，完成 perf-book 规定的任务：把一个串行 `for` 循环用 `rayon` 的 `par_iter` 改写为数据并行版本，用基准测试对比加速比。以下为**示例代码**（书稿中没有，是本讲为练习新写的）。

1. **实践目标**：亲手把一段串行数据并行代码改造为 `rayon` 并行版，测量加速比，并与 4.1 算出的 Amdahl 天花板对照。
2. **操作步骤**：
   - 新建一个 Cargo 项目：`cargo new rayon-demo && cd rayon-demo`。
   - 在 `Cargo.toml` 加入 `rayon`（示例）：
     ```toml
     [dependencies]
     rayon = "1"
     ```
   - 在 `src/main.rs` 写两个版本（示例代码）：
     ```rust
     use rayon::prelude::*;

     fn serial(data: &[i64]) -> i64 {
         data.iter().map(|x| x * x).sum()
     }

     fn parallel(data: &[i64]) -> i64 {
         data.par_iter().map(|x| x * x).sum()
     }

     fn main() {
         let data: Vec<i64> = (0..20_000_000).collect();
         // 用 println 打印，防止编译器把整个计算优化掉
         println!("serial   = {}", serial(&data));
         println!("parallel = {}", parallel(&data));
     }
     ```
   - 用 `cargo build --release` 构建（务必 release，见 u2-l3）。
   - 用 `hyperfine`（u2-l1 引入过）对比加速比。由于 `main` 同时跑两版，可拆成两个二进制，或更简单地用 `std::time::Instant` 各测一次：把 `serial`/`parallel` 各放进一个独立的 bin，用 `hyperfine './serial-bin' './parallel-bin' --warmup 3` 对比。
3. **需要观察的现象**：
   - 两个版本结果应相同（`sum` 相等），证明并行没破坏正确性——这是 Rust「安全并行」的体现。
   - 并行版 wall-time 应明显短于串行版。
   - 实测加速比通常**小于**核心数（如 8 核上往往只有 5–7 倍），差额来自归约合并、线程调度与数据搬运开销。
4. **预期结果**：在 8 核机器上，对这个「平方和」负载，加速比一般落在 4–7 倍区间。把它和你在 4.1.4 用 Amdahl 算出的天花板对照，验证「实测 < 天花板」的规律。**待本地验证**：具体数字依赖你的核心数与内存带宽，请以本地实测为准。
5. 思考题（任务要求）：把数据量从 2000 万改成 200 万，加速比会变大还是变小？为什么？（提示：并行开销是相对固定的，数据量越小，开销占比越大，加速比越低——这也是「取决于设计/负载」的一个例证。）

#### 4.2.5 小练习与答案

**练习 1**：`rayon` 的 `par_iter().map(f).sum()` 为什么几乎不需要锁？

**答案**：`map` 阶段每个线程只处理数据的一个**不相交的子集**，写的是各自的局部结果，没有共享可变状态，自然无需锁。唯一需要协调的是 `sum` 归约，而 `rayon` 用树形归约 + 原子/局部合并把协调降到最低。这种「分而治之、最后归约」的模式，正是数据并行能避开锁竞争（4.3）的根本原因。

**练习 2**：为什么 perf-book 同时推荐 `rayon` 和 `crossbeam`，而不是只推一个？

**答案**：它们处在不同抽象层。`rayon` 是「拿来即用」的高层数据并行框架（一个 `par_iter` 就并行化循环），适合典型的可拆分工作；`crossbeam` 是「自己搭」的底层构件（作用域线程、通道、无锁结构），适合 `rayon` 的固定模式覆盖不了的并发结构（如流水线、生产者-消费者、复杂任务图）。两者互补，按你的并发结构选层——这又印证了「取决于程序设计」。

### 4.3 原语与锁资源：并行引入的同步代价

#### 4.3.1 概念说明

本模块承接 u5-l3（依赖前置）。u5-l3 讲过：`Mutex`/`RefCell` 等**包装类型**把无法在编译期证明的不变量推迟到运行期维护，因此**每次访问都付一笔固定开销 \(c\)**；多个包装值成组访问时，可合并进同一个包装类型，把开销从约 \(N \cdot c\) 降到约 \(c\)。

到了**并行**场景，这个问题会**显著放大**，从「每次访问慢一点」升级为「锁竞争拖垮加速比」。原因在于 `Mutex` 的本质：它保证**同一时刻只有一个线程能进入临界区**，其余线程必须**阻塞等待**。于是：

- **单线程**下，访问 `Mutex` 付的开销 \(c\) 是「锁/解锁的原子操作 + 守卫析构」这一固定成本。
- **多线程**下，若多个线程**同时**抢同一把锁，它们会被**强制串行化**——线程 B 必须等线程 A 释放锁才能进入。这段等待时间完全不在 Amdahl 公式的「并行收益」里，它直接**增大了串行段 \(1-p\)**，把加速比天花板压低。这就是**锁竞争（contention）**：锁越被争抢，并行越退化为串行。

> 待确认：锁竞争的严重程度**完全取决于访问模式**——这正是 u5-l3 末尾「Whether or not this helps performance will depend on the exact access patterns」那句、也是 4.1 节「取决于程序设计」那句的具体落点。

perf-book 用 `wrapper-types` 章的 `Arc<Mutex<u32>>` 例子（见下）演示的虽然是**单线程视角**下的「合并包装值减开销」，但这个例子在并行下有更强的版本：**合并锁能减小竞争粒度**。

#### 4.3.2 核心流程

把「同步代价随并行放大」写成直觉模型：

1. **独立可并行的工作**（4.2 的 `rayon` 模式）：线程各干各的，无共享、无锁，\(p \approx 1\)，加速比接近线性。
2. **共享可变状态 + 细粒度锁**：每个共享值一把锁（如两个 `Arc<Mutex<u32>>`）。多线程频繁访问时，每把锁都是竞争点，等待时间累积，串行段变大。
3. **合并锁（合并包装值）**：把多个相关值塞进同一个 `Mutex<(T, U)>`，**一次加锁访问全部**——这既减少了加锁次数（u5-l3 的 \(N \cdot c \to c\)），也让需要同时访问这些值的线程**只在一把锁上排队**而非多把锁之间产生交叉竞争。
4. **但要警惕锁粒度变大**：若这些值本就常被**不同线程独立**访问（A 线程只用 x、B 线程只用 y），合并成一把锁会让 A、B 在同一把锁上互相阻塞，**反而更慢**。所以「合并」只在「这些值总是一起访问」时才划算——这又回到「取决于访问模式」。

另一条减少同步代价的路线是**根本不用锁**：

- 用**原子操作（atomics）**（`AtomicUsize`/`AtomicBool` 等，见 *Rust Atomics and Locks*）做简单的计数/标志位，避免 `Mutex` 的守卫开销。
- 用 `rayon` 的**归约**模式（4.2），让线程根本不共享可变状态，各算局部结果再合并——这通常是最优解。

#### 4.3.3 源码精读

[wrapper-types.md:11-28](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/wrapper-types.md#L11-L28) 给出了合并包装值的完整例子。第 12–18 行是「两个独立锁」的写法：

```rust
struct S {
    x: Arc<Mutex<u32>>,
    y: Arc<Mutex<u32>>,
}
```

第 20–25 行是「合并为一把锁」的改写：

```rust
struct S {
    xy: Arc<Mutex<(u32, u32)>>,
}
```

第 26–28 行给出**免责声明**：「Whether or not this helps performance will depend on the exact access patterns of the values」——并附了一个真实的 rustc PR 示例链接。这段在并行语境下的含义是：合并锁能减少竞争点，但前提是 x、y **总是被同一线程一起访问**；若它们被不同线程独立高频访问，合并反而引入更粗的锁粒度、更严重的竞争。

注意这里的 `Arc`：它是**线程安全的引用计数**（`Arc` = Atomic Rc），允许多个线程共同拥有同一个 `Mutex`。`Arc<Mutex<T>>` 是 Rust 里「跨线程共享可变状态」的标准骨架——`Arc` 负责共享所有权，`Mutex` 负责互斥访问。本讲 u5-l3 末尾的综合实践（把两个 `Arc<Mutex<u32>>` 合并为 `Arc<Mutex<(u32,u32)>>`）正是本模块代码的原型。

#### 4.3.4 代码实践

这是一个**源码阅读型实践**，目标是把 u5-l3 的合并锁直觉，推进到「并行下锁竞争」的量化理解。

1. **实践目标**：通过阅读真实 PR 与代码，理解「锁竞争」如何抵消并行收益。
2. **操作步骤**：
   - 打开 [wrapper-types.md:28](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/wrapper-types.md#L28) 引用的 rustc PR 示例链接（`rust-lang/rust#68694` 的那条 commit），阅读它合并了什么、为什么更快。
   - 对照思考：如果该 PR 合并的不是「总是一起访问的值」，而是「常被不同代码路径独立访问的值」，合并锁还会更快吗？
3. **需要观察的现象**：你会看到真实项目里「减少锁竞争」带来的实测提速——这正是把 4.1 的 Amdahl 公式中串行段 \(1-p\) 缩小后的收益。
4. **预期结果**：理解一条判据——**「这些共享值是不是总被一起访问」**是「该不该合并锁」的决定性问题。**待本地验证**：具体 PR 的 diff 内容请以链接为准。
5. 进阶思考：在你自己的多线程代码里，找一把被多线程高频抢的锁，评估它能否换成「每线程局部变量 + 最后归约」（即 `rayon` 模式）从而彻底消除竞争。

#### 4.3.5 小练习与答案

**练习 1**：为什么说「并行会让 `Mutex` 的代价从『每次访问慢一点』升级为『拖垮加速比』」？

**答案**：单线程下，`Mutex` 的开销只是锁/解锁的固定成本 \(c\)，每次访问多花一点时间。但多线程下，`Mutex` 强制临界区串行执行——多个线程抢同一把锁时，未获锁的线程必须**阻塞等待**，这段等待时间不算并行收益，反而像串行段一样吃掉加速比，把 \(1-p\) 撑大、把 \(S(N)\) 压低。竞争越激烈，并行越退化为串行，所以「一把被争抢的锁」能毁掉整个并行加速。

**练习 2**：`Arc<Mutex<T>>` 里，`Arc` 和 `Mutex` 各自负责什么？为什么并行代码里这个组合到处都是？

**答案**：`Arc`（原子引用计数）负责**跨线程共享所有权**——多个线程可以各持有一个 `Arc`，引用计数用原子操作保证线程安全；`Mutex` 负责**互斥访问**内部值——保证同一时刻只有一个线程能修改 `T`。组合起来就是「多个线程共同拥有一个可变值」的标准骨架：`Arc` 解决「谁能拿到它」，`Mutex` 解决「谁能改它」。所以它是并行代码里共享可变状态的默认写法。

### 4.4 SIMD：细粒度数据并行的现状

#### 4.4.1 概念说明

perf-book 在讲完线程并行后，又点了**另一种并行形态**：

> If you are interested in fine-grained data parallelism, this [blog post] is a good overview of the state of SIMD support in Rust as of November 2025.

这里要先把两种「数据并行」分清楚，它们处在**完全不同的层次**：

- **4.2 的 `rayon` 并行**是**线程级 / 核心级**的——多个核心各自跑一个线程，每个核心同一时刻仍只处理一个数据元素。
- **SIMD（Single Instruction, Multiple Data，单指令多数据）**是**指令级**的——**一个核心**内，用一条机器指令**同时**对寄存器里的**一批**数据做同一种运算。x86 的 SSE/AVX 寄存器宽 128/256/512 位，一条 `paddd` 指令能一次加 4/8/16 个 32 位整数。它不增加核心，而是加宽「每条指令能处理的数据量」。

所以 SIMD 是「让一个核心自己更快地处理一批数据」，和 4.2 的多核并行是**正交**的，可以叠加（多核 × 每核 SIMD = 极高吞吐）。perf-book 把它单列出来，是因为它的使用方式、工具链、成熟度都和线程并行很不一样。

> 术语提醒：u5-l4 已经引入过 SIMD 的基本词汇——**机器码/汇编**、**宽寄存器（`xmm`/`ymm`/`zmm`）**、**`core::arch` 模块的架构相关 intrinsics**（多为 `unsafe`，如 `_mm256_add_epi32`）。本模块不重复这些，直接承接。

#### 4.4.2 核心流程

Rust 里用 SIMD 有两条路，对应「谁来写并行指令」：

1. **让编译器自动向量化（auto-vectorization）**——这是 u5-l4 的核心结论。LLVM 自带自动向量化器，只要循环满足条件（数据独立、连续访问、无别名干扰），它就会自动把逐元素循环编译成 SIMD 指令。你**几乎什么都不用写**，只要打开 `-C target-cpu=native`（u2-l3）让编译器知道当前 CPU 支持哪些 SIMD 指令集，并确保循环写得「可向量化」。这是首选路径——**手写 SIMD 前必须先看机器码确认编译器没有自动向量化**（u5-l4）。

2. **手写 SIMD intrinsics**——当编译器无法自动向量化（数据布局不友好、算法需要特殊重排），而你确认这是热点瓶颈时，才用 `core::arch`（架构相关，x86/ARM 各一套，多为 `unsafe`）或更高级的封装。手写时要用 `is_x86_feature_detected!` 做运行期 CPU 特性检测、用 `#[target_feature]` 做函数级标注，并**对照机器码验证收益**（u5-l4）。

决策流程（承接 u5-l4）：

```
1. 确认这是「小段、极热」的数值循环（u5-l4）
2. 打开 -C target-cpu=native，用 Compiler Explorer 看是否已自动向量化
   - 已向量化（出现 xmm/ymm/paddd 等）→ 收工，别手写
   - 未向量化 → 进第 3 步
3. 改写循环让它「可向量化」（连续访问、消除别名、用 chunks_exact 等，u4-l3），再看机器码
4. 仍不行且确属瓶颈 → 才手写 core::arch intrinsics，配运行期检测，机器码验证
```

#### 4.4.3 源码精读

[parallelism.md:19-22](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/parallelism.md#L19-L22) 是本模块的原文依据：第 19–20 行把「细粒度数据并行」的现状指向一篇 2025 年 11 月的综述博客，第 22 行是该链接。书里**没有**自己讲 SIMD 的任何细节——连一行 `core::arch` 代码都没贴，完全外链。

#### 4.4.4 代码实践

这是一个**源码阅读 + 验证型实践**，承接 u5-l4 的「看机器码」纪律。

1. **实践目标**：亲眼确认编译器能否对你的数值循环自动向量化，体会「不手写 SIMD」往往就够。
2. **操作步骤**：
   - 打开 [Compiler Explorer（godbolt.org）](https://godbolt.org/)（u5-l4 引入过），选 Rust 编译器。
   - 粘贴一段「对切片逐元素求和」的函数（示例代码）：
     ```rust
     pub fn sum_sq(data: &[i32]) -> i64 {
         data.iter().map(|&x| (x as i64) * (x as i64)).sum()
     }
     ```
   - 在编译参数里加 `-C opt-level=3 -C target-cpu=native`（release + 本机 CPU 特性）。
3. **需要观察的现象**：
   - 观察生成的汇编里是否出现宽寄存器（`xmm`/`ymm`/`zmm`）和 SIMD 指令（如 `pmulld`/`paddd` 之类）。
   - 若出现，说明 LLVM **自动**把你的逐元素循环向量化了——你没写任何 `core::arch` 代码就拿到了 SIMD 收益。
   - 对比去掉 `-C target-cpu=native` 后的汇编，看 SIMD 指令是否消失（说明该开关是自动向量化的关键前提，呼应 u2-l3）。
4. **预期结果**：在开了 `target-cpu=native` 的 release 构建下，这种规整的数值循环**通常会被自动向量化**。这正是 u5-l4「手写前先看机器码」的实操——先确认编译器已经做了，再决定要不要手写。**待本地验证**：具体生成的指令集取决于 godbolt 上选的编译器版本与目标 CPU。
5. 思考：把循环改成「每步依赖上一步」（如 `acc += x[i] - x[i-1]` 之类有跨迭代依赖的），再看机器码——SIMD 指令往往会消失，因为数据不再独立、无法向量化。这就是「取决于程序设计」在指令层的体现。

#### 4.4.5 小练习与答案

**练习 1**：SIMD 和 `rayon` 的多核并行有什么本质区别？它们能叠加吗？

**答案**：SIMD 是**指令级**并行——一个核心内，一条指令同时处理一批数据（靠宽寄存器）；`rayon` 是**核心级**并行——多个核心各跑一个线程。它们处在不同层次、互相正交，可以叠加：N 个核心 × 每核一条 SIMD 指令处理 K 个数据 = 理论上 NK 倍吞吐。所以「先把每核的 SIMD 榨干（u5-l4），再多核并行（4.2）」是数值密集型程序的高吞吐组合。

**练习 2**：perf-book 为什么不自己讲 SIMD 的细节，而是指向一篇外部博客？

**答案**：因为 SIMD 的现状**高度依赖时间与平台**——哪些指令集可用、`portable_simd` 是否进 stable、哪个 crate 是当前最佳选择，都在快速演进。作者若在书里写死，书稿会迅速过时；指向一篇标注了日期（2025 年 11 月）的综述博客，既能给读者权威指引，又把「保鲜」的责任交给会更新的博客。这和本书一贯的「重广度轻深度、深度靠外链」风格（u1-l1）一致。

## 5. 综合实践

设计一个贯穿本讲四个模块的小任务：**把一个串行数值程序改造为并行，并量化每一步的收益与代价**。

**场景**：给定一个很大的 `Vec<i64>`（比如 1 亿元素），要计算所有元素的「平方之和」。这正好是 4.2 的 `rayon` 示例。

**任务步骤**：

1. **建基线（承接 u2-l1/u2-l3）**：写串行版 `data.iter().map(|x| x*x).sum()`，务必 `cargo build --release`，用 `hyperfine` 或 `std::time::Instant` 测出串行 wall-time 作为基线。
2. **算天花板（模块 4.1）**：估算这个负载的 \(p\)（几乎全并行，仅最终归约串行，\(p \approx 1\)），用 Amdahl 公式算出你这台机器（`nproc` 核）的理论最大加速比，作为标尺。
3. **上 `rayon`（模块 4.2）**：把 `iter()` 换成 `par_iter()`，测并行 wall-time，算实测加速比。对照第 2 步的天花板，解释差额来自哪里（归约、调度、数据搬运）。
4. **引入共享状态的对照（模块 4.3）**：故意写一个「错误的并行版」——用多个线程往一个 `Arc<Mutex<i64>>` 里累加平方和（而不是 `par_iter` 归约）。测它的 wall-time，你会看到它可能比串行还慢——这就是锁竞争吃掉全部并行收益、甚至倒挂的实证。思考：为什么 `par_iter().sum()` 能避开这个陷阱？
5. **看 SIMD（模块 4.4 / u5-l4）**：在 Compiler Explorer 上把 `par_iter` 闭包里的 `x*x` 那段看机器码，确认开了 `target-cpu=native` 后它是否被自动向量化（出现 `xmm`/`pmulld` 等）。理解你拿到的并行加速里，有一部分其实是「每核 SIMD 自动向量化」贡献的。

**预期收获**：你会同时看到「数据并行的大收益（`rayon`）」和「锁竞争的大代价（错误版）」两个极端，并理解 perf-book 那句「最佳方式取决于程序设计」的全部含义——同样是「多线程」，归约模式让你接近线性加速，共享锁模式让你比串行更慢。把每一步的实测数字记下来，配合剖析（u2-l2）与机器码（u5-l4），就构成了一套完整的「决定该不该并行、怎么并行」的工程方法。

> 待本地验证：上述所有耗时数字依赖你的核心数、内存带宽与编译器版本，请以本地实测为准。重点是**对照**（串行 vs 并行 vs 锁版）的相对关系，而非绝对值。

## 6. 本讲小结

- perf-book 的「Parallelism」章是一块**刻意简短的路标**：它宣称 Rust 对安全并行的支持「卓越」、能带来「巨大提升」，但明确把深入讲解排除在书外，因为「最佳并行方式高度依赖程序设计」。
- 并行加速比的上限由 **Amdahl 定律** \(S(N)=1/((1-p)+p/N)\) 给出，其中可并行占比 \(p\) 是程序**结构**的属性——这就是「取决于程序设计」的数学含义。
- **线程并行**的入口是 `rayon`（数据并行、工作窃取线程池，`par_iter` 一行改造）与 `crossbeam`（底层并发构件），深入原子与锁看 *Rust Atomics and Locks*。
- 并行必然引入**同步**，而 `Mutex` 等原语的每次访问开销（u5-l3 的固定成本 \(c\)）在多线程下放大为**锁竞争**——它强制临界区串行，吃掉加速比；合并锁、改用归约或原子操作是减竞争的路线。
- **SIMD** 是与多核正交的指令级数据并行；Rust 里应**优先靠 `-C target-cpu=native` 让 LLVM 自动向量化**，手写 `core::arch` intrinsics 是「先看机器码、确认编译器没做」之后的最后手段（u5-l4）。
- 贯穿全程仍是「先测量再优化」纪律：要不要并行、怎么并行、是否值得，都必须靠剖析（找热点）、基准测试（量加速比）、看机器码（验 SIMD）来回答。

## 7. 下一步学习建议

本讲是「更高层次的性能原则」单元（u6）的收尾性能主题。建议按以下顺序继续：

- **补完本单元**：若你尚未读 u6-l1（General Tips，通用性能原则，含 Amdahl 定律原始形式）和 u6-l2（Compile Times，编译耗时），请补上——本讲的 Amdahl 公式直接来自 u6-l1，而编译耗时是优化工作流里常被忽视的一环。
- **顺着本讲的路标深入**：
  - 想上手数据并行：照 [parallelism.md](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/parallelism.md) 指向的 `rayon` 文档，读完它的「Rayon Conference Talks」与 `par_iter` 教程，再尝试 `split`/`par_chunks` 等更细的控制。
  - 想理解锁与原子：读 *Rust Atomics and Locks*（Mara Bos），重点看内存序（memory ordering）与无锁数据结构——这能让你判断「这把锁能不能换成原子」。
  - 想玩 SIMD：读那篇 2025 年 11 月综述博客，再用 u5-l4 的 Compiler Explorer 工作流在自己代码上练自动向量化。
- **进入第七单元（维护与贡献）**：u7-l1 讲 `mdbook test` 与 CI、u7-l2 讲为 perf-book 贡献内容。读完本讲你会发现，「Parallelism」这种「只树路标」的章节，本身就是 perf-book 简洁风格的标本——如果你想把某个外部主题（如「锁竞争实战」或「SIMD 入门」）写进书里，u7-l2 会告诉你它的写作约定与落地步骤。
- **把方法用回你的代码**：挑一个你自己的 CPU 密集热点，按本讲第 5 节的综合实践流程走一遍——先算 Amdahl 天花板，再试 `rayon`，最后看 SIMD 机器码。这套「测量—并行—验证」闭环，是本讲希望你带走的核心能力。
