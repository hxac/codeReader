# Wrapper Types 与日志/调试的性能代价

## 1. 本讲目标

学完本讲，读者应能：

- 理解为什么访问 `RefCell`/`Mutex` 等「包装类型」不是免费的，每次访问都有一笔运行期开销。
- 掌握「把多个总是成组访问的包装值合并进同一个包装类型」这一优化手法及其取舍。
- 识别日志/调试代码（以及喂给它们的数据收集代码）带来的隐藏开销。
- 区分 `assert!` 与 `debug_assert!`，知道何时该把热点断言降级为 `debug_assert!`。

## 2. 前置知识

本讲假设你已具备：

- Rust 的所有权、借用，以及 `Arc`/`Mutex`/`RefCell` 的基本语义（可回顾 u1 系列与 u4-l2）。
- 「先测量再优化」的工作流（u2-l1 基准测试、u2-l2 剖析）。本讲会反复强调：写法改动只是候选优化，是否值得由测量决定。

关键直觉：Rust 的安全检查有些在编译期完成（零开销），有些被推迟到运行期完成（有开销）。包装类型属于后者——它们把一个无法在编译期证明的不变量（「同一时刻只有一个可变借用」「同一时刻只有一个线程持有锁」）放到运行期去维护，于是每次访问都要为「维护这个不变量」付出成本。

两类包装类型的作用回顾：

- `RefCell<T>`：单线程下的「内部可变性」。拿到 `&RefCell<T>` 也能获得 `&mut T`，代价是借用规则改为运行期检查——`borrow()`/`borrow_mut()` 维护一个借用计数。
- `Mutex<T>`：多线程下的互斥访问。`lock()` 通过原子操作获取锁，争用时可能触发操作系统级的睡眠/唤醒（如 Linux 的 futex）。

二者形态不同，但「每次访问都要做额外工作」是共性，这正是本讲的出发点。

## 4. 核心概念与源码讲解

### 4.1 包装类型（RefCell/Mutex）的访问开销

#### 4.1.1 概念说明

「包装类型」(wrapper type) 指 `RefCell`、`Mutex` 这类为内部值提供特殊行为的类型。普通字段 `x: u32` 的访问是「取地址、读内存」，安全由编译期保证、运行期零额外开销；而 `x: Mutex<u32>` 的访问必须先 `lock()`（或 `RefCell` 的 `borrow()`）拿到守卫(guard)，再通过守卫读值，最后守卫析构释放——这条链路上每一步都不是免费的。

为什么要付这个代价？因为这些类型把一个无法在编译期证明的不变量推迟到运行期维护：

- `RefCell` 维护「借用规则」：运行期检查是否存在冲突的可变借用。
- `Mutex` 维护「互斥」：运行期用原子操作保证同一时刻只有一个线程进入临界区。

访问开销的本质，就是「维护这个不变量」所执行的代码。

#### 4.1.2 核心流程

一次 `Mutex<u32>` 的读取大致经历：

1. `lock()`：执行一次原子 CAS（compare-and-swap），把锁状态从「未锁」改为「已锁」。
   - 无争用(uncontended)：只花一次原子操作 + 内存屏障。
   - 有争用(contended)：CAS 失败，陷入操作系统调用（Linux 上是 futex）让线程睡眠，被唤醒后再试。
2. 返回 `MutexGuard`，通过它间接访问内部值（多一层指针解引用）。
3. 离开作用域时 `Guard` 析构，再执行一次原子操作释放锁。

`RefCell` 更轻（单线程、无原子操作，只是普通整数读改写借用计数），但同样有「检查 + 计数维护 + 守卫间接访问」的固定开销。

若把一次直接字段访问记为单位开销 1，把一次包装访问的固定开销记为 \(c\)（\(c\) 明显大于 1），则一段对 \(N\) 个独立包装值各访问一次的代码，其访问相关开销近似为：

\[
\text{开销} \approx N \cdot c
\]

减小 \(N\)（合并包装值，见 4.2）或减小 \(c\)（换更快的实现，见 4.1.3 末尾）都能降低开销。这正是本讲两条优化路线的数学根据。

#### 4.1.4 代码实践

实践目标：直观感受一次 `Mutex` 访问相对直接字段访问的固定开销。

操作步骤：

1. 新建一个小 crate，写两个版本的热点循环（各累加一亿次）：
   - 版本 A：直接累加 `u64` 局部变量。
   - 版本 B：累加 `Arc<Mutex<u64>>`，循环体为 `*m.lock().unwrap() += 1;`。
2. 用 hyperfine（u2-l1 介绍过）分别计时：`hyperfine --warmup 3 './a' './b'`。

需要观察的现象：版本 B 明显慢于版本 A。

预期结果：B 比 A 慢一个数量级左右。具体倍数**待本地验证**——本例单线程无争用，主要开销来自每次循环的原子操作与守卫析构，而非锁争用。

说明：这不是劝你不用 `Mutex`（并发正确性远比这点开销重要），而是建立「每次 `.lock()` 都有固定成本」的直觉，为 4.2 的「合并」铺垫动机。

#### 4.1.5 小练习与答案

**练习 1**：在单线程下，`RefCell::borrow()` 的开销通常比 `Mutex::lock()` 更高还是更低？为什么？

参考答案：通常更低。`RefCell` 是单线程的，借用计数只是普通整数的读改写，没有原子操作与内存屏障；`Mutex::lock()` 即使无争用也要执行原子 CAS 与内存屏障，开销更大。

**练习 2**：若一段代码先 `lock()` 取值、立刻 `drop` 守卫、做些计算、再 `lock()` 取一次值，相比「拿一次守卫贯穿始终」，开销有何不同？

参考答案：前者是两次完整「获取—释放」周期（\(\approx 2c\)），后者是一次（\(\approx c\)）。在临界区可以整体持有的前提下，一次持有通常更省；但若想减小临界区粒度以提升并发，则需权衡——这正是锁设计的基本取舍。

### 4.2 合并多个包装值

#### 4.2.1 概念说明

4.1 给出开销近似 \(\text{开销} \approx N \cdot c\)。当程序里多个包装值「总是成组访问」时，每次访问整组都要付「组大小」次 \(c\)。若把整组合并进同一个包装类型，则单次访问只付一次 \(c\)。

这就是 perf-book 的核心建议：「如果多个这样的值通常被一起访问，把它们放进同一个包装类型里可能更好。」

合并还有两个书中未明说但自然成立的好处：

- **原子性**：一次加锁给出 x 与 y 的一致快照，避免读到「x 已更新、y 未更新」的中间态。
- **缓存局部性**：元组字段常落在同一缓存行，访问第二个值更可能命中缓存，且堆上从两个独立分配变成一个分配。

#### 4.2.2 核心流程

判断「是否该合并」的决策流程：

1. 用剖析（u2-l2）确认这些包装类型的访问是热点。
2. 观察访问模式：这些值是否几乎总是成组读写？
   - 是 → 合并能把整组的 \(N \cdot c\) 降到约 \(c\)，通常有收益。
   - 否（各值经常被独立、并发地访问）→ 合并会**加大锁粒度**，把原本可并发的访问串行化，可能反而更慢。
3. 合并后，访问点改为先 `lock()` 拿到元组守卫，再 `.0`/`.1` 取字段。
4. 用基准测试（u2-l1）验证前后差异。

关键提醒（书中原文）：合并是否提升性能，**取决于值的确切访问模式**。

#### 4.2.3 源码精读

书中用一对经典对照给出合并手法。合并前——两个字段各自独立加锁：

[src/wrapper-types.md:L11-L18](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/wrapper-types.md#L11-L18) —— 这段定义 `struct S { x: Arc<Mutex<u32>>, y: Arc<Mutex<u32>> }`，两个 `u32` 各自包在自己的 `Arc<Mutex>` 里。要同时读 x 和 y，需要两次 `lock()`。

合并后——两个字段共用一把锁：

[src/wrapper-types.md:L19-L25](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/wrapper-types.md#L19-L25) —— 这段把两个字段合并为 `struct S { xy: Arc<Mutex<(u32, u32)>> }`，用一个 `Arc<Mutex>` 包住整个元组。现在同时读 x、y 只需一次 `lock()`，开销从约 \(2c\) 降到约 \(c\)。

书中紧接着给出关键限定：

[src/wrapper-types.md:L26-L28](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/wrapper-types.md#L26-L28) —— 这两行声明「这是否有助于性能，取决于值的确切访问模式」，并附一个真实 rustc PR 示例（rust-lang/rust#68694）。

#### 4.2.4 代码实践

实践目标：亲手把「两个独立 `Arc<Mutex<u32>>`」合并为「一个 `Arc<Mutex<(u32, u32)>>`」，并用基准测试观察差异。

操作步骤：

1. 写一个 `struct Pair { x: Arc<Mutex<u32>>, y: Arc<Mutex<u32>> }`，并提供 `fn inc(&self)`：它对 x、y 各 `lock()` 一次并自增。
2. 在 release 下（`cargo build --release`）循环调用 `inc()` 一千万次，用 hyperfine 或 `std::time::Instant` 计时。
3. 改写为 `struct Pair { xy: Arc<Mutex<(u32, u32)>> }`，`fn inc(&self)` 只 `lock()` 一次，对 `g.0`、`g.1` 各自增。再次计时。

需要观察的现象：合并版耗时更短。

预期结果：合并版每次 `inc()` 少一次「获取—释放」周期，应明显更快。具体倍数**待本地验证**——本例两次锁在同一结构上、无真争用，差异主要来自省掉的那次原子操作与守卫开销。

附加思考：若把 `inc()` 拆成 `inc_x()` 与 `inc_y()`，由两个线程分别高频调用，合并后二者会被同一把锁串行化。此时合并**反而可能更慢**——这正是「取决于访问模式」的含义。

#### 4.2.5 小练习与答案

**练习 1**：合并 `Arc<Mutex<u32>>` × 2 为 `Arc<Mutex<(u32, u32)>>` 后，结构体本身（不含 `Arc` 的堆分配）的体积如何变化？

参考答案：在 64 位平台上，合并前是两个 `Arc` 字段（两个指针宽，约 16 字节），合并后是一个 `Arc` 字段（一个指针宽，约 8 字节）；同时堆上从两个互不相关的分配变成一个分配，缓存局部性更好。

**练习 2**：举一个「合并会带来正确性收益」的例子。

参考答案：当程序需要同时读 x 和 y 并要求二者一致时，合并后一次 `lock()` 自然给出原子快照；若维持两个独立锁，则要么可能读到不一致的中间态，要么需要固定加锁顺序、加两次锁来人为保证原子性，更易出错。

### 4.3 日志/调试代码的隐藏开销

#### 4.3.1 概念说明

日志(logging)与调试(debugging)代码本意是「开发期辅助」，但它们若残留进 release 热路径，会显著拖慢程序。perf-book 指出拖慢有两个来源：

1. 日志/调试代码**本身**慢（例如每条记录都做格式化、序列化、I/O）。
2. 更隐蔽的：**喂给日志/调试代码的「数据收集代码」慢**——即使日志最终没被打印，构造那条日志所需的字符串、遍历、统计也可能已经在跑。

第二条尤其危险：开发者以为「日志关了就没事」，但 `format!(...)` 等参数求值往往发生在「判断日志是否开启」**之前**。原则是：当日志/调试未启用时，绝不应为它做任何多余工作。

#### 4.3.2 核心流程

排查日志/调试开销的流程：

1. 用剖析（u2-l2）定位热点，留意是否出现 `fmt`、`format`、`Display`/`Debug` 的实现，或本应廉价的函数却意外耗时。
2. 检查这些代码是否在为「未启用的日志」构造数据。常见反模式（示例代码，非项目原有）：

   ```rust
   // 示例代码：日志未启用也照样构造字符串
   let detail = expensive_summary(&data);     // 总是执行
   log::debug!("detail = {}", detail);        // debug 关闭时不打印，但 detail 已算完
   ```

3. 改为把昂贵的数据收集放进「先判开关」的分支，或借助会短路求值参数的日志宏（**须查具体日志库文档确认**其是否在求值参数前先判断级别）。
4. 重新剖析、对比。

#### 4.3.3 源码精读

[src/logging-and-debugging.md:L3-L9](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/logging-and-debugging.md#L3-L9) —— 这段指出「有时日志或调试代码会显著拖慢程序」，原因有二：日志/调试代码本身慢，或「喂给日志/调试代码的数据收集代码」慢；并要求「当日志/调试未启用时，确保没有为它做多余工作」。书末附三个真实 rustc PR 示例（#50246、#75133、#147293）。

注意书的写法：它只给原则与真实 PR 链接，不展开具体日志库用法——这符合 perf-book「重广度轻深度、细节靠外链」的风格（见 u1-l1）。读者应点开这些 PR，观察「真实工程里日志开销长什么样、怎么改」。

#### 4.3.4 代码实践（源码阅读型）

实践目标：通过阅读真实 PR，建立「数据收集开销」的直觉。

操作步骤：

1. 打开书中给出的 [Example 2](https://github.com/rust-lang/rust/pull/75133/commits/eeb4b83289e09956e0dda174047729ca87c709fe)（rust-lang/rust#75133）。
2. 阅读 commit diff，回答：这次改动是删掉了「日志本身」，还是删掉了「为构造日志而做的数据收集」？被删的代码在「日志关闭」时是否仍在运行？

需要观察的现象：被删/被门控的多是「为日志准备数据的计算」，而非打印语句本身。

预期结果：你会看到一类典型模式——把昂贵的数据收集包进「仅在对应日志级别开启时才执行」的条件分支，从而让 release 热路径不再为关闭的日志付代价。

#### 4.3.5 小练习与答案

**练习 1**：下面两段（示例代码）在「debug 日志关闭」时，开销有何不同？

```rust
// A
log::debug!("x = {}", fmt_thing(&big));   // 取决于日志库是否短路求值参数
// B
let s = fmt_thing(&big);                   // 无条件先格式化
log::debug!("x = {}", s);
```

参考答案：A 的开销取决于所用日志库是否在求值参数前先判断级别（多数成熟日志宏会先判级别、不匹配则不求值参数，但**必须查具体库文档确认**）；B 则无论日志库如何，`fmt_thing` 都一定会执行。因此 B 是确定的反模式，应避免。

**练习 2**：为什么「数据收集代码」的开销比「日志本身」更难发现？

参考答案：因为数据收集代码常散落在各处、名字看似与日志无关，剖析时容易只盯着 `log`/`println` 调用点；而真正的开销藏在「为构造日志参数而调用的业务函数」里，需要顺着调用链才能找到。

### 4.4 assert! 与 debug_assert!

#### 4.4.1 概念说明

`assert!(cond)` 与 `debug_assert!(cond)` 都是断言：条件为假时 panic。区别在于「在什么构建下生效」：

- `assert!`：**始终**编译进所有构建（dev 与 release 都运行）。
- `debug_assert!`：仅在启用 `debug_assertions` 时运行（dev 构建默认启用，release 构建默认关闭，故 release 下编译为空操作）。

推论：若某个断言位于热点、且**不是安全所必需**（即去掉它不会导致未定义行为），把它从 `assert!` 改为 `debug_assert!`，就能在 release 里完全抹掉这次条件判断，而 dev 里仍保留防护。

#### 4.4.2 核心流程

决策流程：

1. 用剖析（u2-l2）确认某处 `assert!` 是热点。
2. 判断该断言是否「安全所必需」：
   - 是（例如挡住会导致 UB 的越界、空指针解引用）→ 保留 `assert!`，**不能降级**。
   - 否（仅是「逻辑上不该发生」的健全性检查）→ 可降级为 `debug_assert!`。
3. 降级后，dev 仍 panic 防护，release 移除检查提速。
4. 重新基准测试确认收益。

补一句工程常识（书中未展开）：`debug_assertions` 是 cargo profile 的开关，dev 默认开、release 默认关；也可在 release profile 里显式打开它，从而「release 也保留 debug_assert」（代价是失去对应提速）。

#### 4.4.3 源码精读

[src/logging-and-debugging.md:L11-L15](https://github.com/nnethercote/perf-book/blob/a05dd0f15595e98aef45e5a15072c2a71dbe37ba/src/logging-and-debugging.md#L11-L15) —— 这段明确：`assert!` 调用**始终运行**，`debug_assert!` 调用**只在 dev 构建运行**；若某断言是热点且非安全所必需，应考虑改为 `debug_assert!`。书末附两个真实 rustc PR 示例（#58210、#90746）。

#### 4.4.4 代码实践

实践目标：亲手把一处热点 `assert!` 降级为 `debug_assert!` 并观察 release 行为。

操作步骤：

1. 写一个循环一亿次、每次执行一条廉价断言的小程序（示例代码）。
2. `cargo build --release` 并计时。
3. 把该 `assert!` 改为 `debug_assert!`，重新 `cargo build --release` 计时。

需要观察的现象：release 下降级版更快。

预期结果：release 下 `debug_assert!` 不生成任何检查代码，循环少一次条件判断；提升幅度**待本地验证**——断言越在循环深处、条件越复杂，收益越明显。

附加验证（确认「dev 仍生效」）：`cargo run`（dev 构建）下，故意让条件为假，应照常 panic——证明 `debug_assert!` 在 dev 仍是有效防护。

#### 4.4.5 小练习与答案

**练习 1**：为什么「安全所必需」的断言绝不能降级为 `debug_assert!`？

参考答案：因为 `debug_assert!` 在 release 下被完全移除。若该断言挡住的是越界、空指针解引用等会引发 UB 的情形，降级后 release 将失去防护、产生 UB——正确性（尤其未定义行为）必须优先于性能。

**练习 2**：能否在 release 构建里仍保留 `debug_assert!`？

参考答案：能。`debug_assert!` 是否编译进代码由 `debug_assertions` profile 标志控制；在 `Cargo.toml` 的 release profile 里设置 `debug-assertions = true`，即可在 release 里继续运行 `debug_assert!`（代价是失去对应的提速）。

## 5. 综合实践

把本讲四条线索串起来，完成一个端到端的小任务。

背景（示例代码，非项目原有）：一个计数器服务持有两个计数：

```rust
// 示例代码：合并前的结构
use std::sync::{Arc, Mutex};

struct Counters {
    hits: Arc<Mutex<u32>>,
    misses: Arc<Mutex<u32>>,
}

impl Counters {
    fn record(&self, hit: bool) {
        if hit {
            *self.hits.lock().unwrap() += 1;
        } else {
            *self.misses.lock().unwrap() += 1;
        }
        // 一处热点断言：u32 恒非负，校验 < u32::MAX 纯属冗余的健全性检查
        assert!(*self.hits.lock().unwrap() < u32::MAX);
    }
}
```

任务：

1. **合并包装值**：把 `Counters` 改为 `struct Counters { hm: Arc<Mutex<(u32, u32)>> }`，`record` 内只 `lock()` 一次，通过守卫的 `.0`/`.1` 同时更新 hits/misses。说明此举把每次 `record` 的锁获取次数从 2 次（一次更新 + 断言里那次）降到 1 次（\(\approx 2c \to c\)）。
2. **审视断言**：那条 `assert!` 校验的是「u32 是否 < u32::MAX」——u32 语义上恒非负、该检查与任何 UB 无关（非安全所必需），又位于 `record` 热路径，判断它可否降级，并说明理由。
3. **执行降级**：把它改为 `debug_assert!`（合并后改为对守卫字段的检查）。
4. **测量**：对「合并前」「合并后」「再降级后」三个版本分别用 hyperfine 在 release 下计时，给出三个数据点。

预期：

- 合并把锁次数减半，应有可观提速（具体倍数**待本地验证**）。
- 该 `assert!` 校验冗余且非安全必需，可降级；降级后 release 少一次条件判断，且省掉断言本身那次多余 `lock()`。
- 综合来看：原 `record` 每次都触发 **2 次** `lock()`（一次更新计数 + 断言里的一次），合并 + 降级后压到 **1 次**。

完成本任务后，你应能独立完成规格里给的主实践：「把含两个独立 `Arc<Mutex<u32>>` 字段的结构体合并为 `Arc<Mutex<(u32, u32)>>`，并审视代码中一处热点 `assert!` 是否可降级为 `debug_assert!`」。

## 6. 本讲小结

- 包装类型（`RefCell`/`Mutex` 等）把不变量推迟到运行期维护，每次访问都有一笔固定开销 \(c\)（`Mutex` 含原子操作，`RefCell` 较轻但仍非零）。
- 多个「总是成组访问」的包装值，可合并进同一个包装类型，把开销从约 \(N \cdot c\) 降到约 \(c\)；但是否提升取决于访问模式，独立并发访问时合并反而可能更慢。
- 减小单次开销 \(c\) 的另一条路是换 `parking_lot` 等替代实现（回扣 u4-l2），但须先测量。
- 日志/调试代码的隐藏开销常来自「喂给它的数据收集」，而非打印本身；日志未启用时不应为它做任何多余工作。
- `assert!` 始终运行，`debug_assert!` 只在 dev 运行；非安全所必需的热点断言可降级以提速，安全必需的绝不可降级。
- 贯穿纪律：所有写法改动都只是候选优化，是否值得须靠剖析与基准测试确认（承接 u2-l1、u2-l2）。
