本讲（u6-l4 多语言翻译机制）是「配套工具与维护机制」单元第四篇，围绕仓库「为什么这么大、怎么只下载需要的部分」展开，覆盖三个核心模块。模块一讲清 `translations/` 与 `translated_images/` 的组织方式：机器翻译产物采用「路径镜像」设计，源文件整体平移到 `translations/<locale>/` 下、路径一一对应（语言代码遵循 BCP 47，如 zh-CN），翻译后的图则集中存于 `translated_images/<locale>/`，由译文用相对路径回退引用；二者占了绝大部分下载体积。模块二解析微软开源的 `co-op-translator` 自动翻译流水线：它经 GitHub Action 机器人 `localizeflow[bot]` 周期性提交标题为 `chore(i18n): sync translations ... (chunk N/M)` 的同步 PR，核心是其**增量翻译**机制——每语言目录下的 `.co-op-translator.json` 清单记录各源文件的 MD5 `original_hash` 与 `translation_date`，**仅当源文件哈希变化才重译**，对应 `make` 式「内容指纹驱动」思想，避免重复调用翻译 API。模块三给出官方推荐的稀疏克隆方案：组合 partial clone（`--filter=blob:none`，按需下载 blob）与 sparse-checkout（排除两目录），可只拿到课程本体、显著降低下载量；规则随时可改无需重新克隆。关键术语：i18n/l10n、路径镜像、`.co-op-translator.json`、`original_hash`、`localizeflow[bot]`、partial clone、sparse-checkout、`--no-cone`。另指出机器翻译（主线，整目录镜像）与手工翻译（`README.<语言码>.md` 后缀，不译代码）两种约定并存。
