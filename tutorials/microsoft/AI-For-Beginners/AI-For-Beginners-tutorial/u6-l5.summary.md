本讲（u6-l5，配套工具与维护机制单元收尾）回答「这个公开 AI 课程仓库如何被安全维护、如何让人安全贡献」，覆盖三个最小模块：安全扫描工作流、开发约定、贡献流程。核心工具是 GitHub Actions（`.github/workflows/`）下的两个互补工作流——**CodeQL** 做**白盒**语义分析，扫描 actions / javascript-typescript / python 三类代码本身的漏洞（均为解释型，故 `build-mode: none`）；**OpenSSF Scorecard** 做**供应链姿态评估**，检查分支保护、依赖钉扎、维护活跃度等十几项卫生指标。两者都产出 **SARIF** 汇入 GitHub **Code scanning** 面板，触发器含 push/PR/schedule（cron 周一/周五错峰），Scorecard 另有 `branch_protection_rule`。关键安全实践：顶层 `permissions: read-all`（**最小权限原则**）、**SHA 钉扎** action（防标签被篡改）、`persist-credentials: false`（防 token 泄露）。开发约定方面，`AGENTS.md` 规定环境唯一命名为 `ai4beg`、Python 教学代码不强求 lint 而 Vue 工程代码要求 ESLint；最反直觉的结论是**教育仓库「没有传统测试套件」，「测试」即「Notebook 端到端跑通」**。贡献流程设三道门：法律门（**CLA** 由 CLA-bot 自动判定、全微软仓库一次签署）、社区门（Microsoft Code of Conduct）、质量门（PR 清单：跑通 Notebook、改英文内容由 `co-op-translator` 自动重译、改测验应用须 `npm run lint`）；`etc/CONTRIBUTING.md` 的「Looking for Contributions」清单是新贡献者上手入口。
