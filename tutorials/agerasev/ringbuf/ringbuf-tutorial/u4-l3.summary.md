本讲介绍 ringbuf 的第二种同步策略——Frozen（冻结）包装器，对照 Direct 的即时同步。核心是在包装器内用 `Cell` 缓存一份本地 `read`/`write` 索引，让索引变化先停留本地、需要时再一次性同步，从而把 N 次跨核缓存同步摊薄成 1 次。Frozen 仅重写 `read_index`/`write_index`/`set_*_index` 等少数底层原语，使所有派生方法都基于本地索引工作：`try_push` 把元素物理写入底层存储却只前进本地 `write`，对端靠底层 `write_b` 判断可见性故看不到，这就是「冻结」——可见性由索引而非物理写入决定。

Frozen 由 const generic 布尔 `P`/`C` 编码权限，别名 `FrozenProd`（只写）、`FrozenCons`（只读）。它提供四种显式操作：`commit`（本端进度回写底层）、`fetch`（拉取对端进度）、`sync`（双向刷新）、`discard`（FrozenProd 专属，以底层 `write_b` 为回退点丢弃未提交草稿并 drop 元素、不泄漏）；权限不对称，只能 commit 自己一侧、fetch 对端一侧。生命周期上经 `freeze()` 从 Direct 或 Caching 进入并继承 hold 标志，`Drop` 与 `into_rb_ref` 均先 commit 再 close，未提交改动不丢失。代价是须手动管理同步时机：漏 fetch 会误判满，忘 commit 则对端不可见。
