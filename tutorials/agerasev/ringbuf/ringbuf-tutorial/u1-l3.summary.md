本讲从仓库整体视角梳理 ringbuf 项目的三 crate 布局。仓库是 Cargo workspace，根 `Cargo.toml` 身兼两职：既是 workspace 定义（`[workspace]` 段，`members = ["async","blocking"]`），又是核心 crate `ringbuf`（0.5.0）的清单——带 `[package]` 的根是隐式成员，故共有三个 crate：核心 `ringbuf`、`async-ringbuf`（0.3.6）、`ringbuf-blocking`（0.1.0-rc.6），依赖方向为派生→核心。核心 crate 职责纯粹，提供无锁 SPSC 数据结构与非阻塞 `try_*` 接口，刻意把「等待语义」剥离；其库入口 `#![no_std]`，通过 `std`/`alloc` 等 feature flag 控制能力级联。两个派生 crate 奉行「核心 `SharedRb` + 同步原语」模式：`async-ringbuf` 加 `AtomicWaker` 提供 async 接口，`ringbuf-blocking` 加 `Semaphore` 提供阻塞接口。关键设计是 `[workspace.dependencies]` 中对核心设 `default-features = false`，派生 crate 用 `workspace = true` 引用并在自身 `[features]` 按需透传 `ringbuf/std`、`ringbuf/alloc`，从而夺回对核心 feature 的控制、保住 no_std 能力。两者转发核心 trait 的方式不同：blocking 整体转发 `ringbuf::traits`，async 只转发子模块并另建 async trait 层。引入的关键术语：workspace、隐式成员、feature 透传、`default-features = false`、feature 统一、`SharedRb` 包装模式。
