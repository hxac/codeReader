本讲讲解派生 crate `ringbuf-blocking` 的底层地基。`BlockingRb<S, X>` = 核心无锁 `SharedRb<S>` + 两只二值信号量 `read`/`write`：数据搬运全部复用核心，阻塞层仅在「改变对端可见状态」处 `give`——推进 write 索引或 `hold_write` → `write.give` 唤醒消费者，推进 read 索引或 `hold_read` → `read.give` 唤醒生产者，「关闭」复用 hold 通道。这条「信号量名 = 你等待的索引」规律与异步版 `async-ringbuf` 完全对称。

核心可插拔抽象为 `Semaphore` trait，含 `give`（幂等置位、多次合并）、`try_take`（非阻塞取旧值）、`take(Option<Duration>)`（阻塞取，带关联时钟类型 `Instant`）及默认方法 `take_iter`；超时由 `NO_WAIT`/`FOREVER`/`Some(d)` 三态表达。默认实现 `StdSemaphore` 用 `Condvar + Mutex<bool>`，`take` 借循环复查加最后检查应对 condvar 伪唤醒与超时边界；`TimeoutIter` 以 `remaining = dur − elapsed` 统一三态超时，`TakeIter.reset()` 在首次迭代做非阻塞排空，避免二值信号量合并通知导致漏唤醒。`no_std` 下只要实现 `Semaphore`（含时钟）即可让同一套 `BlockingRb` 在嵌入式平台运行。对外阻塞 API（`push`/`pop`/`wait_vacant` 等）留待 u7-l2。
