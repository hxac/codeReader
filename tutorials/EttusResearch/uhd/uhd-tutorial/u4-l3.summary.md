本讲覆盖 UHD 的「包协议」层，回答传输层搬来的原始字节如何被解析成包，含四个模块。

vrt_if_packet 是「线缆字节↔C++ 结构」翻译层：核心结构 if_packet_info_t，对称的 pack/unpack API（be/le 两版）以 link_type（NONE/CHDR/VRLP）切链路层、be/le 切字节序；头部首字编码 packet_type（bits 31–29）、各字段存在位、packet_count 与整包长度。代码生成器 gen_vrt_if_packet.py 把 7 个存在位的 2^7=128 种组合在编译期展开成跳转表，造就快路径。

super_recv_packet_handler（sph 命名空间）代表一组共享采样率的通道，在 recv() 里批量收包、解头、按时间戳 tsf 做多通道对齐（SOB 跨通道取与、EOB 取或）、静默消费流控 ACK 包，并把夹在流里的控制包就地转成 error_code；其包类型判定顺序不可调：inline message → sequence error → timestamp error → if data。

super_send_packet_handler 是接收侧镜像：翻译 tx_metadata_t 为 VRT 头、超单包上限则分片并按采样率累加时间戳、走 pack→convert→commit 三段式，并缓存「SOB + 0 样本」的 metadata。

核心结论：数据包与控制包（context / inline message）共用同一套头部，区别仅在 3 位 packet_type；设备异常（溢出、迟到命令等）不是 C++ 异常，而是夹在数据流里、负载首字节低 8 位为状态码的控制包，经 get_context_code 取出强转成 error_code——这是「出错是常态而非异常」的物理根源。
