本讲是入门篇收尾，以真实可编译示例 `rx_samples_to_file` 把前几讲的公共头文件、命令行骨架、`multi_usrp` 串成一条完整调用链，并建立适用于所有 UHD 收发示例的「四阶段阅读模板」。

三个最小模块：① safe_main——`UHD_SAFE_MAIN` 宏把真正的 `main` 包进 `try/catch`，将未捕获异常转成可读 `Error: ...` 与 `-1` 退出码，你写的函数体其实是 `_main`。② boost::program_options——以「声明选项表 `po::options_description` → `parse_command_line` 写入 `variables_map` → `notify`」三步把命令行解析成 C++ 变量；带 `po::value(...)` 的选项要跟值，不带的是布尔开关，用 `vm.count(...)` 判断，且 `notify` 必须在 `--help` 检查之后调用。③ 主流程四阶段：构造设备（`multi_usrp::make`，须先选子设备再做通道映射）→ 配置射频（采样率/频率/增益/带宽/天线 + 轮询 `lo_locked`）→ 接收写文件（`get_rx_stream`、`issue_stream_cmd`、`recv` 循环、写盘）→ 多线程清理（`std::thread`）。

引入的关键术语：`multi_usrp`（高层设备黑盒，细节留待 u2-l3）、`tune_request_t`、`stream_args_t`（CPU 格式与线上 otw 格式）、`stream_cmd_t`、`rx_metadata_t` 的 `error_code`（TIMEOUT 退出、OVERFLOW 丢弃继续、其它按 `--continue` 决定）、`recv_to_file<T>` 模板函数。核心结论：硬件会把请求采样率四舍五入到最接近支持值，因此必须回读实际采样率。
