本讲围绕 UHD 的自动化测试体系展开，核心是 `host/tests/` 目录。最小模块有四：第一是测试构建体系，UHD 把每个测试 `.cpp` 编译成独立可执行文件，靠 `BOOST_TEST_MAIN` 宏自动生成 `main()`，再用封装函数 `UHD_ADD_TEST` 生成设好 `LD_LIBRARY_PATH` 的 `.sh`/`.bat` 包装脚本并注册给 `ctest`。第二是 API 测试（`block_id_test`、`buffer_test`），只链接公开 `libuhd`、走 `test_sources` 列表，测纯逻辑。第三是 NonAPI 测试与 mock 注入（`client_zero_test`），用 `UHD_ADD_NONAPI_TEST` 把内部 `.cpp` 作 `EXTRA_SOURCES` 编进测试，并通过 `mock_reg_iface_t`（来自 `mock_block.hpp`）预填寄存器、用 `_peek_cb`/`_poke_cb` 主动校验，把硬件依赖隔离掉。第四是子系统级覆盖，按子系统各一组测试：`convert_test`（u4-l1，含 `GET_CONVERTER_SAFE` 软跳过与基准）、`rfnoc_block_tests/`（u3，统一用 `get_mock_block`）、`cal_database_test`（u5-l4，覆盖 RC/FILESYSTEM/FLASH 三源）。关键术语包括 Boost.Test 的断言宏分档（`BOOST_CHECK*` 记录并继续、`BOOST_REQUIRE_*` 立即中止）、API 与 NonAPI 测试之分、mock、`uhd_test` 静态库，以及受 `ENABLE_PYTHON_API` 控制的 Python 测试和需真硬件的 `devtest`。核心结论：UHD 测试按子系统组织，形成「实现—验证」闭环；新增 API 测试用例只需写 `BOOST_AUTO_TEST_CASE` 而无需改 CMake，新增测试文件才需加入 `test_sources`。
