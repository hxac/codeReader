本讲钻进收发流器接口脚下的传输层，回答主机样本缓冲如何被搬过物理链路。核心是 zero_copy_if 抽象：以「帧池 + 侵入式引用计数」暴露统一的取帧、还帧契约——上层取帧后直接在传输层内存上读写，指针析构即自动 release 还帧，发送侧真正发包推迟到 release 时执行。具体落地三支：udp_zero_copy 是 boost::asio 可移植版，自承并非真正零拷贝，每次收发仍有一次内核↔用户态拷贝，靠帧池复用与调大内核缓冲（默认约 20ms 数据量）支撑吞吐；usb_zero_copy 基于 libusb 异步批量传输，是预提交一批、回调驱动、条件变量等待的「推」模型；udp_simple 服务控制事务与设备发现。更宏观上传输是个家族：现代 RFNoC 设备数据通路改用 link_if 抽象（udp_boost_asio_link / udp_dpdk_link / nirio_link），与老的 zero_copy_if 并存、都喂给收发流器；设备层用工厂函数指针选型，use_dpdk 软开关配合 HAVE_DPDK 编译期约束决定是否启用 DPDK 内核旁路。本讲引入的关键术语：frame、buffer_pool、managed_buffer、commit/release、侵入式引用计数、link_if、zero_copy_if、DPDK 内核旁路。
