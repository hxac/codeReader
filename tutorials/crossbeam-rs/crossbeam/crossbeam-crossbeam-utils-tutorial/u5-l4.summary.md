本讲是「深入机制与工程实践」单元收官篇，回答充满 unsafe 的并发库如何验证正确性与度量性能。覆盖三块最小模块：测试组织、cfg 门控、基准写法。核心结论：并发 bug 难以直接断言，`tests/` 采用「一类型一文件」，正确性多用间接断言（Drop 计数追踪生命周期、不变量排除撕裂读、通道 try_recv 编排时序）。三件套互补：loom 穷举线程交错但测不了 `AtomicCell`/`ShardedLock`/`thread::scope`（因 `repr(transparent)` 与 loom 内部布局不兼容），Miri 查 UB 但慢需缩迭代，TSan 查真实数据竞争故强制锁回退。关键术语：`cfg`(属性) 与 `cfg!`(宏) 区别、`miri`(rustc 自动)、`crossbeam_loom`(CI 设)、`crossbeam_no_atomic`/`crossbeam_sanitize_thread`/`crossbeam_atomic_cell_force_fallback`(build.rs 发)、`always_use_fallback()` 让断言跨配置自洽。基准需 nightly（`#![feature(test)]`），测「操作×宽度」矩阵与多线程竞争吞吐，并发 bench 关键是 Barrier 对齐 `b.iter` 边界 + `black_box` 防优化。
