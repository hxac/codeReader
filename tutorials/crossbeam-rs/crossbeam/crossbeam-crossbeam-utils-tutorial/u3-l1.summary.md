本讲解析 crossbeam-utils 的 `Parker`/`Unparker` 线程挂起原语，对应 `src/sync/parker.rs` 与 `src/sync/mod.rs` 两个文件。核心是 **token（令牌）模型**：每个 `Parker` 关联一个初始不存在的 token，`park()` 阻塞并消费 token，`unpark()` 幂等地放置 token；`unpark` 先于 `park` 时后者立即返回，从根本上避免丢失唤醒。

内部状态由单个 `AtomicUsize` 三态机驱动：`EMPTY`（无人等、无 token）、`PARKED`（有线程在等）、`NOTIFIED`（token 已放、待消费）。`Parker` 与 `Unparker` 通过 `Arc<Inner>` 共享同一份状态；`Unparker` 可 `Clone`、`Send + Sync`（多唤醒者），`Parker` 仅 `Send`（单一停车者）。

`Inner::park` 走「快路径消费 NOTIFIED → 加锁 CAS 登记为 PARKED → Condvar 等待循环」四段流程，醒来后再次核对状态以过滤**虚假唤醒（spurious wakeup）**。`Inner::unpark` 有两处反直觉设计：用 `swap(NOTIFIED)` 而非 CAS，确保每次调用都产生 release 写以保证内存可见性；`notify_one` 之前先 `lock.lock()` 再 `drop`，与 `Condvar::wait`「释放锁即睡眠」的原子性配合，闭合**丢失唤醒（lost wakeup）**窗口。本讲建立的「原子计数 + Mutex + Condvar」组合与防漏通知直觉将在 u3-l2 WaitGroup 中复现。
