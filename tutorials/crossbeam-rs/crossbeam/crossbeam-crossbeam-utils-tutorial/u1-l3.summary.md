本讲（u1-l3）讲清 crossbeam-utils 的特性与 cfg 在何处定义、如何编译测试。Cargo.toml 只有两个真特性：`std`（默认开）与 `atomic`（默认关，因依赖 atomic-maybe-uninit 单独要求 Rust 1.74，crate 整体 MSRV 为 1.56）。平台相关判断写不进 feature，改由 build.rs 在编译期读 TARGET 查 no_atomic.rs 黑名单，发出 `crossbeam_no_atomic`，并据 sanitizer 发出 `crossbeam_sanitize_thread` 与 `crossbeam_atomic_cell_force_fallback`（任意 sanitizer 强制 AtomicCell 走全局锁回退）；用 no_ 否定式，使 build.rs 未跑时乐观默认为支持原子。验证有三套入口：tests/（正确性，stable）、benches/（性能，需 nightly 的 `#![feature(test)]`）、loom 模型测试（设 `crossbeam_loom`，crate 内 primitive 抽象层在 loom 与标准库实现间二选一，代价是 AtomicCell/ShardedLock/thread::scope 在 loom 下不可用）。关键术语：feature、build.rs、rustc-cfg、MSRV、primitive 抽象层、loom、force_fallback。完整闭环：feature 定义 → build.rs 发 cfg → src 裁剪代码 → tests 用 `cfg!` 消费。
