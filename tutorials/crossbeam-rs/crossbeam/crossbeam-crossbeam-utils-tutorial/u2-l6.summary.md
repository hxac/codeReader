`CachePadded<T>` 是 `crossbeam-utils` 提供的「缓存行填充」包装器，唯一目标是消除 **false sharing（虚假共享）**：当多线程高频访问的、彼此独立的变量恰好落在同一条 64 字节缓存行里时，一核的写会使整条行在其他核失效，导致反复 cache miss 而性能下降。它通过 `#[repr(align(N))]` 把内部唯一字段 `value: T` 撑大到一整条缓存行。`N` 是按 `target_arch` 编译期硬编码的「悲观猜测」——x86-64/aarch64/powerpc64 取 128（防 Intel 空间预取器成对拉取相邻行），arm/mips/sparc/hexagon 取 32，m68k=16，s390x=256，其余兜底 64；不保证等于真实缓存行。大小为「容纳 `T` 的最小 N 整数倍」，对齐为 `max(N, align_of::<T>())`。API 走「透明包装」路线：`new` 是 `const fn`，配 `into_inner`、`Deref`/`DerefMut`、`From`，`Send`/`Sync`/`Clone`/`Copy` 等均原样转发自 `T`，不引入任何同步语义（防 false sharing，不防 data race）。`crossbeam-utils` 内部两处依赖它：`AtomicCell` 的全局锁池 `[CachePadded<SeqLock>; 67]` 与 `ShardedLock` 的 `Box<[CachePadded<Shard>]>`——都是「数组里装多把独立锁」的典型反 false sharing 场景，其中 `new` 的 `const fn` 性质是 `static` 锁池初始化的前提。本讲是后续 ShardedLock 讲义的硬前置。
