本讲只读 `lib.rs` 与 `atomic/mod.rs`、`sync/mod.rs` 两个门面，建立 `crossbeam-utils` 的模块骨架地图，不进入任何类型实现。`lib.rs` 做三件事：声明 `#![no_std]`、定义 `pub(crate)` 内部抽象层 `primitive`（在标准库与 loom 实现间二选一）、声明并门控五个功能模块。导出分两种风格：`pub mod`（`atomic`/`sync`/`thread` 暴露整个命名空间）与私有 `mod` + `pub use`（`Backoff`/`CachePadded` 及 `sync` 下所有类型仅重导出类型到更短路径）。feature 硬规则：`atomic` 模块需 `feature="atomic"`，`sync`/`thread` 需 `feature="std"`，`Backoff`/`CachePadded` 无门控，对应 README 的 <sup>(no_std)</sup> 标记。另有平台/测试维度 cfg：`AtomicCell` 与私有 `seq_lock` 需 `target_has_atomic="ptr"` 且非 loom，`ShardedLock`/`once_lock`/`thread` 需非 loom，而 `AtomicConsume`、`Parker`、`WaitGroup` 在 loom 下仍可用。核心结论：「文件存在」≠「被编译」≠「对外可见」——孤儿文件 `alloc_helper.rs` 无 `mod` 声明故不参与编译。本讲引入的关键术语供后续承接：`primitive` 抽象层、`pub mod` 与 `pub use` 两类导出、feature/cfg 门控、loom 覆盖盲区。
