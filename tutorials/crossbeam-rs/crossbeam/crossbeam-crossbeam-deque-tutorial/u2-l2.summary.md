本讲拆解 Chase-Lev 队列中 owner 线程独占的 `Worker::push` 与 `Worker::pop` 实现（src/deque.rs），是 u2-l1 骨架之后的「主人端操作」专题，把 `resize` 当黑盒、暂不涉及 `Stealer::steal`。

核心结论：**push 与 flavor 无关**——FIFO 与 LIFO 共用同一段代码，永远写 `back` 槽位再推进 `back`，差异只在 pop 从哪端取。push 的内存序骨架是「Acquire 读 front ↔ 写槽位 ↔ Release fence ↔ Relaxed 写 back」，与 stealer 的「Acquire 读 back ↔ 读槽位」严格配对；ThreadSanitizer 模式下因 tsan 不解 fence，改走 Release store 双路径。

**FIFO pop** 用 `fetch_add`（SeqCst）乐观抢占 `front` 槽位，若越界（任务被判空后被偷走）则 `store` 回退 `front` 返回 `None`，否则直接 `assume_init`，并按 `len <= cap/4`（pop 前长度）缩容。**LIFO pop** 更微妙：「乐观减 back → SeqCst fence → 读 front 复核」；当减 1 后 `len == 0` 即「只剩最后一个任务」时，owner 的最末槽位与 stealer 的队头槽位重合，必须用 `compare_exchange(front)` 仲裁唯一赢家，赢家 `assume_init`、输家 `take()`/`Retry` 丢弃，保证恰好消费一次、无 double-free。关键术语：fetch_add、CAS、fence、MIN_CAP=64、tsan 双路径、assume_init/double-free。后续讲义将承接：u2-l3 讲 stealer 的 epoch::pin + 两步偷取，u2-l5 拆解 resize 的 epoch 回收。
