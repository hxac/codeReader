本讲打开 crossbeam-deque 中 `resize` 与 `reserve` 的黑盒，聚焦「容量如何变化、旧内存如何安全退役」，覆盖三个最小模块：容量触发点、`Worker::resize` 内部、`Worker::reserve`。核心结论：队列容量动态管理遵循两条铁律——恒为 2 的幂（保证掩码寻址）、永不低于 `MIN_CAP = 64`。三个触发点为：`push` 写满时翻倍（`2*cap`）；`pop` 后长度跌破 `cap/4` 时缩半（`cap/2`，引入 1/4 滞后阈值以避免抖动）；批量偷取写入前由 `reserve` 按需倍增到「刚好够」。`resize` 固定五步：读窗口（`front` 用 `Relaxed` 取偏旧快照只多拷不漏拷）→ 分配新 buffer → `copy_nonoverlapping` 逐槽拷贝 → `epoch::pin` 后 `swap` 共享指针并换本地缓存 → `defer_unchecked` 延迟释放旧 buffer。关键术语：只有 owner 改 buffer 的「单写者」不变式；位拷贝靠旧 buffer 的 `dealloc` 走 `MaybeUninit` 不跑 `T::drop` 来避免 double-drop；`swap` 后不能立即释放因 stealer 可能正持旧指针，靠 epoch 把释放推迟到所有线程离开临界区；`#[cold]` 把 resize 移出热路径；`guard.flush` 用于大块内存尽早回收。分工上运行期退役 buffer 归 epoch 管，最终存活 buffer 归 `Inner::drop` 管。
