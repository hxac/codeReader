本讲横向串讲 crossbeam-deque 的内存序、fence 与 volatile 设计。先明确 happens-before、数据竞争（data race）、UB 三者关系：无 happens-before 的并发读写即数据竞争、属 UB，可用原子操作、fence 或 release-acquire 配对建立顺序。正确性建立在 Inner 注释引用的三篇论文上：Chase-Lev 原算法、Le 等的弱内存模型修复（要求 steal 中插 SeqCst fence，本讲所有 fence 即源于此）、CDSchecker 模型检验。Buffer 因泛型 T 无法整体原子读写，改用 ptr::write_volatile/read_volatile，注释坦承是「技术性数据竞争、UB」，可见性靠 fence + 原子索引保证；对照 Injector 的 Slot 用原子 state 门控做到严格无 UB。配对链：push 端「写槽位→fence(Release)→back.store(Relaxed)」，steal 端「front.load(Acquire)→fence(SeqCst)→back.load(Acquire)」。fence 总夹在「预约自己端」与「读对端」之间，FIFO pop 因 fetch_add(SeqCst) 内建语义无需单独 fence。
