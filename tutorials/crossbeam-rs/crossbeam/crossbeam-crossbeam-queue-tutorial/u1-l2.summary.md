本讲义从工程层面剖析 `crossbeam-queue` 的目录结构与 crate 入口，仅精读 `src/lib.rs` 与 `Cargo.toml` 的 `[features]` 段。`src/` 下共四个文件：`lib.rs`（crate 根）、`array_queue.rs`、`seg_queue.rs`、`alloc_helper.rs`。核心结论：`lib.rs` 几乎不含算法，只做三件事——写 crate 级 `//!` 文档、设置 `#![no_std]` 等 crate 级属性、用 `mod`+`pub use` 组装公共 API。引入的关键术语包括：crate 根、inner/outer 文档注释（`//!` vs `///`）、intra-doc link、`#![no_std]`、条件化的 `extern crate alloc/std`、Cargo feature 开关、`#[cfg]` 守卫、`target_has_atomic = "ptr"`、`core`/`alloc`/`std` 三层标准库、feature 透传。依赖链为 `default→std→alloc`，且 `std` 连带打开 `crossbeam-utils/std`。所有模块和 `pub use` 共享同一守卫 `#[cfg(all(feature = "alloc", target_has_atomic = "ptr"))]`：`alloc` 保证堆分配可用，指针原子保证无锁算法可用。三种配置中，默认（`std`）与 `--features alloc` 均导出两个队列，而 `--no-default-features` 编译成无导出的空壳（「不被支持」指无功能而非编译失败）。
