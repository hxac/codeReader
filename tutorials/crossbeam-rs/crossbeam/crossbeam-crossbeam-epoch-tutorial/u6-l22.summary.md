本讲是 crossbeam-epoch 的可移植性专题，阐明同一份算法源码如何同时跑在多个编译目标上。它定义四个彼此正交的移植维度：no_std、loom（并发模型穷举检验）、miri（UB 检测）、target_has_atomic（平台原子能力），各用独立 cfg 处理。围绕四个最小模块展开：primitive 抽象层按 cfg 把 loom 与标准库实现导出到统一名字空间，并手写 UnsafeCell 包装补出 with/with_mut，使上层对运行环境无感；const_fn! 宏用一对互斥 cfg 让同一函数签名在 const 与非 const 间展开，绕开 loom 的 AtomicPtr::new 非 const 限制；alloc_helper::Global 是 no_std 下对 alloc 原语的薄封装，正确处理 ZST，被 Pointable 消费；strict provenance 主线中，without_provenance_mut 与 fetch_and/or/xor 在 miri 下走 transmute/AtomicPtr::fetch_*（严格模型），非 miri 下走 as/AtomicUsize cast（兼容 CHERI 与 MSRV 1.74）。引入的关键术语包括 provenance、strict/permissive provenance、CHERI、loom、miri、crossbeam_loom cfg。核心结论：移植性各维度彼此正交，须分别用对应 cfg 处理，且 miri 下统一走更严格的那条实现，使模型检验覆盖最苛刻的内存模型。
