本讲讲解 `crossbeam-channel` 的三个特殊通道 flavor（`At` / `Tick` / `Never`）及其构造函数 `after` / `at` / `tick` / `never`。核心结论：四者都是**只读、无 `Sender`** 的通道，消息「按需生成（materialized on demand）」而非被 `send` 写入，因此不需要 array/list/zero 的队列、`counter.rs` 引用计数或 `Waker` 阻塞者队列，`Drop` 对这三个 flavor 都是空操作、永不真正断开。`after` 是 `at` 的相对时间包装，`after`/`tick` 用 `checked_add` 防溢出，溢出时退化为 `never()`；`never` 是 `const fn` 的零大小（ZST）占位通道。关键术语：`at` 用 `AtomicBool` 的 `swap(SeqCst)` 保证「恰好投递一次」（`Relaxed` 仅为乐观快查），且阻塞 `recv` 在消息被取走后会永久阻塞；`tick` 用 `AtomicCell<Instant>` 的 CAS（`compare_exchange`）周期性推进投递时刻，阻塞 `recv` 中的 `delivery_time.max(now)` 使其**跳过历史拍**而非补发；`never` 在 `select!` 中完全透明（`try_select` 恒 `false`），用于把「可选超时」统一成固定结构的分支。`At`/`Tick` 靠 `Arc` 共享，真实消息类型是 `Instant`，通过 `transmute_copy` 换装为泛型 `T`；`Never` 是零大小。三者都是「时间驱动」而非「事件驱动」，`SelectHandle` 的 `register`/`watch` 直接返回 `is_ready()`，不进入阻塞者队列，靠 `deadline()` 驱动 select 的定时唤醒——这套对接细节留待 `select!` 宏讲义承接。
