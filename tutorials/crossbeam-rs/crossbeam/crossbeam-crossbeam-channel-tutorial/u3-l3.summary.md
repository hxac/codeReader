本讲聚焦 `src/select_macro.rs`，解析 `select!` 宏在编译期如何把源码改写成 Rust 代码。核心结论有三：其一，用户入口 `select!` 与 `select_biased!` 共享同一个内部宏 `crossbeam_channel_internal!`，唯一差别是入口植入的 `const _IS_BIASED: bool`，它透传给 `internal::select`，在 `run_select` 里决定是否 `shuffle`，从而用编译期常量切换公平与有偏。其二，内部宏是带 `@标签` 的递归宏，分两大阶段——解析阶段（`@list`/`@case`/`@list_errorN`）负责把形态各异的分支**归一化**为 `case(args) -> res => { body },` 并用 `compile_error!` 给出友好诊断；代码生成阶段（`@init`/`@count`/`@add`/`@complete`）负责生成实际调用。其三，**单分支优化**：仅「单个 recv + 可选 default」会退化为 `recv()`/`try_recv()`/`recv_timeout()`（用 `map_err` 归一化错误），send 的优化 arm 被注释掉，故带 send 的分支走通用路径。通用路径在栈上建定长 **handle 三元组数组** `_sel`（上限 32），`@add` 逐个注册分支并按 default 形态调用 `internal::select`/`try_select`/`select_timeout`，`@complete` 用 `if $oper.index() == $i` 链分发并完成操作。引入的关键术语包括 **handle 三元组**、**`internal` 模块**、**归一化**、**标签池**、**生命周期擦除**（`transmute`/`unbind`）；生成代码与 `Select` 动态 API 几乎同构。
