本讲义只读 `src/select.rs` 与 `src/context.rs`（辅以 `utils.rs`、`waker.rs`），讲解 `select!` 宏与 `Select` 动态 API 背后唯一的阻塞式内核 `run_select`，以及与之对应的就绪式内核 `run_ready`。核心结论：select 的本质是「多个线程对同一个原子状态做 CAS 抢占，保证至多一个操作胜出」。引入的关键术语有三件套——`Token`（flavor 间的临时数据口袋，连接「抢占」与「搬运」）、`Operation`（用线程本地引用地址当身份证，断言 `>2` 以避让保留值）、`Selected`（塞进单个 `AtomicUsize` 的四态状态机 `Waiting/Aborted/Disconnected/Operation`，所有迁移靠 CAS，是「至多一个赢家」的根本保证）。`SelectHandle` trait 的七个方法（`try_select/register/unregister/accept/deadline` 与 `is_ready/watch/unwatch`）把算法与六种 flavor 解耦。`run_select` 流程为「shuffle → fast path 抢占 → 登记+防漏唤醒复查 → wait_until → 注销 → 醒后 accept」循环，`Timeout` 三态 `Now/Never/At` 让四个公共入口共用实现；`accept` 仅在 `Operation` 态登场接收 `Waker::try_select` 递来的 packet；`Aborted` 有三种来源（非阻塞自标、登记窗口命中、超时）。`run_ready` 只通知就绪、不占座，可能虚假唤醒需调用者重试。公平性由 `utils::shuffle`（Xorshift + Lemire 无除法取模的 Fisher–Yates）提供，`biased` 模式跳过它实现优先级。各 flavor 如何实现这七方法留待 u3-l2，宏拼装留待 u3-l3。
