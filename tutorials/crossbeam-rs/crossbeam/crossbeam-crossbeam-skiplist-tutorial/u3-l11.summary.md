本讲是 base 层核心算法主链路的收尾，全部源自 `src/base.rs`，专门解决「数据如何彻底离开跳表」。核心结论有四：`SkipList::remove` 采用两步协议——先 `try_acquire` 抢引用计数（保证返回的 `RefEntry` 节点活着），再 `mark_tower` 抢删除权（以 level-0 tag 由 0 翻 1 为线性化点，恰一赢家），输掉删除权必须 `decrement` 退计数以免泄漏；赢得删除权后用 `search_position` 留下的 `left` 数组逐层 CAS 物理摘除，某层失败不重试，改用一次 `search_bound` 让读路径的 `help_unlink` 兜底（协作式清理）。`pop_front`/`pop_back` 是 `front`/`back` + `pin` + `RefEntry::remove` 的循环式弹出，靠重试处理「拿到节点却被并发删」的竞态。`clear` 只标记不 `decrement`，摘除委托给下一批 `lower_bound` 的 `help_unlink`，并以 `BATCH_SIZE=100` 分批、批间 `guard.repin()` 避免长时间 pin 卡住（默认全局）Collector 的 epoch 推进。`Drop for SkipList` 与 `IntoIter` 因独占访问可用 `epoch::unprotected()` 直接释放整链，其中 `finalize`（析构+回收）用于仍拥有 key/value 的节点，`dealloc`（仅回收）用于已被 `ptr::read` 搬空的节点。
