本讲剖析 `select!` 宏如何把用户语法翻译成 u3-l9 的 select 运行时调用。核心是内部 helper 宏 `crossbeam_channel_internal!` 以 `@阶段名` 标记驱动的「解析 + 代码生成」两阶段状态机。解析阶段经 `@list`（宽松切分）、`@list_error`（四级诊断）、`@case`（严格校验参数个数）产出规范 case 列表；代码生成阶段经 `@init`→`@count`→`@add`→`@complete` 翻译为可执行代码。关键结论：`@init` 三条单 recv 短路优化直接改写成 `recv`/`try_recv`/`recv_timeout`，跳过整个 Select 机制；case 数量上限为 32（由标签清单 `_oper0.._oper31` 决定）；宏经 `internal` 模块对接三入口 `select`/`try_select`/`select_timeout`，差别仅在传给 `run_select` 的 `Timeout` 枚举（`Never`/`Now`/`At`）；`select!` 与 `select_biased!` 唯一差别是 `_IS_BIASED` 常量（决定是否 `shuffle` 以保证公平）；`SelectedOperation` 是「必须完成」的半成品，靠 `Drop→panic` 与 `recv/send` 内 `mem::forget` 组合约束，宏的 `@complete` 据此设计成「选中即立即完成」。
