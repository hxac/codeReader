本讲拆解 crossbeam-epoch 的**句柄层**，回答 `epoch::pin()` 背后"连着谁"的问题，分三个最小模块。其一，句柄层全景：`Collector` 本质是引用计数的 `Arc<Global>`（持有共享的全局状态：参与者链表、全局垃圾队列、全局 epoch），`LocalHandle` 本质是指向私有参与者状态 `Local` 的裸指针，二者把共享侧与私有侧隔离；`Global`/`Local` 因含裸指针需手写 `unsafe impl Send+Sync`。其二，`register()`：用 `Owned::new(Local)` 在堆上造一个参与者节点，借 `#[repr(C)]`+首字段布局使 `*const Local` 与 `*const Entry` 数值相等，经无锁 CAS 插入 `Global::locals` 侵入式链表；节点自带一份 `Collector` 克隆顶住 Arc 引用，故 drop 原 Collector 后 handle 仍可用；全程只操作新节点，故可用 `unprotected()` 假守卫。其三，`LocalHandle::pin()` 做四件事：`guard_count` 可重入保护（仅 0→1 第一次才实质发布 epoch）、记录 `pinned` 局部 epoch、插入 `SeqCst` fence、每 128 次 pin 周期性 `collect`。核心结论：pin 后的 `SeqCst` fence 是回收安全的命门，它把"发布 pin"钉成对回收方 `try_advance` 全局可见于"后续对数据结构的读"，杜绝 use-after-free；x86 上用 `lock cmpxchg` 替代 `mfence` 作优化。`is_pinned()` 是 crate 外观察 `guard_count` 的唯一窗口。`try_advance` 与 `collect` 的细节留待 u5-l5。
