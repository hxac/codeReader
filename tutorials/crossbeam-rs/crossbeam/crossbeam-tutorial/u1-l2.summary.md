本讲承接 u1-l1 的全景图，拆解 crossbeam 的 Cargo 工程组织，含三个最小模块：workspace 成员与 path 依赖、特性门控的层层传递、CI 与本地构建测试。

仓库是 8 个 crate 组成的 Cargo workspace，共享一份 Cargo.lock 与一组 lint；成员间用 path+version 双重声明互引，并一律先 default-features=false 再按需开特性。主 crate 以 facade（门面）重导出：crossbeam-utils 必选（始终带 atomic），channel/deque/epoch/queue 可选，skiplist 仅是 workspace 成员而非主 crate 依赖。

关键术语：feature 是编译期开关，通过特性连带（如 std = ["alloc", "crossbeam-channel/std"]）与特性门控 #[cfg(feature)] 实现「同一份源码、三级能力」——no_std、alloc、std 由 default→std→alloc 连带链点亮，并跨 crate 层层传递（crossbeam/std → crossbeam-epoch/std → crossbeam-utils/std）。门面据此兑现可见模块：atomic/utils 永远可用，epoch/queue 需 alloc，channel/deque/scope/sync 需 std。

CI 以 -D warnings 把警告当错误，用宽矩阵（MSRV 1.74 / stable / nightly × 多 OS 架构）与 cargo-hack --feature-powerset 枚举特性幂集，并在真实 no_std 目标（thumbv7m/thumbv6m/riscv32i，代表三种原子能力档位）上构建，保证任意特性组合均可编译；本地等价命令为 cargo test --all --all-features --exclude benchmarks。
