本讲义讲解 crossbeam-channel 三种**只读、无发送端**的特殊 receiver flavor：消息（`Instant`）在 `recv` 时由 flavor 自身「物化」产生，而非缓冲外来数据。**`at`/`after`** 在指定时刻恰好投递一次：状态为 `delivery_time` + `received: AtomicBool`，靠 `swap(true, SeqCst)` 的原子 test-and-set 保证全局唯一赢家，输掉竞争的 `recv` 因通道「永不断开」而永久阻塞（`sleep_until(None)`）。**`tick`** 周期投递，用 `AtomicCell<Align<Instant>>` 存「下次投递时刻」，靠 `compare_exchange` 推进节拍；采用 `max(delivery_time, now) + duration` 的「宁可丢拍、也不积压」策略；`Align` 的 16 字节对齐让 `Instant` 在支持 128 位原子的架构上走 AtomicCell 的 lock-free 快路径。**`never`** 永不投递，仅一个 `PhantomData<T>`，`try_select` 恒 `false`、`deadline` 恒 `None`，作为 `select!` 的零开销占位分支，经典用法是 `duration.map(after).unwrap_or_else(never)` 实现「可选超时」。三者通过 `SelectHandle::deadline()` 与 select 协作：select 合并外部 timeout 与各 flavor 声明的就绪时刻取最小值作为 `wait_until` 睡眠上限，实现「到点即醒、无需轮询」。承接的后续讲义为 u3-l9 select 算法与 u3-l10 `select!` 宏。
