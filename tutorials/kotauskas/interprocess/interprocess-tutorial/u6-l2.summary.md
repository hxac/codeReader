本讲承接 u6-l1 的「镜像」规则，剖析 local socket 异步 Listener 与 Stream 的接口契约与平台后端。核心结论：只有会阻塞等待的操作（`accept`、`connect`、`from_options`）才升级为返回 `impl Future`，创建监听器与配置名称回收保持同步。引入的关键术语包括：`traits::tokio::Listener`（`accept` 取 `&self` 故可多任务并发）与 `traits::tokio::Stream`（supertrait 链 `AsyncRead + RefTokioAsyncRead + AsyncWrite + RefTokioAsyncWrite + StreamCommon`）；`bound_util` 借 GAT 把「`&T: AsyncRead`」编码为 `RefTokioAsyncRead` 的 blanket impl，使 `Arc<Stream>` 或 `&Stream` 可在多任务间按引用共享读写，故异步层推荐用 `Arc` 共享而非 `split`。Windows named pipe 后端直接复用 `tokio::net::windows::named_pipe` 原生类型而不重造 reactor：`PipeListener` 用 `tokio::sync::Mutex` 保护「预装膛实例」（accept 即用掉膛里实例并补建新实例），`PipeStream` 走 `MaybeArc` 的拆分升级/重聚降级、drop 时未刷新数据送入 `linger_pool`；读写遵循 Tokio 就绪模型（`try_read`/`try_write` 遇 `WouldBlock` 即 `poll_*_ready` 挂起）。
