本讲（u8-l1）解读 interprocess Windows 后端的内部设施 `linger_pool`（延迟刷新池）与配套的 `needs_flush` 三态脏标记。核心动机是 **limbo 问题**：管道写端若有未读缓冲，直接 `CloseHandle` 会丢数据、让对端收到 `BrokenPipe`/EOF，因此必须「先 `FlushFileBuffers`、后关闭」；但 `FlushFileBuffers` 会阻塞，故 `drop` 不在调用者线程同步刷新，而是用 `ManuallyDrop::take` 取出脏句柄、移交后台池，立即返回。`NeedsFlush` 用单字节 `AtomicEnum` 的 `No`/`Once`/`Always` 三态决定是否进 limbo：`mark_dirty` 推 `No→Once`，`on_clone` 因克隆后无法可靠追踪而钉死 `Always`，`take` 消费 `Once` 或对 `Always` 恒真，兼判脏与实现「连续 flush 第二次是 no-op」。`linger_pool` 是弹性线程池：1 个持久线程值班、临时线程 500ms 空闲退出，靠高水位 64（队列满则带任务开临时线程）与低水位 8（消费后仍积压则拉帮手）做负载跟随。`QueueEnt` 用**低比特标记指针**把裸句柄（依赖 Windows 句柄恒为 4 的倍数）与堆指针（`assert!(align_of>=2)`）压进单个 `*mut ()`。关键术语：limbo、`linger`/`linger_boxed`/`linger_arc`、`HandleFini`、`DynHandleOwner`、`LingerableArc`、`MaybeArc`（留待 u8-l2 承接）。
