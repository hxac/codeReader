本讲把 u4-l1 的「壳/芯」视角平移到 Windows：local socket 在 Windows 上不是 OS 原语，而是 interprocess 在原生 named pipe 之上构造的抽象。公共枚举 `local_socket::{Listener, Stream}` 是「壳」，后端 `os::windows::named_pipe::local_socket::{Listener, Stream}` 是「芯」，由极薄的 `dispatch_sync` 入口粘合——派发链 `create_sync()` → `create_sync_as`（被调用两次：公共枚举一次、后端一次）→ 后端 `from_options` → `CreateNamedPipe`，与 Unix 后端完全对称。`from_options` 把 `ListenerOptions` 翻译成 `PipeListenerOptions`，只映射 `path`、非阻塞的 accept 维度、`security_descriptor` 三项；Unix 专有的 `reclaim_name`/`try_overwrite`/`max_spin_time`/`mode` 在 Windows 上无意义或不编译。关键术语：named pipe（`\\.\pipe\名字`）、pipe instance（待命实例机制）、ConnectNamedPipe。核心技巧：四态 `ListenerNonblockingMode` 被拆两路——accept 维度给底层，stream 维度存进 `AtomicEnum<ListenerNonblockingMode>`，每次 `accept` 后按需事后调整，这正是 Windows 用 `AtomicEnum` 而非 `AtomicBool` 的原因。`Stream` 包装底层 `DuplexPipeStream<Bytes>`（双工字节流），`set_recv_timeout`/`set_send_timeout` 恒返回 `Err(Unsupported)`、`take_error` 恒返回 `Ok(None)`，体现「同接口、异实现」。后续 u4-l3 将钻进 named pipe 黑盒。
