本讲撬开 u4-l2 留下的黑盒，讲解 interprocess 对 Windows 原生 named pipe 的封装（仅同步，异步留待 u6）。核心是两个正交维度：**模式**（`PipeMode` 的字节流/消息流，决定是否保留消息边界）与**方向**，后者由绝对含义的 `PipeDirection` 与相对含义的 `PipeStreamRole` 共同刻画。`PipeMode` 被进一步「常量泛型化」为 `pipe_mode::{Bytes, Messages, None}` 标记类型，充当 `PipeStream<Rm, Sm>` 的泛型参数，使 `send`/`recv_msg` 成为编译期能力；典型别名有 `DuplexPipeStream`、`RecvPipeStream`。

`PipeListener` 是 interprocess 的发明，不对应任何 Win32 对象：它始终预持一个「待命实例」，每次 `accept` 在该实例上等待连接、再补建新实例装回膛里。Windows 特有的「连接即就绪」使客户端连接直接置实例为已连接；若连后断开而服务端不 `accept`，实例沦为 dead-on-arrival 阻塞新连接，需由 `block_on_connect_clearing_empty_conns` 清理。

模式在 Win32 层拆为两个独立标志：`PIPE_TYPE_*`（管道类型，决定写）与 `PIPE_READMODE_*`（读模式，决定读）；服务端创建时一并设置，客户端用 `CreateFile` 打开后默认字节读，须补设消息读。消息模式下 `send` 发一条、`recv_msg` 收一条，边界由内核保留，此即 local socket（恒用字节流）所不具备的能力。本讲还引出 limbo/linger_pool、MaybeArc、FFI 封装层等后续主题。
