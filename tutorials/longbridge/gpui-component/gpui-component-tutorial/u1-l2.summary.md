本讲解决「如何在本地把 gpui-component 跑起来」这一基本问题，围绕四种运行方式展开。首先建立 Cargo Workspace 概念：仓库是多 crate 工作区，根 Cargo.toml 中 `members` 列出全部可编译 crate（含 12 个示例），`default-members` 决定不指定 `-p` 时默认编译谁，`[workspace.dependencies]` 共享依赖版本；正因 `default-members` 含 `crates/story` 且只有它有 main.rs，所以裸 `cargo run` 启动的是 Story Gallery 而非核心库 `ui`。其次介绍 Story Gallery 入口：`crates/story/src/main.rs` 通过 `std::env::args()` 读取首个参数，支持 `cargo run -- <story_name>` 聚焦特定组件，可选名字来自 `gallery.rs` 内的 Story 列表。第三区分 Gallery（大而全的演示集）与 `examples/`（小而专、每个示例只讲一个特性）：此处示例是独立 workspace 成员包，需用 `cargo run -p <包名>` 运行，而非 `--example`。最后讲解 Web 版：`crates/story-web` 以 `crate-type = ["cdylib"]` 编译成 WASM，经 `cargo build --target wasm32-unknown-unknown` 与 `wasm-bindgen` 两步生成 JS 绑定，再由 Bun/Vite 前端加载，桌面与 Web 共用同一 Gallery。关键结论：桌面入口遵循 `application().run` → `init(cx)` → `open_window` → `Root` 包裹首视图这一统一骨架。
