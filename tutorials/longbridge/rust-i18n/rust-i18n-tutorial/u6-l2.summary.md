本讲承接 u6-l1 的 minify_key 算法，讲清其调用方式与用户取值入口。三个最小模块构成调用链：`_minify_key!` 是函数式过程宏，用 syn 按**位置**解析 `(msg, len, prefix, threshold)` 四参，在编译期调用 `minify_key` 把短键坍缩为字面量常量，输入恒定则输出恒定（确定性哈希），运行期零开销；`tkv!` 是 `macro_rules!` 转发壳，仅接收单个 `$msg:literal`，转发到 `i18n!` 编译期生成的内部宏 `__rust_i18n_tkv`（改名 `_rust_i18n_tkv`），返回 `(key, msg)` 元组，其中 key 在编译期确定、msg 恒为原文字面量；该内部宏经 `quote!` 插值把 `i18n!` 的 `len/prefix/thresh` 以字面量透传给 `_minify_key!`，配置在编译期焊死，可用 `RUST_I18N_DEBUG=1` 查看。关键术语：过程宏、声明宏转发壳、配置透传、阈值门控。核心结论：`tkv!` 总是调用 `_minify_key!`，是否压缩仅由 `minify_key` 的阈值判定；而 `t!`（经 `_tr!`）受布尔总开关 `minify_key` 门控——两者用同一套参数对同一字面量产出相同 key，这是与提取器协作（u6-l3）的基础。
