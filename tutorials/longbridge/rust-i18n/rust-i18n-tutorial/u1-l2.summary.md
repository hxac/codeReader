rust-i18n 采用「根 package 兼任 workspace 根」的结构：最外层 `Cargo.toml` 既是发布的 `rust-i18n` 库，又用 `[workspace]` 纳管四个 `crates/` 子 crate 与若干示例，成员共享同一份 `Cargo.lock` 与 `target/`。能力按「阶段」拆成五个 crate：`rust-i18n`（门面）、`rust-i18n-support`（运行时类型，提供 `Backend`、`SimpleBackend`、`AtomicStr`、`MinifyKey`、`CowStr`）、`rust-i18n-macro`（编译期过程宏 `i18n!`/`_tr!`/`_minify_key!`）、`rust-i18n-extract`（提取器库）、`rust-i18n-cli`（`cargo i18n` 命令行）。依赖关系上，根 crate 只依赖 support 与 macro；macro/extract/cli 均依赖 support 的 `codegen` feature，而 `extract`/`cli` 是开发期工具，不被根 crate 依赖，不进用户应用二进制。`rust-i18n-macro` 必须设 `proc-macro = true`，因过程宏 crate 须在编译期介入且不能导出普通 item，这正是它与 support 必须拆开的根本原因。根 `src/lib.rs` 用三类 `pub use`（过程宏标 `#[doc(hidden)]`、运行时类型公开、按 feature 条件再导出如 `try_load_locales`）实现门面模式，让用户只依赖一个 crate 就拿到全部能力；形成「support 定义 → support `pub use` → 根 crate `pub use` → 用户代码与 `i18n!` 生成代码引用」的四跳自洽链路。
