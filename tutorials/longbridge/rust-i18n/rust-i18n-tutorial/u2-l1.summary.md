本讲是「编译期代码生成主链路」的入口，聚焦 `i18n!` 函数式过程宏如何把 token 流解析成参数结构，不涉及文件加载与代码生成（留待 u2-l2、u2-l4）。核心模块是 `crates/macro/src/lib.rs`：入口 `pub fn i18n` 很薄，仅用 `parse_macro_input!(input as Args)` 触发 `Args::parse`，体现「入口只做解析+分发」的设计。`Args` 是宏参数的「内存模型」，承载 locales_path、fallback、minify_key 系列等字段；注意 `backend=` 对应字段名为 `extend`（类型 `Option<syn::Expr>` 保存表达式 AST），而 `default_locale` 只能来自 metadata、不可由宏参数设置。核心结论是参数的**三级优先级**：硬编码默认值 < `Cargo.toml` 的 `[package.metadata.i18n]` < 宏显式参数。`Args::parse` 借助 `lookahead1()` 决定是否消费路径，并通过 `load_metadata`（把 metadata 表头改名后用 serde 反序列化为 `I18nConfig`）无条件先注入中等优先级基线。难点有二：`consume_options` 靠**逗号递归**（试探性 `parse::<Token![,]>`）支持任意数量的命名参数，未知键名落入 `_ => {}` 被静默忽略；`consume_fallback` 用「`if let Ok` 先试单字符串、失败再试数组」的双形态分支，统一收进 `Option<Vec<String>>`。引入的关键术语：过程宏、token 流、`syn`、`ParseStream`、`Args`、`I18nConfig`、三级优先级、逗号递归、fallback 双形态。
