本讲拆解 `t!` 查不到键时的翻译回退机制。回退链（fallback chain）由 `_rust_i18n_try_translate` 编排为三段递进：精确命中 → territory 自动回退 → 显式 fallback 列表，优先级从高到低，任一段命中即返回 `Some`，三段全空才返回 `None`。三个最小模块：总入口 `_rust_i18n_try_translate`（编排者）；territory 削标签器 `_rust_i18n_lookup_fallback`，依 RFC 4647 Lookup 思路用 `rfind('-')` 逐级削掉末尾子标签（subtag）并 `trim_end_matches("-x")` 清理私有标记，无 `-` 时返回 `None`；显式列表静态量 `_RUST_I18N_FALLBACK_LOCALE`，由 `i18n!(fallback=...)` 编译期生成的 `Option<&[&str]>`，运行期 `find_map` 按声明顺序逐个查。关键结论：`t!` 实走 `try_translate`，全 miss 时返回原始消息而非 `"locale.key"`（后者仅 `_rust_i18n_translate` 测试用，二者兜底策略不同）；fallback 只能在 `i18n!` 配置，`t!(..., fallback = "en")` 会被当成普通插值变量的迷惑写法；显式列表不去重 territory 结果，故 `zh` 可能被试两次；不含 `-` 的未知 locale territory 段一步不走，全靠显式列表兜底。
