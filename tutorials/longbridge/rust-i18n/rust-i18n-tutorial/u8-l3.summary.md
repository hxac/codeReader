本讲义讲解在 Cargo workspace 中让多个业务 crate 共享同一份翻译的工程模式，核心模块为 `examples/share-in-workspace`。结论是：把「加载翻译 + 生成静态后端」收拢到一个专职的共享 `i18n` crate，使译文字面量全局只嵌入一次（内存占用由 N×S 降为 S），多 crate 链接进同一二进制时不重复占用 `.rodata`。关键术语与概念包括：`I18nBackend`——无字段单元结构体，实现 `Backend` 并委托给 `i18n` crate 的 `_RUST_I18N_BACKEND`，在 `translate` 中追加 `en` 兜底；`init!` 宏——声明宏，在业务 crate 内展开为 `i18n!(backend = I18nBackend)`，生成 `CombinedBackend(空 SimpleBackend, I18nBackend)`；业务 crate 因不传路径、无 `locales/` 目录，自身译文为空，最终通过转发委托回共享层。重导出方面，`i18n::t!` 与 `rust_i18n::t!` 是同一个转发壳，配合 `[workspace.dependencies]` 保证依赖树版本一致。该模式本质是转发壳与自定义后端两套机制的真实拼装。
