本讲从「工程组织」角度讲清 axum 仓库的三件事：**workspace 配置**、**crate 划分**、**feature flags**。axum 不是一个 crate 而是一个 **workspace**，根 `Cargo.toml` 用 `members = ["axum", "axum-*"]` 通配把 `axum`、`axum-core`、`axum-extra`、`axum-macros` 四个 crate 统一管理，共享同一份 `Cargo.lock`、`target/` 缓存与 lint 规则（成员用 `{ workspace = true }` 继承 `rust-version` 等配置）。四个 crate 按「越底层越稳定、依赖越少」分层：`axum-core` 只依赖 `http`/`tower-service` 等基础 crate，承载 `FromRequest`/`IntoResponse` 等核心 trait，**库作者应优先依赖它**；`axum` 是面向用户的完整框架；`axum-extra` 提供可选增强；`axum-macros` 因过程宏必须独立成 crate。**feature** 是「条件编译 + 条件依赖」开关：`default` 默认开 9 项，故 `Json`/`Form`/`Query`/`serve` 开箱即用，而 `http2`/`ws`/`multipart` 默认关闭以避免拉入 `tokio-tungstenite` 等重依赖。源码通过 `#[cfg(feature = "xxx")]` 真正裁剪代码，关掉 `json` feature 则 `axum::Json` 整个消失；`axum::serve` 的存在性是一个布尔公式：`tokio` 且（`http1` 或 `http2`）。此外 `dep:` 前缀（resolver v2）可防止可选依赖被同名 feature 意外开启。
