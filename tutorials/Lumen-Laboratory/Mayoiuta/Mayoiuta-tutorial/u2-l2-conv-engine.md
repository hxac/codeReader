# 卷积加速引擎 Conv Engine

> 本讲属于「核心计算通路」单元（u2），承接 [u2-l1 处理单元与脉动阵列 PE Array](./u2-l1-pe-array.md)。
> 上一讲我们看清了「格子」——单个 `Processing_Element` 怎么做一次乘加（MAC）。但神经网络里最常见、也最吃算子的运算是**卷积**：用一个 3×3 的小核去扫过整张特征图。本讲就看 NPU 核里专门干这件事的 `Conv_Engine`：它如何把一张 32×32 的特征图「喂」给九点乘累加电路。

## 1. 本讲目标

学完本讲，你应该能够：

- 说清楚**卷积（convolution）**在图像/特征图上到底在做什么，以及为什么它本质是一堆 MAC。
- 读懂 [`conv_engine.v`](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L1-L46) 里的 `Conv_Engine` 模块：滑动窗口、`window_buffer` 缓冲、九点乘累加。
- 解释 **滑动窗口（sliding window）**的缓存思路，以及为什么 `window_buffer` 要做成 3×3。
- 用卷积输出尺寸公式 \(O = \lfloor(W + 2P - K)/S\rfloor + 1\) 手算输出维度，并发现代码里 `conv_result[0:30][0:30]`（31×31）与公式**对不上**的疑点。
- 识别窗口缓冲移位逻辑（`j%3` 那几行）中存疑的写法，并如实标注「待确认」。

## 2. 前置知识

### 2.1 卷积：拿一个「小窗口」扫过整张图

如果你已经学过 [u2-l1](./u2-l1-pe-array.md)，你就知道 MAC 是「一个激活 × 一个权重，再累加」。**卷积**只是把这件事**重复做很多次、并且按位置摆放结果**：

1. 准备一个小的 \(K \times K\) 权重矩阵，叫**卷积核（kernel）**，本讲里 \(K=3\)，即一个 3×3 的核。
2. 把这个核「盖」在特征图的左上角，盖住一个 3×3 的区域——这个被盖住的区域就是一个**窗口（window）**。
3. 把窗口里的 9 个激活，与核里的 9 个权重**一一相乘再相加**，得到**一个数**。
4. 把这个数写进**输出特征图**的对应位置。
5. 把窗口**向右滑动一格**（步长 stride），重复 3、4；滑到行尾就换到下一行，直到扫完整张图。

所以一次 3×3 卷积 = 每个输出位置做 **9 次 MAC**。整张 32×32 的特征图要扫出成百上千个输出点，对应上千组「9 次 MAC」——这正是需要专用硬件加速的地方。

> 术语速查：**kernel**（卷积核/权重矩阵）、**stride**（步长，窗口每次移动几格）、**padding**（补零，在边界补 0 以控制输出大小）、**feature map**（特征图，卷积的输入/输出）、**MAC**（乘加，见 u2-l1）。

### 2.2 输出尺寸公式（本讲会反复用到）

设输入边长为 \(W\)、核大小 \(K\)、步长 \(S\)、补零 \(P\)，则输出边长：

\[
O = \left\lfloor \frac{W + 2P - K}{S} \right\rfloor + 1
\]

直觉：窗口每滑动一步产出一个输出，能滑动的合法位置数就是输出边长。补零 \(P\) 把输入「撑大」了 \(2P\)，核本身要占 \(K\) 个位置，再按步长 \(S\) 离散取点。

本讲会代入 \(W=32, K=3, S=1, P=1\) 来算，并和代码里声明的输出维度对照——你会发现对不上，这正是本讲要揭示的「待确认」之一。

### 4.1 滑动窗口与 window_buffer

#### 4.1.1 概念说明

`Conv_Engine` 要解决的第一个问题是：**九点乘累加需要同时拿到一个 3×3 的窗口，但特征图是一整张 32×32 的大数组，怎么高效地把「当前窗口」凑齐？**

最朴素的办法是「现用现取」：算到位置 \((i,j)\) 时，直接从 `feature_map[i-1][j-1]` …… `feature_map[i+1][j+1]` 读 9 个数。这能工作，但有浪费——**相邻两个窗口之间，有 6 个元素是重复的**。比如窗口从 \((i,j)\) 滑到 \((i,j+1)\)，三行里各有 2 个元素是共用的，只有最右一列的 3 个元素是新的。

所以硬件里常用一个**滑动窗口缓冲（window buffer）**：把当前窗口的 9 个值存在一个小寄存器阵列里，每滑一格，**只移入新的一列、移出最旧的一列**，就像传送带一样。这样每个时钟只需取少量新数据，节省了对大特征图的重复访问。本模块的 `window_buffer` 就是为此而设——它被声明成一个 3×3 的小阵列（见 4.1.3），正好放下一个窗口。

#### 4.1.2 核心流程

从源码结构看，`Conv_Engine` 把整张特征图的卷积**铺展成两层 `for` 循环**：外层 `i` 遍历输出行，内层 `j` 遍历输出列。对每一个 \((i,j)\)，它做两件事：

1. **更新窗口**（[L19–L27](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L19-L27)）：
   - 若 `j == 0`（每行第一个位置）：**从 `feature_map` 直接装填**当前窗口的第 0 列，并用 `(i>0)`、`(i<31)` 三目运算符在上下边界处补 0。
   - 否则（`j > 0`）：**移位**——用 `j%3` 作下标，把缓冲里「前一列」的内容搬到「当前列」。
2. **算卷积**（[L30–L41](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L30-L41)）：当 `j >= 2` 时，把 `window_buffer` 里的 9 个值与 `weight` 里的 9 个权重一一相乘相加，写进 `conv_result[i][j-2]`。

用伪代码画出「窗口随 j 滑动」的意图（每行 j 递增，窗口整体右移一格）：

```
j=0 :  [装填第0列 f(i-1,0) f(i,0) f(i+1,0)]
j=1 :  窗口右移一格 → 应纳入 f(*,1)
j=2 :  窗口右移一格 → 应纳入 f(*,2)；此时 9 格凑齐 → 算第一个输出 conv_result[i][0]
j=3 :  窗口右移一格 → 应纳入 f(*,3)；算 conv_result[i][1]
...
```

> 注意：上面这段是「设计意图」。4.1.3 会指出，**实际的移位代码并没有真正把新列从 `feature_map` 装进来**，所以意图与实现之间存在落差——这是本讲的第一个「待确认」。

#### 4.1.3 源码精读

**① 模块端口与参数** —— [hardware/rtl/core/conv_engine.v:1-11](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L1-L11)

这段是 `Conv_Engine` 的「对外契约」：

- 三个参数：`KERNEL_SIZE=3`、`STRIDE=1`、`PAD=1`（L2–L4）。
- 输入 `feature_map [0:31][0:31]`（L8）：32×32 的特征图，每元素 8 位。
- 输入 `weight [0:2][0:2]`（L9）：3×3 的卷积核，每元素 8 位。
- 输出 `conv_result [0:30][0:30]`（L10）：输出特征图，每元素 16 位（**维度疑点见 4.2.3**）。

> **待确认（第一个：参数被声明却从未使用）**：在整个模块的 `always` 块里，循环边界写死成 `i<=30`、`j<=30`、`j>=2`、`j%3`，窗口阵列写死成 `[0:2][0:2]`，步长则是「`j` 每次 +1」隐式表达。也就是说，**`KERNEL_SIZE`、`STRIDE`、`PAD` 这三个参数在函数体里一次都没出现**（你可以用 `grep "STRIDE\|PAD\|KERNEL_SIZE"` 验证，它们只出现在 L2–L4 的声明处）。改这三个参数**不会改变模块行为**——它们目前是「死参数」。这一点标注为「待确认」：究竟是预留未实现，还是应当参与循环边界与补零逻辑，需要后续维护者明确。

**② 窗口缓冲声明** —— [hardware/rtl/core/conv_engine.v:13-14](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L13-L14)

```verilog
reg [7:0] window_buffer [0:2][0:2];
```

`window_buffer` 是一个 3×3 的寄存器阵列，每个元素 8 位，正好对应一个 3×3 卷积窗口的 9 个激活。后面九点乘累加就读这 9 个格子。

**③ 窗口滑动的两个分支** —— [hardware/rtl/core/conv_engine.v:19-27](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L19-L27)

```verilog
if (j == 0) begin
    window_buffer[0][0] <= (i > 0) ? feature_map[i-1][j+0] : 0;
    window_buffer[1][0] <= feature_map[i][j+0];
    window_buffer[2][0] <= (i < 31) ? feature_map[i+1][j+0] : 0;
end else begin
    window_buffer[0][j%3] <= window_buffer[0][(j-1)%3];
    window_buffer[1][j%3] <= window_buffer[1][(j-1)%3];
    window_buffer[2][j%3] <= window_buffer[2][(j-1)%3];
end
```

- **`j==0` 分支**：每行的第一个位置，从 `feature_map` 装填窗口的**第 0 列**（三行各取一个）。`(i>0)` 与 `(i<31)` 在上下边界处把越界的行补成 0——这是一种**朴素的垂直补零**，呼应了参数 `PAD` 的意图（虽然 `PAD` 本身没被用上）。注意它只装填了 `window_buffer[*][0]` 这一列，其余两列在这一步并未赋值。
- **`j>0` 分支（移位）**：这是本讲最值得推敲的一段。它用 `j%3` 当目标下标、`(j-1)%3` 当源下标，把缓冲「前一列」的内容搬到「当前列」。

> **待确认（第二个：移位逻辑存疑——`j%3` 写法）**：这段移位有两个层面的问题。
>
> 1. **它从不读 `feature_map`**。一个正确的滑动窗口，每滑一格应当「移入新的一列激活」。但 `else` 分支只做 `window_buffer[k][j%3] <= window_buffer[k][(j-1)%3]`——左右两边都是 `window_buffer` 自身，**没有任何一行去 `feature_map` 取新数据**。结果窗口里只有第 0 列（`j==0` 时装的）真正来自特征图，第 1、2 列只是把第 0 列的内容反复搬来搬去。
>
> 2. **`j%3` 让多个 `j` 写同一个下标**。`j%3` 在 `j=0,3,6,...,30` 时都等于 0。由于这段代码位于 `always @(posedge clk)` 的 `for` 循环里，961 次 \((i,j)\) 迭代在**同一个时钟沿**展开；对同一个 `window_buffer[k][0]`，会有多个非阻塞赋值（`j=0` 时来自 `feature_map`，`j=3,6,...,30` 时来自移位），按 Verilog「**最后一条非阻塞赋值胜出**」的规则，最终生效的是循环里**最后一条**——也就是 `j=30` 的移位结果，而 `j=0` 那次从 `feature_map` 的装填**被覆盖、实际不生效**。
>
> 综合这两点：按字面行为，`window_buffer` 很难保证在计算 `conv_result` 时持有「连续三列」的有效窗口数据。这意味着 `conv_result` 的实际数值与「标准卷积」很可能不一致。**该移位逻辑的正确性与本意，标注为「待确认」，实际数值「待本地验证」。** 4.1.4 的实践会带你亲手追踪一遍来体会这一点。

> 小贴士：遇到这类「循环里对同一变量多次非阻塞赋值」的写法，是 Verilog 行为建模里最容易踩坑的地方。学习时不必急着替原作者「修正」，而是先**如实描述字面行为、标出疑点**，再在有 testbench 时用仿真验证。

#### 4.1.4 代码实践（源码阅读型）

> **实践目标**：亲手追踪 `window_buffer` 在固定 `i` 下随 `j` 变化的取值，验证 4.1.3 的两个论断——「`else` 分支不读 `feature_map`」与「`j%3` 多次写同一格」。

**操作步骤**：

1. 固定 `i = 1`（避开上下边界，方便讨论）。准备一张「示意」特征图，只关心第 0、1、2 列的前三行，比如记 `a = f(0,*)`、`b = f(1,*)`、`c = f(2,*)`，并假设 `f(r,0)` 三行分别为 `a0, b0, c0`，`f(r,1)` 为 `a1,b1,c1`，依此类推。
2. 仿照源码 L19–L27，**按字面**写出 `j=0,1,2` 时 `window_buffer[*][0..2]` 分别被赋成什么（注意非阻塞赋值用「旧值」求右值）。
3. 检查：经过 `j=1`、`j=2` 之后，`window_buffer` 的第 1、2 列里，有没有任何一格等于 `f(*,1)` 或 `f(*,2)`？
4. 进阶：注意到对 `window_buffer[0][0]` 而言，`j=0`（来自 `feature_map`）和 `j=3,6,...,30`（来自移位）都写同一格——按「最后赋值胜出」，哪一条最终生效？

**需要观察的现象 / 预期结果**：

- `j=0` 时 `window_buffer` 的第 0 列被装填为 `(a0, b0, c0)`，第 1、2 列保持原值（未赋值）。
- `j=1` 时 `window_buffer[*][1]` 被赋成「旧的 `window_buffer[*][0]`」即 `(a0,b0,c0)`——**和第 0 列一样**，而不是新列 `(a1,b1,c1)`。
- `j=2` 时 `window_buffer[*][2]` 又被赋成「旧的第 1 列」即 `(a0,b0,c0)`。于是三列内容**雷同**，并非真正的连续三列窗口。
- 第 2 列下标 `0`：`j=0` 的 `feature_map` 装填会被 `j=30` 的移位覆盖（最后赋值胜出）。

**预期结论**：按字面行为，`window_buffer` 不能稳定持有有效的 3×3 窗口；移位逻辑的正确性「待确认」，由此算出的 `conv_result`「待本地验证」。

**运行结果**：本实践为源码阅读 / 手算型，无需运行；若想用仿真确认，需自建 testbench（仓库未提供），结果 **待本地验证**。

#### 4.1.5 小练习与答案

**练习 1**：`window_buffer` 被声明成多大？为什么是这个尺寸？

> **答**：`reg [7:0] window_buffer [0:2][0:2]`，是 3×3 阵列，对应 3×3 卷积核所需的 9 个激活窗口。

**练习 2**：`j==0` 分支里的 `(i > 0) ? ... : 0` 起什么作用？

> **答**：当 `i==0`（顶部边界）时，`feature_map[i-1][...]` 会越界，于是用 0 代替——即**在顶部边界补零**。底部 `(i<31)` 同理。这是一种朴素的垂直 padding，但它只在 `j==0` 的第 0 列装填时生效，水平方向（左右边界）并未补零。

**练习 3**：为什么说 `else` 分支「不读 `feature_map`」是个问题？

> **答**：滑动窗口每右移一格，本应纳入特征图的**新一列**激活。但 `else` 分支三行赋值的右值都是 `window_buffer` 自身（`(j-1)%3` 列），没有任何一处读 `feature_map`。结果只有第 0 列来自特征图，第 1、2 列只是第 0 列的复制，无法凑齐真正的 3 连续列窗口。

### 4.2 九点乘累加与输出维度

#### 4.2.1 概念说明

窗口凑齐后，`Conv_Engine` 的第二件事就是把窗口里的 9 个激活与核里的 9 个权重做**点积**——也就是 9 次 MAC 求和：

\[
\text{out} = \sum_{r=0}^{2}\sum_{c=0}^{2} \text{window\_buffer}[r][c] \times \text{weight}[r][c]
\]

这和 [u2-l1](./u2-l1-pe-array.md) 里 `Processing_Element` 的 `0x2`（乘加）是同一类运算，只不过那里是「一个权重、连续多个激活」的流式累加，这里是「9 对一次性相乘相加」的空间点积。两者本质都是 MAC。

**一个数值例子**：设当前窗口与核分别为

\[
\text{window} =
\begin{bmatrix}1&2&3\\4&5&6\\7&8&9\end{bmatrix},
\qquad
\text{weight} =
\begin{bmatrix}1&0&0\\0&1&0\\0&0&1\end{bmatrix}
\]

（核是一个「对角线核」，只挑窗口里 \(1,5,9\) 三个位置），则输出：

\[
\text{out} = 1{\cdot}1 + 2{\cdot}0 + 3{\cdot}0 + 4{\cdot}0 + 5{\cdot}1 + 6{\cdot}0 + 7{\cdot}0 + 8{\cdot}0 + 9{\cdot}1 = 1 + 5 + 9 = 15
\]

这个 15 就被写进输出特征图的对应格子。把整张特征图每个位置都这么算一遍，就完成了卷积。

#### 4.2.2 核心流程

`Conv_Engine` 把「算出每个输出点」放进 `j>=2` 的判断里（[L30–L41](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L30-L41)）：

1. 循环遍历 `i = 0..30`、`j = 0..30`。
2. 仅当 `j >= 2` 时才计算：把 `window_buffer[0..2][0..2]` 与 `weight[0..2][0..2]` 逐对相乘，9 个积相加。
3. 结果写到 `conv_result[i][j-2]`：当 `j=2` 时写到第 0 列，`j=3` 写到第 1 列……`j=30` 写到第 28 列。

这里 `j-2` 的偏移，是因为窗口需要先「攒够」3 列才能算出第一个有效输出（前两个 `j` 在填窗口）。这条流程也直接决定了**实际写出多少个输出列**——见 4.2.3 的维度分析。

#### 4.2.3 源码精读

**① 九点乘累加** —— [hardware/rtl/core/conv_engine.v:30-41](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L30-L41)

```verilog
if (j >= 2) begin
    conv_result[i][j-2] <=
        (window_buffer[0][0] * weight[0][0]) +
        (window_buffer[0][1] * weight[0][1]) +
        ... // 共 9 项，3×3 全展开
        (window_buffer[2][2] * weight[2][2]);
end
```

9 项正好对应 \(3\times3\) 的全部位置。逻辑直白：**窗口 9 格 × 核 9 格，逐对相乘再相加**。注意它依赖 4.1 的 `window_buffer`——如果窗口内容不正确（见 4.1.3 的「待确认」），这里的和自然也不正确。

> **关于位宽（与 u2-l1 呼应的简化）**：`window_buffer` 与 `weight` 每个元素都是 8 位，单次乘积最多 16 位；9 个这样的积相加，最大可达 \(9 \times 255^2 = 585225\)，需要 20 位才能不溢出（\(2^{19}=524288 < 585225 < 2^{20}=1048576\)）。而 `conv_result` 每元素只有 16 位（[L10](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L10)），于是高位被**截断**。这与 [u2-l1](./u2-l1-pe-array.md) 里 `Processing_Element` 的累加器截断是同一类简化：省面积、省功耗，但累加较大时存在溢出风险，真实 NPU 通常会配 requantization（定点重缩放）或更宽的累加器。

**② 输出维度：31×31 与公式对不上（第三个待确认）**

输出数组声明在 [L10](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L10)：

```verilog
output reg [15:0] conv_result [0:30][0:30]  // 输出特征图
```

`[0:30][0:30]` 是 **31×31**。代入本讲 2.2 的公式（\(W=32, K=3, S=1\)）：

- 若按参数 `PAD=1`：\(O = (32 + 2\times1 - 3)/1 + 1 = 31 + 1 = \mathbf{32}\) → 期望 **32×32**；
- 若按不补零 `PAD=0`：\(O = (32 - 3)/1 + 1 = \mathbf{30}\) → 期望 **30×30**。

**31×31 既不等于补零的 32，也不等于不补零的 30**，与任一标准公式都对不上。此外，循环里真正写入的只是 `conv_result[i][j-2]`，其中 `j=2..30`，即列下标 `0..28`——**只写了 29 列**，声明的第 29、30 两列从未被赋值（仿真里会是 `x` 不定态）。综上，输出维度「31×31」与写入范围（29 列有效）都存在不一致，标注为「待确认」。

> 一句话归纳本讲三处「待确认」：① `KERNEL_SIZE/STRIDE/PAD` 三个**参数声明却未使用**；② `j%3` **移位逻辑不读 `feature_map`、且多赋值同一格**；③ 输出 **31×31 与公式不符、且只写 29 列**。它们都不影响「读懂模块想做什么」，但会影响「模块能否算出正确卷积」——后者需 testbench 验证，待本地验证。

#### 4.2.4 代码实践（手算型，主实践）

> **实践目标**：用公式手算输出尺寸，并与代码声明的 `conv_result[0:30][0:30]` 对照，发现维度落差；同时回顾位宽简化。

**操作步骤**：

1. 代入 \(W=32, K=3, S=1, P=1\)，用公式 \(O = \lfloor(W+2P-K)/S\rfloor + 1\) 算出输出边长。
2. 再算一遍 \(P=0\)（不补零）的情况。
3. 与代码 [L10](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L10) 声明的 31×31 对比，指出差了几个像素。
4. 对照循环写入范围（`conv_result[i][j-2]`，`j=2..30` → 列 `0..28`），指出哪几列**永远不会被写**。

**需要观察的现象 / 预期结果**：

- `PAD=1` → 32；`PAD=0` → 30；代码声明 31。**31 与两者都差 1**。
- 列方向实际写入 `0..28` 共 29 列；声明的第 29、30 列从不被赋值（不定态）。
- 若 9 个最大 8 位输入相乘相加，结果可达约 58 万，需 20 位；输出 16 位会截断高位。

**预期结论**：输出维度声明与卷积公式不一致、与实际写入范围也不一致；加之参数未启用、移位存疑，整个 `Conv_Engine` 的数值正确性「待本地验证」。本步不需要运行仿真，纯手算即可得出上述对照。

#### 4.2.5 小练习与答案

**练习 1**：3×3 卷积、stride=1、不补零，输入 32×32，输出多大？

> **答**：\(O = (32 - 3)/1 + 1 = 30\)，输出 30×30。

**练习 2**：代码里输出声明成 31×31，但只写了 `j-2`（`j=2..30`）这 29 列，那么第 29、30 列在仿真里是什么值？

> **答**：这两列从未被赋值，仿真里是 **不定态 `x`**（`reg` 未初始化的默认行为）。这进一步说明输出维度声明与实际行为不一致，待确认。

**练习 3**：9 个 8 位乘积相加，最多需要多少位才不溢出？`conv_result` 给了多少位？

> **答**：最多需要 20 位（\(9 \times 255^2 = 585225\)，介于 \(2^{19}\) 与 \(2^{20}\) 之间）；`conv_result` 每元素只有 16 位，会**截断高位**，存在溢出风险。

## 5. 综合实践：给 window_buffer 写一份「能读特征图」的移位逻辑

> 这个任务把 4.1（窗口）和 4.2（乘累加）串起来：你将写一份**修正意图**的窗口缓冲（示例代码），再手算一个输出点，理解「为什么原代码算不出标准卷积」。
>
> ⚠️ 本实践**不修改仓库源码**。下面的代码是**示例代码**，请在你自己的临时副本或草稿文件中尝试，结果 **待本地验证**。

**任务背景**：4.1.3 指出原 `else` 分支不读 `feature_map`，导致窗口只有第 0 列有效。经典做法是用**行缓冲 / 移位寄存器**：每来一个新列，把新列的 3 个激活装入最右列，旧两列整体左移一格。

**第 1 步——写一份「移入新列」的示例逻辑**（仅示意思路，未含完整边界保护）：

```verilog
// 示例代码：窗口右移一格，并把新的一列 feature_map 装入最右列
// 行 row 的三列：[0]最旧 → [1] → [2]最新
window_buffer[0][0] <= window_buffer[0][1];                 // 旧[1] → 新[0]
window_buffer[0][1] <= window_buffer[0][2];                 // 旧[2] → 新[1]
window_buffer[0][2] <= feature_map[i-1][j+2];               // 新列(上)
window_buffer[1][0] <= window_buffer[1][1];
window_buffer[1][1] <= window_buffer[1][2];
window_buffer[1][2] <= feature_map[i][j+2];                 // 新列(中)
window_buffer[2][0] <= window_buffer[2][1];
window_buffer[2][1] <= window_buffer[2][2];
window_buffer[2][2] <= feature_map[i+1][j+2];               // 新列(下)
```

对比原代码：这里**每一行都从 `feature_map` 读一个新值**，保证最右列始终是最新纳入的激活；左移则让三列始终是「连续三列」。注意 `feature_map[i±1][j+2]` 仍需边界保护（顶部/底部/右端越界时补 0），这与 `PAD` 参数的用途对得上。

**第 2 步——手算一个输出点**：设窗口与核如 4.2.1 的例子（对角线核），手算结果应为 15。

**第 3 步——对照原代码**：在原代码的 `j%3` 移位下，`window_buffer` 三列内容雷同（见 4.1.4），用同样的核算出来**不会**是 15——这就是「待确认」带来的数值差异。

**预期结果**：示例逻辑能让窗口持有连续三列激活，从而算出标准卷积值（如 15）；原代码因不读 `feature_map` 而算不出该值。两者差异 **待本地验证**（建议用 Icarus Verilog / Verilator 自建 testbench 跑一遍对照）。

## 6. 本讲小结

- **卷积**本质是一组 MAC：用 3×3 核扫过特征图，每个窗口位置做 **9 点乘累加**，写一个输出点。`Conv_Engine` 就是干这件事的专用电路。
- [`Conv_Engine`](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L1-L46) 分两步：用 `window_buffer`（3×3 阵列）缓存当前窗口，再做九点乘累加写入 `conv_result`。
- 滑动窗口的思路是「相邻窗口共享 6 个元素，只移入新列」；但代码的 `else` 分支（[L23–L27](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/core/conv_engine.v#L23-L27)）**只移缓冲、不读 `feature_map`**，且 `j%3` 多次写同一格，导致窗口内容存疑。
- 输出维度用公式 \(O=\lfloor(W+2P-K)/S\rfloor+1\) 衡量：`PAD=1` 应得 32、`PAD=0` 应得 30，而代码声明 **31×31**，且实际只写 **29 列**——与公式和写入范围都不一致。
- 本讲如实标注 **三处待确认**：① `KERNEL_SIZE/STRIDE/PAD` **参数未使用**；② `j%3` **移位逻辑存疑**；③ 输出 **31×31 维度不符**。另指出九点累加被**截断到 16 位**，是与 u2-l1 同类的简化。
