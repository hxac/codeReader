## 2. 前置知识

### 2.1 承接前讲

- **u1-l2 / u1-l3** 告诉我们：RTL 源码按六大功能域分目录（top / core / memory / power / sparse / control），顶层 `NPU_SOC` 是装配原点；同时仓库里有一批"被引用但未提供源码或未被例化"的子模块，统一标注为**待确认**。本讲的 `DVFS_Controller` 就属于 `power/` 域，且同样**未被例化进 SoC**。
- **u2-l1** 引入了 MAC（乘加）与固定功能硬件的概念：NPU 算得快是因为硬件专为 AI 运算定制。本讲从"算得快"转向"**算得省**"——同样的硬件，如何根据负载高低动态调节能耗。

### 2.2 三个通俗概念

| 概念 | 通俗解释 |
|------|----------|
| **workload（工作量）** | 一个 0~255 的整数，描述"当前这颗 NPU 有多忙"。0 表示几乎闲置，255 表示满负荷。 |
| **电压 / 频率档位** | 芯片可以工作在几组不同的"电压+频率"组合上，就像风扇的低/中/高档。低档省电但慢，高档快但费电。 |
| **移动平均（moving average）** | 不直接看"这一拍"的 workload，而是看"最近 8 拍"的平均值，避免被偶发的尖峰带节奏。 |

### 2.3 一条关键物理直觉（不必背公式，理解结论即可）

芯片的**动态功耗**近似满足：

\[
P_{\text{dyn}} \;\propto\; C \cdot V^{2} \cdot f
\]

其中 \(V\) 是供电电压，\(f\) 是时钟频率。注意 \(V\) 是**平方**关系——电压降一点，功耗降很多。所以"想省电，先降电压"。

但降电压有代价：电压越低，逻辑门切换越慢（门延迟随电压下降而上升）。频率必须同步降低，否则电路来不及在 一个时钟周期内完成计算，就会算错。**这就是为什么本模块把 `voltage_level` 和 `freq_level` 成对切换**（`4'b01_01`、`4'b10_10`、`4'b11_11`），而不是只调其中一个。

## 4. 核心概念与源码讲解

本讲把 `DVFS_Controller` 拆成三个最小教学模块：**4.1 动机与整体流程**、**4.2 workload 采样与移动平均滤波**、**4.3 四档调节策略与 case 语法陷阱**。

### 4.1 DVFS 动机与模块整体流程

#### 4.1.1 概念说明

**DVFS（动态电压频率调节）** 是一种按需调节能技术：当 NPU 负载轻时，主动降低电压和频率以省电；负载重时，再拉高电压和频率以保证算力。它与 u4-l1 的稀疏计算是互补的两条节能路线——稀疏计算是"**跳过无用的零**"来省功，DVFS 是"**跑得慢一点、供得低一点**"来省功。

为什么要"动态"？因为 NPU 的负载是时变的：有时在跑大模型推理（重负载），有时在等数据（轻负载）。如果一直跑在最高电压频率，轻负载时就白白费电；如果一直跑在最低档，重负载时又算不过来。DVFS 让硬件在两者间**随负载切换**。

#### 4.1.2 核心流程

`DVFS_Controller` 的数据流是一条直线，三个阶段：

```
workload[7:0]  ──►  [8 级移位寄存器]  ──►  [8 点求和 >>3]  ──►  [case 分档]  ──►  voltage_level / freq_level
  (当前负载)         history_buffer        moving_avg          (00/01/10/11)        (成对输出)
```

伪代码描述：

```
每个时钟上升沿：
    把 workload 推进 8 级移位寄存器（最老的一拍被挤出）
组合逻辑：
    moving_avg = (最近 8 拍 workload 之和) / 8          // 平滑后的负载
    根据 moving_avg 落在哪个区间，选一档 (voltage, freq) 输出
```

关键设计选择：**采样与滤波用寄存器（时序逻辑 `<=`），分档决策用组合逻辑（`always @(*)`）**。这样 `moving_avg` 永远反映"上一拍结束时的缓冲状态"，输出随平均负载即时变化。

#### 4.1.3 源码精读

整体三段式结构一目了然（[dvfs_ctrl.v:12-31](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/power/dvfs_ctrl.v#L12-L31)）：

- **第一段（L12-15）**：`history_buffer` 移位寄存器，每个时钟推入一拍 `workload`。
- **第二段（L18-21）**：`moving_avg` 用连续赋值算 8 点平均。
- **第三段（L24-31）**：`case` 按 `moving_avg` 分档输出。

```verilog
reg [15:0] history_buffer [0:7];              // 8 个槽位，每槽 16 位
always @(posedge clk) begin
    history_buffer <= {history_buffer[6:0], workload};   // 推入最新一拍
end
```

> 这段是本模块**最值得警惕**的一处（详见 4.2.3）：`history_buffer` 被声明为**非压缩数组**（memory），却用 `history_buffer[6:0]` 这种"位选/部分选择"写法去整体拼接——这在标准 Verilog 中并不合法，属于"意图清晰、写法待修正"的骨架代码。

#### 4.1.4 代码实践（功耗估算型）

**实践目标**：用 \(P \propto V^{2} f\) 直观感受"降电压"的节能杠杆。

**操作步骤**：

1. 假设最高档为 \(V=1.2\,\text{V}\)、\(f=1.2\,\text{GHz}\)，最低档为 \(V=0.8\,\text{V}\)、\(f=500\,\text{MHz}\)。
2. 分别计算两档的相对功耗比（把常数 \(C\) 约掉）：

\[
\frac{P_{\text{low}}}{P_{\text{high}}}=\frac{0.8^{2}\times 0.5}{1.2^{2}\times 1.2}
\]

3. 算出比值。

**预期结果**：

\[
\frac{0.64\times 0.5}{1.44\times 1.2}=\frac{0.32}{1.728}\approx 0.185
\]

即最低档功耗约为最高档的 **18.5%**——降档后功耗降到不到五分之一。这正是 DVFS 的价值所在。

> 待本地验证：以上为按公式的手算估算，真实芯片还受漏电功耗、电压转换器效率、切换能耗等影响，结论仅供量级参考。

#### 4.1.5 小练习与答案

**练习 1**：为什么本模块把 `voltage_level` 和 `freq_level` 成对切换（如 `4'b10_10`），而不是只降电压保留高频？

**参考答案**：降低电压会使逻辑门延迟变大（门切换变慢）。若此时仍维持高频，电路在一个周期内来不及完成计算，会出现时序违例、算错结果。因此频率必须随电压一起下调，保证时序收敛；反之升压时也可升频提升算力。

**练习 2**：`DVFS_Controller` 在整个 SoC 中处于什么位置？它的输入 `workload` 由谁产生？

**参考答案**：它属于 `power/` 能效域，理论上应作为 SoC 的"功耗调度器"接收负载反馈。但仓库里 `npu_soc.v` 并未例化它，`workload` 也没有任何生产者模块——所以目前它只是一个**待集成的独立蓝图**，`workload` 的来源待确认。

### 4.2 workload 采样与 8 级移动平均滤波

#### 4.2.1 概念说明

`workload` 是"**这一拍**"的瞬时负载。但 DVFS **不应该**对每一拍的瞬时值都做出反应，原因有二：

1. **电压切换有代价**：片上稳压器（PMIC）改变电压需要若干微秒，切换过程本身耗能。如果负载抖一下就调一次，调节开销可能超过节能收益。
2. **负载会尖峰**：某一拍突然 255、下一拍又回 10，这是噪声，不代表持续高负载。

解决办法是**移动平均（moving average）**：用一个深度为 8 的窗口，输出最近 8 拍的平均值。只有**持续**的负载变化才会显著移动平均值，从而触发档位切换。这就是"平滑（smoothing）"。

#### 4.2.2 核心流程

数学上，8 点移动平均为：

\[
\overline{w}[n]=\frac{1}{8}\sum_{k=0}^{7} w[n-k]
\]

其中 \(w[n]\) 是第 \(n\) 拍的 workload。硬件实现用**移位寄存器**保存最近 8 拍样本，求和后右移 3 位（\(\div 8\)）：

```
每个时钟：
    寄存器整体右移一格，最新 workload 进入 [0]，最老的从 [7] 丢弃
组合逻辑：
    sum = buffer[0]+buffer[1]+...+buffer[7]
    moving_avg = sum >> 3      // 等价于 sum / 8，向下取整
```

**窗口长度的权衡**（本讲综合实践的关键）：

- 窗口越长（如 16、32）：平滑能力越强，越能抑制噪声；但**响应越迟钝**——真正持续的高负载需要更多拍数才能把平均值推过阈值，存在**调节滞后**。
- 窗口越短（如 2、4）：响应快，但容易**抖动（thrashing）**——档位频繁来回切换，既费电又磨损稳压器。

#### 4.2.3 源码精读

移位寄存器声明与推入（[dvfs_ctrl.v:12-15](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/power/dvfs_ctrl.v#L12-L15)）：

```verilog
reg [15:0] history_buffer [0:7];
always @(posedge clk) begin
    history_buffer <= {history_buffer[6:0], workload};
end
```

这里有三处待确认 / 待修正，必须如实指出：

1. **`history_buffer[6:0]` 是非法的部分选择**。`history_buffer` 用 `reg [15:0] ... [0:7]` 声明，是"8 个 16 位字"组成的**非压缩数组**（unpacked array / memory）。标准 Verilog-2001 不允许对非压缩数组做 `array[6:0]` 这种切片再拼接。作者的**意图**显然是"取低 7 个槽位、前面补 workload 凑成 8 个"做移位，但写法不成立。正确写法应是用 `for` 循环逐字搬移，或改用打包（packed）的移位寄存器（见 4.2.4 示例代码）。
2. **槽位宽度 16 位 vs workload 8 位**：每个槽位 16 位，但存入的 `workload` 只有 8 位，高 8 位恒为 0，浪费了一半寄存器。不影响功能，但属资源浪费。
3. **`rst_n` 从未被使用**：端口列表里有 `rst_n`，但两个 `always` 块都不含复位分支。上电后 `history_buffer` 初值为 `x`（不定），在缓冲被填满之前的若干拍，`moving_avg` 也是 `x`，case 会落到 `default`（详见 4.3）。一个可综合设计应在上电时把缓冲清零。

求和与平均（[dvfs_ctrl.v:18-21](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/power/dvfs_ctrl.v#L18-L21)）：

```verilog
wire [10:0] moving_avg;
assign moving_avg = (history_buffer[0] + history_buffer[1] + history_buffer[2] +
                    history_buffer[3] + history_buffer[4] + history_buffer[5] +
                    history_buffer[6] + history_buffer[7]) >> 3;
```

数值核对：8 个槽位每个最大 255，和最大 \(8\times255=2040\)，需要 11 位（\(2^{11}=2048\)）。`>> 3` 后最大 \(2040/8=255\)。所以 `moving_avg` 取值范围是 **0~255**，与下方 case 的阈值域（0~255）对得上。中间求和在 16 位宽度完成（不会溢出），再右移、截到 11 位——比必要的宽一点但无害。

#### 4.2.4 代码实践（修正型源码阅读实践）

**实践目标**：把"意图清晰的骨架"改写成可综合的移位寄存器，并体会窗口长度对滞后性的影响。

**操作步骤**：

1. 阅读上面的 L12-15，确认 `history_buffer[6:0]` 写法待修正。
2. 用下面的**示例代码**（非仓库原有代码）替换移位逻辑，改用显式 `for` 循环逐字搬移，并补上 `rst_n` 复位：

```verilog
// 示例代码：可综合的 8 级移位寄存器（替换 L12-15）
integer i;
always @(posedge clk) begin
    if (!rst_n) begin
        for (i = 0; i < 8; i = i + 1)
            history_buffer[i] <= 16'd0;
    end else begin
        for (i = 7; i > 0; i = i - 1)
            history_buffer[i] <= history_buffer[i-1];   // 整体后移
        history_buffer[0] <= {8'd0, workload};          // 最新值进 [0]
    end
end
```

3. 把窗口长度从 8 改成 16（声明 `history_buffer [0:15]`，求和改成 16 项、`>> 4`），思考：一个**持续**的高负载现在需要多少拍才能把 `moving_avg` 推过阈值？

**需要观察的现象 / 预期结果**：

- 窗口 = 8 时，持续满载 `workload=255`，约需 8 拍 `moving_avg` 才逼近 255。
- 窗口 = 16 时，同样满载约需 16 拍——**响应滞后翻倍**，但单拍尖峰对平均值的影响减半，更不易误触发档位切换。

> 待本地验证：以上拍数为按移动平均定义的手算结论，建议自建一个简单 testbench 喂入阶跃 `workload`，打印 `moving_avg` 观察收敛过程。

#### 4.2.5 小练习与答案

**练习 1**：`moving_avg` 为什么用 `>> 3` 而不是直接写 `/8`？

**参考答案**：两者语义等价（都除以 8、向下取整），但 `>> 3` 是用移位实现除法，综合后是一根硬连线移位，不占除法器资源、时序更好；`/8` 在某些综合工具下也可能被优化成移位，但 `>> 3` 意图更明确、可移植性更好。

**练习 2**：如果 `workload` 长期稳定在 100，但偶尔单拍尖峰到 255，窗口=8 时 `moving_avg` 最多被抬高多少？

**参考答案**：单拍尖峰贡献 \(255/8\approx 31.9\)，其余 7 拍仍是 100（贡献 \(700/8=87.5\)）。尖峰那一拍 `moving_avg` 约 \(87.5+31.9\approx 119\)（粗算），下一拍尖峰移出后又回落到 100 附近。可见移动平均把"255 的尖峰"压缩成了"约 +20 的扰动"，有效抑制了抖动。

### 4.3 四档调节策略与 case 语法陷阱

#### 4.3.1 概念说明

平滑后的 `moving_avg`（0~255）需要被映射到一个具体的档位。本模块采用**四档**策略，电压与频率各用 2 位编码（`00`/`01`/`10`/`11`）：

| `moving_avg` 区间（意图） | `voltage_level` | `freq_level` | 电压（注释） | 频率（注释） |
|---------------------------|-----------------|--------------|--------------|--------------|
| 0 ~ 50（轻载） | `00` | `00` | 0.8 V | 500 MHz |
| 51 ~ 100 | `01` | `01` | （中间档） | （中间档） |
| 101 ~ 150 | `10` | `10` | （中间档） | （中间档） |
| 151 ~ 255（重载，default） | `11` | `11` | 1.2 V | 1.2 GHz |

> 待确认：代码注释只给了**端点**（`00:0.8V ... 11:1.2V`、`00:500MHz ... 11:1.2GHz`），中间两档（`01`、`10`）对应的精确电压/频率**未在文件中给出**。若按等分推测，电压约为 0.8/0.93/1.07/1.2 V，频率约为 500/733/967/1200 MHz，但这是推测，需对照真实寄存器手册确认。

设计直觉：负载越高 → 越需要高电压高频来保证算力；负载越低 → 越应该降档省电。`default` 覆盖 151~255 这一段（105 个值），是四个区间里最宽的——意味着只要负载偏重，就倾向于拉满性能。

#### 4.3.2 核心流程

分档是**纯组合逻辑**（`always @(*)`），输入 `moving_avg`，输出 `{voltage_level, freq_level}` 拼成 4 位。逻辑意图：

```
if      (moving_avg 在 0..50)   → 4'b00_00
else if (moving_avg 在 51..100) → 4'b01_01
else if (moving_avg 在 101..150)→ 4'b10_10
else                            → 4'b11_11
```

下面会看到，作者用 `case` + `0-50:` 的写法试图表达"区间"，但这并非标准 Verilog 的区间语法，导致实际行为与意图不符。

#### 4.3.3 源码精读

调节策略（[dvfs_ctrl.v:24-31](https://github.com/Lumen-Laboratory/Mayoiuta/blob/100706ee7dbee8ff5b05d0ca1fcecc887b38187a/hardware/rtl/power/dvfs_ctrl.v#L24-L31)）：

```verilog
always @(*) begin
    case(moving_avg)
        0-50:   {voltage_level, freq_level} = 4'b00_00;
        51-100: {voltage_level, freq_level} = 4'b01_01;
        101-150:{voltage_level, freq_level} = 4'b10_10;
        default:{voltage_level, freq_level} = 4'b11_11;
    endcase
end
```

**这是本模块最严重的语法陷阱，必须讲透。**

在标准 Verilog 中，`case` 的分支项是**精确匹配单个值或逗号分隔的若干值**（如 `0, 1, 2:`），**并不支持 `0-50:` 这种"区间"写法**。这里的 `0-50` 会被解析为**表达式 `0 - 50`（减法）**，结果是 `-50`。同理：

- `0-50`  → \(-50\)
- `51-100` → \(-49\)
- `101-150` → \(-49\)（注意：\(101-150=-49\)，与上一行**撞值**）

`case` 做的是**按位模式精确比较**。`moving_avg` 是 11 位无符号数，取值 0~255；而 `-50`、`-49` 作为有符号数截到 11 位后分别是 \(2048-50=1998\)、\(2048-49=1999\)，**远大于 255**。于是三个显式分支**永远不会命中**，程序每次都落到 `default`：

\[
\boxed{\text{结论：实际运行时 } \{voltage\_level, freq\_level\} \text{ 恒为 } 4'b11\_11 \text{（最高档）}}
\]

也就是说，**调节策略整段失效**——无论负载多低，输出永远是最高电压、最高频率，DVFS 的节能效果完全无法体现。这正是学习目标里"识别 case 区间语法的待确认/非标准写法"所指。

**正确的区间匹配写法**应当用 SystemVerilog 的 `case ... inside` 配合集合区间 `[lo:hi]`，或退化成 `if-else if` 链（见 4.3.4 示例代码）。

补充：`voltage_level` / `freq_level` 声明为 `output reg`，由组合 `always @(*)` 驱动，**未寄存**——只要 `moving_avg` 变，输出立刻变。这与 4.1 的"采样时序、决策组合"分工一致。

#### 4.3.4 代码实践（本讲核心实践任务）

**实践目标**：把失效的四档 `case` 改写为**正确的三档调节**，并说明移动平均窗口长度对调节滞后性的影响。

**操作步骤**：

1. **重新设计三档阈值**。把 0~255 切成三段，例如：
   - `0 ~ 85` → 第 0 档（最低：0.8 V / 500 MHz）
   - `86 ~ 170` → 第 1 档（中间：1.0 V / 850 MHz）
   - `171 ~ 255` → 第 2 档（最高：1.2 V / 1.2 GHz）
2. **用 `if-else if` 改写**（最稳妥、可综合，不依赖 SystemVerilog）。下面是**示例代码**（非仓库原有代码），替换 L24-31：

```verilog
// 示例代码：三档调节，使用 if-else if 表达区间
always @(*) begin
    if (moving_avg <= 85)        {voltage_level, freq_level} = 2'b00_00; // 最低档
    else if (moving_avg <= 170)  {voltage_level, freq_level} = 2'b01_01; // 中间档
    else                         {voltage_level, freq_level} = 2'b10_10; // 最高档
end
```

> 注意：上面拼接写成了 `2'b00_00` 等（4 位），`{voltage_level, freq_level}` 正好 4 位，与原码一致；三档只用 `00`/`01`/`10`，`11` 不再出现。

3. **（可选）用 SystemVerilog 的 `case ... inside` 表达同样语义**，更贴近作者原意：

```verilog
// 示例代码（SystemVerilog）：用 inside + 区间集合
always @(*) begin
    case(moving_avg) inside
        [0:85]:    {voltage_level, freq_level} = 4'b00_00;
        [86:170]:  {voltage_level, freq_level} = 4'b01_01;
        default:   {voltage_level, freq_level} = 4'b10_10;
    endcase
end
```

4. **回答窗口长度问题**：把 4.2.4 中窗口从 8 改为 16 后，思考调节滞后。

**需要观察的现象 / 预期结果**：

- 改写后，给一个 `moving_avg=40`，输出应为 `4'b00_00`（最低档），而原代码此时会错误地输出 `4'b11_11`。
- **窗口长度对调节滞后性的影响**：移动平均的输出可视为对负载的**低通滤波**。窗口越长（如 8→16），截止频率越低、平滑越强，但阶跃响应的**建立时间**随之变长——持续的高负载需要更多拍数才能把 `moving_avg` 推过阈值，**档位切换更滞后**；窗口越短则响应越快，但更易因噪声在档位间来回抖动。三档相比四档，每档覆盖的负载区间更宽（约 85宽 vs 50宽），切换次数减少，能进一步降低抖动，但档位间的粒度变粗、节能精度下降。

> 待本地验证：建议自建 testbench，让 `workload` 在 30↔200 之间阶跃跳变，分别用窗口 4 / 8 / 16 打印 `{voltage_level, freq_level}` 随时间的变化曲线，直观对比滞后与抖动。

#### 4.3.5 小练习与答案

**练习 1**：原码 `case` 中，为什么说三个显式分支"永远不会命中"？请用具体数值说明。

**参考答案**：`0-50`、`51-100`、`101-150` 在 Verilog 里是减法表达式，结果为 \(-50\)、\(-49\)、\(-49\)。截到 11 位无符号后分别是 1998、1999、1999，都大于 `moving_avg` 的最大可能值 255。`case` 做精确位模式匹配，而 `moving_avg` 永远到不了 1998/1999，所以三个分支都不命中，每次落 `default`，输出恒为 `4'b11_11`。

**练习 2**：如果要保留"四档"但让区间写法合法，最小改动是什么？

**参考答案**：把 `case(moving_avg)` 改成 `case(moving_avg) inside`，并把每个分支项写成区间集合：`[0:50]:`、`[51:100]:`、`[101:150]:`、`default:`。这需要 SystemVerilog 支持；若只能用 Verilog-2001，则改写为 `if-else if` 链最稳妥。

**练习 3**：`voltage_level`/`freq_level` 是寄存输出还是组合输出？这种选择有什么好处和隐患？

**参考答案**：是组合输出（`always @(*)` 驱动 `output reg`，未在时序块里赋值）。好处是 `moving_avg` 一变、档位立刻跟随，没有额外延迟；隐患是若上游 `moving_avg` 存在毛刺（glitch），档位输出也会随之抖动，且组合路径过长可能影响时序。工程上常在组合分档后再加一级寄存器打拍，以获得干净的档位信号。

## 6. 本讲小结

- **DVFS 的本质**：按负载高低成对切换电压与频率，轻载省电、重载保性能；因 \(P\propto V^{2}f\)，降电压是省电的大头，但电压降必须配频率降才能保证时序。
- **三段式结构**：`workload` → 8 级 `history_buffer` 移位 → `moving_avg`（求和 `>>3`）→ `case` 分档输出 `{voltage_level, freq_level}`；采样用时序、分档用组合。
- **移动平均的作用**：低通滤波，抑制瞬时尖峰，避免档位抖动；窗口越长越平滑但调节越滞后，窗口越短越灵敏但越易抖动。
- **四档映射**：`00`~`11` 对应电压 0.8→1.2 V、频率 500 MHz→1.2 GHz，成对切换；中间两档的精确值代码未给，属待确认。
- **多处待确认 / 非标准写法**（本讲重点）：`history_buffer[6:0]` 切非压缩数组非法；`rst_n` 死端口无复位；`VOLTAGE_LEVELS`/`FREQ_LEVELS` 死参数；尤其 `case` 的 `0-50:` 实为减法，三个显式分支永不命中，**输出恒为最高档 `4'b11_11`，策略整体失效**。
- **集成状态**：`DVFS_Controller` 未被 `npu_soc.v` 例化，`workload` 无生产者——目前是待集成的独立蓝图，与 `npu_controller`、`sparse_encoder` 等同类。
