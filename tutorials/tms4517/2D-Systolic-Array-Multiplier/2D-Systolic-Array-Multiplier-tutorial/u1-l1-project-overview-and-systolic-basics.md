# 项目定位与脉动阵列原理

## 1. 本讲目标

本讲是整本学习手册的第一篇，目标是让你在还没有读过任何 RTL 代码之前，先建立两个直觉：

1. **这个项目到底在做什么**——它是一个用 SystemVerilog 写的「二维脉动阵列（2D Systolic Array）矩阵乘法器」，可以把两个方阵相乘。
2. **为什么硬件上要用「脉动阵列」这种结构来做矩阵乘法**——也就是「数据一边流动、一边在每个单元里做局部计算」的核心思想。

读完本讲，你应该能够：

- 用一句话说清项目的输入、输出和可配置范围；
- 解释矩阵乘法为什么是硬件加速的好对象，以及脉动阵列「数据流 + 局部通信」的优势；
- 说出本项目选择 8 位输入 / 32 位输出的位宽取舍，以及它和神经网络量化、Google TPU 的关系。

> 本讲只读 `README.md`，不深入 RTL。具体的 PE、阵列互联、行/列变换的实现细节，会在后续讲义（u2 系列）里逐行讲解。

## 2. 前置知识

本讲面向初学者，不需要你已经会写 Verilog。但下面几个概念会帮助你更好理解：

- **矩阵（Matrix）**：一个二维的数字表格。例如一个 \(4 \times 4\) 矩阵就是 4 行 4 列、共 16 个数的方阵。
- **矩阵乘法**：两个矩阵 \(A\) 和 \(B\) 相乘得到 \(C\)，其中 \(C\) 的每个元素是 \(A\) 的一行与 \(B\) 的一列对应相乘再求和（点积）。
- **时钟周期（Clock Cycle）**：数字电路按节拍工作，每个节拍叫一个周期。硬件加速的核心目标之一，就是「在尽量少的周期里完成尽量多的计算」。
- **乘加运算（MAC, Multiply–Accumulate）**：一次「乘法 + 累加」，即 \(acc \leftarrow acc + a \times b\)。这是矩阵乘法和神经网络推理中最频繁的基本操作。

如果你对矩阵乘法的公式还不太熟，没关系，下一节会从最朴素的算法讲起。

### 4.2 脉动架构背景与数据流思想

#### 4.2.1 概念说明

先回到最朴素的问题：**矩阵乘法到底有多「贵」，为什么值得专门做硬件？**

对 \(N \times N\) 矩阵 \(A, B\)，结果 \(C = A \times B\) 的每个元素为：

\[
C_{i,j} = \sum_{k=1}^{N} A_{i,k} \cdot B_{k,j}
\]

每个 \(C_{i,j}\) 要做 \(N\) 次乘法和 \(N-1\) 次加法；\(C\) 共有 \(N^2\) 个元素。所以总的乘法次数为：

\[
\text{乘法总次数} = N^2 \times N = N^3
\]

对 \(N=4\)，就是 64 次乘法；对 \(N=16\)，就是 4096 次乘法。如果用单个乘法器一个一个串行算，周期数会随 \(N^3\) 增长——这就是瓶颈。

**脉动阵列（Systolic Array）** 是解决这个瓶颈的一种经典硬件结构。它的核心思想是：

> 让**数据像血液在心脏里一样有节律地流动（systolic = 心脏收缩的）**，流过一个个排列成阵列的小计算单元（PE），每个 PE 每个时钟周期只做一次简单的乘加（MAC），并且**只和自己的邻居通信**。

README 在 `## Introduction` 一节专门留了两段来介绍背景，并指向了奠基性论文和 Google TPU：

[README.md:35-37](https://github.com/tms4517/2D-Systolic-Array-Multiplier/blob/59378e8caacdc23086ac09e1c04bc19500d70cfb/README.md#L35-L37) —— 「Systolic architectures」一节，引用了 H.T. Kung 的奠基论文 *Why Systolic Architectures?*（目前是占位待补）。

[README.md:39-41](https://github.com/tms4517/2D-Systolic-Array-Multiplier/blob/59378e8caacdc23086ac09e1c04bc19500d70cfb/README.md#L39-L41) —— 「Existing implementations」一节，引用了 Google TPU 的博客与论文（同样占位待补）。

> 名词解释：
> - **PE（Processing Element，处理单元）**：阵列里最小的计算单元，本项目里每个 PE 做一次 8 位乘法 + 32 位累加。
> - **TPU（Tensor Processing Unit）**：Google 为机器学习定制的芯片，其核心计算单元「矩阵乘法单元（MXU）」就是一个大型脉动阵列。本项目就是这个思想的「教学版迷你实现」。

#### 4.2.2 核心流程

为什么脉动阵列特别适合矩阵乘法？可以从「并行度」和「数据复用」两个角度看：

1. **天然并行**：\(C\) 的 \(N^2\) 个元素彼此独立，\(N \times N\) 个 PE 可以同时工作，每个 PE 负责算一个 \(C_{i,j}\)。
2. **数据复用**：
   - \(A\) 的第 \(i\) 行会参与 \(C\) 第 \(i\) 行所有 \(N\) 个元素的计算——「一行用 N 次」。
   - \(B\) 的第 \(j\) 列会参与 \(C\) 第 \(j\) 列所有 \(N\) 个元素的计算——「一列用 N 次」。
   - 脉动阵列让数据在 PE 之间**流动传递**，一份输入顺次经过多个 PE 被复用，从而极大降低了对存储器带宽的需求。
3. **局部通信**：每个 PE 只和自己左/上/右/下的邻居连线，没有长距离布线，因此可以跑在很高的时钟频率上、面积也规整。

README 的 Design Outline 把这套数据流概括成两步（下面是文字版流程，真正的 RTL 实现在 u2 系列）：

```text
① 准备 row/column 矩阵（set-up）
   - 对 i_a 的每一行：元素顺序反转、高位补零、整体右移对齐  →  row 矩阵
   - 对 i_b 的每一列：先转成一行、高位补零、整体右移对齐    →  column 矩阵
   目的：让数据「错峰」进入阵列，使每个 PE 在正确的时刻拿到正确的乘数。

② 把 row/column 矩阵送进阵列
   - row 矩阵的第一列  →  阵列的水平输入（横向）
   - column 矩阵的第一列 →  阵列的垂直输入（纵向）
   - 每个时钟周期，两个矩阵都左移 1 个元素，给阵列喂入新的一列数据
   - 经过 3N-2 个周期，乘法完成
```

关于「错峰（skew）」的直觉：如果不做错峰，所有数据同一拍涌入，PE 们会在同一时刻争抢同一份数据、时序也对不齐；做了错峰后，数据像波浪一样一波一波推进，每个 PE 每拍都恰好遇到它要乘的那一对数。

**练习 1**：对 \(N=4\)，矩阵乘法总共需要多少次乘法？如果用 1 个乘法器串行算，至少要多少周期（假设 1 拍 1 次乘法）？

### 4.3 输入/输出位宽选择与神经网络精度取舍

#### 4.3.1 概念说明

本项目的位宽选择写在 Design Outline 里短短两句话，但背后有清晰的工程与算法考量：

[README.md:56-60](https://github.com/tms4517/2D-Systolic-Array-Multiplier/blob/59378e8caacdc23086ac09e1c04bc19500d70cfb/README.md#L56-L60) —— 输入元素为 8 位整数（出于简洁、且对神经网络精度足够，参考 TPU）；输出元素为 32 位（为了方便验证 \(2 < N < 17\) 的任意规模）。

拆开看：

- **8 位输入**：每个矩阵元素用 8 位无符号整数表示，范围 \(0 \sim 255\)。这并非随手选的——它是工业界神经网络推理（INT8 量化）的常见位宽，Google TPU 也正是用 8 位整数做矩阵乘法。用 8 位能让乘法器面积小、速度快，同时对神经网络精度损失可接受。
- **32 位输出**：每个输出元素 32 位。这是为了「无论 \(N\) 多大（最大 16），累加结果都不会溢出」，方便测试台直接比对期望值，而不用操心截断。

#### 4.3.2 核心流程

我们来算一下「32 位输出是否真的够用」，这是一个很有代表性的硬件位宽推导。

每个 \(C_{i,j}\) 是 \(N\) 个「8 位 × 8 位」乘积的累加：

1. 单个乘积：\(255 \times 255 = 65025\)，需要 \(\lceil \log_2 65026 \rceil = 16\) 位。
2. \(N\) 个乘积相加：最大值为 \(N \times 65025\)。
3. 对最大 \(N=16\)：\(16 \times 65025 = 1\,040\,400\)，而 \(2^{20} = 1\,048\,576\)，所以 **20 位就足够**。

也就是说，理论上 20 位输出即可覆盖最坏情况；本项目选 32 位是留了充裕余量、并且对齐到常见的机器字长，便于 C++ 测试台处理：

\[
\text{需要位数} = \underbrace{8 + 8}_{\text{乘积}} + \underbrace{\lceil \log_2 N \rceil}_{\text{累加 }} \le 16 + 4 = 20 \le 32
\]

下表汇总了位宽选择：

| 信号 | 位宽 | 取值范围（无符号） | 选择理由 |
| --- | --- | --- | --- |
| 输入元素 `i_a` / `i_b` | 8 位 | \(0 \sim 255\) | 简洁；契合神经网络 INT8 量化（参考 TPU） |
| 输出元素 `o_c` | 32 位 | \(0 \sim 2^{32}-1\) | 覆盖最大 \(N=16\) 时累加（≤20 位即够），留余量、便于验证 |

#### 4.3.4 代码实践

1. **实践目标**：亲手验证「32 位输出在最大规模下也不会溢出」。
2. **操作步骤**：
   - 取最大 \(N=16\)，假设所有输入元素都取最大值 255。
   - 计算 \(C_{i,j}\) 的最大可能值：\(16 \times 255 \times 255\)。
   - 判断它是否小于 \(2^{32}\)，并进一步看它实际需要多少位。
3. **需要观察的现象**：\(16 \times 65025 = 1\,040\,400\)，远小于 \(2^{32} \approx 4.29 \times 10^9\)，甚至小于 \(2^{20}\)。
4. **预期结果**：32 位绰绰有余；理论最少 20 位。这印证了 README「convenience for verifying」的说法——选 32 位是为了方便，不是因为非它不可。
5. 本项为纯计算实践，无需运行仿真。

#### 4.3.5 小练习与答案

**练习 1**：如果把输入从 8 位无符号改成 8 位**有符号**，单个乘积最大会是多少？需要多少位？

**参考答案**：8 位有符号范围 \(-128 \sim 127\)，最大乘积幅度为 \((-128) \times (-128) = 16384\) 或 \(127 \times (-128) = -16256\)，绝对值最大 16384，需要 15 位表示数值、再加 1 位符号，即 16 位带符号数。这也是 u3-l4「把无符号 MAC 改为有符号乘法」会遇到的位宽问题。

**练习 2**：README 说 8 位输入「provides an appropriate level of accuracy for neural network calculations」。请结合「量化（quantization）」的概念解释这句话。

**参考答案**：神经网络推理时，常把训练得到的浮点权重和激活量化成低精度整数（如 INT8）再算，以换取更小的硬件面积、更高吞吐和更低功耗，同时精度损失通常可接受。Google TPU 正是采用 8 位整数矩阵乘法。所以本项目选 8 位输入，既简化了设计，又贴合真实的神经网络加速器实践。

## 6. 本讲小结

- 本项目是一个用 SystemVerilog 写的**二维脉动阵列矩阵乘法器**，输入两个 \(N \times N\) 方阵（\(2 < N < 17\)，默认 4），输出 \(C = A \times B\)。
- 脉动阵列的核心思想是**「数据流动 + 局部通信」**：数据像波浪一样在 PE 阵列里推进，每个 PE 每拍做一次乘加，只和邻居通信。
- 相比串行 \(O(N^3)\)，脉动阵列靠 \(N \times N\) 个 PE 并行 + 数据复用，把矩阵乘法压缩到 \(O(N)\) 量级（\(3N-2\) 拍）。
- 输入选 **8 位**：简洁、契合神经网络 INT8 量化与 TPU 实践；输出选 **32 位**：保证最大 \(N=16\) 时累加也不溢出（理论 20 位即够），方便验证。
- 依赖只有 `Verilator + GNU Make`，运行入口是 `cd tb && make all`；改规模需同步改 RTL 参数 `N` 和 TB 宏 `N`。
- 本讲只读了 `README.md`；项目的端口、RTL 实现和仿真细节在后续讲义展开。
