本讲从「使用者」切换到「贡献者」视角，讲清给 axi 提 PR 必须跨过的门槛。最小模块有四：一是贡献规范——`CONTRIBUTING.md` 三条硬规则：模块名须以 `axi_` 开头、用户面模块用 SystemVerilog `struct` 作 AXI 端口（类型作 `parameter` 传入）、`_intf` 变体只接线不实现功能；二是版本管理——遵循语义化版本（SemVer）与 Keep a Changelog，`VERSION` 为单行机器读版本号，`CHANGELOG.md` 用顶部 `Unreleased` 段登记 PR、破坏性变更单列 `Breaking Changes`；三是 CI 双平台架构——内部 GitLab 跑商业工具（vsim、Synopsys DC）重活，GitHub Actions 跑开源 Verilator lint、yosys-slang elaboration 与文档部署，一条轮询 Action 把内部 pipeline 结果回填到公开 PR，fork 因缺 secret 自动跳过；四是四类质量门——编译（vsim，`-t test -t rtl`）、仿真（多种子，判据 `Errors: 0,`）、lint（Verilator）、综合（DC/yosys-slang，顶层 `axi_synth_bench`，用 `-t synthesis -t synth_test`），本地入口为 `make`。核心结论：GitLab CI 用 `changes: compare_to: 'refs/heads/master'` 取整条分支累计 diff，防止后一条无关 commit 掩盖前一条失败；`--random-seed` 是无参标志（追加 `"random"` 种子），`$TEST_MODULE` 是测试名而非种子。
