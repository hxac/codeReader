本讲围绕 `axi_xbar` 的两大性质展开：**事务保序**与**无死锁**，核心结论是「为什么内部不能流水」。

关键概念：① **同 ID 同方向保序**——AXI 要求同一 master、同 ID 同方向事务的响应按序返回；本库 xbar **无 reorder buffer**，故该约束被翻译成 **slave 端口停顿**：由 `axi_demux_id_counters` 记录占用与目标端口，仅当同 ID 去往**不同 master 端口**的第二笔事务才需等第一笔完成（同端口天然保序）。② **`AxiIdUsedSlvPorts`/`AxiLookBits`**——只比较 ID 低位 N 位建计数器，用偶发**误冲突(false conflict)**换指数级省面积；`UniqueIds=1` 在前提下把面积从 O(2^I) 降至 O(I)，误冲突只损吞吐不损正确性。

核心死锁分析：用 **Coffman 四条件**审视 W 通道，前三条（互斥、占有等待、不可抢占）由 AXI 协议铁律决定不可改，唯一可下手的是第④条**循环等待**。在 demux 输出与 mux 输入间插 spill 寄存器会配合 `rr_arb_tree` 的推进形成 W FIFO 等待环，故 **xbar 内部 demux↔mux 必须纯组合直连**，spill 只能放对外边界（`LatencyMode` 控制）。mux 内部的 `lock_aw_valid` 是切断 AW→W 组合依赖的合法解耦（与禁止内部寄存器不矛盾），`FallThrough` 是可调时序旋钮（推荐 `FallThrough=0` + `CUT_ALL_AX`）。涉及最小模块：`axi_demux_simple`（停顿门）、`axi_mux`（W FIFO 与仲裁）、`id_counters`。
