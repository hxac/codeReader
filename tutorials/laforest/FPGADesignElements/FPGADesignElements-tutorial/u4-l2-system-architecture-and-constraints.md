# Core/Instance/Adapter/Shim 分层与约束

## 1. 本讲目标

本讲是「系统设计标准与模块化哲学」单元的第二篇，承接 [u4-l1](./u4-l1-modularization-and-building-blocks.md)。学完之后，你应该能够：

- 复述自底向上的 **Core / Instance / Adapter / Shim** 四层架构，说清每一层「在谁之上加了一个什么职责」，以及随层升高「从应用逻辑走向对外接口」的抽象方向。
- 把一个真实的 FPGA 工程按这四层切开，并指出每一层在哪个设计阶段（仿真、综合、布局布线）开始接受检查。
- 解释为什么要按**类型**把约束分到不同的文件里（位流、布局、引脚、时序、优化），而不是全塞进一个文件或全写进源码。
- 说出哪些约束**必须**写进源码（如 `ASYNC_REG`、`PRESERVE`、`IOB`），并能用 [u3-l2](./u3-l2-resets-and-register-module.md) 里 CDC 同步器的真实代码证明这一点。

[u4-l1](./u4-l1-modularization-and-building-blocks.md) 回答了「子系统内部怎么拆成处理/控制/接口三模块」。本讲把视角再抬高一层：**整个 FPGA 工程**（甚至跨多块板、多支团队协作的工程）该怎么分层组织，以及那些「不描述设计本身、但 CAD 工具必需」的约束文件该怎么管。

## 2. 前置知识

先用大白话把本讲要用到的术语对齐（部分已在 [u4-l1](./u4-l1-modularization-and-building-blocks.md) 出现，这里只补新概念）：

- **CAD 工具 / 综合工具**：把 Verilog 翻译成 FPGA 上真实硬件（LUT、触发器、BRAM、DSP、时钟缓冲器、I/O 缓冲器等）的工具链，如 AMD/Xilinx Vivado、Intel Quartus。
- **约束（constraint）**：告诉 CAD 工具「该怎么做」的额外信息，但**不描述电路逻辑本身**。例如「这个引脚的电平标准是 LVCMOS33」「这两个时钟彼此异步」「这组寄存器别优化掉」。
- **时序收敛（timing closure）**：让设计在目标时钟频率下满足所有触发器的建立/保持时间要求。
- **CDC（Clock Domain Crossing，时钟域穿越）**：信号从一个时钟域进入另一个不同步的时钟域，需要「同步器」降低亚稳态风险。详见 [u3-l2](./u3-l2-resets-and-register-module.md) 提到的同步器，以及后续 u13 CDC 单元。
- **亚稳态（metastability）/ MTBF**：触发器在靠近时钟边沿采样时，输出可能在 0/1 之间悬停一段时间才稳定。同步器靠多级寄存器把「出错概率」压低，**MTBF（平均无故障时间）**衡量这个概率。
- **属性（attribute）**：Verilog-2001 里 `(* ... *)` 形式的注解，写在某个线网/寄存器/模块声明**前面**，用来给综合工具传递「这条声明该怎么处理」的提示。这是 Verilog-2001 引入的少数「元信息」机制之一。

一句话定位：[u4-l1](./u4-l1-modularization-and-building-blocks.md) 讲的是「模块怎么切」，本讲讲的是「切出来的模块怎么**摞成四层**」以及「配套的约束怎么**分文件**」。

## 3. 本讲源码地图

## 4. 核心概念与源码讲解

本讲拆成三个最小模块：**四层架构**、**约束文件分离**、**源码内约束**。前两个讲「设计与约束怎么组织成文件」，最后一个讲「为什么有少数约束偏要反其道而行、写进源码」。

### 4.1 四层架构：Core / Instance / Adapter / Shim

#### 4.1.1 概念说明

`system.html` 在 System Architecture 一节开篇先说明这套架构的动机：在一个由多支团队分别设计、含多块 FPGA 和多块板卡的工程里，每片 FPGA 上采用**共同的代码架构**，能极大降低「不匹配、错误、临时改动」带来的痛苦，并提升可移植性——单片 FPGA 设计同样受益。

接着作者点出这套架构的总纲，也是本模块最重要的一句话：**自底向上解释，每往上一层，就把下面那一层当作单个模块实例化进来，并只为支撑下层而多做一件具体的事；随着层升高，职责从「创造/支撑应用逻辑」转向「管理对外接口」。**

把它画出来就是一条向上的抽象阶梯：

```text
                  ┌──────────────────────────────────────────┐
   顶层（接 PCB） │  Shim   : 实例化 Adapter，处理 PCB 级问题   │
                  │           （坏引脚、总线散落、电平不匹配）   │
                  └────────────────────▲─────────────────────┘
                                       │ 实例化（端口一一对应）
                  ┌────────────────────┴─────────────────────┐
   接 FPGA 器件   │  Adapter: 实例化 Instance，补齐器件级硬件  │
                  │           （时钟管理、差分缓冲、协议转换）   │
                  └────────────────────▲─────────────────────┘
                                       │ 实例化（给出最终端口）
                  ┌────────────────────┴─────────────────────┐
   缩放到器件     │  Instance: 实例化 Core，定规模/数量/连线    │
                  │           （SIMD 通路数、CPU 个数、片上网络）│
                  └────────────────────▲─────────────────────┘
                                       │ 实例化（只接简单信号）
                  ┌────────────────────┴─────────────────────┐
   纯应用逻辑     │  Core   : 抽象的应用逻辑（零件库的拼装）     │
                  │           （软 CPU 的控制/数据通路、NoC）   │
                  └──────────────────────────────────────────┘
```

四层各自的职责，原文用三条要点概括：

- **Core 与 Instance** 处理**应用逻辑**；
- **Adapter** 把应用逻辑接到**具体的 FPGA 器件**上；
- **Shim** 把 FPGA 接到**它所焊接的那块 PCB** 上。

#### 4.1.2 核心流程

理解四层的关键，是抓住「每层各自在哪个设计阶段开始受检、关心什么」。按从底到顶的顺序：

```text
Core    : 纯逻辑。可仿真验证功能正确性；几乎永不改动（除非换 FPGA 家族）。
          ───────────────────────────────────────────────────
Instance: 把 Core 缩放到「这片」FPGA。用一组临时寄存器测试桩围起来，
          可做仿真 + 初步综合（预估工作频率）。
          ───────────────────────────────────────────────────
Adapter : 接到器件。补上时钟管理、差分缓冲、协议转换等「为了满足 CAD 工具」
          的硬件。在此综合应能暴露最后的错误（时钟缓冲/资源不足、约束缺失）。
          ───────────────────────────────────────────────────
Shim    : 接到 PCB。只做引脚洗牌与标注，不改逻辑。在此做最终综合、
          post-placement 检查 I/O 延时、做最终 floorplanning。
```

两条贯穿全层的线索：

1. **抽象方向**：越往下越「抽象、可复用、永不改」（Core）；越往上越「具体、与某块芯片/某块板绑定、容易因换板而改」（Shim）。
2. **检查深度**：越往下，越靠仿真就能验证；越往上，越必须靠完整综合与布局布线才能验证（因为时钟缓冲、I/O 延时、引脚分配只有到了 Adapter/Shim 才存在）。

#### 4.1.3 源码精读

**（1）四层架构的总纲**——见 [system.html:L125-L143](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L125-L143)。抓三句话：作者「从底向上」讲；每层实例化下层为单个模块并只做一件支撑下层的事；随层升高，职责从应用逻辑转向对外接口。末尾三条要点（Core/Instance 管应用逻辑、Adapter 接器件、Shim 接 PCB）正是上面阶梯图的出处。

**（2）Core——抽象的应用逻辑**：[system.html:L145-L154](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L145-L154)。Core 包含「抽象意义上需要的全部零件」，不关心接口协议、FPGA 大小、实例数量；唯一与 FPGA 相关的是技术映射（用哪种 DSP/BRAM）。除了移植到别的 FPGA 家族，这段代码**永不改动**。举例：一个软 CPU 的控制与数据通路、片上网络（NoC）的构建块——它们都以「零件库」的形式存在（呼应 [u4-l1](./u4-l1-modularization-and-building-blocks.md) 的构建块库）。

**（3）Instance——缩放到具体器件**：[system.html:L156-L170](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L156-L170)。Instance **顶到「一个模块」**，它实例化 Core，并规定 Core 的规模、数量、以及它们之间的全部连线。它主要处理「把 Core 缩放到当前这片 FPGA」。举例：定义一个软 CPU 有几条 SIMD 数据通路、总共有几个 CPU、连接它们的 NoC。此时还**不关心**任何外部接口——总线与时钟都只是普通信号。到这一步，已经可以仿真验证逻辑正确性，并用一组**临时寄存器测试桩**把 Instance 围起来做初步综合、预估最终工作频率（测试桩保证静态时序分析有意义）。

> 注意这里的「临时寄存器测试桩」正是本书 `Synthesis_Harness_Input.v` / `Synthesis_Harness_Output.v` 的用途（详见 u18-l2）。它们是 Instance 阶段的脚手架。

**（4）Adapter——接到器件**：[system.html:L172-L189](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L172-L189)。Adapter 实例化 Instance，定义「板子期望的最终接口端口」，并把 Instance 连到这些端口上。它添加一切**支撑 Instance 同时又满足 CAD 工具**所需的逻辑。举例：时钟产生与管理硬件、给需要的信号加差分缓冲（**即使暂时没用也要加**，只为满足 CAD 工具的约束）、按需做协议转换（AXI leader/follower、双向或三态缓冲等）。到这一步，综合应该能暴露最后的错误——时钟缓冲器或逻辑资源不足、约束缺失/违反等——并尽量消除告警。

**（5）Shim——接到 PCB**：[system.html:L191-L217](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L191-L217)。Shim **只实例化 Adapter，再无其他**。它的端口与 Adapter 相同，且 Shim 的端口就在顶层、物理上直接连 FPGA 引脚。Shim 干的事是处理 **PCB 带来的麻烦**：不可用的连接、原理图错误、不兼容的电平标准等。两个经典场景：

- **坏引脚救场**：若某些顶层引脚因设计或制造错误不可用，Shim 在「顶层引脚」与「Adapter 端口」之间洗牌信号，向 Adapter 呈现一个统一（可能缩减）的接口；有空闲板引脚也在这里用掉。
- **总线散落重组**：若某条总线因 PCB 设计没有连到 FPGA I/O Bank 上连续的引脚，而是散落各处，Shim 把 Adapter 的「逻辑总线引脚」翻译到器件上「物理连续的引脚」，让相关逻辑尽量聚拢。当然，PCB 上另一片 FPGA 也要一个类似的 Shim 把已被打散的总线信号重新组织回来。

Shim 还是标注顶层引脚的好地方（哪些时钟可用、能否做差分对、位于哪个时钟域/I/O Bank 等），方便日后改动。最后，到这里才能做完整综合得到最终工作频率、做 post-placement 综合（检查 I/O 延时线）、做最终 floorplanning。

把四层与「检查阶段」对应起来，就是下表（合并了 [u4-l1](./u4-l1-modularization-and-building-blocks.md) 的三阶段原理图）：

| 层 | 主要职责 | 在哪个阶段开始受检 | 典型问题 |
|----|---------|-------------------|---------|
| **Core** | 抽象应用逻辑 | 仿真、post-elaboration | 逻辑错误、缺失连接 |
| **Instance** | 缩放到器件、规模/连线 | 仿真 + 加测试桩的初步综合 | 功能正确性、频率预估 |
| **Adapter** | 接器件：时钟/差分/协议 | 完整综合 | 时钟缓冲不足、资源不足、约束缺失 |
| **Shim** | 接 PCB：引脚洗牌/标注 | 完整综合 + post-place-and-route | I/O 延时、引脚分配、floorplanning |

#### 4.1.4 代码实践

> **实践类型：源码阅读 + 架构拆分设计**

1. **实践目标**：把一个简单设计按 Core/Instance/Adapter/Shim 四层切开，体会「每层只加一件支撑下层的事」。
2. **操作步骤**：
   - 选一个你熟悉的小设计，例如「一个含 CDC 桥的计数器子系统」：用 `Counter_Binary`（见 [u4-l1](./u4-l1-modularization-and-building-blocks.md)）产数据，经 `CDC_Bit_Synchronizer` 跨到另一个时钟域输出。
   - 画出四层：
     - **Core**：`Counter_Binary` + 跨域逻辑（纯逻辑，不碰具体器件/引脚）。
     - **Instance**：实例化 Core，设好 `WORD_WIDTH`、计数深度等规模参数；两侧时钟都当普通信号接。
     - **Adapter**：补上两个时钟域的时钟缓冲（如 MMCM/PLL）、必要的 I/O 寄存器。
     - **Shim**：把顶层引脚分配到具体 I/O Bank，处理板上的引脚散落。
   - 对每一层，标注「它在哪个阶段开始受检」（参考上表）。
3. **需要观察的现象**：你会看到 Core 里**没有任何**与具体 FPGA/板相关的东西（没有 MMCM、没有引脚号）；所有「脏活」都被往上推到了 Adapter/Shim。这正是「可移植性」的来源——换板时只改 Shim，换芯片时改 Adapter，Core 不动。
4. **预期结果**：得到一张四层方框图，每层只实例化正下方一层为单个模块，且每层有清晰的一句话职责。
5. 完整工程搭建**待本地在 CAD 工具中验证**；本步只要求画出分层方案与连线。

#### 4.1.5 小练习与答案

**练习 1**：作者强调「每往上一层，就实例化下面那一层为**单个**模块」。为什么必须是「单个模块」而不是「把下层代码复制上来」？

> **参考答案**：因为「实例化为单个模块」保住了 [u4-l1](./u4-l1-modularization-and-building-blocks.md) 讲的**设计层次**——在原理图里，下层始终是一个有名字的方块，可以被独立检查、独立替换。如果复制代码，就会重复随机逻辑、丢失层次，且改一处要改多处。单模块实例化是「可移植 + 可维护」的物理保障。

**练习 2**：换了一块 PCB（FPGA 型号不变，只是引脚走线变了），四层里需要改哪一层？换了一片不同家族的 FPGA（PCB 不变）呢？

> **参考答案**：换 PCB 只改 **Shim**（引脚洗牌、电平标准、I/O Bank 分配都在这里）；FPGA 型号不变意味着 Adapter/Core/Instance 可不动。换 FPGA 家族则至少要改 **Adapter**（时钟管理 IP、差分缓冲原语都是器件相关的），可能波及 Instance（若规模/资源变了），但 **Core** 因不含器件相关内容通常不动——这正是把器件相关逻辑上推到 Adapter 的回报。

### 4.2 约束文件分离：按类型分文件管理

#### 4.2.1 概念说明

四层架构讲的是「描述设计的 `.v` 文件怎么摞」。但一个 FPGA 工程里还有一大堆**不描述设计本身、却是 CAD 工具必需**的文件——**约束（constraints）**。`system.html` 在 Project Constraints 一节给出的总原则是：**把不同类型的约束分到不同的文件里**，只把**极少数**约束作为属性直接写在源码的线网/寄存器/模块上。

为什么要分文件？同样一句话：**可移植性 + 可管理性**。当底层 FPGA 或周围 PCB 变化时（这正是 4.1 里 Shim/Adapter 要应对的事），分了类的约束文件能让你只动相关那一类，而不是在一个巨型约束文件里大海捞针。

#### 4.2.2 核心流程

`system.html` 列出了五类该各自成文件的约束：

```text
┌──────────────────────────┬───────────────────────────────────────┐
│ 约束类型                 │ 典型内容                              │
├──────────────────────────┼───────────────────────────────────────┤
│ 1. 位流/配置 (bitstream) │ 配置引脚上下拉、配置接口(SPI/并口/JTAG)│
│                          │ 位流加密                              │
│ 2. 逻辑布局 (placement)  │ floorplan 区域、把模块指派到区域；隔离│
│                          │ 关键逻辑(如 DDR 控制器)保时序；把互相 │
│                          │ 挤占布线的模块物理拉开                │
│ 3. 引脚布局 (pin)        │ 每个 bank 的引脚：源码名、I/O 标准、  │
│                          │ 上下拉、驱动强度；标注未用/危险引脚； │
│                          │ 按 bank 物理顺序排列(噪声/开关电流)   │
│ 4. 时序 (timing)         │ 输入时钟、同步/异步关系、假路径       │
│                          │ (false path)、外部 I/O 延时           │
│ 5. 优化 (optimization)   │ 保留逻辑不被优化。如带半成品/悬空模块 │
│                          │ 时用 DONT_TOUCH(Vivado)保留未用逻辑， │
│                          │ 以获得更准确的面积/时序预估           │
└──────────────────────────┴───────────────────────────────────────┘
```

注意第 3 类「引脚布局」里有个细节：**按 bank 的物理顺序排列引脚**，能让你一眼看出哪些信号在器件上物理相邻——这对噪声与开关电流问题很重要。这与 4.1 里 Shim「向 Adapter 呈现统一接口、把散落总线聚拢」的职责是一体两面：Shim 管「逻辑引脚怎么映射到物理引脚」，引脚约束文件管「这些物理引脚的电气属性」。

#### 4.2.3 源码精读

**（1）分文件的总原则与五类约束**——见 [system.html:L219-L257](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L219-L257)。开篇点题：把不同类型约束分到不同文件，**只留极少数**作为属性直接写在源码上，目的是让设计尽量可移植、在 FPGA 或 PCB 变化时易于管理。随后逐条列出五类约束（位流/配置、逻辑布局、引脚布局、时序、优化）。重点抓两个工程化要点：

- **逻辑布局**可用来**隔离关键逻辑**（如 DDR 控制器）以稳定满足时序，也可用来**物理拉开**原本被放得太近、造成布线拥堵的模块。
- **优化约束**（如 Vivado 的 `DONT_TOUCH`）在「带半成品、有悬空模块」的阶段尤其有用：保留住未用逻辑，才能得到**持续的、更接近最终**的面积/时序估计，而不是被综合器优化掉后盲目乐观。

**（2）与四层架构的呼应**——把 4.1 和 4.2 合起来看：Shim 是「写引脚洗牌逻辑」的地方，而引脚约束文件是「声明这些引脚电气属性」的地方；Adapter 是「放时钟管理硬件」的地方，而时序约束文件是「声明这些时钟及其同步/异步关系」的地方。**设计文件管逻辑，约束文件管「逻辑之外的规则」**，两者职责互补、绝不混写。

#### 4.2.4 代码实践

> **实践类型：源码阅读 + 约束归类**

1. **实践目标**：能把一条具体的约束正确归类到五类之一，并说出它该进哪个文件。
2. **操作步骤**：
   - 想象你在搭 4.1.4 那个「含 CDC 桥的计数器」工程，列出它需要的若干约束，例如：
     - (a) 「`clk_ext` 与 `clk_int` 彼此异步」
     - (b) 「`data_out` 引脚位于 Bank 34，电平 LVCMOS33，驱动 8mA」
     - (c) 「从 `input_pin` 到第一级寄存器是异步路径，设为 false path」
     - (d) 「用 SPI 模式加载位流，配置引脚内部上拉」
     - (e) 「CDC 同步器那两个寄存器别被 retiming 拆散」
   - 把每一条归到五类约束之一，并指出它对应 4.1 的哪一层。
3. **需要观察的现象**：你会发现 (a)(c) 属于**时序**、(b) 属于**引脚布局**、(d) 属于**位流/配置**、(e) 属于**优化**；且 (b)(d) 与 Shim 强相关，(a)(c)(e) 与 Adapter/同步器强相关。
4. **预期结果**：得到一张「约束 → 类型 → 文件 → 对应架构层」的对照表。
5. 注意 (e) 是个**陷阱题**——下一模块（4.3）会说明它其实**不该进外部优化约束文件，而要写进源码**。

#### 4.2.5 小练习与答案

**练习 1**：为什么「按 bank 的物理顺序排列引脚约束」值得专门强调，而不是按字母顺序排？

> **参考答案**：因为噪声与开关电流问题取决于信号在**器件上是否物理相邻**（同一 bank、相邻引脚的串扰和同步开关噪声最大）。按 bank 物理顺序排列，能在约束文件里直接看出哪些信号物理相邻，从而发现潜在的信号完整性问题；按字母排则丢失了这层空间信息。

**练习 2**：在「半成品」阶段为什么要用 `DONT_TOUCH` 这类优化约束保留住未用逻辑？

> **参考答案**：因为综合器会把输出悬空或输入固定的逻辑优化掉，导致你看到的面积/时序估计**虚高乐观**——你以为还剩很多资源、频率很宽裕，其实是被优化掉的假象。保留住未用逻辑，才能得到**持续的、更接近最终**的估计，让项目早期预估可信。

### 4.3 源码内约束：为什么 ASYNC_REG 必须写在源码里

#### 4.3.1 概念说明

4.2 的总原则是「分文件、少写源码」。但 `system.html` 紧接着给了一个**例外**：**有些约束可能不得不写进源码本身**，原因有二：

1. **时序原因**：CAD 工具可能「来不及」在综合/布局/布线的早期阶段读到外部约束文件——这取决于具体工具。而写在源码里的属性，会随模块被精化（elaboration）时**立刻**被读到。
2. **封装/正确性原因**：有些约束（作者点名 Vivado 的 `ASYNC_REG`，它声明「这些寄存器构成一个 CDC 同步器」）**应该**写在 CDC 同步器模块的源码里，这样它们就能在**每一次**实例化同步器时**自动生效**。

第二条是本模块的核心，也是本书「模块化到极致」哲学在约束层面的延续：**同步器的正确性属性应当与同步器模块绑定、随它一起旅行**，而不是每次用同步器都靠人去外部约束文件里补一行——补漏了不会报错，只会悄悄降低 MTBF，留下间歇性现场故障的隐患。

`ASYNC_REG` 到底要求什么？它告诉工具：打了这个属性的寄存器属于一个跨时钟域同步器，因此（a）要被识别为同步器并出现在 MTBF 报告里，（b）这些寄存器之间要**物理紧挨**（maximally close）以最大化 MTBF，（c）**不能**被优化掉或搬进 DSP/BRAM 的输入寄存器，（d）**不能**放进 I/O 寄存器（I/O 寄存器离主逻辑阵列太远）。这些属性本质上是**寄存器实例自身的物理属性**，所以必须贴在声明它的那行源码上。

#### 4.3.2 核心流程

把「属性随声明走」的机制画清楚：

```text
外部约束文件（如 .xdc/.sdc）
   ├─ 引用时必须给出「层次路径」，如 top/inst_cdc/sync_reg[0]
   ├─ 每新增一个同步器实例，就要新增一组约束行
   └─ 忘写一行 → 不报错，只是该同步器 MTBF 变差（静默退化）

源码内属性（(* ... *)）
   ├─ 写在 reg/wire 声明的前一行，物理绑定到该声明
   ├─ 随模块实例化自动复制：实例化一次 = 属性自动生效一次
   └─ 不可能「忘写」：属性与寄存器同生同灭
```

CDC 同步器恰恰是「正确性依赖寄存器物理位置、且会被到处实例化」的典型，因此作者把 `ASYNC_REG` 等属性固化进 `CDC_Bit_Synchronizer.v` 源码，让「安全使用同步器」成为**默认行为**而非「需要记得做的事」。

#### 4.3.3 源码精读

**（1）例外的总纲**——见 [system.html:L259-L265](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/system.html#L259-L265)。原文两层意思：一是 CAD 工具可能读外部约束文件读得不够早（时序原因）；二是像 `ASYNC_REG` 这类约束**应当**写在 CDC 同步器模块源码里，以便「在每个 CDC 同步器模块实例化时**自动生效**」（封装原因）。

**（2）活样本：CDC_Bit_Synchronizer 的属性块**——这是本模块最重要的代码点。先看作者为这些属性写的解释注释：[CDC_Bit_Synchronizer.v:L105-L114](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L105-L114)。注释把属性分成 Vivado 与 Quartus 两组，并点明共同要求：寄存器要**紧挨放置**、要作为同步器出现在 MTBF 报告里、**不能**进 I/O 寄存器位置。

属性本体就紧贴在 `reg sync_reg` 声明的正上方：[CDC_Bit_Synchronizer.v:L115-L124](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L115-L124)。

```verilog
    // Vivado
    (* IOB = "false" *)
    (* ASYNC_REG = "TRUE" *)

    // Quartus
    (* useioff = 0 *)
    (* PRESERVE *)
    (* altera_attribute = "-name SYNCHRONIZER_IDENTIFICATION \"FORCED IF ASYNCHRONOUS\"" *)

    reg sync_reg [DEPTH-1:0];
```

逐条对读（Vivado 与 Quartus 两套是为可移植性并存）：

| 属性 | 工具 | 作用 |
|------|------|------|
| `ASYNC_REG = "TRUE"` | Vivado | 声明这组寄存器是 CDC 同步器的一部分；要求相邻级**紧挨放置**，并出现在 MTBF 报告中 |
| `IOB = "false"` | Vivado | **禁止**把寄存器放进 I/O 寄存器（离逻辑阵列太远，会破坏同步器性能） |
| `PRESERVE` | Quartus | **禁止**优化/搬动这些寄存器（如搬进 DSP/BRAM 输入寄存器） |
| `useioff = 0` | Quartus | **禁止**使用 I/O 寄存器位置 |
| `altera_attribute ... SYNCHRONIZER_IDENTIFICATION "FORCED IF ASYNCHRONOUS"` | Quartus | 强制把这组寄存器识别为同步器（从而紧挨放置） |

**（3）为什么必须紧贴 `reg sync_reg` 声明？**——这是理解「源码内约束」机制的关键。`(* ... *)` 是 Verilog-2001 的属性语法，它**前置于哪条声明，就修饰哪条声明**。这里它修饰的是 `reg sync_reg [DEPTH-1:0];`——也就是构成同步器链条的那组寄存器。所以这些「别优化、别进 I/O、要紧挨」的要求，是**与那组寄存器绑定的**。这正是 `system.html` 说的「自动在每个实例化处生效」的技术基础：每次实例化 `CDC_Bit_Synchronizer`，这组 `sync_reg` 连同它们身上的属性一起被复制，无需调用方操心。

**（4）旁证：Reset_Synchronizer 为什么宁可复制逻辑也不复用 CDC_Bit_Synchronizer？**——`Reset_Synchronizer.v` 带有**完全相同**的属性块：[Reset_Synchronizer.v:L101-L108](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/Reset_Synchronizer.v#L101-L108)。而它的注释直接道出了不得不如此的原因：[Reset_Synchronizer.v:L18-L24](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/Reset_Synchronizer.v#L18-L24) ——「我们**不能**在这里实例化那些模块，因为我们必须把一些属性**直接贴到 `reg` 值上**才能做一个好的同步器，而 CDC 同步器模块没有复位」。换句话说：属性必须**直接**贴在目标寄存器声明上，复用一个不含复位、且寄存器藏在内部的模块，就无法把属性贴到正确的寄存器上——于是作者宁愿把同步器逻辑**内联复制**。这从反面证明了本书对「属性随声明走」的执念有多深。

**（5）与同步器「禁用 I/O 寄存器」说明的呼应**——[CDC_Bit_Synchronizer.v:L76-L85](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L76-L85) 专门有一节「Not Usable as I/O Registers」，解释同步器各级寄存器必须**尽量紧挨**，而 I/O 寄存器离主逻辑阵列太远，所以同步器不能用 I/O 寄存器。这正是上面 `IOB="false"` / `useioff=0` 两条属性要强制的东西。注释（设计意图）与属性（强制手段）在这里严丝合缝。

> 小结：源码内属性不是「写在源码里的普通约束」，而是**绑定到具体声明的物理要求**。`ASYNC_REG`/`PRESERVE`/`IOB` 之所以要写进源码，是因为它们描述的是「**这几个寄存器**必须如何被放置/优化」，这种「与实例绑定、需随实例旅行、漏写即静默退化」的要求，外部约束文件既不方便也容易出错，所以固化进模块源码最安全。

#### 4.3.5 小练习与答案

**练习 1**：假如把 `ASYNC_REG` 从源码挪到外部约束文件，会出现什么风险？为什么它**不会**立刻报错？

> **参考答案**：风险是「漏写某条约束」。外部约束文件靠层次路径引用寄存器，每实例化一个新同步器就要新增一组约束行；漏掉一行，CAD 工具**不会报错**（它只是不知道这几个寄存器是同步器），结果是把它们当普通寄存器布局/优化，可能被拉开、搬进 I/O 或 DSP 寄存器、甚至被 retiming 拆散——MTBF 静默下降，表现为难以复现的间歇性故障。写进源码则让属性随实例化自动生效，从源头消灭「漏写」。

**练习 2**：`Reset_Synchronizer` 注释说「不能实例化 CDC_Bit_Synchronizer，因为要把属性直接贴到 `reg` 上」。这反过来证明了属性的哪条语法规则？

> **参考答案**：证明 `(* ... *)` 属性**前置于哪条声明、就修饰哪条声明**。若 `sync_reg` 藏在被实例化模块的内部，调用方根本「够不着」那条声明，也就无法把 `ASYNC_REG` 贴到正确的寄存器上；所以作者只能把同步器逻辑内联到 `Reset_Synchronizer` 里，让属性紧贴自己的 `reg sync_reg`。这正是「属性必须随声明走」的强制力。

## 6. 本讲小结

- 本书主张用一个**自底向上**的四层架构组织整个 FPGA 工程：**Core**（抽象应用逻辑，永不改）→ **Instance**（缩放到器件、定规模/连线）→ **Adapter**（接器件：时钟/差分/协议，满足 CAD）→ **Shim**（接 PCB：引脚洗牌/标注）。每层只实例化正下方一层为单个模块，只多做一件支撑下层的事。
- 随层升高，职责从「创造/支撑应用逻辑」转向「管理对外接口」；相应地，检查深度也从「仿真即可」加深到「必须完整综合 + 布局布线」。
- **可移植性**是四层架构的核心回报：换板只改 Shim，换芯片改 Adapter，Core 不动。
- 非设计的约束应**按类型分文件**管理：位流/配置、逻辑布局、引脚布局、时序、优化五类各成文件，只留极少数写进源码；目的是在 FPGA/PCB 变化时易于管理。
- **源码内约束是分文件原则的有意例外**：像 `ASYNC_REG`、`PRESERVE`、`IOB` 这类「绑定具体寄存器实例的物理放置/优化要求」必须写进源码，因为 `(* ... *)` 属性前置于声明即修饰该声明，能随模块实例化**自动生效**，避免外部约束文件「漏写一行即静默降 MTBF」的风险。
- `CDC_Bit_Synchronizer.v` 与 `Reset_Synchronizer.v` 是这套理念的活样本：两者属性块逐字一致、紧贴 `reg sync_reg` 声明；`Reset_Synchronizer` 甚至为把属性贴到正确寄存器上而宁可内联复制同步器逻辑。

## 7. 下一步学习建议

- **下一个单元 u5（组合逻辑基础构件）** 会进入本书「零件库」的最底层零件，逐个讲解 `Constant`、`Annuller`、`Multiplexer`、`Address_Decoder` 等构件的具体写法——它们正是 Core 层里被反复实例化的「砖块」。本讲建立的四层与约束视角，会让你在 u5 读这些小模块时知道「它们属于 Core、不带任何器件/板属性」。
- 若想立刻验证本讲的源码内约束，建议本地装一个免费版 Vivado 或 Quartus，把 `CDC_Bit_Synchronizer` 单独综合，观察带/不带 `ASYNC_REG` 时工具是否把它识别为同步器、是否拒绝把寄存器放进 I/O 位置。
- 若对 CDC 同步器本身的工作原理（亚稳态、1.5x/3 边沿频率约束、单位同步规则）感兴趣，可先读 `cdc.html`，本手册在 u13（CDC 理论与基础同步）会系统讲解，届时本讲的 `ASYNC_REG` 会与时钟域理论合流。
- 若想看「四层架构里 Adapter 层器件相关逻辑」的实例，可留意本书与具体器件绑定的模块（如 `IDELAYCTRL_Instance.v`、`Synthesis_Harness_Input.v`），体会「器件相关内容被上推到 Adapter、不污染 Core」的实际写法。
