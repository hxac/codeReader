# 亚稳态与时钟域穿越（CDC）基本理论

## 1. 本讲目标

本讲是「时钟域穿越（Clock Domain Crossing，简称 CDC）」单元的第一篇。CDC 是整本书里最需要敬畏的部分——作者在理论正文里直言：你可以放心地设计单时钟的同步逻辑，但一旦设计 CDC，你将「永远充满怀疑」（always be full of doubt）。

读完本讲，你应当能够：

- 说清楚**亚稳态（metastability）**是什么、它为什么会发生、以及为什么不可能被彻底消除。
- 画出 **CDC 基本构建块——位同步器 `CDC_Bit_Synchronizer`** 的结构，并能解释它那「1 到 3 个时钟周期」的不确定延迟从何而来。
- 掌握两条 CDC 铁律：**每次跨越只能同步一个位**，以及保证可靠采样的 **3 边沿 / 1.5× 频率约束**，并会做简单的频率算术。

本讲只讲最基础的三件事：为什么需要同步器、同步器如何工作、它能在多快的信号上工作。至于用同步器搭出复位同步、标志同步、字同步、脉冲同步、FIFO 等，留给 u13-l2 及以后的讲义。

## 2. 前置知识

在进入 CDC 之前，先用大白话对齐几个概念：

- **时钟域（clock domain）**：一组由同一个时钟信号驱动的触发器（flip-flop）。同一个时钟域里的信号，彼此有确定的时序关系，可以放心地互相采样。
- **触发器与边沿采样**：触发器在时钟的某个有效边沿（本书约定为上升沿 `posedge`）「拍照」采样其输入 `D`，把结果保存到输出 `Q`。采样要可靠，输入必须在边沿前后一段窗口内保持稳定，这段窗口的边界分别叫**建立时间（setup time）**和**保持时间（hold time）**。
- **异步（asynchronous）**：当发送端时钟和接收端时钟没有固定相位关系时（频率不同、或同频但相位漂移），称这两个时钟域彼此异步。CDC 之所以难，根源就是「接收端时钟的采样边沿可能恰好落在发送端信号的跳变沿上」，从而违反建立/保持时间。
- **`default_nettype none`、阻塞/非阻塞赋值、寄存器初始化**：这些 u2-l1、u3-l1 已建立的习惯在本讲的源码里仍然成立。
- **源码内约束（`ASYNC_REG`/`PRESERVE`/`IOB`）**：u4-l2 已经讲过「绑定到寄存器实例的物理放置要求必须写进源码」，本讲会看到这套约束在同步器里的具体用途——这是本讲承接 u4-l2 的接口。

> 一句话定位：u4-l2 教你「约束要随声明走」，本讲告诉你「为什么同步器偏偏是这套约束最不能漏写的地方」。

## 4. 核心概念与源码讲解

本讲按三个最小模块展开：**亚稳态 → 同步器延迟与基本构建块 → 频率约束**。三者环环相扣：亚稳态解释「为什么要同步」，同步器解释「怎么同步、代价是什么延迟」，频率约束解释「同步器能处理多快的信号」。

### 4.1 亚稳态

#### 4.1.1 概念说明

正常情况下，数字逻辑工作在**双稳态（bistable）**模式：一个节点要么是高，要么是低。这个假设之所以成立，是因为我们总在「远离跳变沿」的地方去采样信号。只要采样点离信号的高低切换足够远，双稳态假设就成立，一切太平。

**亚稳态（metastable state）**就是「既不高也不低、停在中间」的状态——就像跳变过程中被冻结的那一瞬间。理论正文用一句很直观的话点明要害：

> 越靠近高低之间的中点，亚稳态**消解（resolve）到合法电平所需的时间越长**，也**越无法预知最终会落到 0 还是 1**。

后果是连锁的：下一级逻辑可能继续看到亚稳态、或者消解得太晚，导致下游触发器采样到错误值，系统就这样崩了。

CDC 场景下亚稳态几乎必然出现：发送时钟与接收时钟通常彼此异步（即便同频，假设它们独立也更简单、更稳健），于是接收时钟的某个采样边沿总有可能恰好撞上发送端信号的跳变，违反建立/保持时间，触发亚稳态。

一个必须接受的哲学结论：**亚稳态不可能被彻底消除**。要彻底消除它，意味着状态切换是「瞬时」完成的，而这在物理上不可能。所以我们不追求「消除」，只追求「把亚稳态发生的概率压到足够低、并且不让它传播出去」——这正是同步器要做的事。

#### 4.1.2 核心流程

把亚稳态的产生与危害串成一条因果链：

1. 发送时钟域里某个信号发生 0→1（或 1→0）跳变。
2. 接收时钟域的某个 `posedge` 恰好落在该跳变附近（违反建立/保持时间）。
3. 接收端第一级寄存器的输出进入亚稳态：电压停在 0 和 1 之间。
4. 该亚稳态电压消解所需时间不确定，且最终落点（0 或 1）不确定。
5. 如果它还没消解就被下一级寄存器采样，亚稳态就会**传播**下去，表现为错误的电平或额外的毛刺。

注意第 4 步里的两个「不确定」：消解时间不确定、最终电平不确定。这两个不确定性正是后续「延迟 1–3 周期」和「每次只能同步一位」的根因。

#### 4.1.3 源码精读

亚稳态的定义来自理论正文 [cdc.html:23-45](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L23-L45)。这段讲了双稳态假设的成立条件、亚稳态「越靠中点越难消解、越不确定」的特性，以及最后那句「不可能设计出无亚稳态的电路」。

实现层面，[CDC_Bit_Synchronizer.v:8-15](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L8-L15) 先给出一段警告（A Warning）：同步器电路本身很简单，但它「在尽力收束后果的同时打破了同步逻辑的惯例」，所以极易用错——轻则时序收敛变差，重则出现仿真里看不到、甚至难以复现的间歇性故障。这正呼应了亚稳态「概率性、不可复现」的脾气。

> 后续 4.2 会看到，模块用一串寄存器把亚稳态「困」在第一级、给它时间消解，从而不让它传到下游。谁是那个「可能亚稳」的第一级，等读到 `sync_reg[0]` 时就一目了然。

**需要观察的现象 / 预期结果**：第一级 `sync_reg[0]` 直接吞下异步输入，是唯一会亚稳态的寄存器；它后面还挂着至少一级 `sync_reg[1]`，正是给亚稳态留出的「消解时间」。结论：亚稳态被允许发生在 `sync_reg[0]`，但不被允许传播到 `bit_out`。

#### 4.1.5 小练习与答案

**练习 1**：为什么说「无法设计出完全无亚稳态的电路」？

**参考答案**：无亚稳态意味着信号状态能瞬时切换，而任何物理电平切换都需要非零时间；只要存在跳变，采样边沿就有可能落在跳变过程中，从而产生亚稳态。所以我们只能降低其概率，不能消除。

**练习 2**：亚稳态的「两个不确定」分别是什么？它们各自会导致什么后果？

**参考答案**：① 消解时间不确定——可能导致下游寄存器也采到亚稳态（传播）；② 最终电平不确定——即使消解了，落到的 0/1 也可能不是真实值（采样错误）。

### 4.2 同步器延迟与基本构建块

#### 4.2.1 概念说明

既然亚稳态不可避免，标准的应对之策是 **CDC 位同步器（CDC Bit Synchronizer）**——理论正文称之为「任何 CDC 方案的核心」。它的结构很朴素：

> 一串**两个或更多、彼此紧挨着放置、中间没有任何组合逻辑**的寄存器，统一由**接收时钟**驱动。第一级吞下来自发送时钟域的位，最后一级吐出「同步到接收时钟域」的版本。

关键在于：第一级寄存器可能亚稳态，但我们给它后面多接几级寄存器，让亚稳态在「抵达下一级之前」有充足的时间消解。每多一级，亚稳态继续传播的概率就**指数级**下降。这就是为什么同步器至少要两级——一级是不够的。

代价是**延迟不确定**：尽管同步器有两级寄存器，但从发送时钟的视角看，一个位从进到出要花几个接收时钟周期，是**不可预测**的，会在 1 到 3 个周期之间漂移。

#### 4.2.2 核心流程

理论正文 [cdc.html:59-119](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L59-L119) 列举了 6 种延迟情形。我们用「发送时钟边沿与接收时钟边沿的相对位置 + 是否亚稳态」把它们浓缩成下表：

| 情形 | 相对位置 / 是否亚稳态 | 延迟 |
| --- | --- | --- |
| 1 | 紧挨跳变之后捕获（恰巧满足建立时间，近同步） | 1 周期 |
| 2 | 跳变之后正常捕获 | 2 周期 |
| 3 | 紧挨跳变之前捕获，本拍错过 | 3 周期 |
| 4 | 撞上跳变、亚稳态，但消解到**正确**（跳变后）的值 | 2 周期 |
| 5 | 撞上跳变、亚稳态，消解到**错误**（跳变前）的值，下一拍才正确捕获 | 3 周期 |
| 6 | 亚稳态消解过慢，传染到第二级寄存器 | ≥2 周期，但概率**指数级**低于前几种 |

几种情形可以归并理解：

- **延迟在 1–3 周期之间漂移**，取决于那一瞬间两个时钟的相位关系和亚稳态的运气。
- 情形 6 理论上存在（亚稳态没来得及消解就被第二级采到），但它比第一次亚稳态的概率**指数级**更低，工程上视为「很多很多年才平均发生一次」，被其他失效模式淹没。这就是「多级寄存器」的价值。
- 情形 1 是个 corner case：只有当接收时钟边沿恰好紧随发送时钟边沿、且满足建立时间时才出现；但若两时钟**准同步（plesiochronous，频率相位非常接近）**，这个 corner case 反而可能成为最频繁的情形——这是初学者容易踩的坑。

这条不确定延迟直接推出 CDC 最著名的设计铁律：

> **每次跨越一个时钟域，只能同步一个位。**

原因：如果你并行同步两个位（或同一个位并行同步两次），没有任何保证它们会经历**相同**的延迟。一个可能 1 周期出来、另一个 3 周期出来，于是接收端看到的两根线在时间上错位，原本的「并行数据」就被撕裂了。而只同步一个位时，最坏情况不过是这次跳变被错过、下一拍才捕获——这只会让接收到的位宽多/少一个接收时钟周期，**不会产生错误值**。

#### 4.2.3 源码精读

先看模块端口与深度定义 [CDC_Bit_Synchronizer.v:87-103](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L87-L103)：

```verilog
`default_nettype none
module CDC_Bit_Synchronizer
#(
    parameter EXTRA_DEPTH = 0 // Must be 0 or greater
)(
    input  wire    receiving_clock,
    input  wire    bit_in,
    output reg     bit_out
);
    localparam DEPTH = 2 + EXTRA_DEPTH;
```

要点：

- 文件开头照例 `default_nettype none`（u2-l1 的规矩），把拼错变成编译错误。
- 端口极简：一个接收时钟、一个异步输入位、一个输出位。注意只有 `receiving_clock`——同步器**只属于接收时钟域**，发送端的时序它管不着。
- `EXTRA_DEPTH` 默认 `0`（本书「参数默认 0」的安全栅栏，见 u1-l2），于是 `DEPTH = 2`，即最小两级。需要更高频率、更好 MTBF 时加大它。`DEPTH` 用 `localparam` 由参数算出（u2-l2 的派生常量用法）。

接着是本模块最关键的一段——布局属性 [CDC_Bit_Synchronizer.v:105-124](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L105-L124)：

```verilog
// Vivado
(* IOB = "false" *)
(* ASYNC_REG = "TRUE" *)
// Quartus
(* useioff = 0 *)
(* PRESERVE *)
(* altera_attribute = "-name SYNCHRONIZER_IDENTIFICATION ..." *)
reg sync_reg [DEPTH-1:0];
```

这正是 u4-l2 所说的「源码内约束」。三个要害：

- **`ASYNC_REG = "TRUE"`**（Vivado）告诉工具这些寄存器构成异步同步器，必须**紧挨着放在一起**，并计入 MTBF 报告。同步器的有效性完全依赖「亚稳态在两级之间有足够时间消解」，一旦两级被工具拉开或塞进别处，MTBF 就垮了。
- **`PRESERVE` / `SYNCHRONIZER_IDENTIFICATION`**（Quartus）既禁止工具把这些寄存器优化掉（比如搬进 DSP/BRAM 的输入寄存器），也标记它们是同步器、需紧挨布局。
- **`IOB = "false"` / `useioff = 0`** 禁止使用 I/O 寄存器做同步级，因为 I/O 寄存器离主逻辑阵列太远——这呼应了模块注释 [CDC_Bit_Synchronizer.v:76-85](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L76-L85) 的「Not Usable as I/O Registers」。

然后是数据通路本身 [CDC_Bit_Synchronizer.v:124-147](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L124-L147)：

```verilog
reg sync_reg [DEPTH-1:0];
integer i;
initial begin
    for(i=0; i < DEPTH; i=i+1) begin
        sync_reg [i] = 1'b0;
    end
end

always @(posedge receiving_clock) begin
    sync_reg [0] <= bit_in;                 // 第一级：吞下异步输入（可能亚稳）
    for(i = 1; i < DEPTH; i = i+1) begin: cdc_stages
        sync_reg [i] <= sync_reg [i-1];     // 后续级：给亚稳态留消解时间
    end
end

always @(*) begin
    bit_out = sync_reg [DEPTH-1];           // 只取最后一级作为输出
end
```

逐行解读：

- `sync_reg` 是一个 `DEPTH` 位的寄存器数组，每个元素就是一级同步寄存器。
- `initial` + `for` 把每一级都初始化为 `1'b0`（u2-l1 的「寄存器必须初始化、避免 X 传播」；这里因为位宽由 `DEPTH` 决定，用循环比手写每个位干净）。
- 时钟块用**非阻塞赋值 `<=`**（u3-l1 的时钟块铁律），构成一条移位寄存器：`bit_in → sync_reg[0] → sync_reg[1] → … → sync_reg[DEPTH-1]`。
- 注释「Peel out the first iteration to avoid a -1 index」解释了为什么要先把 `i=0` 单独写成 `sync_reg[0] <= bit_in`：否则循环从 `i=0` 起会写出 `sync_reg[0] <= sync_reg[-1]`，下标 `-1` 非法。剥出首迭代后，循环从 `i=1` 起，正好衔接。
- `bit_out` 用一个组合块 `always @(*)` 把最后一级引出——因为 `bit_out` 声明为 `output reg`（u2-l1：来自本模块局部逻辑时用 `output reg`），且它只是「读最后一级」的连线，没有额外寄存，所以延迟就是 `DEPTH` 拍。

> 把这段和 4.1 的亚稳态因果链对照：`sync_reg[0]` 是唯一可能亚稳的寄存器，`sync_reg[1..]` 全是给它「争取消解时间」的缓冲。`EXTRA_DEPTH` 越大，消解时间越充裕，MTBF 越高，代价是延迟也越长。

最后，「每次只能同步一位」的规则与延迟说明写在 [CDC_Bit_Synchronizer.v:44-51](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L44-L51)：延迟在 1–3 周期间变化，故并行使用多个同步器**不确定**（无法保证同延迟）；要传多位，应同步一个握手信号，其余位在接收域直接捕获（这是 u14-l1 字同步器的主题）。

2. 默认 `EXTRA_DEPTH = 0` 时，`DEPTH` 是多少？`bit_in` 经过几拍出现在 `bit_out`？
3. 若把实例化改成 `EXTRA_DEPTH = 1`，深度变成几？延迟变成几拍？第一级和最后一级分别是哪个 `sync_reg`？
4. 对照 [cdc.html:65-103](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L65-L103) 的 6 种情形，标出哪种情形对应「撞上跳变、亚稳、消解到错误值」。

**需要观察的现象 / 预期结果**：

- 默认 `DEPTH = 2`，`bit_in` 经 2 拍到 `bit_out`（这是「移位寄存器」本身的固定位移；前述 1–3 周期的不确定延迟是相对发送时钟视角、且含亚稳态影响下的有效到达时间，二者并不矛盾——见练习 3）。
- `EXTRA_DEPTH = 1` 时 `DEPTH = 3`，延迟 3 拍；第一级仍是 `sync_reg[0]`，最后一级变为 `sync_reg[2]`。
- 「亚稳、消解到错误值」对应情形 5，延迟 3 周期。

> 本实践为源码阅读型。若想仿真观察，可在测试台里令 `bit_in` 的跳变故意贴近 `receiving_clock` 边沿，但亚稳态的出现是概率性的，标注：待本地验证。

#### 4.2.5 小练习与答案

**练习 1**：为什么同步器至少要两级，一级为什么不行？

**参考答案**：第一级直接采样异步输入，必然有概率进入亚稳态。只有一级的话，亚稳态会直接出现在输出上并传给下游。多加一级（或更多），是给亚稳态留出消解时间；每多一级，亚稳态继续传播的概率指数级下降。

**练习 2**：并行同步两根独立的控制线有什么危险？

**参考答案**：两个同步器各自独立，无法保证经历相同延迟（一个可能 1 周期、另一个 3 周期），导致两根线在接收域时间上错位，原本的并行语义被撕裂。正确做法是只同步一根握手线，另一根在接收域直接捕获。

**练习 3**：移位寄存器明明固定移 `DEPTH` 拍，为什么前面说「延迟 1–3 周期不确定」？

**参考答案**：`DEPTH` 拍是「信号进入第一级后到从末级输出」的固定位移。但「1–3 周期」是站在**发送时钟视角**看的「有效到达时间」——它还包含了「接收边沿恰好在跳变前/后/上」造成的本拍错过或亚稳态消解时间。两者度量对象不同，并不矛盾。

### 4.3 频率约束：1.5× 与 3 边沿

#### 4.3.1 概念说明

前两节回答了「为什么要同步、怎么同步」。本节回答最后一个基础问题：**同步器能处理多快的信号？**

直觉是：输入信号必须「稳得住」足够久，接收时钟才有把握在两次跳变之间把它采到。如果输入是一个一闪而过的窄脉冲，而接收时钟又不够快，脉冲就可能「来过又走了」却从没被采到。

理论正文给出的硬约束是：

> 输入信号的任意一次跳变之后，必须**保持稳定至少 1.5 倍接收时钟周期**，从而保证期间有**三个接收时钟边沿**（posedge-negedge-posedge，或 negedge-posedge-negedge）经过，才能确保被某个 posedge 可靠采样。

换句话说，「3 个接收边沿」是可靠采样的最低保险。

由此得到两条等价的工程结论：

- **单周期脉冲场景**：若输入是发送时钟的一个单周期脉冲（脉冲宽度 = 一个发送周期 \(T_{\text{send}}\)），则接收时钟必须**至少比发送时钟快 1.5 倍**：
  \[
  T_{\text{send}} \geq 1.5 \times T_{\text{recv}} \quad\Longleftrightarrow\quad f_{\text{recv}} \geq 1.5 \times f_{\text{send}}
  \]
- 等价地，发送时钟频率不得超过接收时钟的 \(1/1.5 \approx 0.66\) 倍：\(f_{\text{send}} \leq 0.66 \times f_{\text{recv}}\)。

> **源码阅读小提示**：`CDC_Bit_Synchronizer.v` 第 28–33 行的注释把这里的 0.66 比值写成「receiving clock … at most 0.66x … sending」，方向读起来与上面的不等式相反。请以理论正文 [cdc.html:121-156](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L121-L156) 的 Performance 节和本节的推导为准——接收时钟要比发送时钟**快**（\(\geq 1.5\times\)），0.66 这个数应读作「发送 ≤ 0.66× 接收」。

#### 4.3.2 核心流程

为什么恰好是「3 个边沿 = 1.5 个周期」？拆开看：

- 一个接收时钟周期 \(T_{\text{recv}}\) 里有 **2 个边沿**（一个 posedge、一个 negedge）。
- 所以 **1.5 个周期**里恰好有 **3 个边沿**。这就是「3 边沿」与「1.5× 周期」的等价来源。
- 为什么需要 3 个边沿才能保险？考虑最坏相位：

  1. **边沿序列 posedge-negedge-posedge**：若第一个 posedge 恰好撞上输入跳变（错过或亚稳），第二个（negedge 中间夹着的下一个）posedge 会在第二次跳变之前把信号稳稳采到。
  2. **边沿序列 negedge-posedge-negedge**：此时 posedge 正好位于两次输入跳变的中点，必然能采到。

两种最坏情况都靠「3 个边沿」兜底。少于 3 个，就存在某个相位下 posedge 错过脉冲的窗口。

「1/3 比例」的另一种等价说法（理论正文 Performance 节）：最坏情况下同步器延迟 3 周期 ↔ 每次输入跳变对应 3 个接收边沿 ↔ 一个完整信号周期（两次跳变）对应 6 个接收边沿，其中必然包含 3 个 posedge。

> 这 1.5× 是个不轻的代价。但实践中外设时钟通常比核时钟慢，代价可摊到多位上：先在发送域把多位凑成一个字，再用**一个**同步过去的 valid 位当锁存触发，在接收域直接捕获整个字（无需对数据线同步）。这就是 u14-l1 字同步器的思路。

两个特例也要记住（见 [CDC_Bit_Synchronizer.v:39-42](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L39-L42) 与 [cdc.html:158-176](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L158-L176)）：

- **输入跳变沿太缓**（rise/fall 时间大于接收周期）：接收时钟会对同一次跳变采样多次，输出表现为多个脉冲。此时同步器后要接 Debouncer。
- **同步器输入前有组合逻辑**：汇聚的多条路径会因延迟不同产生毛刺；同步器的异步时钟可能恰好采到毛刺，把它变成真实的错脉冲。所以**同步器输入必须直接来自一个寄存器**，中间不能有任何逻辑。

#### 4.3.3 源码精读

频率约束的权威表述在理论正文 [cdc.html:121-156](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L121-L156)，其中 Performance 节明确写出「3 edges」「1.5x faster」「1/3 ratio」三者的一致关系，并解释了「6 edges total over a cycle → 3 posedges」的算术。

模块侧的约束表述在 [CDC_Bit_Synchronizer.v:17-33](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L17-L33)：

> 输入信号每次跳变后须保持稳定「至少 1.5 倍接收时钟周期」，以保证 3 个接收边沿（posedge/negedge/posedge 或 negedge/posedge/negedge）经过，从而被某个 posedge 可靠采样。

这段以「保持时间 ≥ 1.5 × 接收周期」的形式给出约束，与本节公式完全一致（这是该注释里**正确**、可作为推导起点的部分）。当无法保证脉冲宽度、或不知两端频率关系时，注释指向了 [CDC_Pulse_Synchronizer_2phase.html](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Pulse_Synchronizer_2phase.html)（脉冲同步器，u14-l1）。

「输入须直接来自寄存器、无中间逻辑」的告诫在 [CDC_Bit_Synchronizer.v:62-74](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L62-L74)（Avoid Logic Glitches）与 [cdc.html:158-176](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/cdc.html#L158-L176)（Special Cases）各讲一遍：汇聚逻辑会毛刺，毛刺被异步采样就成了真实的错脉冲，或使本该采到的高电平提前抖低而被错过。

#### 4.3.4 代码实践

**实践目标**：给定收发时钟频率，判断单周期脉冲能否被可靠采样，并算出最坏情况所需的 3 个接收边沿。

**操作步骤**：

设发送时钟 \(f_{\text{send}} = 100\,\text{MHz}\)（\(T_{\text{send}} = 10\,\text{ns}\)），接收时钟有两种候选：\(f_{\text{recv}} = 200\,\text{MHz}\) 或 \(125\,\text{MHz}\)。

1. 对每个候选，计算比值 \(f_{\text{recv}} / f_{\text{send}}\)，判断是否 \(\geq 1.5\)。
2. 用脉冲宽度条件 \(T_{\text{send}} \geq 1.5 \times T_{\text{recv}}\) 复核。
3. 对「不满足」的那个，计算脉冲至少要展宽到几个发送周期才够（即求最小整数 \(k\)，使 \(k \cdot T_{\text{send}} \geq 1.5 \times T_{\text{recv}}\)）。
4. 写出最坏情况下「3 个接收边沿」对应的物理时长。

**需要观察的现象 / 预期结果**：

- **200 MHz**：\(200/100 = 2.0 \geq 1.5\) ✓；\(T_{\text{recv}} = 5\,\text{ns}\)，\(1.5 \times 5 = 7.5\,\text{ns} \leq 10\,\text{ns}\) ✓。可可靠采样。3 个接收边沿跨 \(1.5 \times 5 = 7.5\,\text{ns}\)，塞进 10 ns 脉冲里绰绰有余。
- **125 MHz**：\(125/100 = 1.25 < 1.5\) ✗；\(T_{\text{recv}} = 8\,\text{ns}\)，\(1.5 \times 8 = 12\,\text{ns} > 10\,\text{ns}\) ✗。**不能**保证可靠采样单周期脉冲。
  - 展宽：\(k \cdot 10 \geq 12 \Rightarrow k \geq 1.2\)，取整 \(k = 2\)，即脉冲至少要在发送域维持 2 个周期（20 ns）。
  - 3 个接收边沿跨 \(1.5 \times 8 = 12\,\text{ns}\)；展宽到 2 周期后 20 ns ≥ 12 ns ✓。

> 这是纯算术实践，可在纸上或 REPL 完成，无需综合。结论可直接套用到「外设慢时钟 → 核快时钟」的常见方向（天然满足 1.5×）与反方向（需展宽或改用脉冲同步器）。

#### 4.3.5 小练习与答案

**练习 1**：发送 50 MHz、接收 80 MHz，单周期脉冲能可靠采样吗？

**参考答案**：\(80/50 = 1.6 \geq 1.5\) ✓。\(T_{\text{send}} = 20\,\text{ns}\)，\(T_{\text{recv}} = 12.5\,\text{ns}\)，\(1.5 \times 12.5 = 18.75\,\text{ns} \leq 20\,\text{ns}\) ✓。可以。

**练习 2**：「3 个边沿」为什么等价于「1.5 个接收周期」？

**参考答案**：一个接收周期含 2 个边沿（posedge + negedge），故 1.5 个周期含 \(1.5 \times 2 = 3\) 个边沿。

**练习 3**：为什么同步器的输入必须直接来自寄存器、中间不能有组合逻辑？

**参考答案**：组合逻辑会汇聚多条路径，各路径延迟不同，输出会毛刺。同步器的接收时钟与发送域异步，可能恰好采样到毛刺，把它变成接收域里真实的错误脉冲；也可能让本该采到的高电平提前抖低而被错过。直接由寄存器驱动可避免毛刺，若无法避免则需在同步器后接 Debouncer。

## 5. 综合实践

把三块知识串起来，完成一次「CDC 可行性评估」小任务。

**场景**：你需要把一个外设控制器（发送时钟域，100 MHz）产生的单周期「数据就绪」脉冲，送到核心逻辑（接收时钟域）去触发处理。核心逻辑可运行在 150 MHz 或 200 MHz。

**任务**：

1. **判断方向与可行性**：分别对 150 MHz 和 200 MHz 两种核时钟，用 \(f_{\text{recv}} \geq 1.5 f_{\text{send}}\) 判断单周期脉冲能否被 `CDC_Bit_Synchronizer`（默认 `EXTRA_DEPTH = 0`）可靠采样。
   - 预期：150 MHz → \(1.5 \times 100 = 150\)，刚好卡在临界（\(150/100 = 1.5\)），理论满足但无裕量，工程上不建议；200 MHz → \(2.0 \geq 1.5\)，安全。
2. **定位亚稳态发生处**：打开 [CDC_Bit_Synchronizer.v:137-147](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L137-L147)，指出哪个寄存器可能亚稳态，并说明 [CDC_Bit_Synchronizer.v:115-124](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CDC_Bit_Synchronizer.v#L115-L124) 里的 `ASYNC_REG`/`PRESERVE`/`IOB` 各自为什么不能省（联系 u4-l2）。
   - 预期：`sync_reg[0]` 可能亚稳态；`ASYNC_REG` 让工具把两级紧挨摆放并计入 MTBF，`PRESERVE` 防止寄存器被优化/搬走，`IOB="false"` 禁止用离阵列太远的 I/O 寄存器——三者缺一都会显著拉低 MTBF。
3. **改方案**：对临界的那档（150 MHz），既然裕量不足，给出两个不改核时钟的补救办法。
   - 预期：① 在发送域把「就绪」脉冲展宽到至少 2 个周期（\(2 \times 10 = 20\,\text{ns} \geq 1.5 \times T_{\text{recv}}\)，对 150 MHz \(T_{\text{recv}} \approx 6.67\,\text{ns}\)，\(1.5 \times 6.67 \approx 10\,\text{ns}\)，单周期其实刚好够；但临界无裕量，展宽更稳）；② 改用脉冲同步器（`CDC_Pulse_Synchronizer_2phase`，u14-l1），它不依赖收发频率比。
4. **画时序**：画出最坏相位下「3 个接收边沿」（posedge-negedge-posedge）落在被展宽脉冲内的示意图，标出可靠的采样 posedge。

> 本任务为设计评估型，重在推理与画图，不要求综合实现。综合验证（如用 `Simulation_Clock` 搭测试台）留到 u18-l2。

## 6. 本讲小结

- **亚稳态不可避免**：异步采样可能撞上信号跳变，触发「非 0 非 1」的亚稳态；越靠中点越难消解、越不确定落点。我们只能压低概率，不能消除。
- **CDC 基本构建块是位同步器**：一串至少两级、紧挨摆放、由接收时钟驱动、中间无组合逻辑的寄存器；第一级允许亚稳态，后续级给它消解时间，使亚稳态传播概率指数级下降。
- **延迟不确定（1–3 周期）**：取决于收发时钟相位与亚稳运气；情形 6（亚稳传染到第二级）概率指数级低。
- **铁律：每次只同步一个位**：并行同步无法保证同延迟，会撕裂并行数据；传多位应同步一个握手位、其余直接捕获（字同步器思路）。
- **3 边沿 / 1.5× 频率约束**：输入每次跳变后须稳 \(\geq 1.5 \times T_{\text{recv}}\)（即 3 个接收边沿）；单周期脉冲要求 \(f_{\text{recv}} \geq 1.5 f_{\text{send}}\)。
- **源码内约束不能省**：`ASYNC_REG`/`PRESERVE`/`IOB` 必须写在源码里（u4-l2 的原则在同步器上的具体落地），否则 MTBF 崩塌；输入还须直接来自寄存器以避毛刺。
