# 仓库结构与开箱即用约束

## 1. 本讲目标

上一篇（u1-l1）解决的是“这本书怎么读、怎么找东西”。本篇往下钻一层，解决“**这个仓库在地上是怎么摆的、有什么看不见的规矩**”。读完后，你应当能够：

- 说清楚仓库的**扁平单目录结构**：几乎所有文件都平铺在根目录，每种后缀（`.v` / `.vh` / `.html` / `.py` / `.css` / `.md`）各司其职。
- 解释**“一模块一文件、文件名等于模块名”**这条约定，以及它为什么是整本书的“承重墙”——`v2h.py` 正是依赖它，才能让 `.v` 与 `.html` 一一对应、并把实例名自动变成超链接。
- 说出本书最反直觉、却最关键的一条工程约束：**所有模块参数的默认值都是 `0` 或空串**，因此模块“按定义无法直接综合”，必须实例化并设置参数后才能用，并且这是**有意为之**。

这一篇仍然不深入 Verilog 语法细节，但会带你读真实的 `README.md`、`v2h.py`、`Register.v` 和 `generate_file_skeleton.py`，把“目录结构、命名约定、参数默认值”这三件事落到源码上。

## 2. 前置知识

本讲承接 u1-l1，默认你已经知道：

- 本书是“一本在线书 + 一套 FPGA 电路零件库”，作者比作“硬件版的 libc”。
- 阅读入口是 `index.html`，每个模块页顶部有 **Source / License / Index** 三个链接。
- 网页是由 Verilog 源码经 `v2h.py` 渲染出来的，注释即正文。

再补充几个本讲要用到的小概念：

- **综合（synthesis）**：把 Verilog 代码翻译成 FPGA 上真实电路（查找表 LUT、触发器、布线）的过程。代码写得不对，综合阶段就会报错。
- **elaboration（精化/展开）**：综合的前一步，工具先把模块层次展开、把参数代进去、算出每个信号的位宽。位宽算不出来或算成 0，往往在这里就炸了。
- **lint（静态检查）**：不生成电路，只检查代码本身有没有可疑写法（位宽不匹配、悬空信号等）。本书用 `verilinter` 做这件事。
- **参数（parameter）**：Verilog 里模块的“编译期常量”，常用来定义端口位宽。实例化时可以覆盖它。
- **实例化（instantiation）**：把一个模块“放进”另一个模块里使用，类似编程里调用函数时要传参数。

如果你对“位宽”还没直觉，先记住一句话：**本书每个模块的端口有多宽，都由参数决定；参数没设，端口就“没有宽度”，工具会拒绝它。** 这正是本讲要讲清楚的核心约束。

## 4. 核心概念与源码讲解

本讲拆成三个最小模块：**目录结构**、**文件命名约定**、**参数默认值约束**。

### 4.1 目录结构

#### 4.1.1 概念说明

绝大多数软件项目都会分门别类地建子目录：`src/` 放源码、`docs/` 放文档、`tests/` 放测试、`build/` 放产物。本书**几乎完全不这么做**——它把所有东西平铺在**同一个根目录**里，源码、生成的网页、工具脚本、样式表、说明文档混在一起。

这并不是“不会组织”，而是**有意为之**，目的是让本书可以直接当“零件库”用：

> 你只要把所有 Verilog 文件一次性导入（import）进你的 CAD 工具，就拥有了整本零件库，不必再去配置一堆路径。

这种“扁平化”换来的是“开箱即库”的便利，代价是根目录文件数量很大（三百多个）。理解了这一点，你就不会在根目录里去找 `src/`——因为它根本不存在。

#### 4.1.2 核心流程

仓库由若干类文件组成，每类后缀职责固定：

| 后缀 | 数量（当前 HEAD） | 职责 | 谁产生的 |
| --- | --- | --- | --- |
| `.v` | 130 | Verilog 模块源码（本书的“正文”） | 作者手写 |
| `.vh` | 10 | Verilog 头文件（可复用函数、运算表） | 作者手写 |
| `.html` | 154 | 渲染后的网页（每个 `.v`/`.vh` 对应一页，外加少量手写概念页） | `v2h.py` 由源码生成 / 少量手写 |
| `.py` | 5 | 辅助工具（渲染、骨架生成、实例生成等） | 作者手写 |
| `.css` | 1 | 网页样式（`style.css`） | 作者手写 |
| `.md` | 2 | 说明文档（`README.md`、`CONTRIBUTING.md`） | 作者手写 |
| 无后缀 / 其他 | 少量 | 如 `verilinter` 脚本、`favicon.ico`、`FPGA_Design_Elements.core` 等 | 作者手写 |

一个值得注意的现象：**`.html`（154）比 `.v`（130）还多**。原因是网页里有一批**手写的概念页**（如 `verilog.html` 编码规范、`system.html` 系统设计标准、`handshake.html` 握手规则、`cdc.html` 跨时钟域理论、`fsm.html` 状态机、`index.html` 目录等），它们并不来自某个 `.v`；同时 10 个 `.vh` 也会各自渲染出一页（如 `clog2_function.html`）。所以“源码文件 → 网页”大致是一对一，但网页总数略多。

#### 4.1.3 源码精读

README 里那句关于布局的说明，是理解整个仓库的钥匙：

[README.md:L16-L17](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/README.md#L16-L17) —— 明确写着“所有文件都在同一个目录里，因此你可以把它当作一个库，在 CAD 工具里直接导入全部 Verilog 文件来使用”。

这句话解释了“为什么扁平”：**扁平 = 可整体导入 = 零件库**。子目录会破坏“一把导入”的简单性。

需要诚实补充的一点：仓库**并非 100% 没有子目录**。根目录之外存在一个很小的 `tests/` 子目录，里面放着示例测试台（如 `tests/Counter_Gray_Tb.py`、`tests/Counter_Gray_Tb.sv`）。也就是说：

- **零件库本身（所有 `.v` / `.vh` 模块）严格扁平**——这部分与 README 的描述完全一致。
- **示例测试台放在 `tests/` 下**——这是一个很小的例外，不影响“把所有 Verilog 一把导入”的用法。

所以当你听到“所有文件同目录”时，心里要把它理解成“**所有模块文件同目录**”，而测试台是个独立的角落。本系列后面的 u18-l2（仿真与测试台）会专门讲 `tests/`。

### 4.2 文件命名约定

#### 4.2.1 概念说明

本书有一条**铁律级**的命名约定：

> **一个文件里只定义一个模块，且文件名（去掉后缀）就等于模块名。**

例如 `Register.v` 里定义的就是 `module Register`；`Multiplexer_One_Hot.v` 里定义的就是 `module Multiplexer_One_Hot`。

这条约定表面上是“整洁”，实际上它是**整本书渲染机制的地基**。`v2h.py` 在把代码包进网页时，会扫描代码里的每一个“词”，如果某个词正好对应一个已存在的 `.html` 文件，就把它变成超链接——这样当你在一个模块里实例化了另一个模块时，那个实例名在网页上就是可点的，点进去直达被调用模块的页面。而这一切之所以能成立，正是因为“文件名 = 模块名 = 网页名”。

换句话说，命名约定不是风格偏好，而是被渲染器**当成 API 在用**的硬约束。

#### 4.2.2 核心流程

`.v` 变成 `.html` 的流水线（由 `v2h.py` 实现）大致是：

```
读入 foo.v
  ├─ 连续以 "//" 开头的行  → 拼成一块 → 当作 Markdown 渲染成正文
  ├─ 其余非空行            → 当作代码 → 包进 <pre> 代码块
  │     └─ 代码里的每个词：若存在 ./词名.html，就把该词替换成超链接
  ├─ 顶部插入 Source / License / Index 三个链接（来自固定 header）
  └─ 输出 foo.html（仅当内容与现有文件不同时才真正写盘）
```

由此得到几个**一一对应**关系，它们是本讲练习任务的核心：

| 关系 | 说明 |
| --- | --- |
| `Foo.v` ↔ `module Foo` | 一个源文件定义一个同名模块 |
| `Foo.v` ↔ `Foo.html` | 一个源文件渲染成一页同名网页 |
| `Foo.html` ↔ `index.html` 里的链接 | 目录页里 `<a href="./Foo.html">Foo</a>` 指向该网页 |
| 代码里写 `Foo ...` 实例 ↔ `Foo.html` | 渲染时实例名自动变成可点链接 |

#### 4.2.3 源码精读

`v2h.py` 开头的 docstring 把这条约定讲得很直白：

[v2h.py:L10-L13](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/v2h.py#L10-L13) —— 说明“非注释行作为代码包进 `<pre>`，并且**只要存在与某个模块同名的 HTML 文件，代码里出现该模块名时就会被替换成链接。这依赖于‘一模块一文件、文件名匹配模块名’的约定**”。

紧接着一句点明了这样做的回报：

[v2h.py:L15](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/v2h.py#L15) —— “这样就能让 Verilog 代码、它的文档、它的呈现三者始终同步。”

再看把“实例名变链接”落到代码里的关键函数 `add_file_links`：

[v2h.py:L108-L122](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/v2h.py#L108-L122) —— 它把一行代码按空白拆成词，对每个词构造 `./词名.html`，**只要该文件真实存在**，就把代码里的这个词替换成 `<a href="./词名.html">词名</a>`。这正是“文件名=模块名”约定被消费的地方：没有这条约定，工具就无法仅凭一个词判断它是不是模块名。

文件名与模块名、网页名之间的转换，由三个小函数固定下来：

[v2h.py:L54-L64](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/v2h.py#L54-L64) —— `base_filename` 去掉 `.v`/`.vh` 后缀；`output_filename` 在其后加上 `.html`；`filename_to_title` 把下划线换成空格作为网页标题。三者合起来保证了 `Register.v` → 模块 `Register` → 网页 `Register.html` → 标题 “Register” 的一致性。

每个生成的网页顶部那三个固定链接，来自脚本里的 `header` 模板：

[v2h.py:L40-L42](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/v2h.py#L40-L42) —— 依次是 **Source**（指向当前 `.v` 源码）、**License**（指向 `legal.html`）、**Index**（指向 `index.html`）。这正好对应 u1-l1 里让你识别的那三个链接——它们不是手写的，而是由 `v2h` 自动加在每一页上的。

最后还有一个很体贴的设计：**只有当生成的内容与原有文件不同时才写盘**：

[v2h.py:L175-L179](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/v2h.py#L175-L179) —— 比较新生成的 HTML 与磁盘上已有文件，不同才覆盖。这让“批量重新渲染整个仓库”不会产生无意义的改动，git diff 保持干净。

这套机制也决定了贡献一个新模块的标准流程。`CONTRIBUTING.md` 把它浓缩成四步：

[CONTRIBUTING.md:L3-L7](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/CONTRIBUTING.md#L3-L7) —— (1) 编辑/新增 `.v` 文件；(2) 用 `v2h.py` 跑出对应的 `.html`；(3) 如有必要在 `index.html` 里加一个指向新页面的链接；(4) 提交 Pull Request。

注意第 (3) 步“如有必要在 index.html 加链接”——目录页是手写的，新增模块后需要人工把链接挂上去；这从侧面说明 `index.html` 不是自动生成的，而那几百个模块页则是自动生成的。

#### 4.2.4 代码实践

**实践目标**：亲手用 `v2h.py` 走一遍“`.v` → `.html`”的渲染，观察命名约定如何生效。为避免改动仓库里的真实文件，我们在 `/tmp` 里做。

**操作步骤**：

1. 复制一个现有模块到临时文件名（注意：临时文件名要避免与仓库里已有模块重名）：

   - 正文注释是否被渲染成了段落（Markdown）。
   - 代码块里凡是指向 `Register_areset` 之类的地方，是否变成了可点链接（因为根目录里存在同名 `.html`）。

- `MyDemoReg.v` 渲染出同名 `MyDemoReg.html`，验证“文件名→网页名”。

**预期结果**：你会看到注释变正文、代码变代码块、实例名变链接。如果运行时报缺少 `markdown` 模块，先 `pip install markdown` 再试（**待本地验证**你的 Python 环境是否预装）。

#### 4.2.5 小练习与答案

**练习 1**：假如你新建了一个模块 `Foo_Bar`，但把文件误存成了 `FooBar.v`（少了下划线），会发生什么？

**参考答案**：模块名 `Foo_Bar` 与文件名 `FooBar` 不一致，破坏了“文件名=模块名”约定。后果是：(1) `v2h` 生成的网页叫 `FooBar.html`，与模块名对不上；(2) 别的模块在代码里写 `Foo_Bar ...` 实例时，`add_file_links` 找不到 `./Foo_Bar.html`（只有 `FooBar.html`），于是**不会生成链接**，跨模块跳转失效。所以文件名必须与模块名逐字符一致。

**练习 2**：为什么 `index.html` 里的链接是 `./Register.html`，而不是 `./Register.v`？

**参考答案**：因为人类阅读的是渲染后的网页。`Register.html` 是由 `Register.v` 经 `v2h` 生成的、带注释讲解和超链接的阅读页；`.v` 才是原始源码。点网页看讲解、点页面顶部 Source 回到源码，是设计好的两层入口。

### 4.3 参数默认值约束

#### 4.3.1 概念说明

这是本讲最重要、也最反直觉的一条规矩：

> **本书所有模块的参数，默认值都是 `0`（或空串）。除非某个设计真的要求某个最小值或常量值。**

也就是说，你拿到一个模块直接综合，**几乎一定会失败**。这不是 bug，而是**故意设计的安全栅栏**。

作者的逻辑很务实：如果给参数填上“看起来合理”的默认值（比如 `WORD_WIDTH = 32`），那么使用者一旦忘记在实例化时设置它，工具不会报错——于是那个本该是 8 位的信号被默默当成 32 位，bug 就悄悄溜进设计里，而且很难发现。

反过来，把默认值设成 `0`：忘记设参数 → 位宽变 0 → 综合/lint 直接报错 → 你立刻知道“漏了参数”。**用“必定失败”来换取“绝不静默出错”**。这是一条典型的“让错误尽早、尽声响”的工程哲学。

代价是：模块“按定义不可综合”，**必须先实例化、在实例化时把参数填上**，才能真正使用。好在“实例化时设参数”本来就是正常用法，所以这个代价几乎可以忽略。

#### 4.3.2 核心流程

以最典型的位宽参数 `WORD_WIDTH` 为例，端口往往写成：

```verilog
input  wire  [WORD_WIDTH-1:0]  data_in,
output reg    [WORD_WIDTH-1:0]  data_out
```

如果实例化时**设置了** `WORD_WIDTH = 8`，那么范围是 `[7:0]`，8 位，一切正常。

如果**忘记设置**，`WORD_WIDTH` 取默认值 `0`，范围变成：

\[
\text{bit range} = [\,\text{WORD\_WIDTH}-1 : 0\,] = [-1 : 0]
\]

`[-1:0]` 是一个非法/零宽度的向量声明（高位下标小于低位下标）。在 elaboration 阶段，工具要么直接报“非法位宽/零宽度”，要么把这个端口当成不存在，随后引发一连串位宽不匹配。最终结果正如 README 所说：**综合几乎总是失败，lint 也会失败**。

于是流程被强制成：

```
想用某模块
  → 必须先实例化它（不能直接综合模块本体）
     → 实例化时把 WORD_WIDTH 等参数显式设成需要的值
        → 现在端口有了正确位宽，才能综合通过
```

#### 4.3.3 源码精读

这条约束的权威出处是 README 里那段以 **IMPORTANT** 开头的说明：

[README.md:L19-L26](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/README.md#L19-L26) —— 直说“模块定义按设计就不能直接使用”；“所有模块参数默认值为 0 或空串，这是有意的”；“这样当使用者在实例化时忘记设参数，综合（几乎总会）失败，lint 也会失败”；“在模块定义里放可用的默认值可能不会被发现，从而引发 bug”；“这意味着模块按其定义是不可综合的，必须单独实例化以设置参数——而这本来就是正常做法”。

再看一个真实模块 `Register.v`，它就是这条规矩的标本：

[Register.v:L41-L52](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/Register.v#L41-L52) —— 模块头里两个参数 `WORD_WIDTH = 0` 与 `RESET_VALUE = 0` 都默认为 0；端口 `data_in` / `data_out` 的位宽都是 `[WORD_WIDTH-1:0]`。若不设 `WORD_WIDTH`，这两个端口就是 `[-1:0]`，无法综合。

连复位初值也遵循“默认 0”：

[Register.v:L54-L56](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/Register.v#L54-L56) —— `initial begin data_out = RESET_VALUE; end`。`RESET_VALUE` 默认 0，所以即便不显式设，至少初值是合法的 0；但 `WORD_WIDTH` 仍必须由使用者设好，否则端口位宽本身就不成立。

这条规矩不仅是某个老模块的遗留写法，而是**新模块的官方模板**。看骨架生成器 `generate_file_skeleton.py` 输出的新模块模板：

[generate_file_skeleton.py:L12-L24](https://github.com/laforest/FPGADesignElements/blob/2450a548eeee2a4dc1c06878b7c00617dd94e2a8/generate_file_skeleton.py#L12-L24) —— 模板里赫然写着 `parameter WORD_WIDTH = 0`，并且用 `localparam WORD_ZERO = {WORD_WIDTH{1'b0}};` 在模块内部构造一个与位宽匹配的零常量。这告诉我们：**贡献新模块时，参数默认 0 是必须遵守的约定**，而不是可选项。

> 小贴士：模板里 `localparam WORD_ZERO = {WORD_WIDTH{1'b0}};` 用的是“复制拼接”惯用法——把 `1'b0` 复制 `WORD_WIDTH` 份再拼起来，得到一个宽度恰为 `WORD_WIDTH` 的全零常量。这个技巧会在 u2-l2（参数化与位宽处理）详细讲；这里只需知道它是为了在“位宽由参数决定”的世界里，安全地写出一个“与端口同宽的 0”。

#### 4.3.4 代码实践

**实践目标**：亲手制造一次“忘记设 `WORD_WIDTH`”的失败，再修复它，从而把这条约束体验到位。

**操作步骤**：

1. 写一个最小顶层（**示例代码，非项目原有文件**），**故意不设** `WORD_WIDTH`：
   ```verilog
   // 示例代码：演示忘记设参数的后果
   `default_nettype none
   module demo_bad;
       wire clock = 1'b0;
       wire [7:0] data_in = 8'h00;
       wire [7:0] data_out;

       // 没有 .WORD_WIDTH(8) —— 取默认值 0
       Register inst
       (
           .clock(clock),
           .clock_enable(1'b1),
           .clear(1'b0),
           .data_in(data_in),
           .data_out(data_out)
       );
   endmodule
   ```
2. 用 lint 工具（或你手头的综合工具）检查它：
   ```bash
   # 若已安装 verilator（本书 verilinter 的后端之一）
   verilator --lint-only demo_bad.v Register.v
   ```
3. 把实例化改成**显式设参数**，再 lint 一次：
   ```verilog
   Register #(.WORD_WIDTH(8), .RESET_VALUE(8'h00)) inst ( ... );
   ```

**需要观察的现象**：

- 第 2 步：报错。错误信息多半与位宽有关——端口 `data_out` 被声明成 `[WORD_WIDTH-1:0]`，而 `WORD_WIDTH=0` 导致范围非法/零宽度；同时 8 位的 `data_in` 连到零宽度端口会触发位宽不匹配。
- 第 3 步：报错消失，端口变成 8 位，连接合法。

**预期结果**：你将亲身体验“忘记设参数 → 工具立刻报错”，这正是作者想要的安全栅栏。不同工具的报错措辞不同，故具体错误文本**待本地验证**；但“必定报错”这一行为是 README 保证的。

> 想深入：把 `WORD_WIDTH` 设成 `1`，观察端口变成 `[0:0]`（合法的 1 位向量），体会 `WORD_WIDTH` 从 0→1→8 时位宽如何变化。位宽与参数的完整惯用法见 u2-l2。

#### 4.3.5 小练习与答案

**练习 1**：为什么不干脆给 `WORD_WIDTH` 一个“合理默认值”比如 32，省得每个实例都要写？

**参考答案**：因为“合理默认值”会在使用者**忘记设参数**时**静默生效**——一个本该 8 位的信号被当成 32 位，工具不报错，bug 却已潜入，且极难排查。默认 0 则把“忘记设参数”变成“必定综合失败”，错误立刻暴露。作者用“吵闹的失败”换取“绝不静默出错”。

**练习 2**：`Register.v` 里 `RESET_VALUE = 0` 也是默认 0，为什么它不会像 `WORD_WIDTH = 0` 那样让综合失败？

**参考答案**：`RESET_VALUE` 是一个**数值**初值，0 是完全合法的复位值；而 `WORD_WIDTH` 是用来计算**位宽**的，`WORD_WIDTH=0` 会让端口范围变成非法的 `[-1:0]`。两者虽然都默认 0，但一个影响“初值”（合法），一个影响“位宽”（非法），后果截然不同。

**练习 3**：假如某个模块确实需要某个参数恒为常量（例如固定 4 路输入），按 README 的说法该怎么做？

**参考答案**：README 说“除非设计本身要求某个最小值或常量值”。也就是说，如果一个参数在物理上就是固定常量（比如反串行化的固定比率），那么可以写死它而不必默认 0；只有那些“由使用者决定、不能猜”的参数（典型如 `WORD_WIDTH`）才默认 0。判断标准是：**这个值能不能被合理猜出——能猜且猜错会出 bug 的，就默认 0 强制要求显式设置。**

## 6. 本讲小结

- 仓库**几乎所有文件平铺在根目录**，目的是“可整体导入为零件库”；唯一的子目录是小小的 `tests/`（示例测试台）。
- 后缀职责固定：`.v`/`.vh` 是源码，`.html` 是渲染网页（含少量手写概念页），`.py` 是工具，`.css`/`.md` 是样式与说明；`.html` 比 `.v` 多，是因为存在手写概念页与 `.vh` 渲染页。
- **“一模块一文件、文件名=模块名”是承重墙约定**：`v2h.py` 的 `add_file_links` 据此把代码里的实例名自动变成跨模块超链接，并保证 `.v` ↔ `.html` 一一对应。
- 每个模块页顶部的 **Source / License / Index** 三链接由 `v2h` 的 `header` 模板自动注入，并非手写；目录页 `index.html` 则是手写、需人工挂链接。
- **所有参数默认值为 0/空串是有意设计**：忘记设参数 → 位宽非法 → 综合/lint 必定失败，从而杜绝“静默用错位宽”的隐患。
- 因此模块**按定义不可直接综合，必须实例化并设置参数**才能使用；贡献新模块时，`generate_file_skeleton.py` 的模板已经把这条约束固化进去。
