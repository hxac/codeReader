本讲精读 `benchmark_inference/` 下两个基准推理脚本，确立其职责边界：**只负责让模型把 Verilog 写出来，评分（编译、仿真、统计通过率）交给外部官方基准仓库**，因此该目录是「跨基准的统一生成层」。两脚本共享同一骨架：`argparse` 参数 → 以 `fp16 + device_map + 左填充`加载模型 → 循环 `model.generate` → 复用 u1-l5 的 Mistral 版抽取（`rsplit('endmodule')` 截断 + 剔除 testbench）→ 落盘，`gen_batch_size` 均写死为 1。公共参数有 `model/temperature/gpu_name/n`（n 即 pass@k 的 k）。

`test_on_verilog-eval.py` 对接 NVlabs/verilog-eval：把「描述文件」与「题目文件」两路数据按 `task_id` 线性对齐，拼成 `detail_description + prompt`，用「列表复制 n 份」拍平成大列表产生候选，结果追加进单个 JSONL；独有 `--bench_type`（Machine/Human）与 `--output_file`。`test_on_rtllm.py` 对接港科大 RTLLM，读自带的 `rtllm-1.1.json`（`Instruction + Input`），用「外层循环 n 次 + `test_i/` 目录」编码候选，靠内置 `design_list` 的子串匹配 + 命中即停把每段代码落成 `{keyword}.v`。本讲重点标记两处源码问题：VerilogEval 抽取得到的代码 `s` 实际未被写盘（生成结果被丢弃，需补 `dic['completion']=s`）；RTLLM 子串匹配对短关键字（如 `alu`/`accu`）与列表顺序敏感，存在误分类风险。本讲为进阶层 u2 收尾。
