本讲拆解 RTL-Coder 数据生成流程的工具箱 `data_generation/utils.py`，含三个最小模块。其一，`askGPT35` 是全流程唯一与 OpenAI 通信的函数，靠 `is_response` 布尔开关复用于两阶段：`False`（默认，造题目）给空 system 提示，`True`（造代码）给 `"I want you act as a Professional Verilog coder."`，两阶段仅 system 消息不同；shipped 主循环目前只走 `is_response=False`。其二，重试循环只对 `'maximum context'` 错误计数并降级——每次将 `max_gen_tokens = int(max_gen_tokens / 1.3)`（约 2048→1575→1211→931），砍满 5 次放弃并返回 `[{finish_reason:'length', text:''}]`；其余错误只 `sleep(2)` 无限重试，且 "rate limit" 日志对所有 `OpenAIError` 都打印、具误导性。其三，`load_json`/`save_json` 采用 JSONL（一行一个 JSON）格式逐行读写，`save_json` 用 `'w'` 每轮整体重写文件以支持断点续跑，`load_json` 流式低内存。关键契约：`askGPT35` 始终返回「单元素列表、元素含 `text` 和 `finish_reason`」的统一形状，放弃时的 `length` 标记被下游 `post_process_gpt3_response_uniq` 解析为空列表优雅丢弃。此外，`engine=model` 配合 `openai==0.28.0` 表明代码面向 Azure OpenAI、用旧版 SDK 写就，迁至 `openai>=1.0` 需改调用入口与异常类。该统一返回形状使 `askGPT35` 可被任意返回同结构的本地模型替换。
