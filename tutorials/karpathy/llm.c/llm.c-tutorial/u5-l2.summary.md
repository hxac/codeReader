本讲讲解支撑 `train_gpt2.cu` CUDA 主线的基础设施层（`cuda_common.h`、`cuda_utils.cuh`、`cublas_common.h`），不展开具体模型层 kernel。

错误检查与启动：`cudaCheck`/`cudaFreeCheck`/`cublasCheck` 采用「内联函数 + 带 `__FILE__`/`__LINE__` 的包装宏」两段式，出错即打印定位信息并 `exit`，绝不带病运行。工程里没有自定义的 kernel 启动包装宏，而是用原生 `<<<grid, block, smem, stream>>>` 三尖括号 + `CEIL_DIV(M,N)` 向上取整宏算 grid，必要时启动后调 `cudaGetLastError()` 抓启动期错误（kernel 内部异步错误须等同步才暴露）。

多 stream 调度：单卡只有一条 `main_stream`，所有 kernel 与异步拷贝都排在其上；多卡才引入第二条流 `nccl_stream`，专跑 NCCL all-reduce，靠 `compute_nccl_sync` event（`cudaEventRecord`/`cudaStreamWaitEvent`）与计算流建立「算完梯度再规约」的最小依赖，争取计算-通信并发。

cuBLASLt：遵循「创建一次、全局复用」。`cublaslt_handle` 与 32 MiB `cublaslt_workspace` 在 `common_start` 创建、`common_free` 销毁、所有 matmul 共享。关键区分数据类型 `CUBLAS_LOWP`（随精度宏变，与 `floatX` 同源）和计算类型 `cublas_compute`（默认 fp32 累加，仅 fp32+Ampere+ 才切 TF32）。

该工具层是各模型层 `.cuh` 共享的「螺丝刀」，承接 u5-l1，为 u5-l3（matmul 详解）与 u6（混合精度、多卡）铺底。
